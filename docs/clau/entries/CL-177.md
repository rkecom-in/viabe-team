# CL-177 — CampaignPlan v0.1 contract locked

## Contract

```python
class CampaignPlan(BaseModel):
    tenant_id: UUID
    subscriber_id: UUID
    template_id: str          # internal template name, e.g. 'team_winback_v1'
    body_params: dict[str, str]   # template variable values
    status: Literal['proposed', 'approved', 'rejected', 'sent', 'failed'] = 'proposed'
    proposed_at: datetime
    proposed_by: str          # agent identifier, e.g. 'sales_recovery_agent'
```

## Lifecycle

- `proposed` — specialist produced the plan; awaiting owner approval
- `approved` — owner approved via Owner Surface
- `rejected` — owner rejected
- `sent` — successfully sent via Twilio
- `failed` — send failed (retryable per VT-3.6)

Agents only emit `proposed`. Owner approval (VT-6) flips to approved/rejected. Sender (VT-5) flips to sent/failed.

## Why these 7 fields and not 6

- `body` → `body_params`: WhatsApp templates take a variable map, not free body text. Avoids contract change once VT-4 ships.
- Status enum expanded beyond `'proposed'`: billing reconciliation + owner approval UI both need non-proposed states. Shipping with just `'proposed'` forces a migration in the next PR.
- `proposed_by` added: Phase 2 has 5 specialists. Without attribution we lose audit trail. Retrofitting requires backfill.

## Storage

Pydantic model lives in apps/team-orchestrator/orchestrator/types/campaign_plan.py.

`campaigns` table schema mirrors this model. Migration ships in VT-4, not VT-3.4 PR 1/3. PR 1/3 produces the Python model and tests that the stub specialist returns a valid CampaignPlan instance.

## Downstream consumers

- VT-4 (Sales Recovery Agent) — produces CampaignPlan instances
- VT-6 (Owner Surface) — displays plans for approval, flips status
- VT-5 (Outbound MCP tools) — reads approved plans, sends, flips status
- Billing reconciliation — reads sent/failed status for ARRR attribution

All four read this contract. Locking it here in PR 1/3 prevents drift.

## Extension policy

The contract is locked at v0.1. Adding fields = Type 2 (joint Clau + Fazal). Removing fields = Type 2. Status enum changes = Type 2. Field type changes = Type 2.

VT-4 may propose v0.2 with additional fields. When that PR lands, this contract gets a SUPERSEDED banner per discipline rule #9.

**SUPERSEDED — 2026-05-20.** This CampaignPlan v0.1 7-field model was a plumbing stand-in for VT-3.4's handoff, written before the Sales Recovery agent existed. The canonical CampaignPlan contract is now **VT-37** (CampaignPlan v1.0 structured-output schema), merged via PR #32. VT-37 is a discriminated union over `status` with three agent-terminal states (`proposed` / `out_of_scope` / `insufficient_data`).
