## ✅ FIXED 2026-07-08 — TWO compounding bugs, both landed + verified on deployed dev

The manager's intermittent verbatim-repeat (byte-identical reply to a repeated owner message,
~2/3 of impatient_repeat runs) had TWO root causes. Both are fixed; measured impatient_repeat on dev
went **~67% → 0/8 shipped repeats**. The audit spine (tm_audit_log) was decisive — thanks Fazal.

### Root cause 1 — shared ConnectionPool starvation (the brain loses context)
`graph.py` built ONE `ConnectionPool(max_size=4)` shared between the LangGraph **PostgresSaver
checkpointer** (holds connections for checkpoint I/O across every graph superstep) AND all ~8
`tenant_connection` reads per dispatch turn (l1/business/directive/lessons/intent/conversation/
onboarding/inflight + the anti-repeat guard). Under checkpointer contention a tenant read
intermittently fail-softed to `[]`, so the brain composed without its own prior turns → repeat.
- **Fix A (graph.py):** `min_size 1→2, max_size 4→10`. Measured: repeat rate **67% → 25%**.
- **Fix B (conversation_log.active_window):** retry a TRANSIENT read error up to 3× with brief
  backoff before fail-softing (a genuine empty window still returns on attempt 1, no added latency).
  Hardens both the brain's context read and the guard's read against a contention blip.
- **Why the earlier RLS-GUC theory was wrong:** the local repro used a DEDICATED pool (no
  checkpointer sharing) → 40/40 clean. The shared-pool + checkpointer contention was the missing
  variable. The tenant-isolation boundary (`tenant_connection` session GUC) was never the bug; the
  proposed SET-LOCAL change would have been a wrong, RLS-risking edit — reproducing first avoided it.

### Root cause 2 — the VT-616 anti-repeat guard NEVER fired on long replies (truncation mismatch)
`record_turn` caps stored `conversation_log` turns at `_TEXT_CAP` (~1000 chars). The guard
(`_reply_repeats_recent`) compared the **FULL untruncated** candidate against the **truncated**
stored prior via `difflib.SequenceMatcher.ratio()`. For any reply past the cap the length mismatch
alone dropped the ratio below the 0.90 threshold — **measured on dev: 1592-char candidate vs
995-char stored prior = 0.77; 2406 vs 994 = 0.58 — dup=False on BYTE-IDENTICAL repeats.** So the
deterministic backstop was blind to exactly the long repeats it existed to catch (fired 0/8).
- **Fix C (dispatch._reply_repeats_recent):** compare on the **common-length prefix**
  (`min(len_cand, len_prior)`) so a truncated prior still matches the full reply it was cut from.
  Byte-identical long repeat → ratio ~1.0 → guard fires → `_compose_progression_reply` recomposes a
  divergent reply; genuinely different reply → prefixes differ → not flagged. +1 unit test (9/9).

## Verification (deployed dev, impatient_repeat ×8 after the full stack)

read reliable). Net **zero verbatim repeats shipped.**
