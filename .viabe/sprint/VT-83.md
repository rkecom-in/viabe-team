## What CoderC must produce
The weekly approval UX. Wraps VT-5.9's pause primitive in the specific UX of weekly campaign approvals. The agent's `CampaignPlan` becomes a structured WhatsApp message; the owner replies; the reply resolves the approval; the campaign moves forward.
1. **Approval message construction** at `apps/team/src/owner_surface/weekly_approval.py`:
	- Function `compose_approval_message(campaign_plan: CampaignPlan, business_profile: BusinessProfile) -> ApprovalMessage`.
	- Returns a message that uses the `weekly_approval` Meta template + structured params:
		- Cohort summary (e.g., "60-90 day dormants, 87 customers").
		- Expected ARRR range (e.g., "u20b915,000u201330,000 if 25% respond").
		- Send window (e.g., "Tuesday 11am").
		- Sample message preview (one variant of the actual outbound).
		- Three-button reply guidance: "Reply YES to approve, NO to skip, or CHANGE with notes".
2. **Owner reply handling**:
	- Webhook (VT-3.3) classifies via VT-5.11. The classifier recognizes:
		- `weekly_approval_response: approved` (variants: "yes", "haan", "go", "approved", "u0939u093eu0902", "u0915u0930u094b")
		- `weekly_approval_response: rejected` (variants: "no", "nahi", "skip", "u0928u0939u0940u0902", "u0930u094bu0915u094b")
		- `weekly_approval_response: needs_changes` (variants: "change", "badlo", "different" + free-text notes; classifier extracts the notes)
		- `weekly_approval_response: defer` (variants: "later", "baad mein", "next week"; the approval timeout extends by 48 hours, max 2 defers per approval)
	- Webhook resolves the `pending_approval` row via VT-5.9 with the decision; orchestrator resumes the run.
3. **Decision routing**:
	- `approved`: orchestrator transitions the campaign to `status='approved'`; scheduled trigger sends at the designated send window via VT-5.6/VT-5.7.
	- `rejected`: campaign transitioned to `status='cancelled'`; orchestrator records reason; weekly cadence skips THIS week (next week resumes normally).
	- `needs_changes`: campaign stays `status='proposed'`; agent re-dispatches with the owner's notes appended to the next context bundle; new approval request sent (counts against owner's per-week messaging budget).
	- `defer`: approval window extended; max 2 defers per approval; after 2nd defer, treat as rejected (with audit note).
4. **Timeout**:
	- Default 48-hour window per VT-5.9; if no reply by timeout, treat as `defer` for first instance, `rejected` after.
	- Owners can configure preferred approval window (24h / 48h / 72h) in their portal (VT-9.7); default 48h.
5. **Per-week messaging budget**:
	- To prevent approval-fatigue spam: max 2 approval requests per owner per week. If a campaign comes back as `needs_changes` and the agent re-proposes, that's the second request; further re-proposals queue until next week.
	- Budget enforcement at orchestrator's weekly cadence trigger; documented in `docs/team/owner-messaging-budget.md`.
6. **Tests** at `apps/team/tests/owner_surface/test_weekly_approval.py`:
	- Approve happy path: synthetic CampaignPlan → approval message dispatched → owner replies "yes" → campaign approved → send scheduled.
	- Reject path: "no" → cancelled.
	- Needs-changes: "change message to be more festive" → campaign re-proposed with notes; new approval request counts as week's 2nd.
	- Defer: "baad mein" → window extended; 2nd defer → rejected with audit note.
	- Timeout: 48h elapsed → first time = defer; second time-elapsed = rejected.
	- Hindi variants recognized for all 4 decisions.
	- Per-week budget: 3rd approval request in same week → queued.
	- Cross-tenant: tenant_A approval reply cannot resolve tenant_B's pending approval.
## Architectural rules to enforce in this subtask
- Pillar 7: owner approval mandatory; campaign cannot send without approved status. CodeX rejects auto-approval paths.
- Pillar 8: one approval UX. CodeX rejects parallel approval flows for sub-types of campaigns.
- 4-decision enum (`approved | rejected | needs_changes | defer`) is structural. Adding a 5th decision is Type 2 governance.
- Per-week budget = 2; raising requires Type 2 (owner messaging fatigue is a known failure mode).
- Hindi variants in classifier are versioned with classifier prompt; this subtask documents the expected variants.
- Sample message preview shown to owner is a single variant; if the agent generates multiple variants (VT-5.13 if approved), only one shown for approval.
