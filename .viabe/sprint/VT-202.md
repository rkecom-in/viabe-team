# VT-202 — Proactive alerts (push complement to VT-201's pull dashboard)

## Why this row exists

VT-201 is the operator-visits-dashboard (PULL) surface. Without proactive notifications, operator awareness depends on dashboard-discipline. For "if anything goes wrong I know what it is" to hold at 2am or during a busy day, the system needs to push.

## Scope

### Alert triggers (configurable per-tenant)

1. **Hard-limit hits** — `status='aborted_hard_limit'` lands → page operator with run_id + reason
2. **Escalations** — `status='escalated'` lands → operator with reason text
3. **Error envelopes** — any error_envelope step kind → page with error details
4. **Cost anomaly** — single-dispatch cost > 2x rolling p95 of last 100 dispatches for same tenant
5. **Latency anomaly** — single-dispatch wall-clock > 2x rolling p95
6. **Privacy audit non-empty** — any privacy_audit_log row → page (currently always empty; non-empty would indicate operator [resolve] action OR a privacy gate violation)
7. **Dispatch volume anomaly** — sudden spike (>3x rolling hourly baseline) suggests spam/attack
8. **Outbound failure** — Twilio send failed → page

### Delivery channels

1. **Telegram** — primary; @viabe_ops_bot DM to Fazal (channel-id configured via env)
2. **Email** — secondary; daily digest of non-critical signals + immediate for criticals (postmark or similar SMTP)
3. **Ops Console banner** — same data the VT-201 sticky banner uses, derived from these triggers

### Anomaly detection mechanism

Rolling-window aggregates computed via DBOS scheduled workflow (every 5 min):
- `tenant_alert_baselines` table — per-tenant rolling p95 latency + cost + dispatch volume
- DBOS workflow updates baselines + compares fresh observations
- Triggered alerts written to `tenant_alerts` table
- Notification dispatcher reads `tenant_alerts` + fires Telegram/email

### Alert routing rules

- Critical (hard-limit / escalation / error / privacy / outbound-fail): immediate Telegram + email
- Warning (cost / latency / volume anomaly): batched into hourly digest unless 3+ within 5 min (then immediate)
- Per-tenant subscription preferences (Phase 1: Fazal subscribed to all; Phase 2: per-operator config)

## Acceptance criteria

1. Synthetic hard-limit trip → Telegram notification arrives within 60s with run_id + reason
2. Cost anomaly (synthetic 2x-p95 dispatch) → Telegram within 5 min of baseline refresh
3. Email digest sent at 09:00 IST daily summarizing previous-day signals (count by type + top 5 most-expensive runs)
4. `tenant_alerts` table is operator-readable via VT-201's history view
5. Alert dedup: same trigger firing repeatedly within 5 min → only 1 notification
6. No alerts during canary runs (whitelist by tenant_id of synthetic test tenants)

## Rule #15 Canary

`apps/team-orchestrator/canaries/vt202_proactive_alerts.py` — 5 assertions covering trigger firing, dedup, channel delivery (mock Telegram API), digest generation.

## Out of scope

- ML-based anomaly detection (statistical rolling p95 only this row)
- SMS / phone-call alerts
- PagerDuty / Opsgenie integration
- Per-customer-level alerts (operator alerts only)
- Auto-remediation (alert → human action; no auto-recovery)
