# VT-75 — Locality coarsening (city → city_tier)

## Parent task
VT-PrivacyArchitecture.

- `concept-team-pillars.md` Pillar 6 (coarsening is structural at construction, not runtime filtering).
- `concept-team-pillars.md` Pillar 7 (coarse-grained data prevents re-identification through location).

## What CoderC must produce
The locality coarsening module. A deterministic function: given a city or locality string, returns its city_tier. Used by L3 construction (VT-7.5) to drop fine-grained location data before aggregation.
1. **Coarsening module** at `apps/team/src/privacy/coarsening.py`:
	- `coarsen_city(city: str, state: str | None = None) -> CityTier`.
	- `CityTier` enum: `tier_1` (8 metros: Mumbai, Delhi, Bangalore, Chennai, Kolkata, Hyderabad, Pune, Ahmedabad), `tier_2` (~50 next-largest cities; specific list in config), `tier_3` (everything else).
	- Lookup driven by `apps/team/config/city_tiers.yaml` — one entry per tier_1 + tier_2 city; everything else falls through to tier_3.
	- State parameter optional but helps disambiguate same-named cities (e.g., "Hyderabad" exists in Telangana and Pakistan; state ensures we mean Telangana).
2. **Locality dropping**:
	- Function `coarsen_locality(locality: str | None, city: str, state: str | None = None) -> CityTier`. Discards locality entirely; only city_tier returned. CodeX rejects PRs that try to retain locality.
3. **Normalization**:
	- City name normalization: case-insensitive, whitespace-trimmed, common variants ("Bangalore" / "Bengaluru" → tier_1; "Bombay" / "Mumbai" → tier_1). Variants live in the YAML.
	- Unknown cities default to `tier_3`. Logged as `city_unknown` event for ops to consider adding to tier_2 if it accumulates.
4. **Tier list source**:
	- tier_1 list is a Type 3 commitment (the 8 metros are constitutionally fixed). Changing requires board approval.
	- tier_2 list is a Type 1 commitment (50 cities by 2025 census; updates as census refreshes, typically PR-reviewable additions).
	- tier_3 is the default fallback (no list).
5. **Tests** at `apps/team/tests/privacy/test_coarsening.py`:
	- 8 tier_1 cities → tier_1.
	- Sample of 10 tier_2 cities → tier_2.
	- Random small towns → tier_3.
	- Variants: "Bangalore" vs "Bengaluru" → both tier_1.
	- Whitespace + case: " mumbai " → tier_1.
	- Unknown city → tier_3 + `city_unknown` event logged.
	- Locality dropped: `coarsen_locality("Andheri", "Mumbai")` returns tier_1 with no Andheri retained.
	- Same-name disambiguation: "Hyderabad" + "Telangana" → tier_1; "Hyderabad" + "Sindh" → tier_3.

## Architectural rules to enforce in this subtask
- Pillar 6: coarsening at construction. CodeX confirms by reviewing call sites in VT-7.5 — locality is dropped before aggregation, not after.
- Pillar 7: locality NEVER stored in L3 patterns. CodeX rejects PRs adding locality fields to `l3_patterns`.
- tier_1 list is Type 3. CodeX rejects PRs modifying it without board approval.
- Unknown cities default to tier_3 — most conservative bucket.
- Same-name disambiguation requires state. CodeX confirms state is plumbed through.

## Acceptance criteria
- Coarsening module exists; tier_1 + tier_2 lists in YAML.
- All test scenarios pass.
- Variants and same-name disambiguation handled.
- Unknown cities log `city_unknown` event.
- locality is provably discarded (regex check on serialized output of coarsen_locality).
- mypy --strict passes.
- PR title references `(VT-PrivacyArchitecture)`; reviewed by CoderX; Fazal personally signs off on tier_1 + tier_2 lists.

## Out of scope
- Adaptive coarsening based on density (Phase 2; e.g., bucket sparse cities differently).
- Pincode-level analysis (out of scope; pincodes are too fine).
- Hex-grid spatial bucketing (Phase 2 if owners ask for it).
- Cross-state same-name handling beyond Phase 1 list (deferred).
