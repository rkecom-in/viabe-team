# VT-54 — Dedup and merging logic (shared primitive)

## Parent task
VT-IntegrationInvention.

## Concept reference
- `concept-team.md` §5 (dedup resolves the same customer across multiple ingestion methods).
- `concept-team-pillars.md` Pillar 4 (deduplication is a deterministic algorithm; no LLM judgment about whether two customers are the same).
- `concept-team-pillars.md` Pillar 7 (when dedup is uncertain, owner is source of truth — ask, don't auto-merge).
- `concept-team-pillars.md` Pillar 8 (one shared dedup module; methods don't reimplement).

## Dependencies
- VT-Foundation VT-2.3 Done (`customers` table).
- VT-MCP-Tools VT-5.8 Done (similar matching algorithm; `match_transactions` and dedup share Levenshtein utilities).
- VT-IntegrationInvention VT-6.2 Done (clarifying flow consumed for ambiguous merges).

## What CoderC must produce
The customer dedup + merging module. Called by every ingestion method after raw extraction. Decides: is this row a NEW customer, an EXISTING customer (merge), or AMBIGUOUS (escalate to clarifying flow).
1. **Module** at `apps/team/src/ingestion/dedup.py`:
	- `dedup_or_create(tenant_id, candidate: CandidateCustomer, source: AcquiredVia) -> DedupResult`.
	- `CandidateCustomer`: fields available from extraction (`phone`, `name`, `upi_handle`, `last_visit_date`, `notes`).
	- `DedupResult`: `action: Literal['created', 'merged', 'ambiguous']`, `customer_id: UUID | None`, `merged_with: UUID | None` (for action='merged'), `confidence: float`, `clarification_id: UUID | None` (for action='ambiguous').
2. **Matching algorithm** (multi-pass, deterministic):
	- **Pass 1 — Exact phone**: [candidate.phone](http://candidate.phone) normalized (E.164) matches existing `customers.phone_e164`. Confidence 0.99. Action: merged.
	- **Pass 2 — UPI handle exact**: candidate.upi_handle matches `customers.upi_handle`. Confidence 0.95. Action: merged.
	- **Pass 3 — Phone last-7 + name fuzzy**: last 7 digits of phone match AND name Levenshtein ≤2. Confidence 0.85. Action: merged.
	- **Pass 4 — Name fuzzy + recency overlap**: name Levenshtein ≤1 AND `last_visit_date` within 30 days of an existing customer's `last_transaction_date`. Confidence 0.70. Action: ambiguous (escalate).
	- **Pass 5 — No match above 0.70**: action: created. New customer row inserted.
3. **Merge semantics** (when action='merged'):
	- Update existing customer row: fill null fields from candidate (additive merge); never overwrite non-null fields.
	- Append source tag: `acquired_via_history` JSONB array gets a new entry with `(source, timestamp)`. The original `acquired_via` field stays as the FIRST source.
	- Bump `last_seen_at`.
	- Insert any extracted transactions tied to the merged customer.
4. **Ambiguity escalation** (when action='ambiguous'):
	- Trigger VT-6.2 clarifying flow with question template: "You uploaded a customer named [Candidate Name]. Is this the same person as [Existing Name] who last visited [date]?".
	- Owner reply (yes/no) resolves: yes → merge; no → create new customer.
	- Until resolved: candidate sits in `pending_dedup_resolution` table (migration 016), NOT committed to `customers`.
5. **Phone normalization**:
	- All phones converted to E.164 (+91XXXXXXXXXX for India). Strip spaces, dashes, country-code variants ("91-9876543210", "+91 98765 43210", "9876543210" all → +919876543210).
	- Use `phonenumbers` library; do not roll-your-own.
6. **Cross-tenant scoping**: dedup runs ONLY within `tenant_id`. Phone collision across tenants is fine — same person can be a customer of multiple businesses.
7. **Tests** at `apps/team/tests/ingestion/test_dedup.py`:
	- Pass 1 exact phone: synthetic candidate matches existing; merged with confidence 0.99.
	- Pass 2 UPI handle: matched; confidence 0.95.
	- Pass 3 phone last-7 + fuzzy name: matched; confidence 0.85.
	- Pass 4 ambiguous: name fuzzy + recency overlap → clarification triggered.
	- Pass 5 no match: new customer created.
	- Phone normalization: 6 input formats all normalize to same E.164.
	- Cross-tenant: tenant_A's customer with same phone as tenant_B's stays separate.
	- Merge additive: existing customer with name=null + new candidate with name set → merged customer has name set.
	- Merge non-overwrite: existing customer with name="Rajesh" + new candidate with name="Rajesh Kumar" → existing name unchanged (don't overwrite).
	- acquired_via_history append: customer originally acquired_via='paper_book', re-ingested via 'upi_phonepe' → history has both, primary stays paper_book.

## Architectural rules to enforce in this subtask
- Pillar 4: deterministic algorithm. CodeX rejects LLM judgments dressed as smart matching.
- Pillar 7: ambiguous = owner decides. CodeX rejects PRs that auto-merge below 0.85 confidence.
- Pillar 8: shared module. Each ingestion method imports from this module; no per-method dedup logic.
- Phone normalization uses `phonenumbers` library, not regex hacks.
- Cross-tenant scope is structural: the function signature requires `tenant_id`; queries are scoped.
- Merge is additive, never destructive.
- The 0.85 auto-merge threshold is config-driven (`apps/team/config/dedup_thresholds.yaml`). Lowering is Type 2 governance.

## Out of scope
- Cross-customer relationship inference (household detection, etc.) — Phase 2.
- Cross-tenant customer matching (explicitly out of scope; Pillar 3 enforces tenant isolation).
- LLM-based dedup (forbidden; Pillar 4).
- Customer ID resolution from raw transaction text — lives in VT-5.8 (`match_transactions`), separate concern.
- Bulk dedup of an entire ledger (Phase 1 dedups one candidate at a time).
