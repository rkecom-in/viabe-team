# CL-157 — Claude Code subagent usage rules

Project knowledge update + usage rule, triggered by CL-152 resolution.

NEW CAPABILITY (project knowledge):

Claude Code's harness includes one or more subagents, including at minimum `pr-review-toolkit:code-reviewer`. Subagents are real in-process Claude Code spawns with their own tool calls, output files, and agent ids — they actually execute, they don't just simulate. They share Claude Code's model family and training, so they're not equivalent to an independent external reviewer.

USAGE RULES (Standing, binding):

1. OPT-IN ONLY. Claude Code does not invoke subagents unless (a) explicitly instructed by Clau or Fazal in plain language (e.g. 'run the code-reviewer subagent on this PR'), or (b) the subagent invocation is a documented part of a workflow we've previously agreed to (none currently exist; this will change as we add them).

2. CORRECT ATTRIBUTION. When Claude Code reports subagent output, the report must (a) state the subagent name explicitly, (b) state that it's a Claude Code in-harness subagent (not an external reviewer), (c) provide the spawn metadata (runtime, tool call count, agent id, output file path) so Clau can request the raw artifact if needed. Never under a label that implies external/independent review.

3. SUBAGENT OUTPUT IS A CLAUDE CODE PRODUCT. Reviews from subagents do not satisfy the "second pair of eyes" requirement (Clau + Fazal review per CL-151). Subagent output is one input to Claude Code's PR-preparation work, not a substitute for the actual review gate.

4. RAW ARTIFACT FETCHABLE. The subagent output file on disk must be referenced in Claude Code's report, and Claude Code must be willing to paste the raw output if Clau asks. No more 'I'll summarize what the subagent said' as the only path.

5. NO SILENT SUBSTITUTION. If Clau or Fazal instructs an unavailable resource (external reviewer, tool that doesn't exist, etc.), Claude Code's required response is: 'X isn't available; I have Y which is the closest substitute — want me to use Y?' Substitution proposal, never substitution silent-execution.

WHAT WE DON'T YET KNOW:

- Full list of subagents available in Claude Code's harness (only `pr-review-toolkit:code-reviewer` surfaced this session).
- Subagent capabilities and limitations beyond the one we've seen.
- Whether subagent invocations cost meaningful runtime or context.

Claude Code is asked to surface, in their next report or at start of next session, a full list of available subagents and their purposes — so we can decide which (if any) we want to opt-in to using as part of standard workflows. Until then: opt-in only, named by user, attributed correctly.

USE CASES TO CONSIDER (not decided yet, just on the table):

- Claude Code running `code-reviewer` subagent as pre-PR self-review, results pasted in PR description with correct attribution. Could catch issues before Clau reviews. Worth piloting on Step 2 (langgraph) PR.
- Other subagents (test runners, security scanners, etc., if they exist) on case-by-case basis.

Decision deferred until we have the full subagent list.
