# VT-68 — L3 pattern construction with k-anonymity at build

## Parent task
VT-KnowledgeArchitecture.

## Concept reference
- `concept-team.md` §9.3 (L3 patterns: cross-tenant generalizations under **k-anonymity ≥10, locked Type 3 commitment per concept doc Section 10**).
- `concept-team-pillars.md` Pillar 6 (k-anonymity is a BUILD-TIME invariant; patterns are constructed only when the eligible cohort meets k-min; ward/city-tier coarsening at construction).
- `concept-team-pillars.md` Pillar 7 (consumer-PII never leaks across tenants; patterns are aggregated, not per-customer).
- `concept-team-pillars.md` Pillar 8 (no patchwork; one construction pipeline; coarsening is structural, not retrofit).

## Dependencies
- VT-7.3 + VT-7.4 Done (L2 is the input for L3 construction).
- VT-PrivacyArchitecture VT-8.3 + VT-8.4 in progress (k-anonymity admission gate + locality coarsening; this subtask is the consumer of those primitives).

## What CoderC must produce
The L3 pattern construction pipeline. Aggregates L2 episodic events across tenants into anonymized patterns the agent can use as priors (e.g., "in cafes in Tier-2-cities, 60-90d dormants who get a discount template approve at 40-50%"). Constructed nightly via a scheduled trigger. **K-anonymity ≥10** enforced at construction; coarsening at construction; never decoded to specific tenants.
1. **Pattern types** (Phase 1 starts with 4):
	- `cohort_response_rate`: response rates for a cohort defined by (business_type, city_tier, recency_band). Output: `{n_tenants_in_cohort, n_campaigns, response_rate_p50, response_rate_p25_p75}`.
	- `attribution_rate_by_recency`: attribution rate for cohorts defined by (business_type, recency_band). Output: similar structure.
	- `template_effectiveness`: relative response rates of approved templates within (business_type, city_tier). Output: per-template ranking with confidence intervals.
	- `time_of_send_effectiveness`: response rate by hour-of-day for (business_type, city_tier). Output: 24 buckets with response_rate stats.
2. **Construction pipeline** at `apps/team/src/kg/l3_construction.py`:
	- Runs nightly (3 AM IST) via scheduled trigger added to VT-3.5.
	- For each pattern type, for each combination of (business_type, city_tier, recency_band):
		a. Identify eligible tenants: those with at least 1 campaign in the past 90 days that match the cohort.
		b. K-anonymity gate (via VT-8.3): require ≥10 distinct tenants. If <10, the cohort is dropped.
		c. Locality coarsening (via VT-8.4): cities are bucketed into city_tiers (Tier-1: Mumbai/Delhi/Bangalore/Chennai/Kolkata/Hyderabad/Pune/Ahmedabad; Tier-2: 50 next; Tier-3: rest). Locality is dropped; only city_tier carries forward.
		c. Aggregate metrics from L2 events (no individual events stored at L3; only aggregates).
		d. Compute statistics (median, IQR, percentile bands).
		e. Write to `l3_patterns` table.
3. **Schema** at migration `022_l3_patterns.sql`:
	- `l3_patterns`:
		- `id` UUID PK
		- `pattern_type` TEXT NOT NULL
		- `cohort_key` TEXT NOT NULL (canonical cohort identifier, e.g., `"cafe|tier_1|60_90d_dormant"`)
		- `n_tenants` INT NOT NULL CHECK (n_tenants >= 10)
		- `n_campaigns` INT NOT NULL
		- `metrics` JSONB NOT NULL
		- `confidence_band` TEXT (low / medium / high based on n_campaigns)
		- `constructed_at` TIMESTAMPTZ
		- `expires_at` TIMESTAMPTZ (90 days from construction; rebuilt nightly anyway, so this is a safety floor)
	- INDEX on `(pattern_type, cohort_key)` for fast retrieval.
	- NO `tenant_id` column. L3 is cross-tenant by design. RLS NOT applied (these aggregates are NOT per-tenant data; they're global priors).
4. **Retrieval** at `apps/team/src/kg/l3_query.py`:
	- `lookup_pattern(pattern_type, cohort_key) -> L3Pattern | None`. Returns None if no pattern matches (cohort didn't meet k-min).
	- The agent's `SalesRecoveryContext` builder (VT-4.3) calls this for the tenant's specific cohort during context build.
5. **Tests** at `apps/team/tests/kg/test_l3_construction.py`:
	- Synthetic L2 events across 15 tenants in cafe + Tier-2 + 60-90d cohort → pattern constructed with n_tenants=15.
	- Cohort below k-min: 9 tenants → pattern NOT constructed.
	- Boundary: exactly 10 tenants → admitted with n_tenants=10.
	- Coarsening: 12 tenants from 8 distinct cities (mix of Tier-2 and Tier-3) → city info dropped; aggregates by city_tier.
	- No-PII test: serialize an L3 pattern row → no phone, no name, no specific tenant identifier present (regex check).
	- Time-of-send test: 24-bucket aggregate for cafe Tier-1.
	- Reconstruction: deleting all l3_patterns + re-running pipeline produces equivalent state.
	- Cross-tenant impossibility: no API path returns L1/L2 data via L3 retrieval.

## Architectural rules to enforce in this subtask
- Pillar 6: k-anonymity is BUILD-TIME. CodeX confirms via test: cohorts below k-min are not constructed at all (not stored with low-confidence). The check is in the construction code; not a runtime filter.
- Coarsening is structural. City names are dropped at construction. CodeX rejects PRs that store city-level data in `l3_patterns`.
- Pillar 7: NO PII at L3. CodeX rejects any PR that adds tenant-identifying or customer-identifying fields to `l3_patterns`.
- Pillar 8: one construction pipeline. Adding a new pattern type is a new function in `l3_construction.py`; CodeX rejects shadow construction pipelines.
- L3 retrieval is cross-tenant by design; no tenant_id check applies. This is intentional and a structural difference from L1/L2.
- 90-day expiry on patterns is a safety floor; nightly reconstruction means most patterns are fresh.
- Adding a new pattern type or changing k-min is Type 2 governance.
