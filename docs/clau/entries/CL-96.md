# CL-96 — Audit brief template needs Decision Authority clause

The Layer 1 audit brief used 2026-05-17 instructed the fresh session to 'find issues and recommend' — which produced question-asking behavior on decisions that were Clau's to make (schema execution order, RLS mechanism, naming, sequencing, mypy gating, attack-test parametrization). Audit-session was acting as code reviewer, not co-founder.

Root cause: audit brief didn't transfer the implicit authority current-session-Clau has accumulated. Fix in the Resurrection File v2.21 (ownership lineage) + Layer 1 system prompt (Decision Authority section) addresses fresh sessions in general. But the AUDIT BRIEF TEMPLATE itself also needs a Decision Authority clause.

## Action: update audit brief template

When drafting any future audit brief (Layer 1, Layer 2 post-VT-3.4 live testing analysis, etc.), include this clause:

---

DECISION AUTHORITY APPLIES TO YOU TOO. When your findings have clear answers, make the call. When they don't, surface them. The earlier session that implemented the code made architectural decisions and logged them; you have the same authority for fix decisions within the same domain.

Specifically:
- For findings in the categories Clau owns (architecture, schema, code design, CI config, sequencing, naming, brief drafting), produce DECISIONS in your findings report, not QUESTIONS. The decision can be tentative ('recommend X unless Y'), but state it as a decision.
- For findings that genuinely require Fazal's input (pricing, policy, regulatory, customer-facing commitments, kill-criteria thresholds, anything in Resurrection File v1), surface as questions.
- If you're unsure which category a finding falls into, default to deciding and documenting your reasoning. Fazal can override; that's how the system works.

Read Resurrection File v2.21 (ownership lineage) for the canonical split.

---

Apply to next audit brief draft. Don't carry forward the older 'find issues and recommend' phrasing.
