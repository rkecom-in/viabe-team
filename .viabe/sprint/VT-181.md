# VT-181 — @observability.tool_step decorator for MCP tools

## Scope

Per design doc §3.2:

1. **Decorator:** `@observability.tool_step(envelope_in: type[BaseModel], envelope_out: type[BaseModel])` at `apps/team-orchestrator/src/orchestrator/observability/decorators.py`
2. **Behavior:** wraps any tool function; on call, validates args against `envelope_in`; on return, validates result against `envelope_out`; calls `write_step(step_kind='mcp_tool_call', ...)` with the validated envelopes + cost_paise + duration_ms.
3. **Boot-time enforcement:** the decorator's envelope types must match registered schemas (VT-179). Drift = boot fails.
4. **Retrofit existing tools:** `self_evaluate.py` (the one existing tool in `agent/tools/`) gets decorated. New tools (composer from VT-30, future tools) MUST use the decorator.

## Dependencies

- VT-180 (writer)
- VT-179 (envelope schemas)
- VT-30 in flight (composer tool will use this; coordination at queue-time)

## Acceptance criteria

- Decorator exists; self_evaluate tool retrofitted; existing self-evaluate tests still pass.
- New decorator-wrapped synthetic tool → pipeline_steps row landed with envelopes.
- Validation failure (args don't match envelope_in) → clear error + step row with `payload_validation_failed: true`.
- Boot-time drift detection works.
