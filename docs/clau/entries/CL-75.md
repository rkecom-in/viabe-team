# CL-75 — CRITICAL audit finding C2: lost-message hazard

Order in twilio_ingress.twilio_ingress():
1. _record_inbound() — INSERT ... ON CONFLICT into twilio_inbound_events. Returns True if new.
2. _build_event() — constructs WebhookEvent.
3. DBOS.start_workflow(...) inside SetWorkflowID block.
All wrapped in `try: ... except Exception: log + return 200 with status='error_logged'`.

Failure mode: if step 1 succeeds (row written) but step 3 throws (DB transient, DBOS not launched, network), the outer except swallows it, returns 200. Twilio considers the message handled. Next time the same MessageSid arrives (Twilio retry or team-web retry), _record_inbound returns False (row exists), event built with dupe_status=True, workflow starts, Pre-Filter Gate routes to dupe_handler — and the original message is permanently dropped.

Fix: move the twilio_inbound_events INSERT inside the workflow as the first @DBOS.step. DBOS's own workflow_id idempotency is the primary dedup; the table is a belt-and-suspenders ledger and belongs INSIDE the durable boundary. The current placement undermines the Pillar 7 promise that webhooks never silently drop messages.

Pair with C3 and C4 in one small PR (~30 lines).
