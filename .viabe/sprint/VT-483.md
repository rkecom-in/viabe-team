# VT-483 — record_business_objective crashes the brain on a bad model-supplied tenant_id

## Defect (found driving the win-back on 63211ce5, deployed dev)
The brain tool `record_business_objective(tenant_id: str, ...)` takes `tenant_id` as the FIRST
LLM-supplied argument. The Opus brain sometimes supplies a malformed/placeholder string (e.g.
"the tenant", a truncated uuid). That reached `UUID(str(tenant_id))` inside
`write_business_objective` (`knowledge/business_context.py:377`) and raised
`ValueError: badly formed hexadecimal UUID string`. LangGraph re-raised the tool error and the
ENTIRE brain run hung at `status='running'`, never reaching the next tool / the
`spawn_sales_recovery` handoff. Reproduced deterministically: every win-back trigger to the real
tenant crashed here before SR could compose a draft.

This is a Pillar-3 violation: `tenant_id` is authoritative context, not a model argument.

## Fix
`orchestrator_agent.record_business_objective` now resolves the tenant from the
`ObservabilityContext` ContextVar (`_observability_context`) — the authoritative ambient dispatch
tenant — and ignores the model-supplied arg. Falls back to parsing the arg only when no context is
present, and returns a structured `{"status": "error"}` (NOT an exception) when neither yields a
valid tenant, so a bad call can never abort the whole run.

## Proof
- 2 unit tests: garbage `tenant_id` arg is ignored in favour of the context tenant; no-context +
  bad arg returns `status=error` without raising. 3/3 skeleton tests green.
- Validated on deployed dev: with this fix, the brain reaches the sales_recovery spawn instead of
  hanging at the objective-recording crash.
