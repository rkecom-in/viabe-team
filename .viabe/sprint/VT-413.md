## Contract (CC-found 2026-06-24; Cowork-confirmed hygiene follow-up).

`.viabe/sprint/.next-id` drifts across branches (a branch cut before recent allocations carries a stale
counter), so `scripts/vt_id_allocate.py` can hand back an ALREADY-USED number (observed: peek returned
VT-408 while VT-408/409 were consumed; the authoritative value lived on dev's .next-id). Same failure mode
the migration counter had.

> **Irony note (2026-06-24):** this drift-guard row was itself FIRST rostered as VT-412 off a stale
> `.next-id`=410 while VT-410/411 already existed — the exact collision it exists to prevent. Cowork
> reconciled (VT-412 → ops console; this row → VT-413; `.next-id` repaired to true max+1). Proof the guard
> is needed.

## Fix (when reached — low priority, mirror the migration-counter approach)
Before allocating, the VT-ID allocator should reconcile `.next-id` against `git show origin/dev:.viabe/sprint/.next-id`
(take the max) — or fetch+read dev's value — so it can never return a stale cross-branch number. Mirror
whatever guard the migration allocator uses. Add a test (stale local + higher dev → allocator returns
dev+1).
