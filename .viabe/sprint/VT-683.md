# VT-683 — session-first owner comms (Fazal ruling, supersedes template-per-message plans)

## The ruling (verbatim intent)
Owner-facing template whitelist = **auth OTP (Twilio-handled) + welcome + wake-up/re-engage**.
Everything else happens ONLY inside the 24h conversation session after the owner responds.
Product loop: make the owner respond once a day → session opens → Viabe Team works for him →
until then, ALL owner communications QUEUE and deliver slowly/asynchronously at idle time
during the session. (Kills the need for hing/hi variants of weekly_approval / monthly_report /
draft_approval — those become in-session sends.)

## Design questions (brief BEFORE build)
1. **Session-window truth**: track last owner-inbound timestamp per tenant → `session_open()`
   predicate (24h). Where: tenants column vs derived from conversation_log (derive first —
   no migration; index exists?).
2. **Owner-comms queue**: table (tenant, kind, payload_ref, queued_at, sent_at, priority) —
   approval asks rank over reports; money/approval semantics unchanged (an approval ask that
   queues must not expire silently — VT-668 honest-expiry interplay).
3. **Idle-pace delivery**: drain trigger = owner-inbound turn completion + a scheduled sweep
   inside open sessions; pace (e.g. one queued item per N minutes, never mid-conversation-turn);
   never interleave into an active back-and-forth (idle detection = no inbound for X min within
   the session).
4. **Daily wake-up**: send the wake-up template ONLY when (a) session closed, (b) queue
   non-empty OR daily-brief due, (c) ≤1/day, (d) owner hours (IST). Route en/hing/hi by
   owner_locale. CONFIRMED SHAPE (Fazal 2026-07-18): the en/hi wake-up templates ALREADY EXIST
   whitelisted — no button, no count variable, {{1}} name only, "reply to this message" CTA;
   the hing variant matches that shape exactly (text provided). Registry drift: the wake-up
   entry is MISSING from templates.md/twilio_templates.yaml — Fazal owes the template NAME +
   en/hi(/hing) SIDs; register all three before build.
5. **Template-send chokepoint**: template sends outside the whitelist FAIL-CLOSED (registry
   enforcement — ties into VT-681 capability truth; the whitelist is a capability declaration).
6. **Existing senders migration**: weekly_approval / monthly_report / draft_approval template
   paths → queue-or-in-session sends. Inventory-first (who sends templates today, VT-632-style
   sweep).

### P1 (S) — session-window module + reactive-send conversion
`owner_surface/session_window.py`: `last_owner_inbound_at`, `session_open(tenant)` (24h),
`idle_minutes(tenant)` — generalized FROM stale_resume (stale_resume re-imports; one truth).
Convert opt_out_confirmation / status_ping / dsr_ack template sends → freeform (window open by
construction; shrinks the live template surface toward the whitelist with zero risk).

### P2 (M) — owner_comms_queue + idle-pace drainer
New table (migration via allocator): id, tenant_id, kind, payload jsonb, priority, status
(queued|sent|dropped), queued_at, sent_at, message_sid. Writers: task_outcome's deferred
'pending' path; arm_pause_request's OUT-of-window approval asks; future senders. Drainer:
(a) post-turn hook — after a completed owner turn, deliver the highest-priority queued item;
(b) scheduled sweep (approval-timeout-sweep template, */10) — for each OPEN-session tenant with
`idle_minutes >= X` (default 5), deliver ONE item; never mid-exchange. Approval asks rank over
reports. **Money interplay (Fazal point A): a queued approval ask's timeout clock starts at
DELIVERY, not at arm** — the owner can't time out on an ask he never saw (VT-668 honest-expiry
spirit); recommend + needs your yes.
### P3 (S/M) — daily wake-up loop
`team_wakeup2` (en/hi/hing, CONTINUE_UPDATES payload): send iff session CLOSED ∧ queue
non-empty ∧ ≤1/day ∧ owner hours (default 10:30 IST slot). Payload/any reply opens the session →
P2 drains. **Fazal point B: merge `team_reengage` into wake-up** (one re-engage surface; the
manager-loop caller switches templates) — recommend yes.
### P4 (S) — whitelist enforcement + registry hygiene
`send_template_message` gains the OWNER-template whitelist {welcome4, wakeup2, error_handler
(system callback)} — SHADOW-LOG first (flag TEAM_TEMPLATE_WHITELIST_ENFORCE), enforce after one
clean week. Registry cleanup: monthly_report orphan flagged deprecated; audience labels fixed.

### Gates
