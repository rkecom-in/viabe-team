# VT-693 — the intelligence exhausts online sources FIRST; asks only the true residual

## Live evidence (Fazal's prod journey, 2026-07-22 ~18:45-18:58 IST screenshots)
Discovery HAD the GST registration payload in business_profile_draft (legal_name, trade_name,
constitution, principal_address, registration_date, nature_of_business, additional_addresses)
— yet the owner saw NONE of it and got a question grind instead: products/services → operating
hours → typical customers → price range → "main services or activities" (a REPEAT of
products/services under a different LLM-invented field name). His direct ask "Can you fetch my
details from online sources?" got a non-sequitur ACTIVATE TEAM consent pitch + another repeat.

## Root causes (diagnosed, file:line)
1. `question_brain._CONFIRMABLE = ("business_type","category","city","about")` — GST-payload
   keys are structurally un-surfaceable as confirms. Fetched, then ignored.
2. `journey._is_identity_anchored` knows only GBP entity-accept / owner-stated website — a
   GST-sourced draft is never anchored → populate/profile-card no-ops.
3. `question_brain.compose_onboarding_questions` gap leg: the LLM invents unbounded new field
   names per turn; dedup is by field NAME so synonyms repeat to the human; no cap; the draft's
   CONTENT doesn't suppress semantically-covered questions.
4. Sequencing wart: the data-inputs ACTIVATE pitch interjected mid-journey (post-profile flow
   beat fired between questions) — non-sequitur to the owner's actual message.

## Design (the Fazal ruling, mapped)
1. **GST identity CARD, confirm-first** (never auto-assert — the GSTIN came from an LLM+web
   HINT, could be the wrong company; DPDP/verify posture unchanged): after discovery lands a
   GST payload, the NEXT beat is "Here's what I found — Rkecom Services Pvt Ltd, registered at
   <principal_address>, nature: <nature_of_business>. Is this your business?" [Yes/No buttons
   — the VT-479 interactive precedent]. YES → draft anchored + populate derivable facts +
   GSTIN routes into the formal Sandbox verify (stamps verification) + covered questions
   suppressed. NO → GST fields discarded, fall back to asking.
2. **Extend confirmable/populate semantics** to the discovery payload (legal/trade name,
   address→city, nature→about/type via reconcile) so found facts COUNT AS KNOWN everywhere.
3. **Residual-question budget**: hard cap (≤3 gap questions post-draft per session slice) +
   semantic dedup (answered VALUES + draft content into the composer prompt with an explicit
   never-re-ask instruction; field-name-synonym dedup belt deterministically).
4. **"Fetch my details online" intent** → answer WITH the card / discovery status (the DF
   direct-handler class), never a consent pitch.
5. Sequencing: the ACTIVATE/data-inputs beat never interleaves mid-question — it waits for
   the profile beats to settle (flow-sentinel ordering).

## Gate
Unit per seam · dev journey re-drive ×3 (real-company name → card → confirm → ≤3 residual
questions → completion; repeat-question assert_not) · adversarial-verify (wrong-company
confirm path, GST verify gate unbent, never-assert) · owner-facing copy slice.
