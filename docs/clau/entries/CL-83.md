# CL-83 — total_arrr_paise typo → total_recovered_paise

CORRECTION (Fazal 2026-05-18 in response to audit drift D-3.2.4 / M1 from CL-70): the SubscriberState field `total_arrr_paise` (three r's) is a typo introduced in Notion VT-25 body. Fazal copied verbatim from the typo'd spec into code.

Intended meaning: 'Annual Recovered Revenue' (Viabe-specific) — NOT 'Annual Recurring Revenue' (SaaS-industry-standard). The ARR acronym is overloaded; using the unabbreviated form removes the collision.

New field name: `total_recovered_paise`.

Fix scope:
- apps/team-orchestrator/src/orchestrator/state.py — rename TypedDict field, default in new_subscriber_state, docstring.
- apps/team-orchestrator/tests/orchestrator/test_state.py — update _NOTION_FIELDS frozenset + test_new_subscriber_state_defaults assertion.
- apps/team-orchestrator/src/orchestrator/transitions.py + invariants.py — grep confirms no reads; only state.py and test_state.py touch it currently. Verify before PR.
- Notion VT-25 (page 356387c2-cc5a-81a8-8668-fea3ee7bc5ec) Expected Outcome + 'What CoderC must produce' field list — update from `total_arrr_paise` to `total_recovered_paise`.
- Future reads (VT-Attribution, VT-Billing dashboards) will be written against the corrected name.

No migration impact: field exists only in TypedDict (in-memory state), not persisted to Postgres yet. Persistence path (when it lands in VT-122 or VT-Attribution) writes the new name.

This is the kind of error that compounds: a typo in Notion gets faithfully implemented, then future code reads from the typo, then renaming becomes risky. Caught early, cost is ~5 lines + Notion edit.
