# CL-127 — VT-CI-fix-2 brief regex errors

VT-CI-fix-2 brief (CL-94 micro-PR) shipped with two errors Claude Code caught and corrected:

1. Used Python non-capturing group syntax `(?:-fix-[0-9]+)?` in a regex that runs in shell `grep -qE` (POSIX ERE). POSIX ERE has no `(?:...)`. Correct form is `(-fix-[0-9]+)?` (capturing group). Root cause: I drafted the regex in Python mental model without reading the gate's actual shell invocation in ci.yml first.

2. Dropped the literal `\(` `\)` anchors that exist in the current gate regex — the gate matches a parenthesized reference, not a bare VT-* token. The brief's §3 Python validator was then internally inconsistent: PASS strings contained literal `(...)` parens, but the pattern in the validator had no `\(...\)` anchors, so the validator would itself misclassify cases. Root cause: I didn't run the validator I authored against the regex I authored.

DISCIPLINE ADDITION (extension of CL-97 Rule 4, not a new rule):
Pre-delivery acceptance scan must include 'brief-internal consistency' — specifically, if a brief contains both an artifact (regex, schema, function signature) AND a validator for that artifact (test cases, example calls, type assertions), the validator must be runnable against the artifact as written. If I can't mentally execute the validator against the artifact and predict 'OK', I haven't actually checked it.

FOR REGEX-LIKE ARTIFACTS SPECIFICALLY: identify which regex flavor the runtime uses (Python `re`, POSIX BRE, POSIX ERE, PCRE, JavaScript) before writing the pattern. Different flavors differ on non-capturing groups, character class escaping, lookahead support, anchor semantics. The default 'Python `re`' mental model is wrong for shell gates.

FOR FUTURE CI-AFFECTING BRIEFS: re-read the existing gate line in ci.yml verbatim before drafting the replacement. Match its syntax and anchors.

NO NEW DISCIPLINE RULE NUMBER — this is an extension of Rule 4. Adding it would dilute the rule count. Logged as a tightening note instead.
