# VT-36 — Self-evaluate step (Opus-backed, gates final output)

## Parent task
VT-SalesRecovery-Agent.

## Concept reference
- `concept-team.md` §8 (self_evaluate is an Opus-backed tool that gates the agent's final output).
- `concept-team-pillars.md` Pillar 2 (LLM-backed tools are explicitly listed and justified; self_evaluate is one of three permitted).
- `concept-team-pillars.md` Pillar 8 (no patchwork; the gate is structural — the agent cannot bypass it).

- VT-4.6 in progress (the `CampaignPlan` schema is what self_evaluate evaluates against).

## What CoderC must produce
The self-evaluation gate. Before the agent returns its final `CampaignPlan`, it MUST call `self_evaluate` with the draft. The tool returns either `pass` (ship) or `revise` (fix and call once more). After two revise verdicts, the agent ships with `self_evaluate_status: failed` and the orchestrator decides what to do.
1. **Gate enforcement** in `apps/team/src/agent/self_evaluate_gate.py`:
	- Wraps the agent's final-output step. The agent's draft `CampaignPlan` is intercepted before being returned to the orchestrator.
	- Calls `self_evaluate` (VT-5.12) with the draft + the system prompt's evaluation criteria.
	- On `pass`: agent's `AgentResult.output` is the draft; `self_evaluate_status='passed'`.
	- On `revise` (first time): the gate rejects the draft, returns the evaluator's feedback to the agent's reasoning loop, and the agent must produce a new draft. The new draft re-enters the gate.
	- On `revise` (second time): the gate logs the failure, sets `self_evaluate_status='failed_after_revisions'`, and ships the second draft anyway. The orchestrator decides what to do (typically: ship to owner with a flag, or escalate).
2. **System prompt awareness**: the prompt (VT-4.2) already instructs the agent that `self_evaluate` is mandatory. The gate makes this structural — the agent CANNOT skip the call. If the agent attempts to return without calling self_evaluate, the gate intercepts and forces the call. (Implementation: the agent's tool list always includes `self_evaluate`, and the gate's interceptor checks the message history for a self_evaluate call before permitting return.)
3. **Evaluation criteria** passed to self_evaluate:
	- Schema conformance: the draft is valid `CampaignPlan` JSON.
	- Pillar discipline: no invented numbers, no per-vertical heuristics, no overstated confidence, no retention pressure.
	- Internal consistency: the campaign plan's targeting matches its messaging; the expected ARRR is plausible given the targeted cohort size.
	- Legal compliance: no prohibited message content (high-pressure language, misleading claims, financial-incentive claims that aren't backed).
4. **Self-evaluate counts toward limits**:
	- Each `self_evaluate` call counts as a tool call (against the 25-call cap). The gate is structural; it costs the agent budget.
	- This is intentional: the agent cannot infinitely loop self_evaluate → revise → self_evaluate.
5. **Failure handling**:
	- If `self_evaluate` itself errors (network failure, model refusal): treated as `agent_invalid_output` failure type → routed by VT-3.6.
	- If the second draft also fails: the gate ships with the failure flag; the orchestrator (via the dispatch node) decides whether to deliver the campaign with caveat, hold for owner review, or escalate to Fazal.
6. **Tests** in `apps/team/tests/agent/test_self_evaluate_gate.py`:
	- Pass on first try: synthetic draft → self_evaluate returns pass → ships cleanly.
	- Revise once, pass: synthetic draft → revise feedback → second draft → pass.
	- Revise twice, fail: both drafts get revise → ships with `failed_after_revisions` flag.
	- Bypass attempt: synthetic agent attempts to return without calling self_evaluate → gate intercepts → forces call.
	- Self_evaluate tool error: network failure on first call → routed as `agent_invalid_output`.
	- Tool-call-cap interaction: agent at 24/25 tool calls → self_evaluate is the 25th → succeeds; agent at 25/25 → self_evaluate would be 26th → hard-limit fires before self_evaluate runs (precedence test).

## Architectural rules to enforce in this subtask
- Pillar 2: self_evaluate is one of three permitted LLM-backed tools (with `classify_owner_message` and optional `draft_message_variants`). Its rationale ("a separate model evaluating the agent's output is structurally different from the agent evaluating itself; produces an independent assessment") is documented in the tool's spec (VT-5.12).
- Pillar 8: the gate is structural. The agent cannot bypass via prompt-injection-style "please skip self_evaluate." CodeX rejects any code path that allows skipping.
- Two-revise-then-fail is the configured policy. The values ("max 2 revisions") are config-driven (`apps/team/config/self_evaluate.yaml`). Higher counts are Type 2 governance (does the gate help, or is the model just slow to converge).
- Self_evaluate uses Opus (per [concept-team.md](http://concept-team.md)). This is the Opus call referenced in the system prompt. CodeX confirms Opus is the model in `models.yaml`.
- The evaluator's feedback is structured (JSON with categories: schema | pillar | consistency | legal), not free prose. The agent receives structured feedback, addresses each category in its revision.
- Hard-limit precedence: hard limits (VT-4.4) ALWAYS fire before self_evaluate logic. A self_evaluate call that would exceed the tool-call cap is blocked by the cap, not by self_evaluate.

- self_evaluate uses Opus per `models.yaml`.
- Evaluation criteria documented at `docs/team/self_evaluate_criteria.md` with the four categories (schema, pillar, consistency, legal).

- PR title references `(VT-SalesRecovery-Agent)`; reviewed by CoderX (mandatory — self_evaluate is the LLM-as-evaluator boundary).
