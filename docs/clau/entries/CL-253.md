# CL-253 — DECISION/MILESTONE: VT-35 hard-limit enforcement complete — four enforcers (toke

PR #31 (VT-35) implements hard-limit enforcement at the orchestrator/agent boundary: four enforcers (TokenMeter 80_000 / ToolCounter 25 / DepthTracker 8 / WallclockTimer 300s) + a first-wins CancellationContext, wired into the hand-written non-streaming Messages-SDK loop. Sync/async resolution = option B: monotonic-clock deadline at the turn boundary catches accumulated wall-clock; a 60s per-turn HTTP timeout (PER_TURN_HTTP_TIMEOUT_S, passed to messages.create(timeout=)) caps any single hung call — no asyncio, no thread, no zombies. GAP CAUGHT IN REVIEW + FIXED (c873993): the 60s HTTP timeout raises APITimeoutError inside _run_one_turn; original code did not catch it, so a hung turn would have surfaced as an uncaught exception rather than a clean terminated_by='wall_clock'. Fix: narrow try/except APITimeoutError around _run_one_turn signals HardLimitAxis.WALL_CLOCK via the coordinator; existing post-loop cancel branch then produces status='terminated', terminated_by=wall_clock, FailureRecord(AGENT_HARD_LIMIT_BREACH) routed. Catch is deliberately narrow — APITimeoutError only; APIConnectionError and other connection faults continue to propagate as external_api_error (not hard-limit material). New test test_per_turn_http_timeout_terminates_as_wall_clock asserts the full termination + routing path and exactly 1 messages.create call (no retry). All four hard-limit termination paths now route cleanly. CI gate gate-vt35-hard-limit-constants greps each of the four Type-3 literals; any change fails CI. 14 tests pass. Constants are Type-3 commitments per concept-team.md §8.3 — changes require board governance.
