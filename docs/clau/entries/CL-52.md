# CL-52 — Migrations applied via Path A

Migrations applied directly by Claude Code using TEAM_SUPABASE_SECRET_KEY from .env.local (Path A), not via Supabase CLI (Path B). Reason: two Supabase accounts (Reports' account + Team's separate account created May 16) introduce CLI auth switching friction. Path A is faster setup, lower overhead.

Risk mitigation:
- TEAM_SUPABASE_SECRET_KEY in .env.local points ONLY at dev project (viabe-team-dev)
- Prod credentials NEVER stored in .env.local; live only in Railway/Vercel env config applied via deploy pipeline
- Migration runner script (built in VT-Foundation) tracks applied migrations to prevent double-apply
- Phase 1 migration cadence is slow (~10-20 migrations total) so Path A's per-migration overhead acceptable

New Supabase auth model (replaces deprecated anon_key + service_role_key):
- TEAM_SUPABASE_URL (project URL)
- TEAM_SUPABASE_PUBLISHABLE_KEY (browser-safe, RLS-enforced reads, replaces anon_key)
- TEAM_SUPABASE_SECRET_KEY (server-only, bypasses RLS, replaces service_role_key)

Discipline rule to bake into CONVENTIONS.md: TEAM_SUPABASE_SECRET_KEY never appears in client code, NEXT_PUBLIC_* prefix, or browser-bound bundles.

Phase 2 trigger to revisit: if migration volume scales >50 migrations or if multi-environment migration verification becomes a problem, evaluate Supabase CLI migration. Migration would be additive (existing migration history preserved).
