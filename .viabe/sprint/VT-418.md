## Contract (Cowork 2026-06-24; Fazal: "build the L2 approve→send driver"). PLAN-FIRST → Cowork gate, then build. Money-send = RISK, idempotency-verified, NOT self-merge.

The VT-396 audit found owner approval flips a batch to `approved` (`agents/approval_glue.py:283-297`) but
**nothing selects approved L2 batches and sends** — dead end. The only production send path is L3 auto-send
(`agents/l3_hold.py:703`, delivery-anchored). This builds the missing L2 driver so owner-approve→send fires.

## Scope
- The driver SELECTS approved L2 batches (approved + L2 + not-yet-sent + in window) → real Twilio win-back send
  (`utils/twilio_send.py:259` + the send-time gates `customer_send.py:507-601` + `team_winback_simple` SIDs).
  Mirror the L3 hold pattern (durable workflow vs scheduled poller — plan recommends).
- **Idempotency (LOAD-BEARING, money-send):** no double-send on re-run/re-select/restart-mid-send. Reuse the
  VT-387/410 ledger-anchored dedup ('sent' anchor write ordering; never re-send on transient error = the VT-410 fix).
- Honor opt-out/consent/caps via the existing send-time gates (don't re-implement). Fails-closed while
  `MARKETING_CONSENT_VERSIONS` empty (VT-396 step-3 enables it — dependency).
- **Canary (Rule #15):** approved L2 batch (synthetic, CL-422) → exactly ONE real send to a controlled test number;
  re-run driver → NO second send (idempotency proof).

## Gate
RISK (MONEY-SEND). Plan-first → Cowork gate; build → Cowork gate, idempotency-verified; Pillar-7 merge. NOT self-merge.
