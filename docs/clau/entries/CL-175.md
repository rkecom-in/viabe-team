# CL-175 — Drop langgraph_supervisor library

## Background

CL-26 chose langgraph_supervisor library for the multi-agent handoff abstraction.
CL-22 made orchestrator-agent IS the supervisor (no separate supervisor node).
CL-136 then chose to keep langgraph_supervisor BUT use only create_handoff_tool (drop create_supervisor).

Audit session for VT-3.4 PR 1/3 brief discovered via Context7 that:
- create_handoff_tool is designed to operate inside a parent graph that create_supervisor builds
- Zero documented examples of using create_handoff_tool outside create_supervisor
- Tool returns Command(goto=agent_name, graph=Command.PARENT) — requires a parent StateGraph at call site with nodes named to match

## Decision

Drop langgraph_supervisor entirely. Write our own spawn tool returning Command directly. Approximately 15-20 lines using langgraph.types.Command, which IS documented.

## Reasoning

1. CL-26 premise (we need a supervisor topology library) no longer holds post-CL-22. Orchestrator-agent IS the supervisor; we never used the library's create_supervisor.
2. CL-136's Option C was already off-pattern — using one factory function from a library whose mental model assumes you also use the other.
3. Undocumented usage = invisible coupling. If langgraph_supervisor bumps minor versions and changes Command.PARENT handling internally, our manual StateGraph could break silently.
4. We control the tool. The Context7 docs literally show us the implementation. Code is more inspectable.
5. One less dep, one less pin to maintain.

## What changes

- Remove langgraph-supervisor from apps/team-orchestrator/pyproject.toml
- Write spawn_sales_recovery as a custom tool in apps/team-orchestrator/orchestrator/tools/spawn_specialist.py (or equivalent location)
- Tool returns Command(goto='sales_recovery_agent', graph=Command.PARENT, update={...})
- Build the multi-agent StateGraph manually in supervisor.py (or merge into existing orchestrator graph file)
- VT-27 spec page wording updated to reflect this; CL-22, CL-129, CL-136 stay valid for the bits they covered; this entry is the canonical source on the library decision

## What stays from CL-129 (3-PR split structure)

- PR 1/3: skeleton + happy-path test only
- PR 2/3: Context Composer + tenant isolation
- PR 3/3: hard limits + cost + escalation

The 3-PR split is unchanged. Only the library-dependency aspect of PR 1/3 changes.
