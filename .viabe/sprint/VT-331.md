# VT-331 — Razorpay /subscribe + plans + price gate (HARD GATE before LIVE)

The VT-89 webhook (#335) resolves tenant via `subscriptions.razorpay_subscription_id`,
set at subscription creation. VT-89 split that creation path out (the webhook is the
self-contained VT-93 keystone; until this lands, subscriptions only get rows via canary
seeding — fine, LIVE is gated). This row builds it. Cowork-approved split (20260605T124500Z).

## Scope
1. **`POST /api/team/razorpay/subscribe`** (team-web): body `{tenant_id, plan_tier,
   customer_card_token}`. Validate tenant scope (owner session / WhatsApp-verify). Call
   Razorpay `/subscriptions` (STUB until LIVE) with the tenant's plan; store
   `razorpay_subscription_id` on the subscription row (the webhook then resolves tenant
   from it). Idempotent.
2. **`config/plans.yaml`** — founding/standard/pro -> plan IDs + prices from env
   (`FOUNDING_PRICE_PAISE` 249900 / `STANDARD_PRICE_PAISE` 499900 / `PRO_PRICE_PAISE`
   1499900; plan IDs `*_RZP_PLAN_ID_DEV/_PROD`). NO rupee literals in code (Pillar 7).
3. **CI gate `gate-no-price-literals`** — grep ₹ + the literal paise values in non-config
   TS/PY (the VT-35 hard-limit-constants gate pattern).

## HARD GATE
Provisions plan IDs = LIVE/NEEDS-FAZAL -> LIVE keys hard-gated (alongside VT-93-N1 +
VT-329 + VT-330 on VT-89.md). plan_tier CHECK already exists (mig 001).
