## Why (Fazal STANDING, CL-2026-07-15)
"Do not hardcode/manage lists unless finite + fully-enumerable; the LLM decides intent, deterministic
code only skips when the outcome is fixed exact-match." The approval/send-intent path enumerates ways a
human phrases "send" (`_SEND_IMPERATIVE_BARE`, F1) and "haven't ordered" (F2 dormancy cues) — INFINITE
sets that always lag reality (a phrasing not in the list → false-negative re-confirm loop). That is the
LLM's job. F1/F2 landed as an interim money-safe deterministic bridge; this row retires the lists.

## Target architecture
- **LLM decides intent** for the pending-send approval reply: structured output {decision:
  approve|reject|hold, cited_cue: "<the exact owner phrase it relied on>", confidence}. Must GROUND the
  decision by quoting the cue (anti-hallucination).
- **Thin deterministic VETO only on hard-stops** that must never be overridden regardless of the LLM:
  a negation bound to the send, opt-out, no consent → force hold/reject. No positive keyword list.
- **Fail-safe unchanged:** uncertain / low-confidence / any veto → re-ask, NEVER auto-send.
- Same treatment for the status-query dormancy/count router (F2) and, in principle, the DF5 nets — but
  scope THIS row to the money-critical send-intent path first; the read-only routers are a follow-on.

## PROOF the keyword approach is broken (F1 adversarial-verify, 4 skeptics, 137 inputs, empirical)
The F1 keyword interim was REJECTED by its own adversarial verify — the deterministic classifier is
broken BOTH ways, and these are PRE-EXISTING holes LIVE on dev (all ≤12 tokens, independent of F1):
- **Auto-sends on a NON-authorization** (money_breach): `kya bhej du` (should I send?), `theek hai
  socho phir batao` (ok, think then tell me), `send hold pe rakho` (put the send on HOLD), `go slow`,
  `proceed with caution` → all return `approved` → an irreversible campaign fires on a question /
  deferral / pause instruction.
- **Misses obvious sends** (false_negative): `dispatch it now`, `bhijwa do`, `chala do`, `भेजिए`,
  `roll it out now` → `None` → re-confirm loop.
A keyword matcher cannot distinguish `bhej du` (send it) from `kya bhej du` (should I send?). This is
the empirical basis for CL-2026-07-15-no-lists. The 137-input suite (money_breach set must NEVER
approve; clear-send set MUST approve) is the ACCEPTANCE GATE for the LLM redesign.

## Verification bar (money gate — HIGHEST)
Adversarial-verify across phrasings (EN/Hindi/Hinglish/Devanagari/code-mix) + a hard "ZERO false
auto-send on any ambiguous/hedged/negated/pause/decline input" proof (the F1 skeptic harness, reused).
Shadow-first against the deterministic classifier before it owns the gate. Do NOT rush an LLM rewrite
INTO the money gate — deliberate, gated, shadow-compared.
