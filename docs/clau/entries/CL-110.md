# CL-110 — VT-3.3c brief §5 referenced pipeline_steps row that §3 didn't write

BRIEF ERROR caught by Claude Code during VT-3.3c implementation: brief §3 spec'd handler return contracts (send_result dict) without persistence, but brief §5 (manual test doc) referenced a pipeline_steps row with step_kind='handler_executed' that no code path would write.

ROOT CAUSE: when drafting §5 manual-test steps, Clau described an idealized observability verification (DB row exists, output_envelope populated) without checking whether §3's scope actually produced that row. Acceptance criteria drifted from implementation spec.

RESOLUTION (Option 1): VT-3.3c does NOT persist handler output. Brief §5 step 4 rewritten to verify structured log line + Twilio Console outbound, not a DB row.

DISCIPLINE LESSON: verify cross-section consistency before delivering a brief. §5 acceptance + manual test must only reference state that §3 implementation produces.
