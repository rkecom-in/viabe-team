# VT-169 — Verify Supabase project regions match VT-18 spec (DPDP residency)

## Why

Surfaced 2026-05-26 01:00 IST by the VT-102 (pipeline_log) canary at commit `957887e`. The canary's preflight stage echoed the resolved Supabase host as:

```
PREFLIGHT OK — resolved host: aws-1-ap-northeast-2.pooler.supabase.com:5432 (env-loaded)
```

`ap-northeast-2` is **Seoul, South Korea**. The brief at `.viabe/sprint/VT-18.md` (migrated from Notion, originally CL-78 / CL-79 era) specifies:

> Two Supabase projects: `viabe-team-dev` (region: ap-south-1, Mumbai) and `viabe-team-prod` (region: ap-south-2, Hyderabad), both with point-in-time recovery and daily snapshots configured.

**The dev project's resolved host is in Seoul, not Mumbai.** Three possible interpretations:

1. **The Supabase project was provisioned in the wrong region** when set up. Real misconfiguration. Customer data + L0/L1 KG vectors + owner_inputs would live in Seoul, not India — a direct DPDP residency violation.
2. **The brief was aspirational; actual hosting differs by design.** Acceptable if explicitly accepted with a DPDP impact analysis. Not currently documented anywhere.
3. **Supabase's APAC pooler routes through Seoul regardless of the underlying DB's location.** The pooler (`aws-1-ap-northeast-2.pooler.supabase.com`) might be a regional aggregation layer while the primary DB sits elsewhere. Confirmable via Supabase dashboard.

Until we know which interpretation holds, **DPDP residency posture is unverified**. The privacy notice (VT-156) cannot be honestly drafted without the answer — it would have to say "your data is stored in India" without proof.

## What

A verification + remediation task. NOT a code change unless interpretation (1) holds.

1. **Open Supabase dashboard** for the `viabe-team-dev` project. Confirm the project's stated region (Project Settings → General → Region).
2. **If region == ap-south-1 Mumbai** AND the pooler hostname is unrelated to primary-DB location (interpretation 3) → document the pooler/DB topology in `docs/team/supabase-topology.md`. Close this row.
3. **If region == ap-northeast-2 Seoul** (interpretation 1 — real misconfig):
   - Surface to Fazal as a Type-3 decision: migrate to ap-south-1 (downtime + data copy) OR accept Seoul with a documented DPDP impact analysis OR leave unfixed for dev and only fix prod (creates dev/prod drift).
   - File the migration as a separate Critical row if migrate is chosen.
4. **Repeat the verification for `viabe-team-prod`** when that project exists (currently it may not — `.viabe/secrets/supabase-prod.env` doesn't exist yet per the secrets README).
5. **Update `.viabe/sprint/VT-18.md` with the actual region** in its Notes section, OR mark it superseded if the spec changed.
6. **Update `docs/clau/decisions-ledger.md`** with the outcome as a Standing decision (region for dev + region for prod).

## Acceptance criteria

- `.viabe/sprint/VT-18.md` reflects the actual provisioned regions (not the aspirational brief text).
- `docs/clau/decisions-ledger.md` has a Standing entry documenting the region decision + rationale.
- If a misconfig was found AND fixed: migration evidence in a new VT row + closeout entry in the ledger.
- If misconfig was found AND accepted: written DPDP impact analysis at `docs/team/dpdp-region-impact-analysis.md` signed by Fazal.
- VT-156 (privacy notice) can now state the actual region (or honest hedge if multi-region).

## Out of scope

- Migrating the Supabase project itself (separate VT row if needed — VT-170+ candidate).
- Touching the prod project (it may not exist yet; verify separately when it does).
- Re-architecting pooler vs primary-DB topology (Supabase platform concern, not Viabe).
- DPDP legal review beyond the impact analysis doc (handled under VT-115 final security/privacy/legal launch review).

## Notes

- This row is **Fazal-owned** because it requires Supabase dashboard access AND a Type-3 decision if interpretation (1) holds. Pipeline Engineer (CC) can do steps 1-2 if Fazal grants dashboard access via a separate credential, but step 3's decision routing is Fazal's.
- Priority is High (not Critical) because: the VT-102 observability table doesn't store customer PII (only PII-redacted payloads + tenant_id UUIDs). The DPDP residency concern is real for the OTHER tables — `tenants`, `owner_inputs`, `pipeline_runs.trigger_payload` JSONB, `pipeline_steps.input_envelope` JSONB, L1 KG tables — which DO store customer-derived content. So this row gates VT-156 privacy notice + VT-115 legal review more than it gates anything CC-implementable.
- The 48.5-second wall-clock of VT-102's canary (vs ~10-15s for a Mumbai-hosted DB at similar latency) is also evidence consistent with Seoul hosting; not proof but a data point.

## Canary (Rule #15 — mandatory)

This row's "implementation" is verification + documentation, not code that hits an external API. The Rule #15 canary in this case is a **fact-check script** rather than an integration test:

`apps/team-orchestrator/canaries/vt169_region_check.py`:

1. Connect to `.viabe/secrets/supabase-dev.env`'s DATABASE_URL.
2. Execute `SHOW server_version;` AND `SELECT current_setting('cluster_name', true);` to gather any region hints.
3. Resolve the hostname via `socket.gethostbyname()` and confirm whether the IP belongs to AWS ap-south-1 vs ap-northeast-2 (AWS publishes IP ranges).
4. Cross-check against Supabase's project metadata via the Supabase management API if `SUPABASE_ACCESS_TOKEN` is available (currently isn't — Fazal-action).
5. Output a JSON block with: `pooler_hostname`, `pooler_region`, `resolved_ip`, `aws_region_for_ip`, `db_advertised_region` (if obtainable), `agreement_with_brief` (true/false).
6. Fail loudly if regions disagree with VT-18's spec OR if interpretation can't be determined.

This canary is itself the row's deliverable — it provides verifiable evidence that the row's verdict (regions match / don't match) is correct.
