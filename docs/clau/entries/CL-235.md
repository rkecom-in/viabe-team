# CL-235 — CORRECTION: collapse path persists CampaignPlan + SubscriberState activity field

Claude Code PR 3/3 pre-flight surfaced that the TRANSITIONS table's 10 events (signup, card_captured, trial_extension_granted, trial_extension_exhausted, weekly_low_engagement, engagement_recovered, cancellation_requested, day39_refund_triggered, day39_continue, manual_cancel) contain NO event representing 'a campaign was proposed'. This is not a gap to fill — it is correct. Proposing a recovery campaign is an ACTIVITY, not a lifecycle phase change.

CORRECTED MODEL (Fazal decision, accepting Clau recommendation): When the sales-recovery specialist returns a CampaignPlan, the subscriber's phase does NOT change at that moment. The collapse path persists the CampaignPlan + relevant SubscriberState activity fields, and makes NO apply_transition call and emits NO event. The phase moves LATER, when the campaign produces a real engagement outcome (subscriber re-engages → engagement_recovered via ingress; does not → stays at-risk). That outcome arrives through existing ingress events in a later subtask, not through the collapse path.

This SUPERSEDES the target described in CL-231 and CL-198 §2: CL-231's principle ('direct write, no campaign_proposed event') stands, but CL-231 wrongly named the target as a phase-transition write. Correct target = direct write to SubscriberState activity fields + CampaignPlan persistence. CL-198 §2 wrongly implied the collapse path owns an apply_transition call; it does not.

OPEN (for PR 3/3 brief Step 1, Claude-Code-verifies): where CampaignPlan is persisted — a campaigns/campaign_plans table, or a field on SubscriberState / pipeline_runs. Schema-existence grep, not a Fazal question.
