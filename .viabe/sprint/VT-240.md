# VT-240 — attribution_method + attribution_confidence write-path substrate

## Why (discovered during VT-43 plan-first, 2026-05-30)
VT-43 `get_attribution_data` was specced to return an attribution-method breakdown (`exact_match` / `window_match` / `manual_owner`), per-method confidence, and `cohort_size`. None of that substrate exists on `main` (verified migrations 000–043): `attributions` (mig 023) has only `id, tenant_id, campaign_id, customer_id(null), razorpay_payment_id, attributed_paise, attribution_at, created_at` — no `attribution_method`, no `attribution_confidence`. There is no `transactions` table, and the cohort source (`customers` table) is blocked on VT-170 (Backlog). VT-43 shipped Option A (graceful-degrade: those fields return `None` + completeness note). This row tracks the substrate that lights them up.

## Scope
1. Migration: add `attribution_method TEXT CHECK(... exact_match|window_match|manual_owner)` + `attribution_confidence NUMERIC(4,3) CHECK(0..1)` to `attributions` (or a normalized attribution-method table if multi-touch is in view).
2. Wire `match_transactions` (VT-46) to populate method + confidence on write (the deterministic attribution-write path — Pillar 1/2).
3. Cohort source: depends on VT-170 (`customers` table). `cohort_size` / `attribution_rate` stay degraded in VT-43 until VT-170 lands.
4. After this lands, lift VT-43's degradation: populate breakdown + confidence + the <0.7 low-confidence note.

## Dependencies
- VT-170 (customers table) — for cohort_size / attribution_rate.
- VT-46 match_transactions (Done) — the write path to extend.
