# VT-317 — wire city-capture → set_tenant_city_tier

Split out of VT-75 (Cowork ruling 20260604T032000Z). VT-75 shipped the coarsening
PRIMITIVE: `coarsen_city` / `coarsen_locality` + `set_tenant_city_tier` (coarsen →
store ONLY the tier on `tenants.city_tier`; raw city discarded). This row WIRES it
at the point a raw city is captured.

## The gap
Nothing WRITES `tenants.city_tier` today (it's only READ — context_builder /
k_anonymity / retrieve_l4_skills). Onboarding (`create_tenant_if_unknown`) takes
`business_contact`, not a city. So `city_tier` stays NULL in prod → VT-74's k-anon
cohorts + VT-68 L3 tier-cohort matching can't deliver. This is a real **launch
prerequisite** (same gate-live posture as VT-313 corpus / VT-314 voyage key).

## What to produce
- Identify/define the onboarding or profile step where the owner's city is
  captured (e.g. an onboarding question, a profile-update flow, or an ingestion
  signal). When a raw city arrives, call `set_tenant_city_tier(tenant_id, city,
  state)` — the raw city is coarsened to a tier and DISCARDED (never persisted;
  CL-390 / Pillar 6/7). NO new column for raw city.
- Backfill city_tier for existing tenants IF a city source exists for them
  (else they stay tier_3-by-absence until they re-provide).

## Dependencies
- VT-75 Done (the coarsening primitive + writer).
- An onboarding/profile city-capture point (may need a small onboarding-question
  addition — confirm with Fazal whether onboarding should ask the city).

## Acceptance criteria
- A captured city → `tenants.city_tier` set to the correct tier; raw city not stored anywhere.
- L3 tier-cohort matching + the k-anon gate see populated city_tier for onboarded tenants.
- Canary: onboard with a city → city_tier set + no raw city persisted.

## Launch gating
- Gated before launch (Reports-Jun15 / VT-231). Without it, L3 cross-tenant tier
  cohorts are empty in prod.
