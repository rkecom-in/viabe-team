# CL-217 — test_dbos_step_resume.py driver lacks stderr capture

=== DEBT ITEM ===
Landmine 2 test driver (test_dbos_step_resume.py) uses subprocess.Popen([sys.executable, str(_WORKER), dsn, run_id]) WITHOUT stderr capture. When the worker subprocess crashes silently before reaching node_b's probe insert, the test only sees a probe-table timeout — the actual error from the worker (e.g., the PostgresSaver.setup() UniqueViolation that surfaced during diagnostics) is invisible to the test output.

During diagnostics on 2026-05-19, the worker's UniqueViolation crash was only visible because Fazal ran the worker directly via python tests/orchestrator/_langgraph_step_resume_worker.py. The test itself could not surface it.

=== IMPACT ===
Future Landmine 2 failures will require an out-of-band 'run worker directly' diagnostic step to surface the worker's stderr. This is process friction whenever the test fails for a worker-side reason.

=== FIX SHAPE ===
Wrap the Popen in stderr=subprocess.PIPE; on timeout, read and print proc.stderr.decode() in the assertion failure message. Approximately 4-6 lines of test driver code. Not urgent because:
  (a) Landmine 2 verdict captured (CL-214 BORING).
  (b) Future failures are diagnostic in nature — the 'run worker directly' workaround is known.
  (c) The committed setup() patch (CL-215) removes the most likely silent-crash failure mode.

=== REPAYMENT TRIGGERS ===
Add stderr capture if:
  - Landmine 2 fails again for a non-obvious reason (worker crashes that aren't the setup() case).
  - We extend the test to cover additional crash windows or composition surfaces.
  - We promote Landmine 2 to CI (CI test output should be self-diagnosing).

=== STANDING IMPLICATION ===
General principle: subprocess-driven tests should capture stderr by default and surface it on failure. Not promoting to a new rule — sub-case of 'tests should be self-diagnosing on failure'.

=== STATE AT TIME OF WRITE ===
test_dbos_step_resume.py committed with bare subprocess.Popen (no stderr capture).
