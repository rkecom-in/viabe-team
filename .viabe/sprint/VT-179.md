# VT-179 — Pydantic envelope schemas for the ~15 step_kinds

## Scope

Build the typed envelope framework at `apps/team-orchestrator/src/orchestrator/observability/envelopes/` per design doc Section 2.2 + 2.3:

1. **Base envelope class** with `input_envelope`, `output_envelope`, `decision_rationale`, common fields.
2. **One Pydantic model per step_kind** (15 initial; extensible):
   - `webhook_received`, `webhook_classified`, `state_transition`, `agent_invocation`, `agent_reasoning_step`, `mcp_tool_call`, `self_evaluate_gate`, `campaign_plan_emitted`, `message_dispatch`, `attribution_match`, `day39_evaluator`, `refund_decision`, `opt_out_processed`, `dsr_processed`, `error`
3. **Schema registry** at `envelopes/__init__.py` — maps step_kind string → Pydantic class, used by the writer (VT-180) and the decorator (VT-181) for validation.
4. **Boot-time schema enforcement** — startup validates the registry has all step_kinds that appear anywhere in source. Drift = boot failure (per design doc §3.2).

## Dependencies

- VT-178 (table schema reviewed; columns ground-truthed)
- Pydantic v2 (already a dependency)

## Acceptance criteria

- 15 envelope classes, each a Pydantic model with documented fields
- Schema registry + boot-time enforcement
- Unit tests: every envelope round-trips JSON → Pydantic → JSON byte-identical
- Schema drift test: synthetic emit_step with unregistered kind → boot fails clearly
- mypy --strict clean

## Out of scope

- Wiring envelopes into the writer (VT-180)
- Decorator integration (VT-181)
- Agent SDK callback (VT-182)
- LangGraph hooks (VT-183)
