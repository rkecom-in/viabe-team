# VT-188 — Operator-role substrate (JWT-with-operator-claim, client-direct-ready)

## Why this row exists

VT-178's STEP-0 audit surfaced that `phone_token_resolutions` stricter access is enforced **by GRANT-EXCLUSION** (not granted to `app_role` in `015_app_role.sql`), not by RLS policy variation. That works for backend-only access via service-role — but **does not work for VT-123 Ops UI** because:

1. Per Fazal decision (2026-05-26): VT-123 Ops UI requires a **client-direct read path** (in addition to backend reads) so the admin console can perform low-latency UI interactions without round-tripping through orchestrator backend for every PII unmask.
2. Per CL-88: the canonical client-direct mechanism is **JWT-with-claim** read against Postgres via Supabase's PostgREST/auth stack (NOT a parallel app-role connection-pool path).
3. Per CL-330 / CL-390: every PII de-anonymization event must be audit-logged.

This row builds the operator-role substrate so VT-123 Ops UI can hit `phone_token_resolutions` directly via JWT, with stricter access enforced by RLS policy + per-resolution audit.

## Scope

### Migration 026 — operator-role + RLS policy variation

1. **Create `app_operator_role`** (new Postgres role; analogous to existing `app_role` from `015_app_role.sql` but with operator-scoped grants).

2. **Grant `app_operator_role`** SELECT on `phone_token_resolutions` + INSERT on `privacy_audit_log` (for the audit-write side). NOT granted INSERT/UPDATE/DELETE on phone_token_resolutions — operator can READ to resolve, can WRITE the audit, cannot mutate resolutions.

3. **Add RLS policy variation** to `phone_token_resolutions` for operator-role:
   ```sql
   CREATE POLICY phone_token_resolutions_operator_select ON phone_token_resolutions FOR SELECT
       TO app_operator_role
       USING (tenant_id = app_current_tenant() AND app_operator_audit_enabled());
   ```
   The `app_operator_audit_enabled()` helper checks the JWT for operator-claim presence + valid audit-session-id; falls back to `false` if absent → RLS denies.

4. **Helper function `app_operator_audit_enabled()`** in `migrations/000c_operator_helpers.sql`:
   - Reads `current_setting('app.jwt.operator_claim')` (set by PostgREST/Supabase from JWT)
   - Checks claim is present + non-empty + within active audit-session window (last 60 min)
   - Returns true → SELECT allowed
   - Returns false → SELECT denied via RLS

5. **Audit-trigger** on phone_token_resolutions SELECT for operator-role:
   - Postgres function `audit_operator_resolve()` invoked via an AFTER-SELECT trigger pattern (or via the application layer if AFTER-SELECT triggers aren't supported on the Postgres version — fall back to the application-layer audit per-resolution call)
   - Writes to `privacy_audit_log` (existing, migration 008): operator_id, tenant_id, token resolved, timestamp, JWT claim hash

### Application layer — JWT validation + audit wrapper

6. **`resolve_phone_token_via_operator_jwt(token, jwt) -> str | None`** at `apps/team-web/src/app/api/ops/resolve-phone/route.ts`:
   - Validates JWT (Supabase auth-issued; operator-claim present)
   - Sets `app.jwt.operator_claim` GUC via Supabase RLS context
   - SELECTs phone_token_resolutions WHERE token = $1
   - Writes audit log row
   - Returns decoded phone OR null (RLS-denied)

7. **CL-88 alignment** — uses the JWT-direct read path Supabase auth establishes for VT-123 client-side reads. Reuses the same `app_current_tenant()` GUC setting mechanism the existing app-role uses, just with an additional operator-claim layer.

### Audit logging — per-resolution

8. Every operator resolution → 1 row in `privacy_audit_log` (existing migration 008). NOT aggregated. Fazal note (2026-05-26): per-resolution is correct for PII forensics; aggregate-by-session would lose the per-token trail.

9. **Backstop:** if the audit-write fails for any reason, the resolution itself MUST fail (no resolve-without-audit). Trigger-based path uses BEFORE-trigger ordering to enforce; application-layer path uses transaction with the audit-write FIRST.

## Out of scope (deferred)

- **Operator-role token issuance** (the auth-flow that mints an operator-JWT for an admin user) — that's in VT-123's authentication scope or a separate row VT-19X if it grows
- **Backend-only resolution path** (orchestrator-internal resolves via service-role) — already works under grant-exclusion; no change needed
- **Multi-operator audit (operator-of-operator)** — single-operator-per-session scope; aggregate audit is a Phase 2 hardening

## Acceptance criteria

- Migration 026 applies cleanly + creates `app_operator_role` + adds policy + helper function
- `tenant_connection_operator()` wrapper (or its application-layer equivalent) usable from team-web
- JWT with operator-claim → SELECT succeeds + audit row written
- JWT WITHOUT operator-claim → SELECT denied via RLS (returns 0 rows)
- Expired audit-session → SELECT denied
- Audit-write failure (synthetic) → SELECT also fails (no resolve-without-audit)
- Cross-tenant: tenant A operator cannot read tenant B's resolutions
- Canary: real Supabase + real JWT round-trip + audit log emission
- mypy --strict clean on the application-layer wrapper
