# CL-155 — Missing CI gate for uv.lock sync

PR #23 is the first PR to commit `apps/team-ingestion-worker/uv.lock`. No CI check currently verifies that the worker's lockfile stays in sync with its `pyproject.toml`. The orchestrator presumably has equivalent coverage (TBD when drafting this PR — confirm against ci.yml).

Risk being mitigated: someone (Claude Code, future-Claude, a contributor) edits `apps/team-ingestion-worker/pyproject.toml` and forgets to regenerate `uv.lock`. CI passes locally because the editor still has the old lock. Merges to main. Next clone's `uv sync --frozen` fails because lockfile is out of sync with manifest. Worker breaks silently in CI for everyone else.

FIX:

Add a CI step to the worker job in `.github/workflows/ci.yml`:

```yaml
- name: Verify worker lockfile is in sync
  working-directory: apps/team-ingestion-worker
  run: uv lock --check
```

`uv lock --check` exits non-zero if the lockfile would change. Same check should exist (or be added in the same PR) for the orchestrator if it doesn't already.

SCHEDULING:

Lower priority than CL-154 because CL-154 fixes an existing wrong-state; CL-155 prevents a future wrong-state. Schedule: after Step 5 (unused cleanup, last step of the dep-audit migration sequence). The unused-cleanup PR will touch both workers' pyprojects; adding the lockcheck step in the SAME PR makes sense — you fix the manifest, you add the gate that prevents future drift, both for the worker and the orchestrator. Bundle.

ALTERNATIVE BUNDLING:

Or fold this into CL-154 (the workaround-removal PR) since both are CI-config edits. Decision when drafting: smaller bundle (each PR = one concern) vs fewer round-trips. Defaulting to bundle with Step 5 because the migration sequence already has that PR scheduled and this rides cleanly with it.

SCOPE GUARD:

This PR adds the check. It does NOT also fix any lockfile drift the check catches (PRs that fix drift would come from running `uv lock` on whatever's stale). Catch-and-report only.
