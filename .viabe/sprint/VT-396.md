## Contract (Fazal 2026-06-22: e2e Sundaram + Sales Recovery, "through a real send", clean reset). PLAN-FIRST.

Goal: run the Team e2e for Sundaram Book Store all the way through a REAL Sales-Recovery win-back
send to a SYNTHETIC test customer — without touching the prod C2 launch gate.

## Steps (plan each before building):
1. **Reset** the dev Sundaram tenant(s) (purge prior signup/data) for a zero-start. Confirm DSR-style
   clean removal (no orphan rows).
2. **Synthetic seed** (AFTER Fazal's fresh signup creates the tenant): one+ lapsed customers with a
   synthetic test phone (Fazal-provided test number, NOT a real customer) + the sales/last-purchase
   rows that make `detect_lapsed_customers` return them + a marketing **consent_record** matching the
   dev test consent version below.
3. **Dev-only C2 enablement.** `MARKETING_CONSENT_VERSIONS` is a hardcoded empty frozenset
   (`sales_recovery_executor.py:80`). Make it **env-driven** so dev can set a test version while
   **prod/main stays empty (locked)**. The dev value is a clearly-named test version
   (e.g. `dev-test-v0`). **BINDING GUARDRAILS:** dev only; default = empty (prod unchanged); this
   change must NEVER reach main with a non-empty default; document as a test harness, not the real C2
   clearance (counsel C1–C3 stays the real gate, CL-438).
4. **Run** Sales Recovery for the tenant → detect → draft (`team_winback_simple`; NOT the
   money-bearing `team_winback_offer`, which stays always-confirm) → owner approval (Fazal approves) →
   real send to the synthetic test number → verify receipt (Twilio SID + delivery).
5. **Floor stays intact:** opt-out/DSR, money-always-confirm, per-customer caps all still enforced.
   Confirm a STOP from the test customer still kills further sends.

## Acceptance
- A real `team_winback_simple` WhatsApp reaches the synthetic test number after owner approval.

- Reset proven clean; seed is synthetic only (CL-422 — no real customer data on dev).
