# VT-34 — SalesRecoveryContext bundle contract

## Parent task
VT-SalesRecovery-Agent.

## Concept reference
- `concept-team.md` §8 (the orchestrator constructs a context bundle and passes it to the agent at invocation; bundle is bounded).
- `concept-team-pillars.md` Pillar 1 (orchestrator builds context, agent receives; tier separation through typing).
- `concept-team-pillars.md` Pillar 3 (every context is tenant-scoped; cross-tenant context is a security failure).
- `concept-team-pillars.md` Pillar 4 (context contains data, not pre-computed conclusions; agent reasons over the data).

## Dependencies
- VT-Orchestrator VT-3.4 Done (context_[builder.py](http://builder.py) is referenced from there).
- VT-PrivacyArchitecture VT-8.1 in progress (typed wrappers for tenant-scoped DB reads; fallback to scoped queries until that lands).
- VT-KnowledgeArchitecture VT-7 partial (L1 KG schema exists; L2 episodic exists; the bundle reads compact summaries from these layers).

## What CoderC must produce
The `SalesRecoveryContext` typed dataclass plus the constructor that builds it from current state and database reads. The constructor is the orchestrator's pre-flight step before invoking the agent. Strict tenant scoping; bounded size.
1. **`SalesRecoveryContext` dataclass** in `apps/team/src/agent/types.py`:
	- `tenant_id: UUID` — always present; the agent receives ONLY this tenant's data.
	- `run_id: UUID` — for trace correlation.
	- `trigger_reason: Literal['weekly_cadence' | 'owner_initiated' | 'edge_case_response' | 'support_followup']` — why the agent is being invoked.
	- `business_profile: BusinessProfile` — from L1 KG (business name, type, locality, hours, owner name, current phase, founding-tier flag).
	- `customer_ledger_summary: LedgerSummary` — NOT the full ledger; a structured summary: total customers, dormant cohort sizes by recency band (30d / 60d / 90d / 180d / 365d+), top-spender list (top 20 customers by lifetime value, anonymized to customer IDs).
	- `recent_campaigns: list[CampaignSnapshot]` — last 5 campaigns with status + ARRR contribution (closed) or pending status.
	- `attribution_snapshot: AttributionSnapshot` — cumulative ARRR, last-7-day ARRR, last-30-day ARRR, attribution-rate trend.
	- `pending_owner_inputs: list[OwnerInput]` — if any (e.g., the owner replied to a clarifying question; the reply is included here).
	- `meta: ContextMeta` — token count, build timestamp, cursor info (which row in ledger / which campaign was last seen).
2. **Constructor** in `apps/team/src/orchestrator/context_builder.py`:
	- `build_sales_recovery_context(tenant_id, run_id, trigger_reason) -> SalesRecoveryContext`.
	- Reads through typed wrappers (VT-8.1; until that lands, scoped queries with explicit tenant assertion + `RaiseIfNotTenantScoped` decorator).
	- Aggregates the ledger summary via SQL (NOT by reading every row — the aggregation is efficient: COUNT GROUP BY recency band).
	- Caps total token count at 8,000 (per VT-3.4's 8K context budget). If a section exceeds budget, truncate that section first (e.g., top-spender list trimmed to top 10 instead of 20). Truncation is logged.
3. **Bounded size enforcement**:
	- Each section has a max-token budget: business_profile 500, customer_ledger_summary 1500, recent_campaigns 1500, attribution_snapshot 800, pending_owner_inputs 1500, meta 200, slack 2000. Total 8000.
	- The constructor measures tokens (via the same tokenizer used by the prompt) and truncates per-section as needed.
	- If even after maximum truncation the bundle exceeds 8000, raise `ContextOverflowError` and the orchestrator's error router (VT-3.6) handles it.
4. **Tenant assertion**:
	- Every DB read inside the constructor passes `tenant_id` and asserts the result is empty OR all rows have matching `tenant_id`. A row with a different `tenant_id` is a security failure that raises immediately + logs to a high-severity alert.
	- Cross-tenant test in CI: synthetic constructor invocation with malicious `tenant_id` substitution attempts cannot return another tenant's data.
5. **`pipeline_log` events**:
	- `context_build_started`, `context_build_completed` with token counts per section, `context_truncation` events when truncation occurred, `context_overflow` events when overflow raised.
6. **Tests** in `apps/team/tests/agent/test_context_builder.py`:
	- Happy path: synthetic tenant with seeded ledger → valid context → token count under 8K.
	- Truncation path: oversized synthetic ledger → truncation kicks in → result still valid.
	- Overflow path: extreme case where even max truncation fails → raises `ContextOverflowError`.
	- Tenant isolation: two synthetic tenants with overlapping data → each gets only its own data.
	- Cursor accuracy: `meta.cursor_info` correctly reflects the data window included.

## Architectural rules to enforce in this subtask
- Pillar 1: the constructor lives in `apps/team/src/orchestrator/context_builder.py`. The agent (`apps/team/src/agent/`) does NOT build its own context. CodeX rejects PRs that move construction into the agent.
- Pillar 3: every DB read is tenant-scoped. The constructor assumes the orchestrator has already verified the tenant before invoking; it adds belt-and-braces assertion.
- Pillar 4: the bundle contains DATA, not conclusions. "Top spenders" is data; "this tenant should target lapsed customers" would be a conclusion and is forbidden in the bundle. The agent reasons; the bundle delivers.
- Pillar 8: token budgets and per-section limits are config-driven (`apps/team/config/context_budgets.yaml`). Adding new sections requires updating the config.
- The dataclass is typed strictly (mypy --strict). Optional fields are explicitly `| None`. No dict-of-anything escapes.
- Truncation is observable. Every truncation event lands in `pipeline_log` so we can monitor whether the 8K budget is too tight in practice.
- The bundle does NOT contain raw customer phone numbers or names. Customer-level data is anonymized to internal IDs; the agent works with IDs, and tools (VT-5.6, VT-5.7) resolve IDs to phone numbers at send time.

## Acceptance criteria
- `SalesRecoveryContext` dataclass typed strictly with all listed fields.
- `build_sales_recovery_context()` exists and is the sole constructor.
- Per-section token budgets enforced; over-budget sections truncated; overflow raises typed error.
- Cross-tenant test passes: malicious `tenant_id` substitution cannot leak other tenant's data.
- Truncation events logged to `pipeline_log`.
- No raw PII (phone numbers, names) in the bundle; verified by regex check on serialized bundle in CI.
- Token count enforcement tested end-to-end with synthetic data.
- mypy --strict passes on the new module.
- PR title references `(VT-SalesRecovery-Agent)`; reviewed by CoderX.

## Out of scope
- L1/L2/L3/L4 retrieval implementations (VT-7).
- Typed wrapper infrastructure (VT-8.1); this subtask uses scoped queries with assertion as bridging until that lands.
- The agent's interpretation of the bundle (the agent decides what to do with the data; this subtask just delivers it).
- Customer-level segmentation logic (lives in MCP tool `query_customer_ledger` VT-5.2).
- Bundle versioning / schema migration (Phase 1 ships v1; v2 is a future concern).
