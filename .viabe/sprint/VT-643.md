## Problem (deterministic misroute, not brain variance — verified 2/3 dev runs)
"Who are my top customers by spend?" was answered with the lapsed_list count template ("N of your
customers are lapsed … Want me to put together the full list?"). Root: `status_query._has_inactivity_cue`
(the SOLE consumer being lapsed_list at classify_status_query:180) had a bare digit+day-unit rule that
tripped on a co-present REVENUE window ("2 sales in 90 days") → `_has_list_cue` ("who are") ∧
`_has_inactivity_cue` → lapsed_list. The one clean run differed only in NOT quoting an "N days" number.

## Fix (deterministic, decision-independent, replay-safe)
1. Tighten the bare digit+day-unit branch: fires only when it co-occurs with a purchase/negation token
   or an elapsed-since phrasing (since/ago/din se/last visit…), so a revenue window can't synthesize a
   dormancy route. Touches ONLY lapsed_list routing (its sole consumer) — no money/consent gate.
2. New `top_spend` qtype: a ranking/value cue + a customer/spender noun, excluding dormancy asks →
   answers deterministically from `CustomersWrapper.top_customers_by_spend` with the ₹ ranking + total
   count. **id/₹ only — names Fazal-gated (CL-2026-07-12 CD2; never inlined), honest "names not in this
   chat view yet" note.** Removes the brain-anchor dependency → Tier-1 guaranteed, not variance-hoped.

## Dev re-drive round 1 (sha 5ead836) — 2/3, exposed 2 residual over-fires of the SAME class
The harness owner-persona varies phrasing per run; round 1 surfaced two revenue/ranking phrasings the
"sales"-only unit test missed, both routing a top-spend ask into a dormancy/count answer:
- **run 2** "top customers by total spend / **order count** … ₹220 for **90 days**" — "order" (a
  RANKING metric) + "90 days" (a revenue back-reference) tripped the digit+day RECENCY inactivity cue.
  Fix: a bare PURCHASE word no longer co-qualifies the day-window (genuine dormancy carries a negation
  or an elapsed-since phrase; a pure-positive "ordered in 90 days" is ACTIVITY, not dormancy).
- **run 3** "top customers by total spend … from **order** history, **not** estimates" — the bare
  `(purchase & negation)` SET cue over-fired (the 'not' negates 'estimates', not 'ordered'). Fix: new
  `_adjacent_negation_purchase(norm)` requires the negation to IMMEDIATELY neighbor the purchase token
  — implementing the adjacency the line-144 comment already stated as intent. `_has_inactivity_cue` is
  private to status_query (contained blast radius; money-adjacency validated on dev x3 next).
All 3 round-1 dev phrasings now classify → top_spend deterministically (was 1×lapsed_list, 1×brain-luck).
