# CL-366 — BLOCKER (process, CRITICAL): main branch has ZERO protection — gh api returns 40

gh api repos/rkecom-in/viabe-team/branches/main/protection returns 404 'Branch not protected'. main has NO branch protection: no required pull request, no required review, no required status checks, no direct-push restriction, no admin inclusion. CI is advisory, not a merge gate.

Impact: every 'Fazal merges / Clau drafts, Fazal approves' assumption in the operating model has been UNENFORCED. PR #44 (L1 KG) merging un-reviewed was not a bypass — it was the default behaviour of an unprotected branch. Commit 560106f landing on main post-#44 was the same. Anything with push access writes straight to main.

Severity: CRITICAL (process integrity). Damage to date is bounded — #44 audited clean bar one HIGH now being closed by PR #49 — but that is luck, not a control.

This is a Fazal-owned repo-governance decision, NOT Clau-owned. Clau recommendation for the protection rule on main: (1) require PR before merge, no direct push; (2) require 1 approving review; (3) require blocking status checks to pass (test, orchestrator, migrations, lint, typecheck, gate-* set); (4) include administrators. Caveat surfaced to Fazal: as sole human, he cannot approve his own PRs under GitHub rules — self-authored PRs would need the admin-merge path; protection still worth it (converts the review gate from habit to enforcement, makes CI blocking).

Status Open until Fazal sets the protection rule.
