## Ask (Fazal, 2026-06-11)

"We must be able to see the /team/signup page on the dev environment." Today the page
redirects to `/team` because `launchMode()` ≠ `live`, and the signup API proxy 404s unless
`ENABLE_PUBLIC_SIGNUP=true`.

## Contract

1. Enable the signup page on the `viabe-team-dev` Vercel deployment: whatever combination of
   launch-mode env + `ENABLE_PUBLIC_SIGNUP` (+ redeploy, since launchMode is build-time)
   makes `/team/signup` render the real form with the OTP step reachable. DEV env vars are
   CC-managed autonomously (CL-431); report changes by variable NAME.
2. **DEV ONLY.** No prod flag, no prod project (doesn't exist yet — VT-231), no main-branch
   change. If any code change is needed (e.g., a dev-mode allowance in launchMode), it is a
   [BUILD] PR into dev.
3. **CL-422 exposure check:** the dev vercel.app URL is reachable by strangers; an exposed
   live form could let a real third party OTP-verify and create a tenant on the Seoul dev DB
   (real-PII vector). VT-326's OTP-before-create + per-IP throttle ARE landed, but prefer
   keeping the form non-public: enable/confirm **Vercel Deployment Protection** (or
   equivalent) on `viabe-team-dev` so Fazal/internal can view it but the public can't.
   If protection is already on, say so and move on. If protection is impossible/disruptive,
   STOP and signal options instead of exposing.
4. Acceptance: `/team/signup` renders the form (EN+HI) on the dev deployment, OTP step
   reachable; `/team` still renders for the public path per VT-372's fix.

## Canary

The acceptance IS the rendered-page check on the live dev deployment (real deploy, real
page; fail-not-skip).

## Merge class

Config-first; code (if any) = [BUILD] self-merge on green per CL-429. Config-only outcome →
task-result signal with var NAMES + protection state, no PR.
