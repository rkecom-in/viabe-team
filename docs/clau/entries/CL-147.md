# CL-147 — Phase 1 pin discipline policy locked

Step 0 of the dep-audit migration sequence (CL-142). Decided once here, applies to all subsequent migration PRs and every future PR that adds or changes a runtime dep.

=== PYTHON DEPS (apps/team-orchestrator, apps/team-ingestion-worker, future Python apps) ===

EXACT PINS (`==`): semantic-affecting runtime libs where minor versions can change behavior, deprecate APIs, or break call sites silently.

- langgraph
- langchain
- langchain-core
- langchain-anthropic
- langgraph-supervisor
- langgraph-checkpoint-postgres
- dbos
- pydantic
- supabase (if/when re-added per CL-143 deliberation conditions)
- psycopg
- psycopg-pool
- anthropic (if/when direct SDK calls justify re-adding)
- sarvamai (post-rename per step 1)
- pgvector (Python type adapters; pinned because vector wire format compatibility matters)

MINIMUM PINS (`>=`): framework/stable-API libs where minor versions are additive, plus dev/test tooling.

- fastapi
- uvicorn
- pyyaml
- twilio (Python SDK; stable wire API)
- pytest, ruff, mypy, and other dev/test deps (declared in `[tool.uv.dev-dependencies]` or equivalent)

=== FRONTEND DEPS (apps/team-web, packages/team-shared) ===

DEFAULT — caret (`^`): npm convention, minor + patch updates allowed within current major. Applies to:

- next, react, react-dom
- tailwindcss
- clsx, class-variance-authority
- lucide-react (note: 0.x means each minor is a major; revisit if it bites)
- recharts (post step 4 v3 bump)
- twilio (node)

EXACT PINS (no caret/tilde): libs where major-version coupling to another framework's major is silent-failure territory.

- tailwind-merge — pin exact post-v3 bump because v2.x silently mis-merges against Tailwind v4 (CL-141 lesson)

=== COROLLARY: NEW-DEP REVIEW GATE ===

Every PR that adds a new runtime dep (any app, any language) MUST in its PR description:

1. State the new dep name, version, and pin style (`==X.Y.Z`, `>=X`, `^X.Y.Z`, exact).
2. Cite Context7 validation per CL-135 — library ID + the specific snippet/page that confirms current API matches our usage.
3. State a one-line rationale: what problem this dep solves that can't be solved with deps already on the project.
4. State pin justification: why exact / minimum / caret per the policy above.

Reviewer (CodeX + Clau) verifies all four before approving.

=== TRANSITIVE DEP RULE ===

Never declare a transitive dep at the top level as a hedge (the anthropic-via-langchain-anthropic mistake from the audit). If we need direct API access to a transitive dep, add it deliberately with a real call site in the same PR, not 'just in case.'

=== ENFORCEMENT ===

Step 1 PR (sarvam rename) is the first PR that applies this policy. Subsequent migration PRs (langgraph, tailwind-merge, frontend bundle, unused cleanup) all conform. Any drift caught in future audits comes back as a Correction entry.
