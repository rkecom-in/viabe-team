# CL-278 — DECISION (Fazal): self-evaluate gate REVISE contract — all-reasons verdict, exac

Fazal locked the self-evaluate gate REVISE-verdict contract for v1. CONTRACT: (1) self_evaluate must return ALL failure reasons in one pass, not break at the first — the verdict model carries a complete list/collection of issues. (2) On a REVISE verdict the gate performs EXACTLY ONE retry: the agent is re-run with the full prior failure-reason list injected into its context so the revision targets the specific issues. (3) Retry PASS -> output ships to owner surface. (4) Retry still REVISE -> run fails as SELF_EVAL_REJECTED and escalates to Fazal (concierge handles the tenant). No second retry. No ship-with-flag. (5) Fixed N=1 retry chosen over an open MAX_REVISIONS ceiling: no non-termination risk, no new tunable constant, cost ceiling is a predictable 2x the eval path. RATIONALE for the minimal terminal: Fazal wants production REVISE-frequency and failure-nature data before building smarter terminal logic; refinement explicitly deferred. SCOPE BOUNDARIES this creates — carried into the wiring brief Step-0 pre-flight as ground-truth gate items (cannot be assumed, per CL-194): (a) the verdict model VT-50/PR#37 actually shipped must be confirmed to expose ALL reasons as a list; if it returns a scalar reason, the wiring subtask must also widen the tool's verdict model or a separate tool-change subtask is required — surface to Fazal before drafting code; (b) VT-35 hard-limit budgets (token + wall-clock) must be confirmed per-run vs per-node — the retry puts two agent runs inside one node invocation; if budgets are per-node they need a documented bump or a legitimate retry trips a hard limit and misreports as AGENT_HARD_LIMIT_BREACH. Pillar 8 holds: one evaluator, one gate, no parallel path. Applies to ViabeTeam_Sprint row 367387c2-cc5a-8108-8119-d14d0ad7c809 (self-evaluate wiring, Exec Order 6.5).
