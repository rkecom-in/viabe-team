# CL-95 — sarvam-ai-client doesn't exist on PyPI

apps/team-ingestion-worker/pyproject.toml declares `sarvam-ai-client` as a dependency, but no package by that exact name is published on PyPI. `uv lock` / `uv sync` fail for the worker. Pre-existing from VT-Foundation (PR #3) — the worker hasn't been exercised at install-time since.

CURRENT WORKAROUND: PR #14 (mypy non-blocking job) uses `uv run --no-project` for the worker mypy step, sidestepping the lock failure.

LIKELY ROOT CAUSE: the official Sarvam Python SDK is published as `sarvamai` (no hyphens, no `-client` suffix). Quick check on pypi.org confirms `sarvamai` exists; `sarvam-ai-client` does not.

FIX SCOPE:
- apps/team-ingestion-worker/pyproject.toml: rename dependency to `sarvamai>=<version>`.
- Verify the import paths used in any sarvam-touching code (likely VT-13 / Sarvam STT subtask, not yet shipped) match the renamed package's import root.
- `uv lock` succeeds.
- Remove the `--no-project` workaround from the worker mypy step in ci.yml.

PRIORITY: Medium. Doesn't block current work — worker has no shipped code that imports sarvam yet, and tests run via --no-project. Becomes blocking when VT-13 (Sarvam STT ingestion) ships and the worker actually needs the SDK installed.

SEPARATE PR: title 'fix(worker): correct Sarvam SDK package name (sarvam-ai-client → sarvamai) (VT-Foundation-fix-2)'. Bundle into VT-13 prep work or ship standalone whenever it's convenient.
