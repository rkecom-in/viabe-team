# CL-111 — Handler output not persisted to pipeline_steps

Direct-handler return values (send_result, side-effect flags) are not persisted to pipeline_steps in VT-3.3c. webhook_pipeline_run runs HANDLERS[name](event, state) and discards the return.

Observability gap: after a handler runs, the only record is (a) a structured log line in stdout, (b) the underlying side effect, (c) the Twilio outbound message in Twilio Console. No pipeline_steps row captures the handler's actual return contract.

For Phase 1 with 10 design partners, log-line + Twilio Console is adequate. But for audit trail / billing reconciliation / Pillar 7 owner-truth at scale, the handler's send_result needs to land in pipeline_steps.

FIX SCOPE: in webhook_pipeline_run, after HANDLERS[name](event, state), wrap into a @DBOS.step that writes pipeline_steps row with step_kind='handler_executed', output_envelope=handler_return_dict. Touches runner.py. Most natural home: VT-122 (observability writer) which already TODOs the broader pattern, OR a small VT-3.3a-fix-3 if VT-122 is too far out.

PRIORITY: Medium. Not blocking soft launch. Becomes important when (a) design partners reach 50+ and Twilio Console review stops scaling, or (b) billing reconciliation (VT-Attribution) needs to verify handler-triggered sends per tenant.

SCHEDULE: bundle with VT-122 if that ships in Sprint 2; else create VT-3.3a-fix-3 standalone.
