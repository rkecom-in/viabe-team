# CL-14 — Three-layer memory system live: project instructions (Layer 1) + Clau_Session_Lo

Three-layer state model: Layer 1 = User-maintained project instructions (read-only at session start; rarely changes; you maintain). Layer 2 = Clau_Session_Log (THIS database; written incrementally by me as decisions land; read at session start). Layer 3 = Live databases (ViabeTeam_Sprint, Viabe_Launch_Tracker; queried as needed; canonical source of current state). Critical rule: I write to this log AT THE MOMENT a decision/blocker/correction happens, not at session end. End-of-session triggers are unreliable (context cutoffs, abrupt session ends); incremental writes mean the log is always current when next session reads it.
