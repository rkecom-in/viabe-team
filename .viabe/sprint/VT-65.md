## What CoderC must produce
The event-driven L1 KG population pipeline. Listens to canonical events from across the system; writes corresponding nodes + edges to the KG. Idempotent on event_id.
1. **Event consumer module** at `apps/team/src/kg/population.py`:
	- Subscribes to a logical event stream (Phase 1: a Postgres LISTEN/NOTIFY channel `kg_events`; consumers run inside the orchestrator process and the ingestion process).
	- Each event has shape: `{event_id, event_type, tenant_id, payload, emitted_at}`. Event types: `tenant_created`, `customer_created`, `customer_updated`, `transaction_created`, `campaign_created`, `campaign_sent`, `attribution_created`, `platform_listing_updated`.
	- Consumer dispatches to the correct KG-write handler.
2. **Per-event handlers**:
	- `tenant_created`: create `Tenant` node + edges to `Locality` and `BusinessType` (creating those nodes lazily if they don't exist).
	- `customer_created` / `customer_updated`: upsert `Customer` node (phone hashed); edge `Tenant -[OWNS]-> Customer`.
	- `transaction_created`: create `Transaction` node + edges `Customer -[MADE]-> Transaction`.
	- `campaign_created`: create `Campaign` node + edge `Tenant -[SENT]-> Campaign`.
	- `campaign_sent`: add `Campaign -[TARGETED]-> Customer` edges for each customer in the cohort.
	- `attribution_created`: add `Campaign -[ATTRIBUTED]-> Transaction` edges; backfill `arrr_paise` aggregate on the Campaign node.
	- `platform_listing_updated`: upsert `PlatformListing` + `HAS_THEME` edges.
3. **Idempotency**:
	- Events table `kg_events_processed` (migration 020) with PK `event_id`. Consumer checks before processing; duplicate is no-op. Lets the system replay events safely (e.g., after a process crash).
4. **Backfill mode**:
	- On first deployment or after KG corruption, a one-shot backfill script reads canonical tables (`tenants`, `customers`, `transactions`, `campaigns`, `attributions`) and synthesizes events in chronological order. Lives at `apps/team/scripts/kg_backfill.py`. Documented in `docs/runbooks/kg-recovery.md`.
5. **Failure handling**:
	- Per-event handler wrapped in try/except. Failure routes to error framework (VT-3.6) as `database_error`. Event marked failed in `kg_events_processed`; backfill can retry.
	- Consumer never crashes the orchestrator process; failed events accumulate without blocking forward progress.
6. **Cross-tenant safety**:
	- Every handler receives the event's `tenant_id` and asserts that all KG writes use it. Mismatched tenant_id between event payload and write-target is a logged security failure.
7. **Tests** at `apps/team/tests/kg/test_population.py`:
	- Each event type produces correct KG state.
	- Event idempotency: duplicate event_id is no-op.
	- Crash recovery: synthetic crash mid-batch, restart ŌåÆ unprocessed events replay correctly.
	- Backfill: synthetic canonical-table state ŌåÆ backfill produces equivalent KG state.
	- Cross-tenant: malicious event with tenant_A payload but tenant_B writes ŌåÆ rejected.
	- LISTEN/NOTIFY plumbing: event emit ŌåÆ consumer receives within 1s.

## Architectural rules to enforce in this subtask
- Pillar 4: handlers write facts as received in events. They do NOT enrich or infer. Inferred fields (e.g., "customer is a regular") are agent reasoning, not KG state.
- Pillar 8: one consumer pipeline. Ingestion methods + orchestrator EMIT events; this subtask CONSUMES them. The two sides talk only via events.
- Idempotency on event_id is mandatory. CodeX rejects handlers without idempotency check.
- Failed events do NOT block forward progress. Backfill recovers them.
- Backfill runbook is a Phase 1 deliverable (`docs/runbooks/kg-recovery.md`); ops needs to know how to recover.
