"why" across ALL progression fails, they are the SAME failure: **the manager loops / stalls / re-asks instead
of advancing.** Three arms:

## Arm 1a — dispatch brain has no anti-loop rule (FIXED here, cheap)
The dispatch/handle-directly prompt (`orchestrator_agent_system.md`) had brevity + honesty rules but NO
"always advance / never re-ask" rule. Added one: never re-ask a fact already given, never repeat a prior
message/link, on an impatient repeat give the result / concrete options / best-effort / escalate (never re-send
"I'm on it"), a specialist hand-off reply must carry the RESULT. Cheap prompt lever; expected to lift some
dispatch-side narrow misses. NOT deployed solo — bundling with the structural fixes to measure once.

## Arm 1b — journey brain re-ask loop is a RECORDING/CONTEXT bug, NOT a prompt gap
`turn_brain.py` ALREADY forbids re-asking (lines 88/105-111 "re-asking is the single most annoying failure").
Yet longhaul_retention_multifact "repeatedly re-asked business name across steps 1-5 without accepting 'Sharma
Store'", happy_path_profile "kept confirming/asking around category without locking the profile", bilingual_
hinglish "hard loop repeating the same confirmation 3x". The owner GAVE the fact; it wasn't ACCEPTED/recorded →
field stays "still needed" → re-ask. Root-cause = the extraction/confirm/record path or the context the brain
sees, NOT the prompt. **Next: root-cause why a stated fact isn't captured/carried.** (Prime suspect: the
confirm_draft/record path or active-window/business_profile substrate — business_profile is stubbed empty.)
