## What CoderC must produce
The Pre-Filter Gate (Stage 1 of the two-stage filter). **REPURPOSED FROM PRIOR 'multi-specialist hooks' subtask** since langgraph_supervisor now handles multi-specialist registration natively.
Deterministic regex/signature-based filter. Catches ~70% of incoming events at ₹0 LLM cost. Only the ~30% requiring judgment passes through to the orchestrator-agent brain (Opus 4.7, VT-3.9).
**Architectural intent:** the orchestrator-agent is expensive (Opus 4.7 per invocation). Most incoming events are routine: `STOP` opt-outs, DSR keyword requests, duplicate webhook deliveries, template error replies, simple status pings. These don't need the brain. A cheap deterministic gate catches them and routes to direct handlers.
1. **Pre-Filter Gate** in `apps/team-orchestrator/src/orchestrator/pre_filter_gate.py`:
	- Function `pre_filter(event: WebhookEvent, tenant: Tenant) -> PreFilterResult`.
	- Wrapped in `@DBOS.step` for durable execution + step record.
	- Returns one of:
		- `RouteToDirectHandler(handler_name: str, payload: dict)` — routine path; goes to a direct handler
		- `RouteToBrain(reason: str)` — needs orchestrator-agent reasoning
		- `Reject(reason: str)` — duplicate, signature failure, rate limit
2. **Direct Handlers** in `apps/team-orchestrator/src/orchestrator/direct_handlers/`:
	Phase 1 implements these deterministic handlers (no LLM):
	- `opt_out_handler.py` — message body matches `STOP / UNSUBSCRIBE / चाहिएं नहीं / बंद करो` (and other approved opt-out keywords from VT-Privacy). Sets tenant opt-out flag, sends opt-out confirmation template, ends workflow.
	- `dsr_handler.py` — message body matches DSR keywords ("data deletion", "my data", "GDPR", "मेरा डेटा"). Sends DSR acknowledgment template ("we received your request; we'll respond within 30 days per DPDP"), creates DSR ticket, ends workflow.
	- `dupe_handler.py` — DBOS workflow_id already executed (idempotency). Returns prior result; ends workflow.
	- `status_ping_handler.py` — message body is simple status query ("how's it going", "any updates"). Sends auto-generated status from current SubscriberState (last campaign, next scheduled action). Deterministic; no LLM.
	- `template_error_handler.py` — Twilio status callback indicates template send failed. Logs to pipeline_steps, triggers retry per VT-3.6 framework if eligible, escalates if not.
3. **Pre-Filter rules** (regex + signature checks, no semantic analysis):
	- **Opt-out keywords (EN + HI):** exact match (case-insensitive) against approved keyword list from VT-Privacy. Approved list lives in `config/opt_out_keywords.yaml`.
	- **DSR keywords (EN + HI):** approved keyword list from VT-Privacy at `config/dsr_keywords.yaml`.
	- **Duplicate detection:** DBOS workflow_id already executed (handled at DBOS layer; pre-filter just confirms).
	- **Template status callbacks:** Twilio sends `delivered`, `read`, `failed` callbacks. `failed` routes here; others to observability writer.
	- **Status pings:** narrow regex pattern matching trivial status queries ("hi", "hello", "any update", "कायसा चल रहा है"). Conservative — only matches obvious status queries. Ambiguous routes to brain.
	- **Everything else:** routes to brain.
4. **Cost discipline:** Pre-Filter target is ~70% routine event capture. Track in observability: ratio of (RouteToDirectHandler + Reject) / total events. If ratio < 60% sustained, add keywords/patterns to broaden the deterministic capture. If > 85%, audit for false-positives (events incorrectly routed to direct handlers that needed reasoning).
5. **Tests** in `apps/team-orchestrator/tests/orchestrator/test_pre_filter.py`:
	- Opt-out keyword in EN → RouteToDirectHandler('opt_out_handler') → opt-out confirmation template sent → tenant opt-out flag set.
	- Opt-out keyword in HI → same behavior.
	- DSR keyword → RouteToDirectHandler('dsr_handler') → DSR ack sent + ticket created.
	- Twilio status callback `delivered` → routes to observability (not pre-filter rejection).
	- Twilio status callback `failed` → RouteToDirectHandler('template_error_handler').
	- Status ping "hi" → RouteToDirectHandler('status_ping_handler').
	- Substantive message ("I want to launch a campaign for dormant customers") → RouteToBrain with reason "substantive owner message".
	- Ambiguous ("how are things going with the cafe customers") → RouteToBrain.
	- Capture-ratio test on synthetic event mix → 60-80% direct handler rate.
6. **Direct handler honesty rules:**
	- opt_out_handler MUST send confirmation template (Pillar 7 — owner-truth).
	- dsr_handler MUST send acknowledgment template + create ticket within same DBOS step.
	- status_ping_handler MUST report accurate state (no padding, no overstatement).

## Architectural rules to enforce in this subtask
- **Pillar 1 (revised):** Pre-Filter Gate is FULLY DETERMINISTIC. NO LLM call. CodeX rejects any PR that adds LLM call to pre-filter or direct handlers.
- **Pillar 8 (no patchwork):** Pre-Filter is ONE gate. Adding new direct handlers = new file in `direct_handlers/` + entry in pre-filter rules. CodeX rejects shadow filtering elsewhere.
- Keyword lists config-driven (`config/opt_out_keywords.yaml`, `config/dsr_keywords.yaml`). Adding keywords = YAML PR. Removing privacy-related keywords (opt-out, DSR) = Type 2 governance.
- Conservative regex matching. False-routing-to-direct-handler is worse than false-routing-to-brain (brain handles routine events fine, just at cost; direct handler mistakenly handling a substantive message is a privacy/UX risk).
- Each direct handler wrapped in `@DBOS.step`. Auto-resume on crash.
- Pre-filter capture ratio monitored in observability. Tunable via YAML keyword adjustments.
