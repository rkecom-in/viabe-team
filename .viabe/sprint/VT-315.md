# VT-315 — strip raw PII from kg_events outbox payloads (CL-390)

Surfaced by the VT-71-capstone adversarial moat-verification workflow (5 confirmed
findings; the moat is otherwise SOUND — k-anon gates, l3_patterns CHECK, RLS
cross-tenant isolation, L1 hash-only, L2/L3/L4 non-propagation all verified clean).

## The one real crack (3 findings, 1 root cause) — CL-390 PII-at-rest
The VT-65 PR-2 atomic-emit pattern treats `kg_events` as an outbox, but rows are
NOT deleted on drain (only `drained_at` is stamped) — so any raw PII in
`payload` persists in plaintext **durably**. Three emit sites leak:
1. **HIGH** — `dedup_merge.py:193-194` CUSTOMER_CREATED emits raw `phone_e164`.
2. **MEDIUM** — `dedup_merge.py:170-172` CUSTOMER_UPDATED emits raw `phone_e164`.
3. **HIGH** — `tenant_provision.py:71-73` TENANT_CREATED emits `business_name`
   that falls back to the raw phone (`business_contact`) when no name is supplied.

The handlers hash correctly into L1 (`_h_customer_upsert` → `hash_phone`; only
`phone_hash` reaches the KG). RLS keeps it tenant-scoped — **NO cross-tenant leak,
NO L2/L3/L4 propagation** (CUSTOMER_*/TENANT_CREATED don't project to
episodic_events). It is a within-tenant PII-minimization / encryption-at-rest
violation — Cowork: MUST-FIX before Reports-Jun15 (CL-390 is a privacy commitment).

## Fix (one coherent PR — CL-424)
**A. Emit-time (new writes clean):**
- `dedup_merge.py`: emit `{"customer_id": ..., "phone_hash": hash_phone(phone)}`
  (not raw `phone_e164`), both CUSTOMER_CREATED + CUSTOMER_UPDATED.
- `kg_population._h_customer_upsert`: read `p["phone_hash"]` directly (no re-hash);
  keep a `phone_e164`→hash fallback for any in-flight legacy event.
- `tenant_provision.py`: emit `{"business_name": business_name}` (the real param,
  None when absent) — NOT the `name` phone-fallback. The DB row keeps the phone
  fallback (tenant identity); the EVENT payload must not carry it.
**B. Backfill (existing rows clean) — mig 088:**
- `UPDATE kg_events SET payload = payload - 'phone_e164' WHERE event_type IN
  ('customer_created','customer_updated')`.
- tenant_created: null `business_name` in payload where it matches a phone regex
  (`^\+?[0-9]{8,}$`) — the L1 node already has it; the payload is vestigial.

## Canary (DR-15, CL-422 synthetic)
- emit CUSTOMER_CREATED/UPDATED → assert kg_events.payload has NO `phone_e164`,
  has `phone_hash`; drain → L1 customer node phone_hash unchanged (== hash_phone).
- TENANT_CREATED with no business_name → payload business_name is NULL (not the phone).
- backfill: seed a legacy row with raw phone → run backfill → phone gone.
