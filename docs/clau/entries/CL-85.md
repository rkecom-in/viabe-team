# CL-85 — Notion subtask Status sync on PR merge

DECISION (Fazal 2026-05-18 in response to audit CL-77 meta-observation): Notion subtask Status sync happens at PR-merge time, not via periodic sweep. The merge ritual gets one extra step.

NEW MERGE RITUAL (post-`gh pr merge N --squash --admin --delete-branch`):
1. Identify the subtask(s) referenced in the PR title (regex (VT-[0-9A-Za-z.]+)$).
2. For each, set Notion Status property to 'Done' via Notion MCP update_properties (single-page targeted update — not bulk — so it survives the tool-approval layer; see CL on update_properties single-page workaround).
3. If the PR spans multiple subtasks, update each.
4. If a PR fixes a Blocker entry in Clau_Session_Log, also mark that entry Status='Resolved' in the same step.

Whose job: Claude Code, as part of the merge sequence. Add to .claude/CONVENTIONS.md so it's instruction-discoverable. Clau (this session class) does the same for any direct PRs.

RETROACTIVE APPLICATION:
The 6 currently shipped-but-still-Backlog subtasks (VT-19, VT-20, VT-24, VT-25, VT-26, VT-31) need status sweep to 'Done' now. Note VT-19 is special — per CL on schema execution order, it's partially done (tenants + DBOS infrastructure) and partially superseded (domain tables). For VT-19 specifically: keep Status='Backlog' or use 'Superseded' if that status exists in the ViabeTeam_Sprint schema; if not, leave Backlog and add a note in the body. The other 5 → 'Done'.

Not logging the sweep itself as separate CL entries — just do it. If something breaks, the Notion audit log captures it.
