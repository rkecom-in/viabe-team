## What CoderC must produce
The Method 9 ingestion adapter. Owner types entries directly into WhatsApp ("Add Rajesh, phone 9876543210, last visit yesterday, spent 800" or "नया customer Sunita, 8765432109, परसों आया था, 1200"). Adapter parses the prose into structured customer + transaction rows.
1. **Adapter module** at `apps/team/src/ingestion/methods/owner_typed.py`:
	- `ingest_owner_typed(tenant_id, message_body, run_id) -> IngestionSummary`.
	- Triggered by the orchestrator when `classify_owner_message` returns intent = `owner_typed_entry` (a 10th classification intent added in VT-5.11; the original 9 intents become 10 with this method).
	- Wait: this changes VT-5.11. Update plan: when this subtask lands, VT-5.11's prompt is bumped to v2 with the 10th intent. Documented as a known follow-up.
2. **Parsing pipeline**:
	- Phase A — Haiku-backed extraction: small LLM call to extract candidate customer entries from the message. Prompt instructs the model to return strict JSON: `[{name, phone (E.164 or null), last_visit_date (ISO or null), total_spend_paise (int or null), notes (string or null)}, ...]`. Model is told: do NOT invent missing fields; null is correct for unknowns.
	- Phase B — Deterministic post-processing: phone normalization via `phonenumbers` library; date phrase resolution ("yesterday" → today − 1, "परसों" → today − 2 with tenant timezone); amount parsing handles "1200" / "₹1200" / "1.2k" / "१२००".
3. **Confidence**:
	- Per-field confidence from the Haiku extraction (the prompt instructs return of confidence per field).
	- Confidence < 0.7 routes to clarifying flow (VT-6.2).
	- Owner-typed input has higher trust default than image extraction; threshold for clarification is 0.6 (slightly more lenient — they typed it deliberately).
4. **Dedup**: per VT-6.3.
5. **Confirmation reply**:
	- On successful commit: short confirmation back to owner: "Added Rajesh (98765*) — last visit yesterday, ₹800 spent." Phone digits partially masked in the confirmation for owner-confidence-without-overexposure.
	- On clarification: question goes via VT-6.2.
	- On parse failure: "I couldn't quite parse that. Could you try again with a format like 'Add [name], phone [number], visited [date], spent ₹[amount]'?" with example included.
6. **Tests** at `apps/team/tests/ingestion/methods/test_owner_typed.py`:
	- English happy path: "Add Rajesh, 9876543210, last week, 800" → 1 customer, 1 transaction created.
	- Hindi happy path: "नया customer Sunita, 8765432109, परसों आया, 1200" → correct extraction.
	- Mixed-language: typical Indian SMB phrasing parses correctly.
	- Multiple entries in one message: "Add Rajesh 98765 800 yesterday, also Mahesh 87654 500 today" → 2 customers, 2 transactions.
	- Missing fields: "Add Rajesh" with no phone or amount → low-confidence fields → clarification.
	- Date phrasing: "yesterday", "परसों", "last Tuesday", "3 days ago", "15 May" all resolve correctly.
	- Amount phrasing: "800" / "₹800" / "1.2k" / "१२००" all → correct paise.
	- Cross-tenant: tenant_A typed entry cannot create tenant_B rows.
	- PII absence in `pipeline_log`: regex check.
	- Dedup integration: typed entry matching existing customer → merged (additive).
	- Real-API integration test (env-gated).

## Architectural rules to enforce in this subtask
- Pillar 4: missing fields = null, not invented. CodeX rejects fallback-to-default-value logic.
- Pillar 7: owner is source of truth; their typed input is canonical. Confidence threshold lower than image methods (0.6 vs 0.7).
- Pillar 8: shared dedup, shared clarifying flow, shared phone normalization. The adapter is thin coordination.
- Haiku model pinned in `models.yaml` under `owner_typed_extraction`.
- Date resolution uses tenant timezone (default Asia/Kolkata; captured at sign-up).
- Amount parsing handles Devanagari numerals via `unicodedata` library, not custom regex.
- Confirmation reply masks phone digits (last 4 visible only) — small privacy gesture; CodeX confirms.
- VT-5.11's prompt versioning bump (v1 → v2 to add 10th intent) is a follow-up action item logged in this PR's description.

## Out of scope
- Voice-typed entries (different method; voice goes through VT-6.8 cash-book + voice path).
- Bulk owner-typed ("add these 50 customers" pasted as a list) — Phase 1 is small inline messages; bulk is paper-book + photo (VT-6.4) or contacts (VT-6.5).
- Updates to existing customers via typed message ("Rajesh's phone changed to 99999") — Phase 2 if owners ask for it.
- Transaction-only entries without customer ("Rajesh paid 200 today") — Phase 1 supports this since dedup will match Rajesh; explicit support for typed-transaction-only is NOT required separately.
- Free-form deletion via typed message ("remove Rajesh") — opt-out flow lives in VT-8.5; explicit deletions go through DSR.
