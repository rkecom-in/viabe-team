# VT-225 — L0 per-tenant k-anonymity admission gate

## Why

VT-196 shipped L0 production write wiring with the consent gate (CL-390) + read-side k-anonymity retention. But the per-tenant admission gate was schema-locked — the brief asked for "k=10 distinct tenant_ids" as the write-admission condition, but the schema for tracking tenant participation per (business_archetype, signal_kind) cell needs a decision before implementation.

Two paths:

**Option A** — add `contributors UUID[]` column to `l0_fragments` aggregate rows; admission gate computes `array_length(contributors, 1) >= 10`. Concurrent-write race possible without `FOR UPDATE`.

**Option B** — new table `l0_cell_contributors(business_archetype, signal_kind, tenant_id)` with UNIQUE constraint; admission gate does `SELECT COUNT(DISTINCT tenant_id) FROM l0_cell_contributors WHERE ...`. Cleaner but adds a join.

## Brief contract

CC drafts a 1-page design doc at `docs/clau/l0-kanon-admission-design.md` covering:
1. Schema option A vs B + recommendation with rationale
2. Concurrency semantics (advisory lock vs FOR UPDATE vs accept-race-with-eventual-consistency)
3. Migration plan (extend migration 040 OR new migration)
4. Canary delta: extend `vt196_l0_prod_writes.py` with admission-gate assertions A7-A10

Fazal reviews design doc → CC implements per locked option.

## Acceptance

- Design doc lands at `docs/clau/l0-kanon-admission-design.md`
- Migration + admission gate code merged
- Canary 4 new assertions PASS:
  - A7: cell with 9 distinct contributing tenants → reject 'k_anon_insufficient'
  - A8: cell with 10 distinct contributing tenants → admit
  - A9: same tenant inserting twice → count unchanged
  - A10: concurrent writes preserve k=10 boundary (no over-count)

## Out of scope

- Read-side k-anon (already in VT-196)
- L0 schema changes beyond admission tracking
- L1 substrate (VT-195 — gated on separate concept doc)
