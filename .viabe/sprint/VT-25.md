# VT-25 — SubscriberState schema and phase transitions

VT-Orchestrator (respec'd 2026-05-12 for orchestrator-as-agent + DBOS + langgraph_supervisor).

## What CoderC must produce
The canonical state schema for a subscriber's lifecycle, plus the deterministic phase-transition machine. The phase transition machine is deterministic state-machine code (no LLM); the orchestrator-agent (Opus 4.7, VT-3.9) reads phase from state and reasons about coordination but never mutates phase directly. The agent (Tier 2) and tools (Tier 3) consume the state read-only.
1. **Expand `OrchestratorState` to `SubscriberState`** in `apps/team-orchestrator/src/orchestrator/state.py`:
	- `tenant_id: UUID`
	- `run_id: UUID`
	- `phase: Literal['onboarding', 'trial', 'trial_extended', 'paid_active', 'paid_at_risk', 'cancelled', 'refunded']`
	- `phase_entered_at: datetime`
	- `trial_started_at: datetime | None`
	- `trial_extension_count: int` (default 0; max 3 extensions to cover 14→60 days)
	- `paid_conversion_at: datetime | None`
	- `last_campaign_at: datetime | None`
	- `attribution_close_pending: list[campaign_id]` (campaigns awaiting T+7 attribution close)
	- `total_recovered_paise: int` (running cumulative attributed recovered revenue) — NOTE 2026-05-18: renamed from `total_arrr_paise` (three r's, typo) per Clau_Session_Log CL-80. Avoids ARR/Annual-Recurring-Revenue acronym collision.
	- `cumulative_fees_paid_paise: int` (running cumulative subscription fees since paid conversion)
	- `escalation_pending: bool` (set when an `escalate_to_owner` event has been emitted but not yet resolved by owner reply)
	- `last_owner_message_at: datetime | None`
	- `history: list[dict]` (append-only event log within the run; complements `pipeline_steps` from VT-122)
2. **Phase transition table** in `apps/team-orchestrator/src/orchestrator/transitions.py`:
	- A `TRANSITIONS` dict mapping `(from_phase, event)` → `to_phase`. Events: `signup`, `card_captured`, `trial_extension_granted`, `trial_extension_exhausted`, `weekly_low_engagement`, `engagement_recovered`, `cancellation_requested`, `day39_refund_triggered`, `day39_continue`, `manual_cancel`.
	- Every transition has a deterministic precondition (e.g., `trial_extension_granted` requires `phase == 'trial'` AND `trial_extension_count < 3`).
	- Invalid transitions raise `InvalidTransitionError` with the from/to/event triple.
3. **Transition function** `@DBOS.step` decorated `apply_transition(state, event, context)`:
	- DBOS step boundary: this is the atomic transition unit. DBOS auto-resumes from here on crash.
	- Validates the (from_phase, event) pair against `TRANSITIONS`.
	- Updates `phase`, `phase_entered_at`, and side-effect fields (e.g., `trial_extension_granted` increments `trial_extension_count`).
	- Writes a `phase_transition` step_record to `pipeline_steps` via VT-122's `@observability.step` decorator (chained with `@DBOS.step` — single atomic boundary).
	- Persists new state to LangGraph checkpoint AND mirrors `phase` column to the `tenants` table (idempotent `UPDATE tenants SET phase = ..., phase_entered_at = ... WHERE id = tenant_id`).
4. **State invariants** in `apps/team-orchestrator/src/orchestrator/invariants.py`:
	- `paid_active` and `paid_at_risk` require `paid_conversion_at IS NOT NULL`.
	- `refunded` requires the day39 refund event in `pipeline_steps` with the corresponding `tenant_id`.
	- `trial_extension_count` cannot exceed 3.
	- `phase_entered_at` is monotonic-non-decreasing for a given tenant (no time travel).
	- Checked AFTER every transition; violation raises → DBOS rolls back the step → checkpoint not committed.
5. **Tests** in `apps/team-orchestrator/tests/orchestrator/test_state.py` + `test_transitions.py`:
	- Every valid transition fires correctly.
	- Every invalid transition raises `InvalidTransitionError`.
	- Invariant violations roll back via DBOS.
	- phase_transition step_records appear in `pipeline_steps`.
	- `tenants.phase` mirror reflects latest state after transition sequence.
	- DBOS crash mid-transition → auto-resume restores prior state cleanly.
	- **Lint test:** orchestrator-agent code AND specialist code cannot import `transitions.py`. CodeX enforces.

## Architectural rules to enforce in this subtask
- **Pillar 1 (revised):** orchestrator-agent and specialists READ phase via state but never mutate. Only `apply_transition` mutates. CodeX rejects any agent code that imports `transitions.py`.
- **Pillar 8 (no patchwork):** phase logic in ONE place (`transitions.py`). New events go there; no shadow phase machines.
- Transitions are deterministic. Same `(from_phase, event)` always produces same `to_phase`. No probabilistic logic.
- Invariant checks run AFTER every transition. Failed invariant → DBOS rollback.
- `tenants.phase` is denormalized mirror, not source of truth. Checkpointed state authoritative; mirror exists for SQL queries.
- `phase_entered_at` monotonic. Backdating requires service-role bypass + manual `pipeline_steps` annotation.
- DBOS step boundary == observability step boundary. Atomic write to both `dbos_workflow_steps` and `pipeline_steps`.
