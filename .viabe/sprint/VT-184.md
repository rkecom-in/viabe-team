# VT-184 — phone_token_resolutions + audit-logged resolution path

## Scope

Per design doc §2.1 (phone_token_resolutions table description) + §2.3 examples:

1. **Phone tokenization on write:** VT-104 already redacts phone numbers to opaque tokens. Extend the redactor (or wrap it) so when a phone is redacted as a `cust_tok_<hash>`, an entry lands in `phone_token_resolutions` mapping `cust_tok_<hash>` → real `phone_e164` + `tenant_id` + `customer_id` (if known; nullable until VT-170 customers table ships).
2. **Resolution API:** `resolve_phone_token(token, operator_id) -> str | None` — fetches `phone_e164`. Stricter RLS (operator role only). Logs the resolution into VT-150's privacy_audit_log (existing table, migration 008).
3. **Idempotency:** same phone hashed twice produces the same token + does not duplicate the resolution row.
4. **Salt management:** the existing `TEAM_PHONE_HASH_SALT` env (used by canary fixtures) is the salt. No new credential needed.

## Dependencies

- VT-178 (phone_token_resolutions table verified + RLS)
- VT-104 (PII redactor canonical entry)
- VT-150 (privacy_audit_log writer for resolution logging)

## Acceptance criteria

- Redactor + tokenization writes resolution row on first encounter; idempotent on subsequent.
- Operator-role resolve succeeds; tenant-app-role resolve denied via RLS.
- Every resolution call → audit log row in `privacy_audit_log`.
- Cross-tenant: tenant A's tokens cannot resolve under tenant B's GUC.
- Canary: real Supabase + redactor + audit log emission.
