## What CoderC must produce
The `self_evaluate` MCP tool. Opus-backed evaluator. Receives a draft `CampaignPlan` and returns `pass` or `revise` with structured feedback per the four evaluation criteria from VT-4.5.
1. **Tool implementation** at `apps/team/src/agent/tools/self_evaluate.py`:
	Input schema:
	- `draft_campaign_plan: dict` (the agent's draft, full `CampaignPlan` JSON).
	- `context_summary: dict` (compact context: business_profile basics, recent_campaigns count, attribution snapshot summary; used to evaluate consistency).
	- `attempt_number: int = 1` (1 or 2; on attempt 2, the prompt instructs the evaluator to be more lenient — if revise feedback was addressed reasonably, lean toward pass).
	Output schema:
	- `verdict: Literal['pass', 'revise']`.
	- `feedback: dict[Literal['schema', 'pillar', 'consistency', 'legal'], CategoryFeedback]` — per category:
		- `verdict: Literal['pass', 'revise']`
		- `issues: list[str]` (max 5 items, each ≤200 chars; empty if pass)
		- `severity: Literal['critical', 'major', 'minor']`
	- `overall_severity: Literal['critical', 'major', 'minor', 'none']` — highest across categories.
	- `model_used: str` (Opus 4.7 — pinned in models.yaml).
	- `tokens_used: int`, `cost_paise: int`.
2. **Evaluation logic via prompt**:
	- **Schema category**: the model checks the draft against the `CampaignPlan` schema's required fields and validators. Most schema violations are caught by the Pydantic model BEFORE this tool runs (cheaper); this category catches semantic issues (e.g., `cohort_size` claim 200 but `customer_ids` list has 87).
	- **Pillar category**: checks for invented numbers, per-vertical heuristics from training data, overstated confidence, retention pressure language.
	- **Consistency category**: cross-references draft fields with `context_summary`. E.g., draft proposes targeting 90-180d dormants but context shows zero customers in that band.
	- **Legal category**: checks message templates for prohibited content (high-pressure, misleading, unsubstantiated financial claims). For Phase 1, focuses on Indian DPDPA implications + Twilio/Meta content policy.
3. **Prompt** at `apps/team/src/agent/tools/prompts/self_evaluate_v1.md`:
	- System prompt: "You are an evaluator. Given a draft campaign plan, return STRICT JSON matching the output schema. Be specific in feedback — cite the exact field path and value when raising an issue. Do NOT rewrite the draft yourself; only critique."
	- Examples of good critique vs bad critique ("revise: cohort_size of 200 contradicts customer_ids list length of 87 — schema violation" is good; "this looks weird" is bad).
	- Token budget: under 3,000 tokens.
	- Frontmatter: version, token count, model.
4. **Model choice rationale** in `docs/team/llm-backed-tools-rationale.md`:
	- Why LLM, not deterministic: most schema violations ARE deterministic (caught by Pydantic). The cases where self_evaluate adds value are the *semantic* ones — "this draft is internally inconsistent" or "this message has subtle pressure language." Those need reading comprehension; deterministic rules can't cover them all.
	- Why Opus, not Sonnet/Haiku: this is the QUALITY gate. The agent's output ships unless this evaluator catches issues. False negatives cost trust with owners. Worth Opus pricing.
	- Cost: estimated ₹10-15 per evaluation. With ~50 evaluations per day across all tenants, ~₹500-750/day — within budget.
5. **Independence from agent**:
	- The evaluator does NOT see the agent's reasoning chain. Only the final draft + compact context. This enforces structural independence — the evaluator can't be fooled by the agent's own reasoning.
	- The evaluator does NOT call other tools. It reads inputs and returns verdict. Pillar 2: tools don't reason about side effects.
6. **Tests** at `apps/team/tests/agent/tools/test_self_evaluate.py`:
	- Pass case: clean synthetic draft → verdict pass, all categories pass.
	- Schema-revise: draft with `cohort_size=200` but `customer_ids` length 87 → verdict revise, schema category critical.
	- Pillar-revise: draft with invented persona traits not in evidence_refs → verdict revise, pillar category major.
	- Consistency-revise: draft proposes 90-180d cohort but context shows 0 such customers → verdict revise, consistency category major.
	- Legal-revise: draft with high-pressure language in template_params → verdict revise, legal category critical.
	- Multiple categories failing: highest severity is `overall_severity`.
	- Attempt 2 leniency: borderline draft on attempt 2 → lean toward pass.
	- Real-API integration test (env-gated): runs against actual Opus.

## Architectural rules to enforce in this subtask
- Pillar 2: LLM-backed; rationale documented. The choice of Opus (not Sonnet/Haiku) is justified for quality reasons.
- Pillar 7: structural independence. The evaluator does NOT see the agent's reasoning. Reading the agent's own thoughts would compromise the independence Phase 1 needs.
- The evaluator is stateless on tenant data beyond what's passed in. ctx.tenant_id logged only.
- Output is strict JSON. Free-form prose is a validation failure.
- Attempt-number-aware leniency is a deliberate Phase 1 design: it prevents infinite revise loops on borderline drafts. The threshold for "borderline" is encoded in the prompt, not in code; Type 2 to change.
- Prompt is versioned (`_v1.md`); evolves through Type 2 governance.
- This tool counts as one tool call against the agent's 25-call cap (per VT-4.5 contract).
