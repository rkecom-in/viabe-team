# VT-196 — L0 production write wiring (moat substrate)

## Why this row exists

VT-126 merged the L0 memory orchestrator-agent integration — the helper exists, the k-anonymity gate works, the cohort-keyed model is tested. But the orchestrator-agent has not been observed writing L0 fragments in production traffic because the brain wasn't wired until VT-193. Now that brain dispatches actually fire LLM calls, the L0 write path needs to actually fire.

Without L0 writes happening, the moat is half-built: agents will eventually need to know "what kinds of campaigns work on what kinds of customers" — that knowledge accumulates in L0. Today no rows are landing.

## Scope

1. **Verify the write path triggers.** Add a canary that runs a sequence of synthetic dispatches and asserts L0 fragments land (subject to k-anonymity gate — may require multiple synthetic dispatches across "tenants" to satisfy k=10 before writes pass).
2. **Audit the WHEN-to-write-L0 guidance in the system prompt.** Current text exists but agent may not be triggering writes. Tighten prompt to make L0 write conditions explicit (and after VT-194 lands, move this to a per-task injection rather than always-on).
3. **Surface L0 write metrics in Ops Console.** Read-only view: count of L0 fragments by cohort over time. Helps validate the moat is accumulating.
4. **L0 → L1 promotion path (forward-looking).** Document the proposed mechanism for promoting high-confidence L0 patterns into per-tenant L1 context. Defer implementation to VT-197 or later.

## Acceptance criteria

1. Synthetic dispatch sequence (e.g., 12 dispatches across simulated cohort) produces ≥1 L0 fragment row when k=10 threshold is met
2. Ops Console L0 panel renders cohort + fragment count + last-write timestamp
3. Privacy audit log shows zero policy violations for L0 writes (no tenant-identifying fragments admitted)
4. Per-dispatch latency overhead < 100ms (L0 write is best-effort; doesn't block agent response)

## Rule #15 Canary (mandatory)

`apps/team-orchestrator/canaries/vt196_l0_production_writes.py` — provisional 5 assertions:
1. Dispatch sequence triggers ≥1 L0 fragment write
2. k-anonymity gate rejects single-tenant fragment attempts (k<10 path)
3. L0 read returns aggregated cohort patterns (no per-tenant leakage)
4. Latency: dispatch wall-clock with L0 write ≈ dispatch wall-clock without (within 100ms)
5. Privacy audit log clean

## Out of scope

- L0 → L1 promotion (separate VT row)
- Cross-cohort L0 analytics (just write + read; no synthesis)
- Backfill from historical pipeline_steps (forward-only writes from this point)
