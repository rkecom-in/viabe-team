# VT-218 — Dev environment setup (Vercel + Railway + Supabase)

## Why this row exists

Fazal flagged (2026-05-28 16:18 IST) that we've been doing integration testing on developer-localhost-only, which can't actually exercise the real-world paths:
- Google Apps Script can't reach `localhost:8001` (VT-212 walk hit this directly)
- Shopify webhook subscriptions can't POST to `localhost`
- Supabase Realtime is per-project, configured separately for dev vs prod
- Resend / Telegram production-bot dispatches need a real environment to validate

Pattern Fazal already proved on Viabe-reports: Vercel + Railway Pro for both apps + two separate Supabase projects (dev/prod). End-to-end testing on dev env with public URLs, then promote to prod for launch.

This row stands up that substrate for Viabe Team **before** continuing VT-212 / VT-213 / any future manual walks.

## Scope

### A. Orchestrator deploy (Railway)

1. **`apps/team-orchestrator/Dockerfile`** — multi-stage Python 3.13 build:
   - Base: `python:3.13-slim`
   - Install `uv` + sync project deps
   - Copy app source
   - Entry: `uvicorn` or whatever the existing main.py launches on `0.0.0.0:$PORT` (Railway sets `PORT` env)
2. **`apps/team-orchestrator/railway.toml`** — service config:
   - build.buildCommand
   - deploy.startCommand
   - deploy.healthcheckPath = `/health`
   - deploy.healthcheckTimeout
3. **Migrations strategy** — `scripts/apply_migrations.py` runs as a Railway pre-deploy step OR as part of `startCommand`. Idempotent + safe to re-run (per the existing migration runner contract).
4. **DBOS schedulers** — already wired; will boot on container start.

### B. team-web deploy (Vercel)

5. **`apps/team-web/vercel.json`** — minimal config:
   - `buildCommand`: `pnpm build`
   - `framework`: `nextjs`
   - `installCommand`: `pnpm install --frozen-lockfile`
6. **Vercel project linkage** — Fazal creates project + links GitHub repo + sets root directory to `apps/team-web` (since this is a monorepo).
7. **No path-based routing changes for dev** — Vercel-provided domain (e.g. `viabe-team-<hash>.vercel.app`) serves at root; team-web's existing route shape (`/team/onboard`, `/team/ops/*`) means URLs are `https://<vercel-domain>/team/onboard` etc. ✓

### C. CI/CD wiring

8. **`.github/workflows/deploy-dev.yml`** — triggered on push to `main`:
   - Job 1: deploy orchestrator to Railway (Railway CLI or `bervProject/railway-deploy@v2` action)
   - Job 2: deploy team-web to Vercel (`amondnet/vercel-action@v25` or Vercel CLI)
   - Both jobs depend on a `pre-deploy-checks` job: `pnpm typecheck` (web) + orchestrator lint/test summary green
   - Concurrency: only one deploy per branch at a time (`concurrency.group: deploy-dev`)
   - Failure mode: deploy fails → workflow fails → Cowork notified via existing channels
9. **Required GitHub secrets** (Fazal sets in repo Settings → Secrets):
   - `RAILWAY_TOKEN`
   - `RAILWAY_SERVICE_ID` (orchestrator service)
   - `VERCEL_TOKEN`
   - `VERCEL_ORG_ID`
   - `VERCEL_PROJECT_ID`

### D. Env-var migration (file-based → dashboard-based)

10. **Stop loading `.viabe/secrets/*.env` files at runtime in deployed environments.** Local dev still uses them; Railway + Vercel use their respective dashboard env-var stores.
11. **Catalog all env vars currently in `.viabe/secrets/*.env`** + provide CC's PR description with a copy-paste matrix:
    - Railway env vars: `DATABASE_URL`, `OPERATOR_JWT_SECRET`, `INTERNAL_API_SECRET`, `TEAM_PHONE_ENCRYPTION_KEY`, `TEAM_PHONE_HASH_SALT`, `SUPABASE_URL`, `SUPABASE_SECRET_KEY`, `FAZAL_OWNER_UUID`, `FAZAL_TENANT_ID`, `GOOGLE_OAUTH_*`, `SHOPIFY_*`, `TELEGRAM_*`, `RESEND_*`, `ANTHROPIC_API_KEY`, `TEAM_CANARY_TENANT_IDS` (empty for dev), `GOOGLE_OAUTH_REDIRECT_URI` (updated to Railway dev URL)
    - Vercel env vars: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`, `OPERATOR_JWT_SECRET`, `INTERNAL_API_SECRET`, `FAZAL_OWNER_UUID`, `FAZAL_TENANT_ID`, `TEAM_ORCHESTRATOR_URL` (= Railway dev URL)
12. **`.viabe/secrets/README.md`** — document the local-vs-deployed split. Local: file-based. Deployed: dashboard-based. Never commit `.env` to git (already gitignored — verify).

### E. Post-deploy updates

13. **Google OAuth redirect URI** — Fazal updates the GCP OAuth client's Authorized redirect URI to add the Railway dev URL pattern (e.g., `https://<railway-domain>/api/orchestrator/integrations/google/callback`). Keep the localhost one too for local dev.
14. **Shopify webhook subscriptions** — update existing subscriptions in the Shopify dev store to POST to the Railway dev URL. CC scripts this via Shopify Admin API in a one-off task post-deploy.
15. **Supabase Realtime** — confirm `pipeline_steps` table has Realtime enabled in the dev Supabase project. If not, enable via Supabase Studio → Database → Replication. (Fazal may have already done this for VT-201 PR-1; verify, don't assume.)

### F. Smoke test on deploy

16. **`scripts/dev-env-smoke.sh`** — runs against deployed dev URL:
    - `curl <RAILWAY_URL>/health` → 200
    - `curl <VERCEL_URL>/team/onboard` → 302 to /login (auth gate working)
    - Mint a test JWT via local script + cookie-set it → `curl <VERCEL_URL>/team/ops/stream` → 200
    - Optionally: trigger one synthetic pipeline_run via internal API → assert it appears in Supabase Realtime stream (validates the full real-time loop)
17. Document in `docs/clau/dev-env-runbook.md`.

### G. Runbook

18. **`docs/clau/dev-env-runbook.md`** — covers:
    - How to view deploy logs (Railway + Vercel)
    - How to rotate secrets in Railway/Vercel
    - How to apply a hot-fix (push to a branch → preview deploy on Vercel; Railway gets only main)
    - How to promote dev → prod (filed as separate row VT-XXX when prod env setup happens)
    - Rollback: Vercel rollback button + Railway rollback CLI
    - Common failure modes + their fixes

## Acceptance criteria

1. Push to `main` → both Railway + Vercel auto-deploy within 5 min, both go green
2. `curl <RAILWAY_URL>/health` returns 200 from anywhere on the internet
3. `curl <VERCEL_URL>/team/onboard` returns 302 to /login (auth gate active)
4. Operator JWT minted locally + cookie-set → can hit `/team/ops/stream` on dev URL successfully
5. DBOS scheduled workflows (ingestion + alerts + purge) run in deployed orchestrator (visible in Railway logs)
6. Supabase Realtime `pipeline_steps` events propagate to dev team-web's stream view
7. `dev-env-smoke.sh` script passes end-to-end
8. Runbook in `docs/clau/dev-env-runbook.md` covers deploy, rollback, secrets rotation
9. No secrets in committed files; `.gitignore` covers `.viabe/secrets/`; verified by `git ls-files | grep secrets/` returning empty

## Rule #15 Canary (mandatory)

`scripts/dev-env-smoke.sh` IS the canary — it runs against the deployed dev URL, exercises the full HTTP + auth + Realtime path, fails-not-skips on any HTTP non-2xx. Doc-test: a tee of the run output appended to `.viabe/queue/VT-218/task_log.md` as proof.
