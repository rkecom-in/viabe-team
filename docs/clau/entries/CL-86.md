# CL-86 — mypy --strict as non-blocking CI job, re-evaluate gating at end of Sprint 1

DECISION (Fazal 2026-05-18 in response to audit CL-77 recommendation 6): add mypy --strict as a NON-BLOCKING CI job that reports errors but doesn't fail the build. After observing the error surface (expected 50-200 issues on first run), triage and decide whether to gate.

FIRST PR scope:
- New CI job 'mypy' in .github/workflows/ci.yml. Runs `mypy --strict apps/team-orchestrator/src apps/team-ingestion-worker/src` plus the packages/team-shared/python tree.
- continue-on-error: true (or equivalent) so the job reports red without blocking branch protection.
- Job output uploaded as a CI artifact for easy review.
- Add `mypy>=1.14` to apps/team-orchestrator/pyproject.toml dev dependencies and apps/team-ingestion-worker/pyproject.toml if missing.

RE-EVALUATION TRIGGER: end of Sprint 1, OR if the strict-mode surface is small enough to gate without major refactoring (likely <50 errors in current Python code given how small it is, ~1500 lines).

Criteria for gating (move to required check):
- Strict-mode error count is stable and low (<20 outstanding).
- Common categories triaged: type stubs missing (install), real bugs (fix), genuine third-party untyped-ness (configure ignore_missing_imports per package), legitimate Any propagation (`type: ignore[<rule>]` with comment).
- No more than ~3 days of cleanup needed to get to zero.

If not gated by end of Sprint 1, log the decision (kept non-blocking or removed entirely).

Notion VT-24 / VT-25 / VT-26 / VT-31 acceptance criteria stated 'mypy --strict passes'. This decision modifies that to 'mypy --strict reports', satisfying the spirit (type discipline) without the cost of blocking PRs on a noisy strict-mode baseline. Update those Notion bodies inline with the retroactive Status='Done' sweep.
