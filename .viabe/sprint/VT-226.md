# VT-226 — webhook_metrics async DBOS workflow

## Why

VT-81 LOCK 1 (Cowork plan-review) raised latency budget concerns about Postgres-backed `webhook_metrics` writes in the Twilio inbound webhook route. CC took option (a) — ship sync, file follow-up. This is the follow-up.

Webhook latency to Twilio matters: 15s response timeout before retry; current inline metrics writes add ~30-80ms per request on top of replay-check + rate-limit-check + actual inbound persistence. Sum can creep into 300-500ms.

## Brief contract

Move `webhook_metrics` table writes from SYNC inline to ASYNC via DBOS workflow. Replay-check + rate-limit-check stay sync (load-bearing for security). After 200 returned to Twilio, fire-and-forget DBOS workflow inserts the metric row.

Substrate:
- `apps/team-orchestrator/src/orchestrator/observability/webhook_metrics_writer.py` (new) — `@DBOS.workflow()` decorated
- team-web webhook route at `apps/team-web/app/api/webhook/twilio/inbound/route.ts` removes inline metrics write; instead enqueues the workflow via an internal HTTP call to orchestrator
- Canary `apps/team-orchestrator/canaries/vt226_webhook_metrics_async.py` measures end-to-end latency < 200ms on the happy path

## Acceptance

- DBOS workflow registered + scheduled-trigger-pattern verified
- 200ms latency budget canary passes
- webhook_metrics rows still land (within 5s of actual webhook event)

## Out of scope

- Webhook handler refactor beyond metrics writes (VT-81 substrate intact)
- Mock-mode for webhook_metrics (orthogonal)
