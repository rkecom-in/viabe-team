## Root cause (live, run #3)

`push_next_question_after_discovery` queued the proactive card push at discovery completion
(21:03:41). The journey then presented the same card itself on the owner's turn; the queued
item went stale but nothing invalidated it, so the */10 owner-comms sweep delivered it at
21:20:11 — 10 minutes AFTER the journey completed. (Run #2 didn't show it only because CC
dropped the duplicate manually.)

## Fix

Drain-time staleness guard in `owner_comms_drainer.drain_one` — the single reliable choke:
a `journey_push` item is DROPPED (reason='stale_journey_push', never delivered) when the
journey is no longer active OR the pushed field is already answered. The push payload now
carries `field` (whatsapp_journey) so the answered-check is precise. Guard fails OPEN (a
journey-read error never blocks a legitimate drain).

## Evidence

4 pins in test_owner_comms_drainer.py (complete→dropped, answered→dropped, pending→delivers,
guard-error→fail-open). Full owner_surface + onboarding sweep green (1054, fresh DB).
