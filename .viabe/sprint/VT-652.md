## Finding (Tier-1: ignored_speech_act, 3/3 runs on dev)
Owner: "Can you set up a win-back offer for my customers who've gone quiet?" → assistant: "You currently
have 10 customers in your ledger." The owner asked to SET UP A WIN-BACK (an ACTION → route to Sales
Recovery); the deterministic count net answered a total customer count instead, never advancing the action.

## Root cause (confirmed locally, pure `classify_status_query`)
The count/status nets fire on the presence of a customer/dormancy NOUN, NOT on the message actually being a
count QUESTION. So ACTION requests that mention customers get swallowed. Confirmed class (3 phrasings):
- "Can you set up a win-back offer for my customers who've gone quiet?" → `customer_count`  (WRONG)
- "set up a win-back for my lapsed customers"                          → `lapsed_count`    (WRONG)
- "run a campaign for my dormant customers"                            → `lapsed_count`    (WRONG)
Correct counters (must still work): "how many customers do I have" → customer_count; "how many customers
haven't ordered in a while?" → lapsed_count (F2).

## NOT VT-648
The enforce send-intent gate touches ONLY send-approval resolution (a later turn). F2's lapsed_count needs a
count cue (absent here) and would answer 8, not the 10 seen. This net over-fire pre-dates VT-648; the enforce
j06 validation merely SURFACED it. (Enforce itself is safe — j01 + j07 Tier-1=0 x3.)

## Fix direction (aligned with CL-2026-07-15-no-lists)
The deterministic count/status nets must fire ONLY on an unambiguous count/status QUESTION (a FINITE,
enumerable interrogative class — "how many / kitne / कितने / number of / count of" + customer noun) and DEFER
to the brain when the message is a REQUEST/ACTION (the brain routes "set up a win-back / run a campaign" to
Sales Recovery). Do NOT enumerate action verbs (infinite → LLM's job); instead REQUIRE the count-interrogative
for the net to short-circuit the brain — anything not matching the tight interrogative falls through to the
LLM. Delicate: this net (DF5, tasks #44/#77) was built to reliably answer counts; tightening it must not
regress the legit count-answer cases.
