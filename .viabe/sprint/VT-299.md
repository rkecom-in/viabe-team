# VT-299 — Customer-facing reasoning agent for WhatsApp inbound (post-v1)

## Why
VT-287 ships the inbound-first customer pipeline as **deterministic v1** (Pillar 1, zero
LLM): intro-once / opt-in(YES) / STOP, with a templated business-voice reply. Cowork
ruling 2026-06-02 deferred a real customer-facing reasoning agent (the brain that composes
genuine business-voice replies to arbitrary customer messages) to this row — the existing
supervisor brain is owner-scoped (owner `SubscriberState`), so a customer-facing brain is
its own context + guardrail surface, not a v1 concern.

## What
A customer-facing reasoning path (separate from the owner supervisor) that, for an
established (consented) customer conversation, composes a business-voice reply within the
24h window — per-customer context, tenant voice config, guardrails (no PII leakage,
no commitments the owner didn't authorize), cost-bounded.

## Acceptance criteria
- Replies only for consented customers (has_consent true) on a live WABA (wa_send_allowed).
- Per-customer context (not the owner SubscriberState).
- Guardrailed: no PII leak (CL-390/CL-426), no unauthorized commitments; escalate-to-owner
  path for out-of-scope asks.
- Cost-bounded (no unbounded LLM on every inbound).

## Out of scope
- The deterministic v1 path (VT-287 — intro/opt-in/STOP/templated reply).
- WABA onboarding (VT-286).

## Notes
Post-v1, low priority. Builds on VT-287's `wa_conversations` + `customer_inbound` path.
Deferred per Cowork ruling 2026-06-02 (VT-287 actor-model ruling).
