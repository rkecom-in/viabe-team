# VT-233 — Hot-fix Supabase magic-link implicit-flow fragment handling

## Why

Fazal manual walk 2026-05-29 17:55 IST against dev. Magic link delivered + clicked. Supabase Auth verified the OTP and issued a valid session JWT (decoded `sub=2fcac3cf-75b9-40bd-b857-5856b65d7c7d`, `email=fazal@viabe.ai`, valid 1h). But the redirect landed at:

```
https://viabe-team-dev.vercel.app/team/ops/login?error=missing_token#access_token=eyJ...
```

The session is in the URL fragment (after `#`). **Browsers never send URL fragments to servers.** Our server-side callback at `apps/team-web/app/api/ops/login/callback/route.ts` only reads `?code=` (PKCE flow) and `?token_hash=` (OTP flow) from query params. Both are absent → callback returns `error=missing_token` even though Supabase Auth succeeded.

This is Supabase's **implicit flow** behavior. It's the default for magic links from supabase-js v2 when neither PKCE nor server-side flow is explicitly configured. Our supabase-client.ts uses bare `createClient()` with no `flowType` option → implicit flow.

## Three options considered

**Option A — Switch supabase-client to PKCE flow (`flowType: 'pkce'`)**
- Requires the SAME client instance (with the same PKCE code verifier) at both signInWithOtp AND exchangeCodeForSession
- Verifier must be stored across requests (typically cookie via `@supabase/ssr` package)
- We don't have @supabase/ssr installed — requires net-new dependency + refactor of how clients are created
- Bigger change

**Option B — Custom Supabase email template bypassing Supabase verify endpoint**
- Template generates link directly to our callback with `?token_hash={{ .TokenHash }}` instead of going through Supabase /auth/v1/verify
- Our existing callback handles `?token_hash=` via `verifyOtp({token_hash, type})`
- Pros: zero code change
- Cons: requires Fazal-side Supabase dashboard email template edit; doesn't cleanly forward our `?next=` param; brittle

**Option C — Client-side fragment handler (recommended)**
- Add a new client component page that reads `window.location.hash` and POSTs the access_token to a new server endpoint
- Server endpoint validates the token via `supabase.auth.getUser(accessToken)` (calls Supabase Auth admin API which verifies signature)
- Mints operator JWT, sets HttpOnly cookie, returns redirect target
- Pros: no Supabase dashboard change required; works with default Supabase config; works for both implicit AND future PKCE flows
- Cons: requires new client component + new server endpoint

**Pick Option C.** Smallest surface change in our control; no Fazal-side dashboard work; doesn't break VT-230 + VT-232 substrate; doesn't introduce new deps.

## Brief contract

CC implements:

### 1. Update callback handler to forward fragment-flow URLs

`apps/team-web/app/api/ops/login/callback/route.ts` — when neither `?code=` nor `?token_hash=` present in QUERY:
- Current behavior: redirect to `/team/ops/login?error=missing_token`
- New behavior: redirect to `/team/ops/login/finalize?next={next-from-query-or-empty}`
- The fragment from the incoming request URL is preserved automatically by the browser across 302 redirects.

### 2. New client page at `app/(auth)/team/ops/login/finalize/page.tsx`

Client component (`'use client'`). On mount:
1. Read `window.location.hash` (substring after `#`)
2. Parse as URL-encoded form (e.g., `URLSearchParams(hash.slice(1))`)
3. Extract `access_token` (required) + `refresh_token` (optional, not used for our cookie)
4. If missing access_token → display "Login link invalid or expired" message
5. POST to `/api/ops/login/finalize-hash` with JSON `{ access_token, next: searchParams.next }`
6. On success → `window.location.replace(json.next || '/team/ops')`
7. On failure → display error message + link back to `/team/ops/login`

Minimal Tailwind UI matching the VT-232 login card aesthetic: centered card on `bg-gray-50`, white card with `shadow-md`, "Finalizing sign-in..." with a spinner during the POST.

### 3. New server endpoint at `app/api/ops/login/finalize-hash/route.ts`

```ts
export async function POST(req: Request) {
  const { access_token, next } = await req.json()

  if (typeof access_token !== 'string' || !access_token) {
    return NextResponse.json({ error: 'missing_token' }, { status: 400 })
  }

  // Use server-secret-key client to verify the access_token via Supabase Auth Admin API
  const supabase = serverSecretClient()
  const { data, error } = await supabase.auth.getUser(access_token)
  if (error || !data.user) {
    return NextResponse.json({ error: 'invalid_token' }, { status: 401 })
  }

  const userId = data.user.id
  const operatorAllowlist = (process.env.FAZAL_OWNER_UUID ?? '').trim()
  if (!operatorAllowlist || userId !== operatorAllowlist) {
    return NextResponse.json({ error: 'not_authorized' }, { status: 403 })
  }

  // Mint operator JWT (same as callback route)
  const opJwt = await issueOperatorJwt(userId)

  // Open-redirect allowlist (same as callback route — extract to shared helper if duplicated)
  const safeNext = computeSafeNext(next)

  const res = NextResponse.json({ next: safeNext })
  res.cookies.set('viabe_ops_jwt', opJwt, {
    httpOnly: true,
    secure: true,
    sameSite: 'lax',
    path: '/team',
    maxAge: 60 * 60,
  })
  return res
}
```

`computeSafeNext()` extracts the open-redirect allowlist check from the callback route into a shared helper at `apps/team-web/lib/auth/safe-next.ts` (DRY).

### 4. Refactor open-redirect allowlist into shared helper

`apps/team-web/lib/auth/safe-next.ts`:
```ts
const NEXT_ALLOWLIST = [
  '/team/ops',
  '/team/ops/stream',
  '/team/ops/stream/history',
  '/team/onboard',
  '/team/dashboard',
]

export function computeSafeNext(nextParam: string | null | undefined): string {
  if (!nextParam) return '/team/ops'
  if (typeof nextParam !== 'string') return '/team/ops'
  if (!nextParam.startsWith('/team/')) return '/team/ops'
  if (nextParam.includes('//') || nextParam.includes('..')) return '/team/ops'
  const ok = NEXT_ALLOWLIST.some(
    (p) => nextParam === p || nextParam.startsWith(`${p}/`) || nextParam.startsWith(`${p}?`),
  )
  return ok ? nextParam : '/team/ops'
}
```

Update both `app/api/ops/login/callback/route.ts` AND new `app/api/ops/login/finalize-hash/route.ts` to import + use this helper. Removes duplication.
