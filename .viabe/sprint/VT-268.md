# VT-268 — enforce the business_profile policy directives (not just context)

VT-195 puts the owner_persona into the agent's CONTEXT (L1 pre-inject). But
L1-as-context ≠ ENFORCEMENT — a model CAN ignore a system instruction. RKeCom's
persona encodes HARD policy that must be enforced structurally:
- **"No discount without my confirmation"** → any discount/price-concession
  action MUST route through `request_owner_approval` (VT-47), not be agent-
  autonomous.
- **"Never update the accounts book"** → a HARD tool boundary: NO write path to
  the accounts book (the google_sheets primary_customer_ledger) from the agent.
  Enforce at the tool layer (no such tool / explicit refusal), not via prompt.

Launch-relevant SAFETY (the moat directive becomes a liability if the agent can
violate the owner's hard rules). Fazal to prioritize. Backlog — do NOT build yet.
Depends on VT-195 (the profile substrate) + interfaces with VT-47 (approval).

## Implementation (PR-D, 2026-06-03) — prove-and-LOCK the boundary

Ground-truth reconcile (Rule #14) corrected the mechanism: the 1:1 send tools
(`send_whatsapp_message`/`template`) are NOT on any agent tool surface and
`send_whatsapp_message` has no production caller; `send_whatsapp_template` is called only by
`execute_approved_campaign` (already campaign_send-approval-gated). Campaign plans carry no
discount field. So there is no live un-gated 1:1 concession to close — VT-268's enforceable
deliverable is to LOCK the boundary closed:

- `agent/tool_guardrail.py` — fail-CLOSED `assert_agent_tools_safe`, wired at BOTH agent builds
  (orchestrator_agent.py + integration_agent.py). RAISES at graph build if any tool exposes a
  send-to-customer / accounts-book(Sheet)-write / ledger-write capability. The agent literally
  cannot be handed such a tool without the build failing.
- `tests/agent/test_no_write_tool_surface.py` — pins the exact agent tool allowlist (any
  addition → CI review), asserts the dangerous standalone fns are not agent tools, proves the
  guard trips on synthetic send/Sheet/ledger tools + passes the real surfaces + that
  `build_orchestrator_agent` rejects a send tool.
- `tests/.../connectors/test_sheets_readonly_lock.py` — locks the Google Sheet "accounts book"
  read-only: scope = `spreadsheets.readonly` only, no Sheet-write method, no Sheets write API call.
- Campaign approval-gate coverage (every proposed campaign → approval) already locked by
  test_collapse/test_routing/approval-integration — referenced, not duplicated.

DEFERRED (not built speculatively): concession DETECTION (owner-set discount terms) needs the
owner-terms UX (VT-195 ph3); no campaign discount field invented. The lock keeps the boundary
safe until that lands.

Cowork ruling 2026-06-03: "accounts book" = the owner's Google Sheet (readonly-locked); internal
Postgres `customer_ledger_entries` is a separate OAuth-authorized ingestion surface, out of scope.
