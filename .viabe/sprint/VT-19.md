## What CoderC must produce
The canonical 8-table base schema for the Team product, defined in versioned SQL migrations. Each migration is forward-only and idempotent.
**The 8 tables:**
1. **`tenants`** — PK `id` (UUID). The root of multi-tenancy. Stores subscriber-level identity: business name, owner name, WhatsApp number, business type, sign-up timestamp, current phase (`onboarding | trial | trial_extended | paid_active | paid_at_risk | cancelled | refunded`), plan tier (`founding | standard | pro`), Razorpay customer ID, KYC status. NO `tenant_id` column — this IS the tenant table.
2. **`customers`** — PK `id` (UUID), `tenant_id` FK NOT NULL. The dormant-customer ledger. Phone number, name, last-transaction date, total-spend cents, segments (JSONB), `acquired_via` enum (paper_book/contacts/upi_*/kot_pos/cash_book/qr_opt_in/apify_*/owner_typed), opt-out status, opt-out timestamp.
3. **`transactions`** — PK `id` (UUID), `tenant_id` FK NOT NULL, `customer_id` FK NULL (some sources have transaction without identified customer). Amount in paise, timestamp, source (`upi_phonepe | upi_gpay | upi_paytm | kot_pos | cash_book | owner_typed | other`), confidence 0–1, raw payload (JSONB).
4. **`campaigns`** — PK `id` (UUID), `tenant_id` FK NOT NULL. Generated `CampaignPlan` JSON (from VT-SR-Agent), status (`proposed | approved | sent | failed | cancelled`), proposed_at, approved_at, sent_at, attribution_close_at, target customer cohort IDs (JSONB array of customer FKs).
5. **`campaign_messages`** — PK `id` (UUID), `campaign_id` FK NOT NULL, `tenant_id` FK NOT NULL (denormalized for RLS). Customer FK, template ID, parameters (JSONB), Twilio message SID, send_status (`queued | sent | delivered | read | failed`), send timestamps, error code if failed.
6. **`attributions`** — PK `id` (UUID), `campaign_id` FK, `tenant_id` FK NOT NULL, `customer_id` FK, `transaction_id` FK NULL. Attribution timestamp, attribution confidence 0–1, ARRR contribution in paise, attribution method (`exact_match | window_match | manual_owner`).
7. **`pipeline_log`** — PK `id` (UUID), `tenant_id` FK NULL (some events are cross-tenant orchestrator events; null tenant means workspace-level). `run_id` UUID (groups events into a single run), `event_type` (e.g. `state_transition`, `tool_call`, `agent_invocation`, `error`), `payload` JSONB, `latency_ms` int, `created_at` timestamp. Append-only enforced by RLS (no UPDATE policy). Used by VT-Observability.
8. **`signup_holding`** — PK `id` (UUID), no `tenant_id` (pre-tenant table for landing-page form submissions). Captures the 4 form fields (business name, owner name, WhatsApp, business type) plus submission timestamp + IP for rate-limiting. Promoted to a `tenants` row when sign-up flow completes (VT-9.2).
**Migration files** under `packages/team-shared/migrations/`:
- `001_create_tenants.sql`
- `002_create_customers.sql`
- `003_create_transactions.sql`
- `004_create_campaigns.sql`
- `005_create_campaign_messages.sql`
- `006_create_attributions.sql`
- `007_create_pipeline_log.sql`
- `008_create_signup_holding.sql`
Each migration is a single SQL file that can be re-run safely (uses `CREATE TABLE IF NOT EXISTS` with explicit column definitions and constraints). Each includes appropriate indexes (`tenant_id` on all multi-tenant tables, `customer_id` on customers/transactions/attributions, `campaign_id` on campaign_messages/attributions, `run_id` on pipeline_log, `created_at` on all).

## Architectural rules to enforce in this subtask
- Pillar 3 (tenant isolation, structural): every table that holds tenant-scoped data has `tenant_id UUID NOT NULL REFERENCES tenants(id)`. The two exceptions (`tenants` itself, `signup_holding`) are explicitly documented in the migration comments. RLS policies (next subtask) will enforce read/write isolation; this subtask must not assume RLS is in place yet but must not block its addition either.
- Pillar 8 (no patchwork): no `ALTER TABLE` in subsequent migrations to retrofit `tenant_id` later. Every multi-tenant table is born with `tenant_id NOT NULL` from migration 1.
- Foreign keys are declared with `ON DELETE` behavior explicit (`CASCADE` for child rows that must die with the parent, `SET NULL` for soft references like `signup_holding→tenants`, `RESTRICT` where deletion should fail loudly).
- `pipeline_log` is append-only by design. The `created_at` column is set via `DEFAULT now()` and not updateable. UPDATE/DELETE on this table will be blocked by RLS in VT-2.4.
- All money is in paise (integer), not rupees (float). Currency conversions are forbidden in the schema.
- Migration files are idempotent at table-creation level (`IF NOT EXISTS`) but NOT at data level. They are forward-only; rollback is via Supabase PITR or fresh dev rebuild.
- `created_at TIMESTAMPTZ NOT NULL DEFAULT now()` on every table. `updated_at TIMESTAMPTZ` where mutable; backed by a trigger that auto-updates on row UPDATE (helper trigger lives in migration 001 and is reused).

## Acceptance criteria
- 8 migration files exist in `packages/team-shared/migrations/` numbered 001–008.
- Migrations apply cleanly to a fresh `viabe-team-dev` schema in order; final state matches the table list above.
- Every multi-tenant table has `tenant_id UUID NOT NULL REFERENCES tenants(id)`. `tenants` and `signup_holding` documented as exceptions.
- All money columns are integer paise; CI grep confirms no `money`, `numeric`, or `decimal` types on amount columns.
- Indexes present on every FK column and on `created_at` on every table.
- `pipeline_log.created_at` defaults via `DEFAULT now()`; column is NOT NULL.
- Migration runner output (or `supabase db push` log) shows zero errors.
- Schema dump committed at `docs/team/schema-snapshots/sprint-1.sql` for future-reference (drift detection).

## SUPERSEDED 2026-05-18 — see Clau_Session_Log CL-78
The 'ship all 8 canonical tables in one go' framing of this subtask was superseded on 2026-05-18 by a deliberate schema execution-order pivot: ship migrations in dependency order of the subtasks that need them, not all-canonical-8 upfront. Status is set to 'Deferred' to reflect this.
What actually shipped (under VT-Foundation parent and the orchestrator subtasks VT-24/25/26/31): the tenants table plus 12 orchestrator-path tables (phase_transitions, subscriptions, razorpay_webhook_events, pipeline_runs, pipeline_steps, phone_token_resolutions, privacy_audit_log, env_config, dsr_tickets, twilio_inbound_events, rate_limit_buckets, plus the DBOS-managed tables that auto-create on first launch).
Deferred tables and their gating subtasks:
- signup_holding → VT-9.2 (sign-up completion flow)
- customers, transactions → VT-Ingestion subtasks (writer landing first)
- campaigns, campaign_messages → VT-SR-Agent + VT-OwnerSurface
- attributions → VT-Attribution (T+7 close logic)
- pipeline_log was SUPERSEDED IN-PLACE by the pipeline_runs + pipeline_steps split (more sophisticated than the original 'one log table'; no retrofit needed)
Rationale: Phase 1 soft launch (July 15) needs the orchestrator path operational, not the full domain schema. Shipping unused tables creates maintenance debt and slows the migration runner. Domain tables get their schema scrutinized in the context of the code that uses them, which is better review hygiene.
The original spec below is preserved for reference but is no longer the acceptance contract for this subtask. Each deferred table's acceptance contract moves to the consuming subtask's body.
