## What CoderC must produce
The Method 7 ingestion adapter. Triggered on demand (e.g., when a tenant onboards or weekly via scheduled trigger). Calls Apify's pre-built actors for Zomato / Swiggy / Magicpin to scrape the tenant's listing details — ratings, review counts, average prices, popular dishes, competitor density in the same locality. Writes to L1 KG. NEVER captures customer reviews verbatim or any PII.
1. **Adapter module** at `apps/team/src/ingestion/methods/apify_food_platforms.py`:
	- `ingest_food_platforms(tenant_id, platform: Literal['zomato', 'swiggy', 'magicpin'], run_id) -> IngestionSummary`.
	- Resolves the tenant's listing URL on each platform (captured during sign-up if available; otherwise skipped with a structured warning).
	- Calls Apify actor with the listing URL.
2. **Apify integration**:
	- Actor IDs configured in `apps/team/config/apify_actors.yaml`.
	- APIFY_TOKEN env var (set in VT-2.5).
	- Sync run with 60-second timeout; longer scrapes → async run with webhook callback (deferred to Phase 1.5 if perf becomes an issue).
3. **Data extracted (CONTEXT ONLY, NO PII)**:
	- Tenant's own listing: rating, review count, average price, popular dishes (if present), photos count, opening hours displayed on platform.
	- Local competitor density: number of similar businesses (same cuisine type) within 1km radius (platform-derived).
	- Aggregate review sentiment: count of reviews (NOT review text), average rating, distribution by star count.
4. **What is NEVER extracted**:
	- Individual review texts.
	- Reviewer names.
	- Reviewer profile photos.
	- Reviewer demographics.
	- Any data that could identify a specific customer.
5. **L1 KG write**:
	- Per-tenant subgraph nodes: `tenant -[has_listing]-> platform_listing { rating, review_count, avg_price_paise, ... }`.
	- Aggregate data: `platform_listing -[has_competitor_density]-> { count, radius_meters: 1000 }`.
	- Last-updated timestamp; cached 7 days (re-scrape only if >7 days stale).
6. **Telemetry**: `pipeline_log` events with run_id, platform, scrape duration, fields extracted; payload contains aggregate counts only — no review text, no reviewer info.
7. **Tests** at `apps/team/tests/ingestion/methods/test_apify_food_platforms.py`:
	- Synthetic Apify response → correct L1 KG writes.
	- PII absence: synthetic Apify response that includes review text + reviewer names → those fields are STRIPPED before any storage. Test verifies no review text in DB or logs.
	- Listing URL absent: tenant without sign-up listing URL → adapter returns warning + skips.
	- Apify timeout: 60s exceeded → structured error.
	- Cache: re-running within 7 days → returns cached, no new scrape.
	- Cross-tenant: tenant_A's scrape data not readable by tenant_B (RLS on KG nodes).
	- 7-day re-scrape: synthetic time advance triggers fresh scrape.

## Architectural rules to enforce in this subtask
- Pillar 7: NO PII extraction. CodeX MUST reject PRs that store individual review text, reviewer names, or any identifiable customer data. Test coverage proves stripping works.
- This method writes to L1 KG, not to `customers` table. Customers table is unaffected by this method.
- Aggregate data only. Counts, ratings, distributions — not contents.
- Apify cost: each scrape is ~₹1-3. Cache 7 days. Per-tenant per-platform cap of 4 scrapes per week (rate limit).
- The agent uses this data to understand competitive positioning, NOT to mention specific reviewer names or reviews in campaign messages.
