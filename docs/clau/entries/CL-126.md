# CL-126 — fix-id grammar allows pseudo-parents VT-CI and VT-Foundation

Convention locked while resolving CL-94 (verbally 'CL-91') regex fix.

VALID PARENT FORMS in fix-id grammar `VT-<parent>-fix-<n>`:
1. REAL SUBTASK ID — the fix targets a single subtask in ViabeTeam_Sprint. Use the subtask ID directly. Examples: VT-3.3a-fix-1, VT-3.3b-fix-1, VT-3.2-fix-1, VT-3.3a-fix-2, VT-3.3a-fix-3.
2. PSEUDO-PARENT — the fix targets infrastructure, build, or schema that doesn't map cleanly to one subtask. Use one of these canonical pseudo-parents:
   - VT-CI — changes to .github/workflows/, CI configuration, gates, lint rules. Example: VT-CI-fix-1 (PR #14 CI hardening), VT-CI-fix-2 (this PR, pr-title regex).
   - VT-Foundation — changes to base schema, migrations not tied to a specific subtask, foundational helpers. Example: VT-Foundation-fix-1 (PR #16 migration 014).

NO OTHER PSEUDO-PARENTS without a logged Decision entry first. Adding 'VT-Docs' or 'VT-DX' silently dilutes the convention.

WHY THIS IS A REAL CONVENTION, NOT A WORKAROUND: cross-cutting infra changes genuinely don't have a single VT subtask home. Forcing them under an arbitrary subtask ID (e.g. 'pretend this CI change is part of VT-3.3a') would be worse — the git log would lie about scope. Pseudo-parents make scope honest.

DOCUMENTATION SURFACE: the regex comment block in .github/workflows/ci.yml. The grammar regex accepts these forms; the comment documents which parents are valid.

FUTURE TEST: if a fix-PR doesn't fit any real subtask OR canonical pseudo-parent, that's a signal the work is broader than 'fix' — it's probably a new VT subtask or a parent rewrite. Don't invent a new pseudo-parent on the fly.
