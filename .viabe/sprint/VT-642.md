## Problem (grounded in the deployed-dev j08 transcript, all 3 runs)

j08 turn-2, owner (Devanagari): "हां, वो 6 वाली लिस्ट भेज दो मुझे। और … वापस-लाने वाला ऑफर भी तैयार कर लो
— लेकिन अभी किसी को भेजना मत" = **"yes, SEND me that list of 6, AND prepare a win-back offer, but don't
send yet."** Two speech-acts: (a) send-me-the-list, (b) prepare-the-draft.

The D3 campaign net (`is_campaign_plan_imperative`) fires on the draft imperative → mints the SR dispatch
→ `skip_legacy_dispatch=True`. The async SR draft answers (b) — "I've drafted a campaign for 6…" — but
(a) the list-send request **vanishes silently**. That dropped speech-act is the Tier-1 breaker
(#4 ignored_speech_act) the blind judge flags on j08 3/3 (distinct from the VT-641 mis-route, now fixed).

Root cause: the D3 net owns the turn but only ever emits the campaign draft; a co-present list-send ask
has no path to the owner. We genuinely can't attach the individual customer names in chat yet — the
file-attachment path is **CD2 / VT-79 (still pending)** — so the honest move is to ACKNOWLEDGE the
list-send request rather than drop it (never a name dump; never a false "here they are").

## Fix (built — deterministic, single seam, money-gate untouched)

`onboarding/campaign_first_contact.py`:
- `mentions_customer_list_request(text)` — a co-present list/names cue detector (EN token set
  {list, lists, names, naam…} + Devanagari `keyword_match` pats {लिस्ट, सूची, नाम…}). Applied ONLY after
  `is_campaign_plan_imperative` already matched. FAIL-OPEN → False.
- `LIST_SEND_ACK_PREAMBLE` — honest interim copy: can't attach the individual names in chat **just yet**,
  confirms the cohort is in hand, bridges to the draft the owner also asked for, re-affirms the money gate
  ("nothing goes out until you say so"). Never a completion claim.

`manager/triage_seam.py` D3 real-cohort branch: when `mentions_customer_list_request(message_text)`, set
`direct_reply_text=LIST_SEND_ACK_PREAMBLE` on the returned `TriageSeamResult`. The runner emits it via the
replay-safe checkpointed `_send_owner_reply_step` (@DBOS.step, at-most-once) BEFORE dispatch; the async SR
draft still emits independently. So the owner gets **(1) the honest list-ack, then (2) the campaign draft**.

Why this is low-risk: it changes a REPLY STRING only — no plan/collapse/money-gate change. The send still
routes through the approval/consent/opt-out rails (`expect_sent_count=0` must still hold). The list-cue path
is strictly better than the pre-fix D1 "I'm on it" fallback (more honest + specific + addresses the drop);
the NON-list-cue path (`direct_reply_text=None`) is byte-identical to pre-VT-642.
