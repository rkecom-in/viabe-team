# VT-56 — Method 2: Phone contacts list import

## Parent task
VT-IntegrationInvention.

## Concept reference
- `concept-team.md` §5 (Method 2: phone contacts list import; structured input, no vision).
- `concept-team-pillars.md` Pillar 7 (consent: importing contacts means importing PII; the owner's act of sharing the file is consent for those phone numbers becoming customer rows).
- `concept-team-pillars.md` Pillar 6 (k-anonymity at L3 layer is unaffected here — this method writes to per-tenant L1/canonical tables, not L3).

## Dependencies
- VT-Foundation Done.
- VT-IntegrationInvention VT-6.3 Done (dedup module — contact list often overlaps with existing customers).

## What CoderC must produce
The Method 2 ingestion adapter. Owner uploads or pastes a phone contacts export (vCard or CSV format from common Android/iOS exports). Adapter parses, dedups, commits.
1. **Adapter module** at `apps/team/src/ingestion/methods/contacts.py`:
	- `ingest_contacts(tenant_id, file_bytes, file_format, run_id) -> IngestionSummary`.
	- `file_format: Literal['vcard', 'csv', 'auto']` (auto detects from content).
	- Parses input; for each contact: extract `name`, `phone` (E.164 normalized), `notes` (any "company" or "label" field). Discard email, address, photo — not relevant to customer ledger.
2. **Parsing**:
	- vCard: use `vobject` library; iterate vCard objects; extract FN (formatted name) + TEL fields.
	- CSV: detect column header row (case-insensitive match for "name" / "phone number" / "first name" / "last name" + "phone"). If columns unclear, return structured error asking owner to adjust the export.
	- Both formats: skip entries without a phone number (a contact without phone is useless for our purposes); skip entries flagged as work numbers if a personal number exists for the same name.
3. **Confidence**:
	- Contacts have HIGH structural confidence (it's structured data, not extracted from an image). All fields default to confidence 0.95 unless ambiguous (e.g., a phone number that doesn't match Indian format).
	- Non-Indian phone numbers (e.g., +1 US numbers) flagged with low confidence; owner asked whether to import.
4. **Bulk processing**:
	- Contacts files are typically large (200-2000 entries). Process in batches of 50.
	- For each batch: dedup (VT-6.3), commit. Yield progress via `pipeline_log` events.
	- Owner gets ONE summary message at end, not per-batch.
5. **Privacy disclosure**:
	- Before processing, send owner a confirmation: "You're about to import [N] contacts as potential customers. They'll receive WhatsApp messages from you only for re-engagement campaigns you approve. Reply YES to proceed." Template `contacts_import_confirmation` (VT-13.3).
	- Wait for confirmation via `request_owner_approval` (VT-5.9). Without explicit yes, drop the import.
6. **Source tagging**:
	- All committed customers get `acquired_via='contacts'`.
	- `last_visit_date` is null (no transaction history from contacts alone).
	- `total_spend_paise` is null.
	- Customers from contacts join the ledger; transactions accumulate as UPI exports / cash entries flow in.
7. **Tests** at `apps/team/tests/ingestion/methods/test_contacts.py`:
	- vCard parsing: synthetic .vcf with 5 contacts → 5 candidates.
	- CSV parsing: synthetic CSV with name + phone columns → candidates correct.
	- CSV ambiguous columns: returns structured error.
	- Dedup integration: 5 candidates, 2 match existing customers → 2 merged, 3 created.
	- Owner confirmation: import without approval → dropped; import with approval → processed.
	- Bulk: 500-contact synthetic file → batched processing, single summary message.
	- Non-Indian phone: flagged low-confidence; owner asked.
	- Cross-tenant: tenant_A's contacts cannot create tenant_B customers.

## Architectural rules to enforce in this subtask
- Pillar 7: explicit owner confirmation before bulk import. CodeX rejects PRs that skip the confirmation.
- Pillar 8: parsing libraries (vobject, csv) are standard, not hand-rolled.
- E.164 normalization via `phonenumbers` library (shared with VT-6.3 dedup).
- Batch size 50 is config-driven (`apps/team/config/contacts_batch_size.yaml`). Larger batches risk timeout; smaller batches risk too many DB roundtrips.
- Cross-tenant scoping enforced; CI test confirms.
- PII (phone numbers, names) never in `pipeline_log` payload — only counts.
- Email, address, photo from contacts are DISCARDED at parse time, not stored. CodeX confirms.

## Acceptance criteria

- Parsing tested with synthetic files of both formats.
- CSV ambiguous-columns case returns structured error.
- Owner confirmation flow integrated via VT-5.9.
- Bulk batching tested at 500 contacts.
- Dedup integration tested.
- Non-Indian phone case tested.
- Cross-tenant attack passes.
- PII absence in `pipeline_log` verified.
- mypy --strict passes.
- PR title references `(VT-IntegrationInvention)`; reviewed by CoderX (mandatory — bulk PII import is privacy-sensitive); Fazal personally reviews owner-confirmation wording.

## Out of scope
- Google Contacts API integration (Phase 1 is file upload/paste only; API integration is Phase 2).
- iCloud Contacts API integration (Phase 2).
- Contact photo / address fields — explicitly discarded.
- Smart-grouping ("family contacts", "work contacts") — owner can manually exclude before upload.
- Periodic contacts re-sync (Phase 1 is one-shot per upload).
- Reverse-lookup (phone → name from external service) — not relevant; we have the name from the contacts file.
