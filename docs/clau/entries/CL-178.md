# CL-178 — Skipped Context7 validation on create_agent

## What happened
While drafting VT-3.4 PR 1/3 brief, I explicitly noted I had not Context7-validated langchain.agents.create_agent's current signature for the stub agent's construction. I wrote: "trust Claude Code; if it hits a signature surprise, surface in PR description. The brief is already long; one more Context7 query is diminishing returns." This is exactly the rationalization CL-140 was written to prevent.

## What CL-140 says
Every brief Clau writes that references external libraries, frameworks, or APIs MUST be Context7-validated on the specific call surfaces before shipping. No exceptions for brief length, no exceptions for 'probably knows what it's doing.'

## What the validation surfaced (proving CL-140 right)
Two real bugs in the brief:

1. langchain.agents.create_agent uses system_prompt= NOT prompt=. The 'prompt' kwarg is the deprecated create_react_agent surface. Per LangChain v1 migration docs the parameter was renamed. Brief had prompt=_STUB_SYSTEM_PROMPT throughout — would have hit Claude Code as a TypeError on first run.

2. The name= kwarg does not exist in create_agent's documented signature. The brief had name="sales_recovery_agent" on the stub agent construction. Agent identification in the parent multi-agent graph comes from the StateGraph node name (graph.add_node("sales_recovery_agent", ...)) not from the agent instance itself. Brief had a phantom kwarg.

Both bugs were caught BEFORE the brief shipped because Fazal flagged the CL-140 violation and forced the validation. If Fazal had not pushed back, the brief would have shipped with these errors and Claude Code would have surfaced them at implementation time.

## What I'm changing in my behavior

CL-140 is now a hard pre-ship gate for every brief, not a 'nice to have when time permits.' Every external library symbol referenced in a brief gets Context7-validated before the brief is presented to Fazal. Brief length is not a valid reason to skip; if the brief is too long to validate every symbol, the brief is too long, period — split it.

Specific implementation: before any brief is presented, I will produce an inline checklist of every external library symbol used in the brief and tick each one against a Context7 result.

## Bonus catch from the validation pass

LangGraph Context7 query for Command(goto=..., graph=Command.PARENT) outside the langgraph_supervisor library context surfaced ZERO canonical examples. Our usage is genuinely off the documented golden path — even using langgraph.types.Command directly without create_supervisor. The primitive is public API, the shape is well-defined (Context7 showed exact example from supervisor library internals), but our integration pattern is novel in our codebase. PR 1/3 happy-path test is the first place this pattern executes. Most likely failure mode if test fails: routing not reaching specialist node. Already covered in brief's STOP conditions. Not a blocker, but worth flagging.
