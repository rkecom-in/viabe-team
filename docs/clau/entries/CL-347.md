# CL-347 — CORRECTION (tool-capability error, Clau-operational): For this session I assumed

CORRECTION (tool-capability error, Clau-operational): For this session I assumed notion-fetch could not read ViabeTeam_Sprint row properties (incl. the Task ID auto-increment), and repeatedly asked Fazal to supply VT IDs — causing avoidable round-trips on VT-144, VT-146, VT-148, VT-149. The assumption was wrong. notion-fetch on a page UUID returns the FULL property set including Task ID. The session-instruction note that 'notion-fetch returns only schema, not rows' applies to DATABASE VIEW URLs, not individual pages — I over-applied it without testing. Verified: notion-fetch on page 368387c2-cc5a-8128-a339-c2eb87a03f82 returned Task ID VT-148; on 368387c2-cc5a-8146-b2da-fd21f1d670a7 returned VT-149. STANDING METHOD going forward: to obtain a VT ID or any property value, notion-fetch the page by UUID — do NOT ask Fazal, do NOT guess, do NOT title a PR from an unverified number. notion-search returns titles/highlights only and is NOT sufficient for property reads. Page UUID remains the canonical reference in briefs and the log; VT-NNN is read via fetch when a PR title needs it. ROW IDs CONFIRMED: VT-148 = DBOS workflow_inputs body retention Step-0 (page ...8128a339...); VT-149 = webhook replay double-write UNIQUE constraint (page ...8146b2da...).
