# VT-320 — agent-action customer-marker L2 emit

Rostered from VT-312 D2 (Cowork 20260604T083000Z). VT-312 retired the fixed-threshold detectors + repurposed
`customer_dormant_threshold_crossed` / `customer_high_value_threshold_crossed` to **agent-action markers**: when
the brain ACTS on a specific customer (targets them in a recovery action), emit a customer-referencing episodic
row (`referenced_entity_type='customer'`, `referenced_entity_id=<customer_id>`) via the kg_events outbox →
L2 dual-projection.

**Why it matters:** this is the path that sets `referenced_entity_type='customer'` on episodic rows. Without
it, **VT-76's reconstitution sweep is a forever-no-op on real data** (nothing to anonymize). So this row makes
both the agent-action audit trail AND VT-76's opt-out reconstitution live.

## Scope
- Identify the agent-action site (the recovery-targeting path) where a specific customer is acted on.
- Emit the customer-referencing episodic event (templated summary, NO raw PII — CL-390; the customer_id is the
  referenced_entity_id, not a name/phone).
- Canary: an agent action on a synthetic customer produces a `referenced_entity_type='customer'` episodic row;
  VT-76 sweep then anonymizes it (the write→act→reconstitute loop).

Partly gate-live (real agent actions need the live dispatch path). Build the emit mechanism + synthetic canary now.
