# CL-224 — RESOLUTION (partial) to CL-209: viabe-team-dev pooler is a substrate sandbox; te

Resolution of CL-209: viabe-team-dev pooler postgres DB is a substrate-only sandbox. No test workflow targets it. Tests that depend on tenant tables (test_migrations.py, test_tenant_isolation.py) self-bootstrap via apply_migrations.apply(dsn=DATABASE_URL) in their module-scoped fixtures, then exercise real RLS via the app_role SET ROLE pattern (PR-fix-7 / CL-120). CI runs these in dedicated jobs against freshly-provisioned pgvector Postgres services. Skip cleanly when DATABASE_URL is unset. No silent failures, no mocked-out RLS paths, no coverage gap.

The dev pooler accumulated state from uncoordinated sources: (1) DBOS auto-init created dbos.transaction_outputs on launch, (2) some prior ad-hoc invocation created public.checkpoint_blobs / checkpoint_migrations / checkpoint_writes / checkpoints, (3) Landmine 2 added *landgraph_node_probe. None of this came from apply_migrations.py — schema_migrations does not exist in this DB.

Open sub-question pending diagnostic: how do the checkpoint tables exist in public without pipeline_runs? Current init_substrate creates the tables AND attaches RLS policies referencing pipeline_runs in the same call. If pipeline_runs missing, CREATE POLICY fails at parse time — we observed this directly. So either (a) tables were created by an earlier version of init_substrate without the RLS clause, or (b) tables were created and policy attachment failed silently somewhere. If (b), checkpoint tables in this DB may not have RLS attached — Pillar 3 concern. Diagnostic queued (pg_tables.rowsecurity + pg_policies on checkpoint).

Resolution path once diagnostic returns: if RLS attached → run apply_migrations against dev pooler, ship Step 11(a) verification, proceed to Step 12. If RLS not attached → drop checkpoint_* tables, run apply_migrations, run setup_checkpoint_tables.py end-to-end, then verification.
