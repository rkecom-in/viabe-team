## What CoderC must produce
The DSR (Data Subject Rights) APIs. Two endpoints: export-my-data (returns a structured archive of all data tied to a customer) and delete-my-data (irreversible erasure). Both verified via WhatsApp confirmation (the customer must reply YES from their registered phone before the action executes). Both auditable.
1. **API endpoints** at `apps/team/src/api/dsr.py`:
	- `POST /api/dsr/export` — initiates export request. Body: `{phone_e164, request_reason: optional}`. Returns `{request_id, status: 'pending_verification', verification_message_sent_to: <masked phone>}`.
	- `POST /api/dsr/delete` — initiates deletion request. Same shape; deletion is irreversible.
	- `GET /api/dsr/status?request_id=...` — returns status of a pending request.
	- `GET /api/dsr/download?request_id=...&token=...` — downloads the export archive (token is one-time, expires 7 days).
2. **Verification flow**:
	- On request initiation: send WhatsApp template `dsr_verification` to the requesting phone. Template includes the request type (export vs delete), request ID, and a one-time confirmation code (6 digits).
	- Customer replies with the code via WhatsApp. Webhook (VT-3.3) classifies as `dsr_verification_response` (12th classifier intent — PR follow-up to bump VT-49 prompt to v4).
	- Code matches → request transitions to `verified`; ineligible operation proceeds.
	- 24-hour verification window; expired requests close with `status='verification_expired'`.
3. **Export pipeline**:
	- Triggers when verification confirmed.
	- Aggregates ALL data tied to the customer across all tenants where they are a customer:
		- Per-tenant: customer profile, transactions, campaigns they were targeted by, attributions, episodic events referencing them, opt-out history, consent records.
		- Cross-tenant: nothing. L3 patterns are aggregates and don't reference the individual.
	- Output format: ZIP archive containing per-tenant folders (`tenant_<id>/`), each with JSON files (`profile.json`, `transactions.json`, etc.) plus a top-level `manifest.json` describing the archive structure + DPDPA disclosure.
	- Stored in a time-bounded S3 bucket (or Supabase Storage equivalent) with the one-time download token. Retention: 7 days. After retention, archive auto-deleted; subsequent download attempts return `request_expired`.
	- Telegram alert to Fazal on every export (low-volume signal; helps spot abuse patterns early).
4. **Deletion pipeline**:
	- Triggers when verification confirmed.
	- Per-tenant deletion (in order): pending approvals → pending clarifications → pending dedup resolutions → attributions → campaign_messages referencing customer → transactions → customer record. Each step is wrapped in a transaction; failure rolls back the whole deletion.
	- Episodic events (L2) handled like opt-out reconstitution: `referenced_entity_id` anonymized to sentinel UUID; events themselves preserved for audit.
	- L3: no per-customer data; nothing to do.
	- Final step: a `customer_deletion_audit` row inserted (migration 026) with the customer's `phone_e164_hash` (NOT plaintext), tenants affected, tables affected, deletion timestamp, deletion reason. This row is never deleted; it's the proof of compliance.
	- Customer receives a final WhatsApp message confirming deletion via VT-5.7 template `dsr_deletion_completed` (13th template).
	- Timeline: deletion completes within 30 days of verified request per DPDPA standard (Phase 1 typically completes within minutes; the 30-day window is a regulatory ceiling).
	- Telegram alert to Fazal on every deletion.
5. **Cross-tenant scope**:
	- DSR is per-customer (per phone). Affects all tenants where that phone is registered.
	- Customer cannot scope a DSR to a single tenant; DPDPA grants the data principal full rights, not partial.
6. **Tests** at `apps/team/tests/privacy/test_dsr.py`:
	- Export happy path: synthetic customer with data across 2 tenants → verification → archive generated with correct structure → download works → expires after 7 days.
	- Delete happy path: synthetic customer → verification → deletion across both tenants → audit row written → confirmation sent.
	- Verification expiry: 25-hour-old unverified request → closed.
	- Wrong code: 5 wrong attempts → request locked, alert to Fazal.
	- Cross-tenant: customer with 2 tenants both affected; partial failure rolls back both.
	- Download token: token used once is invalidated; reuse returns 404.
	- Audit row immutable: attempted DELETE on `customer_deletion_audit` row → RLS-blocked.
	- mypy --strict passes.

## Architectural rules to enforce in this subtask
- Pillar 7: DSR is mandatory and irrevocable once verified. CodeX rejects PRs adding override paths.
- Pillar 8: one DSR pipeline. Adding a new data type requires extending this module, not building a parallel deletion path.
- Verification via separate channel (WhatsApp) prevents API-only attacks. The phone holding the WhatsApp sender ID is the structural authenticator.
- Audit table is append-only. CodeX confirms via RLS DENY on UPDATE/DELETE for `customer_deletion_audit`.
- DPDPA-required disclosure included in `manifest.json`: what data we held, what was returned, retention period, contact info for grievances.
- Phase 1 doesn't expose a self-service web UI; DSR is API-driven with WhatsApp verification. Phase 2 may add a portal.
- 30-day SLA on deletion completion is regulatory; Phase 1 typical completion is minutes.
- Templates 11, 12, 13 added to VT-13.3 list (was 10 after VT-76). Updated count is 13.

## Out of scope
- Granular DSR (export only certain data types): Phase 2 if customers ask.
- Portal / self-service UI: Phase 2.
- Multi-language DSR confirmation copy: Phase 1 ships English + Hindi via the standard template path.
- Bulk DSR (one customer affecting many tenants in one click): out of scope; same per-customer DSR loops over their tenants.
- Right-to-correct (DPDPA): Phase 1 owners can edit their data via the portal; consumer correction is Phase 2.
- Right-to-data-portability beyond raw JSON archive: Phase 2 if needed.
