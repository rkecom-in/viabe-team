## What CoderC must produce
The soak run infrastructure: a synthetic-tenant generator + load driver + monitoring dashboard that runs the full Viabe Team pipeline against 50-100 synthetic tenants for 7 continuous days. Every regression class found is added to the edge-case suite (VT-13.2). Soak is a launch gate; production deployment requires a passing 7-day soak.
1. **Synthetic tenant generator** at `apps/team/tests/soak/synthetic_tenants.py`:
	- Generates 50-100 synthetic tenants spanning 5 business types × 5 city tiers × varying activity levels.
	- Each tenant gets a synthetic Twilio sender ID (sandbox), synthetic Razorpay subscription (test mode), realistic customer ledger (50-500 customers per tenant via Method 6 QR opt-in simulation).
	- Tenants live in the `viabe-team-soak` Supabase project — separate from dev + prod (pre-seeded ap-south-1 instance).
2. **Load driver** at `apps/team/tests/soak/load_driver.py`:
	- Schedule synthetic events into the running pipeline: weekly approval triggers, owner replies (varied: approve / reject / change / defer / hindi variants), edge-case interactions (exclusion / adhoc / status / template-error), opt-out events, DSR requests, refund-decision branches.
	- Targets realistic event mix: 70% normal weekly approvals + 20% edge cases + 10% adversarial (timeouts, malformed inputs, concurrent dispatches).
	- Generates ~500-1000 events/day across all tenants combined.
3. **Monitoring dashboard** at `apps/team/tests/soak/monitor.py`:
	- Reads pipeline_log + cost_dashboard + LangSmith data for the soak project.
	- Surfaces: pipeline error rate, agent termination rate, average tokens/run, average cost/run, queue depths, scheduled trigger lag, memory + CPU on Railway services.
	- Posts daily Telegram digest to Fazal at 9 AM IST.
4. **Failure detection + replay**:
	- Any pipeline_log event with severity=critical fires immediate Telegram alert.
	- Replay tooling (VT-12.4) lets Fazal investigate any specific run_id from the soak.
5. **Soak environment isolation**:
	- Soak Supabase project + Railway services + Twilio sandbox + Razorpay test mode all separate from prod.
	- Soak cannot affect production data; CodeX confirms via lint rule extending VT-8.1's cross-product isolation.
6. **Pass criteria** (documented at `docs/runbooks/soak-pass-criteria.md`):
	- 7 continuous days without operator intervention.
	- Pipeline completion rate ≥99.5% (99.5% of dispatches reach a final phase: completed, refunded, cancelled, or escalated; not stuck).
	- Zero TenantIsolationViolation, zero ContextIsolationViolation, zero PrivacyAuditChainBreak events.
	- Per-run avg cost ≤ ₹50 (within VT-12.3 ceiling).
	- p95 dispatch latency < 2 minutes.
	- Memory + CPU stable (no monotonic growth indicating leak).
	- All cost-runaway alerts investigated; no genuine runaway.
7. **Failure response**:
	- Soak failure (any pass-criterion miss) → analyze, fix, restart soak. Launch is gated on a passing 7-day soak.
	- Issues found get added to VT-13.2 edge-case suite to prevent regression.
## Architectural rules to enforce in this subtask
- Pillar 7: launch is gated on passing soak; no soft-skip.
- Pillar 8: one soak harness. CodeX rejects parallel test infrastructures.
- Soak environment isolation structural; CodeX confirms via cross-product lint extension.
- Pass criteria in runbook are Type 3 commitments; lowering requires board approval.
- 7-day continuous duration is non-negotiable; shorter soaks miss memory leaks + slow drift.
- Synthetic tenant data MUST resemble real tenant patterns; CodeX rejects PRs with unrealistic test data (e.g., all tenants identical).

## Out of scope
- Production load matching (Phase 1 soak is 50-100 tenants; production may be 100+ — this is acceptable Phase 1 risk).
- Chaos engineering / fault injection (Phase 2; Phase 1 soak is steady-state load).
- Multi-region soak (Phase 2).
- Continuous soak post-launch (Phase 2 if launch issues warrant).
- Adversarial / red-team soak (Phase 2).
- Customer-data soak (Phase 1 is synthetic only; never real customer data).
