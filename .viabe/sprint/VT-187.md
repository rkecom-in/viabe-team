# VT-187 — Schema normalization to design-doc §2.1

## Why this row exists

VT-178's STEP-0 audit surfaced significant drift between Clau's 2026-05-12 pipeline-observability design-doc §2.1 (per-field normalized columns) and the actual on-main schema (JSONB-driven envelopes). Inline drift documented in migration 005/006/007 docstrings; full enumeration in `.viabe/notifications/clau-briefing-2026-05-26-vt122-substrate-state.md` §3 Δ1.

Per Clau's review (2026-05-26) + Fazal's decision (CL-417): **α-sequencing — VT-187 schema normalization lands BEFORE VT-180 writer**, so VT-180 writes canonical columns from day one.

Without VT-187 before VT-180: every downstream consumer (decorator/agent-SDK-callback/LangGraph-hooks/Composer/Logfire-export/Day-39-evaluator/campaign-tracking) writes against an interim JSONB shape, then refactors when VT-187 lands. The double-cost is real and exceeds VT-187's own blast radius.

## Scope

### Migration 025 — add canonical per-field columns

For `pipeline_runs`, add:
- `trigger_kind TEXT` (back-fill from `run_type` where compatible)
- `trigger_source_ref TEXT` (back-fill from `trigger_payload->>'source_ref'` where present)
- `final_outcome TEXT` (back-fill from `terminal_state_metadata->>'final_outcome'`)
- `step_count INT NOT NULL DEFAULT 0` (back-fill via `(SELECT count(*) FROM pipeline_steps ps WHERE ps.run_id = pipeline_runs.id)`)
- `error_summary TEXT` (back-fill from `terminal_state_metadata->>'error_summary'`)
- Rename column path: keep `cost_paise` as the source of truth; design doc's `total_cost_paise` is the same field — add a view-side alias if needed for §2.1-compliance, OR rename the column atomically (preferred — cost_paise → total_cost_paise; touches 1 callers, low blast)

For `pipeline_steps`, add:
- `step_name TEXT` (nullable; populated by the writer when known)
- `parent_step_id UUID NULL REFERENCES pipeline_steps(id)` (for Agent SDK think→act→observe nesting)
- `tool_calls JSONB` (separate from input_envelope; lists tool calls within an agent step)
- `status TEXT` CHECK constraint `('running', 'completed', 'failed', 'skipped')` (back-fill from error_envelope presence: NULL error → 'completed'; non-NULL error → 'failed')
- `model_used TEXT` (nullable)
- `tokens_input INT` (nullable)
- `tokens_output INT` (nullable)
- Rename `step_index` → `step_seq` to match §2.1 (touches 1 internal call site + the canary file)
- Rename `rationale` → `decision_rationale` to match §2.1
- Rename `error_envelope` → `error` (or keep error_envelope + add a generated column `error` for §2.1-compliance — decision at plan time)

For `phone_token_resolutions`, add:
- `customer_id UUID NULL` — **NO FK at this row** per Cowork review Condition 1 (originally specified `REFERENCES tenants(id)` which was a brief-author typo: customer ≠ tenant). FK will be added in a future migration when the `customers` table ships (currently TBD; tracked as `TODO(VT-170)` in migration 025 comment).
- Rename `token` → `phone_token` to match §2.1
- Rename `last_resolved_at` → `last_accessed_at` to match §2.1
- Keep `phone_number_encrypted` (encryption-at-rest is a privacy improvement on §2.1's plaintext `phone_e164` — DOCUMENT this delta as an UPGRADE not a regression)
- Keep `resolved_count` (counter not in §2.1 but useful for forensics — DOCUMENT delta as additive)

### Back-fill — preserve all historical data

- Run back-fill SQL inside migration 025 transaction where data is recoverable from JSONB
- Where data is NOT recoverable (e.g., `step_name` was never written), leave NULL — VT-180 writer populates going forward
- Verify count parity post-migration (no row deletions)
