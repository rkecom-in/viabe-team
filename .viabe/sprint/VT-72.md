## What CoderC must produce
The typed wrapper layer that mediates ALL database access from the application. Every read and write goes through a wrapper that requires `tenant_id` as an explicit parameter. Direct DB access bypassing the wrapper is forbidden by lint rule. This is the second of the three independent isolation layers (RLS is first; agent context is third — VT-8.2).
1. **Wrapper module structure** at `apps/team/src/db/`:
	- `apps/team/src/db/wrappers/` — one wrapper class per canonical table. Each wrapper exposes typed methods like `find_by_id(tenant_id, customer_id)`, `list_for_tenant(tenant_id, filters)`, `insert(tenant_id, payload)`, `update(tenant_id, customer_id, patch)`, `delete(tenant_id, customer_id)`.
	- `apps/team/src/db/base.py` — abstract `TenantScopedTable` base class. Every concrete wrapper inherits. The base enforces:
		- Every public method takes `tenant_id: UUID` as the first parameter.
		- Every SQL query the wrapper generates includes `WHERE tenant_id = :tenant_id` in its predicate.
		- Result rows are validated: if any returned row's `tenant_id != input_tenant_id`, raise `TenantIsolationViolation` (a critical security event).
2. **Wrapper coverage** for Phase 1 tables:
	- `TenantsWrapper` (special: scoped by tenant's own ID via PK match; reads/writes always filtered to that one tenant)
	- `CustomersWrapper`
	- `TransactionsWrapper`
	- `CampaignsWrapper`
	- `CampaignMessagesWrapper`
	- `AttributionsWrapper`
	- `EpisodicEventsWrapper`
	- `PendingApprovalsWrapper`
	- `PendingClarificationsWrapper`
	- `PendingDedupResolutionWrapper`
	- `ScheduledTriggerRunsWrapper`
	- `ScheduledFollowupsWrapper`
	- `SendIdempotencyKeysWrapper`
	- `KGEventsProcessedWrapper`
	- `CompositionAuditsWrapper`
	- **Cross-tenant tables (no wrapper, direct access OK with explicit allowlist):** `localities`, `business_types`, `platforms`, `l3_patterns`, `l4_documents` — these are reference/global data. Documented at `docs/team/cross-tenant-tables.md`.
3. **Lint rule**:
	- New ESLint/ruff rule (added to VT-2.7 CI): forbid direct `supabase.from(...)` or raw SQL strings against any table in the tenant-scoped allowlist outside `apps/team/src/db/wrappers/`.
	- Rule name: `no-direct-tenant-db-access`. Documented at `docs/team/lint-rules.md`.
	- Allowed exceptions: migration scripts, seed scripts, ops scripts (path allowlist).
4. **`TenantIsolationViolation` handling**:
	- When raised, the orchestrator's error framework (VT-3.6) routes it as `database_error` with severity=`critical` AND emits an immediate Telegram alert to Fazal regardless of error-budget rules.
	- The violation is logged in `pipeline_log` with full context: run_id, tenant_id_expected, tenant_id_observed, query, stack trace.
	- Phase 1 tolerates zero such violations in production. A single one triggers an incident review.
5. **Tests** at `apps/team/tests/db/test_wrappers.py`:
	- Each wrapper's CRUD methods tested with synthetic tenants.
	- Cross-tenant attack: every wrapper attempted with mismatched tenant_id either returns empty or raises (by design; both are acceptable).
	- Smuggling attack: SQL injection attempt in `tenant_id` parameter (e.g., a UUID-like string with appended OR clause) is rejected by parameter binding.
	- The lint rule fires on a deliberately-broken test commit attempting direct DB access.
	- Cross-tenant table allowlist works: `localities` direct access is permitted (no lint failure).
6. **mypy strictness**: every wrapper method has typed signatures. Generic `dict` returns are forbidden — every method returns a Pydantic model.

## Architectural rules to enforce in this subtask
- Pillar 3: this is the second of three independent layers. The wrapper layer must hold even if RLS were misconfigured. CodeX confirms by reviewing the post-fetch `tenant_id` validation logic.
- Pillar 8: wrappers are mandatory. CodeX rejects PRs that work around the layer.
- The `TenantIsolationViolation` exception is critical-severity. CodeX rejects PRs that catch and silently retry it — it must propagate to Fazal alert.
- Cross-tenant tables (`localities`, `l3_patterns`, etc.) are explicitly allowlisted. Adding a new global table requires Type 2 governance + adding to the allowlist + updating the lint rule.
- Wrapper methods are not generic. Each table gets its own wrapper class with table-specific typed methods. CodeX rejects generic `query(table_name, predicate)` APIs that would defeat the purpose.
- The lint rule path allowlist is documented; ops + migration paths are explicitly listed. Adding a new exception is Type 1 (PR-reviewable change).

## Acceptance criteria
- Every Phase 1 tenant-scoped table has a typed wrapper.
- Cross-tenant table allowlist documented; only listed tables exempt.
- Lint rule `no-direct-tenant-db-access` added to CI; deliberately-broken commit fails the rule.
- Cross-tenant attack test passes for every wrapper.
- Post-fetch `tenant_id` validation raises `TenantIsolationViolation` on mismatch (synthetic test simulates a corrupted return row).
- Telegram alert + critical pipeline_log event triggered on violation.
- mypy --strict passes on all wrappers.
- PR title references `(VT-PrivacyArchitecture)`; reviewed by CoderX (mandatory — privacy-critical foundation).

## Out of scope
- KG query wrappers (VT-7.1's query builder is the equivalent; this subtask covers relational tables only).
- Agent context isolation (VT-8.2; that's the third layer).
- Per-row audit (Phase 2; we have query-level audit via `pipeline_log`).
- Encryption at rest (relies on Supabase defaults; out of scope here).
- Wrapper performance optimization (correctness first; profile in Phase 1.5 if needed).
