# VT-526 — B3 manager decision loop (decision core + situation authoring)

## What (the moat)
After a specialist runs, the manager READS its `SpecialistReturn` and decides what happens next. Before this, `SpecialistReturn` was **defined but never read** (`RETURN_STATE_KEY` unconsumed) and the handoff `situation` was **hardcoded `""`** — the two-way protocol existed only as a shape. This lands the decision REASONING + persistence half.

## Built (0 migrations)
- **`manager/decision.py`** — `decide_next_action(SpecialistReturn, has_next_step) -> ManagerDecision`, the missing consumer of `SpecialistReturn`, pure + deterministic:
  - pushback + `proposed_outcome` → **REVISE** (re-frame + re-dispatch, never force the action);
  - pushback + no path → **ESCALATE**;
  - action-taken, nothing actionable → **CLARIFY** (ask the owner);
  - action-taken, more steps planned → **NEXT_SPECIALIST**;
  - action-taken, plan exhausted → **ACCEPT** (→ verify).
- **`record_decision(...)`** maps each decision onto the B2 `manager_tasks`/`manager_task_steps` spine under the CAS guard (sequential step advancement): ACCEPT→step done+task verifying; NEXT_SPECIALIST→step done + next step running + task current_step advanced; REVISE→step back to pending; CLARIFY→step waiting + task waiting_owner; ESCALATE→step failed + task blocked.
- **Manager-authored `situation`** — `build_handoff_update` gains optional `situation` + `desired_outcome` params (backward-compatible: unset ⇒ the prior empty-situation / spec-default behaviour), so the manager can frame the handoff instead of the static `""`.
- Import-light: `decide_next_action` duck-types `SpecialistReturn` (TYPE_CHECKING only) + lazy-imports `task_store` — dep-less smoke-safe.

## Verify
ruff clean; dep-less import + 8 pure tests pass (incl reading the REAL roster `SpecialistReturn`, not a stub, + situation authoring); 13/13 full on fresh DB (5 `record_decision` CAS transitions verified against the B2 store).
