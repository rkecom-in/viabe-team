# CL-55 — L1 Knowledge Graph drops Apache AGE → Postgres + pgvector + time-aware relational

Apache AGE replaced by Postgres + pgvector + time-aware relational model for L1 Knowledge Graph. Triggered by 2026-05-16 finding that Supabase does not support Apache AGE (verified via Supabase discussion #40285).

Replacement architecture (validated against 2026 production patterns — Mem0 v1.0.0 pattern, Zep/Graphiti temporal model, current Postgres-native agent memory practice):
- l1_entities table: id, tenant_id, entity_type, attributes JSONB, embedding (pgvector), created_at
- l1_relationships table: id, tenant_id, from_entity, to_entity, relationship_type, attributes JSONB, valid_from, valid_to
- Recursive CTEs for 2-3 hop traversals when needed
- Multi-signal retrieval: vector similarity (pgvector) + entity match (relational filter) + BM25 (Postgres full-text)

Gains over Apache AGE plan: works on Supabase today, adds time-awareness (valid_from/valid_to columns) that AGE didn't give us, aligns with 2026 production pattern, single DB simplifies operations.

Losses: Cypher query language; some 3+ hop traversals slower than AGE would have been. Neither is on Phase 1-3 critical path.

IMPORTANT TERMINOLOGY CLARIFICATION: 'time-aware' / 'temporal columns' here refers to the data model property (facts have validity windows). It is NOT related to Temporal-the-product (the workflow orchestration tool we rejected in favor of DBOS). DBOS remains the durable execution substrate per CL-27. The collision of the word 'temporal' between these two unrelated contexts is a documentation hazard — use 'time-aware' or 'validity-window' going forward to avoid confusion.
