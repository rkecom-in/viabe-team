# CL-94 — pr-title regex rejects hyphens

The CI's pr-title check uses regex (VT-[0-9A-Za-z.]+)$ to validate PR titles end with a parenthesized VT-* reference. The character class allows alphanumerics + dot but NOT hyphen. This means fix-IDs like VT-3.3a-fix-1, VT-3.3b-fix-1, VT-3.2-fix-1, VT-CI-fix-1, VT-3.3a-fix-2, VT-Foundation-fix-1 — the exact ID convention from the fix-PR batch — all FAIL the gate.

CURRENT WORKAROUND (all 6 fix PRs): use the parent-task ID in the title (VT-3.3a, VT-3.3b, VT-3.2, VT-3, VT-3.3a, VT-2). Fix-id lives in the PR body and the Notion subtask ID. Works but creates ambiguity in git log/PR list — e.g. PR #11 and PR #15 both end (VT-3.3a) even though they're different fix subtasks.

FIX SCOPE (one-line PR):
- File: .github/workflows/ci.yml (or wherever the pr-title check lives).
- Regex change: (VT-[0-9A-Za-z.]+)$ → (VT-[0-9A-Za-z.-]+)$ (add hyphen to character class, escaped or not depending on position).
- Test: existing PR titles still pass; titles like 'feat(x): foo (VT-3.3a-fix-1)' now pass.

WHY NOT FOLDED INTO PR-fix-4 (VT-CI-fix-1): the audit brief for PR-fix-4 didn't scope it. Claude Code flagged it on #11 originally (anticipated the regex fix landing in #14) and again on #14 (noted it remains unscoped). Discipline call — do not silently expand brief scope.

PRIORITY: Low. Doesn't block any work. Convention-cleanliness issue.

SEPARATE PR: title 'fix(ci): pr-title regex accepts hyphens for VT-N-fix-M IDs (VT-CI-fix-2)'. Bundle into a small grab-bag PR if any other ci.yml tweak comes up. Otherwise micro-PR is fine.
