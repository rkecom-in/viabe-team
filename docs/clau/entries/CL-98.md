# CL-98 — env-rename PRs use pre-merge double-set ritual

DECISION (Fazal 2026-05-18, in response to audit concern 1 about PR #12 deployment coordination): all future env-rename PRs use a pre-merge double-set ritual, not atomic-swap. Generalizable rule, not one-off.

RITUAL (env-rename PR, e.g. RENAME `OLD_NAME` → `NEW_NAME`):
1. BEFORE merging the PR: set both `OLD_NAME` AND `NEW_NAME` in the target deploy environment (Railway, Vercel, etc.) to the same value. Confirm both are set.
2. Merge the PR.
3. Confirm in production logs that the renamed code path is healthy (reading from `NEW_NAME` successfully).
4. Delete `OLD_NAME` from the deploy environment.

WHY NOT ATOMIC SWAP: an atomic rename briefly puts production into a state where the code expects `NEW_NAME` but the env has `OLD_NAME` (or vice versa). For signature-verification paths like #12, this fails closed — Twilio webhooks 403 for the duration. Even brief outages are unacceptable post-launch, and design-partner Phase 1 traffic is a real signal we don't want to lose.

IMMEDIATE APPLICATION TO PR #12:
- Fazal sets `TWILIO_AUTH_TOKEN` AND `TEAM_TWILIO_AUTH_TOKEN` in team-web Railway service to the same value BEFORE clicking merge on #12.
- After merge: confirm team-web Twilio webhook is processing signatures successfully (synthetic test or first real webhook).
- Then delete `TWILIO_AUTH_TOKEN` from Railway.
- Same applies to any other deploy environment (preview deployments, staging) that has the variable set.

SCOPE: this ritual applies to ANY env-var rename, not just secrets. Includes `TEAM_*` prefix corrections, namespace changes, vendor migrations (e.g. when sarvam-ai-client → sarvamai per CL-92 ships, if the dep change requires an env-var rename alongside).

ADD TO CONVENTIONS: include in .claude/CONVENTIONS.md alongside CL-87 (merge ritual). Claude Code should flag env-rename PRs in the PR body with 'Requires pre-merge double-set per CL-93'.
