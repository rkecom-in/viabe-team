# VT-57 — Method 3: UPI transaction history export (PhonePe, GPay, Paytm)

VT-IntegrationInvention.

## Concept reference

- `concept-team-pillars.md` Pillar 4 (deterministic parsing; no LLM unless format is genuinely unstructured).
- `concept-team-pillars.md` Pillar 7 (consent: importing UPI history reveals owner's customer transaction patterns; treat owner upload as consent).

## Dependencies
- VT-Foundation Done.
- VT-IntegrationInvention VT-6.3 Done (dedup module — UPI transactions match many existing customers).
- VT-MCP-Tools VT-5.8 Done (`match_transactions` shares UPI string-matching logic; this method writes raw transactions, that tool attributes them to customers).

## What CoderC must produce
The Method 3 ingestion adapter. Owner uploads a PhonePe / GPay / Paytm transaction history export (PDF or CSV). Adapter parses, normalizes to canonical `transactions` table format, dedups customers via VT-6.3, writes rows.
1. **Adapter module** at `apps/team/src/ingestion/methods/upi_export.py`:
	- `ingest_upi_export(tenant_id, file_bytes, file_format, source: Literal['phonepe', 'gpay', 'paytm'], run_id) -> IngestionSummary`.
	- `file_format: Literal['pdf', 'csv']`. Each provider has its own parser; the adapter dispatches based on `source`.
2. **Per-provider parsers** in `apps/team/src/ingestion/methods/upi_parsers/`:
	- `phonepe_pdf.py`, `phonepe_csv.py`, `gpay_pdf.py`, `gpay_csv.py`, `paytm_pdf.py`, `paytm_csv.py`.
	- PDF parsing uses `pdfplumber` (deterministic table extraction; falls back to vision pipeline VT-6.1 only if structured extraction fails).
	- CSV parsing uses `pandas` + per-provider column mapping.
	- Each parser extracts: `transaction_date`, `amount_paise`, `direction` (credit / debit; tool keeps only credits since they represent customers paying the owner), `payer_vpa` (UPI handle of the customer), `payer_name` (sometimes absent in GPay), `transaction_ref` (provider's reference ID), `notes` (any memo line).
3. **Filtering**:
	- Drop debit transactions (owner paying others, not customer paying owner).
	- Drop transactions to merchant accounts other than the tenant's (some owners have personal + business UPI; tenant's primary VPA is captured at sign-up).
	- Drop refunds and reversals.
4. **Customer matching via VT-6.3 dedup**:
	- For each transaction, build a candidate customer from `payer_vpa` + `payer_name`.
	- Pass to `dedup_or_create()`. Result: existing customer (most common) or new customer (first time we've seen this VPA).
	- Insert transaction row linked to resolved `customer_id`.
5. **Confidence**:
	- UPI exports are structured data with transaction_ref as a unique ID, so duplicate detection is exact: re-uploading the same PhonePe export produces zero new transactions thanks to UNIQUE`(tenant_id, source, transaction_ref)`.
	- Per-row confidence default 0.95 (structured data). Drops to 0.7 if `payer_vpa` and `payer_name` both null (rare but possible — surfaces as an unmatched transaction the owner can resolve later).
6. **Telemetry**:
	- `pipeline_log` events: `upi_export_started` (with source + format), `transactions_parsed` (count), `customers_matched` / `customers_created`, `transactions_committed`, `upi_export_completed` (with summary).
	- Raw payload (which contains UPI handles + names) NOT in `pipeline_log`. Only counts.
7. **Owner summary**: WhatsApp template `upi_export_summary` (added to VT-13.3) sent at completion: "From your [PhonePe] export, we added [N] transactions to [M] customers (incl. [P] new customers)."
8. **Tests** at `apps/team/tests/ingestion/methods/test_upi_export.py`:
	- Each provider × each format = 6 parser tests with synthetic fixtures.
	- Refund / reversal filtering: synthetic with 2 refunds out of 10 → 8 transactions committed.
	- Debit filtering: synthetic with 3 debits out of 10 → 7 transactions committed.
	- Re-upload idempotency: same export uploaded twice → second upload produces 0 new rows (UNIQUE on transaction_ref enforces).
	- Dedup integration: 10 transactions across 5 customers → 5 customer rows, 10 transaction rows.
	- Cross-tenant: tenant_A export cannot create tenant_B rows.
	- PII absence in `pipeline_log`.
	- PDF vision fallback: synthetic corrupt PDF → vision pipeline tried; if it also fails, structured error.

## Architectural rules to enforce in this subtask
- Pillar 4: parsing is deterministic where possible. Vision (VT-6.1) is a fallback for malformed PDFs only, not the primary path.
- Pillar 8: per-provider parsers share a common output schema (`UPITransactionRow`). Adding a new provider is a new parser file, not a rewrite of the adapter.
- UNIQUE constraint on `(tenant_id, source, transaction_ref)` is the structural defense against double-import. Migration 017 adds this.
- Tenant's primary VPA captured at sign-up; transactions to other VPAs filtered out. CodeX confirms this filter is in place.
- Raw UPI handles + names never in `pipeline_log` payload.
- Phone-number resolution from VPA: many UPI handles are `<phone>@upi`. Parser extracts phone where format permits, attaches to candidate.

## Out of scope
- BHIM / other UPI app exports (Phase 2 if owner demand emerges).
- Bank-statement PDFs (different format; out of scope; UPI exports cover most of the customer-payment surface).
- Real-time UPI webhook integration (Phase 2; would require partnership with a payment aggregator).
- Automatic monthly UPI re-import (Phase 1 is one-off uploads; Phase 1.5 may add scheduled import via owner-granted email forwarding).
- VPA-to-customer-name reverse-lookup via external services — privacy concern, not done.
