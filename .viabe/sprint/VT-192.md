# VT-192 — team-web operator-resolve route

## Why this row exists

VT-188 (operator-role JWT substrate) shipped the Postgres role + RLS policy + `app_operator_audit_enabled()` helper + `resolve_phone_token_audited()` stored function. The HTTP wrapper at `apps/team-web/src/app/api/ops/resolve-phone/route.ts` was **deferred from VT-188** because:

1. Substrate is provable via direct `SET ROLE` canary (8/8 PASS without HTTP layer)
2. VT-123 Ops UI owns the auth flow + JWT issuance + UI integration
3. Route is plumbing — natural fit to VT-123's [resolve] button scope

## Two paths

**Option A:** Ship as standalone VT-192 row.
- Pro: small, focused scope; can ship before VT-123 starts
- Con: route has no UI consumer yet; tested via curl + synthetic JWT only

**Option B:** Absorb into VT-123 MVP scope.
- Pro: route + UI + auth flow ship together; tested end-to-end via Playwright
- Con: VT-123 already large; adding ~60 lines of route handler is small relative

**Recommend B** — absorb into VT-123. The route is meaningless without UI. Cowork can drop this row when VT-123 STEP-0 confirms the route is in VT-123 scope, OR keep as separate row if VT-123 declines.

## Scope (if shipped standalone)

`apps/team-web/src/app/api/ops/resolve-phone/route.ts`:

1. POST /api/ops/resolve-phone — body `{ phone_token: str }`
2. Validate Supabase JWT from Authorization header (operator-claim present + within audit-session window via `app_operator_audit_enabled` check)
3. Set Postgres GUC `app.jwt.operator_claim` via Supabase rpc helper
4. Call `rpc('resolve_phone_token_audited', { phone_token, jwt_hash })` — atomic SELECT + audit INSERT
5. Return `{ phone_e164: str }` on success, 403 on no-claim, 404 on token-not-found
6. Audit row is written by the stored function (no double-write)

## Dependencies

- VT-188 (substrate; ✓ on main)
- Supabase auth + `@supabase/supabase-js` (already wired)

## Acceptance criteria

- Synthetic JWT-with-operator-claim → route returns decrypted phone + audit row written
- JWT without operator-claim → 403
- Expired audit-session JWT → 403
- Cross-tenant attempt → 404
- Audit-write failure (synthetic) → 500 + no phone returned (atomic per stored function)

## Rule #15 Canary

`canaries/vt192_team_web_resolve_route.py` — httpx synthetic JWT → route → verify response shape + audit log + cross-tenant deny + ANTHROPIC ABSENT.

(Same canary pattern as VT-188; this row is the HTTP layer the canary previously tested via direct Postgres SET ROLE.)
