# VT-321 — complaint-freeze guardrail (Fazal's ONE non-configurable rule, #20)

Rostered from VT-313 playbook note #20 (Cowork 20260604T084000Z): *"an open complaint freezes ALL selling to
that customer — NON-configurable, no exceptions."* Cowork ruled this must be ENFORCED deterministically, not
left note-only — the owner-approval backstop isn't enough (a complaint customer could ride in a cohort the
owner doesn't scrutinize row-by-row).

## Finding (CC, grounded the state source): there is NO complaint state today
No complaint column on customers (only `opt_out_status`: subscribed/opted_out/blocked), no complaints table, no
open-complaint flag anywhere. So the guardrail has nothing to key on. **Splits (like VT-76):**

### Detection (gate-live — needs the live customer-inbound path / WABA)
A customer gets "open complaint" only via an inbound-message intent classifier (`customer_complaint`) on the
live customer-inbound path. Design choice (plan-first): a `customers.complaint_status` column
('none'/'open'/'resolved') vs a `customer_complaints` table. CC lean: the column (simplest, RLS-scoped).

### Exclusion (buildable NOW, given the state)
Deterministically exclude any customer with an open complaint from EVERY send cohort, fail-closed, in the
collapse/cohort path — independent of L4 retrieval (a missed retrieval must NOT unlock selling to a complaint
customer). Canary: a complaint-flagged synthetic customer is provably excluded from a cohort.

## Acceptance
- complaint-state source exists (column/table) + set by the inbound classifier (detection, gate-live).
- open-complaint customers excluded from every send cohort, fail-closed (exclusion, now).
- canary proves the exclusion.

Plan-first the flag-vs-table choice (Cowork) → build the exclusion now → wire detection at WABA. Customer-data-
go-live-relevant; NOT Reports-Jun15.
