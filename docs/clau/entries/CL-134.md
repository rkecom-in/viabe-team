# CL-134 — create_react_agent deprecated in langgraph 1.2.0

VT-3.9 PR 1/N (PR #21, branch feat/vt-orchestrator-agent-skeleton) uses `langgraph.prebuilt.create_react_agent` on pinned langgraph==1.2.0. This API is DEPRECATED in 1.2.0 and REMOVED in langgraph V2.0. New canonical location is `langchain.agents.create_agent`.

The ==1.2.0 pin protects current PR from breakage. Migration is required before any langgraph bump past 1.x.

Trigger for migration PR: BEFORE any PR that bumps `langgraph` past 1.x in apps/team-orchestrator/pyproject.toml. The bump itself must be blocked until the migration PR lands.

Migration scope:
- apps/team-orchestrator/src/orchestrator/agent/orchestrator_agent.py:
  - Change `from langgraph.prebuilt import create_react_agent` to `from langchain.agents import create_agent`
  - Verify create_agent signature matches (model, tools, prompt, name kwargs) on the target langgraph V2 version via Context7
  - Re-run integration tests with RUN_INTEGRATION_TESTS=1 to verify routing still works
- Same migration applies to ANY future create_react_agent usage. Pre-V2 migration audit: grep entire repo for create_react_agent before bumping.
- The `name="orchestrator_agent"` string is load-bearing for VT-3.4 supervisor wiring; migration must preserve it.
- langgraph V2 may break other langgraph imports (StateGraph, MessagesState, checkpointers). The bump PR is the trigger to do an audit of every langgraph import site in apps/team-orchestrator/, not just create_react_agent.

Fazal: this is a Phase 1 ship-now-migrate-later trade-off. Acceptable because (a) ==1.2.0 pin is hard, (b) V2 is not on the critical path, (c) migration is mechanical when triggered. Trigger discipline matters more than schedule.
