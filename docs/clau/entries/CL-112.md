# CL-112 — VT-3.3a-fix-3 brief: 3 vs 4 reason values drift

BRIEF ERROR caught by Claude Code during VT-3.3a-fix-3 implementation (CL-96 deferred rebase as fresh PR): brief case-C said 'split reason into 3 values (started/dupe/recovering)' in the implementation section, then immediately added a Pillar 7 caveat 'if RETRIES_EXCEEDED is terminal, flag rather than silently bucket into recovering.' These two statements are inconsistent — the caveat implies 4 buckets but acceptance line said 3.

ROOT CAUSE: same drift class as CL on VT-3.3c brief §3-vs-§5 inconsistency (page 364387c2-cc5a-8178-8904-fd7f44ec8214). Wrote literal acceptance count before completing the caveat reasoning. Acceptance line wasn't updated when caveat was added.

RESOLUTION: Option 1 — 4 buckets. Final reason values: started / dupe / recovering / terminal_failure. Mapping per DBOS 2.22 status enum:
- started: prior_status is None
- dupe: SUCCESS
- recovering: PENDING | ENQUEUED | DELAYED (in-flight, will resume)
- terminal_failure: ERROR | CANCELLED | MAX_RECOVERY_ATTEMPTS_EXCEEDED (dead)

DISCIPLINE LESSON (repeating): cross-section consistency check before delivering a brief. When adding a caveat that changes implementation behavior, update the acceptance line in the same edit pass. Both VT-3.3c brief drift and this brief drift hit at the moment of caveat-without-acceptance-update.

PATTERN: I have a tendency to write implementation spec, then add edge-case caveats, then ship acceptance without reconciling. Two instances in two days. Worth elevating from 'lesson' to 'discipline rule'.
