# VT-170 — customers table + tenant customer registry

## Origin

Surfaced as plan-time risk #1 during VT-104 (PII redactor) implementation. VT-104 brief §1 last bullet + canary Group C #8 assumed a `customers` table existed for customer-name tokenization, but no such table was present in migrations 000–022.

VT-104 shipped with a `name_registry: Callable | None` fallback per Cowork's Q1 answer in `.viabe/queue/done/VT-104/review.md`: a callable hook lets the redactor query an injected name source at runtime, but no production code path actually injects one yet. Customer-name tokenization is therefore present in the redactor signature but inert at runtime until a backing store exists.

This row's purpose: design + ship the `customers` table and wire a tenant-scoped customer registry that the VT-104 redactor's `name_registry` callable can consume.

## Why now (or not now)

**Not blocking VT-104 ship.** VT-104 still redacts every pattern-detectable PII type (phone, email, PAN, Aadhaar, IFSC, GST, CC, body) plus named-key fallbacks. Customer-name fuzzy detection without a registry is a Phase-2 feature in the VT-104 brief; the absence of a `customers` table is a graceful degradation, not a leak.

**Sequence this row when** any of the following actually lands:
- L1/L2 KG starts ingesting customer entities (it will want a canonical customers table to join against)
- DSR purge routines need a per-customer scope for "delete all data for customer X" (no table = no scope)
- Owner-facing "view customers" surfaces become reachable (currently they're not)

Until then, the redactor's name_registry callable remaining None is correct behaviour — there's nothing to look up.

## Scope (preliminary; refine when row activates)

1. **Migration** `023_customers.sql`:
	- `customers (id UUID PK, tenant_id UUID FK → tenants(id), display_name TEXT, phone_e164 TEXT NULLABLE, email TEXT NULLABLE, created_at TIMESTAMPTZ DEFAULT now(), updated_at TIMESTAMPTZ DEFAULT now())`
	- Unique index: `(tenant_id, phone_e164)` partial WHERE phone_e164 IS NOT NULL
	- Unique index: `(tenant_id, email)` partial WHERE email IS NOT NULL
	- RLS policy: tenant-scoped SELECT/INSERT/UPDATE/DELETE bound to `app.tenant_id` GUC (mirroring existing tenant-scoped tables — see `015_app_role.sql` pattern).
2. **Customer registry service** (`apps/team-orchestrator/src/orchestrator/privacy/customer_registry.py`):
	- `get_customer_names_for_tenant(tenant_id: UUID, db: Connection) -> list[str]` — returns `display_name` list for redactor's bigram match.
	- In-process LRU cache with explicit invalidation hook on `customers` UPSERT.
	- Plumbed into `redact_for_log` + `redact_for_langsmith` call sites as the `name_registry` callable (currently None per VT-104).
3. **DSR integration** (if VT-PrivacyArchitecture is far enough along):
	- Hard-delete path: `DELETE FROM customers WHERE id = $1 AND tenant_id = $2` cascades to dependent tables (pipeline_log payloads referencing this customer get redacted-not-deleted per Pillar 7's audit requirements; design TBD).
4. **L1/L2 KG hook** (if L1/L2 KG row is sequenced first):
	- Customers table is the join target for entity resolution in L1.
	- Adapter pattern so KG doesn't import privacy-module directly (one-way dep).
5. **Tests**:
	- RLS isolation: tenant A cannot see tenant B's customers.
	- Registry cache invalidates on UPSERT.
	- `redact()` with registry-backed callable redacts known names; unknown names pass through.
6. **Canary** (per Rule #15):
	- Insert real customer row into dev DB.
	- Real call to `redact()` with the registry callable injected.
	- Verify redaction works.
	- Verify RLS denies cross-tenant read with `app.tenant_id` GUC set to a different tenant.

## Out of scope

- Customer merge/dedup logic (separate row).
- Customer state machine fields (separate row, coordinate with VT-billing-state-machine).
- Owner-facing CRM surface (Phase 3+).
- ML-based name detection without a registry (Phase 2 per VT-104 brief).
