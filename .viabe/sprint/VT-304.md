# VT-304 — wire nightly audit-chain verify (DBOS.scheduled) + alert route

Fast-follow split out of VT-80 (Cowork 20260603T152000Z). VT-80 shipped the
tamper-evident hash-chain + `verify_chain` + a ready-to-wire
`run_audit_chain_verify_body` (observability/audit_verify.py), but deferred the
DBOS scheduled registration.

## Why split (not done in VT-80)
Registering a 6th `@DBOS.scheduled` handler shifts DBOS `app_version` — which is
recovery-sensitive (see `scheduled_triggers.register_scheduled_triggers`
docstring: re-registration mid-run breaks the recovery filter). It deserves its
own scheduler canary proving (a) the job fires on cron, (b) a chain break raises
a critical alert, (c) registration doesn't disrupt the existing 5 handlers'
recovery. Meanwhile VT-80's chain is verifiable on demand
(`run_audit_chain_verify_body`) — NOT stale.

## Scope
1. Add `audit_chain_verify_scheduled(scheduled_time, actual_time)` handler +
   `AUDIT_CHAIN_VERIFY_CRON` (nightly, off-peak IST) in scheduled_triggers.py,
   calling `run_audit_chain_verify_body`.
2. Register it in `register_scheduled_triggers` (the 6th handler — EXTEND the
   surface, mirror the VT-47 5th-handler precedent).
3. On `verify_chain` failure → critical alert via the existing alerts path
   (alerts/triggers.py privacy_audit_event posture) so a tamper/corruption is
   surfaced, not just logged.
4. Scheduler canary: synthetic clock fires the body; a tampered row → alert
   emitted; clean chain → no alert; existing handlers' recovery unaffected.

## Dependencies
- VT-80 Done (verify_chain + run_audit_chain_verify_body shipped).
