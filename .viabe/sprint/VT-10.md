## Why this parent exists
Billing is where the product's promises are kept or broken. Three commitments are Type 3 locked: (1) founding-tier ₹2,499/month locked forever for the first 100 paid subscribers (atomic counter, race-safe), (2) 14-day no-card trial extendable up to 60 days, with card capture only at conversion, (3) day-39 deterministic SQL evaluation that triggers a full refund + cancellation if ARRR < 2x cumulative fees.
None of these are negotiable. The day-39 evaluation in particular MUST be deterministic — same inputs produce same output. No LLM. Pure aggregation. If a subscriber qualifies for a refund, they get one, no judgment call. This is the trust contract that makes the product viable; if it leaks, customer reviews end the company before it scales.
## What this parent owns
1. Razorpay Live integration: plans, webhook, settlement bank wiring. Founding (₹2,499), Standard (₹4,999), Pro (₹14,999) tiers. Prices read from `STANDARD_PRICE_PAISE` and tier-specific env vars — never hardcoded.
2. 14-day trial with performance-based extension up to 60 days: extension logic is deterministic, based on ARRR not yet demonstrated.
3. Card capture at conversion to paid, not at sign-up: this is core to the trial experience.
4. Day-39 evaluation engine: deterministic SQL aggregating ARRR since paid-conversion vs cumulative fees paid. ARRR < 2x → full refund + cancellation triggered. Pure SQL, no LLM, same inputs same output.
5. Refund execution flow: triggers Razorpay refund API + Supabase state transition (`paid_active` → `refunded`) + graceful-exit conversation hand-off to VT-9.5.
6. Founding-member counter: atomic counter, race-safe (Postgres advisory lock or SELECT FOR UPDATE). First 100 paid conversions lock the founding tier; 101st pays ₹4,999.
## Architectural rules binding every subtask
- [Concept-team.md](http://Concept-team.md) §6 Type 3 commitments are inviolable. Pricing, trial length, day-39 evaluation cannot be amended without board protocol.
- Day-39 is deterministic. No LLM, no judgment call, no agent involvement. Pure SQL. Test with same inputs always produces same output. CI test verifies idempotency.
- Founding counter is atomic. Race conditions cannot allow the 101st subscriber to slip in at ₹2,499. Stress test confirms.
- Card capture only at conversion to paid. Webhook handlers enforce this; trial sign-up flows never request card.
- Pillar 8 (no patchwork): pricing exceptions go through env-var configuration + Supabase coupons table, never hardcoded in code.
- Refund flow is logged immutably in `billing_audit_log`. Every refund records the day-39 evaluation snapshot, the Razorpay refund ID, the state transition timestamp.
- The trial-extension decision is deterministic: extension granted iff structured ARRR-progress check fails specific threshold. No LLM judgment.
- Razorpay webhook handlers are idempotent: duplicate webhook delivery does not double-charge, double-refund, or double-transition state.

## Definition of done
- All 6 subtasks Done.
- Razorpay Live verified (KYC complete via VT-13.4); test charge to a real card succeeds and settles.
- Trial flow: sign-up does not capture card; conversion to paid captures card via Razorpay's hosted flow; trial-extension test confirms deterministic extension logic.
- Day-39 evaluation: synthetic subscriber with ARRR < 2x triggers refund + cancellation. Same inputs produce same output across 100 runs (no flake).
- Refund execution: Razorpay refund API succeeds, state transitions to `refunded`, audit log written immutably, hand-off to VT-9.5 graceful-exit conversation occurs.
- Founding counter stress test: 110 concurrent paid conversions; exactly 100 pay ₹2,499, 10 pay ₹4,999. No race-condition leakage.
- Webhook idempotency: duplicate Razorpay webhooks do not double-execute any side effect.
- All prices read from env vars; no hardcoded rupee amounts in code (CI grep confirms).
