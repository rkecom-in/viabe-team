# CL-379 — DECISION: DSR purge (PR #51, Brief 2) uses a privileged RLS-bypassing service-ro

PR #51 'VT-DSRPurge' (branch feat/vt-dsr-purge-routine, HEAD 4a7fcc8, 12/12 CI green, 5 substrate tests pass on live PG) delivers the DSR deletion-fulfillment routine. Brief 2.

WHAT SHIPPED: purge routine over 12 tenant-scoped tables in FK-safe order; 3 retention surfaces preserved (tenants anonymized; privacy_audit_log untouched + purge event appended; dsr_tickets status='completed' tombstone). DBOS durability layer + razorpay_webhook_events documented as out-of-scope findings. 5 DB-substrate tests incl. the critical cross-tenant isolation invariant (purging tenant A does not touch B) and DPDP audit-log retention.

ARCHITECTURE DECISION (Clau-owned, decided this session): the purge uses a PRIVILEGED service-role path that bypasses RLS, NOT the RLS-scoped tenant_connection (SET ROLE app_role) path. Claude Code surfaced this; per decision authority (RLS approach + DB schema = Clau-owned) Clau makes the call, not Claude Code. DECISION: privileged path APPROVED. Rationale: migration 015's app_role grants cover only 7 tables explicitly; subscriptions / phone_token_resolutions / privacy_audit_log predate 015 with no grants. Widening app_role grants to cover them would make the DPDP-required append-only privacy_audit_log deletable by any tenant writer — a strictly worse security posture. A DSR purge is inherently an operator/admin action crossing every tenant table; the RLS-scoped path is the wrong shape for it. Privileged path with explicit WHERE tenant_id = %s on every query is the correct fit.

CONDITION ON THE DECISION: because the privileged path switches RLS OFF, a query missing its WHERE tenant_id predicate becomes a silent cross-tenant data-wipe with no RLS backstop. test_purge_preserves_other_tenant_data proves the current happy path is scoped, but does NOT prove a future unscoped query is caught. REQUIRED FAST-FOLLOW (rostered, Critical, Privacy): a guard that makes a tenant-unscoped DELETE on the privileged path impossible or loudly test-failing — mandatory-predicate query builder, lint, or a test that fails on an unscoped delete. May fast-follow #51 rather than block it, but must be a rostered row, not a footnote.

TWO DECISIONS REMAIN FAZAL-OWNED: (1) anonymize-in-place vs hard-delete the tenants row — Claude Code drafted anonymize; Clau recommends anonymize (tenants row is FK-referenced by the retained privacy_audit_log + dsr_tickets; hard-delete would orphan or cascade-destroy the DPDP audit trail). (2) DBOS layer is time-based purge only (~2.5h cycle, not synchronous on DSR request) — confirmed, matches CL-374; privacy notice must state this.

#51 cleared to merge once Fazal confirms (a) the anonymize choice and (b) the unscoped-DELETE guard as a rostered fast-follow.
