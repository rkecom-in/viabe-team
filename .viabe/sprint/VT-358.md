# VT-358 — customer opt-out gate has the SAME whole-body-exact miss VT-329 fixed for the owner

Flagged by CC during the VT-329 fix + ruled gate-class by Cowork (20260606T141500Z).
`integrations/customer_inbound.py` matches opt-out with whole-body-EXACT (`if norm in
_OPT_OUT_KEYWORDS`, ~L61/L136) — so a CUSTOMER texting "please STOP" / "please बंद करो" / a
Hinglish "band karo" MISSES the deterministic opt-out gate. This is the SAME compliance class
VT-329 just closed for the owner pre_filter, but on the **customer** surface — and a customer's
STOP is consent-critical (their right, not the owner's). A missed customer opt-out = an unlawful
continued send (DPDP + WhatsApp policy).

## Scope
1. Port the VT-329 fix to `customer_inbound.py`: replace whole-body-exact opt-out matching with
   boundary-safe **phrase-CONTAINMENT** (the Unicode lookaround `(?<!\w)kw(?!\w)`, re.UNICODE —
   `\b` is dead for Devanagari matras) + **NFC**-normalize the body before matching.
2. Cover the bilingual keyword set (EN + Devanagari + Hinglish: STOP/बंद करो/band karo/roko), the
   same set the owner gate now uses (consider sharing the helper so the two paths can't drift).
3. Same canary style as VT-329: "please STOP" / "please बंद करो" / "band karo" all opt the
   customer out; a benign message doesn't; fail-not-skip pure tests.

## HARD GATE
Customer opt-out is **launch-gating** — it must hold before any real customer messaging goes live
(paired with VT-231 + the consent infra). A missed STOP is a compliance breach.
