# VT-311 — L2 retention (18-month soft-delete) + 100K-event perf test

## Why
The original VT-66 spec called for L2 retention (18-month soft-delete) + a
100K-event index-perf test. Both were consciously deferred out of PR #275 (which
shipped the substrate + dual-projection + retrieval + Composer-wire). This is the
**lifecycle/compliance** add — L2 already works; this caps storage + proves the
read pattern scales. Cowork (20260603T185000Z): keep it as its OWN row, NOT
folded into VT-309 (retention ≠ emit coverage). DPDP storage-limitation concern;
synthetic-only until VT-231, so not launch-blocking, but tracked.

## What to produce
1. **Retention (soft-delete):** add `deleted_at TIMESTAMPTZ` to `episodic_events`
   (new migration via the allocator — never hand-pick). Events older than 18
   months are soft-deleted (`deleted_at` set; row stays for audit). Config-driven
   window (`config/l2_retention.yaml` or a constant — match the project's existing
   retention-config posture; check VT-144/owner_inputs retention for precedent).
2. **Read exclusion:** `l2_query.recent_events` / `events_for_entity` /
   `count_events` MUST filter `deleted_at IS NULL`. Add to the existing queries +
   test the exclusion.
3. **Sweep:** a DBOS.scheduled job (or fold into an existing nightly sweep — see
   VT-304's nightly audit-verify pattern) that sets `deleted_at` on aged rows.
   NEVER hard-delete in Phase 1 (audit trail).
4. **Perf test:** synthetic 100K-event tenant; assert `recent_events(limit=50)`
   returns in <100ms (the spec's bar) on the `(tenant_id, occurred_at DESC)`
   index. Confirm the index is used (EXPLAIN) and that the soft-delete filter
   doesn't force a seqscan (partial index on `deleted_at IS NULL` if needed).

## Dependencies
- VT-66/67 Done (#275) — the `episodic_events` table + `l2_query`.

## Acceptance criteria
- Migration applies; `deleted_at` added; RLS unchanged.
- Aged events soft-deleted by the sweep; reads exclude them; audit rows remain.
- 100K-event recent-50 read <100ms, index-backed (EXPLAIN proof in PR).
- Throwaway-DB validation + ruff + pre-push green. CL-422 synthetic.

## Out of scope
- Hard-delete / GDPR-erasure mechanics (VT-76 reconstitution handles opt-out anonymization separately).
- The agent-decision emit sites (VT-309).
