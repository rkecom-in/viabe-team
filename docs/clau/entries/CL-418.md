# CL-418 — Discipline Rule #17: CC must not stash untracked files during merge tasks

## Decision

During any `task-merge` workflow (CC's merge cycle: checkout → pull → merge → cleanup), Claude Code MUST NOT run `git stash --include-untracked` or `git stash -u`. If a `git checkout` or `git pull` requires a clean working tree and untracked files are blocking it, CC must:

1. **STOP** the merge flow before stashing
2. **Send a signal to Cowork** describing what's in the working tree (file list)
3. **Wait for Cowork to commit / move / cleanup** the working-tree files
4. **Resume** the merge flow only after Cowork confirms working tree is acceptable

Stash-without-include-untracked is fine (it preserves tracked-file changes). The blocked operation is **anything that removes untracked files from the working tree without Cowork's explicit consent**.

This becomes **Rule #17** in `docs/clau/discipline-rules.md`.

## Why this is a Standing decision

Single shared working directory across Fazal (CEO) + Cowork (delivery captain) + Claude Code (implementer) + Claude (Cowork chat sessions). Same git index. **Any untracked files Cowork is mid-curating are visible to CC at merge time.** When CC runs `git stash --include-untracked` to clean the tree for a pull, those Cowork-side untracked files get swept into the stash + vanish from the working tree.

This already happened **twice** in the 2026-05-26 session:

1. **VT-30 PR #66 merge** — CC's `git commit -am` swept Cowork-side AMENDMENTS to existing tracked files into the PR commit. Bundle accepted by Fazal as audit-trail-coherent ("Go on both"), but established the precedent for the discipline carryforward.

2. **VT-178 PR #67 merge** — CC's `git stash --include-untracked` post-merge cleanup swept ~30 untracked Cowork-side files (9 sub-row sprint files + launch-tracker.md + templates.md + queue dirs + architecture mermaid) into `stash@{0}`. They vanished from the working tree. Recovery required Fazal to manually `rm .git/index.lock && git stash apply stash@{0}` (Cowork's sandbox couldn't unlink the lock file at FUSE-mount layer).

Fazal directive (2026-05-26, verbatim): *"I told you last time as well that it's a single folder used by claude code, and you and also me. Any change we make happens on the same git index."*

The recurrence is the trigger for codification.

## Companion rule (carryforward from VT-30)

VT-30 also established: **CC should use explicit `git add <files>` (whitelist), NOT `git commit -am`.** `-a` adds all modified-tracked files, sweeping Cowork-side edits to existing tracked files into CC's PR. The whitelist pattern protects PR scope boundary.

Combined: CC operates ONLY on the file set CC is explicitly building for the current row. Cowork-side untracked files are stashed (without `-u`), committed separately by Cowork, OR left in the working tree if they don't conflict with the merge target.

## Implementation guidance for CC

**Pre-merge checklist (add to merge-task signal acknowledgment):**

1. `git status --porcelain` — capture working tree state
2. Filter for `??` (untracked) entries OUTSIDE the row's scope (e.g., not in `apps/team-orchestrator/canaries/vt<N>_*.py` for the current row)
3. If any out-of-scope untracked files exist → STOP, send signal to Cowork, wait for response
4. If working tree contains modified-tracked files OUTSIDE the row's scope → STOP, send signal to Cowork (likely Cowork-side amendments that need their own commit), wait
5. ONLY when working tree contains only row-scoped changes → proceed with merge cleanup

**Failure mode if violated:** Cowork-side work vanishes. Recovery requires manual stash-apply (and may require Fazal terminal access if .git/index.lock blocks Cowork's sandbox).

**Diagnostic command after a suspected sweep:** `git stash list && git stash show --include-untracked stash@{0}` shows what got swept. `git stash apply stash@{0}` (NOT `pop` — keep stash as backup) restores.

## Cowork-side companion discipline

Cowork commits its own substrate work to its own commit/branch/PR (per Fazal directive 2026-05-26 — direct commit to main acceptable for documentation/state files; PR for anything bigger). Cowork SHOULD NOT leave its untracked working-tree drift to be passively swept by CC's next merge cycle. The bound is: when Cowork has finished a substrate-work session, commit it before the next CC task starts.

## How future sessions must treat this

**DO:** Treat `git stash --include-untracked` and `git stash -u` as forbidden during merge tasks. Surface working-tree obstacles to Cowork via signal. Use `git status --porcelain` as the pre-merge audit.

**DO NOT:** Sweep untracked files. Sweep tracked-but-out-of-scope modifications via `-am`. Re-litigate the rule. Try to "fix" the working tree by deleting things you don't recognize.

**Edge case:** Genuine merge conflicts on tracked files in CC's scope can be resolved as normal. The forbidden operation is **silent removal/sweep of out-of-scope working-tree state without Cowork consent.**
