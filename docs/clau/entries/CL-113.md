# CL-113 — terminal_failure: no downstream alert / DB record / admin recovery path

VT-3.3a-fix-3 introduces a 'terminal_failure' reason in twilio_ingress response when prior workflow is in ERROR / CANCELLED / MAX_RECOVERY_ATTEMPTS_EXCEEDED state. DBOS.start_workflow on a terminally-failed workflow no-ops — nothing recovers it without manual intervention.

Current VT-3.3a-fix-3 PR: returns reason='terminal_failure' in response body (honest per Pillar 7), but no downstream observability fires on it. team-web logs the reason string; no alerting, no escalation, no Slack ping, no owner-notification.

For Phase 1 with 10 design partners: log-only is acceptable. Frequency expected ~zero (workflows aren't reaching MAX_RECOVERY_ATTEMPTS_EXCEEDED in normal operation).

For Phase 2+ at scale: terminal_failure is a real ops signal. Need:
- Slack/email/PagerDuty alert on each occurrence (frequency low enough that no batching needed).
- DB record so we can grep terminal failures per tenant.
- Recovery path: surface to Fazal/admin UI with 'manually retry' or 'mark closed' actions.

FIX SCOPE (future): wire terminal_failure response into:
- pipeline_steps row with step_kind='terminal_failure_detected'
- Alert hook (probably via existing observability pipeline VT-122 when that ships)
- VT-6 (Owner Surface) admin view for stuck workflows

PRIORITY: Low for Phase 1. Medium once design partners > 50 OR if a terminal_failure is observed in soft launch.

SCHEDULE: bundle with VT-122 (observability writer) or surface as VT-6 admin requirement. Not a standalone subtask.
