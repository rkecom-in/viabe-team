# VT-194 — Prompt caching for orchestrator-agent (Sprint 2 anchor)

## Why this row exists

VT-193 brain wiring surfaced observed cost: ₹8.95/dispatch on Opus 4.7 with 5351 input tokens. Most of that input (system prompt + tool inventory) is invariant across dispatches within a tenant. Anthropic's `cache_control: {type: ephemeral}` gives 90% discount on cached tokens. Implementing this is a 1-2 day surface-level change to `build_orchestrator_agent` and likely cuts per-dispatch cost by 5-10x.

CL-419 locks this as Sprint 2 anchor. Fazal directive: "moat first keeping cost in control" — VT-194 IS the "cost in control" tactical work that lands before moat work compounds the burn rate.

## Scope

### Add cache_control to system prompt + tool schemas

1. Modify `build_orchestrator_agent(...)` in `agent/orchestrator_agent.py`:
   - System prompt assembled as message content blocks with `cache_control: {"type": "ephemeral"}` markers on the static portion (system prompt body + tool inventory descriptions)
   - User message stays uncached (varies per dispatch)
2. Verify langchain `ChatAnthropic` propagates `cache_control` to the Anthropic SDK call
3. Confirm cache hit / cache miss metadata captured in `OrchestratorReasoningCallback` for observability

### STUB tool inventory pruning

4. Audit `tools/__init__.py` tool registry — identify STUB tools (write_l0_fragment_stub, etc.) that carry full schemas but no real implementation
5. Either: remove from registry until real implementation lands, OR collapse to a single "placeholder_for_phase_2" entry with minimal schema
6. Target: ~1500 input token reduction independent of caching

### System prompt token reduction (optional, low-risk)

7. Audit `prompts/orchestrator_agent_system.md` for always-on text that could be per-task injection
8. Move WHEN-to-write-L0 guidance to a separate context block injected only when L0 write is contextually relevant

## Acceptance criteria

1. Same canary input as VT-193 (`"can you give me a quick summary of my customers this week"`) — observed cost drops to **<200 paise/dispatch** (vs 895 baseline) — minimum 4x reduction
2. Cache hit metadata visible in `agent_reasoning_step` rows (`usage_data["cache_creation_input_tokens"]` + `usage_data["cache_read_input_tokens"]` populated; first dispatch creates cache, subsequent reads it within TTL)
3. No regression on the 4 load-bearing VT-193 canary assertions (A1-A4 still PASS on brain wiring)
4. No capability regression — agent still routes correctly, still calls tools, still composes output

## Rule #15 Canary (mandatory)

`apps/team-orchestrator/canaries/vt194_prompt_caching.py` — 5 assertions:

1. First dispatch creates cache: `cache_creation_input_tokens > 0` in agent_reasoning_step row
2. Second dispatch (within 5-min TTL) hits cache: `cache_read_input_tokens > 0` + `cache_creation_input_tokens == 0`
3. Observed cost on second dispatch <200 paise (vs ~895 baseline)
4. Capability preserved: dispatch terminates `status='completed'` (same as VT-193 canary A1)
5. Wall-clock not regressed (caching can sometimes ADD latency on cache creation; assert <60s)
