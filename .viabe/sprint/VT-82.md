## What CoderC must produce
The sign-up backend processing pipeline. Owner submits the form on the landing site (VT-11.2); this endpoint validates the submission, creates the tenant, records consent, sends welcome.
1. **API endpoint** `POST /api/signup` at `apps/team/src/api/signup.py`:
	Request body (4 mandatory fields + consent):
	- `business_name: str` (1-100 chars)
	- `owner_name: str` (1-100 chars)
	- `owner_whatsapp: str` (E.164 normalized via `phonenumbers`; rejected if not Indian +91 number)
	- `business_type: str` (free-form; common values populated as suggestions in form UI)
	- `consent_dpdpa: bool` (must be true; rejection on false)
	- `consent_residency: bool` (must be true; rejection on false)
	- `referral_source: str | None` (optional analytics)
	- `preferred_language: Literal['en', 'hi']` (defaulted from form UI selection)
2. **Validation**:
	- Required-field checks return structured errors (400 with `field` + `message`).
	- Phone number must validate via `phonenumbers` and pass +91 country check.
	- Business name + owner name pass profanity / nonsense filter (a small word-list check, NOT LLM; stored in `apps/team/config/signup_blocklist.yaml`).
	- Duplicate detection: phone already registered as another tenant's owner → 409 with helpful guidance ("This number is already linked to a Viabe Team account. To start a new account, use a different number, or contact support to change owners on the existing account.").
3. **Tenant creation transaction**:
	- Insert `tenants` row with `phase='onboarding'`, `signed_up_at=now()`, `paid_conversion_at=null`, `plan_tier='founding'` if founding-counter (VT-Billing VT-10.6) shows space, else `'standard'`.
	- Insert `consent_records` row capturing the dpdpa + residency consent flags + sign-up timestamp + DPDPA disclosure version + residency disclosure version (all from `apps/team/config/disclosure_versions.yaml`).
	- Emit `consent_recorded` event to privacy audit log (VT-8.9).
	- Increment founding-tier counter atomically if founding (VT-10.6).
	- Trial period: `trial_started_at=now()`, `trial_extension_count=0`. The 14-day trial countdown is implicit (trial ends 14 days after `trial_started_at`).
4. **Welcome message**:
	- Send via VT-5.7 (`send_whatsapp_template`) using `welcome` template.
	- Template params: business name, owner name, dashboard URL, founding-tier badge if applicable.
	- Failure to send welcome: tenant creation succeeds, welcome retried by DeliveryBot-equivalent scheduled trigger (VT-3.5).
5. **Response**:
	- 201 Created with `{tenant_id, phase, plan_tier, trial_started_at, trial_ends_at}`.
	- Frontend redirects to a "check your WhatsApp" landing page with step-by-step onboarding.
6. **Observability + analytics**:
	- Per-day sign-ups counter (VT-12.3).
	- Per-source sign-ups (referral_source field).
	- Welcome-send success rate.
7. **Tests** at `apps/team/tests/owner_surface/test_signup.py`:
	- Happy path: valid 4 fields + consent → tenant created, consent recorded, welcome sent.
	- Missing field: 400 with `field` identification.
	- Bad phone format: 400.
	- Non-+91 phone: 400.
	- Profanity in name: 400 with helpful message.
	- Duplicate phone: 409.
	- Missing dpdpa consent: 400.
	- Missing residency consent: 400.
	- Founding-tier saturation: tenant gets `plan_tier='standard'` when founding counter at limit.
	- Welcome-send failure: tenant created, welcome retry scheduled.
	- Consent audit: privacy audit log row written with both disclosure versions.
## Architectural rules to enforce in this subtask
- Pillar 7: consent flags MUST be true. CodeX rejects PRs that allow tenant creation without explicit consent.
- Pillar 8: this is the SOLE sign-up path. CodeX rejects shadow paths (e.g., admin-creating tenants for testing should use a separate marked endpoint that's never exposed publicly).
- Phone normalization via `phonenumbers`; +91 check structural.
- DPDPA + residency disclosures are versioned in YAML; consent records the version. Updating disclosures requires a Type 2 PR (so existing tenants on old versions are visible).
- Welcome-send failure does NOT roll back tenant creation. Welcome is retried; tenant proceeds to trial. This is intentional Phase 1 behavior — owner already submitted; failure to message shouldn't lock them out.
- Founding-tier counter increment is atomic (lives in VT-10.6); this subtask uses the primitive correctly, doesn't reimplement.
