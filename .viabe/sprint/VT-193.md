# VT-193 — Brain wiring (runner → supervisor seam)

## Why this row exists

E2E smoke test 2026-05-27 surfaced that `pre_filter.brain` routes write a `record_brain_pending` placeholder via `runner.py:303-307` and mark the run `status='escalated'`. The orchestrator-agent + supervisor + langchain callback + L0 memory substrate (VT-125/126/27/180/182/183/etc.) all merged but **no production code path imports `OrchestratorAgentDriver` or `build_supervisor_graph`**. The substrate exists but the call site doesn't. This row closes that seam.

VT-3 parent says all 18 sub-items Done. That's bookkeeping correct (the supervisor was built per VT-27; the agent per VT-125) but production-incomplete because the runner never invokes them. VT-193 is the closure row for VT-3 parent's actual DoD: "Smoke run completes end-to-end: synthetic owner message → webhook → pre-filter → orchestrator-agent reasoning → supervisor handoff → unified output."

## Scope

### Single load-bearing change — `runner.py:303-307`

Replace:
```python
elif result.kind == "brain":
    # VT-3.4 brain not yet wired — record a brain-pending step and mark the
    # run 'escalated' so it is not silently reported as completed (Pillar 7).
    record_brain_pending(tenant_id, run_id, result.reason)
    final_status = "escalated"
```

With a real invocation that:
1. Records a `brain_dispatch` envelope step (so we keep observability of "we routed to brain")
2. Enters `observability_context(run_id=UUID(run_id), tenant_id=UUID(tenant_id))` so VT-125's `OrchestratorReasoningCallback` writes `agent_reasoning_step` rows
3. Builds the supervisor graph via `build_supervisor_graph(model=...)` (use the same model env-resolution path as `build_orchestrator_agent`)
4. Invokes the graph with the inbound `event` mapped to a `BaseMessage` and the seed state (`SubscriberState` for the tenant)
5. Captures the graph's terminal output (composed reply payload + any state updates)
6. Writes a `compose_output` envelope step carrying the unified-output composer result (VT-30 substrate)
7. Closes the run with `final_status` determined by terminal node:
   - `orchestrator_terminal` reached with response → `completed`
   - `collapse` reached (specialist returned) → `completed`
   - `escalate_to_owner` tool was called inside the agent → `escalated` (with reason set to the agent's terminal verdict)
   - `HardLimitExceeded` raised by `OrchestratorAgentDriver._enforce_*` → `aborted_hard_limit` (NOT `escalated`)

### Helper extracted

Pull the wiring into `orchestrator/agent/dispatch.py` (new file) — a `dispatch_brain(event, state, run_id, tenant_id) -> DispatchResult` function. Keep `runner.py` thin: it imports `dispatch_brain` and calls it. This isolates the LangGraph + observability_context complexity from `runner.py`'s pre-filter routing logic and makes the dispatch unit-testable.

### Error-taxonomy routing (Pillar 8)

| Failure | Handling |
|---|---|
| `HardLimitExceeded` (tokens / tool calls / depth / wall-clock / cost) | Write `aborted_hard_limit` envelope step with the offending limit; `final_status='aborted_hard_limit'`. Do NOT raise; the run terminates cleanly. |
| Agent calls `escalate_to_owner` tool | Tool return propagates to terminal; write `escalated` envelope step; `final_status='escalated'`. |
| Unhandled exception inside graph invoke | Re-raise to DBOS workflow; DBOS auto-resume handles retries; on retry exhaustion DBOS writes error envelope; `final_status='aborted_hard_limit'` is NOT used here (that's for limit hits, not exceptions). |
| Anthropic API quota / 429 / timeout | Caught by `OrchestratorReasoningCallback.on_llm_error`; written as cost-track step with error metadata; re-raised. DBOS retries the workflow. |

### DBOS step boundaries

The brain dispatch happens INSIDE the existing `process_inbound_event` DBOS workflow. The graph invoke itself does NOT need to be a separate `@DBOS.step` — the supervisor's inner state-transition hooks already write per-node envelopes via VT-180 `write_step`. The `dispatch_brain` function call site stays inside the parent workflow's transaction.

### Idempotency

`process_inbound_event` is already idempotent on `(tenant_id, run_id)`. No new idempotency contract needed at the brain dispatch layer.

## Acceptance criteria

2. `pipeline_steps` for that run includes:
   - `webhook_received` ✓ (already works)
   - `brain_dispatch` (new envelope kind, NOT `agent_invocation` — different semantics: "we routed to brain" vs "placeholder until brain ships")

6. Backward compatibility: `pre_filter.direct_handler` and `pre_filter.reject` paths unchanged; their tests still pass.
