# VT-163 — Approved-templates registry (language-keyed)

## Scope (written from brief 20260530T184226Z, WAVE A)

Replace the inline `_PHASE1_APPROVED_TEMPLATES` placeholder tuple in
`apps/team-orchestrator/src/orchestrator/context_builder.py:425-429`
(`team_winback_v1`, `team_winback_v2`, `team_seasonal_v1` — names that do NOT
exist in `config/twilio_templates.yaml`) with a real **approved-templates
registry module** sourced from the two existing source-of-truth files:

- `apps/team-orchestrator/config/twilio_templates.yaml` — runtime name -> content_sid (8 templates, all currently `en`)
- `.viabe/templates.md` — human-readable variable signatures + bodies (Cowork-managed)

### The registry key MUST be `(template_name, language[en|hi]) -> content_sid`

Fazal confirmed Twilio issues a **separate** content_sid per language variant of
the same template. The current yaml is flat (`name -> {content_sid, audience}`),
implicitly `en`. This row introduces the language dimension so Fazal's incoming
Hindi SIDs drop in as **DATA ONLY** (a yaml edit), with **zero code change**.

### What lands

1. **New module** `apps/team-orchestrator/src/orchestrator/templates_registry.py`:
   - Single resolver `resolve(template_name, language) -> TemplateEntry` (or typed error).
   - Each `TemplateEntry` carries: `content_sid`, `language`, `audience`, and the
     **variable signature** (ordered positional `{{n}}` -> required param name),
     sourced from `.viabe/templates.md` and carried in the yaml so it is
     machine-readable (the markdown is human-only).
   - `60s TTL cache` (timestamped, not bare `lru_cache`) so a yaml edit picks up
     within a minute without a process restart.
   - Typed errors: `UnknownTemplateError(name)`, `UnknownLanguageVariantError(name, lang)`,
     `VariableSignatureMismatchError(name, expected, got)` — all subclasses of a
     base `TemplateRegistryError(ValueError)`.
   - `validate_params(template_name, language, params)` — VT-45 will call this to
     reject param dicts whose keys don't match the carried variable signature.
   - `approved_template_names(language) -> tuple[str,...]` — feeds the serializer's
     "Available WhatsApp templates" block (replaces the inline tuple).

2. **yaml structure evolution** (`config/twilio_templates.yaml`) — additive, must
   NOT break the two existing consumers (`output_composer.load_twilio_templates`,
   `twilio_send._templates`). See plan.md "yaml shape" for the exact nested-vs-flat
   decision and back-compat shim.

3. **context_builder.py** — replace the inline `_PHASE1_APPROVED_TEMPLATES` tuple
   and the `serialize_bundle_for_prompt(..., templates_available=...)` default with
   a registry read (`approved_template_names("en")`). Keep the `templates_available`
   override seam intact (test relies on it).

4. **VT-45 readiness:** variable signatures carried per-template so VT-45's
   `send_whatsapp_template` validates `template_id`/`language`/`params` against the
   registry rather than free-string.

### Out of scope
- No DB migration (Migration: NONE — pure config + Python module).
- No Twilio send path change (that's VT-45).
- No real Hindi SIDs authored here — Fazal supplies those later as a yaml-only edit.

### Tests
- resolve known `name+lang -> content_sid`
- unknown template name -> `UnknownTemplateError`
- known name, missing language variant (e.g. `hi` before SIDs land) -> `UnknownLanguageVariantError`
- `validate_params` signature mismatch -> `VariableSignatureMismatchError`
- existing `output_composer` + `twilio_send` consumers still resolve (back-compat)

### Canary (Rule #15 — persistence/config load)
At boot/import, load + resolve EVERY referenced template in the yaml; fail-not-skip
if any referenced content_sid is missing/malformed (not `HX...`) or any variable
signature is absent. Surfaced as a callable `canary_load()` run in CI and at startup.
