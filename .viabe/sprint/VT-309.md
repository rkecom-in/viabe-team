# VT-309 — full L2 agent-decision emit coverage (~10 sites)

The immediate-next moat row after L2 (VT-66/67, PR #275 merged) and **before
VT-74**. Fazal's call: the moat ships FULL, not thin (Cowork 20260603T182500Z).
VT-66/67 shipped the L2 substrate + dual-projection of the 2 OVERLAPPING outbox
types (campaign_sent, attribution_closed) + retrieval + Composer-wire. This row
adds the ~10 NEW agent-decision/milestone event types that have no outbox
overlap, so they need their own emit sites on the agent path.

## Why its own PR (not bundled into #275)
Cross-cutting live-path surface — incl. `owner_message_received` on the LIVE
dispatch path (`runner.webhook_pipeline_run` → `dispatch_brain`). This is exactly
the VT-65 PR-2 lesson: a multi-site emit pass deserves its own plan-review +
per-site txn-boundary classification + STOP-and-flag, not a bundle into the
schema PR.

## Dependencies
- VT-66/67 Done (#275) — substrate + `record_episodic_event` + drain dual-projection.
- VT-65 Done — the `emit_kg_event`/outbox pattern + per-site boundary discipline.

## What to produce
Emit each of these L2 event types at its real decision point. Two valid
mechanisms per site (state which + why per site):
- **(A) outbox projection** — add the type to the `kg_events` vocab + emit via
  `emit_kg_event` inside the source txn; extend `_L2_PROJECTION` in `kg_emit.py`.
  Use when an L1 projection is ALSO wanted or the site already emits a kg_event.
- **(B) direct `record_episodic_event(conn=...)`** — append in the source write's
  txn. Use for L2-only milestones with no L1 meaning.

The ~10 sites (confirm exact list + boundary in the plan):
1. `campaign_proposed` — collapse / plan-generation.
2. `campaign_approved` — owner-approval handler.
3. `campaign_rejected` — owner-rejection handler.
4. `owner_message_received` — **LIVE dispatch path, `dispatch_brain`. Extra care.**
5. `agent_dispatch_completed` — supervisor/handoff terminal-success.
6. `agent_dispatch_terminated` — supervisor terminal-failure / cancel.
7. `phase_transitioned` — phase machine.
8. `customer_dormant_threshold_crossed` — dormancy detector (feeds `_build_ledger_summary`).
9. `customer_high_value_threshold_crossed` — high-value detector (feeds `top_spenders`).
10. `clarification_resolved` — clarification loop resolution.

## Architectural rules
- **Per-site txn-boundary classification (req from VT-65 PR-2 / Cowork 20260603T181500Z):**
  for EACH site, confirm the emit can be atomic with what it records (shares a
  txn) OR is safely at-least-once via the outbox. **If any site autocommits /
  has no clean boundary, STOP and flag before wiring** — do not emit non-atomically.
  State the per-site boundary table in the PR.
- Summary templates already exist in `l2_types.py` for threshold/dispatch types;
  add any missing ones (PII-free, NO LLM — CL-390).
- Payloads carry NO raw PII; customer refs use `hash_phone` / entity uuids.
- Plan-first → Cowork plan-review BEFORE the PR opens (CL-424 guardrail 3).

## Acceptance criteria
- All ~10 types emitted at their decision points; per-site boundary table in PR.
- `owner_message_received` proven atomic on the live dispatch path (canary).
- Threshold detectors populate `_build_ledger_summary` end-to-end (canary:
  emit → drain/record → `recent_events` + ledger non-empty).
- Tenant-scoped; idempotent (no double-emit on retry/re-drain).
- Throwaway-DB validation + ruff + full pre-push suite green. CL-422 synthetic.

## Out of scope
- L2 retention soft-delete + 100K perf test → **VT-311**.
- L3 pattern extraction (VT-68/69).
