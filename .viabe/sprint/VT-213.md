# VT-213 — VT-208 PR-2: real Shopify webhook manual walk

## Why this row exists

VT-208 PR-1 shipped the Shopify connector substrate (OAuth code paths + checkout/order webhook handlers + HMAC verify + 5/5 canary). But canary asserted structural correctness against canned webhook payloads + canned API responses — real Shopify store round-trip was deferred.

This row closes the gap with a Fazal-driven manual walkthrough using a Shopify development store (or RKeCom's actual storefront once Fazal authorizes).

## Scope

- Manual test session with Fazal driving the OAuth flow
- Create or use existing Shopify dev store
- Configure webhook subscriptions (checkouts/create, checkouts/update, orders/create, orders/paid) to point at viabe.ai/team/webhooks/shopify
- Trigger a real abandoned-checkout event by:
  1. Adding item to cart
  2. Entering email at checkout
  3. Abandoning the page
  4. Verifying webhook hits orchestrator + HMAC verifies + customer row lands tagged `acquired_via='shopify'`
- Trigger a real order event (test order with Bogus Gateway)
- Document any code-path issues caught during real walk
- Fix bugs surfaced (if any)
- Re-run canary post-fix
- Update VT-208 status to Done after successful walk

## Acceptance criteria

1. Shopify dev store install flow grants OAuth → access_token stored encrypted in `tenant_oauth_tokens`
2. Webhook subscriptions registered programmatically via Shopify Admin API
3. Real `checkouts/update` webhook with abandoned cart → orchestrator endpoint → HMAC verifies → customer + event lands in DB tagged `acquired_via='shopify'`
4. Real `orders/create` webhook → customer lands + order persisted (if order schema present)
5. `pull_sample` against real store returns recent abandoned checkouts via Admin API
6. Re-auth flow works when access_token revoked (simulated by uninstalling app)

## Rule #15 Canary

No new canary file. Re-runs `vt208_shopify_connector.py` post-fix + manual verification log appended to `.viabe/queue/VT-213/task_log.md`.
