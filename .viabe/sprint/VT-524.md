# VT-524 — B1 owner-notification delivery ledger (part 1)

## Gap (from VT-519/VT-520)
The owner welcome captured a Twilio `message_sid` (SendResult) at send time but **nothing persisted the async delivery result** — a welcome could be `accepted` yet silently never delivered (exactly Meta's 63049 case). The status-callback webhook already classifies delivered/failed but wrote to no ledger.

## Built (mig 150 + wiring)
- **`owner_notifications` table (mig 150)** keyed by `message_sid`: `owner_notification_status {not_required|pending|accepted|delivered|failed}` + `communication_status {delivered|failed_incident_open}` + timestamps. Tenant RLS + FORCE; **registered in `dsr_purge._PURGE_ORDER` in the SAME PR** (VT-518 lesson) + erasure proven in `test_dsr_purge_substrate`. `run_id` is a soft NO-FK pointer (VT-521 lesson). No raw phone/body/name — only the opaque SID + template_name + status.
- **`owner_notification.py`** — `record_owner_notification` (send → `accepted`) + `record_owner_notification_delivery` (callback → delivered/failed, first-terminal-wins, no regression). Both via the RLS-bypassing service pool, **fail-soft** (never break the send or the webhook).
- **SEND wire** — `send_owner_template` records `accepted` on a successful send (with message_sid).
- **CALLBACK wire** — `runner.py` (before the L3 anchor / pre_filter, so 'delivered' isn't lost to the observability-only Reject) updates the ledger for every status-callback state, keyed by the outbound SID (= the callback's `MessageSid`).

## Verify
ruff clean; mig 150 applies; 19/19 green on fresh DB — accepted-on-send, delivered→delivered, undelivered→failed+failed_incident_open (the 63049 case), terminal-safe (later delivered can't overwrite failed), unknown-sid no-op, + owner_notifications DSR-erasure proof. Real-delivery E2E canary (send to +919321553267 → observe `delivered` on the ledger via the real callback) is queued behind Q2/Fazal — NO self-initiated send.
