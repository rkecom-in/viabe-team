# VT-190 — self_evaluate retrofit with @observability.tool_step

## Why this row exists

VT-181 (decorator) deferred the `self_evaluate` retrofit because `SelfEvaluateTool` is an `MCPTool` subclass with `execute(self, ctx, inputs)` signature that doesn't bind cleanly to `@tool_step` (decorator's `inspect.signature.bind` yields `{self, ctx, inputs}`, not the input model fields).

**Coverage gap is ZERO**: `sales_recovery._emit_self_evaluate_gate` already writes the `self_evaluate_gate` pipeline_steps row directly via VT-179's canonical rename merged in PR #70. The retrofit is uniformity-only — bringing self_evaluate under the same decorator-based pattern as `compose_output` so future tool authors don't need to know about the manual emit fallback.

## Scope

Two options documented in `self_evaluate.py` docstring (VT-181 fix-1):

**Option A:** Extract `execute` body to module-level `_self_evaluate_impl(inputs, model, client)` + decorate that.
- Pro: minimal decorator surface change
- Con: refactors MCPTool to a thin wrapper around module function

**Option B:** Extend `@tool_step` with `argument_picker` parameter that lets caller specify how to extract envelope fields from non-standard signatures.
- Pro: decorator gains general-purpose flexibility
- Con: more complex decorator implementation; may attract more exceptional tool shapes

CC decides at STEP-0; surface as plan-ready Q if it affects scope estimates.
