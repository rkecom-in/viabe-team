## What CoderC must produce
Source-of-truth banners on the 4 canonical documents (plus any others Phase 1 creates), making the documentation hierarchy explicit. Each banner declares: (a) what this document is authoritative for, (b) what it is NOT authoritative for, (c) hierarchy relative to other canonical docs, (d) update protocol.
1. **Banner format**:
	```markdown
> **📖 Canonical document**
>
> This document is the authoritative source for: [what].
> This document is NOT authoritative for: [what; with redirects].
> Source-of-truth hierarchy: [where this fits].
> Update protocol: [who can update, what review].
> Last reviewed: [date]; Next review: [date].
	```
2. **Documents to banner**:
	- [**concept-team.md**](http://concept-team.md) — authoritative for Phase 1 product strategy + scope; NOT for engineering details (redirect to Technical Reference) or operational procedures (redirect to runbooks). Hierarchy: highest-level product doc; supersedes ad-hoc product notes.
	- [**concept-team-pillars.md**](http://concept-team-pillars.md) — authoritative for the 8 inviolable Pillars; NOT for implementation details. Hierarchy: alongside [concept-team.md](http://concept-team.md); both are highest-level. Pillar changes = Type 3 board governance.
	- **docs/Viabe_Team_Technical_Reference_v1_**[**0.md**](http://0.md) (VT-14.1) — authoritative for Team engineering architecture + schema + contracts. NOT for product strategy (redirect to [concept-team.md](http://concept-team.md)) or Pillar definitions (redirect to [concept-team-pillars.md](http://concept-team-pillars.md)). Hierarchy: under [concept-team.md](http://concept-team.md) + [concept-team-pillars.md](http://concept-team-pillars.md); supersedes sub-component READMEs.
	- **docs/adr/** (VT-14.2) — authoritative for architectural decision rationale. NOT for current state (redirect to Technical Reference). Hierarchy: under Technical Reference; ADRs are why, Reference is what.
	- **docs/runbooks/** (VT-14.3) — authoritative for operational procedures. NOT for architectural decisions (redirect to ADRs). Hierarchy: parallel to Technical Reference.
	- **Cross-product banners** on Reports' docs: `Viabe_AI_Technical_Reference_v2_0.md` + `viabe_pipeline_intelligence_architecture_v1.md` get an addendum banner clarifying they are Reports-product-specific and Team has its own equivalents (this Subtask coordinates with Reports' co-maintainer Clau).
3. **Documentation hierarchy diagram** at `docs/documentation-hierarchy.md`:
	- ASCII tree of canonical documents with authority scopes.
	- Cross-product perspective: Reports' docs vs Team's docs vs shared docs ([concept-team-pillars.md](http://concept-team-pillars.md) is Team-specific; Reports has its own pillars in [CLAUDE.md](http://CLAUDE.md) if produced).
	- Update cadence per document.
4. **CodeX shadow-doc detection**:
	- Lint rule that scans for new `.md` files with architecture/strategy/procedure content NOT in the canonical paths.
	- Rule name: `no-shadow-canonical-docs`.
	- Exceptions: README files in component directories (acceptable; must defer to canonical via banner reference).
5. **Review cadence**:
	- Quarterly: Fazal + CoderC + CoderX skim every banner; verify update dates current; identify drift.
	- Annual: full review of canonical docs against actual production system.
## Architectural rules to enforce in this subtask
- Pillar 8: explicit hierarchy. CodeX rejects PRs that create competing canonical docs.
- Banners updated on every PR that materially changes the document.
- `Last reviewed` date kept current; stale dates flag in CI as warning.
- Cross-product banners coordinated with Reports' maintainer (Clau).
- Shadow-doc lint rule prevents drift.
