# VT-140 — Seeded-tenant end-to-end SR-agent canary (Sprint 1+2 E2E harness)

The capstone of the Sprint 1+2 SR-agent loop. Seed a SYNTHETIC tenant and drive
the full owner-facing recovery loop end to end, asserting per-stage state, zero
cross-tenant leakage, recorded dry-run sends, and computed attribution. This is
the LAST row to build in the e2e-batch (depends on Wave A + Wave B + the already-
merged standalone tools).

## Scope

Build a single end-to-end harness (canary + CI test) that:

1. **Seeds a SYNTHETIC tenant** (CL-422 — synthetic only, no real customer data):
   - `tenants` row (business profile: business_name, plan_tier, phase, whatsapp_number).
   - `customers` rows (VT-170 / migration 045) — a small cohort with synthetic
     display_name + phone_e164 + last_inbound_at.
   - a `campaigns` row + `campaign_recipients` linkage (VT-170 cohort integrity)
     OR let the loop create the campaign via collapse; harness seeds the customer
     cohort the SR agent will target.
   - a recent inbound (`twilio_inbound_events` / pipeline_runs trigger) to drive
     the loop.

2. **Drives the full loop** (one synthetic run, thread_id == run_id):
   inbound -> orchestrator (`dispatch_brain`) -> supervisor graph
   (`build_supervisor_graph`) -> SR agent (`run_sales_recovery_agent`) -> tools
   (query_customer_ledger / get_business_profile / get_recent_campaigns /
   get_attribution_data) -> CampaignPlan -> `request_owner_approval` (VT-47, PAUSE
   via checkpointer) -> simulate owner YES (resume) -> `send_whatsapp_template`
   (VT-45, DRY-RUN via `TEAM_TWILIO_MOCK_MODE=1`) -> `match_transactions` ->
   `get_attribution_data`.

3. **Asserts**:
   - per-stage state (pipeline_runs.status, pipeline_steps step_kinds present per
     phase, pending_approvals pending->resolved, campaign persisted, send recorded
     in `send_idempotency_keys` / `campaign_messages` with mock SID).
   - ZERO cross-tenant leakage (seed a 2nd decoy tenant; assert RLS-scoped reads
     never surface the decoy's customers/campaigns/attributions).
   - dry-run sends recorded (mock SID, send_status, customer_id; CL-390 — SID +
     tenant only, never phone/body).
   - attribution computed (`get_attribution_data` returns a snapshot for the run's
     campaign).

4. **Two run modes**:
   - **CI mode (default)**: mock Twilio (`TEAM_TWILIO_MOCK_MODE=1`) + mock Anthropic
     (monkeypatch `sales_recovery._run_one_turn` / `Anthropic` per the
     test_supervisor.py + test_sales_recovery_end_to_end.py pattern). Deterministic,
     no real API spend, no network.
   - **Live mode (behind env flag)**: `RUN_E2E_LIVE=1` + `ANTHROPIC_API_KEY` +
     `DATABASE_URL` -> real Opus dispatch + real DB; Twilio STILL `TEAM_TWILIO_MOCK_MODE=1`
     (no real WhatsApp send even in live mode — dry-run is the contract, Rule #15
     canary verifies the SEND PATH records, not a real message to a customer).

## Wiring prerequisite (NOT yet on main — see Build plan / Risks)

The SR agent loop today runs with `tools: dict[str, Any] = {}` (sales_recovery.py
L447) — the standalone MCP tools are NOT invoked by the agent. The collapse path
persists the CampaignPlan but does NOT send templates or gate on owner approval.
VT-45 (send_whatsapp_template) and VT-47 (request_owner_approval) are NOT built
yet, and the tool-wiring into the SR loop is a separate seam. VT-140 is the
HARNESS — it does NOT build the wiring; it depends on Wave A+B landing the wiring.
See plan.md "Dependency chain" for the exact ordering and what VT-140 asserts vs
what it assumes already exists.
