# VT-679 — §7A proactive planning depth (design-first)

## The gap (objective O4)
The Manager reacts to inbound turns and dispatches from plans it's HANDED (Gap-4 generation exists,
business_plan store + items_for_agent consume exists) — but it does not itself PLAN on a cadence
(monthly refresh / daily initiative) or INITIATE work from the plan unprompted via scheduled
triggers. Owner-visible plan exists (business_plan delivery); the PROACTIVE loop doesn't.

## Design questions (brief to write BEFORE build)
1. Cadence trigger seam: scheduled_triggers.py precedent (cross-tenant sweep → per-tenant work) vs
   a DBOS scheduled workflow per tenant. Cost control: LLM planning per tenant per day = metered
   (VT-619 budget rails apply).
2. Initiation path: plan item (accepted, owning_agent=X, scheduled) → start_manager_task_workflow —
   reuse the EXISTING dispatch machinery; the new piece is only the trigger + item-selection rule.
3. Owner-comms: a proactive initiative MUST surface to the owner before/with action (approval rails
   unchanged — nothing sends without the arm path).
4. Specialists read the plan via read_active_plan (VT-673 — already built, the enabler).
5. Guardrails: never initiate money-path work autonomously beyond arm-for-approval; daily initiative
   cap; quiet-hours.

## Gate
Design brief ratified (Cowork relay) → build → unit + dev-drive proof of one full proactive cycle
(plan item → initiative → owner surfaced → armed approval) → ×3 full-pack re-drive (brain-touch).
