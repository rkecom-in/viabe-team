## What CoderC must produce
The founding-tier counter: a Postgres-backed atomic counter that survives concurrent sign-ups without overcounting. Strict cap of 100. Once a tenant is assigned `plan_tier='founding'`, that pricing is theirs forever (locked into their Razorpay subscription).
1. **Schema** at migration `036_founding_tier_counter.sql`:
	- Table `founding_tier_counter` with a single row (sentinel pattern):
		- `id` INT PK CHECK(id = 1) -- enforces exactly one row
		- `claimed_count` INT NOT NULL DEFAULT 0 CHECK(claimed_count >= 0 AND claimed_count <= 100)
		- `cap` INT NOT NULL DEFAULT 100
		- `last_claimed_at` TIMESTAMPTZ
		- `last_released_at` TIMESTAMPTZ NULL
	- Initial INSERT in migration: `INSERT INTO founding_tier_counter (id, claimed_count, cap) VALUES (1, 0, 100);`
	- Companion table `founding_tier_claims` for audit:
		- `id` UUID PK
		- `tenant_id` UUID NOT NULL UNIQUE -- one founding slot per tenant max
		- `claimed_at` TIMESTAMPTZ
		- `released_at` TIMESTAMPTZ NULL -- set when tenant cancels/refunds; slot returned to pool
		- INDEX on tenant_id
	- Cap of 100 hardcoded in CHECK constraint. Raising the cap requires Type 3 governance (it's a public commitment).
2. **Atomic claim API** at `apps/team/src/billing/founding_counter.py`:
	- `try_claim_founding_slot(tenant_id) -> ClaimResult`:
		- Single SQL: `UPDATE founding_tier_counter SET claimed_count = claimed_count + 1, last_claimed_at = now() WHERE id = 1 AND claimed_count < cap RETURNING claimed_count`. The `claimed_count < cap` predicate makes the increment atomic at the row-locking level.
		- If RETURNING produces a row: claim succeeded. Insert audit row in `founding_tier_claims`.
		- If RETURNING is empty: cap reached or row not found.
	- `release_founding_slot(tenant_id)`:
		- On tenant cancellation/refund (called from VT-9.5 refund flow OR VT-3.2 transition to cancelled): `UPDATE founding_tier_counter SET claimed_count = claimed_count - 1 WHERE id = 1 AND claimed_count > 0`; set `released_at` on the audit row.
		- **Decision: do NOT release slots back to pool.** Once a founding slot is claimed, it stays counted even if the tenant churns. This prevents reopening founding tier later (which would be unfair to original founding tenants). Document this in `docs/team/founding-tier-policy.md`. The `released_at` column exists for audit only; the counter does NOT decrement.
	- `get_founding_status() -> FoundingStatus`: returns `{claimed_count, remaining, cap, all_claimed: bool, public_count_for_widget: int}`. The `public_count_for_widget` is `min(claimed_count, cap)` for landing-site display.
3. **Race safety verification**:
	- The `UPDATE ... WHERE claimed_count < cap` is atomic by virtue of Postgres row-level locking. Two concurrent transactions both attempting to claim slot #100 will serialize; one wins, one returns empty.
	- A test specifically for this: spawn 50 concurrent `try_claim_founding_slot` calls when 99 slots are already claimed. Exactly 1 succeeds; 49 return cap-reached. The `claimed_count` is exactly 100 after the test. CodeX MUST review this test specifically.
4. **Public read API**:
	- GET `/api/team/founding-status` returns `{remaining, cap, public_count}`. No auth required (public landing-site widget consumes this).
	- Cached at the CDN layer for 60 seconds (the count is approximate for display; absolute correctness only matters at claim time).
	- Rate-limited (Cloudflare): 100 req/sec per IP to prevent enumeration / scraping abuse.
5. **Sign-up integration** (VT-9.2 update):
	- VT-9.2 sign-up flow first calls `try_claim_founding_slot(tenant_id)`.
	- On claim success: `tenants.plan_tier = 'founding'`.
	- On claim failure (cap reached): `tenants.plan_tier = 'standard'`.
	- This MUST happen inside the same transaction as tenant creation, so we don't claim a slot for a tenant whose creation later rolls back.
6. **Razorpay plan-tier persistence**:
	- When the tenant subscribes (VT-10.1), the `plan_tier='founding'` resolves to `FOUNDING_RZP_PLAN_ID` (lower-priced plan).
	- The Razorpay subscription is locked at this plan price; future Razorpay plan price increases (if any) do NOT affect existing subscriptions.
	- This is the structural enforcement of "founding pricing forever."
7. **Tests** at `apps/team/tests/billing/test_founding_counter.py`:
	- Sequential claim: 100 sequential claims succeed; 101st fails.
	- Concurrent race (the load-bearing test): 50 concurrent claims at 99/100 → exactly 1 succeeds.
	- Idempotency: same tenant_id calling `try_claim_founding_slot` twice — second call fails with `already_claimed` (UNIQUE constraint on `founding_tier_claims.tenant_id`).
	- No-decrement-on-release: claim 100, release 5 → `claimed_count` stays 100; counter does NOT reopen.
	- Public widget count: returns capped value for display.
	- Public API rate limit: 101st request from same IP in same second is 429.
	- Sign-up integration: tenant with claim succeeds gets plan_tier='founding'; tenant after cap gets plan_tier='standard'.
	- Cap immutability: any attempt to UPDATE `cap` to a different value rejected by RLS DENY UPDATE on cap column (or via service-role-only path).
## Architectural rules to enforce in this subtask
- Pillar 7: 100-tenant cap is Type 3. CodeX rejects PRs raising the cap. The CHECK constraint enforces the upper bound; the RLS DENY UPDATE on cap column enforces immutability via app paths.
- Pillar 8: one claim primitive. CodeX rejects parallel founding-tier logic (e.g., a manual slot grant by Fazal would go through the same primitive with an audit reason).
- No-decrement-on-release: documented + tested. Stops the slot count from reopening which would unfairly create founding tier 2.0.
- Atomic claim via single UPDATE-with-WHERE-cap. CodeX rejects PRs that read-then-write (race condition).
- Tests must include the concurrent-50 test specifically. CI runs this with appropriate concurrency.
- Public widget count is a read-only approximation; cached 60s; minor lag acceptable.
- Founding pricing forever is enforced by Razorpay subscription locking at the FOUNDING plan ID; CodeX confirms VT-10.1's plan resolution honors `plan_tier`.
