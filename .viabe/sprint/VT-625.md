# VT-625 — a 1000-char storage cap was truncating the manager's own memory AND the eval

`conversation_log._TEXT_CAP = 1000` truncates every stored turn at 1000 chars (`record_turn`, :74). That
same capped column is read by BOTH:
- **`active_window()` (:118)** — the manager's always-on brain context. So when the manager sent a >1000-char
  reply, its NEXT-turn context showed its own prior reply **cut mid-sentence** → real context-loss → a direct
  progression/repeat contributor (ties [[VT-621]]). The owner received the full reply; the brain didn't remember it.
- **the eval harness (`convo_harness._new_conversation_turns`)** — so the judge saw truncated replies and
  docked "truncated mid-sentence" on progression/helpfulness for a **logging cap, not a real defect**.

## Evidence (measured on the deployed-dev bundles)
Replies capped at EXACTLY 1000 chars, cut mid-word: `happy_path_profile_collection` step3 (`…wasting mone`),
`hinglish_conversation` step0 (`…aap unnece`), `i_sheets` step1. Verbose romanized Hinglish (more chars/concept)
takes the biggest hit. Both root-cause agents flagged it independently as the top pack-wide lever.

## Fix
`_TEXT_CAP` 1000 → **4096** (one WhatsApp message's max — a single turn can't exceed it anyway; genuine
runaway past one message still bounded). `test_vt616_antirepeat` fixture bumped >4096 so the guard's
truncation path stays exercised (19 tests green). The [[VT-616]] common-prefix guard still applies for the
rare >4096 reply.

Also folded in (zero-risk eval-honesty): `i_sheets_hinglish_connect_and_resume` step2 assert de-brittled —
was an exact Hindi phrase `"nahi dikh raha"` on a non-deterministic LLM reply (manager said synonym
`"nahi ho raha"` + gave the real link → false-fail); now asserts `"http"` (the durable honest-reply-carries-
link invariant).

## Why runtime, not just measurement
The cap fix must DEPLOY + the scenarios RE-RUN to see the lift — the existing bundles' transcripts are already
stored capped-at-1000 (ran against old code). Re-judging them alone won't move; the brain also needs the deploy
to start seeing full priors. Verification = targeted re-run of the truncation-hit scenarios post-deploy.
