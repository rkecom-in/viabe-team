# CL-64 — Discipline rule: Context7 docs occasionally lag

During VT-3.1 implementation, Context7 docs for DBOS specified DBOS.SetWorkflowID and DBOS.SetWorkflowTimeout as class attributes. These do not exist as class attributes — they are module-level imports (from dbos import SetWorkflowID, SetWorkflowTimeout).

Claude Code caught the discrepancy in CI iteration (test failed, error surfaced the import issue), patched, re-pushed, green.

Discipline rule going forward:
- Context7 is the FIRST stop for library docs (efficient, comprehensive)
- When Context7 docs and implementation reality disagree (CI failure, runtime error), the live source wins:
  - PyPI package itself (pip show / inspecting installed code)
  - GitHub source repository at the pinned version
  - Library's own current changelog
- Don't keep guessing against Context7 if implementation contradicts it. Drop to source.

This is not Context7 criticism — it's a known limitation of any docs-index system. Notion-as-source-of-truth has the same property: read the actual subtask body, don't trust summaries.
