## What CoderC must produce
The LangGraph substrate wrapped in DBOS workflows. NOT a state machine — the SUBSTRATE on which the orchestrator-agent (Opus 4.7, spec'd in VT-3.9) and specialist agents run. Every pipeline run is a `@DBOS.workflow`; every state transition inside is a `@DBOS.step`. DBOS auto-resumes on Railway crash from the last completed step.
1. **Install dependencies** in `apps/team-orchestrator/pyproject.toml`:
	- `dbos>=2.0` (durable execution; pinned major version)
	- `langgraph>=0.3` (state graph substrate; pinned major version)
	- `langgraph-supervisor` (latest stable; auto-handoff tools)
	- Document pins in `docs/team/dependency-pins.md`.
2. **DBOS configuration** in `apps/team-orchestrator/src/dbos_config.py`:
	- `DBOS.start()` wired to Supabase Postgres (env: `SUPABASE_URL_DEV` / `_PROD`).
	- DBOS creates its own `dbos_workflows`, `dbos_workflow_steps`, `dbos_queues` tables on first start (idempotent).
	- Workflow timeout: 6 minutes (matches 5-min wall-clock hard limit + 1 min safety).
3. **LangGraph substrate** in `apps/team-orchestrator/src/orchestrator/graph.py`:
	- LangGraph compiled with `PostgresSaver` checkpointer using the same Supabase Postgres. DBOS handles durability + auto-resume; LangGraph checkpointer holds graph state inside a single workflow execution.
	- A `placeholder_node` for plumbing verification only — replaced by real nodes in VT-3.2 through VT-3.10.
	- Typed `OrchestratorState` TypedDict (will be expanded to `SubscriberState` in VT-3.2). For now: `tenant_id: UUID`, `run_id: UUID`, `history: list[str]`.
	- The graph is the substrate; the orchestrator-agent runs ON it. Reasoning lives in the agent (VT-3.9), not in graph nodes.
4. **Workflow entry** in `apps/team-orchestrator/src/orchestrator/runner.py`:
	- `@DBOS.workflow()` decorated function `pipeline_run(tenant_id, run_id, input) -> RunResult`.
	- Inside the workflow: `@DBOS.step` decorated calls that invoke the LangGraph compiled graph.
	- DBOS handles: auto-resume on crash, idempotency by workflow_id, exactly-once semantics for steps.
	- LangGraph handles: graph traversal, conditional routing within a workflow, state mutation.
	- thread_id for LangGraph checkpoint == run_id for joinability with `pipeline_steps` (from VT-122).
5. **Smoke tests** in `apps/team-orchestrator/tests/orchestrator/test_skeleton.py`:
	- Synthetic invocation: `pipeline_run(tenant_id, run_id, input)` returns expected state.
	- **DBOS auto-resume test (CRITICAL):** invoke workflow, kill Python process mid-execution (signal SIGKILL after first DBOS step completes), restart process, verify DBOS auto-resumes the workflow from the next step. Assert final state is correct.
	- **Tenant isolation test:** workflow invocation with `tenant_id_A` cannot read checkpoint state belonging to `tenant_id_B` via direct DB query. RLS enforced.
	- **Idempotency test:** invoking `pipeline_run` twice with the same `run_id` triggers DBOS's idempotency — second invocation returns the first's result without re-executing.

## Architectural rules to enforce in this subtask
- **Pillar 1 (revised):** the LangGraph substrate contains NO LLM calls, NO reasoning, NO tool invocations. Reasoning lives in the orchestrator-agent (VT-3.9) and specialists (VT-4). Graph nodes only route and persist state.
- **Pillar 3 (tenant isolation):** LangGraph checkpoint tables get `tenant_id` and RLS policies. No retrofit.
- **Pillar 8 (no patchwork):** DBOS + LangGraph + langgraph_supervisor versions pinned. Major-version updates require Type 2 governance.
- DBOS is the durability substrate. NO custom retry/recovery logic at the graph level — DBOS handles that.
- The PostgresSaver instance is module-level, reused across invocations. No per-call connection establishment.
- `OrchestratorState` TypedDict is the single source of truth for state shape. Any new field requires TypedDict update; ad-hoc dict access forbidden.

## Acceptance criteria
- DBOS + LangGraph + langgraph_supervisor installed with pinned versions; pins documented.
- DBOS workflow tables auto-created on first start (verify via `\d dbos_*` in psql).

- **DBOS auto-resume test passes:** synthetic crash mid-workflow auto-recovers from last completed step. This is the load-bearing acceptance check.
- Tenant isolation test passes: cross-tenant checkpoint reads blocked at RLS.
- Idempotency test passes: same run_id invoked twice returns same result without re-execution.
- No LLM calls, no tool invocations, no reasoning in the skeleton.
- mypy --strict passes.
