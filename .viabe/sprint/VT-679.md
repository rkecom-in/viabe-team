# VT-679 — §7A proactive planning depth (design-first)

## The gap (objective O4)
The Manager reacts to inbound turns and dispatches from plans it's HANDED (Gap-4 generation exists,
business_plan store + items_for_agent consume exists) — but it does not itself PLAN on a cadence
(monthly refresh / daily initiative) or INITIATE work from the plan unprompted via scheduled
triggers. Owner-visible plan exists (business_plan delivery); the PROACTIVE loop doesn't.

## Design questions (brief to write BEFORE build)
1. Cadence trigger seam: scheduled_triggers.py precedent (cross-tenant sweep → per-tenant work) vs
   a DBOS scheduled workflow per tenant. Cost control: LLM planning per tenant per day = metered
   (VT-619 budget rails apply).
2. Initiation path: plan item (accepted, owning_agent=X, scheduled) → start_manager_task_workflow —
   reuse the EXISTING dispatch machinery; the new piece is only the trigger + item-selection rule.
3. Owner-comms: a proactive initiative MUST surface to the owner before/with action (approval rails
   unchanged — nothing sends without the arm path).
4. Specialists read the plan via read_active_plan (VT-673 — already built, the enabler).
5. Guardrails: never initiate money-path work autonomously beyond arm-for-approval; daily initiative
   cap; quiet-hours.

## Gate
Design brief ratified (Cowork relay) → build → unit + dev-drive proof of one full proactive cycle
(plan item → initiative → owner surfaced → armed approval) → ×3 full-pack re-drive (brain-touch).

## DESIGN BRIEF (CC, 2026-07-18 — seam-grounded, for ratification)

### D1. Cadence trigger — TWO triggers, monthly_impact template
Both registered in `scheduled_triggers.py` via the existing `_register_scheduled` mechanism
(`DBOS.scheduled(cron)(DBOS.workflow()(fn))`, register-before-launch), each following the
`monthly_impact_scheduled` shape: workspace SQL scan → per-tenant loop → per-tenant work, every
iteration try/except so one tenant never halts the sweep.

1. **`plan_refresh_scheduled`** — monthly, 1st 09:00 IST (`30 3 1 * *` UTC). Re-grounds each
   active paid/trial tenant's business_plan: calls a NEW `refresh_business_plan_workflow` thin
   wrapper around the existing generator path but via `store.write_new_version` (append-only
   versioning already built) — the generator's `plan_exists` idempotency-skip is BYPASSED for
   refresh (that skip is the one-shot onboarding rail, not a refresh rail). Delivery of the
   refreshed plan reuses `delivery.deliver_plan` (paced parts, replay-safe bitmap). LLM cost:
   generation already runs through the metered generator seam; per-tenant per-month = bounded.
2. **`daily_initiative_scheduled`** — daily 10:30 IST (`0 5 * * *` UTC — inside owner waking
   hours BY CRON PLACEMENT; scout confirmed no runtime quiet-hours helper exists anywhere, and
   inventing a general one is out of scope. Belt: a one-line IST-hour assert (10:00–19:00) at
   the top of the handler so a mis-set cron can't fire at night).

### D2. Initiation path — deterministic selection, existing dispatch machinery
Per tenant, the daily trigger:
1. `items_for_agent(tenant, agent, statuses=("accepted",))` across `OWNING_AGENTS` (seams.py —
   already IDOR-safe) → candidate items.
2. **Deterministic selection rule (NO LLM):** first `accepted` item, plan-order, whose
   idempotency key has no existing task. Selection is code, not judgment — per the
   LLM-for-intent/deterministic-for-outcomes standing.
3. Create the task via `plan_store.create_plan` with objective from the item;
   `idempotency_key = f"plan-item:{item_id}:{YYYYMM}"` — month-scoped once-ness on the existing
   `manager_tasks_idem` unique index; redelivery/re-fire = structural no-op.
4. `start_manager_task_workflow(tenant, task)` — the EXISTING reactive machinery (workflow id
   already deterministic per (tenant, task)); triage_seam untouched. The new code is ONLY the
   trigger + selection + task-mint. `report_item_status(... "in_progress")` marks the item.

### D3. Owner comms — surface BEFORE effect, rails unchanged
A proactive initiative's FIRST owner-visible act is a surface message ("From your business plan,
I'm starting X — I'll bring you anything that needs your approval."), sent via the existing
owner-notification path. Approval rails untouched: any money step still lands at arm-for-approval
(chokepoints unchanged); nothing sends without the arm path. No silent background effects.

### D4. Guardrails (deterministic, all v1)
- **Daily cap: 1 initiative/tenant/day** (the idempotency key + one-selection rule enforce it).
- **Back-pressure skip:** tenant skipped when it already has a task in
  `('running','waiting_owner','verifying','clarifying','planned')` — never pile proactive work on
  a busy manager.
- **Money floor:** unchanged — proactive work may ARM approvals, never resolve them.
- **Flag-gated:** `TEAM_PROACTIVE_PLANNING` (dev ON, prod unset/Fazal — VT-101 rollout pattern).
- Metering: any LLM in the refresh path already attributes via the generator's seam; the daily
  initiative path itself is LLM-free.

### D5. Build/test plan
- Unit: selection rule (order, skip-busy, idempotency key), refresh bypass-vs-onboarding-skip,
  IST-hour belt. Trigger handlers manually invocable (same as existing sweeps) for dev drive.
- Dev drive: seed accepted plan item → invoke trigger → task minted → owner surface message →
  campaign path arms approval, ZERO sends. Then ×3 full-pack re-drive (per the grant).

### Open point for ratification (only one)
- Initiative surface copy + whether the monthly refresh message re-delivers the FULL plan or a
  delta summary (v1 proposal: delta summary line + on-request full plan; full re-delivery is a
  paced multi-part burst and monthly full-bursts feel spammy).
