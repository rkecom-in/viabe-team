1. **`viabe-team-dev`** — region: ap-south-1 (Mumbai). Postgres 15 + pgvector + Apache AGE extension enabled (AGE may need a support ticket; track timing). Daily snapshots configured with 7-day retention. Point-in-time recovery enabled.

5. **Backup verification runbook** at `docs/runbooks/supabase-backup-restore.md`: documents how to restore from snapshot, how to verify PITR is working, what the recovery point objective (RPO) and recovery time objective (RTO) are. Drilled once on dev (restore a snapshot to a temporary project; document the time taken).
6. **Data residency verification**: confirm via Supabase dashboard that both projects are in their declared regions (ap-south-1, ap-south-2). Document the verification with screenshots in the PR description.

## Architectural rules to enforce in this subtask
- Pillar 8 (no patchwork): credentials only via env vars + 1Password + cloud env settings. Zero secrets in git. CI's `gitleaks` (added in VT-2.7) will catch violations; this subtask must not depend on that catch.
- Pillar 3 (tenant isolation, structural): the `viabe-team-dev` and `viabe-team-prod` projects are SEPARATE Supabase projects, not schemas in a shared project. Strict isolation between dev and prod is non-negotiable.
- Supabase key naming is locked: `SUPABASE_PUBLISHABLE_KEY` (read-only, used by web frontend), `SUPABASE_SECRET_KEY` (full access, used by orchestrator/agent server-side). Legacy names (`SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`) are forbidden in code, env files, and docs.
- Apache AGE extension is required for VT-KnowledgeArchitecture's L1 KG. If AGE is not yet available on Supabase managed Postgres at the time of this subtask, log a vendor-side blocker and proceed with the rest; AGE enablement becomes its own subtask under VT-7.
- pgvector is required for VT-Knowledge L2 episodic memory and L4 skill corpus.
- Backup runbook must be drilled, not just written. "We have a runbook" without a successful drill counts as Not Done.
