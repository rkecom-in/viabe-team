# VT-326 — signup OTP-before-create + per-IP throttle

The launch-blocking security fast-follow for the gap the VT-82 ultracode review found
(Cowork 20260605T063000Z). `/api/signup` is the sole, intentionally pre-auth owner-
acquisition front door with NO rate-limit + NO proof-of-control of whatsapp_number →
(a) unbounded tenant/consent-row creation on the BYPASSRLS pool (flooding); (b) number
SQUATTING — first-writer-wins on the unique whatsapp_number 409-bricks a number an
attacker doesn't own.

## Scope
1. **OTP-before-create**: wire `/api/signup` BEHIND the VT-250 owner-verify OTP — the
   owner must prove control of the whatsapp_number (number → OTP → submit) before
   create_signup_tenant runs. The VT-96 form already structures a phone-entry step for
   this gate to hook.
2. **Per-IP throttle**: a fixed-window rate limit on `/api/signup` (the existing
   rate_limit mechanisms are tenant_id-keyed → inapplicable pre-tenant; needs an
   IP/source-keyed limiter).
3. Remove the NEEDS-FAZAL marker + the public-exposure block on /api/signup + the VT-96
   form once both land.

## HARD CONSTRAINT (carried from VT-82/VT-96): /api/signup must NOT be registered on a
publicly-reachable route, and the VT-96 form must NOT go live, until this lands.

## Canary: real-PG — OTP gate blocks create without a verified number; throttle blocks
the (N+1)th request in the window; a verified+throttled-OK request creates normally.
