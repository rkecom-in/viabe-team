# VT-217 — /team/ops page.tsx Supabase resilience

## Why this row exists

VT-201 PR-3 surfaced that `/team/ops/page.tsx` (VT-123, shipped earlier) does `Promise.all([fetchWorkspaceCounters(), fetchTopTenants(10), fetchInFlightRuns(20)])` without try/catch. Any one of those failing → page 500s → Next.js error boundary → operator sees broken page instead of "Supabase unreachable, retry".

The `/team/ops/stream` page (VT-201 PR-1) added try/catch in commit `afeb9d0` during VT-201 PR-3's CI work. Apply the same defensive pattern to `/team/ops/page.tsx` so the resilience is uniform across all `/team/ops/*` entry points.

Surfaced by CC in VT-201 PR-3 task-result side-finding (2026-05-28 13:49 IST).

## Scope

- `apps/team-web/app/(app)/team/ops/page.tsx` — wrap the three `fetch*` calls in try/catch with fallback shapes (mirror what `afeb9d0` did on stream page).
- On fetch failure, render the page with "couldn't load: <reason>" notes per-section instead of letting the error bubble.
- Keep the banner (in layout) functioning even when this page's data fails — layout's banner already has its own try/catch from VT-201 PR-3.

## Acceptance criteria

1. With Supabase reachable: page renders identical to current behavior (no regression)
2. With Supabase URL forced to an invalid host (env override): page renders shell + per-section "couldn't load" notes; HTTP 200; no Next.js error boundary
3. Banner still renders correctly (zero counts via layout's fallback)
4. No regression on `/team/ops/stream` + `/team/ops/stream/history` + `/team/ops/runs/[runId]`

## Rule #15 Canary

Extend `apps/team-orchestrator/canaries/vt201_ops_stream.py` with assertion A13:
- Boot team-web with `NEXT_PUBLIC_SUPABASE_URL` pointing at `http://127.0.0.1:1` (unreachable); navigate to `/team/ops`; assert response is 200 + page body contains "couldn't load" placeholder text. Restore env in finally.

## Out of scope

- Other pages with similar Promise.all patterns (separate row if surfaced)
- Retry logic / exponential backoff
- Showing detailed Supabase error to operator (just "couldn't load" is enough; full error in browser devtools is sufficient for debug)
