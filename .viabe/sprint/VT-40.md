## What CoderC must produce
The `query_customer_ledger` MCP tool. Returns a paginated, segmented view of the tenant's customer ledger to the agent. Deterministic, no LLM, no business logic.
1. **Tool implementation** at `apps/team/src/agent/tools/query_customer_ledger.py`:
	```python
class QueryCustomerLedger(MCPTool):
    name = "query_customer_ledger"
    description = "Returns a paginated view of the tenant's customer ledger, optionally filtered by recency band, total-spend band, or acquired_via source. Returns customer IDs (anonymized for agent reasoning); phone numbers and names are resolved by send-time tools (send_whatsapp_message/template)."
    is_llm_backed = False
	```
	Input schema:
	- `recency_band: Literal['active_30d', 'dormant_30_60d', 'dormant_60_90d', 'dormant_90_180d', 'dormant_180_365d', 'dormant_365d_plus'] | None`
	- `spend_band: Literal['low', 'mid', 'high', 'top_decile'] | None`
	- `acquired_via: list[Literal['paper_book' | 'contacts' | 'upi_phonepe' | 'upi_gpay' | 'upi_paytm' | 'kot_pos' | 'cash_book' | 'qr_opt_in' | 'apify_zomato' | 'apify_swiggy' | 'apify_magicpin' | 'apify_gbp' | 'owner_typed']] | None`
	- `exclude_opted_out: bool = True` (default; agent typically wants only opt-in customers)
	- `limit: int` (max 200; default 50; the framework caps at 200 hard)
	- `cursor: str | None` (opaque pagination cursor)
	Output schema:
	- `customers: list[CustomerSummary]` ã each item has `customer_id` (UUID, anonymized for agent), `recency_band`, `spend_band`, `acquired_via`, `last_transaction_date`, `transaction_count`, `lifetime_spend_paise`. **NO phone, NO name, NO address.** Agent reasons on IDs; tools resolve at send time.
	- `total_count: int` (full count matching filter, before limit)
	- `next_cursor: str | None` (for pagination)
	- `summary: LedgerQuerySummary` ã `count_by_recency_band`, `count_by_spend_band`, `count_by_acquired_via` (aggregates over the FULL filter, useful for cohort sizing).
2. **Implementation logic**:
	- All filters combine with AND. Empty filters mean no constraint.
	- Recency calculation uses `last_transaction_date` against `now()`. Active = last 30 days; dormant_30_60d = last_transaction 30-60 days ago; etc.
	- Spend bands are tenant-relative percentiles, not absolute amounts: `low` = bottom 25%, `mid` = 25-75%, `high` = 75-95%, `top_decile` = top 10%. Computed per-tenant on each query (cheap because indexed; for large tenants may need precomputation ã deferred to Phase 2 if perf becomes an issue).
	- Pagination uses keyset pagination on `customer_id` (not offset; offset is slow at scale).
	- `exclude_opted_out=True` (default) filters `WHERE opt_out_status IS NULL`.
3. **Tenant scoping**:
	- `ToolContext.tenant_id` is passed to every query. RLS enforces at the DB level. Typed wrapper enforces at the application level.
	- Cross-tenant attack test: invoke the tool with `ctx.tenant_id = tenant_A` against a database containing tenant_B customers; assert zero leakage.
4. **Performance constraints**:
	- Hard timeout: 2 seconds. Tool returns `ToolResult(status='timeout')` if exceeded.
	- Hard limit: 200 customers per call (cap enforced before query). Higher cap requires Type 2 governance (cost: bigger results = more agent context = closer to 8K context cap).
	- Indexed query: `(tenant_id, last_transaction_date DESC)` index already exists from VT-2.3 schema.
5. **Tests** at `apps/team/tests/agent/tools/test_query_customer_ledger.py`:
	- Happy path: synthetic tenant with 100 customers across recency bands; filters return correct subsets.
	- Pagination: cursor-based forward iteration covers all matching customers exactly once.
	- Cross-tenant attack: ctx.tenant_id substitution attempt cannot leak.
	- Empty filters: returns all non-opted-out customers up to limit.
	- Opted-out filter: customers with `opt_out_status` set are excluded by default; included when `exclude_opted_out=False`.
	- Spend-band correctness: synthetic tenant with known spend distribution; `top_decile` returns exactly the top 10%.
	- PII absence: returned `CustomerSummary` objects contain no phone, no name, no address (regex check on serialized output).
	- Performance: synthetic tenant with 10K customers; query completes in < 500ms.

## Architectural rules to enforce in this subtask
- Pillar 2: zero LLM. The tool is pure SQL + Python aggregation.
- Pillar 3: `tenant_id` from context only. Three independent enforcement layers (RLS + typed wrapper + framework rejection of input field).
- Pillar 4: returns data and aggregates, not conclusions. CodeX rejects fields like `recommended_targets` or `cohort_quality_score` ã those are agent reasoning, not tool output.
- PII absence is structural. The tool's output schema does not include phone/name fields. Resolution happens at send time in VT-5.6/VT-5.7.
- Pagination is mandatory. Tool cannot return unbounded results. The 200-cap is a Phase 1 commitment; raising it requires Type 2.
- Spend-band percentiles are tenant-relative by design. "High spender" means high relative to this tenant's distribution, not high in absolute INR. CodeX confirms.
