# CL-56 — LangSmith replaced by Pydantic Logfire

LangSmith replaced by Pydantic Logfire for LLM observability + tracing.

Reason: LangSmith's per-trace pricing model ($39/seat/month + $0.50 per 1K traces) collapses under agentic workloads where one user action generates 20-50 spans. Industry data shows most production teams sample down to 0.1% of traffic to keep LangSmith costs manageable — i.e., discarding 99.9% of telemetry, which defeats the purpose of observability for an AI agent business.

Pydantic Logfire chosen over Langfuse self-hosted because:
1. Flat span-based pricing (~$2/million spans) aligns cost directly with workload, predictable at every scale
2. DBOS natively emits OpenTelemetry spans for every workflow and step — Logfire is the natural destination for those spans (Pydantic + DBOS published a joint blog post on the integration in Feb 2026)
3. Pydantic AI emits spans for agent runs, model requests, tool calls — unified Logfire trace covers the full execution path
4. No new infrastructure to manage (Langfuse self-hosted requires Postgres + ClickHouse + Redis + S3 + workers)
5. MCP server for natural-language queries over observability data

Integration pattern:
- apps/team-orchestrator: DBOS → OTel spans → Logfire
- Pydantic AI agents → native Logfire spans
- Custom application code → logfire.span("...") instrumentation
- Cost projection at scale: at 50M spans/month, Logfire is 40x cheaper than LangSmith equivalent

What we lose vs LangSmith: deep LangChain-specific auto-instrumentation. We mitigate this by using Logfire's general OTel support; LangGraph emits OTel spans we can capture. Less plug-and-play than LangSmith's LangChain integration, but full observability is preserved.
