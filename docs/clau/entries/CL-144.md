# CL-144 — Every new branch starts with four-command sequence

Triggered by PR #22 mis-branch incident (CL-143). Claude Code cut chore/dep-audit-recon off feat/vt-orchestrator-agent-skeleton instead of main because the working tree was still on that branch when `git checkout -b` ran.

STANDING RULE: Every new branch's first action is the four-command sequence:

```
git fetch origin
git checkout main
git pull origin main
git checkout -b <branch-name>
```

No exceptions. No judgment calls. Copy-pasteable. Even when 'I just made the last branch off main 30 seconds ago' — still do it. Cheap; the cost of a wrong-base branch is catastrophic when squash-merged (entire upstream branch's commits collapse into the new PR silently).

Brief template addition: every brief that asks Claude Code to create a new branch MUST start the implementation section with the four-command block above. Non-negotiable.

Defense-in-depth (deferred, not Phase 1 blocker): GitHub Actions check that fails any PR whose merge-base with main is older than 24h — would have caught PR #22's stale base (which was actually feat/vt-orchestrator-agent-skeleton's tip, not main's). File under VT-CI tech debt; not urgent because the four-command rule is the primary defense.

Tech debt entry will be added separately for the CI check.
