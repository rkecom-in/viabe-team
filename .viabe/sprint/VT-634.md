# VT-634 — prod handling of orphaned/failed DBOS workflows

Spec: `.viabe/prod-failed-workflow-handling-spec.md` (authoritative). Summary:
1. **CONTAIN** — prevent auto-re-run on recovery: quarantine/disable/archive; fail-safe default = an EFFECTFUL failed workflow is never auto-re-run.
2. **DIAGNOSE** — a separate process determines cause/tenant/step/effect-state.
3. **REPORT** — surface findings on the VTR console (extends VT-514/515/516) for VTR action.

## Hard part (from the spec)
A workflow that PARTIALLY sent then failed must NOT blindly re-run (double-send) NOR be silently disabled (half-done). Diagnosis reads effect-state FIRST; containment = stop-auto-re-run by default + human/VTR resolves.

## Not to conflate
Dev #53 (harness teardown cancels manager_task workflows before tenant delete, ce124e7-era commit) is HARNESS-specific. This row = a DBOS recovery POLICY + diagnosis + VTR reporting — a different mechanism. #53 does not cover it.
