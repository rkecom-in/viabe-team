# CL-334 — TECH DEBT (carry to VT-7.2 population pipeline): PR #44 ships l1_entities HNSW i

TECH DEBT (carry to VT-7.2 population pipeline): PR #44 ships l1_entities HNSW index (vector_cosine_ops) and search_entities using the <=> cosine operator — operator/opclass are consistent on inspection. But CI cannot prove the HNSW index is actually USED: pgvector silently falls back to exact-distance scan on an operator/opclass mismatch and still returns correct rows, and at PR #44's ~8 test rows the planner will never choose an ANN index regardless. Index-usage is therefore unverified by PR #44's green CI — not a defect, a coverage limit. ACTION at VT-7.2 (when l1_entities is populated): run EXPLAIN on the search_entities retrieval query against a populated table and confirm 'Index Scan using' the HNSW index, not a Seq Scan / exact scan. If it shows exact scan, investigate operator match and HNSW build params before relying on retrieval latency. NOT a PR #44 merge blocker.
