# VT-680 — §7C impact-judge (design-first)

## The gap (objective O4)
The delegation validate-return seam verifies COMPLETION (did the specialist produce the artifact;
VT-611 verify-floor: executed campaign must PASS verification) — but not QUALITY/IMPACT against the
step's defined outcome (manager_step_desired_outcome / acceptance_criteria already thread through
the framework context since VT-667 iter-2). A completed-but-poor outcome passes silently.

## Design questions (brief to write BEFORE build)
1. Judge seam: the validate-return point in the delegation loop (workflow.py verify leg) — add an
   LLM judge call (metered tier; VT-619) scoring result vs desired_outcome/acceptance_criteria.
2. Verdict handling: score below floor → retry-with-feedback (bounded, 1 retry) vs surface-to-owner
   honest partial vs accept-and-log. NEVER silently re-run money steps (idempotency + approval rails
   bind).
3. Logging: verdicts into tm_audit + VT-514 pipeline observability (reasoning_ref joinable — the
   §7D substrate exists).
4. Measurement interplay: the OFFLINE judge (tier_rescore) stays the gate; this is the ONLINE
   in-loop judge — keep rubrics aligned or the two disagree.

## Gate
Design brief ratified → build → unit + a dev-drive where a deliberately-poor specialist result gets
caught (fixture) + a good one passes → ×3 full-pack re-drive (brain-touch).

## DESIGN BRIEF (CC, 2026-07-18 — seam-grounded, for ratification)

### D1. Judge seam — AFTER completion verification, same substrate, additive
New `judge_impact(tenant_id, task_id) -> ImpactVerdict` in `verification.py`, called from
`_verify_completion_step` ONLY after `verify_completion` returns `"verified"` (the completion
floor stays exactly as-is: deterministic floors + DB-fact floors + opus checkpoint untouched).
Inputs = the SAME persisted substrate `verify_completion` already reads (task objective +
`acceptance_criteria` + per-step summaries + `manager_step_desired_outcome` from step detail) —
never a live specialist payload, so replay/recovery semantics are unchanged.
`ImpactVerdict(verdict: Literal["met","partial","unmet"], reason: str)` — pydantic
`extra="forbid"`, mirroring `CompletionVerification`.

### D2. Verdict handling — v1 is JUDGE + HONEST SURFACE, no auto-retry
- `met` → settle exactly as today.
- `partial`/`unmet` → task STILL settles `completed` (completion is factually true; lying about
  that would break money-authority) — but the verdict rides into the owner's terminal
  notification as an honest quality note ("done, though X fell short of Y — want me to take
  another pass?") and into the audit trail.
- **NO automatic re-run in v1.** Retry-with-feedback re-executes specialist work (LLM cost,
  and for money steps a replay hazard the approval rails exist to prevent). The owner's reply
  to the honest note IS the retry trigger — through the normal reactive path, correctly gated.
  Auto-retry (bounded, non-money-only) = explicit phase-2 with its own Fazal call. Judge
  failure/timeout → fail-soft: settle as today, verdict recorded `"unjudged"` — the judge must
  never make a completed task hang.

### D3. Logging — tm_audit, §7D-joinable
`emit_tm_audit(event_layer="decides", event_kind="impact_judged", actor="team_manager",
decision={verdict, reason}, reasoning_ref={"run_id": task_id, "step_name": "impact_judge"},
conn=None)` — fail-soft (advisory judgment must not block settle; the VT-460 fail-closed mode is
for effect-gating audits). Joins VT-514 observability via the existing reasoning_ref mechanics.

### D4. Model + metering — through the SEAM, not a raw client
Unlike verify_completion's legacy raw `Anthropic()` client, the impact judge goes through
`resolve_chat_model(<tier>, agent="team_manager", call_site="impact_judge")` — VT-619 metering
free via `LlmUsageCallback`, luna-portable per the seam-port standing (`text_call=`). Tier:
`verification` class (opus-tier), max_tokens ~400. One call per completed task — bounded.

### D5. Rubric alignment with the OFFLINE judge
The online rubric = a strict SUBSET of tier_rescore's Tier-2 quality dimensions: outcome-vs-
desired_outcome + acceptance-criteria satisfaction ONLY (no tone/language/brevity — those are
measurement-gate concerns). Rubric text lives in ONE module-level constant with a comment
binding it to tier_rescore's categories, so drift is a diff away from visible. The offline ×3
gate remains the authority; the online judge is per-task honesty, never the promotion metric.

### D6. Flag + build/test plan
- `TEAM_IMPACT_JUDGE` (dev ON, prod unset/Fazal).
- Unit: verdict parse (forbid-extra), fail-soft path, settle-unchanged-when-flag-off,
  notification composer carries the partial note.
- Dev drive: fixture task with deliberately-thin result (criteria declared, weak evidence) →
  `partial` verdict + honest note; a good task → `met`, notification unchanged. Then ×3
  full-pack re-drive (brain-touch via the notification copy).

### Open point for ratification (only one)
- The partial-note copy + whether `unmet` should ALSO open a tier-1 escalation incident
  (v1 proposal: no — honest note + audit only; escalation stays reserved for
  completion/limit failures, or O-alert noise returns).
