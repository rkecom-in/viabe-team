## Contract (Fazal 2026-06-23: account created but NO welcome WhatsApp). CRITICAL — surfaced live.

**Root cause (verified in code):** `utils/twilio_send.py` calls `messages.create(from_=<plain
+E.164>, to=<plain +E.164>)` with **no `whatsapp:` prefix** — `send_template_message` (:218-222) and
`send_freeform_message` (:280-283). Twilio requires `whatsapp:+…` on BOTH ends for the WhatsApp
channel; raw numbers misroute (SMS/21910), so the send silently fails (best-effort welcome → no
message). OTP was unaffected because Twilio Verify sets the channel internally. This affects ALL
WhatsApp outbound: welcome, onboarding questions, plan delivery, win-backs.

## Fix
1. **Confirm** via the Twilio message log: the welcome send to Fazal's signup number errored / went
   non-WhatsApp (capture the error code, e.g. 21910). One-line confirmation.
2. Prefix `whatsapp:` on `from_` AND `to` in both send functions — **idempotent** (don't double-prefix
   if the value already starts `whatsapp:`). Keep `TEAM_TWILIO_FROM_NUMBER` as plain E.164 (CL-435);
   the prefix is applied at the call site. Mirror in the mock-send path so tests see the prefixed shape.
3. Test: assert both send functions pass `whatsapp:`-prefixed from_/to to the client.

## Acceptance
- A real `team_welcome` reaches Fazal's number after re-fire; Twilio shows channel=whatsapp delivered.
- `send_freeform_message` (onboarding questions / plan) similarly delivers.
