# CL-146 — Correction to CL-143 wording

In CL-143's plan reply, I wrote PR #21 closure comment and annotation commit text with the phrase "post-hoc review completed" — but CodeX's review runs concurrently with steps 0/1, not before. At time of posting, the correct phrase is "post-hoc review initiated" or "under post-hoc review." Claude Code caught it and held (d)+(e) pending wording call.

Corrected text:

(d) PR #21 closure comment now reads: "Content merged via PR #22 (commit dda82d7) due to mis-branching from feat/vt-orchestrator-agent-skeleton instead of main. No revert; post-hoc review initiated per CL-143. New branching discipline live as CL-144. Closing as superseded."

(e) Annotation commit message now reads: `git commit --allow-empty -m "NOTE: dda82d7 contains both PR #22 audit doc AND PR #21 (VT-3.9 PR 1/N orchestrator skeleton) due to mis-branch. Under post-hoc review; no revert. See Clau_Session_Log CL-143."`

Follow-up obligation: when CodeX's post-hoc review completes, CL-143 gets a status update note (Detail field append) with completion date + findings (or 'no findings'). The 'initiated' wording now obligates that closure entry; 'completed' would have skipped it.

Why not hold (d)+(e) until review completes: holding (d) leaves PR #21 as a phantom-open superseded PR for however long CodeX takes (worse signal than careful wording); holding (e) keeps the git log on main claiming dda82d7 is doc-only until review wraps (misleads anyone who pulls in the interim). Accurate-at-post-time + obligated follow-up is cleaner than delayed-posting.

Class of error: same as the original PR #22 incident — asserting something is in a state it isn't. Catching it in my own wording before it shipped is the system working correctly. Claude Code surfaced it; no defensive reframing.
