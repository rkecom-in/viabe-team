# VT-352 — durable Razorpay webhook dead-letter + programmatic replay (Option B)

VT-330 closed the poison-pill with Option A (record-and-drop): on a parse-error the raw event
is COMMITTED to `razorpay_webhook_events` with a `{"_status":"dropped_parse_error","raw":…}`
payload marker + Fazal-alert + 200 (so Razorpay stops looping). That's the right TIGHT choice
pre-LIVE — nothing is lost; Fazal reconciles manually from the marked row.

Option B (this row, Cowork 20260606T074500Z) is the durable, programmatic-replay form for the
LIVE money path:
- a dedicated `razorpay_webhook_dead_letter` table (event_id, event_type, raw_payload,
  error_reason, retry_count, first_seen/last_retry timestamps) — claim the migration number
  via the allocator;
- a replay path (re-feed a fixed/parsed payload through razorpay_ingress);
- query/observability for stuck events.

**SOFT GATE:** the money path should have programmatic replay before real settlements — this
MUST land before `TEAM_RAZORPAY_LIVE=1`. Behind the queue; roster only.

## HARD pre-LIVE acceptance (VT-330 #370 subagent, Cowork 20260606T080000Z — F1)
VT-330's record-and-drop keys the dropped row on Razorpay's `event_id`. Replaying the CORRECTED
event with the SAME event_id hits the dedup → "duplicate" → the fee is NEVER applied → the
charge is silently lost on replay. (Nothing loses money TODAY — the drop path doesn't touch
fees and LIVE is off — so it's a pre-LIVE blocker, not urgent.) **Before `TEAM_RAZORPAY_LIVE=1`:
the replay path MUST re-process a dropped event** (re-process on `_status=dropped_parse_error`,
or via a distinct replay key that bypasses the dedup) **+ a test proving drop → replay → fee IS
applied.** VT-330 softened its comments so they don't promise a replay this row hasn't built.
