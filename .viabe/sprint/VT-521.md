# VT-521 — tm_audit_log.run_id FK drop

## Problem
VT-514 (mig 147) added `tm_audit_log.run_id UUID NULL REFERENCES pipeline_runs(id)` AND wired `emit_tm_audit` onto `record_escalation`, the support-bot Fazal alert, and the campaign complaint-freeze paths. Any `run_id` not present in `pipeline_runs` (an escalation seeded from a marker, a synthetic/test run, or an emit racing the run INSERT) makes the audit INSERT FK-fail (`tm_audit_log_run_id_fkey`); in fail-closed emit mode that error breaks the primary operation. Surfaced as 2 pre-push failures (test_escalations_substrate, test_support_bot) + contributed to others.

## Fix (mig 149)
`ALTER TABLE public.tm_audit_log DROP CONSTRAINT IF EXISTS tm_audit_log_run_id_fkey;` — keep the `run_id` column + its partial index. Plan principle #2: observability is for detection/learning, NOT safety — it must record events even for not-yet-persisted runs. `trace_id` (= `str(run_id)`, mig 147) is ALREADY the deliberately FK-free join key, so the run_id FK was redundant + inconsistent. NOT a weakening of any correctness/eligibility/effect gate — tm_audit_log gates nothing.

## Verify
Migration 149 applies clean (20/20 migration tests). test_escalations_substrate::test_record_escalation_stores_route + test_support_bot::test_replay_does_not_redupe_fazal_alert now pass on a fresh DB. DSR `_PURGE_ORDER` ordering (tm_audit_log before pipeline_runs) stays as harmless hygiene (no longer FK-load-bearing).
