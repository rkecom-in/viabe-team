# VT-585 — the DSR floor must match deletion INTENT, not the substring "my data"

## Live-harness defect
Scenario topic_switch step 1: "Yes connect my data" → team_dsr_acknowledgment. Cause: dsr_keywords.yaml has bare "my data" + "मेरा डेटा" — matches connect/use/share-my-data. A real owner connecting a store gets told their DELETION request is acknowledged. Compliance misfire, worst class (fail-safe direction was 'over-route deletion' but this over-route causes its own harm).

## Constraint
DSR is a legal floor (fail-closed for genuine deletion). The fix must NEVER miss a real deletion request (delete my data / मेरा डेटा हटाओ / data delete karo / GDPR erase) while not firing on connect/use/share contexts. Type-2 governance surface — tests must prove both directions exhaustively.

## Resolution (CC)
Matcher rewritten in pre_filter_gate.py (`_dsr_match`): dsr_keywords.yaml sectioned into standalone / delete_verbs / data_nouns. Fires on (a) an unambiguous standalone phrase, or (b) a deletion verb AND a data/account noun (any order, EN+Devanagari+Hinglish). Bare 'my data'/'मेरा डेटा' no longer fires. CC follow-up: added account nouns (account/अकाउंट/खाता) so 'delete my account'/'मेरा अकाउंट हटाओ' fire (restored erasure coverage the tightening dropped). 66 pure DSR tests + 35 pre_filter tests green, ruff clean. Accepted residual: 'remove account access for staff' over-fires to DSR (near-zero freq for solo-owner persona; fail-safe erasure direction). Status: DONE, pending batch push.
