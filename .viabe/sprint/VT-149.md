# VT-149 — Webhook replay double-write — add (tenant_id, message_sid) UNIQUE constraint

(This page is blank in Notion; metadata captured above.)

Expected Outcome (from Notion):
A UNIQUE constraint on (tenant_id, message_sid) for the inbound-webhook path so a DBOS workflow replay does not double-write a second pipeline run / owner_inputs row for the same originating WhatsApp message. After this, replay of webhook_pipeline_run is idempotent at the row level for a given message.

Notes (from Notion):
Normal priority — operational correctness, NOT privacy. Surfaced by Claude Code in the PR #46 (VT-146) body: on workflow replay there is currently no (tenant_id, message_sid) UNIQUE guard, so a replay can produce a duplicate run / duplicate owner_inputs row — operationally noisy. Deliberately rostered SEPARATE from the DBOS workflow_inputs retention row: different concern (idempotency vs PII retention), different urgency (normal vs Critical). Do not bundle the two. Migration adds the constraint following the inline-RLS-per-table / migrations-convention pattern; sequence after the current latest migration. Decide handling of any pre-existing duplicate rows (dedupe-then-constrain) at brief time.
