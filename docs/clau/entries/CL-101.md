Verify both functions are updated in the rebase commit; don't drift to only one.

IMPROVEMENT B — docstring correction on `record_brain_pending`:
- Current (line 170-172): 'Idempotency is provided by the DBOS workflow-id boundary (a step never replays); PR-fix-6 adds a UNIQUE (run_id, step_index) constraint as a second guard.'
- Replace with: 'Idempotency is provided by the DBOS workflow-id boundary for COMPLETED steps. A crash between the SQL commit and DBOS recording the step causes re-execution on workflow resume — hence the ON CONFLICT (run_id, step_index) DO NOTHING clause. Migration 014 enforces the UNIQUE constraint that makes ON CONFLICT well-defined.'
- Honesty in code comments matters — Pillar 7 at the documentation surface. Original docstring claims a stronger DBOS guarantee than DBOS actually provides.
- Apply same docstring honesty pass to `record_webhook_received` if its docstring makes the same overclaim.

IMPROVEMENT C — reason-field three-value split on PR #11's pre-check:
- Current (twilio_ingress.py line 138): `'reason': 'dupe' if already_seen else 'started'`.
- New behavior: distinguish three cases by reading the DBOS workflow status returned by `get_workflow_status`:
  - status is None → first time → `'started'`
  - status is SUCCESS → truly a dupe (already-delivered) → `'dupe'`
  - status is PENDING/ERROR/CANCELLED → recovery retry, DBOS will resume → `'recovering'`
- DBOS 2.22 API: verify the actual status enum values via Context7 or live inspection (Context7 lagged on get_workflow vs. get_workflow_status before per #11 CI iteration; assume same risk here).
- Touches twilio_ingress.py only. Test update: existing `test_duplicate_message_sid_short_circuits` still passes (a 2nd ingress of the same SID after the 1st completes → status=SUCCESS → 'dupe'). Add a new test for the 'recovering' case if Claude Code can simulate a PENDING workflow.
- WHY THIS NIT MATTERS: 'dupe' reported for a recovery retry is a Pillar 7 observability lie — team-web logs 'dupe', looks like nothing happened, but DBOS is actually doing recovery work. Three distinct values restore honesty.

LOCATION: all three improvements land in the #15 rebase commit, post-#11 and post-#16 merge. Commit message: 'fix(orchestrator): add targeted ON CONFLICT + correct docstring honesty + split reason field (recovering/dupe/started) [VT-3.3a-fix-2 + #11 followup]'.
