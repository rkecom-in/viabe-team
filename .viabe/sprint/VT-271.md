# VT-271 — non-idempotent CREATE ROLE breaks re-apply

Surfaced during VT-53 canary work (Cowork flagged 2026-06-01). Migrations 015
and 027 run bare `CREATE ROLE app_role ...` which is NOT idempotent — re-applying
`apply_migrations` to an already-migrated DB fails ("role already exists"). That
breaks canary / E2E re-runs against a persistent local pg (the VT-53 standalone
canary couldn't run in-sandbox for this reason) and forces a drop-DB-then-roles
dance.

Fix (pending): guard the CREATE ROLE statements with a `DO $$ BEGIN IF NOT EXISTS
(SELECT FROM pg_roles WHERE rolname='app_role') THEN CREATE ROLE ...; END IF;
END $$;` block (or equivalent), so re-apply is a no-op. NOT in scope for the
Sprint 3 build; filed for Fazal. Recurring local-DB friction (mig 015/027).
