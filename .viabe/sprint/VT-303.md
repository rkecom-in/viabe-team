# VT-303 — owner_inputs consent gate on the brain transmit (Option B) + enable path

Sprint 7 privacy wave, row 1 (Cowork 20260603T140500Z privacy-first ordering;
20260603T141500Z Fazal Option B approved).

## Problem
`dispatch_brain` transmits the owner's inbound WhatsApp body (which may carry
customer PII) to Anthropic (sub-processor) with NO `owner_inputs` consent gate —
while the peripheral surfaces (classify VT-270, L0 writer, vision VT-52, voice
VT-59) ARE gated on `_owner_inputs_enabled`. CL-425: `owner_inputs` is the lawful
basis for that transmit. The core brain transmit was the gap.

## Decision (Fazal, Option B)
Gate the inbound→brain transmit on `tenants.owner_inputs`. Fail-closed = a
conservative NON-LLM reply ("enable data inputs to activate your AI team"), NO
transmit. NOT a drop (would dead-product every tenant — owner_inputs defaults
FALSE and nothing set it). Paired with the ENABLE PATH so Option B is real.

## Scope
1. **Gate** (runner.webhook_pipeline_run, at the `brain` route, inbound_message
   only — status-callback brain routes carry no body): `_owner_inputs_enabled`
   fail-closed (consent-check error → FALSE). FALSE → `consent_required_handler`
   (free-form enable prompt, no Anthropic transmit). TRUE → `dispatch_brain`.
2. **Enable path** (the setter — without it Option B degrades every tenant
   forever): a deliberate opt-in keyword (`config/data_inputs_enable_keywords.yaml`)
   routed by pre_filter → `data_inputs_enable_handler` sets
   `tenants.owner_inputs = true` + free-form confirm. Direct handler (no LLM, no
   consent needed — it's the consent grant itself). Minimal end-to-end loop,
   independent of the deferred D1 web-onboarding wiring.
3. **Live-path canary** (vt303): owner_inputs FALSE → conservative reply + brain
   NOT called (no Anthropic transmit); enable keyword → owner_inputs TRUE → brain
   runs; cross-tenant isolation; fail-closed on consent-check error.

## Pillars
- Pillar 1: gate + handlers are deterministic, zero LLM.
- Pillar 7: owner authority — owner explicitly enables; STOP still opts out.
- Pillar 8: ONE consent source (`_owner_inputs_enabled`); one routing locus.
- CL-390: never log recipient phone plaintext (send util hashes).
- CL-425: owner_inputs is the basis; enabling permits sub-processor transmit.

## Dependency flagged
The web-onboarding consent step (Fazal D1) is the richer enable path; this row
ships the minimal WhatsApp-keyword enable so the loop is complete now. When D1
lands, onboarding completion should ALSO set owner_inputs=true.
