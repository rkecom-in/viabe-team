# VT-328 — refunded tenant must not dispatch during the 30-day graceful window

Surfaced by the VT-93 adversarial review + Cowork PB2 (20260605T095500Z / 100800Z). A
tenant in `phase=refunded` is in a 30-day read-only graceful exit, but VT-93 only ships
the READ-side rule: `billing/graceful_exit.portal_access_allowed(phase, refunded_at)`.
The WRITE/dispatch path is unguarded — `pre_filter_gate` has NO phase gate today, so a
refunded tenant whose webhook fires could still run the pipeline and dispatch customer
messages (campaigns). That must not happen via ANY non-portal path.

## Scope
1. Add a server-side dispatch guard: a refunded (and cancelled) tenant cannot trigger
   campaign dispatch. Enforce at the campaign-execution seam (the single write
   chokepoint — `agent/dispatch.py` / `send_whatsapp_template` / the pipeline entry),
   NOT only the team-web portal reads. Pillar 8: one enforcement point.
2. Derive the refunded state SERVER-SIDE from the tenant's own row (`phase` / the
   graceful_exit rule) — never trust a client scoping field (IDOR lesson, VT-293/294).
3. Inbound owner replies (DSR, portal, refund-decision) MUST still be processed for a
   refunded tenant during the window — only OUTBOUND customer dispatch is blocked. Do
   not blanket-reject at `pre_filter_gate`.
4. Real-PG canary: a refunded tenant's dispatch attempt is refused server-side; a
   co-resident active tenant is unaffected; an inbound DSR request still routes.

## Notes
- The pure rule already exists in VT-93 (`graceful_exit`); VT-328 is the WIRING at the
  write seam (a hot path — map it carefully; VT-93 deferred it rather than risk it).
- Related: VT-87 (read-only owner portal) wires the READ-side `portal_access_allowed`.
