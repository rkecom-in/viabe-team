## Contract (Fazal/Cowork 2026-06-26, e2e Step-1 CRITICAL). The entity-match miss hard-blocked Step 1.

Root cause (proven by live canary): (1) dev `APIFY_API_TOKEN` was dark → 0 candidates → "couldn't find" (fixed: token
set); (2) RKeCom's GBP DOES surface but carries no GSTIN → `isConfirmable=false` → owner stuck (no self-declare path);
(3) the web leg emitted 6 irrelevant "Telecom Services" GST-SERP candidates (fuzzy "RKeCom Services"→"Telecom Services").

Fix:
- **team-web manual-GSTIN-entry path** (`entity-match-step.tsx` + `lib/entity-match.ts`): a `manual_gstin` wizard step —
  the owner types their GSTIN → the EXISTING `confirmCandidate`→`/entity-confirm`→Sandbox verify (the authoritative gate,
  unchanged) → `gstin_verified` → create. Reachable from the picking screen ("Enter my GST number") incl. the 0-candidate
  + found-but-no-GSTIN cases. Client-side `isValidGstinFormat` is a FORMAT gate only (typo retry; never verification). A
  "not GST-registered" link → the same generic reject (no enumeration oracle). Bilingual EN/HI.
- **orchestrator web + GBP relevance filter** (`entity_match.py`): `_significant_tokens` + `_result_is_relevant` drop
  candidates that don't echo a distinctive (non-generic) name token. All-generic name → not over-filtered. The manual
  path is the universal fallback for anything dropped.
