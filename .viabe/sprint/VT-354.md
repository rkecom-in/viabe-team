# VT-354 — enforce the waitlist retention bound (scheduler), not just runbook-manual

VT-97 (#373) shipped the waitlist purge functions — `purge_stale_unnotified(months=6)` (the
6-month retention bound for un-notified rows) + `purge_notified_waitlist()` (post-launch-sweep) —
but they are **orphaned**: nothing calls them. So the DPDP retention bound is runbook-manual, not
enforced (Cowork #373-subagent finding 20260606T090000Z).

## Scope
1. **Wire `purge_stale_unnotified` to a DBOS scheduled job** (e.g. daily) so the 6-month
   retention bound is ENFORCED, not documented-only. (purge_notified_waitlist stays an explicit
   ops/launch-sweep step.)
2. **NIT-3 (folded):** allow erasure by `whatsapp_e164` too, not just `email` — the
   `DELETE /api/waitlist` ops endpoint currently keys on email only; a principal may know only
   their number.
3. Update `docs/policy/waitlist-data.md` to state auto-purge is now ENFORCED (replace the
   "runbook-manual, pending VT-354" note this row removes).

## HARD pre-LIVE gate
The retention bound MUST be enforced (this scheduler live) **before** `ENABLE_WAITLIST_CAPTURE=true`
collects real waitlist PII (VT-231 + Fazal). Pre-launch PII must not sit unbounded with only a
runbook promise.

VT-97 #373 adversarial review (Cowork 20260606T090000Z) — purge fns SQL-correct but unwired.
