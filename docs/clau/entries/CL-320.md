# CL-320 — DECISION (Fazal + Clau, 2026-05-22): owner_inputs substrate — retention policy +

DECISION (Fazal + Clau, 2026-05-22): owner_inputs substrate — retention policy + schema shape locked. FAZAL (Type 3, PII/retention — his call): owner messages WILL be persisted, retention window 90 days. This reverses the codebase's prior deliberate non-persistence of Twilio message bodies; storing owner-sent content is now in-policy with a 90-day ceiling. Privacy-policy / consent language must reflect this — flagged to Fazal as a downstream doc task, NOT auto-actioned. CLAU (schema shape — Clau-owned, follows from the retention decision): Option A selected — new dedicated owner_inputs table. Options B (extend twilio_inbound_events) and C (pipeline_steps envelope) REJECTED: both attach a 90-day-expiry content column to a table whose existing rows have an indefinite/never-expire lifecycle (idempotency ledger / audit ledger respectively) — two retention policies on one table, unenforceable clean purge. Option A is the only shape where 90-day retention is enforceable in isolation. LOCKED SHAPE: new owner_inputs table; columns include tenant_id UUID, received_at TIMESTAMPTZ, owner message body/content, message linkage to twilio_inbound_events.message_sid; RLS tenant-scoped inline-per-table (CL-190 pattern); writer added in twilio_ingress to capture the Twilio Body field currently passed in-memory and dropped; a scheduled 90-day purge mechanism (rows older than received_at + 90d deleted). This is a substrate-CREATION VT row (new tracked subtask under VT-7 Knowledge Architecture or VT-4 — sprint placement TBC), prerequisite to the held owner_inputs read-path brief. Read-path brief re-issues against this real table once the creation PR merges. NOT YET DECIDED: VT-4 ship-thin-vs-complete + the Apache AGE hosting question (separate open thread).
