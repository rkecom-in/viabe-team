# VT-251 — campaign execution seam (DEC-1 from VT-140)

## Why
VT-140 DEC-1: the approved-CampaignPlan → collapse → POST-collapse EXECUTE path
isn't wired end-to-end. VT-47 = approval primitive; VT-45 = send tool; neither
fans the approved campaign out to its cohort. This seam connects them — a real
launch requirement, not just test scaffold. Depends VT-45 + VT-47; before VT-140.

## Scope
On VT-47 resume with decision=approved: `execute_approved_campaign(tenant_id,
campaign_id)`:
1. Load `campaign_recipients` for the campaign (045/241).
2. Per recipient: gate on `customers.opt_out_status` — NEVER message
   opted_out/blocked (CL-421 consent). Resolve template+params via the VT-163
   registry; call `send_whatsapp_template` (VT-45) with idempotency + per-tenant
   rate limit (049 tables). Record `campaign_messages` (template_id, params,
   send_status, message_sid).
3. Advance `campaigns.status` → sent.
4. Hand the sent set to `match_transactions` (VT-46) → attribution (VT-43/240).
5. Partial-failure: per-recipient failures recorded + retried/skipped, NOT
   fatal to the run (mirror VT-241 reject discipline — surface, don't drop).

No migration (reuses 049/045 tables + existing campaigns.status). DEPENDS:
VT-45 + VT-47 merged. Plan: .viabe/queue/VT-251/plan.md. ONE PR, plan-first.
