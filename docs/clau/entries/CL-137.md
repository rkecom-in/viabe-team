# CL-137 — Phase 1 codebase must not run on deprecated APIs

Fazal standing instruction 2026-05-18 after CL-134 (create_react_agent deprecation) surfaced: don't wait for things to deprecate — audit and migrate proactively. Stronger than CL-135 (Context7-validate new briefs). This rule applies to EXISTING code, not just new code.

Nothing currently in the Phase 1 codebase should be running on:
- a deprecated API
- an EOL package
- a meaningfully outdated library version where current canonical API differs from what we use
- a library that is unmaintained or has a more current well-supported replacement

Immediate action: schedule VT-Dep-Audit (pseudo-parent task) BEFORE further forward-feature work. Scope below.

VT-Dep-Audit recon scope:
1. Inventory: every external runtime dep across Phase 1 app pyproject.toml files. Currently confirmed apps: apps/team-orchestrator, apps/team-ingestion-worker, apps/team-web (frontend), possibly others. Claude Code provides full inventory.
2. Per dep, Context7 lookup: (a) current latest stable version; (b) our pinned version; (c) deprecation status of the SPECIFIC API surfaces we use (grep call sites); (d) EOL or maintenance status of the package itself; (e) any newer canonical replacement.
3. Categorize each finding:
   CRITICAL: package EOL or major API we use removed in a version we'll need to bump to.
   HIGH: deprecated API we use (still works, scheduled for removal).
   MEDIUM: old pinned version, current API meaningfully different from ours, no urgency but cleaner to migrate.
   LOW: fine as pinned, on current API path.
4. Output: VT-Dep-Audit findings doc + per-finding ticket for the migrations needed.

Migration sequencing: by blast radius. CRITICAL first, HIGH next, MEDIUM batched. LOW = closed without action.

VT-3.4 PR 1/3 is paused until at least the CRITICAL + HIGH migrations relevant to apps/team-orchestrator land. Specifically, create_react_agent migration (CL-134) is HIGH and is a hard prerequisite for VT-3.4 PR 1/3 because PR 1/3 extends the orchestrator_agent's tool list (touches the same code).

Rule going forward: this audit happens periodically. Cadence to be set after first audit completes. Initial guess: re-audit at end of each Sprint, or any time we bump >3 deps in a single PR.
