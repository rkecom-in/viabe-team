# CL-74 — CRITICAL audit finding C1: DB writers bypass tenant GUC

transitions.py:115 and 9 other call sites open `get_pool().connection()` and write to tenant-scoped tables (phase_transitions, twilio_inbound_events, pipeline_runs, pipeline_steps, tenants, dsr_tickets) without first executing `SELECT set_config('app.current_tenant', <tenant_id>, false)`. All these tables have FORCE ROW LEVEL SECURITY enabled. In CI everything runs as Postgres superuser, which bypasses FORCE — that's why all 63 orchestrator tests pass. In production, if the orchestrator connects as a dedicated app role (best practice; Supabase service-role works around this by bypassing RLS at the auth layer), every write will be rejected.

Three fix options:
(a) Add a typed connection wrapper (e.g. orchestrator.db.tenant_connection(tenant_id)) that sets the GUC on checkout. Every @DBOS.step that takes a SubscriberState or tenant_id uses it. Most code change but cleanest.
(b) Document that the orchestrator runs as the table-owner role and drop FORCE on tenant tables. Less safe (table-owner bypass is implicit).
(c) Connect as Supabase service-role, which bypasses RLS entirely. Defeats Pillar 3 within-app and only works if no other code paths share connections.

Option (a) is the only one consistent with Pillar 3 as documented. Adds an end-to-end test that connects through the app role with the GUC set, which proves the writes actually work — none of the current 63 tests prove this.

BLOCKING WORK: any further DB-touching subtask (VT-3.3c handler send wiring, VT-3.4 supervisor) should not ship until this is resolved or Fazal confirms production runs as service-role/superuser. Confirm role before next PR.
