## What CoderC must produce
The SupportBot escalation handler. When the classifier (VT-5.11) returns `intent='unclear'` with low confidence, OR when the same owner has 2 consecutive unresolved interactions in a 24-hour window, this handler intervenes: sends an honest "let me get Fazal" message and escalates via Telegram with full context.
1. **Handler module** at `apps/team/src/owner_surface/support_bot.py`:
	- `escalate_to_support(tenant_id, message_body, run_id, escalation_reason: Literal['classifier_unclear', 'two_unresolved', 'agent_terminated', 'system_error'], context: dict) -> SupportEscalation`.
	- Migration `032_support_escalations.sql`: table `support_escalations` with `id`, `tenant_id`, `triggered_at`, `escalation_reason`, `original_message_hash`, `agent_run_id`, `fazal_acknowledged_at NULL`, `resolved_at NULL`, `resolution_notes TEXT NULL`. Tenant-scoped RLS.
2. **Owner-facing handoff message**:
	- Template `support_handoff` (added to VT-13.3): "I want to make sure I understand correctly. Let me get Fazal to help with this. He'll reply within \[X\] hours. Your message ID is \[escalation_id\]."
	- X = 4 hours during business hours (10am-7pm IST), 24 hours otherwise. Configurable in `apps/team/config/support_sla.yaml`.
3. **Fazal-facing escalation**:
	- Telegram alert with: tenant business name + owner phone (last 4 only) + the original message + classifier output + agent dispatch history (if applicable) + escalation_reason + escalation_id.
	- Fazal replies via Telegram bot command `/resolve <escalation_id> <notes>` to mark resolved + log resolution_notes. The Telegram bot integration is reused from Reports' OpsAssistant.
	- SLA breach: if not acknowledged within the SLA window, second Telegram alert + email backup. After 2x SLA: alert escalates to a CRITICAL Telegram (loud notification rule).
4. **Repeat-escalation fatigue**:
	- If the same tenant has 3 escalations in a 7-day window: alert mentions "3rd escalation; consider proactive outreach" — prompts Fazal to call the owner rather than respond reactively.
5. **Owner-side resolution feedback**:
	- When Fazal resolves, owner receives template `support_resolved` (added to VT-13.3): "Fazal got back to you. Your support ticket \[id\] is resolved. Anything else?"
	- If Fazal's resolution involved sending a free-form message via WhatsApp (manually), the bot DOES NOT send `support_resolved` (Fazal's message IS the resolution). Decision flag in /resolve command: `--no-followup` skips the auto-template.
6. **Tests** at `apps/team/tests/owner_surface/test_support_bot.py`:
	- Classifier-unclear escalation: synthetic owner message → classifier returns unclear → SupportBot triggered → row written, owner template sent, Telegram fired.
	- Two-unresolved escalation: tenant with 2 prior interactions in 24h that produced no useful response → SupportBot intervenes on 2nd unresolved.
	- Agent-terminated escalation: hard-limit breach (VT-4.4) → SupportBot intervenes.
	- System-error escalation: a database_error or critical exception → SupportBot acknowledges to owner.
	- SLA breach: synthetic 5-hour-old un-acknowledged escalation during business hours → second alert.
	- 3rd-escalation flag: 3 escalations in 7 days → alert includes proactive-outreach suggestion.
	- Resolution flow: synthetic /resolve command → row updated, owner template sent (or skipped with --no-followup).
	- Cross-tenant: tenant_A escalation cannot mark tenant_B's resolved.
## Architectural rules to enforce in this subtask
- Pillar 7: human fallback is honest. Owner is told a human will reply; CodeX rejects PRs that hide the escalation behind "the AI is processing" pretense.
- Pillar 8: one escalation path. CodeX rejects parallel escalation flows. The 4 escalation_reason values cover the legitimate trigger types; adding a 5th is Type 1 governance.
- SLA window is config-driven; business-hours definition (10am-7pm IST) hardcoded for Phase 1 with a config note for Phase 1.5 expansion (multi-timezone owners).
- Repeat-escalation flag prompts proactive Fazal action; not automated outreach.
- Resolution flow uses Telegram bot command pattern reused from Reports' OpsAssistant.
- Templates 17 + 18 added to VT-13.3 list (was 16). New count: 18.
- Phone numbers in Telegram alerts: last 4 digits only.
