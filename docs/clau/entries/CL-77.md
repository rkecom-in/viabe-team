# CL-77 — CRITICAL audit finding C4: Pillar 7 violation in return contract

opt_out_handler returns `{"handler": "opt_out_handler", "opt_out_set": True, "confirmation_sent": True}` while the actual 'send' is a logger.info line tagged `TODO VT-3.3`. Same in dsr_handler (`acknowledgment_sent: True`) and status_ping_handler (`reply_sent: True`). template_error_handler and dupe_handler don't have this issue (no claimed send).

The handlers are honest in their comments — they say 'TODO VT-3.3: replace this logged stub with the real Twilio template send.' But the return contract lies. Anything that reads these handler returns (observability, billing, audit trails, future supervisor logic) will record an opt-out confirmation as sent when none was. That breaks Pillar 7 (owner-truth: report ACCURATE state only) at the contract layer.

Fix: change the return values to `"confirmation_sent": False, "send_pending": True` (or similar) in all three handlers until the real Twilio send wires in. One line per handler.

Note: as of HEAD 4106e29, VT-3.3b shipped but is named 'team-web Twilio webhook + signature verification + tenant lookup + rate limiting' — it doesn't sound like the send-wiring subtask. So either VT-3.3c (or similar) is the send-wiring subtask and these TODOs need their pointer updated (see H5 in audit report), or the send wiring hasn't been scheduled. Confirm with Fazal.
