# CL-208 — Orchestrator subgraph state schema: Option A-narrow

=== SEAM ===
Claude Code STOP at Step 6 seam (well within CL-205 2hr checkpoint; legitimate gate, not escalation): handoff tool body in spawn_sales_recovery cannot read tenant_id / run_id / trigger_reason from InjectedState as Step 6 brief assumed. orchestrator_agent is a create_agent compiled subgraph used as a parent node; create_agent's default state_schema is AgentState (messages + remaining_steps). langgraph filters parent AgentGraphState down to the subgraph schema, so the three propagation-required fields never enter the subgraph. Handoff tool runs INSIDE that subgraph → InjectedState sees only ['messages'].

=== IMPACT IF BUILT AS BRIEFED ===
trigger_reason always falls back to 'weekly_cadence' (read site fallback always fires). tenant_id always None → fail-loud TenantIsolationError raises on every spawn → orchestrator can never hand off. Step 6 unbuildable on the original assumption.

=== OPTIONS CONSIDERED ===

Option A (state_schema=AgentGraphState wholesale):
  Pro: one-line change, keys propagate verified by Claude Code probe.
  Con: loses create_agent's remaining_steps recursion guard — orchestrator loop has no static safety net for pathological tool-call loops. Sales-recovery-shaped fields (campaign_plan, sales_recovery_context, active_agent, terminated_without_spawn) bleed into orchestrator subgraph as inert today; silent state leak surface if future code reads them.

Option A-narrow (dedicated TypedDict extending create_agent's AgentState with the three fields) — SELECTED:
  Pro: preserves remaining_steps recursion guard. No inert-field bleed. Narrowest change that makes CL-195 work as written. CL-195 stays intact (three fields in STATE, fallback at read site).
  Con: requires one extra propagation probe with the narrow TypedDict (Claude Code probed AgentGraphState specifically, not the narrow variant). Cheap, CL-183-aligned.

Option B (runtime context via ToolRuntime, read tenant_id/run_id from context not state):
  Pro: cleanest separation — immutable per-run identity belongs in context.
  Con: reopens CL-195's explicit 'in STATE' decision. Bigger scope, mid-PR scope creep, violates CL-205 single-task-per-PR default.

=== SELECTED: OPTION A-NARROW ===
Define OrchestratorAgentState(AgentState) with run_id, tenant_id, trigger_reason as additional fields (totality matches AgentState's, verify in pre-flight). Pass state_schema=OrchestratorAgentState to create_agent in build_orchestrator_agent. Replicate Claude Code's propagation probe with the narrow TypedDict before proceeding Step 6.

=== RATIONALE ===
1. remaining_steps is real safety, not optional. Losing the recursion guard for one-line convenience is the wrong trade; orchestrator loop SHOULD be small but there is no static guarantee.
2. No silent state leak surface. Narrow TypedDict carries exactly what orchestrator needs, nothing it doesn't.
3. CL-195 stays intact. Option A-narrow is the narrowest change that makes CL-195 functional as originally written.
4. Probe cost is small and aligned with CL-183 discipline.

=== SUPERSEDES NOTHING ===
CL-195 unchanged. This decision is an implementation detail of how the three CL-195 fields actually propagate through the create_agent subgraph boundary — not a redesign of where they live.

=== FOLLOW-ON SEAM SURFACED ===
Step 6 brief specifies a cross-tenant guard: assert state['tenant_id'] == bundle['tenant_id']. If build_sales_recovery_context is called inside the spawn tool body with tenant_id=state['tenant_id'], the guard is tautological (value compared to itself; no-op). Cross-tenant guard only has real signal if the bundle's tenant_id source is independent of state (workflow context, separate DB lookup). Claude Code instructed to grep tenant_id source for build_sales_recovery_context before writing spawn tool body; report result together with narrow-TypedDict probe. Three outcomes: (a) tautological — Clau decides remove or restructure; (b) independent — guard has signal, proceed; (c) unclear — STOP, Pillar 3 question.
