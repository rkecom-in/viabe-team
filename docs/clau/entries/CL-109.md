# CL-109 — VT-3.3c brief named 5 handlers; phantom status_callback_failure_handler

BRIEF ERROR caught by Claude Code 2026-05-18 during VT-3.3c implementation: VT-3.3c brief listed 5 handlers to rewire (opt_out, dsr, status_ping, status_callback_failure, template_error) but repo has only 4 sendable handlers (opt_out, dsr, status_ping, template_error) plus dupe. status_callback_failure_handler does not exist as a separate file.

ROOT CAUSE: when Fazal provided the template renames (status_callback_failure_notice → team_unable_to_complete_request; template_error_handler-template → team_error_handler), Clau treated each rename as implying a separate handler. They aren't separate. pre_filter_gate routes status_callback events with state='failed' to the existing template_error_handler — one handler covers both cases.

RESOLUTION: Option 1 — rewire 4 real handlers, drop the phantom.
Final mapping:
- opt_out_handler → team_opt_out_confirmation
- dsr_handler → team_dsr_acknowledgment
- status_ping_handler → team_status_ping
- template_error_handler → team_error_handler

YAML keeps all 8 template entries. Four (team_welcome, team_weekly_approval, team_agent_stuck_escalation, team_unable_to_complete_request) remain configured-but-unused for VT-3.4 / VT-6 wiring later.

Test count: 8 (4 handlers × 2 tests), not 10.

DISCIPLINE LESSON: when authoring handler↔template mappings, verify handler-file existence in the repo BEFORE writing the brief, not at Claude Code execution time. The audit-prompt's own scope-creep checklist (Pillar 1 verification of handler inventory) would have caught this in 30 seconds. Apply to all future VT-3.x briefs that name handlers.
