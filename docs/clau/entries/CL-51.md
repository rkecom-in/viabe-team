# CL-51 — Tech debt: monorepo has single root .env.example

viabe-team monorepo has a SINGLE .env.example at repo root used by all three apps (team-web, team-orchestrator, team-ingestion-worker). This is structurally problematic for three reasons:

1. **Frontend env var leakage risk.** team-web is Next.js — vars prefixed NEXT_PUBLIC_ get bundled into client JavaScript. If a server-only secret (e.g. TEAM_TWILIO_AUTH_TOKEN) lives in the same file and someone mistakenly prefixes a related var with NEXT_PUBLIC_, the secret becomes browser-readable. Real exposure risk.

2. **Vercel/Railway deployment scoping mismatch.** Vercel takes env vars per-project, Railway takes them per-service. A single root .env means we mentally have to remember which subset goes where during deployment. Easy to over-share or under-share.

3. **Convention drift.** Standard Next.js + Python service monorepos use per-app .env.example files so each service only sees its own surface.

**Fix scope:** ~30-minute Claude Code PR. Move root .env.example → per-app .env.example files under apps/team-web/, apps/team-orchestrator/, apps/team-ingestion-worker/. Update README references. Update CI to verify per-app envs scaffold correctly. Add lint rule that NEXT_PUBLIC_ vars never appear in team-orchestrator or team-ingestion-worker env files.

**Not blocking current work.** Tonight's Twilio + Foundation kickoff proceeds on the single root .env.local. Refactor scheduled for Day 3 morning (May 17) before VT-3.x DBOS work or any code touches env-bound secrets.
