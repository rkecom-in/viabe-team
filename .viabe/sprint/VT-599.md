# VT-599 — model-fillable tenant_id on 33 lane tools

## Finding (deployed dev 3c98a78, VT-598 pack)
marketing_lane.list_recent_campaigns is `@tool (tenant_id: str, ...)` — the LLM fills it and
Sonnet-5 wrote the tenant's business NAME → `UUID()` ValueError in the wrapper → unhandled
exception out of graph.invoke → DBOS retry loop; the run never terminates (harness TIMEOUT; a real
owner would see silence then the reaper). Same pattern on ~33 @tool functions across
marketing/sales/accounting/finance/tech/cost_opt lanes. Worse than the crash: the wrapper scopes
RLS by the PASSED id — a model-authored foreign UUID is the VT-293/294 IDOR class (server-side
scope derivation rule).

## Fix
Authoritative tenant comes from the run context (the observability ContextVar set at dispatch), or
the lane-agent state — NEVER the model: lane tools stop exposing tenant_id as a fillable param
(bind at tool-construction, or read the ContextVar and ignore/override the model value, logging a
mismatch). Any remaining parse of an id validates + fail-softs to a structured tool_error (a tool
error must return to the agent loop, never crash the graph — the VT-484 middleware invariant).
Tests per lane: name-instead-of-uuid → clean tool_error / override, never an exception; foreign
UUID → overridden to the run's tenant + mismatch logged.
