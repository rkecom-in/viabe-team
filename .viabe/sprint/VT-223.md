# VT-223 — Vercel preview deploys + sticky-deploy diagnosis

## Why this row exists

VT-221 surfaced two Vercel-side gaps that didn't block dev env but cost iteration time:

1. **Preview deploys disabled** — PR-level Vercel preview builds don't fire (only main triggers a deploy). Made VT-221 identity validation harder than necessary — we couldn't see preview check until after merge to main.
2. **Sticky main deploy** — Fazal reported a Vercel deploy stuck for 2 hours; manual cancel + redeploy unstuck it. Root cause unknown; could be a one-off or a config gap.

## Scope

### A. Preview deploy enablement (Fazal-side dashboard)

1. Vercel dashboard → viabe-team project → Settings → Git → enable "Deploy preview branches" / equivalent. CC documents the exact toggle path post-investigation.
2. Confirm preview deploys fire for non-main branches by pushing a feature branch + verifying preview URL.
3. Document in `docs/clau/dev-env-runbook.md` "Preview Deploys" section.

### B. Sticky-deploy diagnosis

4. Capture Vercel build logs from the stuck deploy that Fazal saw 2026-05-28 (timestamp around VT-220 merge). If Vercel retains them, file in `.viabe/queue/VT-223/diagnostic.md`.
5. Likely causes to investigate:
   - Build timeout (Vercel default is 45 min; we may have a long-running step)
   - Webhook race during force-push (VT-219's git identity amend + force-push may have triggered a deploy on the orphan commit that hung)
   - `ignoreCommand` exit code ambiguity
6. If a config fix is identified, ship it as `apps/team-web/vercel.json` update.

### C. ignoreCommand reconsideration

7. Current `vercel.json` `ignoreCommand`: `git diff --quiet HEAD^ HEAD ./`. This skips builds when changes are outside `apps/team-web/`. Side effects observed in VT-221:
   - Made debugging the no-op deploy trigger harder
   - Potentially causes confusion when developers expect a deploy on a docs change
8. Reconsider: should we keep the skip-on-non-web-changes optimization, or always deploy on main pushes? Trade-off: build minutes vs. deploy visibility.

## Acceptance criteria

1. Preview deploys fire on push to non-main branches; PR shows Vercel preview check
2. Diagnostic file for the sticky deploy in `.viabe/queue/VT-223/diagnostic.md` with root cause OR "couldn't reproduce; no logs retained"
3. ignoreCommand decision documented (keep or remove or change) in runbook
4. Runbook updated with preview deploy section

## Rule #15 Canary

- A1: push a feature branch with a `apps/team-web/` change → assert preview deploy fires within 2 min
- A2: push a feature branch with a docs-only change → assert preview deploy fires (or doesn't, per ignoreCommand decision)
- A3: smoke probe of preview URL returns expected status
