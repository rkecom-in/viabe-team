## What CoderC must produce
A cookie-free A/B testing framework for the landing site. Variants are deterministically assigned per visitor based on a hash of (visitor IP + user-agent + experiment ID); no cookies, no localStorage, no tracking IDs. Variants tracked anonymously via event counters. Phase 1 ships the framework and ONE example experiment (homepage hero variant); real experiments run post-launch.
1. **Variant assignment** at `apps/team-web/lib/experiments.ts`:
	- `assignVariant(experimentId: string, request: NextRequest, variants: string[]): string` (server-side):
		- Hash input: SHA-256 of `(visitor_ip + user_agent + experimentId)`.
		- Hash output → integer mod len(variants) → picks variant.
		- Same visitor + same experiment + same time → same variant. Different experiment → independent assignment.
		- No cookie, no localStorage. The hash IS the assignment.
	- Visitor IP truncated to /24 (IPv4) or /48 (IPv6) before hashing to reduce identifiability while preserving consistency for most visitors.
2. **Experiment configuration** at `apps/team-web/config/experiments.json`:
	- List of active experiments: `{id, name, variants: [string], traffic_split: [number], active: bool, start_at, end_at}`.
	- Phase 1 ships with one example: `homepage_hero_v1` with variants `["control", "variant_b"]` and 50/50 split, but `active: false` (framework only; no live experiment Phase 1).
	- Adding/modifying experiments is a Type 1 PR with Fazal sign-off.
3. **Variant rendering**:
	- Components access `useExperiment(experimentId)` hook (client-side) or `getExperiment(experimentId, request)` (server-side).
	- Returns assigned variant; component renders accordingly.
	- Phase 1 example: `apps/team-web/app/team/page.tsx` calls `getExperiment('homepage_hero_v1', request)` and renders `<HomepageHero variant={variant} />`. Both `control` and `variant_b` rendered the same Phase 1 (no actual difference; framework demonstration).
4. **Conversion tracking**:
	- Anonymous event counter at `apps/team-web/lib/analytics-events.ts`:
		- `trackExperimentExposure(experimentId, variant)` — fires on page render (server-side; counts in Plausible or similar).
		- `trackExperimentConversion(experimentId, variant, conversionType)` — fires on CTA click / signup completion.
	- NO PII: only counts of (experiment, variant, exposure \| conversion).
	- No cross-session tracking; if same visitor returns next day with different IP, they get re-assigned (acceptable noise).
5. **Statistical analysis**:
	- Phase 1 ships analysis as a manual ops task: Fazal queries the analytics counter and runs a chi-squared significance test in a notebook. Documented at `docs/team/experiment-analysis.md`.
	- Phase 2 may add automated dashboard.
	- Minimum sample size guidance documented (typically 1000 conversions per variant for 10% effect detection).
6. **Privacy posture**:
	- Visitor IP is hashed + truncated; never stored.
	- User-agent contributes to hash; not stored.
	- Cookie-free design respects DPDPA + GDPR.
	- Documented at `docs/team/experiments-privacy.md`: explicit explanation of how A/B testing works without cookies.
7. **Tests** at `apps/team-web/__tests__/experiments/*.test.ts`:
	- Determinism: same (ip + ua + exp_id) → same variant.
	- Distribution: 1000 synthetic visitors → variants split approximately matching traffic_split.
	- Independence: same visitor in two different experiments → variants independent.
	- IP truncation: /24 and /48 truncation works.
	- Inactive experiment: returns 'control' regardless of hash.
	- Conversion tracking: synthetic exposure + conversion → counter incremented in analytics.
	- No PII verification: experiment events contain no IP, no user-agent, no PII.
## Architectural rules to enforce in this subtask
- Pillar 7: cookie-free design. CodeX rejects PRs adding cookies / localStorage / tracking IDs to experiments.
- Pillar 8: one experiment system; CodeX rejects parallel A/B mechanisms (e.g., a server-side flag system + a client-side flag system).
- IP truncation enforced; raw IP never enters the hash.
- Variants accessed via single hook + single server function. CodeX rejects ad-hoc `if (Math.random() < 0.5)` patterns.
- No cross-session tracking. Returning visitor noise is acceptable Phase 1.
- Conversion tracking is anonymous counters; CodeX rejects PRs adding visitor-level conversion attribution.
- Phase 1 deliverable: framework + example. Real experiments post-launch.
- Adding experiments is Type 1 PR with Fazal sign-off; framework changes are Type 2.
## Acceptance criteria
- Variant assignment deterministic + tested.
- Distribution test passes with 1000 synthetic visitors.
- Independence across experiments tested.
- IP truncation tested (/24 + /48).
- Active/inactive experiment toggling works.
- Conversion tracking emits anonymous counters.
- Privacy posture documented.
- Phase 1 example experiment scaffolded as inactive.
- TypeScript strict passes.
- PR title references `(VT-LandingSite)`; reviewed by Frontend Engineer + CoderX (mandatory — privacy-critical); Fazal personally signs off on framework + example.
## Out of scope
