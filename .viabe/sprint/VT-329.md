# VT-329 — Hindi-script + Hinglish DSR/opt-out/negation (the core SMB user base)

The VT-85 re-review subagent (Cowork 20260605T115500Z) found TWO PRE-EXISTING i18n gaps
(on main, NOT VT-85 regressions) that VT-85 now gives money/DPDP teeth on the primary
Hindi/Hinglish SMB user base. No live exposure today (Razorpay stubbed, public signup
gated), so this is a GATE, not a merge-blocker — but it CANNOT be un-tracked.

## Gaps
1. **Devanagari `\b` dead-pattern (`pre_filter_gate.py`):** the DSR/opt-out keyword
   patterns compile `re.compile(rf"\b{kw}\b")` — ASCII `\b` NEVER matches a Devanagari
   word boundary, so every Devanagari DSR/opt-out keyword pattern (`मेरा डेटा`, `हटाओ`, …)
   is DEAD in both `pre_filter` (rule b) AND the new VT-85 `matches_opt_out_or_dsr`
   helper that reuses them. `"मेरा डेटा हटाओ refund"` → DSR silently dropped + (at VT-89
   live) auto-refund. **Affects EVERY DSR/opt-out path TODAY, not just refund.** Fix at
   the pattern-compilation layer: Unicode-aware boundaries (the `regex` module with `\b`
   + UNICODE, or explicit `(?<!\w)`/`(?!\w)` under `re.UNICODE`, or anchored alternation)
   — verify against the actual Devanagari entries in `dsr_keywords.yaml` /
   `opt_out_keywords.yaml`.
2. **Hinglish (romanized Hindi) negation/opt-out uncovered (refund classifier):**
   `owner_inputs/refund_reply.py` covers EN + Devanagari but not romanized Hindi.
   `"mujhe refund nahi chahiye"` → fires REFUND (a false-positive refund, NOT safe
   inaction). Add romanized negation (`nahi`, `nahin`, `mat`, `na`) + opt-out
   (`band karo`, `hatao`, `roko`) tokens. Fold the romanized continue/discuss coverage
   (`jaari rakhein`, `baat karni hai`) into this row too.

## HARD GATE
Must land BEFORE either:
- (a) **VT-89** enables LIVE Razorpay (a Hindi DSR/negation that auto-refunds = real money), AND
- (b) **VT-326** enables PUBLIC SIGNUP for any non-English locale.
Referenced as a blocker on VT-89.md + VT-326.md acceptance (same pattern as the VT-93 N1 gate).

## Canary (real-PG, fail-not-skip)
Hindi-script AND Hinglish DSR/opt-out/negation co-occurring with "refund", for a
refund_offered tenant:
- `"मेरा डेटा हटाओ refund"` / `"data delete karo refund"` → DSR path, NO refund.
- `"mujhe refund nahi chahiye"` / `"refund नहीं चाहिए"` → None (negation), NO refund.
- opt-out `"band karo"` co-occurring with refund → opt-out path, NO refund.
- affirmative `"refund chahiye"` / `"रिफंड"` → still classifies REFUND.

## Scope note
The `pre_filter` `\b` fix is the BROAD one (all DSR/opt-out, every locale path); the
refund-classifier Hinglish tokens are the NARROW VT-85 surface. Both land in this one
i18n correctness pass. CoderX review (privacy + financial); real-PG canary fail-not-skip.
