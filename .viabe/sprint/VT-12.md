## Why this parent exists
Reports product's observability was retrofitted, which made debugging R6→R9 drift painfully slow — each Director decision wasn't logged structurally, so root-cause analysis required reading raw transcripts. Team must launch with structured observability from day one. The agent will produce wrong outputs sometimes; cost-per-subscriber will drift; tool calls will fail. The fast path to fixing all three is a logging substrate that captures every tool call, every state transition, every LLM cost, every reasoning trace, attributable to a run_id and a tenant_id.
This parent also owns the per-subscriber cost dashboard. Team's economics depend on agent cost staying well below subscription revenue; if a subscriber's monthly cost exceeds their fee, the unit economics break. Cost visibility is unit-economics insurance.
## What this parent owns
1. LangSmith integration: every LangGraph state transition and every LLM call instrumented with `run_id` propagated through the tier stack. Reports' run_id discipline ported here.
2. Structured `pipeline_log` table in Supabase: every tool call, state transition, error event written as a single row with run_id, tenant_id, event_type, payload, latency, timestamp. Append-only, indexed for query.
3. Per-subscriber cost dashboard: aggregates LLM cost per tenant per day/week/month. Surfaces tenants where monthly agent cost exceeds, say, 30% of subscription revenue — a churn-risk signal.
4. Agent invocation reasoning trace + redaction: the agent's chain-of-thought is captured but PII-redacted before storage. Enables debugging without logging customer phone numbers or names verbatim.
5. 80K-token enforcement at orchestrator: token measurement for hard-limit termination (VT-3.6 + VT-4.4 reference this). The instrumentation lives here; the enforcement lives in those parents.
## Architectural rules binding every subtask
- Pillar 8 (no patchwork): observability is a first-class concern, not an afterthought. Logging additions go through the structured framework, not random `console.log`/`print` statements.
- Pillar 3 (tenant isolation): observability data is queryable cross-tenant by Fazal/Clau but never by other tenants. RLS policies on `pipeline_log` enforce this.
- PII redaction is mandatory for chain-of-thought traces. Phone numbers, full names, addresses are redacted before storage. Redaction logic is shared, not per-call.
- Cost attribution must be accurate at the run_id level, not estimated. LLM API responses include token counts; those are stored verbatim with USD/INR conversion.
- Reasoning trace storage has a retention cap: traces older than 60 days are crypto-shredded by default (overrideable for ongoing investigations).
- Dashboard queries cannot be free-form SQL exposed to anyone; they go through a typed query layer that enforces tenant scope at read time.
- LangSmith billing: Phase 1 launches on a tier sized for ~100 paid subscribers. Tier upgrade is a Type 2 governance change (VT-13.9 confirms billing).

## Definition of done
- All 5 subtasks Done.
- A synthetic agent run produces a queryable trace in LangSmith with run_id matched in `pipeline_log`.
- Cost dashboard renders for synthetic subscribers; per-day, per-week, per-month aggregates correct against expected values.
- PII redaction test: a synthetic chain-of-thought containing `+91-9876543210` and `Rajesh Kumar from Connaught Place` produces a redacted trace; raw text is never persisted.
- 80K-token enforcement: a synthetic agent run that exceeds 80K terminates via `escalate_to_owner` with a logged event in `pipeline_log` showing the threshold breach.
- Reasoning-trace retention: a 61-day-old trace is automatically crypto-shredded by the cleanup job.
- Cross-tenant query test: a query from tenant A cannot read tenant B's observability data.
