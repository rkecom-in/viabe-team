# CL-181 — Mistook test count aggregate for canary passing

## What happened

Reviewing PR #26 for VT-3.4 PR 1/3, I read the deviations note 'full tests/orchestrator/ → 5 passed, 89 skipped, no errors' and treated it as green confirmation of the local run. I then put 'confirm test passed locally' as condition 1 of merge approval without parsing that the test we actually care about — test_orchestrator_spawns_sales_recovery_returns_campaign_plan — is @pytest.mark.integration + skipif(not ANTHROPIC_API_KEY) and was IN THE 89 SKIPPED, NOT THE 5 PASSED.

Fazal caught it. The canary for the CL-175 novel-pattern Command.PARENT composition has not fired anywhere — not locally, not in CI. The 5 passed are import-collection coverage: they prove the supervisor chain's symbol graph resolves, which is necessary but not sufficient. The actual routing primitive is unverified at runtime.

## Why this matters

CL-175 explicitly flagged the Command.PARENT composition outside langgraph_supervisor as novel in our codebase. The brief flagged the PR 1/3 happy-path test as 'the canary.' The whole point of integration-gating + manual-run-before-merge was to ensure the novel pattern runs at least once before it goes to main. I lost the thread on that and was about to approve a merge with the canary unfired.

## Failure mode in my reasoning

1. Treated aggregated test counts (5 passed / 89 skipped) as the green signal without checking which bucket the canary fell into.
2. Treated the 'orchestrator' CI check (an import-collection job) as testing the routing pattern. It does not — it tests that imports resolve.
3. Did not pattern-match on the integration-gate language in the deviations note: 'Skipped in CI; runs locally' — this should have prompted me to ask 'did it run locally and did it pass' as a YES/NO question, not as a condition I assumed met by the aggregate green.

## What I'm changing

Review discipline addition: whenever a PR's critical-path test is integration-gated or otherwise non-CI-running, my review must explicitly ask 'did the gated test run and pass' as a separate yes/no, BEFORE listing merge conditions. Aggregate test counts (X passed / Y skipped) never count as evidence the integration-gated canary ran — by definition it was in Y, not X.

This pairs with CL-156 (Context7 lag mitigation) and CL-178 (Context7 enforcement) as a third member of the 'don't infer verification from adjacent signals' rule family. Combined principle: a verification step is fired or not fired. Adjacent signals (CI green, import resolves, sibling tests pass, Context7 returns docs) do not substitute for the specific verification that was specified.

## Status

PR #26 merge is BLOCKED until the canary fires.
