# CL-136 — VT-3.4 uses create_handoff_tool ONLY

VT-3.4 PR 1/3 was about to ship a brief using create_supervisor([orchestrator_agent, sales_recovery_agent], ...). Context7 validation pass 2026-05-18 on /langchain-ai/langgraph-supervisor-py revealed this is wrong:

- create_supervisor's agent list is for SPECIALISTS only.
- The supervisor itself is a separate, third LLM built internally by create_supervisor from model= and prompt= kwargs.
- Our orchestrator_agent from PR #21 IS the supervisor (the routing brain) — not a specialist.
- Using create_supervisor as designed would mean two LLMs reasoning about routing in sequence (supervisor-of-the-orchestrator), wasting tokens and doubling latency.
- Additionally, escalate_to_fazal is a non-handoff tool; create_supervisor only generates handoff tools.

Three options surfaced; Fazal picked Option C:
  A) Use create_supervisor as designed; orchestrator_agent compiled-object becomes dead code; lose escalate_to_fazal.
  B) Drop langgraph_supervisor entirely; build multi-agent state graph manually using langgraph.graph.StateGraph.
  C) (CHOSEN) Hybrid: keep langgraph_supervisor library on disk, use only create_handoff_tool from it (well-tested handoff-tool factory), but do NOT use create_supervisor. Register the generated handoff tool(s) directly on orchestrator_agent. orchestrator_agent IS the supervisor, makes routing decisions via its LLM, and calls the handoff tool when it picks spawn_<specialist>.

Implications for VT-3.4 PR 1/3 brief:
- langgraph-supervisor stays as a dep, pinned ==.
- supervisor.py module is now lighter: just imports create_handoff_tool, generates spawn_sales_recovery, returns the tool.
- The orchestrator_agent built in PR #21 needs its tools list extended with the real spawn_sales_recovery handoff tool (replacing the no-op placeholder).
- Multi-agent graph topology: orchestrator_agent calls handoff tool → LangGraph Command(goto="sales_recovery_agent") routes to specialist node → specialist returns → graph terminates or returns to orchestrator.
- The stub sales_recovery_agent stays a create_react_agent (or equivalent post-deprecation purge) with one no-op tool returning a hardcoded CampaignPlan.

This decision reconciles CL-22 (orchestrator-as-agent) with CL-26 (langgraph_supervisor library chosen) which were in latent tension since 2026-05-12. The reconciliation: 'langgraph_supervisor' was chosen for its handoff-tool factory, not its supervisor topology. Spec page VT-27 wording is now partly stale on architecture (talks about create_supervisor wiring); code as shipped is authoritative.
