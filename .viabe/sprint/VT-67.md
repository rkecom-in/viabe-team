# VT-67 — L2 retrieval contract

## Parent task
VT-KnowledgeArchitecture.

## Concept reference
- `concept-team.md` §9.2 (L2 retrieval is the agent's recent-history feed; bounded; cursor-paginated).
- `concept-team-pillars.md` Pillar 1 (orchestrator builds context; agent receives; L2 retrieval is invoked at context-build time, NOT inside the agent).
- `concept-team-pillars.md` Pillar 8 (one retrieval contract; not per-event-type queries scattered across the codebase).

## Dependencies
- VT-7.3 Done (L2 schema + writer).
- VT-SalesRecovery-Agent VT-4.3 Done (`SalesRecoveryContext` builder consumes this; the L2 portion of the bundle comes through here).

## What CoderC must produce
The L2 retrieval contract. Functions the orchestrator calls to fetch episodic memory for a tenant. Used during context-bundle construction (VT-4.3) and by the orchestrator's own decision logic.
1. **Retrieval module** at `apps/team/src/kg/l2_query.py`:
	- `recent_events(tenant_id, limit=20, since=None, event_types=None) -> list[EpisodicEvent]`. Returns most-recent N events filtered by optional time window and event type allowlist.
	- `events_for_entity(tenant_id, referenced_entity_type, referenced_entity_id, limit=20) -> list[EpisodicEvent]`. Returns events about a specific customer / campaign / etc.
	- `events_in_window(tenant_id, start, end, event_types=None) -> list[EpisodicEvent]`. Returns events in a date window.
	- `event_count(tenant_id, event_types=None, since=None) -> int`. Aggregate count without fetching rows.
2. **Bounded results**:
	- All retrieval functions cap at 200 events per call. Higher caps require Type 2 governance (cost: more events = larger context).
	- Default limits chosen to fit within VT-4.3's 8K-token context budget when serialized.
3. **Soft-delete handling**:
	- All retrievals exclude `deleted_at IS NOT NULL` rows by default.
	- Override flag `include_deleted=True` for ops use only; not exposed to agent paths.
4. **Cross-tenant scoping**:
	- Every function takes `tenant_id` and asserts its presence in the WHERE clause via the typed wrapper (VT-8.1).
	- Cross-tenant attack test: malicious tenant_id substitution returns empty.
5. **Summary projection**:
	- For agent context use, retrievals return `EpisodicEvent` with `summary` populated; the bundle builder concatenates summaries when token-budget pressure forces compression.
	- For full-detail use (orchestrator's decision logic), `payload` is also returned. Cost-aware callers ask for `summary_only=True` to drop payload from response.
6. **Tests** at `apps/team/tests/kg/test_l2_query.py`:
	- Each function returns expected results.
	- Bounded cap enforced; attempts to exceed 200 return error or are clamped (decided in spec; default: clamp + log).
	- Cross-tenant attack: empty results.
	- Soft-delete excluded by default; included with override.
	- Performance: 100K-event synthetic tenant, all 4 functions <100ms.
	- `summary_only` reduces serialized response size.

## Architectural rules to enforce in this subtask
- Pillar 1: this module is invoked by the orchestrator during context build, NOT by the agent directly. CodeX rejects PRs that import this from `apps/team/src/agent/`.
- Pillar 8: one retrieval API; no scattered ad-hoc queries. CI grep flags raw SQL queries against `episodic_events` outside this module.
- Bounded results enforced. CodeX rejects PRs raising the cap silently.
- `summary_only` default for agent paths to control token spend.
- All queries use the typed wrapper for cross-tenant safety.

## Acceptance criteria
- Retrieval module exists with 4 functions.
- All functions cap results at 200; cap config-driven.
- Cross-tenant attack passes.
- Soft-delete defaults respected.
- Performance: 100K-event tenant, <100ms per function.
- `summary_only` mode tested.
- mypy --strict passes.
- CI lint enforces no raw SQL on `episodic_events` outside this module.
- PR title references `(VT-KnowledgeArchitecture)`; reviewed by CoderX.

## Out of scope
- L2 writes (VT-7.3).
- L3 patterns (VT-7.5).
- Event search by free-text (Phase 2; full-text indexing on summary is a future ask if useful).
- Aggregations beyond count (Phase 2; e.g., "campaigns per week" is computed by the orchestrator, not the retrieval).
- Pagination cursor (Phase 1 uses limit + offset; cursor pagination is Phase 1.5 if perf becomes an issue).
