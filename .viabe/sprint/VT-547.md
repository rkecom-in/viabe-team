# VT-547 — B1-part-2a: outbound_failure alert on owner-notification failure

## Gap
VT-524 records an owner-notification that fails delivery as `failed`/`failed_incident_open`, but **nothing alerted anyone** — the row just sat flagged. The `outbound_failure` critical trigger (VT-202 `alerts/triggers.py:42`) was **declared but never constructed anywhere** (recon-confirmed dead detector). So a welcome that Meta declines (63049) left the owner silently un-notified with no human in the loop.

## Built (0 migrations)
- On a NEW `failed` transition in `record_owner_notification_delivery` (a real row flipped — a terminal-safe re-callback that no-ops does NOT re-fire), build a `Trigger(outbound_failure, critical)` and `dispatch_alert(...)`. This is the declared-but-never-fired detector, finally wired.
- Uses the existing alert pipeline verbatim: `dispatch_alert` persists to `tenant_alerts` + routes telegram/email, and is **dev-routed** (a dev/canary tenant reaches only the dev bot, never pages Fazal). The message_sid is an opaque Twilio id (not PII) and `dispatch_alert` PII-scrubs the text.
- **Fail-soft in its own guard** — an alert failure never touches the ledger write or the inbound webhook (Pillar-1: the primary op wins). `_update_delivery` now returns rowcount so the alert only fires on a genuine first-failure.

## Verify
ruff clean; 9/9 owner_notification tests on fresh DB — the failed/undelivered callback fires exactly one `outbound_failure` critical alert carrying the message_sid; delivered does NOT fire; a duplicate failed callback fires ONCE (rowcount-0 no-op); an unknown sid does NOT fire.

Auto-retry-send + alternate-WA-path stages (need send-safety design + Fazal sign-off — no real send now). The durable resolvable **owner-contact incident table** + reviewer-takeover lifecycle pair with B6 (VT-529, recovery/safety/ops). The run terminal-outcome enum + `silent_terminal` detector = B1-part-2b (touches load-bearing `pipeline_runs`, own pass).
