# CL-107 — Error/failure handlers do not message the founder

Principle: Viabe Team is the smart system running the founder's business. The founder should not be troubled with system error messages. Error paths are our problem; we log, retry, and self-heal.

Applied to VT-3.3c handler scope:

SENDS A TEMPLATE (founder-facing):
- opt_out_handler → team_opt_out_confirmation (legal requirement, not optional)
- dsr_handler → team_dsr_acknowledgment (DPDP requirement, not optional)

DOES NOT SEND (internal-only logging + retry):
- template_error_handler → log structured failure to pipeline_steps / Logfire when VT-122 ships. NO founder message. DBOS retry policy handles the underlying failure.
- status_callback_failure_notice (i.e., Twilio status_callback with state=failed) → log structured failure. NO founder message. Failed delivery becomes a retry candidate for VT-3.6 framework.
- status_ping_handler → TBD (Fazal to confirm: inside 24h Twilio session window? If yes, free-form reply. If no, need a new template).

Unused templates (registered but not wired in VT-3.3c):
- team_agent_stuck — reserved for future founder-escalation case if/when one is identified. Not for error noise.
- team_welcome — VT-9.2 (sign-up completion)
- team_weekly_approval — weekly approval flow

Net VT-3.3c scope:
- 2 templates send via Twilio SDK (opt_out + dsr)
- 5 handlers rewired: 2 send templates, 3 log internally only
- All 5 handler return contracts become truthful (send_result: SendResult | None)
- TODO VT-3.3 comments removed from all 5 handlers
