# CL-215 — PostgresSaver.setup() NOT idempotent

=== WHAT WAS WRONG ===
CL-213 Step 11(a) resolution: 'setup_checkpoint_tables.py = thin wrapper around init_substrate(database_url) from graph.py... Idempotency inherited from PostgresSaver.setup() + existing _setup_checkpoint_rls; verified incidentally by Landmine 2 run.'

The idempotency-inheritance claim is wrong. PostgresSaver.setup() (langchain-postgres) is NOT idempotent when called against a populated checkpoint_migrations table. It unconditionally inserts into checkpoint_migrations at the end of its setup routine, which fails with psycopg.errors.UniqueViolation on the primary key (v) if a prior call already populated the row.

=== HOW IT WAS CAUGHT ===
Landmine 2 worker run #2 (the second invocation against the same DB) crashed inside the @DBOS.step body with:
  psycopg.errors.UniqueViolation: duplicate key value violates unique constraint 'checkpoint_migrations_pkey'
  DETAIL: Key (v)=(1) already exists.
at the call site: checkpointer.setup() inside _langgraph_step_resume_worker.py.

First pytest run succeeded — fresh DB, no prior migration row. Second direct worker run failed — the prior run's migration row blocked re-insert. The 'idempotent' comment on the call site was misleading: setup() creates tables if absent, but the migration tracking insert is unconditional.

=== ROOT CAUSE ===
Clau assumed PostgresSaver.setup() was idempotent based on the langchain-postgres docstring framing ('create checkpoint tables'). Did not Context7-validate the actual idempotency surface against shipped library version. CL-178 violation — external library symbol claim not validated against shipped version.

This is a sub-case of CL-178's class: not signature drift, but BEHAVIOR drift between docstring framing and actual implementation. Symbol exists, signature is right, but the idempotency property the brief assumed is not the actual property the implementation has.

=== REVISED STEP 11(a) RECOMMENDATION ===

Replace CL-213's 'α: thin wrapper, idempotency inherited from PostgresSaver.setup()' with:

α′: thin wrapper around init_substrate(database_url) from graph.py, AFTER verifying init_substrate's actual idempotency surface. Two sub-questions to resolve at Step 11(a) write time:

  (i) Does init_substrate (graph.py) ALREADY handle the duplicate checkpoint_migrations row case? If yes, how (try/except, table-exists check, conditional INSERT)?

  (ii) If init_substrate does NOT handle it, the script must wrap PostgresSaver.setup() in try/except psycopg.errors.UniqueViolation.

Do not write the script blind. Verify init_substrate's idempotency first.

=== INCIDENTAL EVIDENCE FROM LANDMINE 2 RUN ===
DBOS's own migration applier (_migration.py) IS idempotent in the sense that matters: 'Migration 10 skipped, primary key already exists' on the second run. langchain-postgres's PostgresSaver.setup() does NOT have the same defensive check. This is a library-quality asymmetry, not a usage error.

=== STANDING IMPLICATION ===
CL-178's Context7 pre-brief gate should extend to BEHAVIORAL properties (idempotency, thread-safety, transaction boundaries), not just signatures. Folding into mental model.

=== AFFECTED ENTRIES ===
CL-213 Step 11(a) resolution stands as a recorded decision but the 'idempotency inherited' rationale is struck.
