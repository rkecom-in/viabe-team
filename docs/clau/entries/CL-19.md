# CL-19 — Step records use typed envelopes (declared input/output fields per step) not ful

Step records use TYPED ENVELOPES, not full row snapshots. Each step declares what fields it reads (input_envelope) and what fields it produces (output_envelope). Only those declared fields are persisted into the step_record at execution time. This is cheaper than full-row snapshots, immune to later-step mutations (because the actual values consumed are frozen at write), and forces engineers to be explicit about each step's data contract. Schema example: a step that picks customers gets persisted with `input_envelope: {tenant_id, dormancy_threshold_days, max_candidates}` and `output_envelope: {candidate_customers: [{customer_ref, last_txn_date, total_spend, last_message_sent_at}], filter_applied}`. NOT the full customers table. Implication for CoderC: every MCP tool + every state transition declares a Pydantic envelope schema; CodeX rejects PRs without explicit envelope schemas.
