# VT-598 — the confirmation gate for the whole capability objective

The objective is NOT met until this row passes green on deployed dev. No manual user testing —
everything server-side via convo_harness.

## 1. Harness upgrades
- Per-step FULL multi-line reply captured into the run report (the 6-deploy-phantom lesson:
  never a first-line grep anywhere in the toolchain).
- New assert `assert_not_d1`: for steps marked as real questions/asks, the reply must NOT be the
  D1 fallback line ("Got it — I'm on it…" / hi variant). A green run with D1 replies is a FAIL.
- Scenario runner emits a machine-readable transcript bundle per scenario (for the judge).

## 2. Scenario pack (beyond the existing 8)
Conversation (P1 regression + permutations): onboarding hostile/skeptical; topic-switch mid-flow
and return; consent natural-language variants; integration re-sent link + mind-change
("actually not Shopify, Amazon"); multi-field single message; repeats/impatience; stop then
resume; status/plan/edit requests; Hinglish variants.
Delegation (P2 — the new bar). Fazal clarification 2026-07-04: the objective's "sub-agents" =
the business-function agents (Onboarding, Integrations, Sales Recovery, Marketing, Accounting…).
Ground truth (roster.py): THREE are wired spawn-able specialists today — sales_recovery_agent,
integration_agent, onboarding_conductor; Marketing/Accounting/Finance/Sales/Tech/CostOpt are
manager-held advisory TOOL lanes, not autonomous mini-managers. The pack therefore validates:
- Sales Recovery: seeded lapsed cohort → "make me a plan to win back my lapsed customers" → SR
  spawn → IN-CHAT plan summary + approval prompt (real content asserts, not just non-silent);
  empty-cohort honest reply (no fabrication, no D1); analytical routing (VT-595: "which of my
  customers stopped buying?" must NOT terminate as edge_case:status_query; "how many customers
  do I have?" MUST); approval "yes" → resume → campaign_execute (mocked sends, bogus numbers) →
  owner outcome surface; second plan ask while one approval pending (queue_busy body, single
  message, no false promise).
- Integration agent: connector ask mid-conversation → correct specialist handoff → link minted
  from in-context store address (no re-ask).
- Onboarding conductor: journey progression + off-script handling (overlaps P1 scenarios).
- Marketing/Accounting asks: the manager must respond intelligently VIA its lane tools or
  honestly state capability limits — asserted for honesty + usefulness; the absence of a true
  marketing/accounting mini-manager specialist is a DOCUMENTED GAP in the final report (roster
  expansion = next phase, one SpecialistSpec each per VT-465 — not silently faked as passing).

## 3. Judge
Opus LLM-judge, transcripts BATCHED per call: rubric context-retention / intent / honesty /
helpfulness / progression, threshold ≥4/5 each. Hard asserts run first in code; the judge sees
survivors. Judge verdicts + reasoning recorded in the run report.

## 4. Exit criteria
- Full pack green (hard asserts) on deployed dev at the final sha.
- Judge ≥4/5 on every scenario transcript.
- Zero real sends (send-guard clean, no SM/MM sids on bogus tenants).
- Findings→fixes loop documented; regression: the pack re-runs before any future conversational
  change ships.
- Consolidated report: per-scenario PASS/GAP + evidence + what changed this session and why
  (feeds the P4 ping + CL entry).
