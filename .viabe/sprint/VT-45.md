## What CoderC must produce
The `send_whatsapp_template` MCP tool. Sends a Meta-approved WhatsApp template message via Twilio. No 24-hour window restriction. Template ID + parameters validated against the approved registry before send.
1. **Tool implementation** at `apps/team/src/agent/tools/send_whatsapp_template.py`:
	Input schema:
	- `customer_id: UUID`
	- `template_id: str` — must be in `apps/team/config/approved_templates.yaml`. Failed lookup returns error envelope.
	- `template_params: dict[str, str]` — keys must match the template's parameter spec; values are substitution strings.
	- `language: Literal['en', 'hi']` — must match a language the template was approved in.
	- `idempotency_key: str` — same pattern as VT-5.6.
	Output schema: same shape as VT-5.6 (`status`, `message_sid`, `customer_id`, `sent_at`, `error_envelope`).
2. **Template validation**:
	- On every call, load `approved_templates.yaml` (cached 60s in-process).
	- Validate `template_id` exists; otherwise `error_envelope.code='unknown_template'`.
	- Validate `language` is supported by the template; otherwise `error_envelope.code='unsupported_language'`.
	- Validate `template_params` keys match the template's required parameters; otherwise `error_envelope.code='missing_template_params'` with the specific missing key.
	- Validate parameter VALUES against per-parameter constraints (e.g., max length, no special chars that break Twilio template rendering).
3. **Twilio invocation**:
	- Use Twilio's Content API with the approved template's content SID + variables map.
	- Phone resolution + cross-tenant scoping identical to VT-5.6.
	- Persist `campaign_messages` row with `template_id`, `template_params` (JSONB), `send_status='sent'`.
4. **Idempotency + rate limits**: same patterns as VT-5.6 (reuses `send_idempotency_keys` table). Per-tenant rate limit higher: 5000 templates per day (templates are the primary outbound channel, free-form is the exception).
5. **Tests**:
	- Happy path: known template + valid params + known customer → sends → SID returned.
	- Unknown template: `error_envelope.code='unknown_template'`.
	- Missing param: `error_envelope.code='missing_template_params'` with the specific key.
	- Unsupported language: `error_envelope.code='unsupported_language'`.
	- Idempotency, rate limits, cross-tenant: same coverage as VT-5.6.
	- Cache invalidation: changing `approved_templates.yaml` is picked up within 60s.

## Architectural rules to enforce in this subtask
- Pillar 2: pure transport. Template content is pre-approved; tool does NOT modify or reinterpret.
- Pillar 7: template wording was reviewed during Meta approval. Last-minute edits are forbidden — if a template needs changing, it goes through Meta re-approval (VT-13.3) and the YAML is updated.
- The approved-templates registry is the source of truth. Tools never invent template IDs.
- No 24-hour window check on this tool. Templates are designed for outside-window sends; that's their entire purpose.
- Per-tenant rate limit (5000/day) is conservative but reflects realistic Phase 1 volume; raising requires Type 2.
