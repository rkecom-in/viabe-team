# VT-344 — owner self-serve DSR-delete (request + grace), post-launch

VT-341 (#359) wired self-serve EXPORT but, per Fazal's ruling (2026-06-06), did NOT wire
self-serve DELETE — an INSTANT IRREVERSIBLE synchronous purge from an owner button is too
hot. At launch the settings page shows "contact us to delete" and Fazal/ops runs
admin/dsr/delete out-of-band.

When self-serve delete is wanted, it must be the REQUEST+GRACE model, NEVER instant-from-a-
button:
1. Owner POSTs a deletion REQUEST (POST + confirm + CSRF, like the export proxy).
2. The request is acknowledged + a GRACE WINDOW opens (e.g. 7/14 days — DPDP-compatible);
   the owner gets a "your data will be deleted on <date>; reply CANCEL to stop" ack.
3. After the grace window (a scheduled sweep), the reviewed purge runs (the existing
   purge_tenant_data) — cancellable until then.
4. Double-confirm UX + CSRF (the GET→POST + same-origin discipline from VT-341).
