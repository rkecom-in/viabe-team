# CL-80 — Notion-vs-shipped drift report

Drift findings from comparing shipped code at 4106e29 against Notion subtask bodies for VT-19 (base schema), VT-20 (RLS), VT-24 (VT-3.1), VT-25 (VT-3.2), VT-26 (VT-3.3a+3.3b), VT-31 (VT-3.8). Notion subtask IDs use the flat VT-N scheme; project-level labels VT-3.x are derived (no direct Notion ID).

MAJOR SCHEMA DRIFT (VT-19, VT-20):
The Notion VT-19 'base schema' subtask specified 8 tables: tenants, customers, transactions, campaigns, campaign_messages, attributions, pipeline_log, signup_holding. Shipped at 4106e29: tenants, phase_transitions, subscriptions, razorpay_webhook_events, pipeline_runs, pipeline_steps, phone_token_resolutions, privacy_audit_log, env_config, dsr_tickets, twilio_inbound_events, rate_limit_buckets. ALMOST COMPLETELY DIFFERENT SET. The shipped schema serves the orchestrator's needs; the spec'd schema is the domain layer (customers/transactions/campaigns/attributions) which hasn't been built. Two possible readings: (a) deliberate execution-order pivot to ship the orchestrator path first, (b) execution gap where VT-Foundation was redefined informally. No Decision entry in Clau_Session_Log documents this pivot. Recommend creating one and updating VT-19's body to reflect the new schema, OR explicitly scheduling a separate VT-19-domain subtask.
