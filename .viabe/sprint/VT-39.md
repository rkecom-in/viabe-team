## What CoderC must produce
The MCP tool-contract framework that every individual tool subtask (VT-5.2 through VT-5.13) implements against. Plus the test harness every tool uses. Plus a Type 2 governance decision on whether `draft_message_variants` (VT-5.13) ships in v1 or is deferred.
1. **Framework module** at `packages/team-shared/python/team_shared/mcp/framework.py`:
	- `MCPTool` abstract base class with required methods: `name`, `description`, `input_schema`, `output_schema`, `execute(ctx, inputs)`, `is_llm_backed()` (returns False by default; LLM-backed tools override and provide a written rationale).
	- `ToolContext` dataclass: `tenant_id` (UUID, derived from invocation, never accepted from agent), `run_id`, `agent_id`, `parent_tool_call_id` (for nested calls), `cost_budget_remaining_paise`, `wallclock_remaining_ms`, `db_handle` (typed wrapper from VT-8.1 once available; bridge query layer for now).
	- `ToolResult` dataclass: `status: Literal['ok', 'error', 'rate_limited', 'unauthorized', 'timeout']`, `data: dict | None`, `error: ErrorEnvelope | None`, `tokens_used`, `cost_paise`, `latency_ms`, `metadata` (structured key-value, never a free-text dump).
	- `ErrorEnvelope` typed: `code` (enum), `message` (max 200 chars, no PII), `retry_after_ms` (optional, for rate_limited).
2. **Validation layer**:
	- Input schema validation runs BEFORE `execute()`. Invalid inputs return `ToolResult(status='error')` with structured envelope; `execute()` is never called with invalid inputs.
	- Output schema validation runs AFTER `execute()`. Invalid outputs trigger an immediate error log + return error envelope to the agent; the agent never sees malformed output.
	- Schemas use Pydantic models (consistent with VT-4.6).
3. **Tenant scoping enforcement**:
	- `ToolContext.tenant_id` is set by the orchestrator at the dispatch boundary (VT-3.4). Tools READ it. Tools NEVER accept `tenant_id` as an input field.
	- The framework's input-schema validator REJECTS any tool whose input schema includes a `tenant_id` field. CodeX rejects PRs that try.
	- Every DB read inside `execute()` passes `ctx.tenant_id` to the typed wrapper / scoped query.
4. **Telemetry**:
	- `pipeline_log` event emitted on every `execute()` call: `tool_call_started` with name + tenant_id + run_id + input hash; `tool_call_completed` with status + tokens + cost_paise + latency_ms; `tool_call_failed` on error.
	- LLM-backed tools (those overriding `is_llm_backed()` to True) MUST include the model name + token usage in the metadata of every event.
5. **Test harness** at `packages/team-shared/python/team_shared/mcp/test_harness.py`:
	- `run_tool_test(tool_class, fixtures)` standardized helper. Runs deterministic positive fixture, negative-path fixture (wrong tenant, malformed input, dependency failure simulation), and confirms `ToolResult` shape.
	- Every tool subtask MUST use this harness. CI grep enforces the import.
6. **Tool registry** at `apps/team/src/agent/tool_registry.py`:
	- Central registry mapping tool name → tool class. Populated by VT-5.2–5.13 as those tools land.
	- The agent SDK's tool list is built from this registry at agent construction time (filtered by `tool_subset` from VT-3.8 specialist registry).
	- LLM-backed tools are flagged in the registry; the registry's `validate_subset()` method returns the list of LLM-backed tools in any given subset for review/audit purposes.
7. **`draft_message_variants` decision**:
	- This subtask is the place where the v1 inclusion decision is locked. Run the analysis: does Phase 1 need the agent to produce multiple message variants (Pillar 2 rationale: drafting is a reasoning task, but the agent already drafts the campaign plan; an LLM-backed `draft_message_variants` tool would produce alternative phrasings). Decision options:
		- **(a) Include in v1**: rationale documented in `docs/team/llm-backed-tools-rationale.md`; VT-5.13 is in-scope.
		- **(b) Defer**: rationale documented (e.g., "the agent's own message-template-id selection plus parameter substitution covers v1; variant generation is a Phase 1.5 concern when we have data on which variants land better"); VT-5.13 is moved to Deferred.
	- The decision is Type 2 governance. Fazal signs off. The decision lands as a Notion comment on VT-5.13's page setting Status to either `Backlog` or `Deferred` accordingly.
8. **Documentation**:
	- `docs/team/mcp-tool-framework.md`: framework overview, contract, how to write a new tool, when to use an LLM-backed tool (rare, justified), telemetry contract, error envelope.
	- `docs/team/llm-backed-tools-rationale.md`: enumerates which tools may be LLM-backed and the rationale for each. Phase 1 starts with `self_evaluate` and `classify_owner_message` as locked; `draft_message_variants` decided here.
9. **Tests** in `packages/team-shared/python/team_shared/mcp/tests/`:
	- Framework: a synthetic tool subclass passes/fails contract correctly.
	- Tenant-scoping enforcement: a tool that tries to accept `tenant_id` in its input schema fails framework registration.
	- Output validation: a tool returning malformed data returns the error envelope to the agent.
	- Test harness self-test: a synthetic tool runs through the harness with positive + negative fixtures.

## Architectural rules to enforce in this subtask
- Pillar 2: the framework is where the LLM-backed boundary is enforced. `is_llm_backed()` defaults False. Overriding requires written rationale in code comment + the rationale doc. CodeX rejects LLM-backed tools without rationale.
- Pillar 3: `tenant_id` derivation is structural. The framework REFUSES tools that accept tenant_id as input. This is non-negotiable.
- Pillar 8: framework first, tools second. CodeX rejects tool implementations that bypass the framework (e.g., importing from `mcp/framework.py` partially or reimplementing telemetry).
- The framework is in `packages/team-shared/`, not in `apps/team/`. Tools live where they're consumed; the framework is shared infrastructure.
- Output schema validation is the agent's defense against tool drift. A tool that updates its output without updating its schema breaks validation immediately.
- The framework does NOT enforce business logic. It enforces structural contract. Business logic lives in tool implementations (VT-5.2–5.12).
- The `draft_message_variants` decision is locked here. Re-litigating it in VT-5.13 requires Type 2 governance.

## Acceptance criteria
- Framework module exists with `MCPTool`, `ToolContext`, `ToolResult`, `ErrorEnvelope`.
- Input/output schema validation runs before/after `execute()`.
- Tenant-scoping rejection: a synthetic tool with `tenant_id` in input schema fails registration; test confirms.
- `pipeline_log` events emitted on every tool call (started, completed, failed).
- Test harness runs deterministic positive + negative fixtures; tool subtasks consume it.
- Tool registry exists; supports `validate_subset()` for LLM-backed audit.
- `docs/team/mcp-tool-framework.md` documents the contract.
- `docs/team/llm-backed-tools-rationale.md` documents the 2-or-3-tool rationale; `draft_message_variants` decision recorded.
- VT-5.13 page in Notion has its Status set per decision (Backlog if approved, Deferred if not).
- CI grep confirms tools use the test harness, do not bypass framework, do not accept tenant_id input.
- mypy --strict passes on the framework module.
- PR title references `(VT-MCP-Tools)`; reviewed by CoderX (mandatory — framework defines contracts for 12 downstream subtasks); Fazal personally approves the `draft_message_variants` decision.

## Out of scope
- Individual tool implementations (VT-5.2 through VT-5.13).
- Typed wrapper layer (VT-PrivacyArchitecture VT-8.1; framework references it).
- Agent SDK tool registration logic (lives in `apps/team/src/agent/`; this subtask provides the registry, that subtask wires it).
- Per-tool rate limiting (each tool implements its own; framework provides hooks).
- Tool versioning (Phase 1 uses single-version tools; v2 schema migration is Phase 1.5).
- Performance optimization of the framework (Phase 1 prioritizes correctness; profiling is later).
