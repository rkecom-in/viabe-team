# VT-450 — Entity-match: surface the found company when no GSTIN (don't say "couldn't find")

## Problem (Fazal, live e2e 2026-06-28)
Discovery DOES find the business via the GBP/registry leg (e.g. RKeCom Services Pvt Ltd, with phone), but that
candidate has **no `candidate_gstin`** (GBP carries none; the web/SERP GST leg returns 0 for small biz whose GSTIN
isn't SERP-indexed). The current UI then shows "couldn't find your business" — WRONG: we DID find the company.

## Requirement
When entity-match returns a candidate WITH a name (GBP/registry) but NO GSTIN, the screen must:
- SHOW the found company: "We found **<trade_name>** but couldn't find a GST number for it."
- Offer BOTH recovery paths:
  - (a) **Change the company name** → re-search (the typed name may be off).
  - (b) **Enter your GST number** → the manual-GSTIN path (already primary post-park).
- NOT "couldn't find your business."

## Scope (team-web)
- `app/(marketing)/team/signup/entity-match-step.tsx`: render the GBP/registry candidate (name + detail + phone)
  even when `candidate_gstin` is null; the "couldn't-find" empty-state only when discovery returned ZERO candidates.
- Copy bilingual EN/HI. Keep the manual-GSTIN fallback as path (b).
- The CIN-confirm affordance stays parked (NEXT_PUBLIC_ENABLE_PAN_IDENTIFY off) — this is purely the name+recovery UX.

## Acceptance
A discovery result of [{source:gbp, trade_name set, candidate_gstin:null}] renders "We found <name>…" + the two
buttons; zero candidates renders the existing empty-state. Plan-first → Cowork gate (small).
