# VT-670 — no gate against re-arming a campaign that already sent

## Symptom (j01, judge wrong_action)
After a win-back campaign was approved + SENT ("I sent it to 8 customers"), a fresh identical imperative re-mints a NEW campaign + re-arms a NEW approval for the same 8-customer cohort — risks a confusing duplicate send. Judge: wrong_action ("re-presenting a fresh approval for what is effectively the same campaign… ignores the true system state").

## Root cause (from vt667-rootcause2)
No recency/"already-sent"/cooldown gate exists. campaign_first_contact.py has only is_campaign_plan_imperative (trigger) + campaign_cohort_is_empty (empty honesty). The sole state gate is `not has_active_task` (triage_seam D3 :320 / VT-657 :368-372); TASK_ACTIVE excludes terminal states, so a COMPLETED campaign task is not active → a fresh imperative re-mints. The gate belongs in _dispatch_campaign_first_contact (triage_seam.py:132): before minting, check for a recent proposed/sent campaign to the same cohort → answer status ("your win-back went out N hours ago to M customers — want a DIFFERENT one?") instead of a blind re-arm.

## Fix direction
Deterministic recency check in _dispatch_campaign_first_contact: recent (e.g. <24-48h) sent/proposed campaign for the tenant → do NOT re-mint; route to a status answer + offer a genuinely-new campaign. Money-adjacent (prevents duplicate sends). Gate: j01 x3 clean (no re-arm after send).
