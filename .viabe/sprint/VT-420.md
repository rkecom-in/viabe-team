## Contract (CC-found 2026-06-24 via the VT-418 adversarial idempotency audit). PRE-EXISTING, not introduced by VT-418.

The win-back send path has a bounded double-send residual: the Twilio `messages.create` call
(`send_whatsapp_template.py:576`) and the autocommit idempotency-ledger `'sent'` INSERT (`:580/583`) are NOT in
one transaction (the pool is `autocommit=True`). A process crash in the window AFTER Twilio delivers but BEFORE
the `'sent'` row commits leaves the draft still `'drafted'` + no ledger hit → on recovery it **re-sends** (a
double-charge / double-message). Sub-millisecond, single-statement window; inherited from the VT-45 send path
(documented at `customer_send.py:35-37`), shared by L3 + the new L2 driver (VT-418) — NOT a VT-418 regression.

## Fix (when reached — money-send reliability)
Close the window. Options (plan-first): (a) pull `messages.create` + the ledger `'sent'` INSERT into ONE
transactional unit; OR (b) a Twilio-side idempotency key (Twilio `messages.create` supports an idempotency
header) so a re-send is deduped vendor-side; OR (c) a pre-send `'sending'` ledger marker so recovery can detect
an in-flight send. Recommend (b) or (a). Canary: simulate a crash in the window → recovery does NOT double-send.

## Gate
RISK (money-send reliability). Plan-first → Cowork gate; idempotency-verified. NOT self-merge. Priority: before
heavy win-back send volume (not blocking the VT-396 single-recipient e2e — the window is sub-ms + the e2e is one send).
