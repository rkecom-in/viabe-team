## What CoderC must produce
The L2 episodic memory schema. A per-tenant append-only time-ordered log of agent-relevant events: campaign decisions, owner replies, customer milestones, attribution outcomes. This is the agent's "what happened recently in this tenant" feed, distinct from L1's "what is currently true."
1. **Schema** at migration `021_l2_episodic.sql`:
	- Table `episodic_events`:
		- `id` UUID PK
		- `tenant_id` UUID NOT NULL
		- `event_type` TEXT NOT NULL (enum: `campaign_proposed`, `campaign_approved`, `campaign_rejected`, `campaign_sent`, `attribution_closed`, `customer_dormant_threshold_crossed`, `customer_high_value_threshold_crossed`, `owner_message_received`, `agent_dispatch_completed`, `agent_dispatch_terminated`, `phase_transitioned`, `clarification_resolved`)
		- `summary` TEXT (agent-consumable 1-2 sentence summary of the event; templated, not LLM-generated)
		- `payload` JSONB (structured event-specific fields)
		- `referenced_entity_type` TEXT (e.g., `customer`, `campaign`, `transaction`)
		- `referenced_entity_id` UUID (the entity this event is about)
		- `occurred_at` TIMESTAMPTZ NOT NULL
		- `created_at` TIMESTAMPTZ DEFAULT now()
		- INDEX `(tenant_id, occurred_at DESC)` for the agent's primary read pattern.
	- RLS policy: tenant-scoped read + write via `auth.jwt()->>'tenant_id'`.
2. **Pydantic models** at `apps/team/src/kg/l2_types.py`:
	- `EpisodicEvent` matches the table.
	- Per-event-type structured payloads: `CampaignProposedPayload`, `OwnerMessageReceivedPayload`, etc. — each is a Pydantic model.
	- The 12 event types are an enum; adding a new type is a Type 2 governance change because the agent's prompts may need updating to consume it.
3. **Writer module** at `apps/team/src/kg/l2_writer.py`:
	- `record_episodic_event(tenant_id, event_type, payload, referenced_entity_type, referenced_entity_id, summary, occurred_at) -> UUID`.
	- Validates payload against the per-type model.
	- Generates summary from a Python f-string template (NOT LLM). Templates per event_type live in `apps/team/config/l2_summary_templates.yaml`.
	- Writes to table; emits a `kg_event` of type `episodic_event_recorded` so L3 (later subtask) can consume.
4. **Retention**:
	- L2 events are retained for 18 months. After that, soft-deleted (`deleted_at` set; row stays for audit). Archived rows are excluded from agent reads.
	- Retention is config-driven (`apps/team/config/l2_retention.yaml`).
5. **Tests** at `apps/team/tests/kg/test_l2_episodic.py`:
	- Each event type writes correctly with validated payload.
	- Summary templating: synthetic event → expected templated summary.
	- Cross-tenant: tenant_A's events not visible to tenant_B (RLS).
	- Retention: 18-month-old event soft-deleted; agent reads exclude it.
	- Index performance: 100K-event synthetic tenant; recent-N read in <100ms.

## Architectural rules to enforce in this subtask
- Pillar 3: RLS on `episodic_events` is the structural defense. CodeX confirms.
- Pillar 4: summary is templated, not LLM. Adding a new event_type requires adding a template; CodeX rejects PRs that skip the template entry.
- Pillar 8: one schema, one writer module. Multiple emitters (orchestrator, ingestion, MCP tools) all call the same writer.
- Adding a new event_type is Type 2 governance because the agent's prompt may need updating. Documented at `docs/team/l2-event-types.md`.
- Soft-delete on retention; never hard-delete from L2 in Phase 1 (audit trail).
