# CL-266 — VT-50 blocked on VT-5.1: Path 1 (brief VT-5.1 first). + Dependency-chain ground-

## Trigger

Third pre-flight dependency surprise in a row in Sprint 2:
- VT-32 not on main when VT-35 needed it
- VT-36's tool seam not real when CL-260 said VT-36's dependency was satisfied
- VT-5.1 (MCPTool base surface) not on main when VT-50 needs it

The 2026-05-12 Sprint 2 respec dependency graph does not match what has actually been built and merged. The mismatch is being discovered one STOP at a time at pre-flight, which is the most expensive place to find it.

## Decision A: VT-50 blocker resolution — Path 1

VT-50 (self_evaluate as MCP tool) is blocked on VT-5.1 (MCPTool base surface). Resolution: brief VT-5.1 first, land it, then VT-50 subclasses cleanly.

Rejected Path 2 (build minimal MCPTool base inside VT-50) — a single consumer would unilaterally freeze the API shape of a Critical foundational surface whose owning subtask is VT-5.1. Same anti-pattern as the VT-32/VT-35 fusion rejection.

Rejected Path 3 (ship VT-50 with no MCPTool base, SelfEvaluator Protocol only) — defensible scope boundary, but VT-5.1 is Critical and Sprint 2 needs it regardless. The page's Pillar 2 framing ('one of three permitted LLM-backed tools') indicates self_evaluate is agent-callable mid-reasoning, not gate-only — so it genuinely needs MCP registration. Deferring a Critical foundational surface risks the same surprise resurfacing at the next tool subtask.

## Decision B: Protocol reconciliation (records CL-265)

The lean SelfEvaluateVerdict on main (outcome + four str|None feedback fields) is the contract VT-50's tool must produce. The richer 2026-05-04 page schema (CategoryFeedback objects, overall_severity, model_used/tokens_used/cost_paise) is NOT part of the gate's consumed contract — cut from VT-50's return type. Telemetry (tokens_used, cost_paise) may still be logged internally by the tool; it does not ride on the Protocol return.

## Decision C: Dependency-chain ground-truth pass — approved, pulled earlier

A one-time bounded ground-truth pass over the remaining Sprint 2 SR-agent backlog subtasks (~6-8 rows): for each, confirm whether its named dependencies are actually merged on main. Output is a corrected canonical Sprint 2 sequence.

This is NOT a violation of the 'one audit per sprint' operating mode and NOT pre-hunting. Pre-hunting = fishing for conflicts inside a single brief. This is a scoped correction of a structural artifact (the dependency graph) now proven unreliable by three consecutive failures. The respec graph being wrong is an audit-class problem worth pulling earlier because every brief until then inherits the error. Scope is tight: state-correction only (does the graph match main, per row), NOT code-quality or design review. Result logged as one Decision entry that becomes the canonical Sprint 2 sequence, retiring the 2026-05-12 respec graph as reference.

Run order: dependency pass first (discrete short pass) → then VT-5.1 brief as item 1 of the corrected sequence → VT-50 after.

## Decision D (added 2026-05-20): Board Status reconciliation folded into the same pass

The ViabeTeam_Sprint board has stale Status fields — subtasks with merged PRs still show Backlog/To Do/In Progress instead of Done. Same root problem as Decision C (board vs. main reality), so it is folded into the same ground-truth pass rather than run separately.

Scope addition for the pass: the session running the dependency pass ALSO, in the same sweep, reconciles Status. Mechanism:
1. `gh pr list --state merged --limit 100` to get the authoritative list of merged PRs.
2. For each merged PR, map it to its ViabeTeam_Sprint subtask row (PR title carries the VT-id; board row carries Branch + Task).
3. Flip that row's Status to Done via update_properties (single-property write, allowed).
4. Rows with no merged PR are left untouched — no guessing.

This MUST run in the session with repo access (it needs `gh`). A session without repo access cannot verify merged-PR ground truth and must not flip Status from memory. Confirmed-merged-PR is the only basis for a Done flip.

Known shipped as of 2026-05-20 (flip these if their rows aren't already Done): VT-Foundation, VT-3.1, VT-3.2, VT-3.3a, VT-3.3b, VT-3.3c, VT-3.8, VT-3.4 PR 1/3 + 2/3, VT-32, VT-35, VT-37, VT-39, VT-122. Open question for the pass to resolve from `gh`: VT-3.4 PR 3/3 status, VT-50 (blocked, not shipped).

Vilpower / Twilio-DLT / legal rows are Fazal-owned and not PR-driven — the pass does not touch their Status.
