# VT-232 — Hot-fix Ops Console login: unstyled HTML + banner overlay

## Why

Fazal manual walk 2026-05-29 17:25 IST against `viabe-team-dev.vercel.app/team/ops/login`. Visible symptoms:
- "Completely jumbled and broken"
- "None of the texts are links"

Two root causes compounding:

**Cause 1 — page uses bare HTML, no Tailwind utility classes.** `apps/team-web/app/(app)/team/ops/login/page.tsx` has `<main className="ops-login">` but no CSS defines `.ops-login`. The form renders with browser default styles: no spacing, no centered layout, no button visual affordance. VT-203 brief said "Tailwind defaults" but no Tailwind utility classes were applied — so what landed is browser defaults, not Tailwind defaults.

**Cause 2 — layout inheritance puts the sticky banner on the login page.** `/team/ops/login` lives under `(app)/team/ops/layout.tsx`, which renders `<StickyBannerLive>` for ops-awareness counts. For an unauthenticated visitor, `fetchBannerCounts()` fails (caught), banner renders with zeros — but it's still occupying the top of the page, overlapping the form area. Compounded with no form styling, the result reads as a jumbled stack of unstyled elements.

## Brief contract

CC implements **both fixes** in one PR:

### Fix 1 — Lift login OUTSIDE the ops layout

Restructure the route so `/team/ops/login` does NOT inherit `(app)/team/ops/layout.tsx`. Options:

**1A (recommended):** Move login to a sibling route group:
- New location: `apps/team-web/app/(auth)/team/ops/login/page.tsx`
- New `(auth)/layout.tsx` — minimal shell, no banner, no auth gate
- Old location deleted: `apps/team-web/app/(app)/team/ops/login/`

**1B (fallback):** Keep current location but add a route-group escape via parallel routing — significantly more code, only do if 1A blocks on Next.js limitations.

Pick 1A unless Next.js routing constraints force 1B. Document the decision in the PR description.

### Fix 2 — Apply Tailwind styling to make the form usable

The form should look like a basic centered login card. Suggested baseline (CC may adjust within scope):

```tsx
<main className="min-h-screen flex items-center justify-center bg-gray-50 px-4">
  <div className="w-full max-w-md bg-white rounded-lg shadow-md p-8">
    <h1 className="text-2xl font-semibold text-gray-900 mb-2">Ops Console</h1>
    <p className="text-sm text-gray-600 mb-6">Sign in with your operator email</p>

    {sent ? (
      <p className="text-sm text-green-700 bg-green-50 border border-green-200 rounded p-3">
        Check your email for the sign-in link.
      </p>
    ) : (
      <form action="/api/ops/login" method="post" className="space-y-4">
        <div>
          <label htmlFor="email" className="block text-sm font-medium text-gray-700 mb-1">
            Operator email
          </label>
          <input
            id="email"
            name="email"
            type="email"
            required
            autoComplete="email"
            className="w-full px-3 py-2 border border-gray-300 rounded-md focus:outline-none focus:ring-2 focus:ring-blue-500 focus:border-transparent"
          />
        </div>
        {nextParam ? <input type="hidden" name="next" value={nextParam} /> : null}
        <button
          type="submit"
          className="w-full bg-blue-600 hover:bg-blue-700 text-white font-medium py-2 px-4 rounded-md transition-colors"
        >
          Email magic link
        </button>
      </form>
    )}

    {errorParam ? (
      <p className="mt-4 text-sm text-red-700 bg-red-50 border border-red-200 rounded p-3">
        Sign-in error: {errorParam}
      </p>
    ) : null}
  </div>
</main>
```

This is suggested baseline only — CC can adjust class choices as long as the result renders as a centered usable form.
