# VT-69 — L3 retrieval + 180-day quarantine

## Parent task
VT-KnowledgeArchitecture.

## Concept reference
- `concept-team.md` §9.3 (L3 retrieval; 180-day quarantine for new tenants before they contribute to or read from L3 patterns).
- `concept-team-pillars.md` Pillar 6 (k-anonymity at construction; quarantine prevents new tenants from being identifiable contributors before they're sufficiently anonymized within the cohort).
- `concept-team-pillars.md` Pillar 7 (privacy: a tenant who has 30 days of data is structurally identifiable in a 5-tenant cohort; quarantine ensures the cohort is meaningful before they participate).

## Dependencies
- VT-7.5 Done (L3 patterns table + construction pipeline).
- VT-PrivacyArchitecture VT-8.3 Done (k-anonymity admission gate; quarantine is the time-component of admission).

## What CoderC must produce
The L3 retrieval API plus the 180-day quarantine policy. Tenants younger than 180 days neither contribute to L3 construction (they're excluded from the eligible set in VT-7.5's pipeline) nor query L3 patterns at runtime. They get the empty-pattern fallback path.
1. **Retrieval module** at `apps/team/src/kg/l3_query.py` (extends the stub from VT-7.5):
	- `lookup_pattern(tenant_id, pattern_type, cohort_key) -> L3Pattern | None`.
	- Quarantine check: read `tenants.signed_up_at`; if `now() - signed_up_at < 180 days`, return None and log `l3_quarantine_skip` event with run_id + tenant_id + days_remaining.
	- For tenants past quarantine: standard lookup against `l3_patterns` table by (pattern_type, cohort_key); return matching row or None.
2. **Quarantine in construction pipeline** (modification to VT-7.5):
	- The eligibility filter in `l3_construction.py` adds a `WHERE signed_up_at < now() - INTERVAL '180 days'` clause to its tenant cohort query. New tenants are silently excluded from contributing.
	- Once a tenant crosses 180 days, they automatically join eligible cohorts on the next nightly run. No manual intervention.
3. **Empty-pattern fallback for the agent**:
	- When `lookup_pattern` returns None (whether quarantine or genuine no-match), the orchestrator's context builder (VT-4.3) populates the L3 portion of the bundle with a structured "no L3 prior available" marker, NOT with fabricated default values.
	- The agent's system prompt (VT-4.2) instructs that absence of L3 prior means: "reason from this tenant's own data without cross-tenant priors." CodeX rejects PRs that fabricate fallback values.
4. **Telemetry**:
	- `pipeline_log` events: `l3_query_attempted` (with cohort_key + result), `l3_quarantine_skip` (when quarantine blocked the query), `l3_no_match` (cohort eligible but no pattern existed — typically because the cohort_key didn't meet k-min during construction).
	- The orchestrator's cost dashboard (VT-12.3) surfaces L3 query rates per tenant; drops in rate after quarantine ends are expected and not alerted.
5. **Quarantine override**:
	- There is NO override path. CodeX rejects PRs that add an `override_quarantine=True` parameter or admin bypass. The 180-day window is structural per Pillar 7.
6. **Tests** at `apps/team/tests/kg/test_l3_retrieval.py`:
	- Quarantine: synthetic tenant signed up 30 days ago → lookup returns None + `l3_quarantine_skip` logged.
	- Past quarantine: synthetic tenant signed up 200 days ago → lookup returns matching pattern.
	- Boundary: tenant exactly at 180 days — inclusive boundary documented; test confirms behavior matches.
	- No-match path: tenant past quarantine but cohort_key has no row → returns None + `l3_no_match` logged (distinct event from quarantine).
	- Construction exclusion: synthetic 10 tenants where 4 are <180 days → only 6 used in cohort eligibility; if the 6 fall below k-min=5 alone, the cohort is dropped (verifies the quarantine excludes from contribution as well).
	- Override impossibility: any test attempting to bypass quarantine by parameter or context manipulation fails to do so.

## Architectural rules to enforce in this subtask
- Pillar 6+7: 180-day quarantine is structural; CodeX MUST reject any override mechanism. The window applies to both contribution and consumption.
- Pillar 4: empty-pattern fallback returns structured "no prior" marker; CodeX rejects fabricated default values in the fallback path.
- The 180-day value is a Type 3 commitment. Lowering it requires board governance.
- Quarantine check happens at retrieval time; the agent never sees a pattern from a quarantined source. The check does NOT happen inside the agent.
- Tenants past quarantine automatically join. No manual flag.
- L3 quarantine is independent of phase: a tenant in `paid_active` phase but only 60 days old is still quarantined.

## Acceptance criteria
- `lookup_pattern` enforces quarantine via `signed_up_at` check.
- Construction pipeline (VT-7.5) excludes <180-day tenants from cohort eligibility.
- Empty-pattern fallback returns structured marker; agent prompt addresses absence.
- All 5 test scenarios pass.
- No override mechanism exists; verified by test attempting bypass.
- Boundary behavior documented and tested.
- mypy --strict passes.
- PR title references `(VT-KnowledgeArchitecture)`; reviewed by CoderX (mandatory — privacy-critical); Fazal personally signs off on the 180-day value as a Type 3 commitment.

## Out of scope
- Adjustable per-tenant quarantine duration (rejected by Pillar 7; not a Phase 1 concern).
- L3 caching at the agent layer (Phase 1 hits the table directly; cheap given low cardinality of cohort_key).
- L3 effectiveness measurement (Phase 2; once Phase 1 ships, observe whether L3 priors improve agent quality).
- L3 versioning (Phase 2; current L3 reconstructs nightly; versioning is a future ask).
- Tenant-facing visibility into their L3 contribution status (out of scope; ops-only).
