## What CoderC must produce
The instrumentation that makes the 4 hard limits (80K tokens, 25 tool calls, depth 8, 5min wall clock) observable + alertable. VT-4.4 enforces; this subtask measures, charts, and alerts on approach + breach.
1. **Per-run instrumentation** at `apps/team/src/observability/limits_instrumentation.py`:
	- At agent dispatch start: emit `limits_initialized` event with the 4 thresholds.
	- During run: emit `limit_progress` events at 25%, 50%, 75%, 90% of each limit.
	- On breach: emit `limit_breached` event with severity=critical; agent terminates per VT-4.4.
	- Wall-clock measurement via async monotonic clock, NOT system time (avoids drift / timezone bugs).
2. **Token counting**:
	- Anthropic returns token counts in API responses; sum these per run.
	- For tools that don't use Anthropic (Twilio, Razorpay): treat as 0 tokens (their cost is in VT-12.3 separately).
	- Pre-flight estimate before each Anthropic call using Anthropic's tokenizer SDK; if cumulative + estimated would exceed 80K, terminate before the call.
3. **Tool-call counting**:
	- Increment per `tool_invoked` event in pipeline_log.
	- The 25-tool-call cap is across ALL tools combined (not per-tool).
4. **Depth tracking**:
	- Depth is reasoning loop iterations: each `agent_reasoning_step` event increments depth counter for that run.
	- 8-deep cap reflects "agent reasoned 8 times before producing output". Beyond 8 is structural runaway.
5. **Wall-clock tracking**:
	- Start time captured at dispatch entry.
	- 5-minute soft warning at 4 minutes; 5-minute hard termination.
6. **Alerts**:
	- 90% threshold breach for any limit → Telegram warning to Fazal (informational).
	- 100% (hard breach) → critical Telegram alert + agent terminated + tenant flagged for review.
	- Pattern detection: tenant with 3+ limit breaches in 7 days → separate alert suggesting pipeline review for that tenant's data quality.
7. **Dashboard integration**:
	- VT-12.3 cost dashboard adds limits panel: per-tenant 7-day breach counts + average token usage + average tool calls.
	- Telegram `/limits <tenant_id>` returns the same view.
8. **Tests** at `apps/team/tests/observability/test_limits_instrumentation.py`:
	- 80K token breach: synthetic run accumulating to 81K tokens → limit_breached event + terminated.
	- 25 tool calls: synthetic run with 26 tool invocations → breach.
	- Depth 8: synthetic run with 9 reasoning loops → breach.
	- Wall clock: synthetic 6-minute run → breach.
	- 90% warning: synthetic run reaching 72K tokens → warning event + Telegram.
	- Pre-flight estimate: synthetic dispatch where next call would push to 81K → terminated before the call.
	- Pattern alert: synthetic 4 breaches in 7 days for one tenant → separate alert fired.
	- Cross-tenant: tenant_A breaches don't affect tenant_B accounting.
	- Telegram /limits command: synthetic data → correct breakdown.
## Architectural rules to enforce in this subtask
- Pillar 4: deterministic enforcement. CodeX rejects PRs adding LLM-judged dynamic limits.
- Pillar 7: limits protect tenant from runaway. CodeX rejects per-tenant limit overrides.
- Pillar 8: one instrumentation primitive. VT-4.4 enforces; this subtask observes. No parallel observability.
- Token estimate uses Anthropic's tokenizer SDK; CodeX rejects custom tokenizers.
- Wall-clock via monotonic clock; CodeX rejects time.time() for measurement.
- Pattern alert window (7 days) + threshold (3 breaches) is Type 2.
