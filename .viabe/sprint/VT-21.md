	- `SUPABASE_URL`, `SUPABASE_SECRET_KEY` (per environment, dev/prod separated)

	- `ANTHROPIC_API_KEY`, `OPENAI_API_KEY` (existing keys reused, but with usage tagged "team" via API key labels where supported)
	- `TWILIO_ACCOUNT_SID`, `TWILIO_AUTH_TOKEN`, `TWILIO_WA_FROM` (existing Reports Twilio account; Team uses the same with WhatsApp Business templates added in VT-13.3)

3. **GitHub Actions workflows** in `.github/workflows/`:
	- `team-web-ci.yml` — runs on PRs touching `apps/team-web/` or `packages/team-shared/`. Pytest is irrelevant here; runs vitest, eslint, type-check, build.
	- `team-services-ci.yml` — runs on PRs touching `apps/team/` or `packages/team-shared/`. Runs pytest, ruff, mypy, build.
	- `team-deploy-dev.yml` — on push to `dev`, triggers Railway deploy of `team-orchestrator-dev`. Vercel auto-deploys via GitHub integration without explicit workflow.
	- `team-deploy-prod.yml` — on push to `main`, triggers Railway deploy of `team-orchestrator-prod`. Same auto-deploy assumption for Vercel.
4. **Branch protection rules** on the GitHub repo:
	- `main`: requires PR + green CI + 1 approving review (Fazal). No direct pushes (including from Fazal). No force-push.
	- `dev`: requires green CI. Direct pushes allowed only from Fazal account; CoderC and CoderX use PRs.
	- Both: linear history required (no merge commits; squash or rebase only).
5. **Deploy verification**: a stub endpoint in `apps/team/` (e.g., `GET /healthz`) returns `200 OK`. Stub page in `apps/team-web/` (e.g., `/healthz`) returns `200`. Both verified live on dev URLs.

## Architectural rules to enforce in this subtask
- Pillar 8 (structural separation): dev and prod environments are separate Railway services + separate Vercel preview/production targets. They DO NOT share a deployment unit. A bug pushed to dev cannot accidentally hit prod.
- Branch protection is mechanical: it cannot be bypassed by convention. CoderC and CoderX cannot push directly to `main`. Even Fazal does not push directly to `main` (he merges PRs).
- GitHub Actions workflows must be path-filtered: `team-web-ci.yml` only runs when Team Web files change, not on Reports app changes. Avoids burning CI minutes on unrelated work.
- Production credentials live ONLY in Vercel/Railway env settings + 1Password. Never in workflow files (workflows reference secrets by name).
- A workflow that deploys to prod (`team-deploy-prod.yml`) requires the `production` GitHub environment, gated by manual approval from Fazal. Not just "push to main" — also a deploy-time gate.
- Linear history requirement reduces drift in incident postmortems (no diamond merges).
