# CL-125 — Verbal CL-numbers don't always match page Entry IDs

Discovered during CL-91 / CL-94 micro-PR drafting: verbal CL-numbers used in conversation (e.g. 'CL-71', 'CL-84', 'CL-91') do not match the Entry ID auto-increment values on the corresponding pages (which are CL-74, [unknown], CL-94 respectively). The verbal numbering appears to come from an earlier ordering that doesn't match what the auto-increment generates. Page UUIDs are authoritative; CL-numbers in either form (verbal or Entry ID) are nicknames.

IMPACT ON PRIOR LOGGING: CL-124 (PR #19 merge State Snapshot) references 'CL-71/CL-84/CL-117/CL-120/CL-122' — the page UUIDs in 'Linked Tasks' are correct, but the CL-numbers in the prose may not match those pages' Entry IDs. Not worth retroactively patching CL-124 prose (cosmetic, log entries are self-linking via UUID), but flagging for future searches.

DISCIPLINE GOING FORWARD: when referencing prior CL entries in new log writes, use the page UUID (which I do) AND either (a) the actual Entry ID from the page (authoritative) or (b) the verbal nickname Fazal uses (familiar but ambiguous). Prefer (a) for new entries; tolerate (b) when echoing Fazal's verbiage in the same turn.

NOT FIXING: the historical verbal numbering isn't worth normalizing. The system works via UUIDs; CL-numbers are human shorthand. Trying to rationalize a single canonical numbering across all prior log entries is a snake-eating-its-tail exercise.
