## Root cause (the prod mystery, solved — NOT a crash)

Enforce-gate rule D routes every interrogative to "the brain" (return None). A PRE-ACTIVATION
tenant has no brain downstream — owner_inputs=false sends the inbound to the consent gate,
whose represent guard re-asks the pending question robotically. By-design routing hole, live
on prod run #4 ("What does that mean?" → "Let's finish setting up first — …") and reproduced
on dev by the sim.

## Fix

Rule D now checks activation: interrogative + ACTIVE journey + owner_inputs FALSE →
delegate to the journey turn-brain (`maybe_handle_journey_reply` — carries the VT-701
never-deflect/explain rules; signup consent covers onboarding processing). Activated tenants
keep the brain routing (the measured-good privacy-question path). Fail-open on the
owner_inputs read (falls back to brain routing).

## Evidence

3 new pins in test_enforce_journey_gate.py (pre-activation delegates; activated stays None;
read-failure fail-open). 44-test gate suite + onboarding sweep green. Sim re-run on dev
post-deploy = the live proof.
