# CL-346 — DECISION/FINDING (DBOS workflow_inputs Step-0 — Branch B confirmed, evidenced): 

DECISION/FINDING (DBOS workflow_inputs Step-0 — Branch B confirmed, evidenced): DBOS 2.22.0 persists workflow inputs (incl. our twilio_fields.Body) to the workflow_status.inputs column via _core.py start_workflow -> _sys_db.init_workflow. Default serializer is pickle+base64 — NOT encryption, trivially reversible. On recovery, inputs ARE re-read from sys DB: _recovery.py recover_pending_workflows -> _core.py execute_workflow_by_id:707 deserialize_args; no in-memory fallback. webhook_pipeline_run does NOT re-read Body (consumed once at runner.py:158/218) — BUT the workflow FUNCTION BODY (pre_filter + conditional branches) re-executes from the top on recovery, so a redacted-before-start Body => empty event.body on the resumed run => routing divergence. Body is replay-critical to the workflow LOGIC, not to DBOS itself. BRANCH B: redaction-before-start_workflow is unsafe. FIX SHAPE (Clau-scoped): post-completion purge — dbos._sys_db.garbage_collect EXISTS (_sys_db.py:3481, deletes completed workflow_status rows incl. inputs, excludes PENDING/ENQUEUED/DELAYED) but NO scheduler wires it on main. Fix = wire garbage_collect to a recurring schedule with retention cutoff N. Encryption (custom Serializer) set aside as separate optional hardening, not bundled. OPEN (Fazal): N = retention window, goes in the privacy notice as a customer-facing commitment. WORKFLOW_TIMEOUT_SECONDS is 6 min (hard in-flight ceiling) => Clau recommends N=1h (loose enough for recovery safety, tight enough to state honestly); report floated 24-48h which is unnecessarily loose. PRIVACY NOTICE CONSEQUENCE: RETAINED line now has 3 sub-cases — pipeline_runs/steps (closed, VT-144), owner_inputs derived rows (tenant-lifetime), DBOS workflow_status.inputs (transient <=N hours then purged). Notice cannot say 'raw body not retained' — must say 'transiently retained <=N then purged'.
