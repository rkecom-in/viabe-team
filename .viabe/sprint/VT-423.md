## Contract (from the VT-420 adversarial money-send verify, 2026-06-25). Two NON-BLOCKING residuals — POST-e2e.

VT-420's 'sending' marker closes the crash-window double-send (canary-proven, CANNOT double-charge on any
reachable path). The adversarial verify found two non-blocking residuals worth hardening later:

1. **Self-serializing marker:** `_write_inflight_marker` (`send_whatsapp_template.py`) uses
   `INSERT ... 'sending' ON CONFLICT DO NOTHING` WITHOUT a `RETURNING`/rowcount check — so the TOOL doesn't
   serialize two true-parallel first-attempts on the same key (both could pass `_check_idempotency` → double-send).
   UNREACHABLE today (DBOS workflow-id single-flight + the serial draft loop prevent true-parallel same-key sends),
   but the tool shouldn't DEPEND on upstream. Fix: `RETURNING id` / rowcount so the marker self-serializes.
2. **24h marker window:** `_check_idempotency`'s `created_at` 24h window means a 'sending' marker stops blocking
   after 24h — a narrow re-send tail if a draft is somehow re-driven >24h post-crash. PRE-DATES VT-420. Note/decide.
