## What CoderC must produce
The Method 8 ingestion adapter for Google Business Profile reviews. Structurally identical to VT-6.10 but a different platform with a different actor and different listing-URL structure. Same PII discipline.
1. **Adapter module** at `apps/team/src/ingestion/methods/apify_gbp.py`:
	- `ingest_gbp(tenant_id, run_id) -> IngestionSummary`.
	- Resolves the tenant's GBP URL (captured at sign-up; format: `https://www.google.com/maps/place/<name>/@<coords>/...`). If absent, returns warning + skips.
	- Calls Apify GBP actor with the URL.
2. **Data extracted (CONTEXT ONLY, NO PII)**:
	- Tenant's own GBP listing: rating, review count, photos count, popular times (if Google provides), amenities/attributes (e.g., "wheelchair accessible", "cash only"), opening hours displayed, business categories.
	- Aggregate review distribution: count by star (1-star count, 2-star count, ..., 5-star count), recent review count (last 30 / 60 / 90 days), aggregate keyword themes from a privacy-preserving extraction (see below).
	- Local competitor density: count of similar GBP listings within 1km radius, in the same category.
3. **What is NEVER extracted**:
	- Individual review texts (verbatim).
	- Reviewer names, profile URLs, photos, locality tags.
	- Owner-replied review text.
	- Reviewer rating histories (their other GBP reviews).
4. **Privacy-preserving keyword themes**:
	- The Apify actor returns review snippets (Apify can't unsee them). The adapter, BEFORE writing anything, runs a small deterministic transformation:
		- Strips reviewer names + photos.
		- Aggregates the snippets into theme counts via simple keyword matching (a small built-in keyword list per business_type, e.g. for cafes: "coffee", "food", "service", "ambiance", "price", "wifi"). Each theme gets a count + average sentiment (positive/neutral/negative based on star rating of the source reviews, NOT NLP).
	- The original review snippets are DISCARDED in-memory after theme extraction. Never stored, never logged.
5. **L1 KG write**: per-tenant subgraph: `tenant -[has_gbp_listing]-> gbp_listing { rating, review_count, ... }` plus `gbp_listing -[has_theme]-> { theme: 'service', count: 42, sentiment_avg: 'positive' }`. Last-updated timestamp; cached 7 days.
6. **Telemetry**: `pipeline_log` events with run_id, tenant_id, actor used, themes extracted (counts only). Raw review text NEVER in payload.
7. **Tests** at `apps/team/tests/ingestion/methods/test_apify_gbp.py`:
	- Synthetic Apify response with reviews → correct theme extraction → correct L1 KG writes.
	- PII absence: synthetic Apify response with reviewer names + verbatim review text → ALL stripped before storage. Test verifies no review text in DB or `pipeline_log`.
	- GBP URL absent: tenant without GBP URL → warning + skip.
	- Apify timeout: 60s exceeded → structured error.
	- Cache: re-running within 7 days → cached, no scrape.
	- Cross-tenant: tenant_A's data RLS-blocked from tenant_B.
	- Theme extraction: 10 synthetic reviews about service quality → 'service' theme count = 10 with correct sentiment_avg.
	- 7-day re-scrape: synthetic time advance triggers fresh scrape.

## Architectural rules to enforce in this subtask
- Pillar 7: NO PII. Same discipline as VT-6.10. CodeX MUST reject any PR that stores review text, reviewer names, or reviewer profile data. Test coverage proves stripping.
- Theme extraction is deterministic keyword counting, NOT LLM-based sentiment analysis. Sentiment is derived from the star rating of the source review, which is a structural fact, not a model judgment.
- The keyword list per business_type is config-driven (`apps/team/config/gbp_themes.yaml`). Adding a new business_type's keyword set is Type 1.
- Cache + rate limit + TTL discipline shared with VT-6.10.
- The agent uses theme counts to understand customer perception, NOT to quote specific reviews. CodeX confirms by reviewing how the agent's prompt (VT-4.2) constrains usage.
