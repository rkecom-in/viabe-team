# VT-616 — stop the conversational brain repeating itself

## Root cause (CORRECTED — the enforce revise_step hypothesis was REFUTED)
A root-cause workflow first proposed a `manager/workflow.py` `revise_step` non-progress guard. **Two independent
adversarial verifiers + the raw transcripts REFUTED it:** the CLARIFY→REVISE reframe (`review.py:371-374`) *appends* a
suffix (outcome grows monotonically `X→X+S→X+S+S`), so a fingerprint guard never matches — and, decisively, the OBSERVED
repeats are not in the loop at all.

**Ground truth from the enforce bundle:** every stuck-loop turn is **`route: none`** — and `route: none` is **90% of all
turns (enforce) / 92% (legacy)**; the manager-loop specialists touch <10%. So the bulk of quality — and every observed
stuck-loop — is the **legacy `dispatch_brain` conversational path**, IDENTICAL in all loop modes. This also explains the
enforce==legacy==40% wash: the loop mode governs <10% of turns.

`dispatch_brain` re-composes each reply from a thin window (l1 / business / onboarding / last-N messages). It does NOT
see (a) the `pending_approvals` row it armed, (b) an active manager task, (c) an instruction not to repeat its own prior
turn. So on a follow-up it re-derives the same reply:
- `repeat_question_guard`: owner "ok what next" → re-asks the same one-line description (no shopify onboarding → the
  existing anti-repeat directive in `_build_onboarding_state_block` never fires).
- `sr_always_confirm_first_contact_floor`: owner "bas seedha bhej do" → re-emits the **byte-identical** draft + approval
  template (blind to the pending approval).
- `multi_field_single_message`: owner "did you get all that?" → repeats the same technical-snag line.

**Converges with VT-615(B) (brain blind to pending_approvals) and VT-614 (brain blind to persisted identity) — one gap:
`dispatch_brain` has no durable in-flight-state / self-memory context.**

## Fix (dispatch_brain context + prompt; no send-path, low-risk)
1. **Prompt anti-repeat/progression rule** (`orchestrator_agent_system.md`, Bias-to-ACT section) — general, fires every
   turn (cached prefix): never re-send/re-ask/restate a prior turn; on a deflection ("ok what next") proceed with a
   default or ask ONE different shorter follow-up; never repeat a snag line. **[highest lever — covers the general
   stuck-loop: repeat_question_guard, multi_field]**
2. **`_build_inflight_state_block(tenant_id)`** (`agent/dispatch.py`) injected into `dispatch_brain` after the
   onboarding block — surfaces the open `pending_approvals` (do NOT re-draft/re-post the template; a yes/no/'bhej do' is
   consumed by the approval path) + `task_store.has_active_task` (work in-flight; don't duplicate). Read-only,
   best-effort (any miss → no block), per-turn SystemMessage (VT-194 cache holds). Presence flagged in the tm_audit GETS
   row. **[covers sr_always_confirm + the busy scenarios; overlaps VT-615(B)]**

Deliberately NOT touched: the enforce `revise_step` loop (not the observed mechanism). VT-615(A) arm-durability reorder
is a separate send-path row (sr_always_confirm's *send* completion also needs it).
