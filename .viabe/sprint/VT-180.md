# VT-180 — pipeline_observability.write_step() writer (LOAD-BEARING)

## Scope

**The core data-log writer.** Companion to VT-102's event-log writer (`observability/log.py`), but for the DATA log. Per design doc §3:

1. **Module:** `apps/team-orchestrator/src/orchestrator/observability/pipeline_observability.py`
2. **API:** `write_step(step_kind, run_id, tenant_id, step_name, input_envelope, output_envelope, status, parent_step_id=None, decision_rationale=None, tool_calls=None, error=None, cost_paise=0, model_used=None, tokens_input=None, tokens_output=None) -> None`
3. **Atomic step + run update:** Each write inserts a `pipeline_steps` row AND updates `pipeline_runs.step_count` + `pipeline_runs.total_cost_paise` in the same transaction.
4. **Envelope validation:** Inputs validated against VT-179 schema for the given step_kind. Drift = explicit error logged + step write proceeds with `payload_validation_failed: true` flag (soft-fail per pattern; same as VT-102).
5. **SQLite buffer fallback:** If prod DB INSERT fails, write to a local SQLite buffer at `apps/team-orchestrator/.observability_buffer.db`; background flusher retries when DB recovers. **Pipeline MUST NOT crash on observability failure.** Per design doc §3.5.
6. **Step_seq derivation:** monotonic within a run, atomic via SELECT-MAX-then-INSERT pattern (or a Postgres SEQUENCE keyed by run_id).
7. **PII handling:** input/output envelopes flow through VT-104 redactor before serialization. Phone numbers → tokens via VT-184 (when shipped); for now, use the redactor's existing phone-hashing.
