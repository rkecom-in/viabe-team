## Why this parent exists
This parent is the launch gate. Two distinct workstreams converge here. First, hardening: soak runs, edge-case suite expansion, security/privacy/legal review under DPDPA — the work that turns a working pipeline into a production-ready one. Second, vendor approvals: Meta WhatsApp templates (each takes 2-7 days), Razorpay Live KYC, Apify actors, Twilio DLT, KYC vendor selection, Resend DMARC, LangSmith billing. Vendor approvals are not negotiable; they sit on the critical path and run on external organizations' calendars, not Team's. Most are 1-7 day async waits.
The Vendor Approvals Buffer (June 12-19 in the source brief) exists because vendor approvals can fail in ways code-side fixes cannot route around. Meta can reject a template; Razorpay can flag a KYC anomaly; Twilio DLT can stall behind regulatory queues. The buffer absorbs that variance. Without the buffer, a single rejection delays launch by a week.
## What this parent owns
1. Soak run infrastructure: synthetic-load, multi-day runs that surface memory leaks, slow drift, scheduled-trigger reliability.
2. Edge-case test suite expansion: failure modes uncovered during Sprints 1-9 get codified as regression tests here.
3. Meta WhatsApp Business templates: 8 templates required for Phase 1 (welcome, weekly approval, campaign send, attribution close, day-39 evaluation, refund offer, monthly report, support escalation).
4. Razorpay Live verification: entity KYC, settlement bank confirmation, plan approvals.
5. Apify actor confirmations: scrapers for Zomato, Swiggy, Magicpin, GBP must be approved or built and deployed.
6. Twilio approvals: DLT registration for Indian SMS/WhatsApp, sender ID approval.
7. KYC vendor selection + integration: Phase 1 candidates DigiLocker, Karza, Signzy. Decision documented; integration shipped.
8. Resend domain DMARC: [viabe.ai](http://viabe.ai) DMARC configured for impact-report email deliverability.
9. LangSmith billing tier confirmation: tier sized to expected subscriber count, billing live.
10. Final security/privacy/legal review (DPDPA pass): outside legal counsel reviews the architecture, opts policy, DSR APIs, and signs off.
## Architectural rules binding every subtask
- Pillar 8 (no patchwork): hardening fixes route through architecture. If soak surfaces a memory leak, fix the resource lifecycle, do not paper over with a restart cron.
- Vendor approvals run on vendor calendars; subtasks here track status, do not pretend they can be expedited.
- Each vendor subtask has a fallback plan documented: if Meta rejects template X, what's plan B? If Razorpay KYC stalls beyond N days, what's the escalation path?
- DPDPA review is sign-off gated. Without legal counsel sign-off, launch is blocked. Internal opinions do not substitute.
- Soak runs are reproducible: same seed produces same load profile.
- Edge-case tests added during hardening are codified as regression tests, not as one-off scripts.
- The KYC vendor decision is a Type 2 change with documented rationale (cost, coverage, integration complexity, DPDPA compliance).

## Definition of done
- All 10 subtasks Done.
- 72-hour soak run completes without failures, memory leaks, or scheduled-trigger drops.
- Edge-case suite covers all uncovered-during-sprint scenarios; regression suite green.
- 8 Meta templates approved and integrated. Plan-B documented for any rejected template.
- Razorpay Live: real test charge succeeds and settles. KYC complete.
- Apify scrapers operational on a synthetic subscriber location.
- Twilio DLT registered, sender ID approved.
- KYC vendor integrated; synthetic owner-onboarding KYC flow passes end-to-end.
- Resend DMARC verified; test impact-report email deliverable to a real Gmail/Outlook inbox.
- LangSmith billing tier confirmed; usage projections documented.
- DPDPA legal counsel sign-off recorded in Notion. Architecture documents shared with counsel.
