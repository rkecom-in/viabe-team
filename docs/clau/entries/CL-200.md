# CL-200 — emit_pipeline_step shipped as log emitter only

Location: apps/team-orchestrator/src/orchestrator/_tenant_guard.py (new in VT-3.4 PR 2/3).

SITUATION: §3.2 of the brief calls emit_pipeline_step from inside assert_tenant_scoped to record tenant_isolation_breach events. emit_pipeline_step does not yet exist as a real DB-writing function — VT-122 (observability pipeline_steps writer) hasn't shipped. Additionally, pipeline_steps rows require a run_id, which the guard helper doesn't have in its current signature (guard receives rows + expected_tenant_id only).

CURRENT STATE (Claude Code build 2026-05-19): emit_pipeline_step shipped as a structured-log emitter (logger.error with security_tag + structured fields). DB write to pipeline_steps deferred to VT-122. The fail-loud TenantIsolationError still raises, so security-relevant behavior is intact — the gap is observability/audit-trail only, not the security guarantee.

FIX when VT-122 lands:
1. emit_pipeline_step becomes a real writer that calls into the observability module's @observability.step or equivalent.
2. The guard's signature may need to accept run_id (or pull it from a contextvar) so the pipeline_steps row has run_id populated. Decide at VT-122 integration time which pattern (explicit arg vs contextvar) fits the rest of the observability surface.
3. Existing structured-log calls become parallel — keep the log, add the DB write — OR get replaced by the observability decorator, depending on VT-122's API.

GREP TARGET when VT-122 lands: `rg "emit_pipeline_step" apps/team-orchestrator/src/` should surface every call site. Currently only _tenant_guard.py and (per §3.3) context_builder.py truncation-event paths. 

NOT BLOCKING PR 2/3. Security guarantee (raise on breach) is intact; only audit-trail DB write is deferred. PR description will note the deferral.

LINKED: VT-122 (observability pipeline_steps writer).
