# VT-505 — DBOS durable layer broken on Railway dev (LAUNCH-RELEVANT)
Verified: the dev Supabase DB's dbos schema has ONLY transaction_outputs; dbos.workflow_status does NOT exist. So DBOS.start_workflow can't persist → (a) start_l3_hold silently fails (BLE001-caught) → an armed auto_send_pending batch is left hold-unstarted with NO recovery sweep for that state (the code claims 'recovered by next sweep' but no such path exists for auto_send_pending without a running workflow); (b) the @DBOS.scheduled crons — incl the VT-431 daily autonomous coordinator + the alert sweeps + the VT-481 orphan reaper — likely don't actually FIRE unattended (they need the DBOS runtime). So 'the autonomous loop is active' is wired but NOT durably executing on dev.
# G35 nuance: the autonomous DECISION + send PATH is proven (kick → auto_send_pending, no owner pause, mocked); the durable EXECUTION (unattended cron-driven + durable holds) is what's broken. The drive triggered the path manually (kick); production-unattended needs DBOS working.
# Likely cause (INVESTIGATE, don't assume): DBOS schema-init/migration didn't fully run on the Supabase dev DB; OR the transaction-mode pooler (pgBouncer) connection can't host DBOS's session-scoped workflow tables; OR DBOS 2.22 init fails silently at Railway startup. Check the Railway dev orchestrator boot logs for a DBOS init error + whether DATABASE_URL points at the pooler vs a direct/session connection. Possibly an infra/console fix (connection string / DBOS init), not just code. Plan-first → Cowork/Fazal (infra decision).

## CONFIRMED canary/live-e2e BLOCKER (Rule #14): start_l2_send (l2_send.py:201,210) — the OWNER-APPROVED human-run send path — ALSO uses DBOS.start_workflow(l2_send_workflow), same dependency as start_l3_hold. So with dbos.workflow_status absent, Fazal's human-run win-back (owner approves → start_l2_send) would NOT durably fire either. launch_dbos() IS called (main.py:145) so DBOS init is failing silently on Railway dev — investigate the boot logs + the DBOS SYSTEM-DB config (DBOS stores workflow_status in a system DB which may be a separate DB/connection from the app DATABASE_URL; transaction_outputs present but workflow_status absent = partial/failed init or wrong system-DB). This BLOCKS the pre-run canary + Fazal's live e2e until fixed.

## CORRECTED (Rule #14, definitively verified): the original 'DBOS not initialized' premise was WRONG — a wrong-DB query. DBOS IS working on dev: postgres_dbos_sys DB exists, dbos.workflow_status present (91 workflows), crons fire, the autonomous loop durably runs. workflow_status lives in the SEPARATE postgres_dbos_sys DB (DBOS appends _dbos_sys), not the postgres app DB (which has only transaction_outputs — what I mis-queried). So L2 owner-approved send + the canary + Fazal's human run are NOT blocked.
## THE REAL BUG (confirmed in code, l3_hold.py:717-741): module globals `_hold_demote_step=None`/`_hold_send_step=None` are immediately rebound by `def _hold_demote_step(...)`/`def _hold_send_step(...)` (SAME names) → `_ensure_hold_steps()`'s `if _hold_demote_step is None` is never True after import → the DBOS-wrapped step never replaces the wrapper → the wrapper calls itself → RecursionError. Confirmed live: l3_hold workflow uuid l3_hold_63477e2c... = ERROR RecursionError in postgres_dbos_sys. l2_send.py is CORRECT (distinct `_l2_send_step_decorated` name). FIX (contained code, no infra): rename the DBOS-wrapped globals to `_hold_demote_step_decorated`/`_hold_send_step_decorated`, set those in _ensure_hold_steps, call the decorated versions from the wrappers (mirror l2_send.py). Blocks ONLY the L3 AUTONOMOUS terminal send (G35 durable path); L2 unaffected.
