# VT-318 — inbound consumer STOP handler + opt-out immediate effects

Split out of VT-76 (Cowork ruling 20260604T032500Z). VT-76 builds the
reconstitution SWEEP + SLA + columns NOW (the privacy mechanism over the VT-66
hook). This row is the RECEIVE side — it needs the live customer-inbound path
(WABA), so it's gate-live (same posture as the rest of customer-comms).

## What to produce (when WABA/customer-inbound is live)
1. **Inbound STOP classifier** (`consumer_opt_out` intent): STOP / "रोको" / "band
   karo" / "unsubscribe" / variants, on the customer-inbound path. Resolve the
   sender phone → customer_id (cross-tenant: a phone may be a customer of several
   tenants; opt-out applies per-tenant by the prompting listing/campaign; ambiguous
   → all tenants, conservative).
2. **Set opt-out:** `customers.opt_out_status='opted_out'`, `opt_out_at=now()` for
   the affected tenant(s). (The columns + the 7-day sweep land in VT-76.)
3. **Immediate effects:** campaign send-skip (partly done, mig 045 — confirm the
   send gate checks opt_out_status); `query_customer_ledger` `exclude_opted_out`
   default; pending clarifications/approvals auto-resolve with
   `decision='customer_opted_out'`.
4. **Templates:** `opt_out_confirmation` + `reconstitution_completed` — free-form
   interim (VT-272/Meta), same as VT-303.

## Dependencies
- VT-76 Done (the opt_out columns + the reconstitution sweep + SLA).
- Live customer-inbound path (WABA) — the blocker.

## Acceptance criteria
- Inbound STOP → opt_out set + confirmation sent; Hindi/mixed recognized.
- Cross-tenant phone → conservative all-tenant opt-out when ambiguous.
- Active campaign skips the opted-out customer; ledger excludes; clarifications auto-resolve.
