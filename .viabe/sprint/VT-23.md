## What CoderC must produce
The canonical CI pipeline that every Team-side PR must pass to merge. Five gates, each runs on every PR touching Team paths, all green required. CI runs on `dev` and `main` branches plus PRs against either.
1. **`.github/workflows/team-ci.yml`** — the umbrella workflow that orchestrates all five gates as parallel jobs. Path-filtered: only runs when files in `apps/team-web/`, `apps/team/`, `packages/team-shared/`, `.github/workflows/team-*.yml`, or `docs/team/` change. Reports paths trigger the existing Reports workflows separately.
2. **Gate 1 — Python tests (pytest)**:
	- Job `python-tests`. Runs `pytest apps/team/tests/ packages/team-shared/python/tests/` with coverage report.
	- Coverage minimum: 75% for new code in this PR (delta coverage, not absolute). Below 75% → PR fails.
	- Includes the RLS attack tests (VT-2.4), the secret-comparison tests (VT-2.6), and any additional security tests added later.
	- Runs against the `viabe-team-dev` Supabase project (read-only secrets in CI env vars).
3. **Gate 2 — TypeScript tests (vitest)**:
	- Job `typescript-tests`. Runs `vitest run` in `apps/team-web/` and `packages/team-shared/` (TypeScript side).
	- Coverage minimum: 75% delta on new code.
4. **Gate 3 — Linting**:
	- Job `lint`. Runs ESLint + Prettier on TS, ruff + black on Python.
	- The cross-product import rule from VT-2.1 is part of this gate.
	- The constant-time secret comparison rule from VT-2.6 is part of this gate.
	- Type-check: `tsc --noEmit` for TypeScript, `mypy --strict` for Python (where applicable; new code is mypy-strict).
5. **Gate 4 — Notion task ID checker**:
	- Job `notion-task-id`. Custom GitHub Action (or shell script) that parses the PR title and asserts it matches the regex `\(VT-[A-Za-z]+(?:-[A-Za-z0-9]+)*\)$` — must end with a parenthesized VT identifier (e.g. `(VT-Foundation)`, `(VT-MCP-Tools)`, `(VT-Pillars)`).
	- On failure: PR comment from the bot explaining the rule and pointing to `concept-team.md` §16.
	- This gate runs on PR open + edit + reopen. Force-pushing a fix re-triggers it.
	- Also asserts the branch name matches `^(feat|fix|chore|docs|refactor|test|hotfix)/vt-` prefix.
6. **Gate 5 — gitleaks**:
	- Job `gitleaks`. Scans the diff (and full repo on first scan) for committed secrets.
	- Configuration at `.gitleaks.toml` includes patterns for: 64-char hex secrets (catches `INTERNAL_API_SECRET` leaks), Supabase publishable keys (`sb_publishable_*`), Supabase secret keys (`sb_secret_*`), AWS keys, Razorpay keys, Twilio auth tokens, generic API key shapes.
	- Allowlist ONLY the `.env.example` files (which contain placeholders) via path exclusion.
	- Negative test: a deliberate test commit on a throwaway branch with a fake `INTERNAL_API_SECRET=abc...` value triggers gitleaks failure. Verified once during this subtask, never re-tested in CI.
7. **Failure reporting**: when any gate fails, GitHub status check is red and the PR is blocked from merge. Branch protection rules from VT-2.5 require all five gates to pass.
8. **Documentation**: `docs/runbooks/ci-pipeline.md` documents how to read CI failures, common causes, who to escalate to. Includes a section on bypassing CI in true emergencies (which requires Fazal's explicit approval and a follow-up PR to fix the bypass cause).

## Architectural rules to enforce in this subtask
- Pillar 8 (no patchwork): CI gates are non-bypassable except by Fazal in documented emergencies. CoderC and CoderX cannot merge a PR with red CI; branch protection from VT-2.5 enforces this.
- The Notion task ID gate is mechanical, not aspirational. A PR title without `(VT-N)` fails immediately. This is what prevents the lineage drift Reports suffered.
- Coverage minimums are delta-based, not absolute. A PR adding 100 new lines must have 75 of them covered; existing-code coverage is not retroactively required.
- Path-filtering is mandatory. Reports paths must not trigger Team CI and vice versa. CI minutes are billed; cross-firing is wasteful and confuses signal.
- Gitleaks is the LAST line of defense, not the only one. The `INTERNAL_API_SECRET` log-scrubbing from VT-2.6 is the first; gitleaks catches what slipped past.
- Lint rules are enforced by the existing toolchain (ESLint, ruff, black) where possible. Custom rules (cross-product import, constant-time secret compare) are added as plugins or as a single shell script gate, not as bespoke per-rule jobs.
- Coverage data is uploaded to a coverage service (Codecov, Coveralls, or similar) for trend visibility. Phase 1 picks one and documents the choice in `docs/runbooks/ci-pipeline.md`.

## Acceptance criteria
- `.github/workflows/team-ci.yml` exists, path-filtered to Team paths, runs all 5 gates as parallel jobs.
- Sample PR (touching `apps/team/` or `apps/team-web/`) triggers all 5 gates and they all pass.
- Sample PR with a deliberate bug in each gate's domain (e.g., a failing pytest, a wrong PR title format, a committed fake secret) fails the corresponding gate and blocks merge.
- Coverage minimum 75% delta enforced; a PR with low delta coverage fails Gate 1 or Gate 2.
- Notion task ID checker rejects PRs without `(VT-N)` suffix; sample failing PR exists as evidence.
- gitleaks negative test (throwaway branch with fake secret) caught the secret and reported it.
- Cross-product import lint rule from VT-2.1 fires when a Team file imports a Reports file.
- Constant-time secret comparison lint rule from VT-2.6 fires on a deliberate `===` comparison.
- `docs/runbooks/ci-pipeline.md` exists, covers reading failures, emergency bypass procedure, escalation.
- Branch protection on `dev` and `main` requires all 5 CI checks to pass before merge.
- PR title references `(VT-Foundation)`; reviewed by CoderX.

## Out of scope
- Production deploy gates (Fazal's manual approval for prod env) — lives in VT-2.5.
- Cost dashboards for CI minute consumption — if it becomes a concern, separate Observability ask.
- Mutation testing, fuzz testing, contract testing — deferred. Phase 1 ships with unit + integration coverage; advanced testing is a Phase 1.5 consideration.
- Linting tools beyond ESLint, ruff, black, Prettier — keep the toolchain minimal in Phase 1.
- Performance benchmarks in CI — too noisy for PR-level gates; surfaced via Observability dashboards instead.
- Soak runs — those are VT-13 hardening; CI runs only fast checks.
