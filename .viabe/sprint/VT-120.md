# VT-120 — Deployment shape (sibling repos, shared accounts, separate projects)

## Parent task
VT-Foundation — repo, Supabase, RLS, secrets, dev/prod, CI.

## Why this subtask exists (and supersedes parts of earlier subtasks)
This subtask is the canonical architecture-of-record for how Viabe Team coexists with Viabe Reports on shared infrastructure. It was added AFTER VT-17 through VT-21 were authored under the (incorrect) assumption that Team and Reports would share a monorepo. The corrected deployment shape, decided 2026-05-04 by Fazal:
1. **`viabe.ai/team` path-based routing** (not subdomain). Phase 1.5 also migrates Reports to `viabe.ai/report/*`. Homepage at `viabe.ai/` becomes a descriptor offering entry to both products.
2. **Sibling repos**: `github.com/rkecom-in/viabe-team` (this repo) and `github.com/rkecom-in/viabe-reports` (existing, currently named `viabe`; renamed as part of the Phase 1.5 work). NOT a monorepo.
3. **Two production Postgres**: `viabe-reports-prod` (existing Supabase project) and `viabe-team-prod` (new, ap-south-1). Acceptable cost trade-off for isolation.
Where VT-17 / VT-21 / VT-22 / VT-23 (Foundation subtasks) say "monorepo" or imply Reports-coexistence inside the repo, this subtask is the authoritative correction. The Implementation team reads this subtask alongside the original Foundation subtasks; conflicts resolve in favor of this subtask.

## What CoderC must produce
### Deployment shape (the authoritative architecture)
**Vercel team (one) → three projects**
- `viabe-router` (NEW; thin routing shell) — owns `viabe.ai` apex DNS. A minimal Next.js project. Routes:
	- `/team/*` → rewrite to `viabe-team-web` Vercel deployment URL.
	- `/report/*` → rewrite to `viabe-reports-web` deployment URL (this rewrite is added in Phase 1.5 when Reports migrates; Phase 1 may leave Reports routing as-is or add the rewrite preemptively, Fazal's call).
	- `/` — descriptor homepage with two cards: "Sales Recovery Agent for SMBs" → `/team`, and "Location Intelligence Reports" → `/report`.
	- `/api/team/*` → rewrite to `viabe-team-web` (so Razorpay webhook, sign-up API, DSR APIs all land on Team's deployment).
	- `/api/report/*` (Phase 1.5) → rewrite to `viabe-reports-web`.
	- Owns: SPF, DKIM, DMARC, MX (if any) records for the apex; VT-113 DNS edits land on the router's domain config.
- `viabe-team-web` (NEW) — deploys this repo's `apps/team-web/`. Vercel-internal URL (e.g., `viabe-team-web.vercel.app`). Never directly user-facing; always behind the router.
- `viabe-reports-web` (existing; currently named `viabe-web`) — Reports' Next.js. Phase 1 may continue owning `viabe.ai` apex; Phase 1.5 detaches its apex ownership and serves only behind the router.
**Why three Vercel projects, not two**: a thin router project that owns ONLY routing keeps the apex stable across Reports and Team deploys. Without it, a Reports deploy could in principle break the `/team` rewrite if the rewrite rule lived in Reports' `vercel.json`. The router is deploy-coupled to neither product.
**Railway organization (one) → new Team services**
- `viabe-team-orchestrator` (NEW) — LangGraph orchestrator, agent SDK, MCP tools, all FastAPI internal endpoints (`/internal/*` consumed only by Team itself).
- `viabe-team-ingestion-worker` (NEW) — separate process for Apify-driven Methods 7-8 + heavy ingestion jobs (Method 5 cash book photo + voice multimodal, Method 6 QR opt-in surge handling). Separate service to isolate latency from orchestrator hot path.
- Reports' existing Railway services (`viabe-crewai-factory`, `openclaw`, `n8n` per Reports' Section 2) are untouched. They share the org but not env namespaces.
**Supabase organization (one) → separate projects**
- `viabe-reports-prod` (existing) — Reports' Postgres. NOT migrated. NOT shared.
- `viabe-team-prod` (NEW; ap-south-1 Mumbai) — Team's Postgres. Owns migrations 001-039+ from the Team repo. PITR + daily logical snapshots to ap-south-2 Hyderabad per VT-78.
- `viabe-team-dev` (NEW; ap-south-1 Mumbai) — dev environment.
- `viabe-team-soak` (NEW; ap-south-1 Mumbai; per VT-106) — soak environment.
- **Cost note**: ~2x Postgres production cost vs single-project. Accepted trade-off per Fazal's 2026-05-04 decision.
- **Cross-project queries forbidden Phase 1.** Foreign data wrappers / cross-project SQL = Phase 2 conversation.
**Razorpay (one merchant entity) → two webhook URLs**
- Same [Viabe.ai](http://Viabe.ai) merchant KYC (PAN/GST/bank) covers both products. Owned by Fazal; one set of paperwork.
- Reports' webhook continues at `viabe.ai/api/webhook` (or whatever its current path is) with its existing secret.
- Team's webhook at `viabe.ai/api/team/razorpay/webhook` (via router rewrite to Team deployment) with separate `RZP_WEBHOOK_SECRET_TEAM_PROD`.
- Three Team plan IDs created in the existing Razorpay dashboard: `FOUNDING_RZP_PLAN_ID_PROD` (₹2,499), `STANDARD_RZP_PLAN_ID_PROD` (₹4,999), `PRO_RZP_PLAN_ID_PROD` (₹9,999) per VT-89.
- GST line-item accounting separated per product in Razorpay's invoice configuration.
**Twilio (one account) → separate sender ID**
- Same Twilio account as Reports. Reports' existing WhatsApp sender stays.
- Team's WhatsApp Business sender is a NEW phone number with NEW Meta Verified Business Sender approval (display name 'Viabe Team' per VT-108), NEW DLT entity registration extension (VT-111 — same [Viabe.ai](http://Viabe.ai) DLT entity covers both products; per-template registration is per-product).
- Sandbox sender retained for Team dev.
**Anthropic, OpenAI, Resend, Apify (shared accounts; namespaced API keys)**
- Same accounts as Reports. Separate API keys per environment per product (e.g., `ANTHROPIC_API_KEY_TEAM_PROD` vs `ANTHROPIC_API_KEY_REPORTS_PROD`).
- Cost attribution: VT-103 cost dashboard tags every external_api_call with `product=team`; Reports' equivalent tags with `product=reports`. Vendor invoices reconciled by tag.
- Resend: sender domain `viabe.ai` shared (one DMARC posture per VT-113); separate sender ADDRESSES (Team: `noreply@viabe.ai` for transactional + `support@viabe.ai` for replies; Reports may use different).
**LangSmith (one org) → separate projects**
- `viabe-reports-prod` + `viabe-reports-dev` (existing or to-be-created if Reports doesn't currently use LangSmith).
- `viabe-team-prod` + `viabe-team-dev` (NEW per VT-101).
- Tier upgrade (VT-114) applies to the LangSmith ORG; Reports gets paid-tier tracing as a side effect with no additional cost.
**GitHub: sibling repos**
- `rkecom-in/viabe` → RENAMED to `rkecom-in/viabe-reports` (Fazal's action; GitHub auto-redirects old URLs for ~30 days).
- `rkecom-in/viabe-team` → NEW repo (this is where the 105 subtasks live as code).
- No shared package; if a utility is needed in both, it's duplicated Phase 1. Extraction to a third package (`@viabe/shared` on a private npm registry, or a separate `rkecom-in/viabe-shared` repo) is Phase 1.5+ ONLY if drift becomes painful.
- Cross-repo coordination is human governance (Fazal + Clau), not tooling.

## Architectural rules to enforce in this subtask
- Pillar 3: cross-product isolation via separate repos, separate Supabase projects, separate Vercel projects, separate webhook secrets, separate API keys. Sharing happens ONLY at account / org / merchant level where infrastructure cost or human paperwork makes sharing necessary.
- Pillar 8: one source of truth for deployment shape — this subtask. Conflicts between this subtask and any earlier subtask resolve in favor of this subtask. After Phase 1.5 (Reports also migrates to `/report/*`), this subtask is updated; until then, Reports may continue owning `viabe.ai` apex.
- The router project deploys independently. A Reports deploy cannot break Team's `/team/*` rewrite. A Team deploy cannot break Reports' apex.
- Sibling-repo discipline: shared utilities are duplicated Phase 1. Extraction = Phase 1.5+ decision, not Phase 1 default.
- ap-south-1 is prod region; ap-south-2 is backup region. The Foundation subtask's original "prod in ap-south-2" phrasing is wrong; this subtask corrects it.
