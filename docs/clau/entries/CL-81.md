# CL-81 — Schema migrations are path-first, not canonical-8-upfront

DECISION (Fazal 2026-05-18, in response to audit findings CL-77): schema migration execution order is deliberately path-first, not canonical-8-upfront. Ship migrations in dependency order of the subtasks that need them — orchestrator path first (DBOS, RLS infrastructure, idempotency, DSR machinery, rate limits), domain tables as their consuming subtasks land.

Deferred tables and their gating subtasks:
- signup_holding → VT-9.2 (sign-up completion flow). DC2's signup_holding requirement (route unknown_sender to sign-up completion) becomes tractable once this lands.
- customers, transactions → VT-Ingestion subtasks (the ingestion path is what populates them; they shouldn't exist before there's a writer).
- campaigns, campaign_messages → VT-SR-Agent + VT-OwnerSurface (campaign generation + approval flow).
- attributions → VT-Attribution (T+7 close logic).
- pipeline_log was superseded by pipeline_runs + pipeline_steps (already shipped) — the run/step split is more sophisticated than the original spec; no retrofit needed.

ACTION ITEMS:
- Update Notion VT-19 body to reflect this execution order (mark as superseded or revise sections).
- Each deferred table inherits its consuming subtask's body for schema definition.
- VT-3.3c (handler-side Twilio sends) does NOT need any of these tables.
- Audit drift items D-3.3-signup_holding closes when VT-9.2 ships.

RATIONALE: Phase 1 soft launch (July 15) needs the orchestrator path operational, not the full domain schema. Shipping unused tables creates maintenance debt and slows the migration runner. Domain tables get their schema scrutinized in the context of the code that uses them, which is better review hygiene than scrutinizing them in isolation 2 months before any code touches them.

SUPERSEDES: Notion VT-19 'Base schema: 8 core tables + tenant_id' acceptance criteria for shipping all 8 tables in one go. Status of VT-19 should be split: 'tenants + DBOS infrastructure' part is Done; 'domain tables' part is split across VT-9.2 / VT-Ingestion / VT-SR-Agent / VT-OwnerSurface / VT-Attribution.
