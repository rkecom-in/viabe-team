## What CoderC must produce
The `CampaignPlan` Pydantic model (or equivalent strict dataclass) that defines exactly what the agent returns. Validation is structural ā€” a draft that fails validation cannot ship.
1. **`CampaignPlan` model** in `apps/team/src/agent/schema/campaign_plan.py`:
	Top-level fields:
	- `status: Literal['proposed', 'out_of_scope', 'insufficient_data']` ā€” the three legal terminal states (defined in system prompt VT-4.2).
	- `version: Literal['1.0']` ā€” explicit schema version. Major bumps are Type 2 governance.
	- `tenant_id: UUID` ā€” must match the context's tenant_id; mismatch is a hard validation failure.
	- `run_id: UUID` ā€” trace correlation.
	- `generated_at: datetime` ā€” timestamp when the agent produced this draft.
	- `self_evaluate_status: Literal['passed', 'failed_after_revisions'] | None` ā€” populated by the gate (VT-4.5), not the agent.
	When `status == 'proposed'`, the following are required:
	- `campaign_window: CampaignWindow` ā€” `start_at`, `attribution_close_at` (computed T+7), `send_at` (within window).
	- `target_cohort: TargetCohort` ā€” `customer_ids: list[UUID]` (the explicit list, not a query), `cohort_label` (e.g. "60-90 day dormants"), `cohort_size`, `selection_reason` (one paragraph max 200 words explaining why these customers were chosen).
	- `message_plan: MessagePlan` ā€” `template_id` (must match an approved Meta template from VT-13.3), `template_params: dict[str, str]` (key-value substitutions), `language: Literal['en', 'hi']`, `personalization_strategy` enum.
	- `expected_arrr: ARRRProjection` ā€” `low_paise: int`, `high_paise: int`, `confidence: Literal['low', 'medium', 'high']`, `basis` (200 words max explaining the projection's evidence). Point-estimate fields are FORBIDDEN ā€” always a range.
	- `exclusion_list: list[UUID]` ā€” customers explicitly excluded (opted out, recently messaged, etc.) with reasons in `exclusion_reasons: dict[UUID, str]`.
	- `escalation_conditions: list[EscalationCondition]` ā€” structured triggers under which the orchestrator should escalate to the owner before sending (e.g., "if cohort_size > 500, request owner approval first").
	- `evidence_refs: list[EvidenceRef]` ā€” every claim in `selection_reason`, `basis`, and `personalization_strategy` must have a corresponding evidence_ref pointing to a tool call result, an L4 skill corpus doc ID, or an L2 episodic memory entry. No unsupported claims. CodeX rejects schemas without this enforcement.
	When `status == 'out_of_scope'`:
	- `out_of_scope_reason: str` (max 500 chars) explaining why the request fell outside Sales Recovery scope, plus a `suggested_specialist` field if a Phase 2 specialist would handle it (currently `'reputation' | 'marketing' | 'operations'`; orchestrator routes this to a holding queue since Phase 2 is disabled).
	When `status == 'insufficient_data'`:
	- `missing_data: list[MissingDataItem]` ā€” structured list of what's missing (e.g., "customer_ledger has 0 rows", "no campaigns sent in past 90 days"). Each item has `category`, `description`, `suggested_remediation` (e.g., "prompt owner to onboard via Method 1 paper-book ingestion").
2. **Validators** as Pydantic field validators or equivalent:
	- `tenant_id` matches context's tenant_id (cross-validation requires context passed in).
	- `campaign_window.attribution_close_at == campaign_window.send_at + 7 days` (hard rule).
	- `expected_arrr.low_paise <= expected_arrr.high_paise`.
	- `len(target_cohort.customer_ids) == target_cohort.cohort_size`.
	- Every `evidence_refs[].claim_id` matches a string ID present in the relevant prose field (parser scans `selection_reason`, `basis`, etc. for `[E1]`, `[E2]` markers).
	- `template_id` is in the approved-template registry (loaded from `apps/team/config/approved_templates.yaml`; failed lookup is a validation error).
	- `template_params` keys match the template's parameter spec (e.g., a template with `{customer_name}` requires a `customer_name` param).
3. **Serialization helpers** in `apps/team/src/agent/schema/serializer.py`:
	- `to_orchestrator_dict(plan)` ā€” produces the exact dict shape consumed by the orchestrator's persistence layer (`campaigns` table schema in VT-2.3).
	- `from_agent_output(raw)` ā€” parses the agent's raw JSON return into `CampaignPlan`; raises typed validation errors with structured field-paths.
4. **Schema documentation** in `docs/team/campaign-plan-schema.md`:
	- Full markdown spec of the schema.
	- Examples of valid plans for each status.
	- The contract for evolving the schema (Type 2 for major version bump; Type 1 for adding optional fields).
5. **Tests** in `apps/team/tests/agent/test_campaign_plan_schema.py`:
	- Valid `proposed` plan parses cleanly.
	- Valid `out_of_scope` and `insufficient_data` plans parse.
	- Missing required field on `proposed` raises validation error with correct field path.
	- `attribution_close_at != send_at + 7d` raises.
	- `low_paise > high_paise` raises.
	- `tenant_id` mismatch raises.
	- `template_id` not in approved registry raises.
	- `template_params` missing required key raises.
	- Evidence-ref claim IDs that don't appear in prose raises.
	- Evidence-ref claim IDs that DO appear but lack a matching evidence entry raise.
	- Round-trip: serialize ā†’ deserialize ā†’ equal.

## Architectural rules to enforce in this subtask
- Pillar 1: the schema is the typed contract between the agent and the orchestrator. The agent's `output` field is `CampaignPlan` typed; raw dicts are forbidden once parsed.
- Pillar 7: structural honesty. The schema makes it impossible to omit confidence ranges or evidence refs. Pillar 7's principles ā€” honesty about ARRR, evidence-backed claims, no false certainty ā€” are enforced by validators, not by code reviewer judgment.
- Pillar 8: schema lives in ONE module. Schema-version-bumps are Type 2 governance. Adding optional fields is Type 1; making fields required, removing fields, or changing semantics is Type 2.
- Schema validation runs in the self-evaluate gate (VT-4.5) BEFORE the agent's draft can ship. A draft that fails schema validation never reaches the orchestrator's persistence layer.
- `tenant_id` validation is cross-context. The validator receives the context's tenant_id at runtime and rejects any draft where the agent fabricated a different one.
- `template_id` registry is config-driven. New approved templates from VT-13.3 land in `approved_templates.yaml`; the schema picks them up without code changes.
- `evidence_refs` enforcement is real, not aspirational. Tests prove that a draft with an evidence-less claim is rejected.

## Acceptance criteria
- `CampaignPlan` Pydantic model exists with all listed fields and discriminated union over `status`.
- All listed validators implemented and tested.
- Approved-template registry config exists; schema validates against it.
- Evidence-ref enforcement: prose claims without backing refs fail; refs without prose mentions fail.
- Serialization helpers produce the expected dict shape for `campaigns` table persistence.
- All test scenarios pass (valid happy paths, all listed failure modes, round-trip).
- Schema documentation at `docs/team/campaign-plan-schema.md` includes examples and evolution contract.
- mypy --strict passes on the schema module.
- A draft with hallucinated `tenant_id` fails validation in test.
- PR title references `(VT-SalesRecovery-Agent)`; reviewed by CoderX (mandatory ā€” schema is the structural enforcer of Pillars 1+7).
