# VT-300 — VTR live run-control (DBOS pause / override / steer)

## Why
VT-293 Activity ships Pause/Steer/Override as **intent-recording** controls: the action is
logged to `ops_audit` (observable, auditable) + the UI says "… requested", but it does NOT
yet enforce on the live agent. Cowork ruling 2026-06-02: a control that silently no-ops is
the dead-end the "nothing dead-ended" rule forbids — recording intent + an honest label + a
tracked follow-up (this row) is the acceptable v1. This row is the actual enforcement.

## What
Wire the recorded control intents to real orchestrator action on a RUNNING DBOS workflow:
- **Pause** — suspend an in-flight pipeline_run (DBOS pause / hold).
- **Steer** — inject a corrective directive into the agent's next step.
- **Override** — force a step outcome / redirect.
A VTR's request (ops_audit `control_requested`) becomes an actual orchestrator control,
gated by the VT-290 assignment scoping (only on assigned tenants) + audited.

## Acceptance criteria
- A pause request actually suspends the running workflow (DBOS), reversible.
- Steer/override change the run's subsequent behaviour, audited.
- Scoped (VTR → assigned tenants only) + every control audited (ops_audit).
- The Activity UI reflects real state (Paused, not "pause requested") once enforced.

## Out of scope
- The intent-recording + listing (VT-293, shipped).

## Notes
Backlog, post-launch. Needs DBOS workflow-control primitives (pause/resume/redirect on a
running workflow). Rostered per Cowork ruling 2026-06-02 (VT-293 honesty fix).

## Implementation — 2026-06-03, Review (Fazal build-go + adversarial-review honest scoping)

**Honest semantics (Fazal ruled):** override IS VTR-issuable; pause = **coarse stop-at-next-node**,
NOT mid-step (the agent brain runs in ONE synchronous DBOS step → cancel_workflow can't suspend
mid-step). Target = the REAL `webhook_pipeline_run → dispatch_brain` path. The meaningful coarse
control point is the **send boundary**: a VTR's pause/steer/override HOLDS the campaign fan-out
before any customer send.

- `migrations/078_run_controls.sql` — control queue (server-derived tenant, control_type, PII-scrubbed
  directive, status). Deny-all RLS.
- `api/ops_runcontrol.py` `POST /api/orchestrator/ops/run-control` — the ENFORCEMENT-LEG authz the
  review flagged (team-web auth is fail-open here): **NO tenant param** — re-derives the run's tenant
  from pipeline_runs + re-checks operator_assignments server-side, fail-CLOSED; audits control_executed
  on success, control_denied on refusal; directive PII-scrubbed.
- `run_control_handler.py` `consume_pending_control` — atomic (FOR UPDATE SKIP LOCKED) single-claim;
  wired into `_campaign_execute_node` → ANY pending control HOLDS the send.
- team-web `lib/ops/activity.ts` `flagRunControl` → fast pre-check + forwards to the authoritative
  endpoint (`forwardRunControl`).
- Tests: 7 real-PG canary (assigned→recorded+audited server-derived tenant; unassigned→403+deny-audited
  +no-write; cross-tenant impossible/no-tenant-param; 404; bad-secret/type; PII-scrub; consume claims-
  oldest-once) + 11 team-web vitest (flagRunControl forward/pre-check) + 20 regression (campaign/e2e/
  routing). ruff/tsc/eslint green.

**FLAGGED (rostered follow-up if needed):** finer mid-ReAct-loop control (pause mid-step / steer
between sub-steps) requires splitting dispatch_brain into per-node DBOS steps (XL restructure) — out
of scope; v1 is honest coarse send-boundary control.
