# VT-209 — Field-mapping reasoner + dedupe primitive

## Why this row exists

Every connector ends with a sample of arbitrary-shaped rows. Two shared steps before they land in canonical tables:
1. Map source fields → canonical fields (name/phone/cart_value/last_seen/etc.)
2. Dedupe against existing customers (phone_hash)

Both are connector-agnostic. Build once. Used by VT-207, VT-208, every future connector + VT-6 manual methods.

## Scope

### Field-mapping reasoner

`apps/team-orchestrator/src/orchestrator/integrations/field_mapping.py`:
- Input: sample data (first 50 rows) + connector spec's `canonical_fields_hints`
- Step 1: heuristic match on column name (exact + fuzzy)
- Step 2: where ambiguous (confidence < 0.85), invoke orchestrator-agent (Opus 4.7 + caching) to propose mapping
- Step 3: render proposal to owner via Integration Agent conversation; owner confirms or corrects
- Confirmed mapping persisted in `tenant_field_mappings` table (per tenant per connector)

### Dedupe primitive

`apps/team-orchestrator/src/orchestrator/integrations/dedupe.py`:
- Input: canonical row about to be inserted
- Lookup: phone_hash (VT-184 substrate already exists)
- If match: merge (don't duplicate); track `merged_from` lineage
- If no match: insert new customer row
- Tagged with `acquired_via` so downstream observability knows the source mix

### Confidence thresholds (uniform per CL-19 / VT-6 architectural rules)

- < 0.7 → ask owner via clarifying question
- 0.7-0.85 → commit with notification to owner (visible in Ops Console next time owner opens)
- ≥ 0.85 → commit silently

## Acceptance criteria

1. Heuristic match handles common cases: "Phone" / "Mobile" / "Contact Number" → canonical `phone`
2. LLM-assisted match handles harder cases: "Last Touch Date" → canonical `last_seen` (Opus reasons)
3. Owner corrections persisted: same Sheet, second pull → uses saved mapping, doesn't re-ask
4. Dedupe: same phone via Sheet then Shopify → 1 customer row with 2 `acquired_via` tags, NOT 2 rows
5. Confidence telemetry: per-field confidence recorded in pipeline_steps for observability
