## What CoderC must produce
A consolidated runbook set: ≥9 runbooks covering known operational scenarios. Each runbook is tabletop-drilled before launch. Lives at `docs/runbooks/`. Indexed at `docs/runbooks/README.md`.
1. **Runbook format** (template at `docs/runbooks/0000-template.md`):
	- Scenario title (clear single-sentence description).
	- When to use (specific triggers / detection signals).
	- Severity classification (P0 / P1 / P2 / P3).
	- Response procedure (numbered steps with expected duration each).
	- Verification (how to confirm resolution).
	- Communication template (owner / customer / authority where applicable).
	- Post-incident actions (postmortem requirements; edge-case suite addition).
	- Tabletop drill record (date + participants + findings).
	- Last updated + author.
2. **Required runbooks (≥9)**:
	- **runbook-0001: Breach response** (from VT-8.8) — P0/P1/P2/P3 severity classification, DPDPA 72h notification, communication templates EN+HI.
	- **runbook-0002: Region migration** (from VT-8.7) — ap-south-1 to ap-south-2 failover; Phase 1 documented but not drilled (acceptable risk).
	- **runbook-0003: DLT renewal** (from VT-13.6) — annual entity renewal + per-template renewal on content change.
	- **runbook-0004: Resend cutover** (from VT-13.8) — DNS edits, sender domain verification, rollback to dev account.
	- **runbook-0005: Refund execution failures** (from VT-10.5) — partial refund failure path; Fazal manual investigation steps.
	- **runbook-0006: Soak failure response** (from VT-13.1) — categorize failure, fix, restart soak; launch gate impact.
	- **runbook-0007: Day-39 refund conversation escalation** (from VT-9.5) — owner replies DISCUSS; Fazal calls owner; outcome documentation.
	- **runbook-0008: Razorpay Live cutover + rollback** (from VT-13.4) — production env var swap, first-5-subscription monitoring, rollback to sandbox.
	- **runbook-0009: SupportBot escalation handling** (from VT-9.8) — Telegram /resolve workflow, SLA breach handling.
	- **runbook-0010: Email reputation warmup** (from VT-13.8) — W1→W4 ramp; deliverability anomaly response.
	- **runbook-0011: Founding-tier saturation** (from VT-10.6 + VT-11.5) — widget + sign-up flow behavior at 100/100.
	- **runbook-0012: Tenant cost runaway** (from VT-12.3 + VT-12.5) — ₹50 ceiling breach response; pattern alert investigation.
	- At least 9 of the above 12 must be authored Phase 1; remaining can be Phase 1.5.
3. **Tabletop drill protocol** (documented at `docs/runbooks/tabletop-drill-protocol.md`):
	- Each runbook drilled before launch via simulated scenario.
	- Participants: Fazal + CoderX (minimum); Quality Engineer + Pipeline Engineer where relevant.
	- Drill: walk through procedure with synthetic incident; identify ambiguity / gaps; revise runbook.
	- Drill record appended to runbook itself.
	- Quarterly re-drill post-launch.
4. **Index** at `docs/runbooks/README.md`:
	- List of runbooks with severity coverage + last-drill date.
	- Categorical view: by trigger type (security / compliance / billing / availability / quality).
	- Cross-reference to ADRs (VT-14.2) where applicable.
5. **Maintenance**:
	- Every production incident → runbook updated within 7 days of resolution.
	- Quarterly review: stale runbooks (no drill in 6 months) flagged.
6. **Tests**:
	- Markdown linting passes.
	- Each runbook has all required sections per template (synthetic check).
	- Tabletop drill records present for all Phase 1-required runbooks.
