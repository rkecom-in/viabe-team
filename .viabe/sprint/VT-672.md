# VT-672 — one common `escalate` tool

## Gap (from `scripts/check_capability_gaps.py`, `KNOWN_CAPABILITY_GAPS[unified_escalate]`)
Escalation is duplicated per-lane — integration/finance/tech/accounting/sales/onboarding each own a
bespoke escalate. There is NO single common `escalate` tool a specialist reaches to hand a decision
back to the Manager/owner. Every specialist needs it; the framework's common-tool surface lacks it.

## Build (DONE 2026-07-18)
ONE common `escalate` tool (ADVISORY, structured marker return) on the new
`tools_common.COMMON_ADVISORY_TOOLS` surface; cataloged (`_COMMON_ADVISORY_ANN` + defining surface);
`required_tools_reachable` unions the advisory surface so a specialist may `required_tools=("escalate",)`.
Owner-comms stays Manager-only (§1.2).

**Scope deviation (deliberate, CC call):** the 8 per-lane escalate sites were left UNTOUCHED —
NOT migrated to delegates. Rationale (scout survey): #1 `escalate_to_fazal` is the Manager's LIVE
terminal signal keyed by NAME in `dispatch._classify_terminal` (must not move); #4-8 are DEAD
scaffolds de-wired from the roster (unreachable to any live LLM); #2/#3 (integration/onboarding,
live) are one-line log+string bodies — delegation adds indirection + import surface for zero
behavior. Consolidation-by-delegation is optional cleanup IF the advisory lanes are ever re-wired.
