# CL-143 — PR #22 mis-branch incident resolution

Resolution to PR #22 mis-branch incident (Claude Code cut chore/dep-audit-recon off feat/vt-orchestrator-agent-skeleton instead of main; PR #22 squash-merged VT-3.9 + audit doc together as commit dda82d7 on main, bypassing pre-merge review of VT-3.9).

Chosen path — Option 4 (not Claude Code's recommended Option 1):

(a) NO REVERT. Main stays as-is. dda82d7 holds. Code is byte-identical to what PR #21 would have merged; dep pins (langgraph, langchain, langchain-anthropic) are correct; doc is approved. Reverting punishes correct work for a process error and introduces revert-commit log noise + a fiddly partial pyproject/uv.lock revert that risks new bugs.

(b) FORCED POST-HOC REVIEW. CodeX + Fazal review the VT-3.9 code as it sits in main with the same rigor as a pre-merge gate. Any findings ship as fix-forward PRs (VT-CI prefix for build/lint, VT-3.9-fix-N for code-level). If review clean, we've at least closed the loop honestly.

(c) DISCIPLINE RULE (separate CL entry) — every new branch's first action is `git fetch origin && git checkout main && git pull && git checkout -b <name>`. Boilerplate, no judgment required. Adds to brief template.

(d) CLOSE PR #21 with note pointing to this CL entry, not a buried 'Claude Code mis-branch error' comment. Future search for 'why was #21 closed' finds the lesson.

(e) ANNOTATION COMMIT on main: `git commit --allow-empty -m "NOTE: dda82d7 contains both PR #22 audit doc AND PR #21 (VT-3.9 PR 1/N) content. Merged together via mis-branch; see CL-143. Post-hoc review completed; no revert."` — makes the git log honest about what dda82d7 actually contains.

Why this over Option 1 ("recommended, lowest churn"): Option 1 is the right outcome shape (no revert) but with no learning baked in. "We'll just review post-hoc next time" is how review gates erode. (c)+(d)+(e) make the failure mode harder to repeat and the git history honest.

Why not Option 2 (full revert): punishes correct doc work, introduces revert noise, partial-revert problem still exists on re-PR.

Why not Option 3 (surgical revert): pyproject.toml partial-revert (drop create_react_agent code but keep pins) is hand-surgery, not a revert. New bug risk for marginal governance gain over Option 4.

CL-131 sequencing log update: VT-3.9 is merged (not 'pending review'). PR #21 closed. VT-3.4 PR 1/3 unblocked once dep migrations land.
