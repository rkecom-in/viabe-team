# VT-216 — pipeline_steps tsvector free-text index

## Why this row exists

VT-201 PR-2 shipped free-text search across `envelope_payload` using `ILIKE '%' || $q || '%'` because no tsvector substrate existed. At Phase-1 scale (~10K steps/day × 30-day window = ~300K rows) ILIKE is acceptable. Past that scale, full-table-scans on every search dominate latency.

This row replaces the ILIKE fallback with a proper PostgreSQL FTS index that the PR-2 search endpoint can switch over to without breaking the API contract.

## Scope

### Migration

```sql
-- 037_pipeline_steps_fts.sql
ALTER TABLE public.pipeline_steps
    ADD COLUMN envelope_search_tsv TSVECTOR
    GENERATED ALWAYS AS (
        to_tsvector('english', coalesce(envelope_payload::text, ''))
    ) STORED;

CREATE INDEX pipeline_steps_envelope_search_tsv_gin
    ON public.pipeline_steps USING GIN (envelope_search_tsv);
```

(Decision on `english` analyzer: Phase-1 is English-only for operator-facing search; multi-lingual analyzer = own row when needed.)

### Query swap

`apps/team-web/lib/ops/data-access.ts::fetchHistoricalSteps` — when `q` is non-empty, switch query from:
```sql
... WHERE envelope_payload::text ILIKE $q ...
```
to:
```sql
... WHERE envelope_search_tsv @@ plainto_tsquery('english', $q) ...
```

### Backfill

`GENERATED ALWAYS AS ... STORED` populates the column on INSERT and on ALTER TABLE for existing rows automatically — no separate backfill needed. Confirm via `EXPLAIN ANALYZE` post-deploy.

## Acceptance criteria

1. Migration applies cleanly + populates `envelope_search_tsv` for all existing pipeline_steps rows
2. GIN index exists + is used by the query planner (`EXPLAIN` shows index scan, not seq scan)
3. VT-201 PR-2 history-view free-text search returns same row set as ILIKE fallback for a known synthetic dataset (5-row matrix covering: phrase match, single-word, no-match, special-character, case-difference)
4. Query latency p95 < 200ms on a synthetic 1M-row table
