# VT-601 — the journey queue didn't cross-fill or skip answered fields

## Finding (VT-598 opus judge, deployed dev)
The walker applies an owner reply ONLY to the presented question and advances +1: a descriptive
business_type correction carrying the 'about' substance still got the 'about' gap question next
turn (judge: context_retention 2/5). The flow-beat path (multi_field_single_message) handles
multi-fact messages fine; the journey-queue path didn't.

## Fix (deterministic, journey.handle_reply)
1. Narrow cross-fill: a business_type/category answer with ≥4 tokens fills a still-pending 'about'
   gap (only when 'about' is unanswered + unskipped; a bare 'yes' confirm never triggers it).
2. Record-and-move-on invariant: cursor advance skips EVERY queue entry whose field is already in
   answers/skipped — an answered field is never re-presented (also covers recomposition-duplicate
   entries).
Tests: descriptive-correction → about filled + city presented next; bare-yes → about still asked;
duplicate-field entry skipped. 24/24 journey suite green vs throwaway DB.
