VT-KnowledgeArchitecture.

## What CoderC must produce
The composition module that takes a tenant_id + cohort context and produces the knowledge slice of the bundle: relevant L1 facts + recent L2 events + applicable L3 priors + retrieved L4 docs. Plus an audit log so any agent decision can be traced back to the knowledge inputs that informed it.
1. **Composition module** at `apps/team/src/kg/composition.py`:
	- `compose_knowledge_bundle(tenant_id, cohort_key, run_id, budget_tokens=4000) -> KnowledgeBundle`.
	- Token budget split: L1 (1000), L2 (1500), L3 (300), L4 (1200). Slack 0; if a layer is empty, the freed budget does NOT redistribute (keeps composition deterministic).
	- Cross-layer dedup: if an L4 doc references the same pattern an L3 prior covers, only the L3 number is included (data); the L4 doc is included for the heuristic context. No content collisions.
2. **Per-layer composition**:
	- L1: business profile (compact projection from VT-5.3) + customer ledger summary (from VT-5.2) + last 5 campaigns (from VT-5.4).
	- L2: 20 most recent episodic events (via VT-7.4 `recent_events` with `summary_only=True`).
	- L3: lookup matching pattern via VT-7.6; quarantine respected.
	- L4: query VT-7.7 with derived natural-language query from cohort_key + business_type + recency_band; top 3 docs.
3. **`KnowledgeBundle` typed structure**:
	- `l1_facts: dict` (business profile + ledger summary + recent campaigns)
	- `l2_events: list[EpisodicEventSummary]`
	- `l3_prior: L3Pattern | None` (None when quarantined or no-match)
	- `l4_docs: list[L4Document]` (3 max)
	- `composition_audit: CompositionAuditEntry` — metadata about what was retrieved, token counts per layer, budget pressure (was L4 trimmed because budget exhausted?)
4. **Audit logging**:
	- Migration `024_composition_audit.sql`: table `composition_audits` with `id`, `tenant_id`, `run_id`, `composed_at`, `cohort_key`, `l1_token_count`, `l2_event_count`, `l2_token_count`, `l3_pattern_id` (nullable), `l4_doc_ids: UUID[]`, `total_token_count`, `budget_pressure_layers TEXT[]` (which layers got trimmed).
	- Tenant-scoped RLS.
	- Every composition writes one row. The orchestrator can later look up "what knowledge did the agent see for run X?" and reconstruct.
	- Retention: 18 months matching L2 episodic.
5. **Failure handling**:
	- L1 query failure: composition fails the run (L1 is non-optional).
	- L2/L3/L4 query failure: composition continues with empty layers; failure logged in `composition_audit.budget_pressure_layers` (with marker like `'l4_failed'`).
	- The orchestrator's error framework (VT-3.6) routes total composition failure as `database_error`.
6. **Tests** at `apps/team/tests/kg/test_composition.py`:
	- Happy path: synthetic tenant with all 4 layers populated → valid bundle within budget; audit row written.
	- Budget pressure: synthetic tenant with massive L2 (500 events) → L2 trimmed to fit; budget_pressure includes 'l2'.
	- Quarantine: tenant <180 days → l3_prior is None; audit reflects.
	- L4-only fallback: tenant past quarantine but cohort_key has no L3 pattern → l3_prior=None, L4 docs retrieved, audit reflects.
	- L1 failure: simulated DB error on L1 query → entire composition fails (not partial).
	- L2 failure: simulated DB error on L2 → composition succeeds with empty l2_events; audit logs failure.
	- Audit completeness: every test composition produces a complete audit row.
	- Cross-tenant isolation: tenant_A composition queries cannot leak tenant_B data via any layer.

## Architectural rules to enforce in this subtask
- Pillar 1: composition is in the orchestrator (`apps/team/src/kg/`); the agent receives the bundle. CodeX rejects PRs that move composition into agent code.
- Pillar 8: one composition function. Adding a new knowledge layer (Phase 2) requires extending this function and updating the budget split. CodeX rejects PRs that bypass composition.
- Pillar 7: audit log is mandatory. Every composition writes an audit row. CodeX rejects PRs that skip audit on success path or failure path.
- Token budget split is config-driven (`apps/team/config/composition_budgets.yaml`). Changes are Type 1 except for adding new layers (Type 2).
- Per-layer failure isolation: L2/L3/L4 failures don't kill the run; L1 failure does (L1 is the floor).
- Cross-layer dedup is structural; CodeX confirms by reviewing the L3+L4 interleaving logic.
- Composition is deterministic given identical inputs (modulo timestamps): re-running with same tenant_id + cohort_key + frozen DB state → same bundle. Useful for replay and debugging.

- Quarantine respected through composition.
- Cross-tenant isolation tested.
- Audit completeness: every composition logged.
- Reproducibility: identical inputs → byte-identical bundle (modulo timestamps).

## Out of scope
- The agent's interpretation of the bundle (the agent reasons; this subtask delivers).
- Composition for Phase 2 specialists (each Phase 2 specialist gets its own composition function).
- Owner-facing visibility of audit logs (Phase 2; ops-only in Phase 1).
- Real-time composition streaming (Phase 1 is one-shot at context-build time).
- Composition caching (Phase 2 if perf becomes an issue; correctness first).
- Cross-run learning from audit ("compositions that led to high-FAS reports use these L4 docs more") — deferred to long-after-Phase-1 once Phase 1 has produced enough audit data.
