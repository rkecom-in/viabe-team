# CL-168 — Mis-cited CL-137 as strict VT-3.4 blocker

Earlier this session I twice referenced CL-137 (which I mis-numbered as CL-138 in conversation, the actual rule about 'Phase 1 codebase must not run on deprecated APIs') as if it strictly blocked VT-3.4 behind ALL CRITICAL+HIGH dep-audit items including Step 3 (tailwind-merge, frontend). That framing was wrong.

CL-137's actual wording: 'VT-3.4 PR 1/3 is paused until at least the CRITICAL + HIGH migrations RELEVANT TO apps/team-orchestrator land.' The rule already scopes by app. Step 3 (frontend, apps/team-web) is not relevant to the orchestrator. VT-3.4 is already unblocked by Step 2's merge (PR #24).

Failure mode: I quoted the headline of CL-137 without re-reading the body. Same class as CL-152 (state-assertion without verification) and CL-155-class (PR review without diff inspection). The fix: when citing a CL entry to support a recommendation, re-fetch the entry verbatim if it's load-bearing.

IMPLICATION: no amendment to CL-137 is needed. VT-3.4 is unblocked. The (VT-3.9-fix-1) → VT-3.4 → Step 3 sequence stands, and Step 3 does NOT block VT-3.4. Steps 4-5 (frontend bundle + UNUSED cleanup) also don't block any orchestrator work and can run in parallel.

This correction doesn't change any locked decisions — it cleans up my reasoning in-session.
