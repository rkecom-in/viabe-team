# VT-620 — ops management: make alerts actionable

## Why (Fazal, 2026-07-08)
"An alert that I don't understand is just useless. We need better ops management system.
This way we won't be able to control the system or even know what's happening." The alert
stream is an unactionable firehose — Fazal can't triage or see system state.

## Root cause (evidence)
- **326 leaked `convo-harness-pack-%` TEST tenants** in dev DB (purged this session) —
  the test packs don't clean up (FK-blocked deletes). Test traffic **polluted the anomaly
  p95 baselines** AND **fired prod-grade alerts**.
- Alert layer has: **no test/prod split** (test runs page like prod), **no correlation**
  (one struggling scenario → 5+ loose alerts: schema_rejection + self_eval_rejected +
  cost_anomaly + 2× latency_anomaly, all one root event), **no context** (a raw number,
  no scenario/model/what-happened/action), **miscalibrated severity** (a correctness-gate
  escalation pages as CRITICAL).
- Alerts module = `src/orchestrator/alerts/` (triggers.py detect + `_SEVERITY_BY_KIND`;
  baselines.py p95; dispatch.py + clients.py routing→Telegram; scheduler.py cadence).
  Already has a suppression pattern to mirror: `_volume_is_mock_only` (VT-489).

## Scope (this row — highest-value core; correlation + ops-console are the next layer)
- **P1 test/prod segmentation** (biggest noise kill): `_is_test_tenant` suppressor at the
  top of `detect_critical_for_run` + `detect_slow_triggers` (fail-safe: never suppress a
  real prod alert on read error); exclude test runs from the baseline p95.
- **P2 severity recalibration**: `self_eval_rejected` / `schema_rejection` /
  `aborted_hard_limit` → warning/info (a correctness gate escalating is NOT critical);
  reserve CRITICAL for real crash / stuck-never-terminal / data-loss / outbound_failure.
- **P4 test-tenant reaper**: scheduled FK-safe reaper for leaked `convo-harness-pack-%`
  tenants (mirror `orphan_reaper`); fix the `run_full_pack`/`convo_harness` per-scenario
  cleanup FK bug (tenant-scoped, FK-ordered delete).

## Next layer (VT-620 follow-on or new row)
- P3 correlation + enrichment: group alerts by run/tenant into ONE incident with root
  cause + scenario/model/tenant-type/action.
- P5 Ops Console surface: grouped incidents + live cost/latency per tenant (extends the
  VT-619 `get_agent_usage_breakdown` ops read).
