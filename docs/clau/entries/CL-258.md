Lifecycle-state field stays a SEPARATE owner-surface subtask. Recommend this migration goes before VT-4.2. ViabeTeam_Sprint row pending sprint-schema confirmation.

## Trigger
VT-37 (CampaignPlan v1.0, PR #32) shipped the canonical schema at `orchestrator.agent.schemas.campaign_plan` but intentionally did NOT migrate consumers (one-subtask-one-PR). `main` is now split-brain: v1.0 schema exists, all consumers still on v0.1 at `orchestrator.types.campaign_plan`.

## Six v0.1 call sites on main
Production (4): supervisor.py (:31, :37, :92 — sales_recovery_node parses stub JSON), collapse.py (:32, :50 — tenant guard + INSERT INTO campaigns), agent/sales_recovery_stub.py (:23 — hardcoded_campaign_plan returns v0.1 shape), state/agent_graph_state.py (:23, :47 — AgentGraphState.campaign_plan field type).
Tests (2): test_supervisor.py (:42), test_collapse.py (:79).

## Why this is NOT a type-swap
- v1.0 is a discriminated union — `CampaignPlan.model_validate(...)` call sites must switch to the `parse_campaign_plan(...)` TypeAdapter accessor.
- v1.0 has no top-level template_id / body_params / proposed_at / proposed_by / subscriber_id. template_id→MessagePlan; proposed_by dropped (run_id covers correlation); single subscriber→target_cohort.customer_ids list.
- collapse.py's `INSERT INTO campaigns` writes v0.1-shaped columns. Migrating the type alone breaks the write path — the campaigns table schema (or the INSERT) must be reconciled to v1.0 IN THE SAME SUBTASK.

## Subtask scope (when created)
IN: six call-site migrations to the v1.0 discriminated-union accessor + campaigns INSERT/column reconciliation to v1.0 shape. Area: Specialist Agent (or Database, brief-author call at creation). Sprint 2.
OUT: lifecycle-state field (approved/rejected/sent/failed) — remains a separate campaigns-schema / owner-surface subtask. If the INSERT reconciliation turns out to hard-depend on the lifecycle field existing first, Claude Code surfaces that at pre-flight; not pre-hunted now.

## Sequencing recommendation
Migration subtask should go NEXT, before VT-4.2 (system prompt). Reason: split-brain touches the campaigns write path; stacking further agent subtasks on a base whose contract its callers don't honor compounds risk. Fazal's call — may have a launch-gate reason to prefer VT-4.2 first.
