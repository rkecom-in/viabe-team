## Why this parent exists
Reports product had `Viabe_AI_Technical_Reference_v2_0.md` as its engineering reference — the human-readable companion to `CLAUDE.md`. It was written long after most code was already in production, which meant it documented a frozen snapshot rather than an evolving spec. Reviewers had to cross-reference live code against stale docs. Team avoids that by writing the reference incrementally, in lockstep with code: when an architectural component lands, its section in the reference lands in the same PR. By the end of Sprint 9, the reference is current with reality.
The parent also owns the ADR set (architecture decision records) and runbook set (operational procedures). ADRs capture the *why* of irreversible choices; runbooks capture the *how* of recurring operations (secret rotation, breach response, refund execution, KYC onboarding, Razorpay reconciliation). Without runbooks, every operation depends on Fazal's memory.
## What this parent owns
1. `Viabe_Team_Technical_Reference_v1_0.md`: the human-readable engineering reference. Sections mirror Reports' v2.0 structure (Executive Overview, Architecture, Data Flows, per-component sections, Database Schema, Workflows, Deployment, Operations).
2. ADR set: at minimum, ADRs covering LangGraph-as-orchestrator vs alternatives, Anthropic Agent SDK choice, Apache AGE for KG, k-anon-at-construction-time, day-39-as-deterministic-SQL, Razorpay tier structure, KYC vendor selection.
3. Runbook set: secret rotation, Supabase backup restore, breach detection response, refund execution, day-39 evaluation manual override (for legal/exceptional cases), KYC vendor escalation, Razorpay reconciliation, Meta template re-submission, monthly impact report regeneration.
4. Source-of-truth banners across the four canonical docs: `concept-team-pillars.md`, `concept-team.md`, `CLAUDE.md`, `Viabe_Team_Technical_Reference_v1_0.md`. Each banner restates the hierarchy and points readers at the higher-authority doc.
## Architectural rules binding every subtask
- Pillar 8 (no patchwork): documentation is a first-class deliverable, not optional. PRs that add or change a component must update the relevant reference section in the same PR. CI rejects PRs that touch a component without touching its doc section (after VT-14.1 lands).
- The Engineering Reference cannot supersede `concept-team-pillars.md`. When the two disagree, pillars win and the reference is updated.
- Each runbook is drilled at least once before launch. "We have a runbook" without a drill is theater.
- ADRs are immutable once accepted. Reversing an ADR requires a new ADR that supersedes the old one, with both kept in the repo for lineage.
- The four-doc hierarchy banners must be present at the top of each canonical doc, with consistent wording.
