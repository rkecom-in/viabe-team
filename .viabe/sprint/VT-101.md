## What CoderC must produce
LangSmith integration for the entire pipeline. Every orchestrator dispatch generates a run_id; that run_id propagates through LangGraph nodes, agent SDK invocations, every MCP tool call, every DB write, every Telegram alert. End-to-end: one run_id traces from inbound webhook to outbound WhatsApp message.
1. **run_id generation**:
	- Orchestrator entry point (`apps/team/src/orchestrator/entry.py`) generates a UUID v4 at the top of every dispatch: webhook intake, scheduled trigger fire, recovery retry. Stored as `run_id` in the LangGraph state.
	- The run_id is also the LangSmith trace ID (LangSmith accepts external trace IDs via `LANGSMITH_TRACING_PROJECT` + `langsmith.trace_id` parameter).
	- For nested dispatches (e.g., agent invocation calling sub-tools), child spans inherit the parent run_id.
2. **LangSmith client setup** at `apps/team/src/observability/langsmith.py`:
	- Initialize LangSmith client with `LANGSMITH_API_KEY`, `LANGSMITH_PROJECT='viabe-team-dev'` or `'viabe-team-prod'`.
	- Wrap every LangGraph node + tool call in `@traceable` decorators.
	- For non-LangGraph code paths (webhook handler, DeliveryBot, scheduled triggers), wrap with explicit `langsmith.trace()` context manager.
3. **Propagation rules**:
	- HTTP outbound from orchestrator (Twilio, Razorpay, Resend, Anthropic): include `X-Trace-Id: <run_id>` header where the destination supports it (Anthropic does; Twilio/Razorpay/Resend ignore).
	- DB writes: every row in `pipeline_log` (VT-12.2) carries the run_id.
	- Telegram alerts: include run_id in the message footer.
	- Frontend submissions (sign-up): generate a frontend_run_id; the backend webhook generates its own run_id and links the two via `parent_run_id`.
4. **LangSmith project hygiene**:
	- Two projects: `viabe-team-dev` (development) and `viabe-team-prod` (production). Env-var-driven.
	- Daily quota monitoring: LangSmith billing tier (configured in VT-13.9) limits free tier to ~5K traces/day; production exceeds this so paid tier required at launch. CodeX confirms env config + billing tier.
5. **PII discipline in traces**:
	- LangSmith traces include prompt content + tool args + responses. PII redaction MUST apply before traces are sent.
	- Apply VT-12.4 PII redaction to all values entering LangSmith. Phone numbers hashed; message bodies hashed; customer names tokenized.
	- Documented in `docs/team/langsmith-pii-policy.md`. CodeX rejects PRs sending raw PII to LangSmith.
6. **Tests** at `apps/team/tests/observability/test_langsmith.py`:
	- Synthetic dispatch → run_id generated → LangSmith span created (synthetic LangSmith client mock).
	- Nested dispatch (orchestrator → agent → tool) → child spans correctly parented to root run_id.
	- run_id appears in pipeline_log + Telegram alert footer.
	- PII redaction: synthetic prompt with phone number → LangSmith span contains hashed phone, not plaintext.
	- Project isolation: dev environment writes to `viabe-team-dev`; prod to `viabe-team-prod`.
	- LangSmith API failure: trace continues; orchestrator doesn't crash on tracing failure (graceful degradation).
## Architectural rules to enforce in this subtask
- Pillar 1: orchestrator generates run_id; agent + tools receive it. CodeX rejects PRs where agent generates its own run_id.
- Pillar 8: one run_id namespace. CodeX rejects parallel correlation IDs (e.g., a separate `request_id` for HTTP layer).
- PII never raw in LangSmith. CodeX rejects PRs that bypass redaction.
- Graceful degradation: tracing failure does not kill the pipeline. CodeX confirms via test.
- Project separation dev vs prod is structural; mixing data is an audit failure.
