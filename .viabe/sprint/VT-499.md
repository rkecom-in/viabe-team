# VT-499 — campaign_window computed server-side (LAUNCH-BLOCKER, closes the win-back)
The dominant win-back parse failure (VT-496-named: proposed.campaign_window value_error, 3/3) is the LLM failing the CampaignWindow._window_validity validator (start>=now, tz-aware, end>start) — i.e. the model botches current-date arithmetic. VT-493 already injected today's date + a valid example + the validator rule, yet Haiku fails it. The window is MECHANICAL (now → now+~7d), not business judgment. Fix: the SYSTEM owns campaign_window — compute it server-side at the coercion/parse layer (now, tz-aware → +7d) and override/fill the LLM's value, so the LLM focuses on cohort/message/objective (its judgment) and never has to produce a valid date window. Robust across ALL model tiers (Haiku incl). Supersedes the need for VT-497 (Opus tier) for THIS field. Plan-first-ish (money-path schema/coercion) → Cowork audit-on-diff. Don't weaken the validator for genuinely-bad OTHER fields.
