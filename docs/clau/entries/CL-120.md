# CL-120 — PR-fix-7 writeup ingested. (C) leading

Claude Code writeup (gist: https://gist.github.com/rkecom-in/2e60326f0f446cfd02da36d61246f18b) read in full. Summary of findings:

INVENTORY: 14 sites, not 10. 12 TENANT_SCOPED + 1 PRE_TENANT (_lookup_tenant) + 1 CROSS_TENANT (_within_rate_limits writes tenant bucket + workspace sentinel on one conn). Sites 6/7/12 new since CL-71 (PR-fix-1, #15, VT-3.3c). _record_inbound is dead. Revised brief must re-grep at implementation time.

GUC NAME BUG: brief said `app.current_tenant_id`. Actual schema (000b) uses `app.current_tenant`. Brief as written would have shipped a silent no-op — CI passes (superuser bypass), prod silently fails.

000b VERBATIM: schema's own design note already establishes service-role bypass as INTENDED path. Single-role uniformity was an artifact of working off CL-71 summary instead of 000b.

DBOS REALITY: dbos/_app_db.py:273 issues CREATE SCHEMA on launch. DBOS needs schema-create + table-owner privileges. Brief's `DATABASE_URL → app_user` swap would break DBOS outright. Any plan that demotes the global connection role is wrong. Same for LangGraph PostgresSaver (graph.py) — second infra consumer the brief missed.

OTHER FINDINGS: rate_limit_buckets missing from grants; @DBOS.step serializes args so connection-injecting decorators break checkpointing (concrete reason to prefer ctx manager primitive); close_run/close_webhook_run UPDATE by id only — silent no-op under real role without GUC (Pillar 7 observability hole); 63 tests today prove zero real isolation coverage.

THREE OPTIONS PRESENTED:
(A) Two-pool: app_role pool for 12 writers + service-role pool for sites 13-14 + DBOS + checkpointer. 1 migration. Doubled pool sizing. Per-site accessor choice greppable.
(B) Single app_role pool + SECURITY DEFINER functions for pre-/cross-tenant. Still needs separate privileged DBOS connection — not actually single-pool. SECURITY DEFINER is an RLS bypass mechanism, arguably against CL-88's spirit even if technically not a policy variant.
(C) NEW — not in CL-116/CL-117. Single privileged pool (unchanged). tenant_connection() does SET ROLE app_role + set GUC + paired RESET ROLE/clear GUC on exit. DBOS and checkpointer untouched. 1 migration. Smallest diff.

Claude Code recommends (C); falls back to (A) only if SET ROLE fights the pool.

CLAU PUSHBACKS LOGGED:
1. SET ROLE leak class is worse than GUC leak class (silent permission_denied for next borrower, non-deterministic). psycopg pool reset hooks may make it impossible — needs verification, not hope.
2. 'One real gate' framing for (C) is rhetorical not structural. Audit surface is identical to (A). Decide on actual merits (DBOS-safe, single migration, smaller diff) not framing.
3. Forcing test for SET ROLE + psycopg pool + DBOS reuse should be spiked BEFORE committing to (C), not 'if things go wrong'. Cost: ~30 LOC. Saves whole implementation throwaway if (C) is broken.

FAZAL CALL PENDING: approve spike-first approach, or take (C) on Claude Code's evidence and ship.
