## What CoderC must produce
The Razorpay integration for Team. Three plan tiers configurable via env. Webhook handler for `subscription.activated`, `payment.captured`, `payment.failed`, `subscription.cancelled`, `subscription.charged`. All routes through the orchestrator's state machine (VT-3.2) for phase transitions.
1. **Plan configuration** in `apps/team/config/plans.yaml`:
	- Plan IDs (created in Razorpay dashboard, IDs captured here):
		- `founding` (env var `FOUNDING_RZP_PLAN_ID_DEV` / `_PROD`)
		- `standard` (env var `STANDARD_RZP_PLAN_ID_DEV` / `_PROD`)
		- `pro` (env var `PRO_RZP_PLAN_ID_DEV` / `_PROD`)
	- Display prices read from env: `FOUNDING_PRICE_PAISE` (default 249900), `STANDARD_PRICE_PAISE` (default 499900), `PRO_PRICE_PAISE` (default 1499900). Never hardcoded.
	- Plan currency: INR only Phase 1.
2. **Webhook endpoint** at `POST /api/team/razorpay/webhook` (apps/team-web/app/api/team/razorpay/webhook/route.ts):
	- HMAC signature verification using `RZP_WEBHOOK_SECRET_DEV` / `_PROD` env var.
	- Idempotency on Razorpay's `event.id` via `razorpay_webhook_events` table (migration 033, NOT cross-product with Reports' equivalent).
	- Routes events:
		- `payment.captured`: triggers conversion-from-trial flow if first payment; transitions phase via VT-3.2 from `trial` (or `trial_extended`) to `paid_active`; writes `paid_conversion_at`.
		- `subscription.activated`: confirms subscription; persists `razorpay_subscription_id`.
		- `subscription.charged` (recurring monthly billing): updates `cumulative_fees_paid_paise` on the tenant.
		- `payment.failed`: logs + Telegram alert; phase transitions to `paid_at_risk` after 3 consecutive failures.
		- `subscription.cancelled` (Razorpay-initiated, not via day-39): phase transitions to `cancelled`.
3. **Subscription creation flow**:
	- `POST /api/team/razorpay/subscribe` body: `{tenant_id, plan_tier, customer_card_token}`.
	- Validates tenant_id matches the session (owner authenticated via VT-9.7's session cookie OR via WhatsApp verification flow for trial-to-paid conversion).
	- Calls Razorpay's `/subscriptions` API to create the subscription with the tenant's chosen plan.
	- Stores `razorpay_subscription_id` on the tenant.
	- Returns subscription ID + status to frontend.
4. **Cross-product isolation**:
	- Reports product's existing Razorpay webhook is a SEPARATE webhook URL with a SEPARATE secret. Team has its own `RZP_WEBHOOK_SECRET_TEAM_*`.
	- Reports' tables (`reports.orders`, etc.) are not touched by Team's webhook. CodeX confirms via lint rule (`no-cross-product-db-access`) extending VT-8.1's wrapper rules.
	- Both products bill separately; a customer who subscribes to both has 2 distinct subscriptions in 2 distinct Razorpay accounts (or 2 distinct plan IDs in one account; documented decision at `docs/team/razorpay-account-strategy.md`).
5. **Tests** at `apps/team/tests/billing/test_razorpay_webhook.py`:
	- Each event type: synthetic webhook → correct phase transition + DB writes.
	- Bad signature → 403.
	- Idempotency: same [event.id](http://event.id) twice → second call no-ops.
	- 3 consecutive payment.failed → paid_at_risk transition.
	- subscription.cancelled → cancelled transition.
	- Cross-product safety: Team webhook receives a Reports event by mistake → rejected (different signature secret).
	- Cross-tenant: webhook for tenant_A can't update tenant_B.
	- Plan tier resolution: subscribe call with `plan_tier='founding'` resolves to correct Razorpay plan ID from env.
## Architectural rules to enforce in this subtask
- Pillar 7: prices NEVER hardcoded. CodeX rejects PRs with rupee literals. CI grep for `₹499` / `₹2499` / `₹14999` / number `1499900` etc. as substrings in non-config files.
- Pillar 8: one webhook handler; one subscription creation endpoint. Reports' equivalents stay in their own product.
- Cross-product DB access lint extension: Team code can't import from `reports/*` paths and vice versa. Documented at `docs/team/cross-product-isolation.md`.
- Razorpay sandbox vs Live: env-driven (`RZP_KEY_ID_DEV` vs `_PROD`); CodeX rejects PRs hardcoding `rzp_test_*` or `rzp_live_*` keys.
- 3 consecutive payment.failed = paid_at_risk: structural threshold; raising/lowering is Type 2.
- Webhook always returns 200 to Razorpay even on internal errors (per Razorpay spec; we log + retry internally).

- **VT-93 N1 (HARD GATE — Cowork 20260605T104200Z, financial-subagent audit):** concurrent-caller double-refund window. The VT-93 refund advisory lock (`db.refund_executions.claim_or_get`) commits BEFORE the external Razorpay refund HTTP call, so two concurrent `execute_refund` callers for one `(tenant_id, refund_reason)` could each fire a vendor refund; today only the deterministic `idempotency_key` prevents it, and there is no concurrency canary. Before Razorpay Live keys are provisioned: (a) a concurrent-caller integration canary asserting EXACTLY ONE vendor refund call per `(tenant, reason)`; (b) verify the live Razorpay client honors the deterministic `X-Idempotency-Key` (a re-used key returns the prior refund, not a new one). **Live keys MUST NOT go in until both pass.**
- **VT-329 (HARD GATE — Cowork 20260605T115500Z, i18n):** the Devanagari `\b` DSR/opt-out patterns are DEAD (`pre_filter_gate.py`) and Hinglish negation/opt-out is uncovered in the refund classifier — a Hindi/Hinglish DSR or opt-out can be silently dropped and (with live Razorpay) auto-refund a customer who declined. **Razorpay Live keys MUST NOT go in until VT-329 lands** (Unicode-aware DSR/opt-out matching + Hinglish negation, real-PG canary).
