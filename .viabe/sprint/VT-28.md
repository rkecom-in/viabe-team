## What Claude Code must produce
Four scheduled triggers, each implemented as a `@DBOS.scheduled` workflow. **DBOS handles cron scheduling + idempotency natively** — no external scheduler (no apscheduler, no n8n, no separate cron container). Each trigger is idempotent on `(tenant_id, trigger_type, window_id)` via DBOS's exactly-once semantics.
**Architectural shift from prior framing:** the old design used apscheduler-in-process or LangGraph cron nodes with a custom `scheduled_trigger_runs` table for idempotency. DBOS replaces both: it provides scheduled workflow primitives natively with native idempotency. The custom idempotency table is no longer needed (DBOS workflow_id handles it).
1. **Weekly cadence trigger** (`@DBOS.scheduled(cron='0 9 * * MON', timezone='Asia/Kolkata')`):
	- Fires every Monday 9:00 AM IST.
	- For each active subscriber (phase ∈ {`paid_active`, `paid_at_risk`}): start child DBOS workflow `weekly_campaign_proposal(tenant_id, week_id)` with workflow_id = `weekly:{tenant_id}:{iso_week}`.
	- Child workflow goes through the orchestrator-agent + supervisor pipeline (since this requires reasoning about which campaign mode to propose). Trigger reason: `weekly_cadence`.
	- Skip condition: subscriber has an outstanding `proposed` or `approved` campaign older than 3 days.
	- DBOS idempotency on workflow_id = exactly-once per (tenant, week).
2. **Attribution close trigger** (`@DBOS.scheduled(cron='0 2 * * *', timezone='Asia/Kolkata')`):
	- Fires daily 2:00 AM IST.
	- Scans for campaigns where `attribution_close_at <= now()` AND status `sent` AND `attribution_closed_at IS NULL`.
	- For each: start child workflow `close_attribution(campaign_id)` with workflow_id = `attribution_close:{campaign_id}`.
	- **Fully deterministic SQL.** NO orchestrator-agent invocation. NO supervisor handoff. NO LLM. Aggregates `attributions` table for the campaign window, finalizes ARRR, updates `total_arrr_paise`, marks `attribution_closed_at`, emits `attribution_closed` step_record.
3. **Day-39 evaluation trigger** (`@DBOS.scheduled(cron='0 6 * * *', timezone='Asia/Kolkata')`):
	- Fires daily 6:00 AM IST.
	- Scans for tenants where `paid_conversion_at + 39 days <= now()` AND `phase ∈ {paid_active, paid_at_risk}` AND no day-39 evaluation already recorded.
	- For each: start child workflow `day39_evaluation(tenant_id)` with workflow_id = `day39:{tenant_id}`.
	- **Fully deterministic SQL** (calls VT-Billing VT-10.4 evaluator). NO LLM. Based on result:
		- ARRR ≥ 2x cumulative fees: emit `day39_continue` event; phase stays.
		- ARRR < 2x: emit `day39_refund_triggered` event; transition to `refunded` via apply_transition; refund execution + refund-conversation flows take over.
	- DBOS idempotency = same tenant on consecutive days does not re-evaluate after decision recorded.
4. **Monthly impact report trigger** (`@DBOS.scheduled(cron='0 8 1 * *', timezone='Asia/Kolkata')`):
	- Fires 1st of month, 8:00 AM IST.
	- For each `paid_active` subscriber: start child workflow `monthly_impact_report(tenant_id, month_id)` with workflow_id = `monthly:{tenant_id}:{YYYY-MM}`.
	- **Fully deterministic** data prep + PDF generation (no LLM). Aggregates previous month's metrics; generates PDF via VT-OwnerSurface VT-9.6; emails via Resend.
	- Skip if subscriber's `paid_conversion_at` is < 30 days ago.
5. **DBOS scheduled workflow registration**:
	- All 4 triggers registered in `apps/team-orchestrator/src/orchestrator/scheduled_triggers.py`.
	- Cron expressions in source for type safety; documented also in `docs/team/scheduled-triggers.md` for reference.
	- Phase 1: IST-only timezone. Per-tenant timezone support is Phase 2.
6. **Trigger reasoning** (for weekly cadence only):
	- When the weekly cadence fires, the child workflow goes through orchestrator-agent reasoning (NOT pre-filter — scheduled triggers are trusted internal events).
	- Orchestrator-agent decides which mode (Personalized Win-Back / Segment Offer Burst / High-Value Targeting) based on tenant context (L0 + L1-L4).
	- Then handoff to Sales Recovery specialist via supervisor.
7. **Tests** in `apps/team-orchestrator/tests/orchestrator/test_scheduled_triggers.py`:
	- Each trigger tested with synthetic time-clock injection via DBOS testing harness.
	- Idempotency: re-running the same workflow_id is a no-op (DBOS short-circuits).
	- Skip conditions correctly applied.
	- Day-39 both branches tested.
	- Attribution close: synthetic campaign + transactions → correct ARRR aggregation → `attribution_closed_at` updated.
	- **DBOS auto-resume:** crash mid-trigger-execution → DBOS resumes from last completed step.

## Architectural rules to enforce in this subtask
- **Pillar 1 (revised):** scheduled triggers that need reasoning (weekly cadence) go through orchestrator-agent + supervisor. Triggers that are pure SQL (attribution close, day-39, monthly impact) DO NOT invoke the orchestrator-agent or any specialist. They're deterministic.
- **Pillar 8 (no patchwork):** DBOS handles cron + idempotency. No external scheduler. No custom idempotency table (DBOS provides it).
- Day-39 evaluation DETERMINISTIC. Any LLM call in the day-39 path is a Type-3 contract violation — Claude Code MUST reject it; Cowork MUST bounce a plan-ready signal that allows it.
- Attribution close aggregation DETERMINISTIC SQL. No LLM, no probabilistic logic.
- DBOS scheduled workflow registration is the single mechanism. Adding a new trigger = new `@DBOS.scheduled` function.
- Each trigger child workflow is itself a `@DBOS.workflow` with `@DBOS.step` granularity. Auto-resume on crash.
- workflow_id derivation deterministic and documented per trigger.
