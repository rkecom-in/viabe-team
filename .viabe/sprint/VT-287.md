# VT-287 — Inbound-first WhatsApp message pipeline + auto-reply

## Why
The customer-comms model is **inbound-first** on the owner's WABA: the customer
messages the business, which (a) anchors consent as customer→business (fiduciary;
Viabe = processor) and (b) opens the 24h session window for a free-form reply. This
is the runtime that turns a connected WABA (VT-286) into actual conversations.

## What
Inbound webhook → orchestrator brain pipeline:
- First-contact intro: business-forward, **Viabe disclosed as operator**,
  **automated-assistant disclosed**. Localized per tenant.
- Opt-in capture: customer replies YES → `consent.record_consent(...,
  consent_method='wa_inbound_optin', consent_text_version=<from
  .viabe/consent-text.md>)` (VT-8.5 machinery, method-agnostic — no rework).
- Business-voice reply within the 24h session window.
- STOP / opt-out → `consent.opt_out(...)`.

## Acceptance criteria
- inbound → intro sent → customer "YES" → `record_of_consent` row written →
  `has_consent` true (real-PG, synthetic CL-422).
- "STOP" → `opted_out_at` set → `has_consent` false.
- Intro discloses Viabe-as-operator + automated-assistant.
- Reply respects the 24h window (template vs free-form path).

## Out of scope
- WABA onboarding/provisioning (VT-286).
- Email/SMS hook channels that drive the inbound (VT-288).
- The consent machinery itself (VT-8.5 / #226 — this row calls it).
