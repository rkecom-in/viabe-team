# VT-123 — VT-OpsConsole

## Why this parent exists
Fazal needs to see what's happening across 10 design-partner tenants in real time during the July 15 soft launch, and replay any incident step-by-step. Telegram alerts (VT-121) and SQL queries alone won't scale beyond the first 3-5 tenants. This parent delivers the Ops UI at `viabe.ai/team/ops`, gated to Fazal-only auth, showing live runs, tenant dashboards, and n8n-style step replay.
Reference design document: 'Viabe Team — Pipeline Observability Architecture' section 5 (Notion page under [Viabe.ai](http://Viabe.ai) parent, created 2026-05-12). Locked decision: Clau_Session_Log CL-21.
MVP scope for July 15 soft launch = 3 views: workspace overview, per-tenant dashboard, run replay. Deferred views (cost dashboard, alerts, incidents) ship Phase 1.5 or later. Telegram surfaces (VT-103 /costs, VT-121 alerts) fill the gaps for soft launch.

## What this parent owns
1. The `apps/team-web/ops/` Next.js route group with auth-gated access.
2. The `/ops` workspace overview view (live counters, in-flight runs panel, top tenants).
3. The `/ops/tenants` list view + `/ops/tenants/<tenant_id>` per-tenant dashboard.
4. The `/ops/runs/<run_id>` n8n-style run replay (the load-bearing view).
5. Phone-token resolution flow: click `[resolve]` in step record, logs to VT-80 audit log, displays phone number.
6. 'Export step as test fixture' button: copies the input_envelope as an edge-case suite scenario (VT-13.2 fixture).
7. Server-side data access via service-role connection to viabe-team-prod (NEVER exposed to client).
8. Auth: hardcoded to Fazal's user UUID for Phase 1; Phase 2 adds RBAC.

## Architectural rules
- Read-only Phase 1 (no mutations from Ops UI). Phase 2 may add force-replay, abort-run, refund-override.
- Server-side data access only. Client never sees raw Supabase queries; the Next.js API routes mediate.
- Phone-token resolution ALWAYS logs to VT-80 audit log. Every resolve = one audit row.
- Pillar 7: Ops UI is a debugging surface, not a customer surface. PII appears (resolved phone numbers) only when operator explicitly requests + audit-logged.
- Pillar 8: one Ops UI. CodeX rejects parallel admin surfaces. Telegram surfaces are CLI complements, not duplicates.
- Performance: per-tenant dashboard loads in <2s for tenants with up to 100 runs in last 30 days. Run replay loads in <1s for runs with up to 50 steps.

## Subtasks under this parent
1. **VT-123.1** — Auth gate + `apps/team-web/ops/` route group + Fazal-only UUID check + dev/prod env discrimination.
2. **VT-123.2** — `/ops` workspace overview: live counters via Supabase realtime subscription; top tenants; in-flight runs panel.
3. **VT-123.3** — `/ops/tenants` list + per-tenant dashboard at `/ops/tenants/<id>` (profile, 30-day pipeline_runs timeline, recent customers, recent campaigns, privacy access log).
4. **VT-123.4** — `/ops/runs/<run_id>` run replay view: vertical waterfall of pipeline_steps, click to expand each, formatted JSON envelopes, decision_rationale text, tool_calls list, phone-token resolve button.
5. **VT-123.5** — 'Export step as test fixture' button: serialize input_envelope to edge-case-suite-compatible format, copy to clipboard, log the action.
6. **VT-123.6** — Phone-token resolution UI flow: click [resolve], confirm dialog, server-side resolution + audit log write, render phone in-place.
7. **VT-123.7** — Ops UI smoke tests + Playwright end-to-end test on dev environment.

- All 7 subtasks Done.
- Auth gate verified: non-Fazal user sees 403.
- Workspace overview shows live counters and updates within 5s of new run starting.
- Per-tenant dashboard renders for a tenant with 30 days of synthetic data in <2s.
- Run replay renders a 30-step run in <1s; each step card expands to show full envelope JSON formatted.
- Phone-token resolution: clicking [resolve] writes one row to VT-80 audit log and displays the phone number.
- Export-step-as-test-fixture button copies a valid edge-case suite scenario to clipboard.
- Playwright end-to-end test passes against dev environment.
- Lighthouse perf >85 on Ops UI (lower bar than landing site; this is internal).
- Fazal personally smoke-tests on Vercel preview deployment.
