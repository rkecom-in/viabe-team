# CL-182 — temperature=0 rejected by Claude Opus 4.7

## What happened

VT-3.4 PR 1/3 brief specified, in tests/orchestrator/test_supervisor.py:

    model = ChatAnthropic(model="claude-opus-4-7", temperature=0)

Fazal ran the integration canary. The test failed at the orchestrator_agent's first LLM call:

    anthropic.BadRequestError: Error code: 400 - 'temperature' is deprecated for this model.

Claude Opus 4.7 (the current frontier model) does not accept the temperature parameter at all. It's not 'accepts and ignores' or 'accepts with a warning' — it's 'rejects the request.'

## Why I wrote it that way

I carried 'temperature=0 for deterministic test runs' as a reflex pattern from general Python testing experience without validating against the target model's accepted-parameters list. I did not Context7-check the ChatAnthropic kwargs against the Opus 4.7 surface. I did not grep the existing codebase for how _MODEL was constructed before writing the test setup.

The production _MODEL constant in orchestrator_agent.py (pre-existing from PR #21) is:

    _MODEL = ChatAnthropic(model="claude-opus-4-7", max_tokens=4096)  # type: ignore[call-arg]

No temperature. The author of PR #21 already knew. My brief regressed on knowledge that was already in the codebase — a one-grep check would have caught it.

## Fix

One-line in test_supervisor.py:

    model = ChatAnthropic(model="claude-opus-4-7")  # type: ignore[call-arg]

Pushed to same branch (feat/vt-orchestrator-supervisor-handoff), no new PR. CL-149-clean since the fix is internal to PR #26's scope and PR has not merged.

## What the failure usefully proved

The Command.PARENT canary is still unfired — we never reached the routing primitive because the upstream LLM call was rejected. BUT we did learn:
- build_supervisor_graph composed successfully
- Graph entered orchestrator_agent as the first node correctly
- All imports + topology + node registration work

So when the temperature fix lands and the canary re-runs, we're testing only the LLM-decision + handoff + Command.PARENT routing + stub-execution path. Everything upstream is already verified.

## What I'm changing

New discipline: when a brief specifies model-construction code (ChatAnthropic, OpenAI clients, any model-specific SDK call), I must:
1. Grep the existing codebase for prior usage of the same model identifier and copy its kwarg pattern as the default.
2. Only deviate if I have a specific reason AND have Context7-validated the kwargs against the model's accepted-parameters page.

The deeper pattern across CL-178, CL-181, and this entry is: I keep inferring verification from adjacent signals. The unifying discipline rule: when verification requires touching a specific surface (this model's kwargs; this test's run-status; this library's exact signature), check that specific surface. Adjacent signals don't transfer.
