## What CoderC must produce
The privacy audit log: a dedicated, append-only, tamper-evident log of privacy-relevant events. Append-only via RLS DENY UPDATE/DELETE. Hash-chained: each row's hash is computed from the prior row's hash + current row content, so detection of tampering is mechanical.
1. **Schema** at migration `027_privacy_audit_log.sql`:
	- Table `privacy_audit_log`:
		- `id` UUID PK
		- `event_type` TEXT NOT NULL (enum: `consent_recorded`, `opt_out_received`, `reconstitution_completed`, `dsr_export_requested`, `dsr_export_verified`, `dsr_export_completed`, `dsr_delete_requested`, `dsr_delete_verified`, `dsr_delete_completed`, `tenant_isolation_violation`, `context_isolation_violation`, `breach_alert_fired`, `breach_alert_resolved`, `k_anonymity_admission`, `data_residency_verified`)
		- `tenant_id` UUID NULLable (some events are workspace-level)
		- `customer_phone_e164_hash` TEXT NULLable (hashed per VT-7.1; never plaintext)
		- `payload` JSONB (event-specific structured data, NEVER PII)
		- `severity` TEXT (info / warning / critical)
		- `prev_hash` TEXT (hash of the immediately prior row in the chain)
		- `current_hash` TEXT (sha256(prev_hash || event_type || tenant_id || timestamp || payload_canonical_json))
		- `created_at` TIMESTAMPTZ DEFAULT now()
	- INDEX `(event_type, created_at DESC)`, `(tenant_id, created_at DESC)`, `(customer_phone_e164_hash)`.
	- RLS: DENY UPDATE, DENY DELETE (append-only). SELECT scoped: tenant-bound rows visible to the tenant's owner; workspace-level events visible to Fazal only.
2. **Writer** at `apps/team/src/privacy/audit_log.py`:
	- `log_privacy_event(event_type, tenant_id, customer_phone_hash, payload, severity)` — the sole writer.
	- Atomically reads the latest row's `current_hash` (under SELECT FOR UPDATE on a sentinel row to serialize chain construction), computes the new row's hash, inserts.
	- On read after fork (concurrent inserts): the database's serializable isolation OR the sentinel-row lock prevents fork. Documented decision: Phase 1 uses an advisory lock (`pg_advisory_xact_lock(123456789)`) to serialize.
3. **Verification function** at `apps/team/src/privacy/audit_verify.py`:
	- `verify_chain(start_id, end_id) -> VerificationResult`. Walks the chain forward from start to end; recomputes hashes; reports any mismatch.
	- Run nightly (added to VT-3.5) on the full chain; mismatch fires critical alert. The chain shouldn't grow huge in Phase 1 (~10K events/month estimate); full nightly verification is feasible.
4. **Coverage**:
	- All 8 prior subtasks emit at least one event type each into this log.
	- VT-8.1 → `tenant_isolation_violation` (when raised).
	- VT-8.2 → `context_isolation_violation`.
	- VT-8.3 → `k_anonymity_admission`.
	- VT-8.5 → `opt_out_received`, `reconstitution_completed`.
	- VT-8.6 → `dsr_*` events.
	- VT-8.7 → `data_residency_verified` (on every orchestrator startup).
	- VT-8.8 → `breach_alert_fired`, `breach_alert_resolved`.
	- Sign-up flow (VT-9.2) emits `consent_recorded`.
5. **Read API** at `apps/team/src/privacy/audit_query.py`:
	- `list_events(tenant_id, event_types, limit, since) -> list[PrivacyAuditEvent]` for owner-facing queries (tenant-scoped via RLS).
	- `list_workspace_events(event_types, limit, since) -> list[PrivacyAuditEvent]` for Fazal-facing queries (no tenant filter; service-role only).
	- `verify_event_integrity(event_id) -> bool` exposes the chain-walk for individual event verification.
6. **Tests** at `apps/team/tests/privacy/test_audit_log.py`:
	- Append-only: synthetic UPDATE attempt → RLS-blocked; DELETE attempt → RLS-blocked.
	- Hash chain: 10 synthetic events; chain verification confirms; tampering with one row's payload makes verification fail at that row.
	- Concurrent inserts: 100 parallel writes → chain remains intact; advisory lock serializes correctly.
	- Cross-tenant: tenant_A's owner cannot read tenant_B's privacy events.
	- Workspace events: tenant owners cannot read workspace-level events.
	- Coverage: every prior subtask's primary event type is recorded by a synthetic exercise.
	- PII absence: serialized payloads never contain plaintext phone or name (regex check).
	- Performance: 100K-event chain verification under 30 seconds.

## Architectural rules to enforce in this subtask
- Pillar 7: append-only is structural (RLS DENY UPDATE/DELETE). Hash chain catches tampering even if RLS were misconfigured.
- Pillar 8: one writer, one query API. Adding a new event type requires updating the enum + writer; CodeX rejects shadow audit paths.
- PII never in payload — only hashes. CodeX rejects payloads containing plaintext customer fields.
- Nightly verification is structural; mismatch alerts to Fazal.
- The advisory lock for serialization is documented; concurrent insert correctness verified by test.
- Owner-facing read access scoped via RLS to their own tenant's events; workspace-level events visible only via service role.
