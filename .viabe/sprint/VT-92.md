## What CoderC must produce
The day-39 evaluator: a deterministic SQL function that, given a tenant, computes their ARRR (recovered revenue attributed to Viabe campaigns) over the period since `paid_conversion_at`, computes their cumulative fees paid, and returns a structured decision: `continue` (ARRR ≥ 2x fees) or `refund_eligible` (ARRR < 2x fees). NO LLM. Reproducible byte-for-byte across runs given identical DB state.
1. **Evaluator module** at `apps/team/src/billing/day39_evaluator.py`:
	- `evaluate_day39(tenant_id) -> Day39Decision`.
	- `Day39Decision`: `tenant_id`, `evaluated_at`, `paid_conversion_at`, `days_since_conversion`, `arrr_paise`, `cumulative_fees_paise`, `ratio` (arrr / fees), `decision: Literal['continue', 'refund_eligible']`, `reasoning: str` (deterministic templated explanation), `evaluator_version: str`.
	- Threshold: `decision = 'refund_eligible' if ratio < 2.0 else 'continue'`. Threshold is a Type 3 commitment (the 2x is in [concept-team.md](http://concept-team.md) §6).
2. **ARRR aggregation**:
	- Reuses VT-5.5 `get_attribution_data` with `since=paid_conversion_at`, `until=evaluated_at`.
	- Sums `arrr_paise` across all campaigns whose `attribution_close_at` falls in the window.
	- Pending attributions (not yet closed) are NOT counted — only finalized attribution.
3. **Cumulative fees aggregation**:
	- SQL on `razorpay_webhook_events` filtered to `subscription.charged` events for this tenant, summing `payment_amount_paise` since `paid_conversion_at`.
4. **Reproducibility**:
	- Same `tenant_id` + same DB state → byte-identical decision (modulo `evaluated_at` timestamp).
	- `evaluator_version` is a constant string (e.g., `'1.0.0'`); changing the evaluator logic requires bumping this version + Type 3 governance.
	- Decision rows persisted to `day39_evaluations` table (migration `034_day39_evaluations.sql`) so future re-runs can verify against historical decisions.
5. **Trigger integration**:
	- VT-3.5 day-39 trigger queries eligible tenants (paid_conversion_at + 39 days).
	- For each: invokes `evaluate_day39(tenant_id)`; persists the result; based on decision:
		- `continue` → emit `day39_continue` event; phase stays.
		- `refund_eligible` → emit `day39_refund_triggered` event; VT-9.5 refund-conversation engine takes over.
6. **Suppression after CONTINUE**:
	- VT-9.5 records 90-day suppression after a CONTINUE response. The trigger respects this: tenants with `last_day39_continue_at < 90 days ago` are skipped.
	- Re-evaluation on day 129 (39 + 90) if criteria still met.
7. **Tests** at `apps/team/tests/billing/test_day39_evaluator.py`:
	- Synthetic tenant with ARRR ₹10K, fees ₹5K → decision=continue (ratio 2.0; boundary).
	- Boundary at exactly 2.0: continue (\>= 2.0).
	- ARRR ₹4999, fees ₹5K → refund_eligible (ratio 0.9999).
	- Reproducibility: same synthetic state, two evaluation calls → identical fields (modulo timestamp).
	- Persistence: result written to `day39_evaluations`; subsequent reads verify match.
	- Pending attribution: synthetic with ₹5K closed + ₹5K pending → only ₹5K counted; ratio against fees uses closed only.
	- Cross-tenant: tenant_A evaluation cannot include tenant_B's data.
	- Suppression: tenant with continue 60 days ago → trigger skips.
	- Suppression expiry: same tenant, 91 days post-continue → trigger fires.
	- Refund branch → emits day39_refund_triggered event consumed by VT-9.5.
## Architectural rules to enforce in this subtask
- Pillar 4: deterministic SQL only. CodeX rejects LLM invocation paths in this evaluator.
- Pillar 7: 2x threshold is structural. CodeX rejects any conditional / per-tenant override.
- Pillar 8: one evaluator. CodeX rejects parallel implementations (e.g., a "manual override evaluator" for ops).
- 2x ratio is Type 3; lowering or raising requires board governance.
- 90-day suppression after CONTINUE is in VT-9.5; this evaluator respects it.
- Pending attribution NOT counted — only what's actually finalized within the 39-day window. Documented in `docs/team/day39-evaluation.md`.
- Reproducibility test re-runs evaluation against same DB state and asserts byte-identical decision.
- `evaluator_version` constant; bumping it requires Type 3 governance.

## Out of scope
- Pro-rata refund calculations (Phase 1 is full-refund-or-continue; Phase 2 may add partial).
- Partial-month fees (Phase 1 monthly billing only).
- Multi-tenant household evaluation (out of scope).
- Quarterly evaluation reps (Phase 2; Phase 1 is one day-39 per tenant lifecycle, with re-eval after 90-day suppression expiry).
- Owner contesting the evaluation outcome (Phase 2; Phase 1 is take-it-or-leave-it via VT-9.5 refund flow).
