## What CoderC must produce
The L4 skill corpus: a per-document table backed by pgvector for semantic retrieval. The Sales Recovery Agent retrieves relevant documents at context-build time (VT-4.3) for inclusion in the bundle. Phase 1 ships with at least 30 hand-authored documents covering Indian SMB sales-recovery patterns.
1. **Schema** at migration `023_l4_skill_corpus.sql`:
	- Table `l4_documents`:
		- `id` UUID PK
		- `title` TEXT NOT NULL
		- `body` TEXT NOT NULL (markdown; max 4,000 chars per doc; documents that exceed get split into multiple)
		- `body_embedding` VECTOR(1536) (pgvector; OpenAI ada-002 or Anthropic embeddings; chosen at authoring time and pinned in `models.yaml`)
		- `tags` TEXT[] (e.g., `['cafe', 'tier_2', 'dormant_cohort', 'whatsapp_template_design']`)
		- `applies_to_business_types` TEXT[] (NULL = applies to all)
		- `applies_to_city_tiers` TEXT[] (NULL = applies to all)
		- `priority` INT (1-5; higher = more likely to be retrieved when relevant)
		- `authored_by` TEXT (the human who wrote it; for accountability)
		- `version` INT DEFAULT 1
		- `superseded_by` UUID NULL (when a document is replaced; old doc remains for audit)
		- `created_at`, `updated_at`
	- INDEX `(applies_to_business_types) gin`, `(applies_to_city_tiers) gin`, IVFFLAT on `body_embedding`.
	- NOT tenant-scoped; corpus is global.

## Architectural rules to enforce in this subtask
- Pillar 4: corpus is hand-authored facts/heuristics, NOT model-generated. CodeX rejects PRs that auto-generate corpus content.
- Pillar 5: no fine-tuning. Corpus is retrieval-augmented; the model is unchanged.
- Pillar 8: one corpus, one retrieval API. CodeX rejects per-method shadow corpora.
- Embedding model is pinned. Switching requires full re-embedding (one-shot).
- Authorship is recorded. Every document has `authored_by`. PRs without it fail review.
- Documents are versioned via `superseded_by`. Updates create new rows; old rows stay for audit.
- Adding a new authored document is a routine PR. Removing or superseding is Type 1 (record + summary in PR description).
- The seed script must be re-runnable safely: re-running on existing rows is a no-op (idempotency on `id` derived from filename hash).
