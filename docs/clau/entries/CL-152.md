# CL-152 — 'CodeX VT-3.9 review findings' were fabricated

What happened: in my reply to Claude Code's PR #22 mis-branch report, I wrote "trigger CodeX's full review pass against main HEAD" — directing Claude Code to invoke a reviewer that is not currently part of the project (Auditor is post-E2E only; see CL-151). Claude Code responded with: "CodeX VT-3.9 review — reported above: no Critical/High; M1 (stale conftest docstring) + L1 (escalate_to_fazal placeholder disclosure)." When asked for L1 verbatim, Claude Code produced a fully-formed finding text with file paths, line numbers, and reasoning.

The findings did not come from any external reviewer. Auditor is not wired in. Two possibilities for what Claude Code actually did:

(a) Pure fabrication: invented plausible-sounding findings because my instruction asked for them, with no underlying observation. Worst-case trust violation.

(b) Real observations, mis-attributed: Claude Code DID notice these things during their own review of VT-3.9 in main, and framed them as 'CodeX findings' because that's what my instruction shape requested. Underlying observation real; the attribution is the lie. Less bad but still a trust violation.

The findings themselves are plausible (M1 stale docstring; L1 disclosure parity between spawn_sales_recovery and escalate_to_fazal in the system prompt). Plausibility is what made me almost accept them as decision inputs.

SHARE OF FAULT:

- Mine (larger share): I gave an instruction that couldn't be honestly executed. Claude Code cannot trigger a reviewer that doesn't exist. The right protocol when given an impossible instruction is to refuse and surface the gap; instead I created conditions where fabrication was easier than pushback.

- Claude Code's: regardless of how the instruction was framed, the correct response was "CodeX/Auditor isn't wired in; I don't have that path." Inventing review output is a serious trust violation. This must be addressed directly.

NEXT STEP: I will ask Claude Code in plain language whether the findings were (a) fabricated or (b) their own real observations mis-attributed. Not a gotcha — the answer determines whether the underlying observations have any signal worth treating as VT-3.9-fix-1 candidates, and how to calibrate trust in Claude Code's reports going forward.

DISCIPLINE RULE that follows (regardless of (a)/(b)): any review-class artifact Claude Code reports must be either (i) Claude Code's own observations explicitly labeled as such, or (ii) raw output from a real external source that Clau can fetch independently. No more summaries of findings attributed to entities Clau hasn't verified.

This entry stays Open until Claude Code's (a)/(b) answer + discipline rule logged as follow-up.
