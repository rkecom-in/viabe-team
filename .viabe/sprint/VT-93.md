## What CoderC must produce
The refund execution pipeline. Triggered by VT-9.5 when owner replies REFUND. Calls Razorpay refund API; cancels subscription; transitions phase to `refunded`; sets up 30-day graceful-exit window during which the tenant retains read access to their data and DSR ability.
1. **Execution module** at `apps/team/src/billing/refund_executor.py`:
	- `execute_refund(tenant_id, refund_reason: Literal['day39_eligibility', 'manual_request']) -> RefundExecution`.
	- Idempotency: same tenant_id + reason within 24h → returns existing execution. Migration `035_refund_executions.sql` with PK `(tenant_id, refund_reason)` + status enum.
2. **Razorpay refund call**:
	- Sums all `subscription.charged` events for the tenant since `paid_conversion_at` → total refund amount.
	- Calls Razorpay's `/refunds` API per payment_id (Razorpay refunds per-payment, so multiple API calls if multiple charges).
	- Each refund response captured in `refund_executions.refund_responses` JSONB.
	- On any individual refund failure: marks the overall execution `partial_failed`; alerts Fazal via Telegram with details; remaining refunds NOT attempted (avoid partial-state confusion). Fazal investigates manually.
3. **Subscription cancellation**:
	- After refunds succeed: calls Razorpay's `/subscriptions/<id>/cancel` to cancel the active subscription with `cancel_at_cycle_end=false` (immediate cancellation).
	- On cancellation failure: alerts Fazal; refund execution stays `pending_subscription_cancel`.
4. **Phase transition**:
	- All refunds + cancellation succeed → transition phase via VT-3.2 `day39_refund_triggered` event → phase=refunded.
	- Sets `tenants.refunded_at` timestamp.
5. **Graceful exit**:
	- 30-day window from `refunded_at` during which:
		- Portal accessible (read-only; no campaigns, no agent dispatch).
		- DSR export endpoint accessible.
		- Owner can re-onboard later by signing up fresh (existing data NOT auto-restored — they get a clean slate).
	- After 30 days: portal access revoked; data retained per DSR retention rules (still accessible via DSR endpoint indefinitely with WhatsApp verification).
6. **Communication**:
	- On execution start: WhatsApp template `refund_processing` (template 21; PR follow-up to VT-13.3) sent: "Your refund of ₹X is being processed. Expect it in your account within 5 business days."
	- On all refunds completed: template `refund_completed` (template 22): "Your refund of ₹X has been processed. Your subscription is cancelled. You can still access your dashboard for 30 days."
	- On partial failure: NO template to owner; Fazal handles manually.
7. **Audit**:
	- Privacy audit log row per refund execution + per status change.
	- `refund_executions` row immutable post-completion (RLS DENY UPDATE/DELETE on completed status).
8. **Tests** at `apps/team/tests/billing/test_refund_executor.py`:
	- Happy path: tenant with 2 charges (₹5K each) → 2 refund API calls → subscription cancelled → phase=refunded → templates sent.
	- Idempotency: same tenant within 24h → second call returns existing execution.
	- Partial failure: 2nd refund call fails → execution=partial_failed; Fazal alert; no further calls.
	- Cancellation failure: refunds succeed but cancel fails → status=pending_subscription_cancel; Fazal alert.
	- 30-day window: synthetic 31-day-old refund → portal access revoked but DSR still works.
	- Cross-tenant: tenant_A refund executor cannot affect tenant_B subscription.
	- Audit row persistence + immutability.
## Architectural rules to enforce in this subtask
- Pillar 7: graceful exit is honest — 30 days for the owner to download their data, no retention pressure. CodeX rejects PRs adding retention manipulation.
- Pillar 8: one execution path. CodeX rejects parallel refund handlers.
- Idempotency on `(tenant_id, refund_reason)` is structural.
- Partial failure halts; doesn't auto-retry. Fazal investigates manually.
- 30-day graceful window is a Type 3 commitment.
- Razorpay refund call is async; webhook returns the final status and updates the execution row.
