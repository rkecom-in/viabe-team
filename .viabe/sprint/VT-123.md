# VT-123 — VT-OpsConsole (MVP, launch-essential)

## Why this parent exists (v2.0)

Fazal needs visibility into what's happening across design-partner tenants during the Reports-Jun15 soft launch + ability to replay any incident step-by-step. v1.0 of this row was scoped at Sprint 8 with a broad capability surface; **v2.0 reframes to Sprint 1 MVP** carrying ONLY the launch-essential subset:

1. **Workspace overview** — live counters + in-flight runs panel
2. **Per-tenant dashboard** — profile + 30-day pipeline_runs timeline + recent campaigns
3. **Run replay** — n8n-style step-by-step trace (the load-bearing view)
4. **[resolve] button** — phone-token unmask (per-resolution audit-logged)

The **full admin control panel** (debug / trace / track / train / override / monitor / empower) is split to **VT-189 (VT-OpsConsole-V2)** as a Sprint 2+ parent.

This v2.0 reframe was made 2026-05-26 per Fazal direction: ship the launch gate (Reports-Jun15) with the MVP subset that's actually launch-blocking; build the full vision after.

Reference design document: 'Viabe Team — Pipeline Observability Architecture' §5 (Notion page 35e387c2-cc5a-81c1-b5d8-e7392ab4ff24, created 2026-05-12).

## What this parent owns (MVP scope)

1. **`apps/team-web/ops/`** Next.js route group with auth-gated access
2. **`/ops` workspace overview** — live counters (in-flight runs, total runs today, escalations today, cost burn today), top tenants by activity, in-flight runs panel
3. **`/ops/tenants/<tenant_id>`** per-tenant dashboard — profile, 30-day pipeline_runs timeline, recent campaigns, privacy access log
4. **`/ops/runs/<run_id>`** n8n-style run replay — vertical waterfall of pipeline_steps, click to expand each, formatted JSON envelopes (input/output), decision_rationale text, tool_calls list, error envelope, cost per step
5. **`[resolve]` button** — phone-token unmask flow: click → confirm dialog → server-side JWT-with-operator-claim resolution via VT-188 → audit log row (privacy_audit_log) → render phone in-place
6. **Server-side data access** via JWT-direct (operator-claim) for hot-read views (per CL-88 client-direct pattern) + service-role for cold-read views (per-tenant dashboard aggregate queries)
7. **Auth** — Fazal-only UUID check for Phase 1 (hardcoded); RBAC deferred to VT-189

## Architectural rules

- **Read-only Phase 1.** No mutations from Ops UI (override / abort / refund deferred to VT-189)
- **Pillar 7 — every resolve is audit-logged.** No silent admin actions. Per-resolution audit row (not aggregate) per Fazal direction 2026-05-26
- **Pillar 3 — tenant isolation preserved.** Cross-tenant views (workspace overview) are read-aggregate only; per-tenant views require explicit tenant selection in URL
- **CL-88 client-direct** — hot reads (live counters, in-flight panel) via JWT-direct; cold reads (per-tenant 30-day) via Next.js API route + service-role
- **Performance** — per-tenant dashboard <2s for ≤100 runs in last 30d; run replay <1s for ≤50 steps
- **No parallel admin surfaces** — Telegram /costs from VT-103 + Telegram alerts (when filed) are CLI complements, not duplicates
