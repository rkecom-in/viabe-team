# CL-360 — TECH DEBT (PR #48 review — merged with known weak test): test_owner_inputs_ship_

TECH DEBT (PR #48 review — merged with known weak test): test_owner_inputs_ship_gate.py::test_runner_does_not_invoke_extraction_when_gate_off is named as a behavioral test but is NOT one. It sets a MagicMock spy on run_extraction_for_event, never drives webhook_pipeline_run, then asserts spy.call_count==0 — trivially true since the spy was never given a chance to fire; the assertion would pass even with no guard at all. The test's real assertion is a source-string check ('if OWNER_INPUTS_EXTRACTION_ENABLED:' in inspect.getsource) — the fragile grep-the-source pattern (soft dead-canary). PR #48 MERGED ANYWAY — deliberate call: the gate's actual safety rests on (a) constant defaults False (test 1 genuinely verifies this via imported-symbol value check) and (b) the guard literally wraps the sole call site (verified by direct read of runner.py). Blocking the merge to fix test hygiene would prolong the gateless state on main from the PR #46/#47 bypass — wrong trade. FOLLOW-UP REQUIRED when owner_inputs work resumes: add a REAL behavioral test to test_owner_inputs_substrate.py — drive webhook_pipeline_run against the live substrate with OWNER_INPUTS_EXTRACTION_ENABLED toggled both ways; assert run_extraction_for_event IS called when True and NOT called when False. webhook_pipeline_run is a @DBOS.workflow needing the live pool, which is why a true behavioral test belongs in the substrate suite, not pure-Python — that is the legitimate reason it wasn't done in #48, but the named-as-behavioral/actually-source-grep test should not be left as the gate's only behavioral-looking coverage.
