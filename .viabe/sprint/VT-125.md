## What CoderC must produce
The Orchestrator-Agent system prompt + tool inventory. This is the Opus 4.7 brain that reasons about coordination + routing for the ~30% of events that pass the Pre-Filter Gate. NOT a specialist — specialists reason about domain. The orchestrator-agent decides per event: spawn specialist / run deterministic pipeline / respond directly / observe only.
**Architectural intent:** the orchestrator-agent is the Claude-Code-equivalent harness for Viabe Team. Its job is the routing-and-coordination problem space, not the campaign-design problem space (that's Sales Recovery's job).
1. **System prompt** at `apps/team-orchestrator/src/orchestrator/prompts/orchestrator_agent_system.md`:
	Structured into sections:
	- **Role:** "You are the Orchestrator-Agent for Viabe Team. You coordinate work across specialists and direct deterministic pipelines. You DO NOT design campaigns, write WhatsApp copy, or make domain decisions — specialists do that."
	- **Decision framework:** for each invocation, decide one of: `spawn_<specialist>` / `respond` / `pipeline` / `observe`.
	- **Tools available:** enumerated tool inventory with descriptions.
	- **Memory access:** read from L0 (workspace patterns) + L1-L4 (tenant-scoped) via Context Composer.
	- **Hard limits:** orchestrator-agent invocations capped at 10K tokens, 5 tool calls, depth 3, 2-minute wall clock, ₹5 cost. (Lower than specialist limits — orchestrator-agent decisions should be fast.)
	- **Escalation criteria:** what triggers escalate_to_owner vs escalate_to_fazal.
	- **Out of scope:** owner-facing message text (use compose_owner_output tool); campaign design (spawn Sales Recovery); domain reasoning of any kind.
2. **Tool inventory** registered with the agent:
	Phase 1 tools (limited — orchestrator-agent has narrow scope by design):
	- `spawn_sales_recovery(context_summary, trigger_reason)` — from langgraph_supervisor (VT-3.4)
	- `compose_owner_output(specialist_result, intent_or_trigger)` — from VT-3.7
	- `send_whatsapp_template(tenant_id, template_id, params)` — from VT-MCP-Tools VT-5.7
	- `get_subscriber_state(tenant_id)` — read-only state query from VT-MCP-Tools VT-5.2
	- `query_pipeline_history(tenant_id, lookback_days=7)` — from VT-MCP-Tools VT-5.3
	- `escalate_to_fazal(run_id, reason, context)` — from VT-3.6 error framework
	- `write_l0_fragment(fragment_type, cohort_key, content, observation_count)` — from VT-7.N (orchestrator-only write)
	- `query_l0(fragment_type, cohort_key)` — from VT-7.N
3. **Tool subset enforcement:**
	- Orchestrator-agent's allowed tools are exactly the list above. CodeX rejects PRs that grant additional tools without Type 2 governance.
	- Specialists' tool inventories are DIFFERENT (specialists have domain-specific tools); orchestrator-agent cannot call specialist tools directly.
	- Pre-Filter Gate handlers do NOT call this agent (deterministic only).
4. **Context Composer integration:**
	- Before each orchestrator-agent invocation, Context Composer (VT-71) builds the bundle:
		- Current SubscriberState
		- L0 fragments matching ([tenant.business](http://tenant.business)_type, [tenant.city](http://tenant.city)_tier, current_phase)
		- L1-L4 relevant subset for the event
		- Last 5 pipeline_steps for the run
	- Bundle bounded: 4K tokens max (within orchestrator-agent's 10K hard limit).
	- Bundle assembled via index + pointers pattern (Claude Code [MEMORY.md](http://MEMORY.md) inspiration) — not raw content dump.
5. **L0 write hook:**
	- After each orchestrator-agent invocation, the agent MAY emit an L0 fragment via `write_l0_fragment` tool call.
	- Fragment types: `routing_decision` (why I chose to spawn vs respond), `specialist_outcome` (how the handoff went), `trigger_pattern` (cross-tenant observation).
	- Every L0 write routes through VT-7.N k-anonymity gate; tenant-identifying fragments rejected.
6. **Tests** in `apps/team-orchestrator/tests/orchestrator/test_orchestrator_agent.py`:
	- System prompt loads correctly; agent can be invoked with synthetic event.
	- Hard limits enforced: 5 tool calls = termination; 10K tokens = termination; 2-min wall-clock = termination; ₹5 cost = termination.
	- Decision routing: synthetic 'weekly_cadence' trigger → spawn_sales_recovery; synthetic 'opt_out' (despite Pre-Filter normally catching) → respond directly; synthetic status query → respond directly.
	- Tool subset enforcement: agent attempting to call out-of-subset tool (e.g., a Sales Recovery domain tool) → structured error.
	- L0 write integration: synthetic decision → fragment written → k-anonymity gate test passes.
	- DBOS auto-resume: crash mid-agent-invocation → resume from last completed tool call.

## Architectural rules to enforce in this subtask
- **Pillar 1 (revised):** orchestrator-agent reasons about coordination + routing ONLY. NO domain reasoning. NO campaign design. NO WhatsApp copy generation (use compose_owner_output tool).
- **Pillar 8 (no patchwork):** ONE system prompt in ONE file. ONE tool inventory. No drift. Tool additions require Type 2 governance.
- Orchestrator-agent hard limits are LOWER than specialist limits (intentionally — orchestrator-agent should be fast).
- Context bundle bounded at 4K tokens (within 10K agent limit). Bundle expansion = Type 2.
- L0 writes pass through k-anonymity gate. CodeX enforces.
- Specialists cannot invoke this agent. Orchestrator-agent is upstream of specialists, never downstream.
