# VT-78 — Data residency config (ap-south-1 primary, ap-south-2 backup)

## Parent task
VT-PrivacyArchitecture.

## Concept reference
- `concept-team.md` §11 (data residency: ap-south-1 Mumbai primary, ap-south-2 Hyderabad backup; DPDPA + Indian SMB market expectation).
- `concept-team-pillars.md` Pillar 7 (data residency is part of consumer-trust posture; deviation requires Type 3 governance).
- `concept-team-pillars.md` Pillar 8 (one residency configuration; ad-hoc region overrides forbidden).

## Dependencies
- VT-Foundation VT-2.2 Done (Supabase projects exist; this subtask configures their region pinning + backup region).
- VT-Foundation VT-2.5 Done (Vercel + Railway configurable per region).

## What CoderC must produce
The data residency configuration: pinning all data-bearing infrastructure to ap-south-1 (Mumbai) primary with ap-south-2 (Hyderabad) as backup. Documentation, verification scripts, and runbook for ops.
1. **Supabase region pinning**:
	- Both `viabe-team-dev` and `viabe-team-prod` Supabase projects MUST be created in `ap-south-1`. Verified via the Supabase project's region setting.
	- For Phase 1, ap-south-2 backup is via Supabase's PITR (Point-in-Time Recovery) snapshots if available in the chosen tier. If not, daily logical backups via `pg_dump` to an ap-south-2-region S3 bucket (or equivalent).
	- Configuration documented at `docs/team/data-residency.md`.
2. **Vercel region pinning**:
	- `viabe-team-web` deployment region: `bom1` (Mumbai). Set in `vercel.json` under `regions: ['bom1']`. Vercel's automatic edge cache may serve from other regions for non-data-bearing static assets; documented as acceptable.
	- API routes (which are data-bearing) execute in `bom1`. Verified via Vercel deployment logs.
3. **Railway region pinning**:
	- `viabe-team-orchestrator` and any other Railway services configured in Railway's `asia-southeast1` if `ap-south-1` not directly supported (Railway's regional offering is constrained; document the next-best choice with rationale at `docs/team/data-residency.md`).
	- If Railway doesn't offer an Indian region: document the Phase 1 acceptable risk + mitigation (encrypted-in-transit; minimal data crossing); flag for Phase 2 review.
4. **External services**:
	- Anthropic API: data-in-flight to AWS regions Anthropic uses; Anthropic's data-handling policy linked in residency doc; explicit caveat that LLM calls leave India for processing.
	- OpenAI / Whisper (if used for VT-6.8 voice): same caveat.
	- Twilio: WhatsApp messages route through Twilio's infrastructure; Twilio India presence documented; routing path noted.
	- Razorpay: India-native; full ap-south-1 stack.
	- Apify: international actors; explicit caveat that scraping crosses regions; only context-only data leaves (no PII per VT-6.10/6.11).
5. **Verification script** at `apps/team/scripts/verify_residency.py`:
	- On startup of the orchestrator process, verifies:
		- Supabase project region == ap-south-1.
		- Vercel-deployed region (via env or runtime check) == bom1.
		- All env vars referencing external endpoints match the documented endpoints.
	- Mismatches log a critical alert to Fazal and refuse to start.
6. **DPDPA disclosure**:
	- The DSR export's `manifest.json` (VT-8.6) includes a residency statement: "Your data is stored in Mumbai (ap-south-1). Backup data is in Hyderabad (ap-south-2). LLM processing for agent reasoning may occur outside India per Anthropic/OpenAI's documented infrastructure."
	- Sign-up flow (VT-9.2) shows a residency disclosure to owners during onboarding.
7. **Tests** at `apps/team/tests/privacy/test_residency.py`:
	- Verification script: synthetic mismatch → critical alert + refuse to start.
	- DPDPA disclosure text matches documented version (regression test on the manifest.json template).
	- Migration script for cross-region data move (in case prod ever needs migration): scaffolded but not run; documented at `docs/runbooks/region-migration.md`.

## Architectural rules to enforce in this subtask
- Pillar 7: residency posture is part of consumer trust. CodeX rejects PRs adding non-Indian regions for data-bearing services without Type 3 governance.
- Verification on startup is structural; CodeX confirms it runs unconditionally before serving traffic.
- LLM API endpoints are documented as exceptions; Phase 1 accepts that the agent's reasoning processes data outside India in real-time. This is disclosed; not hidden.
- Phase 2 considerations: Anthropic India region availability would let us bring LLM calls in-country; revisit if Anthropic offers it.
- Backup region (ap-south-2) is part of the Type 3 commitment.

## Acceptance criteria
- Supabase projects in ap-south-1 confirmed via Supabase dashboard screenshot in PR description.
- Vercel `regions: ['bom1']` set; deployment region verified.
- Railway region documented (best available; ap-south-1 if supported, fallback documented).
- `apps/team/scripts/verify_residency.py` runs on orchestrator startup; mismatch → critical alert + refuse-to-start.
- DPDPA disclosure in DSR manifest + sign-up flow.
- `docs/team/data-residency.md` documents all regions, exceptions, and Phase 2 considerations.
- `docs/runbooks/region-migration.md` exists (scaffolded; Phase 1 doesn't run it).
- mypy --strict passes.
- PR title references `(VT-PrivacyArchitecture)`; reviewed by CoderX; Fazal personally signs off on the residency configuration.

## Out of scope

- Self-hosted Postgres (Phase 2 if compliance demands).
- Region-specific DR drills (Phase 1 documents the runbook; Phase 1.5 runs the drill).
- Geo-fenced API access (Phase 2; restricting API to Indian IP ranges is a separate decision).
- Data sovereignty audit (Phase 2; once Phase 1 ships, an external audit can verify configuration).
