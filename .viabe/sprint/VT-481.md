# VT-481 — Reap orphaned 'running' pipeline_runs + restart-mid-run guard

## Problem
A `pipeline_runs` row opens `status='running'` at webhook-inbound start and closes
(`close_webhook_run`) at the end. If the process DIES mid-run (Railway deploy-restart / OOM /
crash) the close never fires → the row is stranded `'running'` forever. DBOS recovery does NOT
heal it: on restart DBOS recovers only workflows matching the CURRENT `executor_id` + `app_version`,
and a redeploy changes `app_version`, so a prior-deploy run is never re-invoked (never closed).
14 such orphans had accumulated on dev (oldest 37 days; e.g. SM20d29, webhook_received seq0 done,
never closed).

## Fix
`orchestrator/orphan_reaper.py::reap_orphan_runs` — closes every `status='running'` run older than
a 1-hour floor to `status='aborted_hard_limit'` (an existing terminal CHECK member, mig 052 — no
schema change), stamping `terminal_state_metadata.reaped_by='vt481_orphan_reaper'`. Wired into the
FastAPI lifespan startup AFTER `launch_dbos()` (so DBOS's own same-version recovery fires first),
best-effort (never blocks boot — mirrors `warm_pause_cache`).

## Why the 1-hour floor is provably safe
A `running` row is ONLY the webhook-inbound path, hard-bounded: 6-min invoke timeout + 30-min max
run-control hold (`_RUN_CONTROL_MAX_HOLD_S`). The genuinely long-lived states (owner-approval parks,
L3 auto-send holds) sit `status='paused'`, NOT `running`. So any run still `'running'` >1h is, with
certainty, an orphan — the reaper can never race a live in-flight run or a DBOS mid-recovery row
(recovery fires within seconds of launch, same-version only).

## Verification
- Unit/real-DB: `tests/orchestrator/test_orphan_reaper.py` (5 tests) — reaps old `running`; leaves a
  RECENT `running` (live in-flight), a `paused` (long-lived), and terminal runs UNTOUCHED; idempotent.
- Live cleanup: ran the reaper against dev → closed the 14 existing orphans → 0 remaining. The
  deployed startup hook now auto-reaps any future restart-orphan.

## Out of scope
Not changing DBOS recovery itself or the run lifecycle — purely a janitor for rows the lifecycle
leaks on process death. `.next-id` already at 482.
