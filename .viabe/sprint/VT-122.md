# VT-122 — VT-PipelineObservability

## Why this parent exists
Viabe Reports' biggest operational pain was inability to reconstruct what data flowed between agents when something broke. This parent prevents the same failure mode for Team. Every step in every pipeline persists a typed envelope (declared input/output fields) at execution time, keyed by run_id, queryable for 90 days. Together with the Ops UI (VT-OpsConsole), this gives Fazal the n8n-style step-by-step replay he needs to debug production issues and to justify agent decisions to clients.
Reference design document: 'Viabe Team — Pipeline Observability Architecture' (Notion page under [Viabe.ai](http://Viabe.ai) parent, created 2026-05-12).
Locked decisions (Clau_Session_Log CL-18 to CL-21):
- Typed envelopes, not full-row snapshots.
- Only customer phone numbers tokenized; all other fields plaintext for debuggability.
- 90-day full retention, then aggregate to Pipeline_Run_Summary.
- Ops UI (VT-OpsConsole) is Phase 1 launch-blocking; this parent's data model is its foundation.

## What this parent owns
1. The three Postgres tables: `pipeline_runs`, `pipeline_steps`, `phone_token_resolutions` (schema defined in the design document, section 2.1).
2. The enumerated step_kinds (~15 kinds, design doc section 2.2) and per-kind Pydantic envelope schemas at `apps/team/observability/envelopes/<kind>.py`.
3. The `pipeline_observability.write_step(...)` writer with atomic step + run cumulative-state updates.
4. The MCP tool decorator `@observability.tool_step(envelope_in=..., envelope_out=...)`.
5. The Anthropic Agent SDK observability callback (one `agent_reasoning_step` per think/act/observe loop iteration).
6. The LangGraph node hooks for state transitions.
7. The phone-tokenization logic (extends VT-104 PII redactor; uses shared sha256 hashing).
8. The local SQLite fallback buffer for observability writes when prod DB is down.
9. The nightly aggregation job that compacts >90d step records into Pipeline_Run_Summary.
10. CI checks that prevent regression: new LangGraph node without hook = fail; new MCP tool without decorator = fail; envelope schema drift from code = fail at boot.

## Architectural rules
- Pillar 7 (consumer trust): step records persist agent reasoning so we can justify decisions to clients. This is the primary accountability surface.
- Pillar 8 (single source of truth): one canonical step record schema. CodeX rejects parallel observability schemes.
- Pillar 3 (tenant isolation): every step record has tenant_id, RLS-enforced. phone_token_resolutions has stricter RLS than pipeline_steps.
- Failure resilience: observability MUST NOT crash the pipeline. SQLite buffer + recovery.
- Atomicity: each step write updates `pipeline_runs.step_count` and `total_cost_paise` in the same transaction.
- Envelope schema enforcement at write time, not runtime: schema drift fails at boot, not in production.

## Definition of done
- All 9 subtasks Done.
- A full pipeline run on the dev environment writes ~20-30 step records covering all step_kinds at least once.
- Querying `SELECT * FROM pipeline_steps WHERE run_id = '<uuid>' ORDER BY step_seq` returns the complete chronological trace.
- Phone numbers in step records appear as `cust_tok_<hash>` and resolve correctly via the resolution table.
- The local SQLite buffer kicks in correctly when DB is unreachable (test by blocking the connection).
- CI fails when an engineer adds a new MCP tool without the decorator (verified by intentional negative test).
- Aggregation job dry-run on synthetic >90d data produces correct Pipeline_Run_Summary rows.

## 2026-05-16 Updates (audit-driven respec)
**LangSmith → Pydantic Logfire.** Observability tracing stack changed per stack audit decision. DBOS natively emits OpenTelemetry spans → Logfire is the destination. Pydantic AI agents emit spans natively. Flat span-based pricing (~$2/million spans) replaces LangSmith's per-trace pricing that collapses under agentic load. Subtask bodies that reference LangSmith must be updated to Logfire. See Clau_Session_Log CL-Logfire-decision.
**Schema deviation already shipped in VT-Foundation PR #3:** `pipeline_steps.tenant_id` column was added directly (not just reachable via run_id join). Pillar 3 requires direct tenant filter for RLS to work cleanly. VT-122.1 spec did not originally include this column. The implemented schema (in /migrations/006_pipeline_steps.sql on main) IS the source of truth; this parent's spec language is the drift, not the migration. Implementation team should read the migration, not this body, for column-level truth.
**Anthropic Agent SDK observability callback** (subtask reference in section above) needs updating: 2026 Anthropic SDK has new instrumentation surfaces (prompt caching, defer_loading on tools) that should be captured in step records. Specific subtask brief will spec which fields per agent_reasoning_step.
