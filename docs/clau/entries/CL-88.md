# CL-88 — Correction to CL-79: dual RLS mechanism (GUC for backend, JWT for client direct reads)

CORRECTION to CL-79 (DECISION: RLS canonical mechanism). The prior entry's framing was too narrow — 'GUC is canonical, no JWT anywhere' overstated. The accurate answer is: dual mechanism, scoped to runtime.

FINAL DECISION (Fazal 2026-05-18 after pushback):
- **Backend runtime** (orchestrator + ingestion worker, direct Postgres DSN, no Supabase Auth): GUC pattern. Connection wrapper sets `app.current_tenant_id` GUC on checkout; RLS policies read via `app_current_tenant()`.
- **Client runtime** (team-web frontend reading own data via Supabase Auth + publishable key): JWT pattern. RLS policies read tenant via `auth.uid()` joined to the tenant mapping (mechanism TBD — see open schema decision below).
- **Helper function pattern** when JWT path activates: `COALESCE(current_setting('app.current_tenant_id', true)::uuid, <jwt-derived-tenant>)`. Standard Supabase + Postgres RLS pattern.

RATIONALE: Path A (route every dashboard read through orchestrator) was rejected because (i) it's a substantial unplanned scope expansion (8-12 dashboard queries in VT-123 alone, each needs a Python endpoint), (ii) re-implements auth verification that Supabase already provides battle-tested, (iii) the dual-helper pattern is standard not novel.

OPEN SUB-DECISION (Fazal to resolve before PR-fix-7 design solidifies): does `tenants.id` map 1:1 to a Supabase Auth user, or is there a `user_tenants` mapping table?
- **B1 (1:1)**: add `auth_user_id UUID UNIQUE REFERENCES auth.users(id)` to tenants. Cheap. Constrains future multi-user-per-tenant.
- **B2 (mapping table)**: create `user_tenants(user_id, tenant_id, role)`. More flexible, ~15 lines, fits Pillar 8 (no auth-structure retrofits later).
- Clau recommends B2. Push back if reason for B1.

PR-fix-7 SCOPE NARROWS: implement GUC pattern only (the helper `app_current_tenant()` already does this correctly). DO NOT add COALESCE or JWT variants to policies until the first dashboard read activates them — invariant clutter otherwise. The JWT side gets wired when VT-OpsConsole (or first tenant-facing dashboard subtask) lands, with the user_tenants/auth_user_id decision driving the policy shape.

SUPERSEDES: CL-79 (single-mechanism framing).
