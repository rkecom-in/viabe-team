# VT-246 — remove redundant Railway CLI deploy job

## Why
The orchestrator Railway service already auto-deploys natively from GitHub
(source=rkecom-in/viabe-team, branch=main, "Auto deploys on push" ON,
"Wait for CI" ON). The `railway up` CLI job in deploy-dev.yml was redundant
— and worse, its failure (Invalid RAILWAY_TOKEN; wrong token type, railway
cli#845) **blocked the native deploy**, because "Wait for CI" gates the
native deploy on every Action passing. Removing it fixes the outcome
without needing a project-scoped token.

## Change
- Deleted the `deploy-orchestrator` (`railway up`) job.
- Deleted `post-deploy-smoke`: it `needs:` the removed job and curled
  RAILWAY_DEV_URL/health — but the native deploy fires only AFTER this
  workflow is green (Wait-for-CI), so a CI-time orchestrator smoke can't
  see the fresh deploy and would re-block the gate. Railway deploy is now
  verified Railway-side. `scripts/dev-env-smoke.sh` kept in-repo (manual).
- Kept the Vercel (team-web) job + pre-deploy-checks unchanged.

deploy-dev is now: pre-deploy-checks → deploy-team-web (Vercel). Green →
Railway "Wait for CI" unblocks → native orchestrator deploy fires.

## Now-unused GitHub secrets (Fazal may delete)
RAILWAY_TOKEN, RAILWAY_SERVICE_ID, RAILWAY_DEV_URL.

## Acceptance
- actionlint exit 0; deploy-dev green (Vercel) on the merge push.
- Railway native deploy fires post-CI (dashboard-verified by Fazal).
