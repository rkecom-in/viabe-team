# VT-35 — Hard-limit enforcement at orchestrator/agent boundary

## Parent task
VT-SalesRecovery-Agent.

## Concept reference
- `concept-team.md` §8.3 (hard limits per agent invocation: 80K tokens, 25 tool calls, depth 8, 5-minute wall clock; exceed any → terminate with escalate_to_owner).
- `concept-team-pillars.md` Pillar 1 (orchestrator measures; agent does not police itself).
- `concept-team-pillars.md` Pillar 8 (no patchwork; structural enforcement, not best-effort).

- VT-Orchestrator VT-3.4 Done (dispatch node has structural hooks for hard limits).
- VT-Orchestrator VT-3.6 Done (error framework can route hard-limit breach as a known failure type).

## What CoderC must produce
The four hard-limit enforcers, wired into the dispatch boundary. Each runs continuously during agent execution. Any one breach → hard-cancel the in-flight Anthropic SDK call → set `AgentResult.status='terminated'`, `terminated_by=<which limit>` → emit `agent_hard_limit_breach` failure to error router → router escalates to owner.
1. **Token meter** in `apps/team/src/agent/limits/token_meter.py`:
	- Hooks into Anthropic SDK's per-message usage callback. Sums input + output tokens across all turns within the run.
	- Maintains running total in a per-run state object (keyed by `run_id`).
	- On each callback: if `total_tokens > 80_000`, signal cancel.
	- The meter is reset on every new dispatch (per-invocation, not cumulative across invocations of the same tenant).
2. **Tool-call counter** in `apps/team/src/agent/limits/tool_counter.py`:
	- Hooks into the Agent SDK's tool-dispatch event. Increments on each tool invocation.
	- On increment: if `tool_calls > 25`, signal cancel.
	- Counts both successful and failed tool calls; a tool that errored still consumed budget.
3. **Reasoning-depth tracker** in `apps/team/src/agent/limits/depth_tracker.py`:
	- Tracks the chain of (think → tool → think → tool) cycles. Each tool dispatch followed by another reasoning step increments depth.
	- On increment: if `depth > 8`, signal cancel.
	- This catches runaway loops where the agent calls tools without converging.
4. **Wall-clock timer** in `apps/team/src/agent/limits/wallclock_timer.py`:
	- Asyncio task started at `run_sales_recovery_agent` entry. Sleeps 5 minutes; if not cancelled by completion, signals cancel.
	- Cancellation propagates to the Anthropic SDK call via the SDK's cancellation token (or `asyncio.CancelledError` wrapping if the SDK doesn't expose one cleanly).
5. **Cancel coordinator** in `apps/team/src/agent/limits/coordinator.py`:
	- Centralizes the four cancel signals into a single `CancellationContext`. First signal wins.
	- On cancel: stops the Anthropic SDK call cleanly (no orphan API requests), populates `AgentResult.status='terminated'` and `terminated_by`, emits the failure record.
6. **Wiring**: the four limit modules are instantiated in `run_sales_recovery_agent` (VT-4.1) and wired to the SDK via the appropriate callbacks. The dispatch node (VT-3.4) receives the populated `AgentResult` and the error framework (VT-3.6) routes the breach.
7. **Tests** in `apps/team/tests/agent/test_hard_limits.py`:
	- Token-cap test: synthetic agent run exceeds 80K — terminates with `terminated_by='tokens'`, no further turns processed.
	- Tool-call-cap test: synthetic agent invokes 26 tools — terminates with `terminated_by='tool_calls'` on the 26th.
	- Depth-cap test: synthetic agent runs a 9-deep reasoning chain — terminates with `terminated_by='depth'`.
	- Wall-clock test: synthetic agent runs longer than 5 minutes (mocked time) — terminates with `terminated_by='wallclock'`.
	- Cancel coordinator: simultaneous breach (e.g., tokens + wall-clock at the same instant) — first signal wins, second is no-op.
	- Clean termination: no orphan Anthropic API calls after cancel (verified by tracing in test).
	- Resource cleanup: the wallclock asyncio task is cancelled on early completion (no zombies).

## Architectural rules to enforce in this subtask
- Pillar 1: the orchestrator (this enforcement layer) measures and decides termination. The agent does NOT have access to the limit state; it cannot ask "how many tokens have I used?" The agent acts; the orchestrator measures.
- Pillar 8: the four limits are enforced unconditionally. No "warn at 80%" thresholds, no "please finish your current step" graceful-stop. Hard cancel.
- Hard limits are NOT a config concern. The 80K / 25 / 8 / 5min values are Type 3 commitments ([concept-team.md](http://concept-team.md) §8.3); changing them requires board governance.
- Termination is unilateral. The agent does not get a chance to argue.
- Resource cleanup is mandatory. Asyncio tasks are cancelled cleanly. No zombie HTTP requests, no leaked timers.
- Cost continues to accumulate during a terminated run (Anthropic charges for tokens already produced). The cost is logged accurately to `pipeline_log` even on termination.
- Per-invocation reset: the next dispatch starts fresh budgets. Budgets do NOT carry across dispatches even within the same `run_id` (a `run_id` is one dispatch).

## Acceptance criteria
- All four limit enforcers exist as separate modules with clean tests.
- Cancel coordinator centralizes signals; first signal wins.
- All four limits independently tested with synthetic breaches.
- Simultaneous-breach test passes (no double-firing, no race condition).
- Clean cancellation: zero orphan Anthropic API calls after termination.
- Resource cleanup verified: no zombie asyncio tasks.
- Cost attribution accurate even on termination.
- A deliberately-changed limit constant (e.g., 80K → 100K) fails CI — confirms the values are version-pinned (Type 3).
- PR title references `(VT-SalesRecovery-Agent)`; reviewed by CoderX (mandatory — hard limits are reliability-critical).

## Out of scope
- The instrumentation that produces the cost numbers (VT-12.5) — this subtask consumes it.
- The owner-facing escalation message (VT-OwnerSurface; this subtask just emits the failure event).
- Per-tenant or per-trigger limit overrides — NOT a Phase 1 concern. All tenants get the same envelope.
- Soft-warning thresholds — explicitly forbidden by the architecture.
- Cross-dispatch cumulative limits ("this tenant has burned 1M tokens this month") — cost-monitoring concern, lives in VT-12.3.
