# VT-206 — Integration Agent onboarding conversation flow

## Why this row exists

The Integration Agent IS a specialist agent — same architecture as orchestrator-agent and (future) sales_recovery. Lives in the supervisor graph. Walks owner through phases 1-4 of the use-case diagram. This row is the agent itself (prompt, tools, decision tree, conversation state).

## Scope

### Specialist registration

- `apps/team-orchestrator/src/orchestrator/agent/integration_agent.py` — new specialist
- Registers `spawn_integration` handoff tool in supervisor (per VT-27 pattern)
- Uses same Opus 4.7 + cache_control (VT-194 substrate)
- Tools available: `list_connectors`, `start_connector_setup`, `pull_sample`, `propose_field_mapping`, `confirm_field_mapping`, `setup_recurring_ingestion`, `dedupe_against_existing`, `escalate_to_fazal`

### Conversation state schema

- `tenant_integration_state` table: tracks per-tenant onboarding progress (phase 1/2/3/4, current connector, pending owner-input, last decision)
- State transitions logged as pipeline_steps per VT-180

### System prompt

`apps/team-orchestrator/src/orchestrator/prompts/integration_agent_system.md`:
- Identity: "You're the onboarding specialist. Your only job is getting business data INTO Viabe."
- Behavior: ask non-technical questions; never assume the owner knows API/OAuth/JSON; offer screenshots/walkthrough links
- Hard rules: never ask for raw passwords; always use OAuth where available; if owner stuck, escalate to Fazal not abandon

### Onboarding triggers

Owner can start onboarding by:
- Opening `/team/onboard` in Ops Console (web flow)
- Messaging "add integration" / "connect my data" / "I want to use Shopify" via WhatsApp → pre_filter routes to brain → orchestrator spawns integration specialist

Both paths converge to the same agent.

## Acceptance criteria

1. Owner starts onboarding from web OR WhatsApp; same agent flow runs
2. Agent reads connector registry (VT-205); does NOT hard-code tool names in prompt
3. Phase transitions persisted in `tenant_integration_state`; resumable if owner disconnects mid-flow
4. Agent escalates to Fazal when stuck (e.g., owner can't find a Shopify admin token) — does NOT loop or fabricate
5. End-to-end RKeCom canary: synthetic owner answers walk → Sheet OAuth (mocked or stubbed in canary) → sample pull → field-map confirmation → tenant_config row created

## Rule #15 Canary

`apps/team-orchestrator/canaries/vt206_integration_onboarding.py` — 6 assertions covering: agent spawned, phase transitions logged, tenant_integration_state populated, escalation path works, dedupe across resume, end-state tenant_config valid.
