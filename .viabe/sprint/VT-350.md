# VT-350 — pin `algorithms: ['HS256']` in the JWT verify helpers (defense-in-depth)

Flagged by the VT-326 #368 auth subagent (Cowork 20260606T065000Z). PRE-EXISTING (NOT
introduced by VT-326): all four `jwtVerify` call sites omit `algorithms: ['HS256']` —
`verified-number-token.ts`, `owner-jwt.ts`, `operator-jwt.ts`, `verify-trial-end-token.ts`.

Not currently exploitable (jose rejects `alg=none` by default + an RS-vs-HS confusion needs a
public key the attacker doesn't control for a symmetric secret; the audience guard separates
token types regardless) — but pinning the accepted algorithm explicitly is standard JWT
hardening + removes a future-footgun.

**Fix:** add `algorithms: ['HS256']` to each `jwtVerify(token, secret, { audience, ... })`
call. One-line per helper + a test that a non-HS256 token is rejected.

## SOFT GATE
This MUST land before `ENABLE_PUBLIC_SIGNUP` is flipped to `true` (public signup go-live).
Behind the queue; roster only — do NOT build now.
