## Contract (Cowork OPT1, Fazal 2026-06-26 "fix it properly, then run"). Surfaced by the e2e seed-verify.

Onboarding `pull_and_ingest_shopify` pulled `/customers.json`+`/checkouts.json` (abandoned checkouts) as the SR
sales-history. Wrong substrate: SR recovers lapsed BUYERS → must read real ORDERS. ALSO: the order mapper read
`created_at`, which Shopify forces to NOW on API-created orders (a backdated seed order can't register) — `processed_at`
is the real transaction date AND the field that sticks. Fix:
- Onboarding ingests `pull_orders` (`/orders.json`) via `shopify_order_to_canonical` (the existing order path/seam).
- The order mapper reads `processed_at` (fallback `created_at`) — more correct for real merchants + backdatable for the seed.
- `pull_sample` + `shopify_sample_row_to_canonical` (checkouts) RETAINED for a future abandoned-cart-recovery agent — not deleted.
- Phase-A safety intact: counts-only PII, ingest gated behind connected, names-only to LLM, RLS-scoped.

## Verify
ruff clean; 31 pure tests (incl 2 new processed_at tests); 14 onboarding tests incl the resume canary (orders→2
customers+2 sales) + PII canary (counts-only) GREEN on a real DB.
