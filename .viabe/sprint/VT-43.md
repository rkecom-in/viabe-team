## What CoderC must produce
The `get_attribution_data` MCP tool. Returns the structured attribution snapshot for a campaign window: how many recipients transacted, total ARRR contribution, attribution method breakdown.
1. **Tool implementation** at `apps/team/src/agent/tools/get_attribution_data.py`:
	Input schema:
	- `campaign_id: UUID | None` — if provided, returns attribution for this campaign
	- `window_start: datetime | None`, `window_end: datetime | None` — if provided (and campaign_id is None), returns aggregate attribution for all campaigns whose `attribution_close_at` falls in window
	- exactly one of `campaign_id` or window range must be provided; framework input validation enforces.
	Output schema (campaign_id mode):
	- `campaign_id: UUID`
	- `attribution_status: Literal['pending', 'closed']`
	- `attribution_close_at: datetime`
	- `cohort_size: int`
	- `transacting_count: int` — customers in cohort who transacted within attribution window
	- `attribution_rate: float` — transacting_count / cohort_size
	- `arrr_paise: int` — sum of attributed transaction amounts
	- `attribution_breakdown: dict[Literal['exact_match', 'window_match', 'manual_owner'], AttributionBreakdownEntry]` — per method: count, arrr_paise contribution, average confidence
	- `notes: list[str]` — structured caveats: e.g. `"3 transactions ambiguously matched between this campaign and a parallel one; attributed to first-sent campaign"`. Generated deterministically from `attributions` table flags, NOT LLM.
	Output schema (window mode): aggregates the same fields across multiple campaigns:
	- `window_start`, `window_end`
	- `campaign_count: int`
	- `total_cohort_size: int`
	- `total_transacting_count: int`
	- `aggregate_attribution_rate: float`
	- `total_arrr_paise: int`
	- `aggregate_attribution_breakdown` (same structure)
	- `per_campaign_summary: list[CampaignAttributionSummary]` (one-line summaries per campaign in window)
2. **Implementation**:
	- SQL aggregation over `attributions` table joined to `campaigns` and `transactions`.
	- Confidence averaging: weighted by attribution paise (more weight to bigger attributions).
	- `notes` are deterministic: generated by inspecting `attributions.attribution_method` and `attributions.attribution_confidence` fields and emitting structured caveats from a fixed mapping.
3. **Honesty requirements** (Pillar 7):
	- When `attribution_status == 'pending'`, ALL fields tagged with `confidence: low`. Even cohort_size is potentially incomplete (transactions still arriving).
	- `attribution_breakdown.manual_owner` entries are flagged separately because they're owner-asserted, not system-detected.
	- Average confidence below 0.7 triggers a `notes` entry: `"average attribution confidence below 0.7; treat ARRR as indicative range, not point estimate"`.
4. **Tenant scoping**: ctx.tenant_id; cross-tenant attack tested.
5. **Tests**:
	- Happy path: synthetic campaign with seeded attributions → correct ARRR + rate + breakdown.
	- Window mode: 5 campaigns in window → correct aggregate.
	- Pending vs closed: correct flags.
	- Confidence-low note: synthetic attributions with avg confidence 0.65 → note appears.
	- Manual_owner separation: synthetic owner-asserted attributions tagged distinctly.
	- Ambiguous-match note: synthetic case where two campaigns overlap in time and a transaction matched both → note describes the disambiguation.
	- Cross-tenant attack: tenant_A cannot see tenant_B attribution data.

## Architectural rules to enforce in this subtask
- Pillar 2: pure SQL + aggregation.
- Pillar 4: returns data + deterministic notes; no judgment fields.
- Pillar 7: pending status + low-confidence cases generate honest caveats. CodeX confirms `notes` generation is exhaustive (every case in the mapping is covered).
- The day-39 evaluator (VT-Billing VT-10.4) reads from this tool. The data must be reproducible: same inputs always produce same output. CI test re-runs identical query and asserts byte-identical result.
- The 7-day window from `sent_at` to `attribution_close_at` is fixed in [concept-team.md](http://concept-team.md); this tool does NOT compute the window. It reads what the orchestrator's attribution-close trigger (VT-3.5) wrote.

- All output fields populated correctly for synthetic test data.

- `notes` mapping covers all flagged cases (test enumerates them).
- Cross-tenant attack passes.

## Out of scope
- The attribution-close logic itself (VT-3.5).
- The transaction-matching logic (VT-5.8 `match_transactions`).
- Predictive attribution ("this customer is likely to attribute in the next 3 days") — Phase 2 if useful.
- Multi-touch attribution (campaigns A + B both touched a customer; how to split credit) — Phase 1 ships first-touch attribution; multi-touch is Phase 2 design discussion.
- Channel attribution (WhatsApp vs UPI organic vs walk-in) — Phase 1 attributes only WhatsApp-prompted transactions; channel breakdown is Phase 2.
