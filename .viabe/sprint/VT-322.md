# VT-322 — reaper ordering fix (drop/revoke before DROP ROLE)

Rostered from the VT-313 pre-push cycle (Cowork 20260604T094500Z). `scripts/reap_test_dbs.sh` (VT-310) bit
AGAIN: leftover throwaway test DBs hold GRANTs on the 3 shared roles (app_role / app_operator_role /
rls_tester), so the reaper's `DROP ROLE` silently fails (role still referenced) → stale grants persist → the
RLS migration/isolation tests (k_anonymity / checkpoint-RLS / twilio-tenant-number) false-fail "permission
denied" on the next pre-push. Had to force-clean by hand (drop every test DB first, then the roles).

## Fix
The reaper must DROP (or REVOKE the grants of) the test DBs BEFORE / together with the shared roles — i.e. the
role-drop can only succeed once nothing depends on the roles. Mirror the pre-push hook's own sequence
(`scripts/git-hooks/pre-push` lines ~133-137: DROP DATABASE WITH FORCE, then DROP ROLE, then recreate). Make
`reap_test_dbs.sh` order-correct + idempotent so a stale-grant state self-heals.

## Acceptance
- Run the reaper against a cluster with a leftover test DB holding role grants → it drops the DB(s) first, then
  the 3 shared roles succeed (0 remaining) → a subsequent RLS test run passes (no "permission denied").

Low priority (a hand force-clean works), but it keeps re-costing pre-push cycles. bash-3.2-portable (VT-310).
