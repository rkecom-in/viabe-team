# VT-201 — Operations live stream + history dashboard

## Why this row exists

Fazal directive 2026-05-27 21:43 IST: "I need a clear running live stream giving me clear idea of what the Orchestrator is doing, what a particular agent is doing — with the option to filter and view the stream of a particular tenant, particular agent, etc. Just like live stream, I should be able to see past streams from the same or earlier day. At least last 30 days stream records should be accessible to view. Options that can make me well aware of whats happening, so if anything goes wrong I know what it is."

The data substrate exists today — pipeline_runs + pipeline_steps + privacy_audit_log + cost views all capture every operation. **The UX doesn't.** Current Ops Console (VT-123) is run-replay-oriented, not stream-oriented; no real-time updates, limited filtering, no time-range navigation.

Without this surface, operators learn about agent misbehavior post-hoc by reading database rows. Operationally unacceptable for production traffic.

## Scope

### Page 1 — `/ops/stream` (live, real-time)

Real-time feed of pipeline_steps as they land. Each row shows:
- Timestamp (HH:MM:SS)
- Tenant (business_name from join)
- Run ID (clickable → run detail)
- Step kind (color-coded: webhook_received / agent_invocation / agent_reasoning_step / tool_call / state_transition / compose_output / aborted_hard_limit / escalated / error_envelope)
- Agent role (orchestrator / sales_recovery / specialist-name)
- Step summary (extracted from envelope_payload — for agent_reasoning_step: first ~80 chars of reasoning; for tool_call: tool_name + args; for compose_output: response preview)
- Cost (paise, if applicable)
- Wall-clock for the step
- Status indicator (success/in-progress/failed)

**Real-time mechanism:** Supabase Realtime subscription on pipeline_steps INSERT events. Filter applied server-side.

**Filters (sidebar):**
- Tenant (multi-select dropdown)
- Agent role (multi-select)
- Step kind (multi-select)
- Status (success / failed / aborted / escalated)
- Time window (live / last 1h / last 24h / last 7d / last 30d / custom range)
- Free-text search across envelope_payload (PostgreSQL `to_tsvector` on a generated column; needs migration)

### Page 2 — `/ops/stream/history` (past streams, replay)

Same UI as live stream but with historical data + scrubber:
- Date picker (default: today)
- Hour scrubber (jump to specific hour of selected day)
- "Play" button to replay events at real-time speed OR 10x / 100x speed
- 30-day retention per CL-416 — full data available
- Pagination via cursor (run_id-based, not offset-based; handles high-volume)

### Page 3 — `/ops/stream/run/[runId]` (single-run timeline)

Already partially exists at `/ops/runs/[runId]` (VT-123). Enhance:
- Vertical timeline of all steps for one run, ordered by step_seq
- Expandable rows show full envelope_payload + reasoning text
- Token/cost/latency visible per step
- "Jump to next run" / "previous run" for same tenant

### Operator-aware-ness affordances

- **Sticky banner** at top: "Last 24h: N escalations, M hard-limits, K errors" — surfaces problems without operator hunting
- **Quick-filter pills:** one-click filters for "Show only failures" / "Show only escalations" / "Show only this tenant"
- **Per-run latency / cost summary** visible in stream view without clicking through

### NOT in this row (separate VT-202 for proactive alerts)

- Telegram / email alerts on threshold breaches
- Anomaly detection
- Slack-style digest summaries
- Mobile push

This row is PULL-based (operator visits the dashboard). VT-202 will be PUSH-based (system notifies operator). Both are needed; splitting keeps scope manageable.

## Acceptance criteria

1. `/ops/stream` shows live pipeline_steps INSERT events with <2s lag from DB write to visible UI
2. All 6 filter dimensions work (tenant / agent / step_kind / status / time_window / free-text search)
3. `/ops/stream/history` allows date selection within last 30 days; loads <3s for any day
4. Replay button works at 1x / 10x / 100x speed
5. Single-run view shows complete step timeline with expandable envelope details
6. Sticky banner accurately counts last-24h escalations / hard-limits / errors
7. Quick-filter pills toggle correctly
8. RLS isolation: operator (Fazal) sees all tenants; future per-tenant operators (Phase 2) would see only their own (DEFERRED — Phase 1 has only Fazal as operator)
9. Per-run latency + cost summary correct (verified against pipeline_runs aggregates)
10. Free-text search returns relevant matches within 1s for 30-day window

## Rule #15 Canary (mandatory)

`apps/team-orchestrator/canaries/vt201_ops_stream.py` — provisional 6 assertions:
1. Synthetic pipeline_step INSERT triggers Supabase Realtime event within 2s
