## What CoderC must produce
The sign-up form on [viabe.ai/team](http://viabe.ai/team). 4 fields + 2 consent checkboxes + submit. Inline validation; submission posts to VT-9.2 backend; success redirects to "check WhatsApp" page; error states inline.
1. **Form layout** at `apps/team-web/app/team/signup/page.tsx`:
	- 4 fields:
		- Business name (text, 1-100 chars).
		- Owner name (text, 1-100 chars).
		- Owner WhatsApp number (tel, with country code dropdown defaulting to +91; only +91 accepted Phase 1).
		- Business type (select dropdown with autocomplete; common Indian SMB types as suggestions: cafe, restaurant, qsr, kirana, salon, gym, retail, electronics_shop, pharmacy, etc.).
	- 2 consent checkboxes (BOTH MANDATORY):
		- DPDPA consent: "I agree to the Privacy Policy and consent to processing per DPDPA 2023" with link to full disclosure.
		- Residency consent: "I understand my data is stored in India (Mumbai); LLM processing may occur outside India per our infrastructure disclosure" with link.
	- Optional referral_source (hidden): pre-filled from UTM params.
	- Optional language preference: pre-filled from active locale (EN or HI).
	- Submit button: disabled until all 4 fields valid + both consent boxes ticked.
2. **Inline validation**:
	- Real-time on blur: format checks (phone E.164, lengths, profanity check via client-side blocklist matching VT-9.2's server-side list).
	- Server-side validation on submit: full validation per VT-9.2 with error responses mapped to inline error messages.
	- Bilingual error messages (EN + HI).
3. **Submission flow**:
	- POST to `/api/signup` (VT-9.2 backend).
	- Loading state during request (spinner; submit disabled).
	- Success → redirect to `/team/welcome` page with instructions: "Check your WhatsApp for the welcome message from Viabe Team. Click the link in the message to access your dashboard."
	- Error responses → inline messages mapped from VT-9.2's structured errors.
4. **Cross-tenant + privacy**:
	- No tenant_id in form (server creates).
	- No PII in client-side analytics events (only "signup_submitted", "signup_succeeded" — no field values).
	- Form values cleared from session/state after successful submission.
5. **Accessibility**:
	- All fields labeled; ARIA attributes for error states; keyboard-only navigation tested.
	- Bilingual labels (EN + HI).
6. **Tests** at `apps/team-web/__tests__/team-signup/*.test.tsx`:
	- Happy path: 4 fields filled + both consents ticked → submit enabled → POST to /api/signup → success redirect.
	- Validation: bad phone format → inline error.
	- Missing consent: form remains disabled.
	- Server error response: error message displayed inline.
	- Hindi locale: all labels + errors in Hindi.
	- Accessibility: axe-core passes.
	- Cross-locale: form fields work identically in EN and HI.
## Architectural rules to enforce in this subtask
- Pillar 7: consent checkboxes are mandatory + visible. CodeX rejects PRs hiding them in collapsed sections or pre-checking them.
- Pillar 8: one form, one submission endpoint. Variations per source (e.g., "founding-tier-only signup") forbidden — founding tier is server-decided based on counter, not URL.
- Real-time validation client-side as UX assist; server-side validation is authoritative. Both must agree.
- No PII in client-side analytics.
- WCAG 2.1 AA.
- Bilingual labels required.
