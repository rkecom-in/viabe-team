# VT-20 — 32 RLS policies + 7 attack tests proving cross-tenant isolation

## Parent task
VT-Foundation.

## Concept reference
- `concept-team-pillars.md` Pillar 3 (tenant isolation is structural; three independent enforcement layers required — RLS is the first).
- `concept-team.md` §10 (the rule: every row carries tenant_id; cross-tenant access impossible by construction).
- Reports' RLS patterns are NOT a reference — Reports does not implement multi-tenancy; Team's RLS posture is built fresh.

## Dependencies
- VT-2.3 Done (8 tables exist with `tenant_id NOT NULL` on multi-tenant ones).
- VT-2.2 Done (Supabase projects exist with `auth.uid()` available).

## What CoderC must produce
A migration file (`009_rls_policies.sql`) that enables RLS on every multi-tenant table and adds explicit policies. Plus a separate test directory containing 7 attack tests proving cross-tenant isolation holds.
**RLS pattern for multi-tenant tables** (`customers`, `transactions`, `campaigns`, `campaign_messages`, `attributions`, `pipeline_log`):
```sql
ALTER TABLE <table> ENABLE ROW LEVEL SECURITY;

CREATE POLICY <table>_select_own ON <table> FOR SELECT
  USING (tenant_id = auth.jwt()->>'tenant_id'::uuid);

CREATE POLICY <table>_insert_own ON <table> FOR INSERT
  WITH CHECK (tenant_id = auth.jwt()->>'tenant_id'::uuid);

CREATE POLICY <table>_update_own ON <table> FOR UPDATE
  USING (tenant_id = auth.jwt()->>'tenant_id'::uuid)
  WITH CHECK (tenant_id = auth.jwt()->>'tenant_id'::uuid);

CREATE POLICY <table>_delete_own ON <table> FOR DELETE
  USING (tenant_id = auth.jwt()->>'tenant_id'::uuid);
```
That's 4 policies per multi-tenant table × 6 multi-tenant tables = 24. Plus 8 policies for the two non-multi-tenant tables (`tenants`, `signup_holding`):
- `tenants`: SELECT/UPDATE on own row only (`id = auth.jwt()->>'tenant_id'`); INSERT/DELETE blocked from client; service-role-only (server-side via typed wrappers).
- `signup_holding`: INSERT allowed unauthenticated (landing page); SELECT/UPDATE/DELETE service-role-only.
4 + 4 = 8. Total: **32 policies**.
**Append-only enforcement on `pipeline_log`**: drop the UPDATE and DELETE policies entirely (or replace with `USING (false)`). Service-role bypass is allowed; tenant-scoped clients cannot mutate log rows.
**The 7 attack tests** in `apps/team/tests/security/test_rls_attacks.py`:
1. **Cross-tenant SELECT**: tenant A's JWT cannot SELECT tenant B's customers.
2. **Cross-tenant INSERT**: tenant A's JWT cannot INSERT a customer row with `tenant_id = tenant_B`.
3. **Cross-tenant UPDATE**: tenant A cannot UPDATE tenant B's transaction (e.g., overwrite amount).
4. **Cross-tenant DELETE**: tenant A cannot DELETE tenant B's campaign.
5. **Pipeline_log append-only**: tenant A cannot UPDATE or DELETE its own pipeline_log row.
6. **JWT tampering**: a JWT with a forged tenant_id (signed by a wrong key) is rejected by Supabase auth, not silently accepted.
7. **Service-role boundary**: a query with `SUPABASE_PUBLISHABLE_KEY` (frontend key) cannot perform actions that require `SUPABASE_SECRET_KEY` (server key) — e.g., insert into `tenants` directly.
Each test runs against `viabe-team-dev` with two synthetic tenants seeded for the test, and asserts the expected denial. Tests run in CI on every PR (added in VT-2.7).

## Architectural rules to enforce in this subtask
- Pillar 3 (structural isolation): RLS is enforced at the database level, not at the application level. Bypassing the application layer (e.g., direct Postgres connection with publishable key) must still respect RLS.
- Pillar 8 (no patchwork): RLS policies live in a single migration file (`009_rls_policies.sql`), not scattered across multiple. Future RLS changes for new tables go into NEW migrations (010+), never edit migration 009.
- Every multi-tenant table has all 4 CRUD policies. Missing a policy means default-deny, which is correct, but explicit policies are clearer for review.
- `auth.jwt()->>'tenant_id'` is the canonical tenant-resolution method. The orchestrator (VT-3) is responsible for stamping `tenant_id` into the JWT before invoking the agent. Tools (VT-5) inherit this scope via the JWT — they do NOT accept `tenant_id` as a parameter.
- Service-role access (using `SUPABASE_SECRET_KEY`) bypasses RLS by Supabase design. This is correct: typed wrappers (VT-PrivacyArchitecture) re-introduce tenant scoping at the application level. RLS is layer 1; typed wrappers are layer 2; agent context isolation is layer 3.
- Attack tests must use realistic JWTs (signed via Supabase test helpers), not mocked auth. The test runs against the actual database to verify the actual policies.

## REVISED 2026-05-18 — dual RLS mechanism (GUC for backend, JWT for client direct reads). See Clau_Session_Log CL-79 (superseded by CL-85).
This subtask shipped (Status='Done') with a GUC-based RLS pattern, not the `auth.jwt()->>'tenant_id'::uuid` pattern originally specified below. The reason is runtime-mechanism: the orchestrator + ingestion worker connect via direct Postgres DSN as a dedicated Postgres role; they never see a Supabase Auth JWT.
FINAL DECISION (CL-85, supersedes CL-79):
- **Backend runtime** (orchestrator + ingestion worker): GUC pattern. RLS policies use the helper `app_current_tenant()` which reads `current_setting('app.current_tenant_id', true)::uuid`. Connection wrapper sets the GUC on checkout. Currently the only active path.
- **Client runtime** (team-web frontend reading own data via Supabase Auth + publishable key): JWT pattern, activated when the first tenant-facing dashboard read query lands. NOT YET WIRED — invariant clutter to add it before a code path uses it.
- **Helper function when JWT path activates**: `COALESCE(current_setting('app.current_tenant_id', true)::uuid, <jwt-derived-tenant>)`. Standard Supabase + Postgres RLS pattern.
The 32-policy count and the attack-test count of '7' are aspirational from the original spec; the shipped reality at 2026-05-18 has policies on a different table set (per the schema execution-order pivot under VT-19) and 1 parametrized attack test covering 4 sub-attacks on 1 table. The breadth gap is tracked as VT-3.3d (parametrize RLS attack test across all 13 tenant-scoped tables + negative test) per CL-84.
OPEN SUB-DECISION (Fazal to resolve before the first tenant-facing dashboard subtask): does `tenants.id` map 1:1 to a Supabase Auth user (add `auth_user_id` column to tenants) or is there a `user_tenants` mapping table? Clau-recommended B2 (mapping table), pending Fazal decision. Affects the JWT side of the COALESCE helper.
