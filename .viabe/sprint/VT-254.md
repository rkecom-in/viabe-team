# VT-254 — systemic `SET LOCAL app.current_tenant = %s` bug (RLS launch risk)

Discovered by the VT-140 E2E harness. `SET LOCAL app.current_tenant = %s` is a
Postgres SYNTAX ERROR — params ($1) cannot bind into a SET statement. Every
tenant-scoped RLS read/write through these raises SyntaxError against a REAL
postgres role; MagicMock-cursor unit tests hide it. VT-45 + VT-43 fixed in
VT-140 (#175). The SAME bug remains in 7 tools:

- query_customer_ledger
- get_business_profile
- get_recent_campaigns
- schedule_followup
- send_whatsapp_message (VT-44)
- privacy/cohort
- privacy/customer_registry

## Fix (mechanical, proven)
Replace `cur.execute("SET LOCAL app.current_tenant = %s", (tid,))` →
`cur.execute("SELECT set_config('app.current_tenant', %s, false)", (tid,))`
everywhere (the fix used in VT-45/VT-43/#175 + the VT-241 canary). Add a
real-DB (non-mock-cursor) test per tool so the regression can't recur (the mock
cursor is what masked it).

## Why Critical
Reports-Jun15 / E2E launch risk — these tools break the moment the SR agent
calls them against real RLS. ONE PR; grep `SET LOCAL app.current_tenant` to find
all sites; canary = a real-DB call per tool. HOLD merge for Pillar-7.
