## What CoderC must produce
The agent reasoning trace plus the PII redactor that all observability components share. Reasoning trace captures every prompt + response + tool args + tool results from the agent SDK. Redactor strips PII deterministically before any of this enters logs/traces.
1. **PII redactor module** at `apps/team/src/privacy/pii_redactor.py`:
	- `redact(value: Any, depth: int = 0, max_depth: int = 5) -> Any`. Recursive walker over dicts, lists, strings.
	- Patterns redacted:
		- **Phone numbers**: `+91XXXXXXXXXX` and 10-digit Indian phone patterns → hashed via E.164 hash from VT-7.1. Format: `<phone:hash:abc123>`.
		- **Email addresses**: regex `[\w.+-]+@[\w.-]+\.\w+` → `<email:hash:xyz789>`.
		- **Indian PAN**: `[A-Z]{5}[0-9]{4}[A-Z]` → `<pan:redacted>`.
		- **Indian Aadhaar**: 12-digit patterns → `<aadhaar:redacted>` (extra cautious; even partial display is risky).
		- **Bank account / IFSC / GST**: corresponding patterns → redacted.
		- **Credit card numbers**: 13-19 digit patterns matching Luhn algorithm → `<cc:redacted>`.
		- **Customer / owner names**: heuristic detection via VT-12.4's name registry — NOT possible deterministically. Phase 1: tokenize known customer names from the tenant's customers table; replace exact matches with `<customer_name>` or `<owner_name>`. False negatives accepted for Phase 1; PR follow-up for stronger detection if needed.
		- **Message bodies**: any string longer than 200 chars that matches likely-message patterns (campaign content, owner replies) hashed: `<body:hash:def456>`.
	- Redaction is structural; CodeX rejects PRs that disable it conditionally ("only in dev", "only for tenant X", etc.).
	- Output preserves structure: dict keys preserved, only values redacted; arrays preserve length.
2. **Reasoning trace capture** at `apps/team/src/observability/reasoning_trace.py`:
	- The agent SDK (VT-4) emits trace events: `agent_reasoning_step`, `tool_call_args`, `tool_call_result`. Each captured via `pipeline_log.log_event` with appropriate event_type.
	- The trace is a directed graph: each tool call has a parent reasoning step; each reasoning step has a parent agent invocation.
	- Visible via `query_run(run_id)` (VT-12.2's API).
3. **Replay tooling**:
	- `apps/team/scripts/replay_run.py <run_id>`: prints the redacted reasoning trace as a readable timeline.
	- For debugging: Fazal can ask "what did the agent do for run X?" and get a clean trace.
	- Tool not exposed to tenants Phase 1.
4. **Redaction tests** at `apps/team/tests/privacy/test_pii_redactor.py`:
	- Phone E.164 patterns redacted.
	- 10-digit Indian phone patterns redacted.
	- Email patterns redacted.
	- PAN, Aadhaar, IFSC, GST patterns redacted.
	- Credit card with valid Luhn checksum redacted.
	- Long message bodies hashed.
	- Customer name tokenization: synthetic tenant customer "Rajesh Kumar" → redacted in test message containing his name.
	- Customer name false negative: name not in tenant's customers table → NOT redacted (acknowledged Phase 1 limitation; documented).
	- Recursive structures: deeply nested dict/list → all PII at all levels redacted.
	- Max depth: 6-level nested structure → redaction stops at depth 5 with marker `<redaction_truncated>`.
	- Idempotency: redact(redact(x)) == redact(x).
5. **Reasoning trace tests** at `apps/team/tests/observability/test_reasoning_trace.py`:
	- Synthetic agent run → reasoning steps + tool args + results captured in pipeline_log.
	- Replay script: synthetic run_id → readable timeline output.
	- PII in trace: synthetic agent invocation with phone in customer name → trace contains hashed phone.
	- Cross-tenant: reasoning trace queries are run-scoped, not tenant-scoped (since run_id is unique); confirm leak-proof.
## Architectural rules to enforce in this subtask
- Pillar 7: redaction always-on. CodeX rejects conditional disabling.
- Pillar 8: one redactor. CodeX rejects parallel redactors.
- Customer name detection is best-effort; false negatives acknowledged + documented; not blocking.
- Reasoning trace flows through pipeline_log; no parallel store.
- Replay script ops-only Phase 1; tenant-facing replay is Phase 2.
- Idempotent redaction: re-redacting an already-redacted string is safe.

- ML-based PII detection (Phase 2; Phase 1 is regex + tenant registry).
- Cross-language PII patterns beyond English/Hindi (Phase 2; e.g., Tamil/Marathi/Bengali names).
- Address redaction (Phase 2; addresses don't typically appear in agent traces).
- Image PII redaction (Phase 2; if traces ever capture images).
- Tenant-facing replay (Phase 2).
- Fuzzy name matching beyond exact (Phase 2).

Bank-account redaction is KEY-DRIVEN ONLY (`bank_account` / `account_number` / `acct_no`). Pattern-driven catches still cover phone, Aadhaar, CC, IFSC, GSTIN, PAN cases — only an unstructured pure-digit bank-account string outside a known key escapes. (Cowork review §Condition 1.)

### Ten required assertions (all must PASS; any fail = canary FAILED)

**Group A — Regression (existing callers unchanged):**

1. **VT-101 LangSmith path still redacts.** Build the same synthetic input VT-101's canary used (`+919876543210` in string + `customer_name: Rajesh Kumar` + nested `body: Hi I want to cancel`). Run it through the LangSmith `@traceable_node` decorator. Fetch the trace via `client.read_run(run_id)`. Assert the trace's input field contains the redacted markers (NOT raw values) — i.e., VT-101's canary assertion #5 still passes after the redactor consolidation.

2. **VT-102 pipeline_log path still redacts.** Build the same synthetic payload VT-102's canary used. Call `log_event(...)`. Direct SQL SELECT the persisted row's JSONB `payload`. Assert no raw PII; redacted markers present — VT-102's canary assertion #2 still passes.

**Group B — 7 PII pattern types (per brief §1):**

3. **E.164 + 10-digit Indian phone redacted.** Synthetic input strings: `"+919876543210"`, `"9876543210"`, `"call me at 98765 43210"`. After redaction: no raw 10-digit sequences remain; each match replaced with `<phone:hash:HEX>` marker.

4. **Email redacted.** Inputs: `"fazal@viabe.ai"`, `"customer.support@gmail.com"`. After redaction: no `@` characters in raw form; markers `<email:hash:HEX>` present.

5. **Indian PAN + Aadhaar + IFSC + GST + Bank patterns redacted.** Inputs: `"ABCDE1234F"` (PAN), `"123456789012"` (Aadhaar pattern), `"HDFC0001234"` (IFSC), `"22AAAAA0000A1Z5"` (GSTIN). After redaction: raw values absent; format-specific markers present (`<pan:redacted>`, `<aadhaar:redacted>`, etc.).

6. **Credit card with valid Luhn passes; without Luhn doesn't trigger.** Input: `"4532015112830366"` (valid Luhn) → redacted as `<cc:redacted>`. Input: `"1234567890123456"` (fails Luhn) → NOT redacted (avoids false positives on order IDs, tracking numbers, etc.). Assert both behaviours.

7. **Long message body redacted.** Input: a 250-char synthetic message body string. After redaction: replaced with `<body:hash:HEX>` marker. Input: a 150-char string (below 200 threshold) → NOT redacted (proves the threshold logic).

**Group C — Customer-name tokenization (per brief §1, tenant registry):**

8. **Customer name from tenant registry redacted; unknown name not redacted.** Insert a real `customers` row for canary tenant_A with name `Rajesh Kumar`. Run redactor on input containing both `"Rajesh Kumar"` (registered) and `"Random Person Name"` (unregistered). Assert: `Rajesh Kumar` → `<customer_name>`; `Random Person Name` → unchanged (Phase 1 false-negative behaviour, documented per brief).

**Group D — Idempotency + recursion:**

9. **Idempotent: `redact(redact(x)) == redact(x)`.** Build a complex nested dict with PII at multiple levels (phone in a string, email in a dict value, customer name in a list element, all wrapped 3 levels deep). Redact once, assert structure; redact the result, assert byte-identical output to first redaction. Recursive structures handled correctly (5-level nesting → all levels redacted; 6-level → truncates at depth 5 with `<redaction_truncated>` marker).

**Group E — On-the-wire proof via real Anthropic:**

10. **Real Anthropic call with PII in prompt → Anthropic receives only redacted content.** Build a prompt containing `+919876543210` and `Rajesh Kumar`. Pass through redactor BEFORE calling `client.messages.create(model="claude-haiku-4-5", ...)`. Capture the `redacted_prompt` and assert no raw PII in it. Make the real API call (cost: ~$0.0001 with Haiku — negligible). Receive response. Assert the call succeeded AND the captured redacted_prompt was what was actually sent (not the original). This is the "on the wire" proof — the redactor must produce the sent payload, not be a post-hoc no-op.
