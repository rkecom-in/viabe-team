## 2026-05-16 CLARIFICATION (audit-driven, L1-L3 architecture changed)
**L0 stays custom. L1-L3 now use Mem0 OSS library. L4 stays custom.**
This subtask's scope is UNCHANGED — it's still about the orchestrator-agent's L0 tool registration + Context Composer hook. But the broader memory architecture changed on 2026-05-16:
- **L0 (workspace knowledge, orchestrator-only)** — stays fully CUSTOM. This subtask is correct as written.
- **L1-L3 (tenant business knowledge, customer profiles, anonymized patterns)** — now use Mem0 OSS library configured against our Supabase Postgres. See VT-KnowledgeArchitecture parent page "2026-05-16 ARCHITECTURE REVISION" section.
- **L4 (hand-authored skill corpus)** — stays CUSTOM.
**What this means for VT-3.10 implementation:**
- The `write_l0_fragment` and `query_l0` tools called out below remain CUSTOM (not Mem0). L0 is Viabe-specific orchestration patterns (admission gate decisions, escalation rules, routing memory) — not generic agent memory.
- The Context Composer in VT-71 will eventually integrate BOTH L0 (custom) and L1-L3 (Mem0 client calls) and L4 (custom). This subtask only wires L0.
- Tool inventory in VT-3.9 will eventually include Mem0-backed retrieval tools for L1-L3 reads alongside L0 tools. Those tools come in VT-7.x respec, not here.
No body changes needed to this subtask's main content. The clarification above is sufficient for the implementation team.

## What CoderC must produce
The orchestrator-side integration with L0 memory. This subtask is the bridge between the orchestrator-agent's reasoning loop (VT-3.9) and the L0 infrastructure (VT-7.N). Three pieces:
1. **Tool registration in orchestrator-agent's tool registry:**
	- `write_l0_fragment(fragment_type, cohort_key, content, observation_count)` — wraps `apps/team-orchestrator/src/memory/l0_write.py:write_l0_fragment`.
	- `query_l0(fragment_type, cohort_key, semantic_query=None, k=5)` — wraps `apps/team-orchestrator/src/memory/l0_read.py:query_l0`.
	- Both registered in `apps/team-orchestrator/src/orchestrator/tools/__init__.py` exclusively for orchestrator-agent (lint rule rejects specialist imports).
2. **L0 fragment generation guidance in agent system prompt** (extends VT-3.9):
	- Section in system prompt explaining WHEN to write L0:
		- `routing_decision`: when orchestrator-agent makes a non-obvious routing choice (e.g., "decided to respond directly instead of spawning specialist because tenant signaled urgency")
		- `specialist_outcome`: after specialist returns, record what worked or didn't (e.g., "weekly_cadence triggered SR; SR proposed segment_offer_burst; campaign approved")
		- `trigger_pattern`: cross-tenant observations (e.g., "tenants in cafe + tier_2 routinely escalate weekly_approval on Sundays")
	- Explicit guidance: NEVER write tenant-identifying content. K-anonymity gate (VT-7.N) will reject; better to not even attempt.
	- Confidence band guidance: only write fragments with observation_count >= 3 to ensure quality.
3. **Context Composer integration** (extends VT-71):
	- Context Composer queries L0 with ([tenant.business](http://tenant.business)_type, [tenant.city](http://tenant.city)_tier, current_phase) cohort key.
	- Returns top-5 fragments by recency + semantic similarity.
	- Fragments included in orchestrator-agent's bundle as `prior_observations` section (capped at 1K of the 4K bundle).
4. **Tests** in `apps/team-orchestrator/tests/orchestrator/test_l0_integration.py`:
	- Orchestrator-agent can call `write_l0_fragment` end-to-end; fragment lands in L0.
	- K-anonymity gate rejection: synthetic tenant-identifying fragment → rejected at gate → step_record documents rejection → agent's next reasoning step does NOT see the rejected fragment.
	- `query_l0` returns relevant fragments for a synthetic cohort.
	- Context Composer integration: orchestrator-agent invocation with populated L0 → bundle contains `prior_observations` section.
	- Lint rule: specialist agent code attempting to import `write_l0_fragment` → PR rejected at CI.

## Architectural rules to enforce in this subtask
- L0 write is ORCHESTRATOR-AGENT ONLY. CodeX lint rule enforces.
- All L0 writes pass through k-anonymity gate (VT-7.N).
- L0 fragments aggregate, never tenant-identifying. If content scan finds PII, write rejected.
- Context Composer L0 read is read-only; bundle size capped.
- L0 tools registered in orchestrator-agent tool inventory; not exposed to specialists.
