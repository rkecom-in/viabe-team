TECH DEBT (context, not a fix): VT-122 (PR #33) reconciled the campaigns table to CampaignPlan v1.0 via a `plan_json` JSONB column carrying the v1.0 plan dict — not individual typed columns. Defensible: v1.0 is a discriminated union, flat-column unpacking is brittle. ## What landed
VT-122 (PR #33) reconciled the campaigns table to CampaignPlan v1.0 by adding a `plan_json` JSONB column that carries the v1.0 plan dict, rather than reconciling individual typed columns to the v1.0 shape. Migration 018.

## Why it's defensible
CampaignPlan v1.0 is a discriminated union (three variants, different field sets). Unpacking a union into flat typed columns is awkward and brittle — every variant change would mean a migration. JSONB stores the agent's structured output as-is. Pragmatic and correct for the agent-output contract.

## Consequence the owner-surface subtask author MUST know
The campaigns table is now a MIXED model: `plan_json JSONB` (agent output) + whatever typed columns the lifecycle-state field adds later. The separate lifecycle-state subtask (approved/rejected/sent/failed, owner-surface scope) will add a TYPED column alongside the JSONB blob. That is fine — lifecycle state is queried/filtered by the owner surface and benefits from being a real column — but the author should not assume the campaigns table is all-typed or all-JSONB. It is both.

## Not a fix item
No action required now. This entry exists so the owner-surface / campaigns-schema subtask inherits the context instead of rediscovering it.
