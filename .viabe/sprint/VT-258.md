# VT-258 — query_customer_ledger real wiring (forward row)

VT-257 made `query_customer_ledger` degrade gracefully (returns empty, no crash)
because BOTH its backing schema elements are unlanded:
- `customer_ledger_entries` table is not in main.
- `customers.phone_token` does not exist (landed customers has `phone_e164`;
  phone tokens live in `phone_token_resolutions.token`, not joined to customers).

When the ledger schema lands (the migration that adds `customer_ledger_entries`
+ the token-keyed customer lookup), wire the tool to real data:
- Resolve `customer_phone_token` → customer via the landed token path.
- Read the real `customer_ledger_entries` rows.
- Flip the VT-254 real-DB test (`test_query_customer_ledger_degrades_gracefully`)
  from a graceful-empty assertion to a success + cross-tenant-denial assertion
  (mirror the other RLS-enabled tool tests).

**Do NOT build until the ledger-schema row lands** — there is nothing to join to
today (ingestion methods 1–9 largely pending). Cowork confirmed graceful-empty
is the correct interim (2026-05-30). Depends on the ledger-schema migration row.
