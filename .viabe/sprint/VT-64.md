## What CoderC must produce
Apache AGE installed and configured on the Supabase Postgres instance. The L1 KG schema defined as a Cypher-compatible graph backed by AGE. Migration scripts. RLS-equivalent enforcement on graph tables. Smoke tests proving the graph works end-to-end.
1. **AGE installation**:
	- Migration `019_apache_age_install.sql`: `CREATE EXTENSION age; LOAD 'age';`. Verify via Supabase support that AGE is permitted on the project tier; if not, document the alternative (graph-on-relational simulation) in `docs/team/kg-backend-decision.md` before this subtask proceeds.
	- Per-environment graph: `viabe_team_kg_dev` and `viabe_team_kg_prod`. Created via `SELECT create_graph(...)`.
2. **L1 node + edge types** (defined in `apps/team/src/kg/schema.py` as Pydantic models that mirror Cypher labels):
	Node labels:
	- `Tenant` { tenant_id, business_name, business_type, locality, city, state, signed_up_at }
	- `Customer` { customer_id, tenant_id, phone_e164_hash, name_initials, acquired_via, last_seen_at, opt_out_status } — phone HASHED at the KG layer (full phone lives in `customers` table, hash in KG enables joins without exposing the value)
	- `Transaction` { transaction_id, tenant_id, amount_paise, transaction_date, source }
	- `Campaign` { campaign_id, tenant_id, status, sent_at, attribution_close_at, arrr_paise }
	- `Locality` { locality_id, name, city, state, pincode } — SHARED across tenants (a locality is not tenant-scoped data)
	- `BusinessType` { business_type_id, label } — SHARED
	- `Platform` { platform_id, name } — SHARED (Zomato, Swiggy, Magicpin, GBP)
	- `PlatformListing` { listing_id, tenant_id, platform_id, rating, review_count, last_updated_at }
	Edge types:
	- `Tenant -[OPERATES_IN]-> Locality` (cross-tenant safe; locality is shared)
	- `Tenant -[CLASSIFIED_AS]-> BusinessType` (shared)
	- `Tenant -[OWNS]-> Customer` (tenant-scoped)
	- `Customer -[MADE]-> Transaction` (tenant-scoped)
	- `Tenant -[SENT]-> Campaign -[TARGETED]-> Customer` (tenant-scoped)
	- `Campaign -[ATTRIBUTED]-> Transaction` (tenant-scoped)
	- `Tenant -[HAS_LISTING]-> PlatformListing -[ON]-> Platform` (tenant-scoped on the listing; platform is shared)
	- `PlatformListing -[HAS_THEME]-> Theme { theme: text, count: int, sentiment: text }` (tenant-scoped)
3. **Tenant-scoping enforcement**:
	- Every tenant-scoped node has `tenant_id` as a property. AGE doesn't have RLS natively, so enforcement is at the application layer via typed wrappers (VT-8.1).
	- The wrapper's KG query helper REJECTS Cypher queries that don't include `WHERE n.tenant_id = $tenant_id` for tenant-scoped node types.
	- Phase 1 acceptable approach: a query-builder API in `apps/team/src/kg/query.py` that programmatically constructs Cypher with mandatory tenant_id binding. Raw Cypher strings are forbidden in app code (CodeX lint rule).
4. **Phone hashing**:
	- At KG-write time, `phone_e164` from `customers` is hashed (SHA-256 with a per-environment salt) and stored as `phone_e164_hash` on the KG `Customer` node.
	- The salt is in `INTERNAL_API_SECRET`-equivalent env vars (`KG_PHONE_HASH_SALT_DEV`, `KG_PHONE_HASH_SALT_PROD`); never in code, never committed.
	- This lets graph traversals match customers by phone without leaking phone values into graph storage.
5. **Smoke tests** at `apps/team/tests/kg/test_l1_schema.py`:
	- Create a synthetic tenant + customer + transaction + campaign + attribution edge.
	- Run a Cypher query: "all customers of tenant_A who transacted in the last 30 days" — returns expected count.
	- Cross-tenant attack: same query with tenant_id swap returns 0.
	- Wrapper enforcement: a Cypher string without tenant_id WHERE clause raises `MissingTenantScopingError`.
	- Phone hashing: customer phone in `customers` table differs in shape from KG node's `phone_e164_hash` (hash form, not plaintext).
	- Cleanup: tenant deletion cascades to all owned nodes + edges (foreign-key-equivalent in graph form).

## Architectural rules to enforce in this subtask
- Pillar 3: tenant scoping is structural. The query-builder wrapper makes scoping mandatory; raw Cypher is forbidden. CodeX rejects PRs introducing raw Cypher strings.
- Pillar 8: one graph schema across the whole team product. Methods don't get their own subgraphs; they write into the shared schema with appropriate node + edge types.
- Phone numbers in plaintext NEVER in KG. Hash at write-time; original value lives only in `customers` table behind RLS.
- Shared nodes (Locality, BusinessType, Platform) are NOT tenant-scoped. They're reference data. Edges from tenant-scoped nodes to shared nodes carry the tenant scoping.
- AGE backend choice is documented; if AGE is unavailable on the Supabase tier, the graph-on-relational fallback (`graph_nodes` + `graph_edges` tables with JSONB properties) is documented and used. The application API stays the same.
- Cypher queries are programmatically constructed; Phase 1 does not expose a Cypher REPL or admin UI.
