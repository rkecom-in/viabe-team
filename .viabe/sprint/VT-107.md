## What CoderC must produce
The edge-case suite: a curated set of 100+ test scenarios that exercise the pipeline's known failure modes. Run on every PR via CI. Failures here block merge.
1. **Suite organization** at `apps/team/tests/edge_cases/`:
	- Subdirectories per major component: `webhook/`, `orchestrator/`, `agent/`, `mcp_tools/`, `ingestion/`, `kg/`, `privacy/`, `owner_surface/`, `billing/`, `landing/`, `observability/`.
	- Each test scenario is a Python file with synthetic setup + expected outcome + Pillar reference (which Pillar the scenario tests).
2. **Scenario coverage** (target 100+, sourced from):
	- Hindi/English/mixed variants for every classifier intent (13 intents × 3 lang × 3 variants = ~117 scenarios).
	- Cross-tenant attack tests for every tenant-scoped table (15 tables from VT-8.1; one attack scenario each = 15 scenarios).
	- Concurrent operation races (founding counter, opt-out + active campaign, refund + cancellation = ~10 scenarios).
	- Malformed inputs for every external surface (webhook, sign-up form, DSR endpoints, snapshot widget = ~20 scenarios).
	- PII redaction across all 7 patterns + 5 nesting levels (~15 scenarios).
	- Hard-limit edge cases (just-under-limit, just-over-limit, pre-flight estimate boundaries = ~12 scenarios).
	- Timeout + retry scenarios for every external API integration (~15 scenarios).
	- Total estimated: 200+ scenarios at suite maturity.
3. **CI integration**:
	- Suite runs on every PR via GitHub Actions; failures block merge.
	- Suite split into fast (<5 min) and slow (full ~30 min) tiers; fast on every PR, slow nightly + on labeled PRs.
	- Coverage report posted as PR comment.
4. **Maintenance protocol**:
	- Every soak failure → new scenario added to suite within 24 hours of root-cause identification.
	- Every production incident (post-launch) → new scenario.
	- Every PR introducing a new feature → at least one edge-case scenario added to relevant subdirectory.
	- Scenario curation: review quarterly to retire scenarios for code paths no longer in use.
5. **Tests** (the suite IS the tests; this subtask delivers the framework + initial scenarios):
	- Framework smoke test: synthetic scenario passes/fails as expected.
	- Coverage check: every component subdirectory has ≥5 scenarios.
	- CI-blocking: deliberate failure in a scenario blocks merge.
	- Fast/slow tier classification works.
## Architectural rules to enforce in this subtask
- Pillar 8: one suite. CodeX rejects parallel test corpora.
- Every scenario references the specific Pillar(s) it tests.
- Every soak failure → suite addition within 24h is documented in suite README; CodeX rejects PRs that add scenarios without referencing source (soak run ID, incident ID, or feature PR).
- Scenario quality: each scenario must have a clear synthetic setup, expected outcome, and Pillar reference. CodeX rejects vague or incomplete scenarios.
- Suite is committed to repo; no external test scenario stores.
## Acceptance criteria

- ≥100 scenarios across all 11 components on initial commit; targeting 200+ at maturity.
- CI integration with fast + slow tier split.
- Scenarios reference Pillars + sources.
- Maintenance protocol documented.
- Coverage report on PRs.
- Suite passes on a clean PR.
- mypy --strict passes.
