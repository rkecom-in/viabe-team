# CL-36 — DBOS chosen over Temporal for durable execution substrate

DBOS chosen as durable execution substrate for Phase 1. Decided over Temporal on three grounds: (1) Time to productive build is 4-7 days vs 3-4 weeks for Temporal — doesn't push July 15 soft launch. (2) Uses existing Postgres (Supabase viabe-team-prod) as durability backend; no new infrastructure to operate or pay for. (3) Lighter code paradigm (decorators on Python functions) vs Temporal's workflow/activity rewrite; if we ever need to leave DBOS, the porting cost is contained. Trade-off accepted: DBOS is younger than Temporal, less battle-tested at extreme scale (1000+ subs), smaller community. Phase 4+ revisit trigger: if we hit DBOS limits at 1000+ subscribers, evaluate Temporal migration at that point. DBOS-LangGraph integration exists first-class. Mike Stonebraker + Matei Zaharia behind DBOS (2024 founding); Postgres veterans, credible technical leadership.

Implementation impact:
- Replaces 'checkpointer-only durability' framing throughout architecture diagrams.
- VT-3 Orchestrator parent gets a new subtask for DBOS workflow wrapping.
- pipeline_runs / pipeline_steps tables still ours; DBOS workflow state lives in its own dbos_\* tables in same Postgres.
- Every pipeline run becomes a DBOS workflow; every step within a run becomes a DBOS step. Auto-resume on crash.
- Hard-limit checks and revise-loop logic stay where they are; DBOS only adds durability + auto-recovery underneath.
