# VT-286 — WhatsApp WABA onboarding via Meta Embedded Signup

## Why
Customer comms run on the **owner's own WABA** — Meta mandates client-owned WhatsApp
Business Accounts; On-Behalf-Of is dead. Business-forward identity, Viabe disclosed
as operator/processor. This is the launch-critical entry point: without an owner WABA
connected + verified, no customer messaging happens. Inbound-first model (VT-287)
builds on top. Zero manual paste after the owner approves (CL-421).

## What
Owner-owned WABA connect via Meta **Embedded Signup** (Twilio tech-provider):
- `build_embedded_signup_url(tenant, ...)` → Meta popup (owner's ~5-min flow:
  Embedded Signup + business verification + display-name = their shop name + privacy
  URL).
- callback (uses the **VT-289 hardened state** nonce; tenant from stored record) →
  token exchange → provision a **dedicated NEW number per tenant** (Fazal's choice) →
  submit display name → persist tokens encrypted per-tenant (reuse the
  `tenant_oauth_tokens` pattern; Fernet `encrypt_value`).
- System user automates all post-grant config (no further owner steps).
- Track per-tenant status: `pending → verifying → name-approved → live`. A tenant
  CANNOT send until `live` (Jan-2026 Meta rule: business verification + privacy URL
  before templates).

## Acceptance criteria
- Embedded Signup exchange persists an encrypted per-tenant token (real/sandbox,
  fail-not-skip — DR-15).
- Status-gate: send is blocked when status != `live`.
- Callback derives tenant from the VT-289 nonce, not the URL.
- Zero-paste after the owner approves (CL-421).
- Dedicated number provisioned per tenant.

## Out of scope
- The inbound message pipeline + auto-reply (VT-287).
- The state-CSRF hardening itself (VT-289 — this row consumes it).
- Email/SMS hook channels (VT-288).
- Meta Tech Provider track approval — **Fazal initiates separately; do NOT block the
  build** (build against Embedded Signup / Twilio tech-provider API, canary vs
  sandbox/mock, live walk E2E-deferred like Shopify VT-283).
