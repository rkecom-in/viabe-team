# CL-91 — Stack PR-fix-5 on PR-fix-1's branch

DECISION (Fazal + Clau 2026-05-18, in response to Claude Code's branching question): when a fix-PR has a real semantic dependency on another open PR (edits the same function in the same way), stack on the open PR's branch rather than branching off main.

IMMEDIATE APPLICATION:
- PR-fix-1 (PR #11, open): branch `fix/vt-orchestrator-ingress-hardening`, off main. Rewrites webhook_pipeline_run to move dedup + event-build inside the workflow.
- PR-fix-5: branch `fix/vt-orchestrator-brain-pending-record`, off `fix/vt-orchestrator-ingress-hardening` (NOT main). Edits the same function PR-fix-1 just rewrote. PR base in GitHub = PR #11's branch. After PR #11 merges to main, GitHub auto-retargets PR-fix-5's base; one-click rebase, no manual intervention.

MERGE ORDER: PR #11 first, PR-fix-5 second. PR-fix-5 cannot merge until PR #11 lands.

RATIONALE:
- Option 2 (branch off main, accept conflict) buys a merge conflict that resolves to 'do what option 1 already did' — wasted Fazal-time relearning the dependency.
- Option 3 (wait for #11 to merge) serializes work that doesn't need to be serial; Claude Code can start PR-fix-5 immediately against the stacked base.
- Stacking is the standard tool when semantic dependency exists. GitHub renders it cleanly: reviewer sees only the brain-pending diff against PR-fix-1's rewritten file.

GUARDRAILS:
- Stacking is acceptable for 2-deep. At 3+ deep, the rebase pain compounds and the review surface gets confusing. If a third PR depends on PR-fix-5, prefer waiting for merge over stacking 3-deep.
- The other 5 fix PRs (PR-fix-1, 2, 3, 4, 6) touch independent files. They branch off main in parallel — no stacking needed.
- Document the dependency in the PR description ('Stacked on #11; merge order: #11 then this').

WHO ENFORCES: Claude Code sets the base branch correctly when creating PR-fix-5. Fazal merges in dependency order. If GitHub's auto-retarget fails post-merge, Claude Code rebases on request.
