## What CoderC must produce
L0 workspace-wide operational memory for the orchestrator-agent. Distinct from L1-L4 (which are tenant-scoped). L0 is k-anonymized aggregate; never tenant-identifying. Only the orchestrator-agent writes to L0. Specialists never touch L0.

**Contents of L0:**
- Cross-tenant trigger-pattern memory (e.g., 'weekly_approval replies at 7am Mondays are typically APPROVE; replies at 11pm Sundays are typically SKIP').
- Known operational states across all tenants ('tenants in their day-39 window need extra attention').
- Orchestrator-agent's own reasoning history about routing decisions.
- Aggregated patterns about specialist invocations that worked vs failed.

1. **Schema** at migration `028_l0_workspace_memory.sql`:
	- `l0_fragments`:
		- `id` UUID PK
		- `fragment_type` TEXT NOT NULL (e.g., 'trigger_pattern', 'operational_state', 'routing_decision', 'specialist_outcome')
		- `cohort_key` TEXT NOT NULL (canonical cohort identifier; same format as L3 patterns)
		- `n_tenants` INT NOT NULL CHECK (n_tenants >= 10) -- enforces k-anonymity at schema level
		- `content` TEXT (the actual fragment content; plaintext, no PII)
		- `embedding` VECTOR(1536) (for pgvector retrieval)
		- `confidence_band` TEXT (low / medium / high based on n_tenants + observation count)
		- `created_at` TIMESTAMPTZ
		- `expires_at` TIMESTAMPTZ (90-day expiry, reconstructed nightly)
	- INDEX on `(fragment_type, cohort_key)` for fast retrieval.
	- HNSW vector index on `embedding` for semantic retrieval.
	- NO `tenant_id` column. L0 is cross-tenant by design. RLS NOT applied (L0 is aggregate, not per-tenant data).
2. **Write path** at `apps/team-orchestrator/src/memory/l0_write.py`:
	- `write_l0_fragment(fragment_type, cohort_key, content, observation_count) -> L0WriteResult`.
	- Hard guard: only callable from orchestrator-agent code (CodeX enforces via lint rule: import allowed only in `apps/team-orchestrator/src/agents/orchestrator/`).
	- Routes EVERY write through VT-8.3 k-anonymity admission gate with k_min=10.
	- If gate rejects: write a `pipeline_step` documenting rejection; do NOT write to L0.
	- If gate admits: compute embedding (Anthropic embeddings API or sentence-transformers), insert row.
	- Atomic with the orchestrator-agent's enclosing DBOS step.
3. **Retrieval path** at `apps/team-orchestrator/src/memory/l0_read.py`:
	- `query_l0(fragment_type, cohort_key, semantic_query=None, k=5) -> list[L0Fragment]`.
	- Used by Context Composer (VT-71) at orchestrator-agent invocation time.
	- Returns top-k fragments by recency + semantic similarity.
4. **Tests** at `apps/team-orchestrator/tests/memory/test_l0.py`:
	- Synthetic write with n_tenants=10 → admitted, row inserted.
	- Synthetic write with n_tenants=9 → rejected via k-anonymity gate.
	- Tenant-identifying content (phone number, business name in content text) → rejected at content scan.
	- Write attempt from specialist agent code path → lint rule fires; PR rejected.
	- Read returns top-k by semantic similarity.
	- 90-day expiry: synthetic row past expires_at → not returned in queries.

## Architectural rules to enforce in this subtask
- L0 is orchestrator-agent ONLY. Lint rule rejects writes from specialist code.
- k≥10 enforced at schema level (CHECK constraint) AND at write-time (gate).
- No PII in content. CodeX scans for phone-number patterns, tenant_id patterns, customer-name patterns; rejects PRs that allow PII through.
- 90-day expiry. Nightly reconstruction job (added to VT-3.5 scheduled triggers).
- Pillar 6: build-time k-anonymity (gate enforced at write).
- Pillar 7: aggregate-only; per-tenant data never lands here.
- Pillar 8: one write path; one read path; CodeX rejects shadow L0 stores.
