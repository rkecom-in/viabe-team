=== WHERE SPEED COMES FROM (operational rules, effective immediately) ===

1. SINGLE-PR-PER-TASK DEFAULT. Split a task into multiple PRs ONLY when the failure surfaces are genuinely independently bisectable AND Clau can name each surface in one sentence.

Override test: 'Could a bug in PR N specifically be diagnosed without PRs N+1 and N-1 being involved?' If yes — split is legitimate. If 'sort of, depends on which subsystem' — single PR.

2. MINIMUM VIABLE BRIEFS. Default brief is 200-300 lines, not 900-1270.

Length check: if Clau's draft brief exceeds 400 lines, stop and identify what's filler.

3. GROUND-TRUTH PRE-FLIGHT BEFORE BRIEF DRAFTING, NOT AFTER. First move on any task that depends on current repo state: ask Fazal to paste the actual file(s) Clau will modify, OR ask Fazal to run rg/grep/git-log checks Clau can't run.

4. ONE TASK PER SESSION, CLOSED. Session shape: ground-truth verify → brief → Claude Code builds → review → PR open → merge. If a session can't close one PR, the task scope was wrong — fix scope, don't multi-session.

5. LOG ENTRY DISCIPLINE. Log ONLY:
  - Decisions Fazal locks
  - Blockers Clau can't resolve in-session
  - State Snapshots at session close
  - Tech Debt items
  - Questions for Fazal that don't get resolved in-session
  - Corrections to prior load-bearing entries

Do NOT log: process introspections, standing-discipline-from-self-reflection, 'I noticed I should be better about X' meta-observations, redundant entries restating the same fact.

Log-count check: a healthy session produces 1-3 log entries. A session producing 10+ entries is a sign Clau is treating logging as productive work.
