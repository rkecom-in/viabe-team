## What CoderC must produce
The `request_owner_approval` MCP tool. Sends a structured approval request to the owner via WhatsApp template, persists the pending state to LangGraph's checkpointer, and pauses the agent's run until the owner replies. The orchestrator's webhook (VT-3.3) detects the reply and resumes the run with the owner's decision in the state.
1. **Tool implementation** at `apps/team/src/agent/tools/request_owner_approval.py`:
	Input schema:
	- `approval_type: Literal['campaign_send', 'cohort_size_exceeded', 'sensitive_data_access', 'other']` — categorizes the request for telemetry.
	- `summary: str` (max 500 chars) — brief human-readable summary of what's being approved.
	- `details: dict` — structured data the owner needs to make the decision (e.g., for `campaign_send`: cohort_size, expected_arrr_range, message_preview).
	- `template_id: str` — must be in approved registry; typically `weekly_approval_request` for campaigns.
	- `template_params: dict[str, str]` — substitutions for the template.
	- `timeout_hours: int = 48` — if no reply by then, the run resumes with `decision='timeout'`.
	Output schema:
	- `status: Literal['paused', 'error']`
	- `approval_id: UUID` (the pending-approval record's ID)
	- `pause_token: str` (the LangGraph checkpoint resume token)
	- `error_envelope: ErrorEnvelope | None`
2. **Pause-state persistence**:
	- Migration `013_pending_approvals.sql` adds a table: `id UUID PK`, `tenant_id`, `run_id`, `approval_type`, `summary`, `details JSONB`, `requested_at`, `timeout_at`, `resolved_at NULL`, `decision` enum NULL, `owner_message_sid`.
	- On invocation: insert a row with `decision=NULL`, send the WhatsApp template via VT-5.7, capture `message_sid`.
	- The agent run is paused via LangGraph's checkpointer. The `pause_token` is the checkpoint thread state.
	- The agent does NOT poll; it returns control to the orchestrator with a typed pause result. The dispatch node (VT-3.4) sees the pause and exits the run loop without timing out.
3. **Resume flow** (consumed by webhook, NOT by this tool but documented here for clarity):
	- Owner replies via WhatsApp — orchestrator's webhook (VT-3.3) classifies as `weekly_approval_response` (or other approval type) intent.
	- Webhook looks up unresolved `pending_approvals` for the sender. Updates `decision` based on owner's reply (typically: `approved | rejected | needs_changes`).
	- LangGraph state is resumed via checkpoint; the agent's next invocation sees the decision in `state.pending_owner_inputs` and proceeds.
4. **Timeout handling**:
	- Scheduled trigger (added via VT-3.5 timeout sweep — if not already in scope, add a 5th trigger): every 30 minutes, scans for `pending_approvals WHERE timeout_at <= now() AND resolved_at IS NULL`. Resolves them with `decision='timeout'` and resumes the affected runs.
	- Default timeout is 48 hours; configurable per-call up to 7 days (max).
5. **Tenant scoping**: ctx.tenant_id; pending approvals are tenant-scoped. Cross-tenant attempt (e.g., resuming a different tenant's pause) impossible by RLS.
6. **Tests** at `apps/team/tests/agent/tools/test_request_owner_approval.py`:
	- Happy path: tool emits template, persists row, returns `paused`. Simulated owner reply resolves the row, agent resumes.
	- Timeout: synthetic 49-hour-old approval triggers timeout sweep, resumes with `decision='timeout'`.
	- Multiple pending approvals: a tenant has 2 concurrent pauses; resolving one doesn't affect the other.
	- Cross-tenant: tenant_A cannot resolve tenant_B's pending approval.
	- Template send failure: VT-5.7 returns error → `request_owner_approval` returns error_envelope, no pending_approvals row written.
	- LangGraph resume: pause_token correctly resumes the agent's run from where it paused.

## Architectural rules to enforce in this subtask
- Pillar 1: this tool is a pause primitive. It does NOT manage the resume — that's the orchestrator's webhook + scheduled trigger. CodeX rejects PRs that try to make this tool poll for the answer.
- Pillar 7: owner approval is authoritative. The agent cannot proceed without it (or without timeout). CodeX rejects any code path that auto-approves on the agent's behalf.
- Pause state is durable. LangGraph's Postgres checkpointer ensures that even if the orchestrator process restarts, the run resumes correctly when the owner replies.
- Approval records are auditable. `pending_approvals` is read by VT-9.7 portal for owner-facing pending-approval visibility.
- The tool counts as one tool call (against the 25-call cap from VT-4.4). Pause does not exempt it.
- Webhook resume logic (VT-3.3 + VT-3.4) is in scope of those tasks; this subtask provides the pause primitive only.
