# VT-618 — onboarding_conductor 400: orphaned tool_use after a mid-loop hard-limit abort

## Symptom (owner-visible)
`multi_field_single_message` scenario: owner sends business name + category + store URL in one message.
Manager replies *"I've hit a small snag saving your setup, so I've flagged it to my team"* then next turn
*"all saved now"* — with **no real save** (judge honesty=2, "contradictory action claim"). Stable across
clean3/4/5 runs (means 3.4/3.8/3.8). The "snag" is a REAL failure, not fabrication.

## Root cause (evidence-based, from deployed-dev `railway logs`, 553bc0e→7ba6962)
The onboarding_conductor specialist (`onboarding_conductor.py:512 build_onboarding_conductor_agent`,
langgraph `create_agent` ReAct loop, model **claude-sonnet-5**) dies with:
```
anthropic.BadRequestError: 400 - messages.2: `tool_use` ids were found without `tool_result` blocks immediately after: toolu_01BBAH…
onboarding_conductor: specialist invocation failed — deterministic floor engaged   (onboarding_conductor.py:701)
```
The chain:
1. A **hard-limit fires MID-loop** — with VT-617 (tool-call 5→10) the binding limit is now **cost** (`cost_paise observed=518 limit=500` = ₹5); before VT-617 it was tool_calls=5. Either aborts the ReAct loop.
2. The abort happens **after** an assistant `tool_use` block is emitted but **before** its `tool_result` is appended → the message state has an **orphaned/unpaired tool_use**.
3. The conductor is **retried** (`spawn_onboarding_conductor` fired **3×** in one scenario) and re-invokes with that broken state → Anthropic **400** (tool_use without immediately-following tool_result).
4. `_node` (line 701) catches it → `_deterministic_floor_reply` → the scripted "snag/trouble" line. Owner never gets a real save.

This is a VT-602-class message-validity bug, but the TRIGGER is the hard-limit abort path, not normal reasoning.

## Proposed fix (design — NOT yet built; do not start tired)
Two independent layers (defense in depth):
1. **Sanitize on abort / before invoke.** When a hard-limit aborts the loop (or in `_node` before `sub_graph.invoke(state)`), repair `state["messages"]`: for any trailing assistant `tool_use` lacking a following `ToolMessage`, either drop the unpaired `tool_use` or append a synthetic `tool_result` ("aborted: hard limit") so the next invoke is Anthropic-valid. This is the durable fix — a mid-loop abort must never leave invalid message state.
2. **Cost-limit headroom for onboarding.** ₹5/turn is now the binding truncation for legitimate multi-field onboarding (opus/sonnet round-trips × ~6-10 tools). Decide: raise the per-turn cost budget for the onboarding path, OR make onboarding cheaper (batch `record_answer`, fewer round-trips). Raising tool-calls to 10 (VT-617) just moved the wall from tool-count to cost — the real issue is the manager makes too many model round-trips per turn.
