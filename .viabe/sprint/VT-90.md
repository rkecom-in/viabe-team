## What CoderC must produce
The trial state machine. New tenant gets 14 days. Day 12: warning template. Day 14: trial ends with payment prompt OR (if eligible) extension offer. Up to 3 extensions, each adding 14 days; max trial duration 14 + 3×14 = 56 days (close to the 60-day cap mentioned in [concept-team.md](http://concept-team.md)).
1. **Trial-end scheduled trigger** added to VT-3.5:
	- Daily 7:00 AM IST: scans tenants where `phase IN ('trial', 'trial_extended')` and `trial_started_at + 14*(1 + extension_count) days <= now() - 2 days`. (i.e., trial ends in 2 days).
	- Fires `trial_ending_warning` event → sends template `trial_ending` via VT-5.7. Template includes: trial end date, plan options, dashboard link to subscribe.
2. **Trial-end day**:
	- Daily 8:00 AM IST trigger: scans tenants where trial actually ends today.
	- For each:
		- **Engagement check** (eligibility for extension): tenant has approved ≥1 campaign in this trial period. Engagement query is deterministic SQL on `campaigns.status`.
		- **Engaged**: send `trial_extension_offered` template (template 19 of 19; PR follow-up to VT-13.3) with explanation: "Your trial ends today. Since you've been actively using Viabe Team, we're extending your trial by 14 more days at no charge. Use this time to see more results before deciding."
		- Auto-extends: applies `trial_extension_granted` event via VT-3.2; `trial_extension_count++`; phase → `trial_extended`.
		- **Not engaged**: send standard `trial_ending` template (different copy) with subscription prompt; phase stays `trial` until they subscribe or 7-day grace period elapses.
		- **At max extensions (count == 3)**: send template `trial_max_reached` (template 20; PR follow-up): "You've used all 3 extension periods. To keep using Viabe Team after \[date\], please choose a plan." No further auto-extensions; manual `discuss` only.
3. **Grace period after trial ends**:
	- 7 days after trial ends without subscription: phase transitions to `cancelled` automatically. Tenant can re-onboard later but their trial slot is consumed.
	- During grace period: campaigns paused (no sends); portal still readable for the tenant to download their data.
4. **Extension max enforcement**:
	- 3 extensions cap is structural in VT-3.2's invariant (`trial_extension_count <= 3`). Bypassing requires raw SQL + service role.
	- Raising the cap is Type 2 governance.
5. **Tests** at `apps/team/tests/billing/test_trial.py`:
	- Day 12 warning: synthetic 12-day-old trial → `trial_ending` template sent.
	- Day 14 engaged: 14-day-old trial with 1 approved campaign → auto-extension; phase = trial_extended; extension_count = 1.
	- Day 14 not engaged: 14-day-old trial with 0 approved campaigns → standard prompt; phase stays trial.
	- Max extensions: 56-day-old trial (3 extensions used) → `trial_max_reached` template; no further extensions.
	- Grace period: 21-day-old trial (post-grace) without subscription → phase = cancelled.
	- Cross-tenant: tenant_A trial trigger doesn't affect tenant_B.
	- Race: simultaneous extension trigger + subscription event → idempotency holds (extension granted then subscription captures payment; both succeed).
