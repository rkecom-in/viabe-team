## What CoderC must produce
A waitlist mode for the landing site. When `NEXT_PUBLIC_TEAM_LAUNCH_MODE=waitlist`, the sign-up form is replaced with a waitlist email form. When `=live`, full sign-up flows. Toggle controlled via Vercel env var; redeploy switches modes.
1. **Mode-aware rendering** in `apps/team-web/app/team/page.tsx` and `/team/signup/page.tsx`:
	- Reads `NEXT_PUBLIC_TEAM_LAUNCH_MODE` (build-time env; values: `waitlist | live | maintenance`).
	- **Waitlist mode**: Hero CTA changes from "Start free trial" to "Join waitlist"; sign-up form replaced with a 2-field form (email + WhatsApp). Day-39 promise section + pricing section visible (informational); sign-up form sections show a "Get notified at launch" form instead.
	- **Live mode**: Full sign-up flow per VT-11.2.
	- **Maintenance mode**: Single-page "We're temporarily down for maintenance, back soon" with no form. Used during launch hot-fixes.
2. **Waitlist data capture**:
	- Email + WhatsApp + optional referral_source.
	- POST to `/api/team/waitlist` (lightweight endpoint; doesn't create a tenant).
	- Stored in `waitlist_signups` table (migration `037_waitlist.sql`): `id`, `email`, `whatsapp_e164`, `referral_source`, `created_at`, `notified_at NULL`. Email + WhatsApp UNIQUE per signup (deduplication).
	- On launch (mode flip from waitlist to live): manual ops job sweeps the waitlist and sends a launch announcement template to all entries; sets `notified_at`.
3. **Cross-mode discipline**:
	- The toggle is build-time. CodeX rejects PRs that read `NEXT_PUBLIC_TEAM_LAUNCH_MODE` at runtime in client code (causes hydration mismatches).
	- Each mode is a distinct rendered output; the toggle picks the rendering tree at build.
4. **Mode transition runbook**:
	- `docs/runbooks/launch-mode-transitions.md` documents how to flip modes (Vercel env var + redeploy + waitlist sweep).
	- Estimated downtime per transition: <60 seconds (Vercel deployment time).
5. **Tests** at `apps/team-web/__tests__/team-launch-mode/*.test.tsx`:
	- Waitlist mode: hero shows "Join waitlist"; form has 2 fields.
	- Live mode: hero shows "Start free trial"; form has 4 fields + consent.
	- Maintenance mode: only maintenance page shown.
	- Waitlist submission: synthetic POST → row in waitlist_signups; deduplication on duplicate email.
	- Mode change at build: changing env var + rebuild renders different mode.
## Architectural rules to enforce in this subtask
- Pillar 7: waitlist is honest — owner is told they're being notified later, not given a trial.
- Pillar 8: one toggle; CodeX rejects scattered conditional rendering.
- Toggle is build-time, not runtime. Hydration mismatches are bugs.
- Waitlist data is opt-in for marketing comms (single confirmation message at launch + opt-out instructions in that message).
- Maintenance mode is a Phase 1 deliverable for launch hotfixes; documented runbook.
- Mode transitions documented in runbook; not done ad-hoc.
