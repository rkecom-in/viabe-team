## Parent task
VT-Billing-Trial-Refund.
## Concept reference
- `concept-team.md` §13 (card capture happens at trial-to-paid conversion, NOT at sign-up; reduces sign-up friction; Razorpay-tokenized; PCI-compliant via Razorpay).
- `concept-team-pillars.md` Pillar 7 (no card capture upfront = honest signal that the trial is genuinely free; tenant only commits when they convert).
- DPDPA + Razorpay PCI-DSS compliance.
## Dependencies
- VT-Billing VT-10.1 Done (Razorpay subscription creation requires a card token).
- VT-OwnerSurface VT-9.7 Done (portal hosts the card-capture form).
## What CoderC must produce
The card-capture flow at trial-to-paid conversion. Owner clicks "Subscribe" in their portal (or via WhatsApp deep link); a Razorpay-hosted page or in-context modal collects card details; Razorpay returns a tokenized payment_method_id; backend creates the subscription.
1. **Frontend integration** (apps/team-web):
	- Subscription page at `/team/subscribe?plan=<tier>`.
	- Razorpay Standard Checkout integration (Razorpay-hosted; we never see card details).
	- Or: Razorpay Subscription Checkout with their hosted form.
	- On success: Razorpay returns subscription_id; frontend POSTs to `/api/team/razorpay/subscribe` to confirm tenant binding.
2. **PCI compliance posture**:
	- Card details NEVER touch our infrastructure. Razorpay's iframe / hosted page handles capture.
	- Documented at `docs/team/pci-posture.md`: Viabe Team is a SAQ-A merchant (Self-Assessment Questionnaire A) by virtue of fully outsourcing card data to Razorpay.
	- No PAN, no CVV, no expiry stored. Only `razorpay_subscription_id` and `razorpay_customer_id` retained.
3. **Card-on-file mechanics**:
	- Razorpay Subscriptions auto-charge on the billing cycle. We don't initiate charges manually.
	- Razorpay sends `subscription.charged` webhook on each successful charge (handled by VT-10.1).
	- Card-update flow (when card expires): owner gets WhatsApp prompt with deep link to `/team/update-payment-method`; same Razorpay-hosted update flow; new subscription_id captured.
4. **Storage**:
	- `tenants.razorpay_customer_id` stores the Razorpay customer reference.
	- `tenants.razorpay_subscription_id` stores the active subscription.
	- `tenants.payment_method_last_four` (display-only; safe to store per PCI; useful for portal display "•••• 4242").
5. **Trial-end conversion path**:
	- When trial ends without engagement (per VT-10.2), template includes deep link: `viabe.ai/team/subscribe?plan=standard&token=<short-lived-jwt>`.
	- Token authenticates the owner without portal login (one-time-use, expires 7 days).
	- On click → subscribe page → Razorpay capture → webhook fires → phase=paid_active.
6. **Tests** at `apps/team/tests/billing/test_card_capture.py`:
	- Synthetic Razorpay success callback → subscription created → tenant updated.
	- Razorpay failure callback → phase stays trial; logged.
	- Card-update flow: existing subscription, new card → new subscription_id stored.
	- Last-four extraction: synthetic Razorpay payment_method response → last 4 stored.
	- Cross-tenant: subscription event for tenant_A doesn't bind to tenant_B.
	- PCI verification: no card data in `pipeline_log`, in DB columns, or in any non-Razorpay storage.
## Architectural rules to enforce in this subtask
- Pillar 7: card capture deferred to conversion is the honest signal. CodeX rejects PRs that move card capture to sign-up.
- PCI: zero card data in our infrastructure. CodeX rejects any column or log holding PAN / CVV / full expiry.
- Last-four is allowed (not PCI-restricted); useful for portal display.
- Razorpay tokens are the only identifier; we never decrypt or transform them.
- Update-payment-method flow uses the same Razorpay primitives; we don't roll our own.
- Documentation at `docs/team/pci-posture.md` mandatory; CodeX confirms it exists with merchant SAQ category stated.
## Acceptance criteria

- Razorpay hosted capture; our code never sees PAN.
- Subscription confirmation flow ties back to tenant via webhook.
- Last-four stored for display.
- Card-update flow tested.
- Cross-tenant attack passes.
- PCI verification: no card data anywhere.
- `docs/team/pci-posture.md` documents SAQ-A posture.
- mypy --strict + TypeScript strict pass.
- PR title references `(VT-Billing-Trial-Refund)`; reviewed by CoderX (mandatory — financial + compliance critical); Fazal personally signs off on PCI posture document.
## Out of scope
- Card capture at sign-up (explicitly forbidden by Pillar 7 + [concept-team.md](http://concept-team.md)).
- Alternate payment methods (UPI, netbanking) — Phase 2 if owners demand.
- International cards (Phase 2; Phase 1 is INR cards from Indian banks).
- Full PCI compliance audit (Phase 2; SAQ-A self-assessment is Phase 1 posture).
- Fraud detection beyond Razorpay's defaults (Phase 2).
