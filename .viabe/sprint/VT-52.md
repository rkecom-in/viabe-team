# VT-52 — Vision-LLM extraction pipeline (shared primitive)

## Parent task
VT-IntegrationInvention.

## Concept reference
- `concept-team.md` §5 (the 9 ingestion methods; methods 1, 4, 5 use vision; method 8 uses image OCR for review screenshots).
- `concept-team-pillars.md` Pillar 4 (no business-type defaults to fill missing fields; extract or ask, never invent).
- `concept-team-pillars.md` Pillar 8 (no patchwork; vision pipeline is one shared module, not duplicated per method).

## Dependencies
- VT-Foundation Done.
- VT-MCP-Tools VT-5.1 Done (the vision pipeline is consumed by ingestion tools that follow the framework contract).
- ANTHROPIC_API_KEY env var set.

## What CoderC must produce
The shared vision-LLM extraction pipeline that methods 1 (paper book), 4 (KOT/POS export), 5 (cash book + voice), and 8 (GBP review screenshots) all consume. One module, multiple consumers. Returns structured fields with per-field confidence.
1. **Module** at `apps/team/src/ingestion/vision_extraction.py`:
	- `extract_from_image(image_bytes: bytes, schema: ExtractionSchema, context: ExtractionContext) -> ExtractionResult`.
	- `ExtractionSchema` is a Pydantic model declaring expected fields (e.g., for paper book: `customer_entries: list[{name: str, phone: str | None, last_visit_date: date | None, total_spend_paise: int | None}]`).
	- `ExtractionContext` provides hints: `language` (en/hi/mixed), `image_type` (paper_book/kot/cash_book/review_screenshot), `tenant_id` for telemetry only.
	- `ExtractionResult`: `extracted: dict` (structured per schema), `per_field_confidence: dict[str, float]` (every leaf field gets a 0-1 score), `overall_confidence: float`, `warnings: list[str]`, `tokens_used`, `cost_paise`.
2. **Model selection**:
	- Phase 1 uses Sonnet 4.6 for image extraction. Vision-capable, cheaper than Opus, accurate enough for handwritten Indian script + table layouts. Pinned in `models.yaml` under `vision_extraction`.
	- Image preprocessing: resize to max 2048px on longest edge; convert HEIC → JPEG; reject corrupt files with structured error before LLM call.
3. **Prompt** at `apps/team/src/ingestion/prompts/vision_extraction_v1.md`:
	- System prompt: "You are an OCR + structured-extraction model. Given an image, return ONLY a JSON object matching the provided schema. For each leaf field, also return a confidence 0-1. If a field is not present in the image, return null AND confidence 0. Do NOT invent values. Do NOT use general knowledge to fill blanks. If the image is illegible, return all-null with overall confidence 0 and warnings list explaining what's wrong."
	- Examples for handwritten Hindi/Devanagari and English script.
	- Token budget under 3,000 (vision tokens dominate; system prompt should be tight).
4. **Confidence semantics** (Pillar 4 enforcement):
	- Confidence < 0.7: do NOT auto-commit. Caller routes to clarifying-question flow (VT-6.2).
	- Confidence 0.7–0.85: commit + notify owner ("we extracted X; let us know if anything's wrong").
	- Confidence ≥0.85: commit silently.
	- These thresholds are uniform across all methods (Pillar 8) and config-driven via `apps/team/config/vision_thresholds.yaml`.
5. **Privacy**:
	- Image bytes are NEVER persisted by this pipeline. They're sent to Anthropic API in-memory and discarded after the call.
	- Owners may upload images that contain PII (customer phone numbers, addresses). Those become customer/transaction rows downstream. The pipeline does NOT log image contents or extracted PII to `pipeline_log` payload — only counts and confidences.
6. **Tests** at `apps/team/tests/ingestion/test_vision_extraction.py`:
	- Synthetic paper-book image (small fixture) → returns expected entries with reasonable confidence.
	- Illegible image → all-null with overall confidence 0 + warnings.
	- Mixed-script image (English + Hindi) → both extracted correctly.
	- Confidence threshold respected: synthetic high-confidence vs low-confidence cases route correctly.
	- PII absence in `pipeline_log`: regex check after a synthetic extraction with phone numbers in the image.
	- Real-API integration test (env-gated): runs against actual Sonnet vision endpoint with a fixture image.
	- Image preprocessing: HEIC → JPEG conversion test; oversized image → resized; corrupt file → structured error before LLM call.

## Architectural rules to enforce in this subtask
- Pillar 4: invented values are forbidden. Confidence 0 + null is the correct response for unknown fields, NOT a best-guess.
- Pillar 8: the pipeline is shared. Method-specific tools (VT-6.4, VT-6.7, VT-6.8, VT-6.11) call this; they do NOT reimplement vision extraction.
- Confidence thresholds are uniform. CodeX rejects per-method threshold overrides.
- Images are NEVER persisted. CodeX rejects PRs that save image bytes to disk or DB.
- The pipeline is method-agnostic. The schema parameter tells it what to extract; it doesn't have method-specific branches.
- Logged payload contains counts + confidences, never extracted PII.
- Real-API integration test is env-gated to avoid CI burning vision-API quota on every run; budget allocated once per Sprint cycle.

- Confidence thresholds config-driven; default 0.7/0.85.
- Image preprocessing handles HEIC, oversized, corrupt files.
- Tests pass for synthetic + illegible + mixed-script + PII absence + threshold routing.
- Real-API integration test passes (env-gated).
- mypy --strict passes.
- PR title references `(VT-IntegrationInvention)`; reviewed by CoderX.

## Out of scope
- Method-specific schemas (VT-6.4, VT-6.7, VT-6.8, VT-6.11 each define their own).
- Voice transcription (separate pipeline; VT-6.8 will pair voice with this for cash-book + voice).
- The clarifying flow that fires on low confidence (VT-6.2).
- Dedup logic that handles repeated entries across uploads (VT-6.3).
- ML model training / fine-tuning — explicitly forbidden by Pillar 5.
- Image storage / archival (out of scope; images are discarded post-extraction).
