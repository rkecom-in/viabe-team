## What CoderC must produce
The `match_transactions` MCP tool. Given a candidate set (typically a campaign cohort + window), find which transactions in that window match which customers via UPI string analysis. Returns matches with confidence scores. Writes confirmed matches to the `attributions` table.
1. **Tool implementation** at `apps/team/src/agent/tools/match_transactions.py`:
	Input schema:
	- `campaign_id: UUID` — the campaign whose attribution we're computing.
	- `window_start: datetime`, `window_end: datetime` — the attribution window (typically `sent_at` to `sent_at + 7 days`).
	- `confidence_threshold: float = 0.7` — matches below this are returned but NOT auto-written to `attributions` (they need owner review or remain in low-confidence state).
	- `commit: bool = False` — if True, write matches at or above threshold to `attributions` table. If False, return matches without writing (preview mode).
	Output schema:
	- `matches: list[TransactionMatch]` — each item: `transaction_id`, `customer_id`, `confidence` (0-1), `match_method` (`exact_phone | partial_phone | name_substring | upi_handle | manual_owner_assertion`), `evidence` (the matching string from raw payload).
	- `unmatched_transactions: list[UUID]` — transactions in window with no candidate match above threshold.
	- `unmatched_customers: list[UUID]` — cohort customers with no matching transaction.
	- `committed_count: int` — how many were written to `attributions` (0 if `commit=False`).
	- `committed_arrr_paise: int` — sum of attributed transaction amounts written.
2. **Matching algorithm** (deterministic, multi-pass):
	- **Pass 1 — Exact phone match**: UPI payload `payer_vpa` or extracted phone == `customers.phone_e164`. Confidence 0.98.
	- **Pass 2 — Partial phone match**: last 7 digits of UPI payload phone match last 7 digits of customer phone. Confidence 0.85.
	- **Pass 3 — UPI handle match**: payload's `payer_name` field contains a substring of the customer's known UPI handle (if recorded during ingestion). Confidence 0.75.
	- **Pass 4 — Name substring match**: payload's `payer_name` exact-or-fuzzy matches `customers.name`. Levenshtein distance ≤2 → confidence 0.65.
	- **Pass 5 — Manual_owner assertion only**: not run by this tool; an owner-asserted attribution comes through a separate owner-edit flow (VT-9.4 edge cases) and writes directly to `attributions` with method='manual_owner'.
	- Each transaction gets the highest-confidence match from any pass. Ties broken by pass order (1 > 2 > 3 > 4).
3. **Conflict resolution**:
	- A transaction matched to multiple customers across passes: highest confidence wins; ties go to the customer in the campaign cohort (campaign-scoped match preferred).
	- A transaction matched to a campaign cohort customer AND a non-cohort customer at same confidence: cohort customer wins (this is attribution).
	- A transaction in window matching a customer in TWO concurrent campaign cohorts: both campaigns flag it, but attribution goes to the campaign sent first (per VT-5.5 deterministic rule).
4. **Write path** (when `commit=True`):
	- Insert one `attributions` row per match ≥ threshold.
	- Fields: `tenant_id`, `campaign_id`, `customer_id`, `transaction_id`, `attribution_method`, `attribution_confidence`, `arrr_contribution_paise` (the transaction amount).
	- Idempotency: composite UNIQUE constraint on `(campaign_id, transaction_id)`; duplicate writes are no-ops (handled by `ON CONFLICT DO NOTHING`).
5. **Tenant scoping**: ctx.tenant_id; cross-tenant attack tested (a transaction from tenant_B cannot be attributed to a tenant_A customer).
6. **Performance**:
	- Hard timeout: 30 seconds (this is heavier than other tools because it does string matching across hundreds of transactions).
	- For cohorts > 500 customers, the tool returns `error_envelope.code='cohort_too_large'` and the caller (typically scheduled trigger VT-3.5) breaks the work into batches.
7. **Tests** at `apps/team/tests/agent/tools/test_match_transactions.py`:
	- Synthetic dataset: 100 cohort customers, 200 in-window transactions, mix of match types.
	- Verify each pass produces correct matches with expected confidence.
	- Conflict resolution: transaction matching two customers at different confidences → higher wins.
	- Cohort vs non-cohort tie: cohort customer wins.
	- Concurrent campaigns: transaction matching two cohorts → first-sent campaign wins; both attempts write attempted but only one succeeds (UNIQUE constraint).
	- `commit=False`: matches returned, no `attributions` rows written.
	- `commit=True`: matches written, idempotent on re-run.
	- Cross-tenant: tenant_A's call cannot create attributions for tenant_B.
	- Performance: 500-customer/2000-transaction synthetic dataset completes in <30s.
	- Levenshtein correctness: Hindi-English transliteration cases tested ("Rajesh" vs "Raajesh" should match; "Rajesh" vs "Mahesh" should not).

## Architectural rules to enforce in this subtask
- Pillar 2: pure algorithm. The matching uses deterministic string operations + Levenshtein distance. CodeX rejects any LLM call dressed as a "smart matcher."
- Pillar 4: confidence scores are honest. CodeX rejects threshold gymnastics that inflate scores to look more matched than they are.
- Pillar 7: low-confidence matches are returned but not auto-committed. The agent decides; the owner can override via manual_owner.
- The `manual_owner` method is owner-driven, not algorithmic. This tool never produces method='manual_owner'.
- UNIQUE constraint on `(campaign_id, transaction_id)` is the structural defense against duplicate attribution.
- Levenshtein implementation uses an established library (e.g. `python-Levenshtein`), not a hand-rolled one. Reduces correctness risk.
- Hindi-English transliteration is a real constraint. Tests must cover it; failure to match common transliteration variants is a correctness bug.

## Out of scope
- ML-based matching (Phase 2 if Phase 1 false-match rate is unacceptable).
- Owner-asserted attributions (VT-OwnerSurface VT-9.4 edge case).
- Cross-channel attribution (UPI vs cash vs walk-in) — Phase 1 is UPI-only.
- Transaction enrichment / cleanup pipeline (different concern; separate ingestion tooling in VT-6.6).
- Real-time attribution (Phase 1 is batch; called by scheduled trigger or on-demand).
