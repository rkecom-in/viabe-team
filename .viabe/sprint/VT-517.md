# VT-517 — ownership verification → VTR-human-only (launch)

## Problem
The "Confirm you own this business" screen (no-public-number-found state) asks the USER to TYPE a public number then OTPs it = ZERO ownership proof (user-entered contact proves nothing; signup OTP already proved number-control, never linked person→entity). Kill the theater.

## Build (no automated verification method now)
1. **Real, non-bypassable gate.** `ownership_verified` = declarative activation prerequisite, default FALSE at tenant create. Wire into the EXISTING send/execute rail (Gate-0 in agent_send_draft, VT-421) — agent CANNOT send/act until ownership_verified=true. Rail-level, NOT a soft flag.
2. **Remove the self-entered-number OTP step.** Replace the screen with honest copy (business set up; ownership pending a quick Viabe team review; agent won't act until verified). NO fake verify/OTP. KEEP GST verify + signup-number OTP unchanged.
3. **VTR Ops Console surface:** view of `ownership_pending` tenants; VTR marks Verified/Rejected + note/evidence; the action is AUDITED (ties into the VT-514 audit log). Ownership status visible. Respect VTR PII boundary.
4. **Setup proceeds while pending** — onboarding/connect/configure all work; only EXECUTION (send/act) gated.

## Reconcile
SUPERSEDES VT-411 (OTP-to-listed-number) — fold/replace, do NOT leave two ownership paths.

## Out of scope (fast-follow, do NOT build)
PAN↔GSTIN match, domain-email OTP, public-phone OTP, bank penny-drop, DigiLocker.

## Guardrails
Real non-bypassable prereq (prove non-bypass like VT-460/VT-514); accurate copy only; main untouched; NO real send; dev; light-mode lock. Self-gate (PII/rail-touching → adversarial + no-bypass proof before landing). Next migration = 148.
