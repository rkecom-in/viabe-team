## What CoderC must produce
The `classify_owner_message` MCP tool. LLM-backed (one of three permitted by Pillar 2). Takes a Hindi/English/mixed-language owner message and returns a structured intent + confidence. The orchestrator's webhook (VT-3.3) calls this on every inbound message.
1. **Tool implementation** at `apps/team/src/agent/tools/classify_owner_message.py`:
	```python
class ClassifyOwnerMessage(MCPTool):
    name = "classify_owner_message"
    description = "Classifies an inbound owner WhatsApp message into a structured intent. Used by the orchestrator's inbound webhook to route messages. LLM-backed for robust handling of Hindi/English/mixed-language input."
    def is_llm_backed(self): return True
	```
	Input schema:
	- `message_body: str` (1-4096 chars; Twilio's max).
	- `recent_context: list[ContextMessage]` (max 5; the last 5 messages from this owner from `pipeline_log`, oldest first; helps disambiguate "yes" replies).
	- `pending_approval_id: UUID | None` (if there's an outstanding `request_owner_approval` for this owner from VT-5.9, the classifier knows the message is most likely an approval response).
	Output schema:
	- `intent: Literal['weekly_approval_response', 'exclusion_request', 'adhoc_campaign_request', 'status_query', 'template_error_followup', 'refund_decision', 'sign_up_completion', 'support_followup', 'unclear']`
	- `confidence: float` (0-1).
	- `extracted_fields: dict` — e.g. for `weekly_approval_response`: `{decision: 'approved' | 'rejected' | 'needs_changes', notes_for_agent: str | None}`. For `exclusion_request`: `{customer_phone_or_name: str, reason: str | None}`.
	- `original_language: Literal['en', 'hi', 'mixed', 'other']`
	- `model_used: str` (e.g. `claude-haiku-4-5-20251001` — use Haiku for cost; classifying is a small task).
	- `tokens_used: int`, `cost_paise: int`.
2. **Prompt** at `apps/team/src/agent/tools/prompts/classify_owner_message_v1.md`:
	- System prompt instructs the model: "You are a classifier. Given an inbound WhatsApp message from a small-business owner in India, return ONLY a JSON object matching the output schema. Do NOT add prose. Be conservative: if unclear, return `intent='unclear'` with low confidence rather than guess."
	- The 9 intent categories listed with examples in Hindi + English + mixed.
	- Frontmatter same as VT-4.2 (version, token_count, model). Token budget: under 2,000 tokens for this prompt.
3. **Model choice rationale** (mandatory documentation in `docs/team/llm-backed-tools-rationale.md`):
	- Why LLM, not regex: Hindi+English+mixed-language messages don't pattern-match cleanly. Owners may write "haan bhai approved", "हां करो", "go ahead", "OK ji". Regex would need hundreds of patterns and still miss novel phrasings. An LLM with a small prompt handles all of this in one tool call.
	- Why Haiku, not Opus: classification is a small task. Haiku is ~10× cheaper and accurate enough for the 9-class task. Opus is reserved for self_evaluate (VT-5.12) where reasoning quality matters.
	- Cost cap: estimated ~₹0.05 per classification at Haiku pricing. With ~30 inbound messages per tenant per week and 100 tenants: ~₹15/day total. Within Phase 1 budget.
4. **Caching**: identical (message_body, pending_approval_id) within 5 minutes returns the cached classification. Prevents double-classification on Twilio retries (which the webhook should already idempotency-guard, but defense in depth).
5. **Confidence handling**:
	- If `confidence < 0.7`: orchestrator's webhook (VT-3.3) routes to `unclear` flow even if the LLM picked a specific intent. Prevents acting on low-confidence guesses.
	- The threshold is config-driven (`apps/team/config/classifier_thresholds.yaml`).
6. **Tests** at `apps/team/tests/agent/tools/test_classify_owner_message.py`:
	- Synthetic inputs for each of the 9 intents (English + Hindi + mixed); model returns correct intent.
	- Pending approval context: "haan" with `pending_approval_id` set → `weekly_approval_response`, `decision='approved'`. Without `pending_approval_id` → likely `unclear`.
	- Low-confidence handling: ambiguous input returns `unclear` or low confidence; webhook routes correctly.
	- Mixed-language: "customer 9876543210 ko exclude karo, woh angry hai" → `exclusion_request`, extracted phone correct.
	- Cache hit: identical input within 5 minutes returns cached result, no API call.
	- Cost attribution: `cost_paise` populated in every result.
	- Cross-tenant: ctx.tenant_id has no effect on classification (this tool is stateless on tenant data) but is logged for observability.
	- Real-API integration test (env-gated): runs against actual Haiku endpoint with synthetic messages.

## Architectural rules to enforce in this subtask
- Pillar 2: rationale documented in `docs/team/llm-backed-tools-rationale.md` and at the top of the tool file. CodeX rejects PRs without the rationale.
- Model selection in `apps/team/config/models.yaml` under `classify_owner_message: claude-haiku-4-5-20251001`. Switching models is config, not code.
- Output schema is strict. Free-form prose from the model is a validation failure (the system prompt instructs JSON-only).
- Caching is a cost-control measure, not a correctness one. Cache TTL 5 minutes; longer TTLs risk acting on stale context.
- The tool is stateless on tenant data. ctx.tenant_id is used for logging only. CodeX confirms.
- Confidence threshold is config-driven and conservative (0.7). Lowering it is Type 2 governance.
- Prompt is versioned (`_v1.md`); changes go through versioned files.
