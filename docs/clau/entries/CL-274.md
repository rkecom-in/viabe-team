# CL-274 — DECISION (test strategy): two-mode canary pattern for real-API tests of LLM-back

Test-strategy decision (Clau-owned; test strategy is Clau authority). Establishes a TWO-MODE canary pattern for real-API tests of LLM-backed MCP tools. Originated on PR #37 (VT-50 self_evaluate) but is a GENERAL pattern — every LLM-backed tool's real-API canary should follow it, and the E2E testing stage must apply it to any end-to-end path that makes live model calls.

THE PATTERN — a real-API canary has two modes, selected by env var VIABE_CANARY_MODEL ('haiku'|'opus', default 'haiku'); the separate VIABE_RUN_SELF_EVALUATE_CANARY=1 (or per-tool equivalent) still gates whether the canary runs at all:
- HAIKU mode (default, cheap, run freely during iterative fix rounds): real SDK round-trip on Claude Haiku. Asserts PLUMBING ONLY — live response received, JSON parses, return conforms to the expected verdict/result shape, no exception. Does NOT assert semantic correctness (a cheaper model judges differently; correctness asserts would be flaky).
- OPUS / PRODUCTION-MODEL mode (run rarely, pre-merge): real round-trip on the actual production-pinned model. Asserts plumbing PLUS judgment — a deliberately-flawed input yields the correct verdict/category. This is the real verification.
When the run-gate env var is absent, the canary must SKIP (pytest 'skipped'), never PASS.

WHY: real-API tests get run many times across fix rounds; burning production-model (Opus) tokens on rounds that fail for syntax/plumbing reasons is wasteful — a plumbing bug fails identically on Haiku and Opus, so Haiku is sufficient for those rounds. CRITICAL GUARDRAIL: the two modes differ in WHAT THEY ASSERT, not just which model. A Haiku pass is explicitly NOT production verification. This guardrail exists because PR #37's first canary attempt shipped a test that reported PASSED while making no live call at all (CL-272) — the failure mode to avoid is a cheap green check mistaken for real verification.

E2E-STAGE ACTION (the reason this is logged now, not at #37 merge): when Sprint 9 / E2E testing builds end-to-end paths that invoke live models, apply this two-mode split — cheap Haiku plumbing runs for iteration, production-model fidelity run as the pre-merge/pre-release gate. Do NOT let an all-Haiku E2E suite become the verification of record, and do NOT run every E2E iteration on Opus. Production self_evaluate model pin stays Opus 4.7 — unchanged; this decision is about TEST execution only, not the production gate.

Implementation for VT-50 lands in a follow-up commit on PR #37 branch feat/vt-mcp-self-evaluate (HEAD f25e223 at time of logging).
