## Parent task
VT-LandingSite.
## Concept reference
- `concept-team.md` §14 (founding-tier counter widget: live count of remaining founding slots displayed prominently on landing page; creates urgency without manipulation since the cap is real and structural).
- `concept-team-pillars.md` Pillar 7 (counter shows real numbers; never inflated; never decrements when tenants churn — see VT-10.6 no-decrement-on-release policy).
- `concept-team-pillars.md` Pillar 8 (one widget; one data source; one read API).
## Dependencies
- VT-Billing VT-10.6 Done (founding-counter atomic primitive + public read API at `/api/team/founding-status`).
- VT-LandingSite VT-11.1 Done (landing page that hosts the widget).
## What CoderC must produce
A client-side React component that fetches `/api/team/founding-status` every 60 seconds, displays the live count of remaining founding slots, and degrades gracefully when the API is slow or unavailable. Embedded in the pricing section's Founding card.
1. **Component** at `apps/team-web/components/team/FoundingCounter.tsx`:
	- Initial render: SSR-fetched count (avoid layout shift on hydration). The landing page server-fetches `/api/team/founding-status` once at request time and passes initial state as props.
	- Client hydration: re-fetches every 60 seconds via SWR or React Query.
	- Display states:
		- **Available** (claimed_count < cap): shows "X of 100 founding spots claimed" + countdown bar visualization. Underneath: "Founding pricing is locked in forever for these first 100 customers."
		- **Almost full** (claimed_count >= 90): adds urgency framing "Only Y spots left" but NEVER fabricates urgency at lower counts.
		- **Full** (claimed_count >= 100 / `all_claimed: true`): shows "All 100 founding spots are claimed. Standard pricing applies." The Founding pricing card transitions to a faded / unavailable state. The Subscribe-to-Founding CTA disappears.
	- Loading state: skeleton UI matching final layout dimensions (no jank).
	- Error state (API unreachable): falls back to last known good count from SSR; if no SSR data, shows "Loading availability..." without a number.
2. **API integration**:
	- Fetches `/api/team/founding-status` (public, CDN-cached at edge for 60 seconds).
	- Standard fetch with 5-second timeout; on timeout, retains existing state.
	- No PII in request; no auth required.
3. **CTA wiring**:
	- When count < cap: "Claim founding spot" CTA links to `/team/signup?plan=founding`.
	- When count == cap: CTA is replaced with a "View Standard plans" link OR the entire founding card visually de-emphasizes (decision: de-emphasize; the Standard tier becomes the visual default).
	- The actual founding-tier assignment happens server-side in VT-9.2 (counter atomic claim during sign-up); this widget is purely informational. Even if a user clicks "Claim founding spot" while the counter shows 99/100 but races to sign-up after slot 100 is taken, the server falls back to Standard tier with a friendly message.
4. **Honest framing**:
	- The widget does NOT use deceptive urgency tactics (no fake countdown timers, no fabricated "3 people viewing", no manipulated counts).
	- The 100-cap is real (VT-10.6 enforces). The displayed number is the actual count. This is the structural honesty.
	- CodeX rejects PRs adding urgency manipulation patterns.
5. **Accessibility**:
	- Counter announced to screen readers via ARIA live region (polite, not assertive — doesn't interrupt).
	- All visual states have text alternatives.
	- Reduced-motion respected: countdown bar animation suppressed.
6. **Performance**:
	- Component bundle <5KB gzipped.
	- 60-second polling does NOT block other UI; happens in background.
	- No-poll mode if user is not on landing page (avoid wasteful requests; use `IntersectionObserver` to pause polling when widget is off-screen).
7. **Tests** at `apps/team-web/__tests__/team-landing/founding-counter.test.tsx`:
	- SSR render with initial count: rendered HTML contains the correct count.
	- Client polling: synthetic 60-second tick triggers re-fetch; updated count rendered.
	- Available state: 50/100 → standard display.
	- Almost-full state: 92/100 → urgency framing appears.
	- Full state: 100/100 → Founding card de-emphasized; CTA replaced.
	- API timeout: synthetic 5s timeout → retains existing state.
	- SSR-only fallback: client API unreachable, SSR initial count used.
	- Race-to-claim graceful: synthetic claim that races and loses → server returns "plan_tier=standard" with friendly message; widget reflects on next poll.
	- Accessibility: axe-core passes; ARIA live region works.
	- Off-screen: IntersectionObserver pauses polling.
## Architectural rules to enforce in this subtask
- Pillar 7: no urgency manipulation. CodeX rejects fake timers, fabricated viewer counts, etc.
- Pillar 8: one widget; CodeX rejects parallel founding counters (e.g., a separate Hindi-language counter component).
- Real number, real cap. No decoration of the count beyond visual presentation.
- SSR for initial paint; hydration takes over for live updates. No layout shift.
- Polling pauses off-screen.
- Race-to-claim is server-side; widget is informational.
- Bundle size <5KB gzipped enforced by CI.
