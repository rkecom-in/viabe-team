## Contract (Fazal 2026-06-24, found via direct curl). THE blocker for the verified-flow e2e.

The Sandbox `/authenticate` response carries the token TWICE: top-level `access_token` (WORKS) and nested
`data.access_token` (500s on the subsequent /search). `_authenticate` grabbed the NESTED (dud) token, so
every auth 200'd but every GSTIN search Internal-Server-Error'd — which masqueraded as an account/product
issue. Also: the search call wants `x-api-version: 1.0.0` (Fazal's proven-good curl), not `1.0`.

## Fix
1. `_authenticate` prefers top-level `access_token`, falls back to nested for compat.
2. Search uses `_SEARCH_API_VERSION = "1.0.0"` (auth stays `1.0`).
3. Regression test: both-tokens-present → top-level extracted; fallback when top-level absent.
