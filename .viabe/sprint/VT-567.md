# VT-567 — whatsapp: prefix normalization at the ingress boundary

## Root cause (live drill, third layer)
A real WhatsApp inbound arrives channel-prefixed (`From=whatsapp:+91…`) while every downstream identity (tenants.whatsapp_number, phone tokenisation, consent/opt-out) keys plain E.164. Nothing stripped it → `_lookup_tenant` missed EVERY live inbound → reason=unknown_sender → 200-and-drop. Tests only posted plain E.164, so the suite never caught it. (Layers 1+2 were config: Messaging-Service webhook pointed at the old openclaw bot; Vercel dev missing TEAM_TWILIO_AUTH_TOKEN + stale TEAM_TWILIO_WEBHOOK_URL.)

## Fix
`_normalize_wa_fields` strips `whatsapp:` off From/To/WaId ONCE at the ingress boundary — lookups and the workflow both see plain E.164 (a partial strip would fork the phone-token space). Plain payloads pass byte-identical. Tests: pure normalize + a real-format inbound resolves the tenant and completes a run (36/36 ingress suite green).
