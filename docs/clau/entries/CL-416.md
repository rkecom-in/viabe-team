# CL-416 — DECISION (Fazal, LOCKED/STANDING): Pipeline-observability retention is lifetime-of-relationship

## Decision

For the three pipeline observability tables — `pipeline_runs`, `pipeline_steps`, `phone_token_resolutions` — retention is **lifetime-of-relationship**. No time-based deletion. No nightly aggregation-and-drop. **DSR-purge becomes the sole deletion path** for these tables (matching the CL-330 pattern for `owner_inputs` and the CL-390 "Consequence to track" clause for L1 KG vector rows).

## Why this is a Standing decision

CL-21 (2026-05-12, the original pipeline-observability design-doc §2.1 retention spec) said "90-day full retention then aggregate to Pipeline_Run_Summary." That decision was **superseded by the CL-390 privacy cluster** (decisions-ledger line 23 already marks it: *"retention now governed by LOCKED standing privacy decisions"*), but no successor retention number for `pipeline_steps` had been put on the books.

Cowork surfaced the gap 2026-05-26 while filing VT-185 (which had been built against the now-superseded 90-day timer). Clau flagged it for outranking the feature-question stack ("a nightly job that aggregates-and-drops `pipeline_steps` at 90 days may directly contradict the privacy posture and the DSR-purge architecture"). Fazal made the call: lifetime-of-relationship retention.

## Consequences across the codebase

- **VT-185 reframed v1.0 → v2.0** — old scope "nightly aggregation >90d → Pipeline_Run_Summary + drop source rows" is invalid. New scope: "DSR-purge coverage for the 3 pipeline observability tables" (extending `scripts/dsr_purge.py:_PURGE_ORDER`). Priority: Critical (DSR coverage is launch-blocking). Sprint: 1 (was Phase 1.5).
- **Privacy notice (per CL-389)** — must disclose lifetime retention of pipeline observability data + the DSR-driven deletion path. The privacy notice is now the customer-facing contract for this.
- **DSR architecture (PR #51 / `scripts/dsr_purge.py`)** — `_PURGE_ORDER` must extend to cover `pipeline_runs`, `pipeline_steps`, `phone_token_resolutions`. VT-185 builds this.
- **No nightly cron job** — DO NOT file a follow-up VT row for time-based aggregation of pipeline_steps. If query performance on full-history retention becomes an Ops UI problem, file a separate row for compaction/rollup tables that DO NOT delete source data (additive, not destructive).
- **Cost projection** — full-history retention at the projected step volume per tenant (~100-500 steps/day/tenant; 10 tenants design partners) is ~36k-180k rows/year/tenant; well within Supabase Pro pricing. Cost is not a constraint at projected scale.
