## What CoderC must produce
The k-anonymity admission gate. A reusable function called by L3 construction (VT-7.5) and any future cross-tenant aggregation. Given a candidate cohort definition, returns admit/reject + the eligible tenant set.
1. **Gate module** at `apps/team/src/privacy/k_anonymity.py`:
	- `check_admission(cohort_predicate: CohortPredicate, k_min: int = 10) -> AdmissionResult`.
	- `CohortPredicate`: structured filter (e.g., `{business_type: 'cafe', city_tier: 'tier_2', recency_band: '60_90d_dormant', signed_up_before: now() - 180 days}`). The predicate excludes quarantined tenants automatically (180-day rule from VT-7.6).
	- `AdmissionResult`: `admitted: bool`, `tenant_count: int`, `eligible_tenant_ids: list[UUID]` (returned to caller for aggregation), `reason: Literal['admitted', 'below_k_min', 'predicate_invalid']`.
2. **k_min handling**:
	- Default k_min = 10 (concept doc Section 10 locked Type 3 commitment; "kâ‰Ą10, well above typical regulatory k=3-5").
	- k_min is a parameter so the construction pipeline can pass higher values for sensitive aggregations. Raising kâ‰Ą10 is always allowed.
	- The value 10 itself is a Type 3 commitment. Lowering below 10 is forbidden. CodeX enforces: `assert k_min >= 10` in the gate.
3. **Predicate validation**:
	- Every CohortPredicate must include `signed_up_before: timestamp` to enforce quarantine. Missing this field returns `predicate_invalid`.
	- Predicate fields are restricted to a Phase 1 allowlist: business_type, city_tier (NOT city or locality), recency_band, signed_up_before. Adding a new field is Type 2 governance.
	- CodeX rejects PRs that pass `city`, `locality`, `phone`, or any per-customer field â€” those would defeat the anonymity guarantee.
4. **Eligible tenant query**:
	- Runs a SQL query against `tenants` joined with relevant predicate sources. Returns tenant_ids matching the predicate.
	- Cap the query at 10,000 tenants for safety (admission for more than 10K is unusual; if it happens at scale, Type 2 governance discussion).
5. **Logging**:
	- Every admission check logged to `pipeline_log`: `event_type='k_anonymity_check'`, payload contains `predicate_hash`, `k_min`, `tenant_count`, `admitted`. The `eligible_tenant_ids` are NOT logged (would be a privacy issue itself).
	- Below-k-min rejections are logged at `info` level (frequent, not concerning).
6. **Tests** at `apps/team/tests/privacy/test_k_anonymity.py`:

	- Boundary: exactly 10 tenants â†’ admitted.
	- Synthetic 15 tenants matching predicate â†’ admitted with `tenant_count=15`.
	- Predicate without `signed_up_before` â†’ `predicate_invalid`.
	- Predicate with forbidden field (e.g., `city`) â†’ `predicate_invalid`.
	- k_min < 10 â†’ assertion error.
