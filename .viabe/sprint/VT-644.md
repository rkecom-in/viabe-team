## Problem (real prod money-safety window; not reproduced by any non-crash eval)
`customer_send.agent_send_draft` (L2 path): the `agent_drafts.status='sent'` flip + audit capture commit
inside `with conn.transaction()` (customer_send.py:877-885), but the `agent_customer_contacts` INSERT
(the SOLE substrate `check_agent_send_caps` counts for the 30d recontact / weekly / 90d suppression)
ran as a SEPARATE autocommit statement AFTER. A crash in the gap leaves the draft 'sent' but the
suppression row ABSENT; the DBOS re-drive early-outs on `draft_status=='sent'` (replay guard :510) and
never re-reaches the INSERT → the cap record is permanently lost → a later batch re-sends to that
customer inside their suppression window. The L3 path already writes this INSERT inside its FOR UPDATE txn.

## Fix
Moved the `agent_customer_contacts` INSERT INSIDE the `with conn.transaction()` block so the draft->'sent'
flip + suppression row commit atomically (both-or-neither), matching L3. `_finalize_batch_if_terminal`
deliberately stays OUTSIDE (VT-382 externalization, backed by the daily outbox_redaction sweep).

## Verify
Regression rail: the enlarged txn makes a rollback re-drive the send → leans on `send_idempotency_keys`;
mitigation confirmed ('sent' idempotent hit inside send_whatsapp_template → no 2nd Twilio call). Existing
real-DB exactly-once test asserts success-path atomicity.

## Crash-injection regression (LANDED) + the trace that proved no residual gap
`test_crash_at_contacts_insert_rolls_back_flip_then_redrive_completes` (real-DB) injects a crash AT the
`agent_customer_contacts` INSERT via a connection proxy, and asserts: (1) the draft->'sent' flip ROLLS
BACK (draft stays 'drafted', contacts=0) — the atomicity guard; (2) a clean re-drive completes flip+
contacts with NO second wire send. Passes by name against an ephemeral migrated DB.

**Why it actually works (deep trace, customer_send.py):** the L2 wire send commits `send_idempotency_
keys='sent'` in its OWN txn (line 820, upstream of the flip txn). The re-drive short-circuit is the
DRAFT_STATUS early-out at line 510 (`draft_status=='sent'`), NOT the idempotency ledger. So keeping the
contacts INSERT inside the flip txn is load-bearing: on failure the flip rolls back → draft stays
'drafted' → line-510 early-out does NOT fire → the re-drive proceeds, the wire send is idempotent
(send_whatsapp_template's own ledger blocks a 2nd Twilio call), and the flip txn re-runs to write
contacts. Pre-fix the flip committed 'sent' first → line-510 stranded the re-drive → contacts lost
forever. No new stuck state introduced. L3 path was already atomic (send+flip+contacts in one FOR UPDATE txn).
