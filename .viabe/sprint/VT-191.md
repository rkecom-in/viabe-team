# VT-191 — Phone number encryption-at-rest

## Why this row exists

VT-184 (Cond 1) shipped `phone_token_resolutions.phone_number_encrypted` as **PLAINTEXT** in Phase-1 dev with a 3-layer ⚠️ WARNING (migration 026 COMMENT ON COLUMN + module docstring + function docstring). The column name is misleading; the lie must be made honest before any real production phone numbers land in the table.

Per CL-390 privacy posture: phones are PII; storing plaintext locally requires either encryption-at-rest OR explicit policy acknowledgment. Encryption is the cleaner posture.

## Pre-prod gate

**Cannot be skipped before:**
- First real production tenant onboarding (Reports-Jun15 → Team-Jul15)
- Any data-import that brings real customer phones into dev DB

## Scope

1. **Add `cryptography` dep** to `apps/team-orchestrator/pyproject.toml`
2. **Add `TEAM_PHONE_ENCRYPTION_KEY` env** to `.viabe/secrets/supabase-dev.env` + production secret mgmt
   - Fernet symmetric key (256-bit)
   - Key rotation strategy (per-tenant key? master key + per-row salt? document choice)
3. **Migration 028:** back-fill `phone_number_encrypted` from existing plaintext rows
   - DECRYPT(EXISTING_PLAINTEXT, NULL) → ENCRYPT(PLAINTEXT, KEY) atomically
   - Drop the `COMMENT ON COLUMN` warning (column now honestly encrypted)
4. **Update `phone_tokens.py`:**
   - `register_phone_token`: encrypt phone before INSERT
   - `resolve_phone_token`: decrypt phone_number_encrypted on SELECT
   - Module docstring: remove ⚠️ WARNING; replace with encryption-mechanism note + key-rotation pointer
   - Function docstrings: remove WARNING
5. **Verify rotation path** — synthetic key rotation in canary; back-fill from old-key to new-key
