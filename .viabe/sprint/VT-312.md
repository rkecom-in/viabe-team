# VT-312 — customer-state detectors (dormancy + high-value) → L2 emit

Split out of VT-309 (Cowork ruling 20260603T191000Z). The two threshold L2 event
types (`customer_dormant_threshold_crossed`, `customer_high_value_threshold_crossed`)
have **no decision-point in the codebase today** — they are net-new product logic,
NOT an emit-wire into an existing site. They feed `context_builder._build_ledger_summary`
(the L2→Composer wire shipped in #275): until these detectors emit, `dormant_cohorts`
+ `top_spenders` are empty in production. NOT dropped — correctly separated from the
VT-309 emit-coverage pass.

## BLOCKED on Fazal threshold decision
- **Dormancy:** what makes a customer dormant? (e.g. N days since last transaction —
  N=90? per-tenant? per-business-type?)
- **High-value:** what crosses the high-value threshold? (e.g. lifetime spend ≥ ₹X
  paise? top-N percentile of the tenant's customers? absolute or relative?)
Cowork will surface both to Fazal with context when this row comes up.

## Dependencies
- VT-66/67 Done (#275) — `episodic_events` + `record_episodic_event` + `_build_ledger_summary` consumer.
- VT-309 — establishes the per-site emit pattern (in-txn `record_episodic_event(conn=...)`, deterministic event_id idempotency).

## What to produce (after thresholds decided)
1. A dormancy detector + a high-value detector. Design call needed: **scheduled batch
   re-evaluation** (e.g. nightly, like the audit-verify sweep VT-304) **vs on-write**
   (emit when a transaction/ledger write crosses the threshold). State the choice + why.
2. Emit `customer_dormant_threshold_crossed` / `customer_high_value_threshold_crossed`
   at the detector's decision point, atomic with the source write (or at-least-once via
   the outbox with a deterministic event_id) — same discipline as VT-309.
3. Confirm `_build_ledger_summary` populates `dormant_cohorts` + `top_spenders`
   end-to-end (canary: detector fires → episodic row → ledger non-empty).

## Architectural rules
- PII-free payloads (CL-390): customer entity id/hash, cohort bucket, days_dormant,
  lifetime_paise — never names/phones.
- Idempotency: a customer crossing a threshold must not emit twice per crossing
  (deterministic event_id or detector-state tracking).
- DR-15 canary; CL-422 synthetic; plan-first (its own plan-review).
