# CL-87 — Create VT-3.3d: parametrize RLS attack test across all 13 tenant-scoped tables

NEXT ACTION (Fazal 2026-05-18 in response to audit H4 / CL-77 recommendation 7): create Notion subtask VT-3.3d.

SCOPE: parametrize the cross-tenant RLS attack test across every tenant-scoped table in the shipped schema. Currently the only RLS attack test (test_migrations.py::test_rls_blocks_cross_tenant) covers phase_transitions; one of 13 tenant-scoped tables.

TABLES NEEDING COVERAGE (13 confirmed tenant-scoped via migration audit):
1. tenants (special: keyed on id, not tenant_id; needs adapted test)
2. phase_transitions — already covered
3. subscriptions
4. pipeline_runs
5. pipeline_steps
6. phone_token_resolutions
7. privacy_audit_log (special: nullable tenant_id)
8. dsr_tickets
9. twilio_inbound_events
10. rate_limit_buckets (special: workspace sentinel UUID)
11. checkpoints (LangGraph) — already covered
12. checkpoint_blobs (LangGraph)
13. checkpoint_writes (LangGraph)

Workspace-only tables (razorpay_webhook_events, env_config) get a separate 'deny-all-from-tenant-role' test — they have policies `USING (false)` and FORCE RLS, so a tenant-scoped role must fail every operation.

IMPLEMENTATION:
- Pytest parametrize over (table, tenant_id_column, special_handling) tuples.
- Each iteration: seed 2 tenants, write a row for each as superuser, then under a non-superuser scoped role: assert SELECT visibility limited to scoped tenant; assert UPDATE/DELETE of other-tenant rows touches 0 rows; assert INSERT with other-tenant tenant_id is rejected by WITH CHECK.
- Special handling for tenants (id, not tenant_id), privacy_audit_log (nullable), rate_limit_buckets (workspace sentinel), and LangGraph checkpoint tables (subquery-based policy via thread_id → pipeline_runs).
- Plus a deliberately-broken-policy NEGATIVE test (audit drift D-Foundation.4 / VT-20 spec): temporarily change one policy from `=` to `!=`, assert attack test now FAILS, restore. Confirms tests have teeth.

ESTIMATED EFFORT: 1-2 hours Claude Code.
DEPENDENCIES: blocked on the C1+DC4 GUC fix PR (Decision CL above on RLS mechanism). The new role-based attack tests are the very thing that catches GUC absence. So VT-3.3d ships AFTER PR-fix-3 (the GUC connection wrapper PR).

ACCEPTANCE:
- Every tenant-scoped table has SELECT + UPDATE + DELETE + INSERT-foreign-tenant attack assertions.
- The 2 workspace-only tables have deny-all-from-tenant-role assertions.
- Negative test (broken policy) makes attack tests fail.
- All passes on CI.

Notion subtask to create under VT-Orchestrator parent. Schedule alongside VT-3.3c but after the GUC fix PR.
