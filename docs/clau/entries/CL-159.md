# CL-159 — Built-State Snapshots discipline

Triggered 2026-05-19 by Fazal's request: he needs visibility into what's actually built vs planned, separate from the target-architecture engineering reference. The Architecture Diagrams page (35f387c2-cc5a-81c2-a1cf-ed73b080f931) shows the target state; the new Built-State Snapshots page (364387c2-cc5a-8141-b59f-c06a097bcc39) shows what exists today.

This entry has been UPDATED IN PLACE (not superseded by new entries) to reflect format and procedural refinements made during the same session. Edit history: original = SVG-only; refined to dual SVG+Mermaid pairing after Fazal tested both formats; session-start ritual extended to include reading the Built-State Snapshots page directly.

=== STANDING RULE ===

WHEN to snapshot:
- New component reaches functional state → add new section to Built-State Snapshots page.
- Existing component changes materially → update its section (Last updated date, Status one-liner, lists, Honest summary, SVG, Mermaid).
- Triggers: first merge that materially adds functionality (scaffolded → has real code); new integration wired; scope expansion (2 tools → 4+, 1 ingestion method → 2+); anything changing 'what does this component do today?'
- Non-triggers: bug fixes within existing functionality; refactors without capability change; documentation-only PRs; dep bumps without capability change; test additions for existing code.

=== FORMAT (dual artifacts) ===

Every snapshot has TWO artifacts on the Built-State Snapshots page:

1. SVG file — the primary visual. Clau generates via file creation, presents in chat, Fazal manually attaches to the Notion page section. This is what Fazal looks at when he wants to understand what's built.

2. Mermaid code block — written directly to the Notion page by Clau via update_content. Diff-friendly, in-page editable, version-history tracked.

Both are kept in sync. When a snapshot is updated, BOTH are refreshed.

Color coding (both formats): green = built and working; yellow = scaffolded or placeholder (exists but not functional); grey-dashed = planned, not present yet.

=== WORKFLOW per snapshot ===

1. Clau drafts the SVG using file creation. Color-coded per above.
2. Clau presents the SVG file to Fazal in chat.
3. Clau writes the equivalent Mermaid + section text to the Notion page section via update_content. Same color coding via classDef.
4. Clau writes the surrounding text: Last updated date, Status one-liner, Built/Placeholder/Not-built lists, Honest summary.
5. Fazal manually attaches the SVG to the Notion page section.
6. Clau confirms attachment happened.

=== OWNERSHIP ===

Clau owns: detecting triggers; drawing the SVG; writing the Mermaid; updating the page text; producing the file for Fazal.

Fazal owns: SVG file attachment; calling out missed triggers; requesting ad-hoc snapshots when he wants the current picture of any component.

=== SESSION-START INTEGRATION ===

At session start, in addition to Clau_Session_Log Standing+Open entries, Clau reads the Built-State Snapshots page (364387c2-cc5a-8141-b59f-c06a097bcc39) to know current state of each component. If a recent merge or change crossed a snapshot trigger and the page wasn't updated, Clau flags it to Fazal at session open.

=== HISTORY RETENTION ===

When a snapshot is updated, the previous Mermaid block goes into a Notion toggle/collapsible block labeled with the previous Last-updated date. Previous SVG stays attached (Fazal's call whether to delete or stack). Notion's page-version-history is the deeper fallback if needed.
