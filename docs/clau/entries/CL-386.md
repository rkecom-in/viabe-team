# CL-386 — DISCIPLINE RULE #14 (Fazal-approved, in force 2026-05-23): Any closeout tracker,

DISCIPLINE RULE #14 — added 2026-05-23, in force immediately. Fazal-approved.

RULE TEXT: "Any closeout tracker, status summary, or session-end handoff must be reconciled against ground truth before it is trusted or relayed: run gh pr list --state merged for PR state, and a log reconciliation (latest snapshot + relevant Standing entries) for decision state. A summary written from memory is not authoritative until reconciled. This applies to Clau's own summaries, not just Claude Code's."

EVIDENCE / RATIONALE: three stale-summary failures in the single 2026-05-23 session, each caught by luck rather than process:
(1) CL-373 closeout tracker (written 2026-05-23 15:33) listed DBOS Step-1b as 'NOT STARTED' — PR #47 had already merged the fix ~17h earlier (caught at CL-374).
(2) The same tracker + the DSR-purge brief stated owner_inputs 'does not exist yet' — migration 020_owner_inputs.sql was already on main (caught at CL-377, only because a tenants-FK grep happened to surface it).
(3) Claude Code's PR-merge summary listed the Voyage canary and DPA/ZDR as open when both were closed (CL-334 / the compliance decision), and omitted the owner_inputs Component 2 rewrite (caught at CL-381).
Each catch depended on an incidental verification, not a standing check. Rule #14 converts that luck into procedure.

SCOPE NOTE: the rule explicitly binds Clau's own summaries and snapshots, not only Claude Code's outputs — the failure class is 'written from memory without reconciliation', which Clau is equally prone to. Before relaying or trusting any session-end handoff, Clau runs the gh + log reconciliation.
