# VT-249 — formalize ultracode + migration-number allocator

## Why
Fazal 2026-05-30: "use ultracode workflows for all big and small tasks" + "Yes
formalize it". Cowork guardrail: serialize allocation under fan-out. The real
concurrency gap: migration numbers were chosen by unlocked directory scan →
parallel subagents collide on the same number (VT-240 + VT-86 both reached for
047). The VT-ID allocator already serializes via flock; migration numbers did
not.

## Part A — formalize ultracode (docs)
- **CL-424** added to `docs/clau/decisions-ledger.md` (Fazal-issued STANDING):
  CC runs xhigh ultracode for all tasks; dynamic fan-out when warranted;
  binding guardrails (allocate IDs+migration-numbers up-front; one PR per row;
  plan-first review on big rows; Pillar-7 merge unchanged).
- CLAUDE.md: "Ultracode + parallel fan-out" discipline note + "Migration
  numbers via the allocator only" + a "How to find a thing" row.

## Part B — migration-number allocator
- `scripts/migration_id_allocate.py` — mirrors `vt_id_allocate.py`: flock on
  `migrations/.lock`, counter `migrations/.next-migration` (seeded 048 — 047 is
  VT-240/#158 in-flight), `--peek` non-consuming, zero-padded 3-digit output.
- `apps/team-orchestrator/tests/test_migration_id_allocate.py` — 6 tests incl.
  a 32-thread concurrency test proving no collision (the gap it closes).
