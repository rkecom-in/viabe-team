# CL-183 — Adjacent verification anti-pattern

## Discipline rule (load-bearing)

Across CL-178 (skipped Context7 validation), CL-181 (aggregate test counts mistaken for canary firing), and CL-182 (temperature=0 reflex without checking model's accepted kwargs), the same failure mode keeps producing real bugs: I infer a specific verification has been done when only an adjacent signal is present.

Unifying rule: a verification step targets a specific surface. Adjacent signals do not substitute, no matter how strong they are.

Concrete examples (each is a real bug caught this session):

| Adjacent signal | Specific verification | Bug if conflated |
|---|---|---|
| Context7 returns docs for library | The specific kwarg you're using is on the library's current API | name= kwarg, prompt= vs system_prompt= |
| pytest run reports X passed / Y skipped, no errors | The specific test (which may be in Y) actually ran | Canary unfired but reported as 'green local run' |
| Familiar Python pattern (temperature=0 for deterministic tests) | The specific model you're calling accepts that kwarg | temperature= rejected by Opus 4.7 |
| Imports resolve in CI's import-collection job | The actual routing primitive executes at runtime | Command.PARENT composition still unverified |
| The library README shows pattern X | Your usage of pattern X actually works given your library version | langgraph_supervisor.create_handoff_tool standalone usage — zero documented examples outside create_supervisor |

## Implementation in review/brief workflow

Before declaring any verification complete, ask: 'is the signal I have the SPECIFIC one needed, or is it an ADJACENT one that I'm hoping transfers?' If adjacent, do the specific check or flag the gap.

For briefs specifically: every external library symbol referenced gets:
1. Context7 query on the current API (CL-140 / CL-178)
2. Codebase grep for prior usage of the same symbol (NEW, this entry)
3. If model-specific: model identifier accepted-kwargs check

For reviews: every claimed verification gets:
1. Was the specific test/check named and did it report a specific outcome (pass/fail/skipped)? (CL-181)
2. Are aggregate counts hiding the specific test in the 'skipped' bucket? (CL-181)
3. Does CI status cover the runtime concern or only collection-time? (NEW, this entry)

This rule supersedes nothing — it stands as a meta-rule alongside CL-140/CL-156/CL-178. Future Corrections that fit this pattern get linked here rather than re-derived.
