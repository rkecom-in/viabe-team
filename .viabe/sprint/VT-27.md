## What CoderC must produce
The langgraph_supervisor integration. **Replaces the prior 'custom dispatch_to_specialist function' design.** The orchestrator-agent (Opus 4.7, from VT-3.9) decides whether to spawn a specialist; the supervisor library auto-generates the `spawn_<specialist>` handoff tools.
**Architectural shift from prior framing:** the dispatch logic is no longer custom Python code that the orchestrator state machine calls. It's now a **handoff tool** registered with `langgraph_supervisor` that the orchestrator-agent calls via its LLM reasoning. Adding Phase 2-6 specialists = register additional specialists with the supervisor; the library auto-generates `spawn_reputation`, `spawn_marketing`, etc. NO custom routing rewrite.
1. **Supervisor setup** in `apps/team-orchestrator/src/orchestrator/supervisor.py`:
	```python
from langgraph_supervisor import create_supervisor, create_handoff_tool

spawn_sales_recovery = create_handoff_tool(
    agent_name="sales_recovery_agent",
    name="spawn_sales_recovery",
    description="Hand off to Sales Recovery Agent for dormant-customer winback campaign work"
)

# Phase 2-6 handoff tools added by registering specialists with create_supervisor()

supervisor_workflow = create_supervisor(
    agents=[orchestrator_agent, sales_recovery_agent],
    model=opus_4_7,
    prompt=ORCHESTRATOR_AGENT_SYSTEM_PROMPT,  # from VT-3.9
    handoff_tools=[spawn_sales_recovery]
)
	```
2. **Context bundle for the handoff**:
	- When `spawn_sales_recovery` fires, the supervisor passes the conversation history + current state to the specialist agent.
	- The Context Composer (VT-71) is consulted to build the `SalesRecoveryContext` for the specialist (L0 + L1-L4 + run state → bundle).
	- Bundle bounded: max 8K tokens of context to leave room for agent reasoning + tool calls within 80K hard limit.
3. **Hard-limit enforcement at handoff boundary** (Pillar 1 — orchestrator enforces, agent doesn't self-police):
	- Wrap the specialist invocation with limit meters:
		- Token meter (Anthropic SDK usage callbacks)
		- Tool call counter
		- Reasoning depth tracker
		- Wall-clock timer (5min hard cap)
		- Cost meter (₹50 ceiling)
	- All 5 metered atomically via the same `@DBOS.step` + `@observability.step` wrapper.
	- On any limit breach: terminate the agent (cancel in-flight request), set `state.escalation_pending = True`, emit `escalate_to_owner` step_record, return partial result with `terminated_by: <limit_type>`.
4. **Result handling**:
	- On normal return: specialist produces a `CampaignPlan` (or other structured output). Persisted to `campaigns` table with `status='proposed'`.
	- State updated: `last_campaign_at`, `history` appended.
	- DBOS step boundary committed atomically with LangGraph checkpoint.
5. **Cost attribution per handoff**:
	- Track LLM cost in `pipeline_steps` step_record with run_id + tenant_id + cost_paise.
	- Cost dashboard (VT-12.3) consumes these. Telegram /costs query also surfaces.
6. **Phase 2-6 readiness**:
	- The supervisor pattern means adding new specialists is additive: register the specialist module with `create_supervisor(agents=[...])` and create a handoff tool. NO change to dispatch logic, NO change to the orchestrator-agent.
	- Phase 1 ships with ONE specialist (Sales Recovery). Phase 2 adds Reputation, etc.
	- Specialist enum is NOT used (the supervisor library handles registration); attempting to invoke a non-registered specialist via direct tool call raises a runtime error.
7. **Tests** in `apps/team-orchestrator/tests/orchestrator/test_supervisor.py`:
	- Happy path: orchestrator-agent decides to spawn Sales Recovery; handoff tool fires; stub specialist returns `CampaignPlan`; campaigns row created; state checkpointed.
	- Hard-limit termination: each of the 5 limits (tokens, tool calls, depth, wall-clock, cost) tested independently; each produces `escalate_to_owner` step_record + state flag.
	- DBOS auto-resume: simulate crash during specialist execution; verify DBOS resumes from last completed step (NOT from beginning of workflow).
	- Cost attribution: synthetic handoff logs cost step_record with non-zero paise.
	- Tenant isolation: handoff under tenant A cannot pass tenant B's context.
	- Library version pinned: `langgraph-supervisor` version locked in pyproject.toml.

## Architectural rules to enforce in this subtask
- **Pillar 1 (revised):** the supervisor handoff is the SOLE entry point from orchestrator-agent to specialist. CodeX rejects PRs that introduce alternate dispatch paths (e.g., specialist invoked directly from a graph node without going through supervisor).
- **Pillar 8 (no patchwork):** hard-limit checks wired at supervisor handoff boundary once. Adding new limits requires updating this file + the limit enum.
- The handoff is opinionated about context size: 8K tokens max for the specialist context bundle. Phase 2 expansion = Type 2 governance.
- Hard-limit termination is hard-cancel, not graceful-stop. CodeX rejects "please finish current step" handling.
- `escalate_to_owner` emitted exactly once per breach. Re-handoff after escalation requires explicit owner reply (gated by VT-3.3 webhook → Pre-Filter → orchestrator-agent reasoning).
- Cost attribution mandatory. A handoff that fails to log cost is a CI failure (test enforces).
- langgraph_supervisor library version pinned. Major updates Type 2.
