## Contract (Fazal 2026-06-23: welcome said "Hi Raj Cafe", not the real owner). CRITICAL — live e2e.

**Root cause (verified in code):** `utils/twilio_send.py:234`
`content_variables=json.dumps(params)` sends **named keys** (`{"owner_name":…, "trial_end_date":…}`),
but Twilio Content templates use **positional** placeholders `{{1}}`,`{{2}}`. Twilio doesn't match the
named keys → the variables render the template's **sample values** ("Raj Cafe" is team_welcome's {{1}}
sample). This breaks personalization on **every** template send (welcome, winbacks, approval, DSR,
escalation, …); it only surfaced now because this is the first real template that actually went out.

The registry already has the answer: `config/twilio_templates.yaml` declares
`variables:` as the **ordered positional list** ("index i = positional {{i+1}}") — e.g. team_welcome:
`[owner_name, trial_end_date]`.

## Fix
1. In `send_template_message`, map the named `params` → positional `content_variables` using the
   resolved template entry's ordered `variables` list:
   `{str(i+1): params[name] for i, name in enumerate(entry.variables)}`.
2. **Fail-closed:** if any declared variable is missing from `params`, do NOT send (return
   success=False / error) — never silently emit Twilio sample values to a real owner/customer.
3. Test: assert the positional mapping for team_welcome (owner_name→"1", trial_end_date→"2") and at
   least one multi-var template; assert a missing var refuses to send.

## Acceptance
- Re-fired welcome to Fazal's tenant reads "Hi <real owner_name>" + correct trial-end date (no sample).
- No template can emit Twilio sample values for unmapped variables.
