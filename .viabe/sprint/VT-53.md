# VT-53 — Clarifying-question flow (shared primitive)

## Parent task
VT-IntegrationInvention.

## Concept reference

- `concept-team-pillars.md` Pillar 7 (owner-truth: when uncertain, ask; never invent).
- `concept-team-pillars.md` Pillar 8 (no patchwork; one clarifying flow shared across methods).

## What CoderC must produce
The clarifying-question flow. When vision extraction (VT-6.1) returns confidence <0.7, the ingestion tool calls this flow to ask the owner via WhatsApp. Owner reply resolves the uncertainty, the resolved data commits to the canonical tables.
1. **Flow module** at `apps/team/src/ingestion/clarifying_flow.py`:
	- `request_clarification(extraction_result, ambiguous_fields, original_image_ref) -> ClarificationRequest`.
	- `ClarificationRequest` writes a row to `pending_clarifications` table (migration 015): `id`, `tenant_id`, `extraction_run_id`, `ambiguous_fields` (JSONB), `original_extraction` (JSONB), `requested_at`, `timeout_at`, `resolved_at NULL`, `resolution_data NULL`, `owner_message_sid`.
	- Sends a `clarifying_question` WhatsApp template to the owner via VT-5.7. Template includes: 1-2 specific ambiguous fields (e.g., "Was the total spend for Rajesh ₹1200 or ₹1500?"), reply format guidance.
2. **Resolution path** (consumed by webhook VT-3.3, but contracted here):
	- Owner reply classified by VT-5.11 with intent `clarification_response`.
	- Webhook looks up `pending_clarifications` by sender + recency. Parses owner reply via a small LLM call (Haiku, prompt narrowly scoped) to extract the specific values for the ambiguous fields.
	- `resolution_data` JSONB written; `resolved_at` set. Triggers the original ingestion tool to commit using the resolved data.
3. **Timeout handling**:
	- 7-day timeout. After timeout, the clarification is closed with `resolution_data='owner_no_response'`. The original extraction is dropped (NOT committed with low confidence — Pillar 4 forbids inventing the missing values).
	- Scheduled trigger (added to VT-3.5) sweeps stale clarifications.
4. **Ambiguity bundling**:
	- If multiple ambiguous fields are below threshold, bundle them into ONE clarifying question per upload. Owner gets one message asking 1-3 questions, not 10 separate messages.
	- Max 3 ambiguous fields per request. Beyond 3: drop the upload, ask the owner to retry with a clearer image (the extraction quality is too low to repair via Q&A).
5. **Owner experience**:
	- Question wording is owner-friendly (Hindi/English/mixed per `tenant.preferred_language`).
	- Always include the source context: "From the paper book photo you uploaded on [date]…"
	- Reply parsing is lenient: "₹1500" / "1500" / "fifteen hundred" / "१५००" all extract to 150000 paise.
6. **Tests** at `apps/team/tests/ingestion/test_clarifying_flow.py`:
	- Single-field ambiguity: synthetic extraction with 1 field <0.7 → clarifying request sent, row persisted.
	- Multi-field bundling: 3 ambiguous fields → one message with 3 questions; 4 ambiguous fields → drop upload + ask retry.
	- Resolution: synthetic owner reply parsed correctly for various phrasings; resolution_data populated; original ingestion tool re-invoked.
	- Timeout: synthetic 8-day-old clarification → closed with `owner_no_response`; original extraction discarded.
	- Cross-tenant: tenant_A cannot resolve tenant_B's clarification (RLS).
	- Hindi reply parsing: `₹१५००` → 150000 paise.
	- Numeric word parsing: "fifteen hundred" → 150000.

## Architectural rules to enforce in this subtask
- Pillar 4: timeout = drop, NOT commit-with-best-guess. CodeX rejects fallback-to-low-confidence-commit.
- Pillar 7: owner is source of truth. Their reply resolves; agent never decides on their behalf.
- Pillar 8: shared flow; ingestion methods don't have their own clarifying logic.
- Max 3 questions per request. CodeX rejects PRs that bundle more.
- 7-day timeout is config-driven (`apps/team/config/clarifying_timeouts.yaml`); changes are Type 2.
- Reply parsing uses Haiku LLM (cost-justified); pattern is similar to VT-5.11 but narrowly scoped to numeric/categorical extraction. Pillar 2: this is an LLM-backed dependency, not a new tool.
- Cross-tenant attack: clarification rows scoped by RLS.
