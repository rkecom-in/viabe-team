## Bug

`signup.py:33` has a stale `_TRIAL_DAYS = 14`, used at ~line 309 to compute the `trial_end`
passed to the `team_welcome` template variable {{2}}. The authoritative trial length is
`config/trial.yaml` → `trial_days: 30` (CL-433; the yaml comment even says "was 14"). The
evaluator/sweep read the yaml, so the trial actually runs 30 days — but the FIRST message a
new owner receives states a trial-end date 16 days early. Owner-visible, found in the
2026-06-11 Sundaram journey review (verified directly in code, not from the handoff).

## Fix contract (one source of truth — do NOT just change 14→30)

1. Welcome `trial_end` MUST derive from the same `config/trial.yaml` `trial_days` value the
   evaluator uses (reuse/extract the existing loader pattern from
   `billing/trial_evaluator.py`; CC's call on the exact shape — shared accessor preferred).
   A second hardcoded constant recreates this exact drift; that is the failure mode being
   fixed, not the number.
2. Delete `_TRIAL_DAYS` from `signup.py` (grep-zero on the constant).
3. Config-read failure inside the welcome path must stay non-terminal (the welcome send is
   best-effort and must never fail the signup — preserve the existing guard semantics).
4. Test: signup welcome receives `trial_end == trial_started_at + config trial_days` (assert
   against the yaml value, not a literal 30, so a future tunable change can't re-drift it).
