# VT-241 — wire cohort resolution into the campaign collapse path

## Why (split from VT-170, 2026-05-30)
VT-170 ships the cohort-integrity primitive: `campaign_recipients` table + composite FKs (cross-tenant linkage physically impossible) + `resolve_cohort_recipients(campaign_id, customer_ids, tenant_id) -> {resolved, rejected}`. But VT-170 deliberately does NOT invoke it from the agent write path, because doing so carries a product behavior decision that's Fazal's to make. This row does the wiring.

## Scope
1. Call `resolve_cohort_recipients` from the collapse node (the path that persists a `proposed` campaign with its `target_cohort.customer_ids`), so every new campaign populates `campaign_recipients` with verified, same-tenant members.
2. **Reject-behavior — FAZAL DECISION REQUIRED:** when one or more cohort customer_ids are unresolved or cross-tenant (returned in `rejected`), the collapse node must either:
   - (A) **fail-closed** — reject the whole campaign proposal, surface the rejected ids to the owner; or
   - (B) **proceed-with-note** — persist the campaign with only the resolved cohort, attach a note listing dropped ids for owner review.
   Do not implement until Fazal rules A vs B. (A is the conservative default; B is more permissive.)
3. After this lands, VT-43 cohort_size/attribution_rate (VT-240) can read `campaign_recipients` as a trustworthy COUNT.

## Dependencies
- VT-170 (customers + campaign_recipients + resolve fn) — must land first.
- Sequencing: VT-241 before VT-240's VT-43 lift (else cohort_size reads an empty table → false 0).

## Out of scope
- The table/FK/resolve primitive (VT-170).
- VT-43 lift (VT-240).
