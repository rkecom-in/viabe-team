# VT-276 — imported_transactions raw-import surface

The two-surface transaction model (CC systemic-fork proposal; Cowork APPROVED +
plan-reviewed 2026-06-01). The RAW import surface complementing the clean
attributed ledger (VT-273 customer_ledger_entries). Rostered after the fact —
the plan-approved substrate that had no row (Cowork merge-signal 07:51, "allocate
a VT-ID … don't fold under VT-275 — that's the separate bridge").

POS/UPI imports routinely have NO resolvable customer (no phone/name) and need a
stable per-row idempotency key (provider_ref) the clean ledger can't hold. So:
- `imported_transactions` (migration 062): every imported row lands here,
  idempotent on UNIQUE(tenant_id, source, provider_ref); customer_id NULLABLE
  composite-FK (MATCH SIMPLE); direction CHECK(credit|debit); FORCE RLS.
- `customer_ledger_entries` stays the clean attributed ledger; ATTRIBUTED credits
  ALSO write a ledger row via the single-source ledger.record_ledger_entries.

Cowork plan notes implemented + canary-asserted:
- N1 (refunds): direction=debit RETAINED raw, NOT promoted (ledger has no refund
  type yet — follow-up). Methods retain refunds.
- N2 (no double-count): ledger write fires ONLY for rows newly inserted this call;
  re-import no-ops BOTH surfaces. Ledger entry_key idempotency is the second net.

Canary (DR-15, real Postgres, no mock cursors), 9 passed: attributed-credit→both,
unattributed→imported-only, idempotent-both-surfaces (N2), attributed-debit-
retained (N1), cross-tenant isolation. mypy --strict clean.

Enables: VT-57 (UPI) / VT-58 unattributed-extend / VT-59 (cash-book) route through
record_imported_transactions; VT-275 match_transactions bridge reads this
WHERE customer_id IS NULL.
