CORRECTION (supersedes the owner_inputs substrate claim in DECISION 368387c2-cc5a-8180-be0b-d1e64e0366de — '90-day retention' was never decided by Fazal): Earlier this session Clau logged DECISION ...8180 stating the owner_inputs substrate was settled as 'new dedicated owner_inputs table, Option A, 90-day retention'. The '90-day retention' figure was NOT a Fazal decision — it was lifted from the Claude Code Step-0 option menu and written into a Decision entry as if ruled on. Treating an unmade Type 3 decision as settled. Fazal caught and rolled it back 2026-05-22. The Option-A 'new dedicated table' shape in ...8180 was also premature — substrate shape was undecided pending this ruling. AUTHORITATIVE owner_inputs DECISION (Fazal, Type 3, 2026-05-22) — this entry supersedes the substrate/retention content of ...8180: (1) owner_inputs stores STRUCTURED INTENT, not raw message bodies. The codebase's existing deliberate drop of Twilio's Body field is PRESERVED, not reversed. Table persists extracted structured fields — e.g. {intent, segment, occasion, captured_at, source_message_sid} — never verbatim owner text. (2) Rationale: DPDP purpose-limitation (store only what the stated purpose needs — the agent needs intent, not text); data minimization is the more defensible posture to a customer/regulator; keeps the existing no-raw-body architecture intact. (3) RETENTION: because only derived structured fields are stored — not conversational content — the 90-day question dissolves. owner_inputs rows are operational data tied to an active subscriber relationship; retain for the LIFETIME OF THE TENANT RELATIONSHIP; purge on tenant offboarding / DSR. NO 90-day timer. (A 90-day timer would have been required only if raw bodies were stored — a further reason not to store them.) (4) PRIVACY-NOTICE LINE: owner-facing privacy policy must state, BEFORE owner_inputs ships, that the system records structured information derived from owner messages (intent, requested actions) to operate the service, and that raw message text is not retained. Exact wording is PENDING Fazal sign-off — owner_inputs must NOT ship until Fazal confirms the wording. (5) PRE-FLIGHT CHECK (mandatory, real): confirm WhatsApp Business Platform / Meta terms do not impose their own constraint on deriving-and-storing from message content, before the read-path brief executes. NEXT: Clau drafts the owner_inputs substrate + read-path brief on these terms — structured-intent table, lifetime-of-relationship retention, privacy-notice line pending Fazal wording sign-off, Meta-terms check at pre-flight.
