# VT-297 — Ops Console — Telegram VTR bot

## Why
Fazal: "all ops functions/analysis/actions can also be carried out by the VTR through
Telegram." A VTR is often mobile/away from the web console; Telegram gives parity-by-function.

## What
- Telegram bot exposing ops functions: alerts, quick-actions, simple queries.
- Confirmed split: alerts + quick-actions + simple queries on Telegram; deep analysis on
  web — but **every ops function reachable by some path** (parity-by-function, not
  pixel-parity).
- Actions route through the same orchestrator RLS/audit paths as the web console.

## Acceptance criteria
- VTR identity → assigned-tenant scope enforced on Telegram (RLS fail-closed; a VTR can't
  query/act on an unassigned business via the bot).
- A quick-action via Telegram writes through the same path + audit-log as the web action.
- PII de-identified in bot output (CL-426).
- (Bot token: sandbox/mock acceptable until the token lands — flag if needed.)

## Out of scope
- The autonomous watchdog→alert push (VT-298); web pages (VT-290..296).

## Notes
Build deferred behind launch queue. Reuses the existing Telegram MCP/bot plumbing if
present. One PR (CL-424).
