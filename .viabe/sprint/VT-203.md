# VT-203 — Ops Console login surface

## Why this row exists

VT-123 shipped the operator auth gate (`requireFazal()` reads `viabe_ops_jwt` cookie + verifies via OPERATOR_JWT_SECRET) and all 4 ops pages `catch (UnauthorizedError) { redirect('/login') }`. **But `/login` doesn't exist.** Operator cannot access any ops page without manually pasting a cookie via DevTools.

Surfaced 2026-05-27 ~23:55 IST when Fazal tried to open `/team/ops/stream` and got 307→404. Same gap blocks `/ops`, `/ops/tenants/[id]`, `/ops/runs/[id]`, `/ops/stream`.

## Scope

### Phase 1 — minimal credential-based login

For Phase 1 (Fazal as sole operator), simplest workable login:

- `apps/team-web/app/login/page.tsx` — React form: email + password
- `apps/team-web/app/api/auth/login/route.ts` — POST endpoint validates against env-stored credentials (`OPS_LOGIN_EMAIL` + `OPS_LOGIN_PASSWORD_HASH` via bcrypt), issues operator JWT via `issueOperatorJwt(FAZAL_OWNER_UUID)`, sets `viabe_ops_jwt` httpOnly cookie
- `apps/team-web/app/api/auth/logout/route.ts` — clears the cookie
- Logout button in ops layout header

**Cookie config:** httpOnly, Secure (prod) / not-Secure (dev), SameSite=Lax, Path=`/`, Max-Age = 1 hour (longer than the 5-min [resolve] TTL — operator session vs operator action are different scopes).

### Phase 2 deferred (not this row)

- Multi-operator support
- OAuth (Google / GitHub) login
- Magic-link email
- Per-operator subscription preferences

## Acceptance criteria

1. `/login` renders a form with email + password fields
2. Valid credentials → cookie set + redirect to `/team/ops`
3. Invalid credentials → 401 + error message rendered
4. `/api/auth/logout` clears the cookie + redirects to `/login`
5. Logout button in ops layout header works
6. Cookie config: httpOnly + Lax + correct Path
7. Operator session TTL = 1 hour; expired session → redirect to `/login`
8. Backward compat: existing 4 ops pages continue to work; redirect target `/login` now resolves

## Rule #15 Canary (mandatory)

`apps/team-web/tests/e2e/ops-login.spec.ts` — Playwright spec (uses existing VT-123 e2e harness):

1. Unauth GET /team/ops/stream → redirects to /login
2. POST /api/auth/login with valid creds → 200 + cookie set
3. POST /api/auth/login with wrong password → 401 + no cookie set
4. After login, GET /team/ops/stream → 200 + sticky banner rendered
5. POST /api/auth/logout → cookie cleared + subsequent GET redirects to /login
6. Expired JWT (TTL=1s test fixture) → next request redirects to /login

## Env additions

`.viabe/secrets/supabase-dev.env` needs 2 new vars Fazal drops:
- `OPS_LOGIN_EMAIL=fazal@rkecom.in` (or whichever)
- `OPS_LOGIN_PASSWORD_HASH=<bcrypt hash of chosen password>` (generate via `node -e "console.log(require('bcrypt').hashSync('your-password', 10))"`)

## Out of scope

- Password reset flow (Phase 1 = email Fazal directly if forgotten)
- Account creation surface (single-operator Phase 1)
- 2FA / TOTP (defer to multi-operator phase)
- Rate limiting on login (defer; low-volume internal surface)
