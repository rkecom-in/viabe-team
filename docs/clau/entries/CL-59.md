# CL-59 — Next.js upgraded 15 → 16 NOW before more code is written

Next.js upgraded from 15 to 16 NOW (Day 3 / May 16-17 2026), before more application code is written.

Reason: Per Fazal directive 'don't think of deferring anything to later phases, as that becomes impossible to switch.' Upgrading now while apps/team-web is essentially empty (scaffold only, no routes, no business logic) costs hours; upgrading later, after Sprint 7-8 builds out the owner surface, would cost days.

Next.js 16 brings:
- React 19.2 features: View Transitions, useEffectEvent, Activity component
- Async page props API (PageProps<'/route/[slug]'> helper for type-safe params/searchParams)
- Continued fetch-cache-default-off behavior (carried from 15)
- Vercel's current optimization target (Vercel ships Next.js 16-specific runtime features)

Breaking changes from 15 → 16 in our context:
- Page props (params, searchParams) now must be awaited: `await props.params` instead of accessing directly
- React 19.2 imports may shift slightly
- No major API surface breakage; mostly tightening of async semantics

No existing pages in apps/team-web to migrate — scaffold currently has only:
- app/(marketing)/team/page.tsx (placeholder Team product landing)
- app/(app)/ route group structure (no routes yet)
- app/api/team/health/route.ts (GET handler shipped in VT-Foundation)

Action items (next Claude Code task):
1. Bump next from 15.x to 16.x in apps/team-web/package.json
2. Bump react and react-dom to 19.2.x
3. Bump @types/react and @types/node to compatible versions
4. Update health route handler if any signature changes
5. Run pnpm install at root, verify build, verify lint+typecheck pass in CI
6. Open small PR (feat(team-web): upgrade Next.js 15 → 16 + React 19.2)

Low-risk change because scaffold has no business logic. Estimated 30-60 min Claude Code work.
