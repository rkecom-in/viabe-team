## What CoderC must produce
Owner-portal-specific hardening on top of the existing Twilio webhook. VT-3.3 wired the basics (signature verification + tenant lookup + idempotency on `MessageSid`). This subtask adds: replay-attack defense, signature-verification edge cases, rate-limiting beyond per-tenant, structured PII redaction in logs, observability metrics for ops.
1. **Replay attack defense**:
	- Twilio webhooks include the `X-Twilio-Signature` header but no nonce. An attacker who captures a real webhook (e.g., via compromised log) could replay it. Defense: `MessageSid` idempotency (already in VT-3.3) catches this within the dedup window, but not after expiry.
	- Add: a 24-hour rolling window of seen `MessageSid` values stored in the existing `send_idempotency_keys` table (or a new `inbound_message_ids` table; migration `028_inbound_message_replay.sql`). Replays after the expiry window return 200 + log + drop, never re-process.
2. **Signature verification edge cases**:
	- URL canonicalization: Twilio's signature is computed against the EXACT URL Twilio called. Differences in trailing slash, query string ordering, or proxy header rewriting break verification. Document the canonical URL form in `docs/runbooks/twilio-webhook.md` and add a verification function that handles common proxy variations (X-Forwarded-Proto, X-Forwarded-Host) before computing the expected signature.
	- Body parsing: Twilio sends `application/x-www-form-urlencoded`. Empty-body or oversized-body (>4096 chars total) requests rejected with 403 before signature verification (signature on empty body would be valid but useless).
3. **Rate limiting beyond per-tenant**:
	- Per-sender (E.164 phone) rate limit: 60 inbound messages per hour per sender. Excess returns 200 + log + drop. Prevents a malicious owner-equivalent sending floods.
	- Per-Twilio-sender-id (the tenant's WhatsApp business number) rate limit: 5000/hour. Catastrophic protection.
	- Both rate limits stored in Postgres (`rate_limit_counters` table from VT-2.3) with sliding-window counters.
4. **PII redaction in logs**:
	- Webhook log payload retains `From` (sender phone) and `Body` (message text). Both potentially PII.
	- Phone in logs: stored as E.164 hash (matching VT-7.1 KG hash discipline). Plaintext phone never appears in `pipeline_log.payload`.
	- Body in logs: hashed for log; full body lives in a separate `inbound_message_bodies` table (migration `029_inbound_message_bodies.sql`) with restricted-access RLS (service-role only). Engineers debugging an incident can join via message_id; ops has access; agent does not.
	- This makes `pipeline_log` itself fully PII-free.
5. **Observability metrics**:
	- Counter: inbound webhooks per minute, per Twilio sender ID. Surfaced via VT-Observability VT-12.3 dashboard.
	- Counter: signature verification failures per minute. Spike triggers alert.
	- Counter: replay-attack drops per hour. Spike triggers alert.
	- Gauge: webhook processing latency p50/p95/p99. Used for SLA monitoring.
6. **Tests** at `apps/team/tests/owner_surface/test_webhook_hardening.py`:
	- Replay: synthetic webhook with valid signature but seen `MessageSid` from 25 hours ago → 200, dropped, log event.
	- URL canonicalization: synthetic request with X-Forwarded-Proto rewrite → signature verification still passes.
	- Empty body: 403 before signature check.
	- Oversized body: 403 before signature check.
	- Rate limit per-sender: 61st message in same hour → dropped.
	- Rate limit per-sender-id: 5001st in hour → dropped.
	- PII redaction: verify `pipeline_log` row contains hashed phone + hashed body, not plaintext.
	- Body retrieval: synthetic ops query joins `inbound_message_bodies` via message_id; works.
	- Cross-tenant: messages routed to wrong tenant's `inbound_message_bodies` rejected (table tenant-scoped).
## Architectural rules to enforce in this subtask
- Pillar 7: signature verification is structural; no bypass paths, no "sandbox mode" override. CodeX rejects PRs adding either.
- Pillar 8: hardening is layered onto VT-3.3, not a parallel handler. The webhook is one function; the hardening adds checks in order (signature → replay → rate limit → process).
- All rate limits config-driven (`apps/team/config/webhook_rate_limits.yaml`).
- PII redaction is non-negotiable. CodeX rejects PRs that log plaintext phone or message body to `pipeline_log.payload`.
- 24-hour replay window matches Twilio's documented webhook retry window (Twilio retries up to 24h on 5xx); replays after that are unambiguously suspicious.
- Body-storage table is tenant-scoped + RLS service-role-only. Ops queries route through service role.
