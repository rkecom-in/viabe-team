## What CoderC must produce
The `get_business_profile` MCP tool. Returns the tenant's business identity — the persistent facts the agent needs at every invocation.
1. **Tool implementation** at `apps/team/src/agent/tools/get_business_profile.py`:
	Input schema: empty (no parameters; the tool returns the calling tenant's profile, period).
	Output schema:
	- `business_name: str`
	- `business_type: str` (free-form; tenant declares at sign-up; common values: cafe, restaurant, qsr, kirana, salon, gym, retail, electronics_shop, etc.; the agent reasons over this label)
	- `owner_name: str`
	- `owner_whatsapp: str` — included in the profile because the owner-side surface needs it; NOT customer phone numbers
	- `locality: str | None` — free-form (e.g., "Bandra West, Mumbai")
	- `city: str | None`
	- `state: str | None`
	- `pincode: str | None`
	- `coordinates: tuple[float, float] | None` (lat, lng) — if captured during onboarding; many tenants won't have it
	- `operating_hours: dict[Literal['mon'|'tue'|'wed'|'thu'|'fri'|'sat'|'sun'], OperatingHours | None]` — each day either has open/close times or is None (closed)
	- `preferred_language: Literal['en', 'hi']`
	- `phase: Literal['onboarding', 'trial', 'trial_extended', 'paid_active', 'paid_at_risk', 'cancelled', 'refunded']`
	- `plan_tier: Literal['founding', 'standard', 'pro']`
	- `signed_up_at: datetime`
	- `paid_conversion_at: datetime | None`
	- `meta: ProfileMeta` — `kg_data_freshness` (when L1 was last updated for this tenant), `fields_with_uncertain_confidence: list[str]` (any fields where confidence at ingestion was 0.7-0.85 and the owner was notified but didn't correct).
2. **Implementation**:
	- Phase 1 (interim): SQL JOIN over `tenants` + `business_profile_facts` view.
	- Post-VT-7: query Apache AGE graph via Cypher; refactor wrapped behind the same tool interface so the agent doesn't notice the change.
	- Cache: tenant profile rarely changes; cache for 60 seconds in-process. Cache invalidation on `tenants` UPDATE event (orchestrator phase transition or sign-up flow update).
3. **Tenant scoping**:
	- `ToolContext.tenant_id` is the only way the tool resolves which profile to return. No input parameter.
	- Cross-tenant attack test: ctx tenant_id substitution.
4. **Tests** at `apps/team/tests/agent/tools/test_get_business_profile.py`:
	- Happy path: synthetic tenant returns expected profile.
	- Cross-tenant attack: profile of tenant A cannot be returned when ctx.tenant_id = tenant_B.
	- Cache: two consecutive calls return same data; underlying DB is hit only once.
	- Cache invalidation: simulated `tenants` UPDATE invalidates cache; next call re-reads.
	- Missing optional fields: tenant without coordinates / pincode returns null for those fields, not an error.
	- Phase mirror correctness: returned `phase` matches `tenants.phase` after a transition (VT-3.2).

## Architectural rules to enforce in this subtask
- Pillar 2: zero LLM. Pure read.
- Pillar 4: returns stored facts only. CodeX rejects inferred fields ("likely target demographic", "competitor density") — those are agent reasoning or come from L1 KG enrichment, not from this tool.
- The output schema is stable. Adding new fields is Type 1; removing or changing semantics is Type 2.
- Cache invalidation must be observable. CI test confirms updates are picked up within 60s.
- The eventual KG-backed implementation must satisfy the same interface. Refactor PR (post-VT-7) does not change the schema.

## Acceptance criteria

- Cross-tenant attack test passes.
- Cache test confirms 60s window + invalidation on update.
- Returns expected profile for synthetic tenants; missing optional fields handled.
