# VT-330 — VT-89 webhook hardening (HARD GATE before LIVE Razorpay)

VT-89 (#335) financial-subagent nits, all touching the refund-feeding fees counter or
the verify seam. HARD GATE before LIVE Razorpay (alongside VT-93-N1 + VT-329 on
VT-89.md). #1 + #2 are the priority — they MUST land before LIVE keys.

## Scope
1. **Poison-pill 500-loop (priority):** `api/razorpay_ingress.py:_amount_paise` does
   `int(amount or 0)`; a NON-integer `amount` raises ValueError INSIDE the dedup txn ->
   500 -> Razorpay retries the SAME bad payload forever (the dedup row rolls back each
   time, so it never dedups). Add a parse-failure path: record-and-drop (or dead-letter
   to a `razorpay_webhook_dead_letter` table) + Fazal alert, instead of a 500-loop.
2. **Silent under-count (priority):** a `subscription.charged` resolving `amount==0` adds
   0 + returns "fees_incremented" with NO signal. cumulative_fees_paid_paise drives
   refunds (VT-93), so log + Fazal-alert on a charged event with amount==0 (a real charge
   is never 0 — it signals a payload/parse problem -> under-refund risk).
3. **Route<->HMAC end-to-end test (REAL crypto):** the route test mocks
   verifyRazorpaySignature; the Q1-502 test mocks the forward. Add an e2e test asserting
   the raw bytes survive `req.text()` -> verify intact (real HMAC), and a real
   orchestrator-down -> 502.
4. **UPDATE rowcount guard:** the fees/counter UPDATEs assume the subscription row
   exists; guard on `cur.rowcount` (defense-in-depth; subscription-row-deleted race).
5. **plans.yaml price-consistency unit test (VT-331 #337 subagent):** assert
   `resolve_plan("founding").amount_paise == 249900` (+ standard 499900 / pro 1499900).
   Nothing today catches a typo in plans.yaml; a wrong price can't charge until LIVE (env
   plan IDs NEEDS-FAZAL) and VT-330 lands before LIVE, so this is the right home.
6. **price-literal gate self-test (VT-331 #337 subagent):** a CI step that plants a
   synthetic plan-price literal and asserts `gate-no-price-literals` FIRES — regression-
   protects the gate's own regex.
7. **payment_method_last_four population (VT-91 #339, CL-390):** the VT-89 webhook extracts
   ONLY `card.last4` (4 digits, display) from the Razorpay payload and writes
   `subscriptions.payment_method_last_four` (mig 103, added by VT-91). The redaction must
   WHITELIST last4 surgically — NEVER relax the routing-only payload to admit PAN/CVV/expiry.
8. **founding_tier_counter CHECK should reference `cap` (VT-94 #341 subagent):** the
   `claimed_count <= 100` CHECK (mig 104) hardcodes 100; if `cap` is ever raised (Type-3),
   the literal CHECK silently errors. Change it to `claimed_count <= cap` in the next
   migration that touches the table. Low priority.

## Canary
Poison-pill (amount="abc") -> recorded-and-dropped, NO 500-loop, Fazal alerted.
charged amount==0 -> fees unchanged + Fazal alert. Real-HMAC route e2e. rowcount-0 handled.
