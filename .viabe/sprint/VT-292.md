# VT-292 — Ops Console — Escalations

## Why
Escalations are the VTR's core workload (CL-426 decaying human-on-the-loop). The queue
must let a VTR resolve/override/reassign inline without leaving the listing.

## What
- Listing: escalation queue with inline filters + inline actions (resolve / override /
  reassign). Deep view → overlay.
- Every action writes through the orchestrator's RLS/audit paths (no back-door); audit-logged (Pillar 7).

## Acceptance criteria
- VTR sees only assigned-tenant escalations (RLS fail-closed; real-PG canary).
- An override/resolve action writes + is audit-logged (canary asserts the audit row).
- Reassign respects VTAdmin-vs-VTR permissions.
- PII de-identified (CL-426).

## Out of scope
- VTR↔business assignment management (VT-295, VTAdmin); behaviour training (VT-294).

## Notes
Build deferred behind launch queue. Feeds the escalation-rate decay metric (VT-282). One PR (CL-424).
