# CL-249 — DECISION: sales-recovery agent built on the anthropic Messages SDK (pure Python,

VT-32 Notion page (dated 2026-05-04) says 'Anthropic Agent SDK'. Context7 surfaced that a literally-named package, claude-agent-sdk (pip install claude-agent-sdk), now exists — but it did not exist when the spec was written, so the spec phrase is generic-descriptive ('build on Anthropic's SDK'), not a product reference. claude-agent-sdk REJECTED because: (a) it bundles the Claude Code CLI and requires a Node runtime on the image — a real infra expansion for our Railway-hosted Python orchestrator service, taken on to honor a year-old terminology guess; (b) it is designed for coding/CLI-driving agents, not a context-bundle-reasoning + MCP-tool specialist; (c) using it would put the agent loop inside vendored code we do not control. CHOSEN: anthropic Messages SDK — already in pyproject, pure Python, no Node, no CLI bundle, no Railway image change; max_tokens + max_thinking_tokens are native Messages params. We write the agent loop ourselves. This is REQUIRED for clean VT-35: the four hard-limit enforcers (token meter / tool counter / depth tracker / wall-clock) instrument the think->tool->think loop directly, satisfying Pillar 1 (orchestrator measures) by attaching to OUR code rather than a vendor hook abstraction. CONSEQUENCES: (1) VT-32 must pin the currently-unpinned anthropic dep (pre-existing tech debt, fixed in passing). (2) VT-35 enforcer attach-points change — no claude-agent-sdk HookMatcher; enforcers attach via streaming-response events + a custom tool-dispatch wrapper around our loop. Held VT-35 brief to be revised accordingly when VT-35 unblocks. Decision authority: SDK/library choice within the locked stack + code architecture = Clau owns; Type 2, recommended strongly to Fazal, standing-maintenance implication (we own the loop) flagged to Fazal.
