# VT-325 — platform_listings per-listing SOURCE

Fazal-authorized 2026-06-05 ("roster the platform_listings source and implement it
once CC is done with all in-progress tasks"). VT-308 was blocked on a stale VT-6
pointer — VT-6 shipped the AGGREGATE `business_profile` (one `l1_entities` row per
tenant, all platform context MERGED into one JSONB), never a per-listing source.
This row builds the missing source: a `platform_listings` table + wrapper + the
ingestion writer + the transactional-outbox emit that fires the EXISTING (dormant)
`_h_platform_listing_updated` KG consumer.

PLAN-FIRST (net-new): plan-ready sent to Cowork; no build until reviewed.

## Scope (the SOURCE only — VT-308 owns the consumer-side HAS_THEME logic)
1. **Table** `platform_listings` (new hot table): RLS ENABLE + FORCE, tenant-scoped
   policies on `app_current_tenant()` (like 045_customers). Migration via the
   allocator. One row per (tenant, platform, external_listing_id).
2. **Wrapper** `PlatformListingsWrapper(TenantScopedTable)` from day one (VT-306
   pattern) — `_table="platform_listings"`, an `upsert(conn=...)` that composes
   atomically with the emit, reads validated via `_validate`.
3. **Ingestion writer**: extend the in-orchestrator apify method path
   (`apify_gbp.ingest_gbp`, `apify_food.ingest_swiggy/zomato`) to ALSO write the
   per-listing row (in addition to the existing aggregate business_profile write).
   The ingestion-worker is a SystemExit stub — NOT needed; ride the apify path.
4. **Emit**: on listing upsert, `emit_kg_event(conn, "platform_listing_updated",
   tenant_id, {listing_id, platform, rating})` in the SAME txn (atomic), then
   `drain_kg_events(tenant_id)` post-commit. CL-390: PII-free payload.
5. **Canary** (Rule #15, real PG, CL-422 synthetic): per-listing upsert → emit →
   drain → assert the KG projection (PLATFORM_LISTING node + HAS_LISTING edge via
   the existing `_h_platform_listing_updated`) lands; + cross-tenant negative.

## Boundary vs VT-308
VT-325 = SOURCE (write the listing row, emit `platform_listing_updated`). VT-308 =
CONSUMER (extend `_h_platform_listing_updated` to add HAS_THEME edges). No overlap:
VT-325 emits the minimal per-listing fact; VT-308 reads the listing / extracts
themes on consume. VT-308.blocked_by repointed [VT-6]→[VT-325].

## Done when
Table + wrapper + writer + emit land; canary green; gate EXIT 0 (new table is
wrapper-only, not a new direct-SQL site); Cowork review → Pillar-7. VT-308 unblocks.
