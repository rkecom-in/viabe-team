## Contract (Cowork 2026-06-24, from the VT-412 PR-D audit — the deferred live-stream gap). PLAN-FIRST → Cowork gate.

`apps/team-web/app/(app)/team/ops/stream/page.tsx` is a **direct browser Supabase Realtime subscription to RAW
`pipeline_steps`**. The mig-030 RLS policy admits ANY operator claim to **ALL tenants' raw, un-projected rows**
(the real hole — an over-permissive grant). VT-412 PR-D (#497) did NOT open this surface; it CONTAINS the gap
(a VTR sees a role-restricted notice; the Realtime operator JWT is minted ONLY on the VTAdmin/Fazal branch).
This row builds the proper fix so VTRs can have a live feed safely.

## Fix (plan-first)
1. **Server-side SSE proxy** — replace the direct browser Realtime sub for VTRs with a server endpoint:
   service-role read of `pipeline_steps` → **de-identify** (the `deIdentifyStepForVtr` / `vtr_step_timeline`
   projection — keys-only, no decision_rationale/error/tool_calls, no raw customer name/phone/body) →
   **per-operator assignment filter** (only the VTR's assigned tenants) → stream to the client. VTAdmin/Fazal
   may keep the raw firehose.
2. **Tighten mig-030 raw-stream RLS** — the policy must NOT admit a generic operator claim to all tenants' raw
   rows. Scope it (assignment-aware) or restrict raw access to the service role + VTAdmin. Allocate the
   migration via `scripts/migration_id_allocate.py` (never hand-pick).
3. **Reconcile** with the existing run-replay de-id path (VT-412 PR-D) — reuse the same projection, don't fork.

## Gate
RISK (auth + PII + RLS + migration). **Plan-first → Cowork's gate**, Pillar-7 Fazal-authorized merge. NOT
self-merge. Surfacing the mig-030 RLS over-grant to Fazal.
