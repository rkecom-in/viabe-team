# CL-26 — L0 memory tier added: workspace-level operational memory for the orchestrator-ag

Concept doc defines four knowledge layers L1-L4 all tenant-scoped. Orchestrator-as-agent needs a fifth tier above them: L0 workspace-level operational memory. Contents: cross-tenant trigger-pattern memory ('weekly_approval replies at 7am Mondays are typically APPROVE; replies at 11pm Sundays are typically SKIP'); known operational states across all tenants ('tenants in their day-39 window need extra attention'); orchestrator's own reasoning history about routing decisions; aggregated patterns about specialist invocations that worked vs failed. L0 is k-anonymized aggregate; never contains tenant-identifying data. Storage: Postgres + pgvector, same substrate as L2 episodic. Retention: matches L3 cross-tenant patterns. The orchestrator's context bundle at each invocation = L0 (always) + relevant slices of current tenant's L1-L4 (composed at invocation time).
