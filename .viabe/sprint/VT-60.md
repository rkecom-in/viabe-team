## What CoderC must produce
The Method 6 ingestion adapter, ledger half. The privacy half (consent capture, opt-out flow, DSR) lives in VT-8.5. This subtask is the post-consent ledger write: once a customer scans the tenant's QR, accepts the terms, and provides their phone number, this method writes them to the `customers` table.
1. **Adapter module** at `apps/team/src/ingestion/methods/qr_optin.py`:
	- `ingest_qr_optin(tenant_id, customer_phone, customer_name, consent_record_id, run_id) -> IngestionSummary`.
	- Called by VT-8.5's consent-capture endpoint AFTER consent is verified.
	- `consent_record_id` is a foreign key to the consent record (in VT-8.5's `consent_records` table). The customer row stores this for audit / DSR.
2. **Customer record write**:
	- `acquired_via='qr_opt_in'`.
	- `consent_record_id` populated.
	- `last_visit_date` = consent timestamp (proxies for "first interaction").
	- `total_spend_paise` = null.
	- Default `opt_out_status=null` (customer is opted IN by definition; opt-out path is a separate flow in VT-8.5).
3. **Dedup via VT-6.3**:
	- Customer may already exist (e.g., previously imported from contacts or paper book). Dedup runs; if matched, update existing row's `consent_record_id` (multi-source) and `acquired_via_history` append.
	- If new: create.
4. **Welcome message** (post-opt-in):
	- Send free-form WhatsApp message via VT-5.6: "Thanks for opting in to [Business Name]'s updates. We'll only send you offers you'll find useful, no spam. Reply STOP to opt out anytime."
	- This is within the 24h Twilio window because the customer JUST scanned the QR and presumably initiated the conversation (the QR landing page sends a [wa.me](http://wa.me) link that opens a chat).
5. **Tests** at `apps/team/tests/ingestion/methods/test_qr_optin.py`:
	- Happy path: synthetic consent_record + new customer → customer created, welcome message sent.
	- Existing customer: dedup matches → acquired_via_history appended, consent_record_id updated.
	- Missing consent_record_id: returns error (privacy gate; this method must NEVER write without consent).
	- Cross-tenant: tenant_A consent_record cannot create tenant_B customer.
	- Welcome message failure: row written, welcome flagged for retry by DeliveryBot-equivalent (VT-OpenClaw equivalent doesn't exist for Team; instead, retry via scheduled trigger in VT-3.5).
	- Opt-out timing: a customer who opts in then immediately opts out (within minutes) — row written, `opt_out_status` set by separate opt-out flow.

## Architectural rules to enforce in this subtask
- Pillar 7: this method is downstream of consent. CodeX rejects PRs that bypass `consent_record_id` requirement.
- Pillar 8: thin adapter; consent capture lives in VT-8.5.
- The welcome message is REQUIRED to make the relationship explicit. Skipping it would be a soft-consent dark pattern.
- Tenant scoping: consent_record → customer → tenant chain enforced.
- `acquired_via='qr_opt_in'` is distinct from other methods and lets analytics differentiate consent-explicit customers from passively-imported ones.

## Acceptance criteria

- Missing consent_record_id is a hard error.
- Dedup integration with `acquired_via_history` append.
- Welcome message sent via VT-5.6.

## Out of scope
- Consent capture form UX (VT-OwnerSurface for owner-side; consent landing page for customer-side is a small mobile web view; lives in VT-8.5).
- Opt-out handling (VT-8.5).
- DSR APIs (VT-8.6).
- QR generation per tenant (VT-8.5 or VT-OwnerSurface).
- Multi-tenant QR sharing (out of scope; each tenant has their own QR).
- Repeat-scan handling (a customer scanning the same QR twice = idempotent; existing customer is updated, not duplicated).
