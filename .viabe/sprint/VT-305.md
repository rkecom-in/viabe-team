# VT-305 — wire nightly PII-in-log sweep (DBOS.scheduled) + alert route

## Why split (not done in VT-79)
Registering a new `@DBOS.scheduled` handler shifts DBOS `app_version`
(recovery-sensitive — see `scheduled_triggers.register_scheduled_triggers`
docstring) and deserves its own scheduler canary. `detect_pii_in_logs` is
runnable on demand meanwhile (not stale).

## Scope
1. Add a nightly handler (off-peak IST) sweeping `detect_pii_in_logs` across
   `all_active_tenant_ids` + `dispatch_alert` each `pii_in_log` trigger (the
   existing VT-202 dispatch path — Telegram/email, PII-scrubbed).
2. Register it (the alerts scheduler `register_alert_scheduler` or
   `scheduled_triggers`, whichever owns the nightly cron) — EXTEND, don't fork.
3. Scheduler canary: synthetic clock fires the sweep; a seeded unredacted-PII
   step → alert dispatched; clean → none; existing handlers' recovery unaffected.
