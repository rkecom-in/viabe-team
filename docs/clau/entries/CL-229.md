# CL-229 — State Snapshot template locked: 5 fixed fields (Critical Path / In Flight / Bloc

## Problem

Fresh sessions ran the start ritual but didn't own the what/how/when. They hedged, asked Fazal what to work on, bounced back Clau-owned decisions. Root cause: a semantically-ranked pile of log entries is raw material, not a briefing. The synthesis has to be pre-written in ONE place — the State Snapshot.

## Fix

Every State Snapshot now follows a fixed 5-field template (full spec in Resurrection File v2.23):

- CRITICAL PATH: the single thing that matters now, one sentence
- IN FLIGHT: what's mid-execution — subtask, branch, PR#, concrete state
- BLOCKED ON: the specific gate + who owns clearing it
- NEXT ACTION: the literal next move + whose move it is
- DO NOT: traps a fresh session might wrongly re-open

Entry title format: 'STATE SNAPSHOT <date>: <one-line critical path>'.

## Ritual change

SESSION-START RITUAL step 1 now fetches the latest State Snapshot FIRST (query 'state snapshot', recent date range, newest). That snapshot is the briefing. The broader Standing/Open search is supplementary. A session opening with a structured snapshot states the critical path and proposes the next action instead of asking what to work on.

## Lifecycle rule

Exactly one State Snapshot is Status=Standing at a time — the latest. When a new session writes its own snapshot, the prior one flips to Superseded. Keeps session-start search returning exactly one current snapshot.

## Fazal action

Replace SESSION-START RITUAL step 1 in Layer 1 Project Instructions with the snapshot-first version (provided alongside this log entry).
