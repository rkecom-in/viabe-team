# VT-170 — customers table + tenant customer registry (ACTIVATED, expanded)

## Status note (2026-05-30)
**Activated + expanded by Fazal directive:** "None of our actions can complete without a customers table. As our agents are managing business actions, business actions can't complete without a defined customers dataset." Re-scoped from the original thin redactor-name-registry to the **foundational customer entity** the SR-Agent depends on, **with cohort integrity**. Priority Medium→Critical. Migration number corrected 023→**045** (023 taken by attributions; latest is 044).

## Why now
The customers table is the substrate multiple rows are blocked on:
- **VT-44** send_whatsapp_message needs `customers.last_inbound_at` for the 24-hour WhatsApp window.
- **Consent gating** — agents must not message opted-out customers (`opt_out_status`).
- **VT-43** get_attribution_data cohort_size/attribution_rate: campaigns store the cohort as `target_cohort.customer_ids` (mig 018), but those IDs reference customers that don't exist as rows. This table + cohort integrity make cohort_size/rate trustworthy (currently degraded to None in PR #144 — Option A).
- **VT-104** redactor `name_registry` callable (shipped inert) gets a backing store.

## Scope (activated — Foundational + cohort integrity)

1. **Migration `045_customers.sql`**:
   - `customers (id UUID PK, tenant_id UUID NOT NULL FK→tenants(id) ON DELETE CASCADE, display_name TEXT, phone_e164 TEXT NULL, email TEXT NULL, last_inbound_at TIMESTAMPTZ NULL, opt_out_status TEXT NOT NULL DEFAULT 'subscribed' CHECK (opt_out_status IN ('subscribed','opted_out','blocked')), source TEXT NULL, created_at TIMESTAMPTZ DEFAULT now(), updated_at TIMESTAMPTZ DEFAULT now())`
   - Partial unique indexes: `(tenant_id, phone_e164) WHERE phone_e164 IS NOT NULL`, `(tenant_id, email) WHERE email IS NOT NULL`.
   - Index `(tenant_id, last_inbound_at)` for the 24h-window lookup (VT-44).
   - RLS: `ENABLE` + `FORCE ROW LEVEL SECURITY` + 4 tenant-scoped policies keyed `tenant_id = app_current_tenant()` (mirror 016/023/044 template).

2. **Cohort integrity** (Fazal: "cohort integrity needs to be in place"):
   - Make `campaigns.target_cohort.customer_ids` referentially sound against `customers.id` (same tenant). CC proposes the mechanism in the plan — options: write-time validation in the campaign-create path + a consistency-check function, OR a normalized `campaign_recipients(campaign_id, customer_id)` linkage with FKs. Pick the lower-risk one that makes VT-43 `cohort_size` a trustworthy COUNT of real, same-tenant cohort members and is joinable.
   - Do NOT silently drop unresolved/cross-tenant cohort ids — reject or surface them.
   - Out of scope: rewriting historical campaign cohorts; just establish the forward integrity path + a validation the consistency-check can run.

3. **Customer registry service** (`apps/team-orchestrator/src/orchestrator/privacy/customer_registry.py`):
   - `get_customer_names_for_tenant(tenant_id, db) -> list[str]` for the redactor bigram match.
   - In-process LRU cache + explicit invalidation on customers UPSERT.
   - Plumb into `redact_for_log` + `redact_for_langsmith` as the `name_registry` callable (currently None per VT-104).

4. **Tests**:
   - RLS isolation: tenant A cannot see tenant B customers.
   - Registry cache invalidates on UPSERT; redact() with registry redacts known names, unknown pass through.
   - Cohort integrity: a campaign cohort referencing a non-existent / cross-tenant customer id is rejected or flagged (not silently accepted).
   - opt_out_status default + transitions.
   - Cross-tenant attack on every path.

5. **Canary (Rule #15)**:
   - Real dev-DB insert of a customer + real `redact()` with the registry callable injected; verify redaction.
   - Real RLS cross-tenant denial with `app.current_tenant` set to another tenant.
   - Cohort integrity: a real campaign cohort referencing the inserted customer resolves; a bogus id is caught.
   - **CL-422 HARD CONSTRAINT: SYNTHETIC data ONLY on dev** — no real customer phones/emails/names. customers is tenant-identifying PII; real data waits for VT-231 (Mumbai prod). fail-not-skip; cleanup.
