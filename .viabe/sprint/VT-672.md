# VT-672 — one common `escalate` tool

## Gap (from `scripts/check_capability_gaps.py`, `KNOWN_CAPABILITY_GAPS[unified_escalate]`)
Escalation is duplicated per-lane — integration/finance/tech/accounting/sales/onboarding each own a
bespoke escalate. There is NO single common `escalate` tool a specialist reaches to hand a decision
back to the Manager/owner. Every specialist needs it; the framework's common-tool surface lacks it.

## Build
Add ONE common `escalate` tool (ADVISORY kind, no gated capability — it hands off, it doesn't act)
to the Manager-scoped common set (mirror `tools_common.COMMON_READ_TOOLS` shape). Catalog it (`_Ann`
in `tool_catalog.py`). Migrate the per-lane escalates to delegate to it (or deprecate them). Owner-
comms stays Manager-only (§1.2) — escalate returns to the Manager, never messages the owner directly.

## Gate / auto-close
Once `escalate` is on a cataloged surface, `open_capability_gaps()` drops this gap; the registry-
honesty test (`test_capability_gap_registry_is_honest`) then fails RED until the `unified_escalate`
entry is deleted from `KNOWN_CAPABILITY_GAPS` — do that in this PR. `check_capability_gaps.py` goes
green for this row. Deny-list + conformance suites stay green.
