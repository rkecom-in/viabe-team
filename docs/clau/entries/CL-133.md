# CL-133 — Orchestrator-agent system prompt describes only wired behaviors

Standing decision derived from CL-138 session close: orchestrator-agent system prompt only describes behaviors actually wired in the current PR. No aspirational tools, no unenforced limits.

This is a Standing discipline for all future iterations of orchestrator_agent_system.md (and any other system prompts we author). The prompt is honest about the agent's actual capabilities at any point in time. As tools are added (VT-3.4 spawn_sales_recovery wiring, future MCP tools), the prompt is extended in lockstep. As limits are enforced (VT-3.9 PR N+ hard-limit suite), the prompt mentions them.

The reverse — telling the model about tools it doesn't have or limits we don't enforce — leads to (a) the agent attempting calls that fail at runtime, (b) the agent declining work because it thinks a non-existent limit applies, (c) prompt drift where every edit becomes 'is this still aspirational?' instead of 'is this accurate?'

Related: CL-158 (placeholder disclosure design question for the spawn_sales_recovery + escalate_to_fazal Note in current prompt).
