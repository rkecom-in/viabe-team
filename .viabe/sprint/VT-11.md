## Why this parent exists
Landing is the conversion funnel. The product is invisible until launch; [viabe.ai/team](http://viabe.ai/team) is the first surface where prospective subscribers learn what Team is and decide to sign up. The page must do four jobs: explain the product in language Indian SMB owners understand (Hindi + English), capture sign-ups with a 4-field form, gate access in waitlist mode pre-launch and remove the gate at launch, and run the founding-tier counter widget that gives the page urgency. None of these are complex individually; together they are the only thing standing between traffic and revenue.
Reports product's landing was simpler — single CTA to a checkout page. Team's landing must support a trial-not-purchase flow, which requires a different conversion grammar. The waitlist mode + flag mechanism is a pre-launch necessity: marketing can warm the audience before the product surface is ready.
## What this parent owns
1. The [viabe.ai/team](http://viabe.ai/team) page itself: Hindi + English copy, hero, value props, founding-member proof, FAQ. Built in `apps/team-web/`. Reuses Reports' brand tokens where applicable; never imports Reports application code.
2. Sign-up form with 4 fields: business name, owner name, WhatsApp number, business type. No card capture (Pillar from VT-10).
3. Waitlist mode + flag mechanism: pre-launch the form captures email/WA only; post-launch the flag flips and the form provisions a tenant + starts the trial.
4. Social proof placeholders: testimonials, case studies, logos. Empty at launch (no real customers yet); structured so they can be populated as evidence accumulates.
5. Founding-member counter widget: live-reads the atomic counter from VT-10.6 and displays "X of 100 founding spots remaining." Updates in near real time without refresh.
6. A/B testing framework: cookie-based, no PII, supports headline/CTA/copy variants on the landing page. Conversion attributed by signup-form completion.
## Architectural rules binding every subtask
- Pillar 8 (no patchwork): copy strings live in i18n resource files, not inline JSX. Pricing values come from env vars (`STANDARD_PRICE_PAISE`, etc.), never hardcoded in component code.
- Pillar 3 (tenant isolation): the landing page is unauthenticated; sign-up form data is captured into a holding table, not into a tenant-scoped table. Tenant provisioning happens after server-side validation in VT-9.2 sign-up flow, not from the landing-page client.
- The waitlist flag is a single env var or feature flag read at request time. Flipping it must be reversible without redeploying.
- A/B testing cannot capture PII. Variant assignment uses an opaque cookie ID, not the user's email or phone. Conversions are attributed by the signup form, not by tracking pixels.
- Hindi and English are first-class. Neither is a translation of the other; both are authored. (Tamil/Telugu deferred per [concept-team.md](http://concept-team.md) §6.)
- The founding counter widget must read live from the atomic counter — not from a cached value that could lie. If the API call fails, the widget shows the last-known count with a freshness timestamp and degrades gracefully.
- No third-party tracking pixels (Facebook, GA, etc.) on the landing page until DPDPA review. Phase 1 launches with first-party analytics only via the existing observability stack.

## Definition of done
- All 6 subtasks Done.
- [viabe.ai/team](http://viabe.ai/team) renders correctly in both Hindi and English on mobile and desktop. Lighthouse score ≥ 90 on performance, accessibility, SEO.
- Sign-up form validates the 4 fields client-side and server-side. Submission writes to the holding table; a synthetic submission completes end-to-end.
- Waitlist flag flip: with flag ON, form captures email/WA only. With flag OFF, form provisions tenant + starts trial via VT-9.2. Both modes verified in dev.
- Founding counter widget: pulls live count from VT-10.6 atomic counter; degrades gracefully on API failure with last-known + timestamp.
- A/B framework: synthetic A/B test runs across two headline variants; conversion attribution captured without PII; framework supports adding new variants without redeploying.
- Brand tokens consistent with Reports' tokens; no application code imported from Reports apps.
