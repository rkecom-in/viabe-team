# VT-479 — Rich-media in-window — Twilio Content API buttons/lists/images once 24h window open

## Plan-first design (CC, 2026-06-28) — send-path = risk-adjacent, so plan before build

### Goal (Fazal directive)
Once an owner inbound has opened the 24h WhatsApp customer-care session window, our REPLIES
should be RICH — quick-reply BUTTONS (confirm Qs → Yes/No/Skip), list-pickers (enumerable gaps),
images, links/CTAs — not plain `Body=` text. Buttons also remove the free-text "yes" parse that
made the confirm flow brittle (the VT-477 / VT-478 UX root).

### ⟹ CORRECTED MODEL (Fazal via Cowork 141500Z) — in-window = FREE-FORM, NO Meta whitelisting
- **IN-WINDOW (≤24h since the tenant's last inbound):** ALL agent→tenant messages — onboarding Qs, Yes/No
  prompts, rich media — are FREE-FORM session messages. **NO Meta template, NO whitelisting.** (Dropped the
  earlier "3 Fazal-approved Content templates" framing.)
- **OUT-OF-WINDOW (>24h):** send ONE fixed Meta-approved re-engagement template ("are you available / ready
  for updates?"). The tenant's reply RE-OPENS the 24h window → back to free-form. This single re-engagement
  template is the ONLY Meta-approval ask in the whole flow.

### CANARY FINDING (Rule #15 — the exact Twilio mechanism, verified against Twilio docs, NOT asserted)
Cowork asked: can true tappable in-session buttons be sent as pure free-form, or do they need a Twilio
Content object? **Definitive answer from twilio.com/docs:**
- **In-session quick-reply buttons do NOT require WhatsApp/Meta template APPROVAL** — Twilio doc (twilio-quick-reply):
  "When you send an in-session WhatsApp message that does NOT require template approval, only **three buttons**
  are allowed" (vs up to 10 for an approved template). So buttons in-window = NO Meta approval. ✓ (matches Fazal's model)
- **BUT a Twilio Content OBJECT (HX content SID) IS still required** to render true tappable buttons. `messages.create`
  accepts EITHER `Body`/`MediaUrl` (plain text/media) OR `ContentSid` — there is **no inline-interactive-buttons
  parameter**. Interactive content types (`twilio/quick-reply`, `twilio/list-picker`, `twilio/card`) are deliverable
  ONLY by referencing a pre-created Content object's `ContentSid`. The Content object is created ONCE via the Twilio
  **Content API** (`POST /v1/Content`) — this is a Twilio-side registration, **NOT a Meta approval** and **NOT a
  whitelisting step**. (A Content object MAY be submitted for Meta approval for OUT-of-session use, but that's optional
  and separate; for in-session use it just needs to EXIST.)
- **So the two options for Fazal:**
  - (A) **Plain free-form text** ("Reply 1 for Yes / 2 for No") — ZERO setup, pure `Body=`, works today. No buttons.
  - (B) **True tappable buttons** — needs a one-time Twilio Content-object create per button-template (CC can do this
    via the Content API with the dev Twilio creds; no Meta approval, no whitelisting). The HX SIDs go in `.viabe/templates.md`.
- **Recommendation:** do (B) — CC creates the Content objects via the Content API on dev (one-time, no Fazal-Meta
  step), records the HX SIDs in the registry. The earlier "Fazal must create 3 approved templates" blocker was WRONG;
  the only Fazal-Meta ask in the whole flow is the SINGLE out-of-window re-engagement template.
- The inbound button reply arrives as a normal inbound webhook whose Body is the button's payload/title → the journey
  reads it as the answer (no free-text "yes" parse needed) — this is what kills the VT-477 brittleness.

### Window-state seam

   Record each `HX…` SID in `.viabe/templates.md` (NO hardcoded SIDs). The button PAYLOAD encodes the field+value.

   payload as the answer — REMOVES the "yes" token parse for confirms (the VT-477 brittleness fix).
3. Out-of-window → ONE re-engagement template; tenant reply re-opens the window → free-form again.
