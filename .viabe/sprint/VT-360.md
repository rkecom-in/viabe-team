# VT-360 — team-web VTR surface onto the DB-enforced de-identified views

Rostered by Cowork (20260607T131500Z, VT-280 plan-ack). VT-280 made CL-425 DB-enforced on the
ORCHESTRATOR/digest path (reads via `app_vtr_role` + the VT-281 views). This row completes the
guarantee on the **team-web ops surface**, which still uses app-side `maskForVtr` (lib/ops/de-
identify.ts) — and maskForVtr leaks the raw row id as REF# (the VT-281 subagent finding).

## Scope (>1h — its own row per anti-proliferation + the plan-ack's >1h clause)

  escalations page + escalations-list component) with reads through an **app_vtr_role** Supabase
  connection + the VT-281 views (`vtr_customers`, `vtr_escalations`).

- RETIRE maskForVtr (+ its raw-id REF#) when the call sites are migrated.
- team-web currently reads via the Supabase service-role (RLS-bypassing) — this is a real re-plumb
  (a separate app_vtr_role connection + routing the ops reads through it), which is WHY it's a
  separate row, not folded into VT-280.

## MULTI-VTR precondition (cross-note — same surface, same gate as VT-281's)
Phase-1 = Fazal is VTR#1 (sees all tenants). BEFORE a SECOND VTR exists, the views MUST be
assignment-scoped (a `vtr_assignments(vtr_id, tenant_id)` table + a `WHERE tenant_id IN (...)` on
each view, the app_current_tenant() GUC pattern). This row + that scoping are BOTH preconditions for
any second VTR (multi-VTR console = VT-189, post-launch).
