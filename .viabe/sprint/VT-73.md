# VT-73 — Agent context isolation (3rd layer)

## Parent task
VT-PrivacyArchitecture.

## Concept reference
- `concept-team.md` §11 (third isolation layer: agent context contains only one tenant's data; structural isolation, not just discipline).
- `concept-team-pillars.md` Pillar 3 (three independent layers each sufficient on its own).
- `concept-team-pillars.md` Pillar 8 (one composition layer enforces; ad-hoc context construction forbidden).

## Dependencies
- VT-SalesRecovery-Agent VT-4.3 Done (`SalesRecoveryContext` builder).
- VT-KnowledgeArchitecture VT-7.8 Done (composition layer).
- VT-PrivacyArchitecture VT-8.1 Done (typed wrappers; the structural defense agent context relies on).

## What CoderC must produce
The agent context isolation enforcement layer. The agent SDK's invocation receives a `SalesRecoveryContext` constructed from one tenant's data. This subtask adds the structural guarantees that make multi-tenant context contamination impossible: pre-flight validation, in-flight isolation, post-flight audit.
1. **Pre-flight context validation** at `apps/team/src/agent/context_validator.py`:
	- Function `validate_context_isolation(ctx: SalesRecoveryContext) -> None`. Called by `dispatch_to_specialist()` (VT-3.4) immediately before invoking the agent SDK.
	- Validates: every customer_id in `ctx.customer_ledger_summary` belongs to `ctx.tenant_id` (cross-checked via wrapper); every campaign_id in `ctx.recent_campaigns` belongs to `ctx.tenant_id`; every event in any L2 episodic data has `tenant_id == ctx.tenant_id`; every L4 document referenced (cross-tenant by design — these are global) is in the L4 allowlist.
	- Failure raises `ContextIsolationViolation` — critical severity, alerts Fazal, kills the dispatch.
2. **In-flight isolation**:
	- The agent SDK invocation runs in an asyncio Task with a context-local `tenant_id` that's set at dispatch entry and unset at exit. Any tool call within the agent that doesn't match the context-local `tenant_id` raises `ContextIsolationViolation`.
	- Implementation: `contextvars.ContextVar` named `current_tenant_id` is set by `dispatch_to_specialist()`. MCP tools (VT-5) read this at execute() entry; if their `ctx.tenant_id` doesn't match, hard error.
	- This is belt-and-braces over VT-8.1's wrappers: even if a tool somehow got a wrong `tenant_id`, the contextvar mismatch catches it.
3. **Post-flight audit** at `apps/team/src/agent/context_audit.py`:
	- After agent dispatch completes, scan the run's `pipeline_log` events: collect every `tenant_id` that appears in any event for this `run_id`. Assert all match `ctx.tenant_id`. Anomalies trigger `ContextIsolationViolation` post-hoc and a Fazal alert.
	- This catches violations that escaped pre-flight + in-flight (e.g., a logging path that wasn't covered).
4. **No-cross-tenant-prompt-content rule**:
	- The system prompt (VT-4.2) and the agent's reasoning never mention any tenant other than the current one. CodeX rejects PRs that add cross-tenant references to prompts.
	- L3 priors are the only cross-tenant data the agent receives — and they're aggregates (≥5 tenants), not identifiable. They have no `tenant_id` field; they're scoped by `cohort_key` only.
5. **Tests** at `apps/team/tests/privacy/test_context_isolation.py`:
	- Pre-flight: synthetic context with cross-tenant customer_id leak → `ContextIsolationViolation` before agent dispatch.
	- In-flight: synthetic tool call with mismatched `ctx.tenant_id` → tool raises immediately.
	- Post-flight: synthetic run with anomalous pipeline_log event → audit detects.
	- Three-layer fail-safe: simulate VT-8.1's wrapper bypassed somehow + RLS misconfigured → context isolation layer still catches the cross-tenant attempt.
	- L3 access: agent receives L3 prior with no tenant_id field → no violation, prior is consumed correctly.
	- Concurrent dispatches: two simultaneous dispatches for two different tenants → contextvar correctly scoped per asyncio Task; no cross-talk.

## Architectural rules to enforce in this subtask
- Pillar 3: this is the third of three independent layers. CodeX confirms by reviewing pre/in/post-flight independence.
- Pillar 8: one validator, one audit, one contextvar. CodeX rejects shadow validators.
- `ContextIsolationViolation` is critical-severity. Cannot be caught and retried.
- Pre-flight, in-flight, and post-flight checks are independent — passing one doesn't excuse failing another. CodeX confirms each runs unconditionally.
- L3 priors are explicitly cross-tenant and exempt; they have no tenant_id by design. CodeX confirms the validator's L3 exemption is narrow (only L3 patterns, not anything else).
- Concurrent dispatches must be safely scoped via `contextvars.ContextVar` (which is asyncio-Task-scoped). No globals.

## Acceptance criteria
- Pre-flight validator exists and is called from VT-3.4's dispatch.
- In-flight contextvar enforcement in every MCP tool's `execute()`.
- Post-flight audit exists and runs after every dispatch.
- All 6 test scenarios pass.
- Three-layer fail-safe test confirms isolation holds with the other two layers misconfigured.
- Concurrent dispatch test confirms asyncio Task scoping.
- Critical alert + log on violation.
- mypy --strict passes.
- PR title references `(VT-PrivacyArchitecture)`; reviewed by CoderX (mandatory).

## Out of scope
- Cross-tenant aggregations beyond L3 (Phase 2 if needed; Phase 1 is L3-only cross-tenant).
- Per-tenant agent-config customization (Phase 2).
- Multi-specialist dispatch isolation (Phase 2; Phase 1 has one specialist).
- Read-only "observation mode" for ops debugging (out of scope; debugging happens via pipeline_log + composition audit, not agent observation).
