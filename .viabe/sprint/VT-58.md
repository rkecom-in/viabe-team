# VT-58 — Method 4: KOT/POS export

## Parent task
VT-IntegrationInvention.

## Concept reference
- `concept-team.md` §5 (Method 4: KOT/POS export where it exists; for the ~30% of SMBs that do run a POS).
- `concept-team-pillars.md` Pillar 4 (POS exports are structured; deterministic parsing primary; vision fallback for printed-receipt photos).

## Dependencies
- VT-Foundation Done.
- VT-IntegrationInvention VT-6.1 Done (vision pipeline for printed-receipt photo path).
- VT-IntegrationInvention VT-6.3 Done (dedup).

## What CoderC must produce
The Method 4 ingestion adapter. Owner uploads a KOT (Kitchen Order Ticket) or POS export. Two flavors: (a) digital export from a POS system in CSV/JSON/XML format, (b) photograph of printed thermal-paper KOT receipts.
1. **Adapter module** at `apps/team/src/ingestion/methods/kot_pos.py`:
	- `ingest_kot_pos(tenant_id, file_bytes, file_format, run_id) -> IngestionSummary`.
	- `file_format: Literal['csv', 'json', 'xml', 'image_kot', 'auto']`. `auto` sniffs format from content-type header + first bytes.
2. **POS export parsing** (digital path):
	- Phase 1 supports the most common Indian POS systems: PetPooja, Posist, Slick (these are the SMB-favored POS vendors in India). Parser stubs for each in `apps/team/src/ingestion/methods/kot_parsers/`.
	- Each parser maps the vendor's schema to canonical fields: `transaction_date`, `total_amount_paise`, `customer_phone` (often null in POS — most POS doesn't capture phone), `customer_name` (often null), `items` (line-item array — stored as JSONB but not used for customer matching), `bill_number`.
	- Generic CSV/JSON fallback for unrecognized vendors: column-header sniffing finds amount + date columns; phone/name only matched if column names are explicit.
3. **KOT image parsing** (photograph path):
	- Calls `vision_extraction.extract_from_image()` with `KOTReceiptSchema`: `transaction_date`, `total_amount_paise`, `customer_phone` (rare on KOT receipts), `customer_name` (rare), `items`, `bill_number`.
	- Confidence routing per VT-6.1's standard thresholds.
4. **Customer linking**:
	- When phone or name is present: dedup via VT-6.3, link transaction to customer.
	- When BOTH absent (common in POS): transaction row created with `customer_id=NULL`. The transaction is unattributable until later matched by `match_transactions` (VT-5.8) against a UPI export or owner-asserted attribution.
5. **Idempotency**:
	- UNIQUE`(tenant_id, source, bill_number)` (migration 018) prevents re-import of the same POS export.
	- For KOT photos: less reliable; use `(tenant_id, source, transaction_date, total_amount_paise, bill_number)` as the dedup key, since photos may be retaken.
6. **Tests** at `apps/team/tests/ingestion/methods/test_kot_pos.py`:
	- PetPooja CSV parser: synthetic export → correct rows.
	- Posist JSON: synthetic → correct.
	- Slick XML: synthetic → correct.
	- Generic CSV with explicit phone column: rows linked to customers via dedup.
	- Generic CSV without phone column: rows committed with `customer_id=NULL`.
	- KOT image: synthetic → vision pipeline + extraction → row written.
	- Re-import idempotency: re-uploading same export → 0 new rows.
	- Cross-tenant: tenant_A POS export cannot create tenant_B rows.
	- PII absence in logs.

## Architectural rules to enforce in this subtask
- Pillar 4: deterministic parsing for digital exports; vision fallback only for image inputs.
- Pillar 8: per-vendor parsers share canonical output schema. Adding a new POS vendor is a new file under `kot_parsers/`, not a rewrite.
- Most POS exports lack customer phone; transactions with `customer_id=NULL` are valid — they get attributed later by `match_transactions`.
- UNIQUE constraint structural for re-import.
- Vision pipeline thresholds uniform with VT-6.1.

## Acceptance criteria
- 3 vendor parsers (PetPooja CSV, Posist JSON, Slick XML) plus generic CSV fallback.
- Image path uses VT-6.1 vision pipeline with KOTReceiptSchema.
- Migration 018 adds UNIQUE constraint.
- Dedup integration when phone/name available; null customer_id otherwise.
- All test scenarios pass.
- Cross-tenant attack passes.
- PII absence in logs verified.
- mypy --strict passes.
- PR title references `(VT-IntegrationInvention)`; reviewed by CoderX.

## Out of scope
- POS API integrations (Phase 2; e.g., direct PetPooja API).
- Item-level analytics on `items` JSONB (deferred; agent doesn't reason about specific menu items in Phase 1).
- Inventory / stock data from POS exports (out of scope; only customer + transaction data).
- KDS / Kitchen-Display-System integration (Phase 2).
- POS vendor partnerships / certified integrations (Phase 2 GTM concern).
