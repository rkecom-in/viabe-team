## What (Fazal 2026-07-21, prod smoke insight)

Today a cold WhatsApp from a number with no tenant hits `unknown_sender` and drops — the only
front door is the public signup page. For a WhatsApp-FIRST product (CL-443) the natural front door
IS the inbound WhatsApp. Add it: an unknown_sender inbound BECOMES a signup.

**Fazal's key point — NO OTP:** a WhatsApp inbound is already phone-verified by Meta, so the Twilio
Verify OTP the public page uses is redundant. Skip it; record verification basis = `whatsapp_inbound`.

## The seam + flow

- SEAM: the `/api/orchestrator/twilio-ingress` `unknown_sender` branch (no tenant for the number).
- FLOW (consent-gated — see below):
  1. unknown inbound -> reply a welcome + a CONSENT request (freeform-first; e.g. "Reply YES to
     agree to our Terms + data-use policy <link>"). NO tenant yet.
  2. owner replies consent (YES/AGREE, LLM-primary intent + hard-stop veto per no-lists rule) ->
  3. `create_signup_tenant(phone, consent_dpdpa=True, consent_residency=True, source='whatsapp')`
     (records consent proof + trial) -> `start_journey` -> collect the rest (business details) via
     the journey over WhatsApp, same details the public page asks.

## Non-negotiables (correctness gates — never bend)

- **CONSENT IS NOT SKIPPED (DPDP).** `create_signup_tenant` requires consent_dpdpa+residency; the
  public page captures them as checkboxes. Over WhatsApp the explicit consent REPLY is that capture.
  A tenant is created ONLY after consent -> a random/spam inbound never creates a tenant.
- **Abuse gate.** Any number can trigger this. Guard behind a flag (e.g. ENABLE_WHATSAPP_SIGNUP,
  distinct from ENABLE_PUBLIC_SIGNUP), a per-number cooldown/rate-limit on the consent-prompt, and
  the consent step itself filters spam (spammers won't consent). Idempotent: repeated inbounds from
  the same un-onboarded number don't spawn duplicate pending signups.
- **No fabricated data.** Only the number we actually received; everything else asked over WhatsApp.

## Gate

Unit per seam + a dev journey (unknown inbound -> consent -> tenant -> journey -> welcome) + ×3
full-pack (owner-facing). Money/consent-adjacent -> adversarial-verify the consent gate.
