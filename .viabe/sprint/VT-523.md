# VT-523 — DSR data-principal access/correction (Track A)

## Landed this pass — ACCESS is now a tracked ticket
Recon: `dsr_export.export_tenant_data` + `build_export_zip` existed (VT-77) and `/admin/dsr/export` ran them, but — unlike `/delete` — it created **no `dsr_tickets` row**, so an access fulfilment was off-the-books (mig 010 already allows `request_type='access'`; only 'deletion' was ever written).

- `/admin/dsr/export` now creates/loads a `dsr_tickets` **access** ticket (reusing an open one; one fulfilment path, Pillar 8), builds the ZIP, then stamps the ticket `status='completed', completed_at=now()` — a regulator-facing record that the access request was answered + when. Ticket id returned in `X-DSR-Access-Ticket`. Fail-open: a failed export leaves the ticket `acknowledged` (retryable), never a silent completed.
- No migration (dsr_tickets already supports access). RLS/DSR posture unchanged.
- **Canary (real Postgres, deployed-dev-equivalent):** `test_export_endpoint_tracks_access_ticket` (ticket created + completed + completed_at) + `test_export_endpoint_reuses_open_access_ticket` (no duplicate) — 7/7 green, ruff clean.
