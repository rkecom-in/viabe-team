# VT-669 — the framework enforces tool SAFETY, never tool SUFFICIENCY

## The gap (Fazal's instinct, sharpened)
`assert_conforms`'s 8 checks + `assert_agent_tools_safe` enforce that a specialist holds NO forbidden tool (deny-list: send/paste/write/spend). NOTHING checks it holds the tools its job REQUIRES. A specialist can silently lack a read/effect tool it needs and still register clean. And the tools themselves are scattered with no canonical inventory: ~90 `@tool` fns across 11 surfaces + 2 GateFacade doors, no single catalog of name→kind→capability→PII→scope→who-holds-it.

Inventory (2026-07-17): integration_agent 11 · orchestrator_agent (Manager-held) 10 · onboarding_conductor 10 · tech_lane 9 · marketing_lane 7 · finance_lane 7 · cost_opt_lane 6 · accounting_lane 6 · sales_lane 5 · agent/tools/* (self_evaluate, query_customer_ledger, get_attribution_data, get_business_profile, schedule_followup, classify_owner_message) · COMMON_READ_TOOLS 3 (VT-669-adjacent, just built) · GateFacade: request_customer_send + perform_business_action (gated effect doors) + gate_business_action (decision-only).

## The three pieces
1. **TOOL CATALOG** — the canonical inventory (a code registry preferred over a doc so it can't drift; doc generated FROM it). Every tool + GateFacade door with: name, surface (file:line), KIND (READ / GATED-EFFECT / INTEGRATION / ADVISORY / EVAL), the `Capability` it maps to, PII posture (CL-390), RLS/tenant scope (§3), gated? (Y/N), and which module(s)/agent(s) declare or reach it. Single source of truth; ARCHITECTURE §1.3 made concrete.
2. **PER-SPECIALIST REQUIRED-TOOLS MANIFEST** — for each specialist (SR, Onboarding, future Marketing/Finance/Compliance/Data-I-O), the tools its job REQUIRES to deliver its defined outcome: the READ tools it must reach (operational data per §1.1) + the gated effect DOOR it arms. This is the "can it actually do its job" spec — the answer to Fazal's question. Declared on the manifest (e.g. `AgentManifest.required_tools` or a companion map), verifiable at boot.
3. **CONFORMANCE SUFFICIENCY CHECK** — a 9th `assert_conforms` check `required_tools_reachable`: a registered specialist must hold/reach every tool in its required-tools manifest (positive coverage), not just pass the deny-list. Fail-loud at registration. Closes the exact hole: a specialist missing a needed tool dies at boot, never silently at 10am.
