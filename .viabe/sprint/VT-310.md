# VT-310 — canary-DB reaper / local test-cluster hygiene

## Problem (observed in the VT-66/67 build, 2026-06-03)
Throwaway/canary DB runs (and DBOS, which spawns a `<db>_dbos_sys` sidecar per
launch) accumulate on the local Postgres cluster and **never get reaped**. The
migrations create the shared cluster roles `app_role` / `app_operator_role` /
`rls_tester` and GRANT table privileges in each DB. Because roles are
cluster-level but those grants live per-DB, a leftover canary DB holding grants
makes `DROP ROLE app_role` fail:

```
ERROR: role "app_role" cannot be dropped because some objects depend on it
DETAIL: 58 objects in database vt303_canary_38251
```

The pre-push hook's migrations job silently swallows that failed `DROP ROLE`,
then re-`CREATE ROLE` no-ops on the still-present role → the fresh throwaway DB
inherits a stale role with the wrong grants → RLS tests fail with
`permission denied for table pending_approvals` (and similar). This is an
**environment** failure masquerading as a code failure — it cost a debug cycle
in the L2 build and will re-bite any DB-job pre-push.

## What to produce (pick the lightest that holds)
1. A reaper script `scripts/reap_test_dbs.sh` (or Python) that drops every
   non-`postgres`, non-template DB matching the canary/prepush patterns
   (`viabe_*`, `vt*_canary*`, `viabe_prepush_*`, `*_dbos_sys`) **with FORCE**,
   then drops the 3 shared roles. Guard hard against touching any real DB (this
   local cluster is test-only; real dev/prod is remote Supabase — confirm before
   widening the glob).
2. Wire it into the pre-push hook step 4 **before** the throwaway-DB create, OR
   as a `make reap-test-dbs` target + a one-line note in CLAUDE.md's
   throwaway-DB recipe. (Pre-push integration is cleaner — the failure happens
   there — but must stay fast + opt-out-able like the existing DB-job block.)
3. Canary runners (`init_substrate`/`launch_dbos` in tests) should drop their
   `_dbos_sys` sidecar on teardown if cheap — investigate.

## Acceptance criteria
- Running the reaper on a polluted cluster frees the shared roles (subsequent
  `DROP ROLE` succeeds) and a fresh migrations job passes.
- Pre-push DB job no longer false-fails on accumulated canary cruft.
- No path by which the reaper can drop a non-test DB.

## Out of scope
- CI cluster hygiene (CI uses fresh ephemeral pg per run — not affected).
