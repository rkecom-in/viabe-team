# VT-694 — every question infers first and ships with suggested-answer buttons

## Evidence
Second live-transcript set (pre-VT-693 deploy): services → pricing → busy season → annual
revenue → storefront → headcount, incl. a broken "Last one:" promise — an interview, not an
assistant. Fazal's ruling: fetch/infer from public sources + the LLM's own industry knowledge;
each residual question carries suggested answers as buttons (typing still works).

## Built (2026-07-22, same session)
- **Canary-proved the enabler**: Twilio quick-reply Content objects ACCEPT VARIABLE BUTTON
  TITLES → ONE generic object `journey_suggest_3` (en HX41f744c4e398e5d09fd69243a766871c;
  {{1}}=question, {{2}}..{{4}}=suggestions) serves every question with dynamic buttons.
  Canary: canaries/vt694_suggest_buttons_create.py.
- **Composer** (question_brain): the gap prompt now demands 2-3 suggestions per question with
  the MOST LIKELY answer FIRST (inferred from business type + all known context; ≤20 chars) —
  the infer-don't-ask UX without fabrication (the owner's tap is the record; nothing is
  auto-asserted). Question dataclass + queue dicts carry suggestions_en/hi.
- **Delivery**: shared `journey._send_suggestion_buttons` (pads with "Skip" — a native journey
  token; freeform fallback). Wired into BOTH paths: the deterministic walker (`_send`, gap
  questions with suggestions) and the turn-brain (`_send_turn` — dynamically-titled button
  sets now deliver as real buttons; inline "(A / B / C)" text demoted to fallback).
- **Turn-brain discipline** (prompt): every question ships 2-3 suggested answers most-likely
  first; infer before asking; the whole onboarding is "a handful" of questions max; "last one"
  MUST be last; stop asking when the still-needed list is empty.
- A tap echoes the suggestion text as the Body = the recorded answer (zero parse changes).
