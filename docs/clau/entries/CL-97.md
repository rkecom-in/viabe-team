# CL-97 — Env-rename PR ritual: ALWAYS pre-merge double-set

RULE: When a PR renames an environment variable that production reads (Railway, Vercel, or any deployed runtime), follow this ritual in order. Never atomic-swap.

## The ritual

1. **Before merging the PR**: in every environment where the variable is read (Railway service env, Vercel project env, .env.local files if applicable), set BOTH the old name AND the new name to the same value. Both vars now exist; both resolve to the same secret.

2. **Merge the PR**. Code now reads the new name; old name is unread but still present. Zero downtime.

3. **Verify** the production caller works against the new name. For webhook routes, a quick Twilio sandbox test or a synthetic webhook fixture is sufficient. For other endpoints, the regular smoke tests.

4. **Delete the old variable name** from all environments. Code never reads it; safe to remove.

## Why not atomic-swap

Atomic-swap (delete old var + set new var at exact merge moment) creates a sub-second window where neither exists in some environments. For webhook routes specifically, Twilio retries exponentially for ~24 hours then drops messages. Even a 60-second window during a high-message-rate moment can lose customer signal. Not acceptable post-soft-launch; not worth the risk pre-launch either.

## Generalizes beyond PR #12

This applies to all future env-rename PRs in any app (team-web, team-orchestrator, team-ingestion-worker, viabe-reports). The pattern is the same regardless of which secret or which environment.

## What this is NOT

This is not for NEW env vars (no old name to remove; just add the new one and document). Not for env-var VALUE changes (rotation — different concern; same name, different value, no rename involved). Only for renames of vars that production reads.

## Implementation note

Claude Code's merge ritual should now include 'verify pre-merge double-set is in place for any env-rename PR' as a check. Fazal sets the vars in Railway/Vercel manually before approving merge; this is a 30-second click per env. The check is: 'Does this PR change the name of an env var the runtime reads? If yes, is the double-set in place?'

If no, block merge until double-set is done.
