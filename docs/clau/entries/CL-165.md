# CL-165 — model_provider kwarg is OPTIONAL

Context7 deep-query on /websites/langchain_oss_python_langchain (2026-05-19) returned three accepted patterns for the model argument:

1. Provider-prefixed string: create_agent("openai:gpt-5.4", tools=tools)
2. Separate model + model_provider kwargs: create_agent(model="anthropic.claude-3-5-sonnet...", model_provider="bedrock_converse", tools=[...], system_prompt="...")
3. BaseChatModel instance: model = ChatOpenAI(model="...", temperature=0.1); create_agent(model, tools=tools)

The audit's claim that model_provider is a new required kwarg is wrong. Only one of the three patterns is needed. Pattern 3 means the current orchestrator approach (ChatAnthropic instance) still works in create_agent with NO model-passing change.

IMPLICATION for Step 2 brief: scope is smaller than the audit framed. Likely just (a) import change from langgraph.prebuilt.create_react_agent to langchain.agents.create_agent, (b) call-site change to create_agent(...) preserving existing kwargs, (c) add langchain to pyproject with == pin. system_prompt is now first-class but if current code passes a system prompt some other way (e.g. via initial messages), that may also need migration — verify against verbatim orchestrator_agent.py.

NOT a contradiction of CL-134 — deprecation/removal is still real, the migration is still required. Just smaller than the audit doc indicated.

REINFORCES CL-156: Context7 lag was the failure mode the audit warned about; here the audit itself was downstream of incomplete Context7 reading. Standing rule: before drafting any brief off an audit-doc finding, re-validate the specific call surface against current Context7 — even if the audit cited Context7.
