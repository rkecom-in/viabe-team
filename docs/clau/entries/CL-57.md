# CL-57 — Memory layer: Mem0 OSS for L1-L3 substrate, custom L0+L4

Memory layer L1-L3 substrate uses Mem0 OSS Python library (Apache 2.0). L0 and L4 remain custom-built.

Architecture clarification (terminology corrected):
- Mem0 OSS is the FREE Python library (`pip install mem0ai`) — NOT the Mem0 Cloud managed service
- Library runs IN our team-orchestrator Python process on Railway
- Library writes to OUR existing Supabase Cloud Postgres database (no new infrastructure)
- Mem0 creates its own tables (memories, entity collection, relationships) in our Supabase DB via migrations we apply

Reason: Mem0 v1.0.0 implements exactly the pattern we'd be building for L1 (Postgres + pgvector + entity linking + multi-signal hybrid retrieval). Their dual-store architecture (vector + entity collection in Postgres) is the 2026 production-standard pattern they explicitly moved to from external graph stores. Building it ourselves would be writing the same code with worse results (52.8K GitHub stars, $24M raised, SOC 2 compliance, MCP server for Claude Code).

What Mem0 handles (L1-L3 substrate):
- Multi-tenant scoping via user_id, agent_id, run_id, app_id filters with AND/OR
- Entity extraction (uses Claude as LLM for extraction)
- Multi-signal retrieval: vector similarity + BM25 + entity matching
- Metadata filtering for context-scoped memory (matches our L1/L2/L3 boundaries)
- L1 = tenant business knowledge → Mem0 user_id=tenant_id, app_id='l1_business'
- L2 = customer profiles → Mem0 user_id=tenant_id, app_id='l2_customer', metadata.customer_id
- L3 = anonymized cross-tenant patterns → Mem0 app_id='l3_anonymized' (no user_id scoping; k≥10 enforcement at admission gate stays custom)

What stays custom (differentiating work):
- L0 = workspace knowledge for Viabe-specific orchestration patterns (admission gate logic, k≥10 enforcement, escalation rules, terminal states). Specific to our business, not memory-as-substrate.
- L4 = closed-loop feedback unique to our owner-approval flow. Specific to our business.

Row-Level Security preserved (Pillar 3): Mem0's tables are regular Postgres tables in our Supabase DB. We apply RLS policies to them in a migration immediately after Mem0's schema deploys. Tenant isolation enforced at DB layer AND at Mem0 query layer (user_id/app_id filters). Belt-and-suspenders.

Trade-off accepted: Mem0's schema lives in Mem0's source code. We can extend with columns (Postgres permits it), but core schema is theirs. If Mem0 changes schema in a future version, we migrate with them. This is the cost of using a maintained library instead of fully owning the schema. Acceptable.

Not-natively-supported gap: Mem0 doesn't have first-class temporal data model (valid_from/valid_to). For Phase 1-3 not critical. If it becomes critical we add columns or layer Zep/Graphiti later.

LLM cost for entity extraction: Mem0 uses an LLM call on each memory.add() to extract entities. Cost is modest (few hundred tokens per write) but non-zero — quantifiable in production. Configure to use Claude Haiku for extraction to minimize cost.

Time impact on Sprint 7: probably saves 1-2 weeks of L1 build time. Doesn't push July 15 soft launch.
