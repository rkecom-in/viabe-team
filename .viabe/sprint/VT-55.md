## What CoderC must produce
The Method 1 ingestion adapter. Owner sends a photograph of their paper customer book via WhatsApp. The adapter receives the image, extracts entries via vision pipeline (VT-6.1), routes ambiguous entries through clarifying flow (VT-6.2), dedups via VT-6.3, and writes resolved customers + transactions.
1. **Adapter module** at `apps/team/src/ingestion/methods/paper_book.py`:
	- `ingest_paper_book(tenant_id, image_url, run_id) -> IngestionSummary`.
	- Fetches image bytes from Twilio media URL (auth via Twilio credentials).
	- Calls `vision_extraction.extract_from_image()` with `PaperBookSchema`: `customer_entries: list[{name: str, phone: str | None, last_visit_date: date | None, total_spend_paise: int | None, notes: str | None}]`.
	- For each extracted entry: route by per-entry confidence:
		- All fields ≥0.85 → dedup + commit.
		- Any field 0.7-0.85 → dedup + commit + notify owner ("we extracted X; verify if needed").
		- Any field <0.7 → clarifying flow (VT-6.2) before commit.
	- Returns `IngestionSummary`: `entries_extracted`, `entries_committed`, `entries_pending_clarification`, `entries_dropped`.
2. **Schema definition** for paper book (passed to VT-6.1):
	- Customer entries are typically rows in a handwritten ledger: name, phone (sometimes), date of last visit, running total or per-visit spend.
	- Hindi (Devanagari) and English mixed; phone numbers in Devanagari numerals also possible.
	- Schema permits `notes` field for unstructured marginalia ("prefers afternoon", "family of 4", etc.) — captured but not parsed further in Phase 1.
3. **Telemetry**:
	- `pipeline_log` events: `paper_book_ingestion_started`, `vision_extraction_completed` (with confidence stats), `entry_committed`, `entry_pending_clarification`, `entry_dropped`, `paper_book_ingestion_completed` (with summary).
	- PII NEVER in payload — only counts.
4. **Owner feedback after ingestion**:
	- Send a WhatsApp summary message: "From your paper book photo, we added 23 customers, asked you 2 clarifying questions, dropped 1 unclear entry." Template: `paper_book_summary` (added to VT-13.3 list).
	- If dropped count > 0: include guidance on retaking the photo for the dropped entries.
5. **Tests** at `apps/team/tests/ingestion/methods/test_paper_book.py`:
	- Synthetic image fixture with 5 entries (3 high-conf, 1 medium, 1 low) → 3 committed silently, 1 committed + notify, 1 escalated to clarification.
	- Image fetch failure (Twilio 404): structured error, no partial commit.
	- Vision pipeline failure: error envelope; ingestion fails clean (no orphan rows).
	- Dedup integration: synthetic image with entries that match existing customers → merged correctly.
	- Cross-tenant: tenant_A image cannot create tenant_B rows.
	- Owner summary: synthetic ingestion → correct summary message dispatched.
	- Real-API integration test (env-gated): real fixture image → vision extraction + dedup + commit end-to-end.

## Architectural rules to enforce in this subtask
- Pillar 4: confidence-driven routing is uniform. The thresholds match VT-6.1's defaults. CodeX rejects per-method threshold overrides.
- Pillar 8: this method is a thin adapter. Heavy lifting is in vision pipeline (VT-6.1), clarifying flow (VT-6.2), dedup (VT-6.3). The adapter coordinates; it does not reimplement.
- All commits use the typed wrapper (VT-8.1) for tenant scoping.
- PII never in `pipeline_log` payload.
- Image bytes are never persisted (delegated to vision pipeline).
- The owner-summary message is sent via VT-5.7 (template), not free-form, since onboarding may be outside the 24h window.
