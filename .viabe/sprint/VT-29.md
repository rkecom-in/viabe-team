## What CoderC must produce
The orchestrator's BUSINESS error-handling framework. Layered on top of DBOS's system-level auto-resume.
**Architectural shift from prior framing:** with DBOS, system-level errors (Railway crash, transient DB connection drop, network timeout mid-step) are handled BELOW this framework by DBOS auto-resume. This framework handles BUSINESS-LEVEL errors: agent hard-limit breach, agent refusal, agent invalid output, persistent external API errors after DBOS retries exhausted, webhook signature failures, unknown errors requiring human classification.
Failure taxonomy + strategy library + deterministic router. No exceptions swallowed. No silent retries.
1. **Two error layers (made explicit):**
	- **System errors (DBOS-handled):** Railway crash, transient DB drop, Python process kill, ephemeral network failure. DBOS auto-resumes the workflow from the last completed step. NO classification needed at this layer; DBOS handles it.
	- **Business errors (this framework):** semantic failures that DBOS resume cannot fix. Agent breached hard limits. Agent refused. External API persistently failing after DBOS retries. Webhook signature mismatch. Unknown unclassified exception.
2. **Failure taxonomy** in `apps/team-orchestrator/src/orchestrator/failures.py`:
	Phase 1 enumerates 9 business failure types:
	- `tool_call_timeout` — MCP tool exceeded individual timeout (after DBOS retries exhausted).
	- `tool_call_error` — MCP tool returned structured error envelope (not an exception).
	- `agent_hard_limit_breach` — one of 5 limits tripped (80K tokens / 25 tool calls / depth 8 / 5min wall-clock / **₹50 cost**).
	- `agent_refusal` — model returned refusal pattern (`I cannot`, `I'm sorry, I'm unable`, etc.).
	- `agent_invalid_output` — agent returned result that fails schema validation (e.g., CampaignPlan).
	- `external_api_error` — Twilio/Razorpay/Resend/Apify persistent error after DBOS retries.
	- `database_error` — RLS rejection / constraint violation / business invariant failure (NOT transient connection drop — that's DBOS-handled).
	- `webhook_signature_failure` — inbound signature didn't verify.
	- `unknown_error` — catch-all for unclassified exceptions; ALWAYS escalates to Fazal.
	Each type has structured fields: `severity (critical|major|minor)`, `retryable (bool)`, `default_strategy`, `max_retries`, `escalation_threshold`.
3. **Strategy library** in `apps/team-orchestrator/src/orchestrator/strategies.py`:
	Five strategies for Phase 1:
	- `retry_with_backoff(attempts, base_delay, max_delay)` — exponential backoff with jitter. For transient `tool_call_timeout` + `external_api_error` cases DBOS didn't auto-handle.
	- `retry_after_owner_clarification` — pause workflow (DBOS supports this via signals), send owner clarifying message, resume on owner reply.
	- `escalate_to_owner` — send WhatsApp template + escalation path. Default for `agent_hard_limit_breach`, `agent_refusal`, repeated failures.
	- `escalate_to_fazal` — alert via Telegram (reuse Reports' OpsAssistant channel). Default for `unknown_error`, `webhook_signature_failure`, repeated `database_error`.
	- `accept_and_log` — record failure and proceed (some errors don't need retry).
4. **Router** in `apps/team-orchestrator/src/orchestrator/error_router.py`:
	- `route_failure(failure: FailureRecord, run_state: SubscriberState) -> Strategy`.
	- Deterministic (Pillar 8); no LLM in routing decision.
	- Logs routing decision to `pipeline_steps` with structured fields.
5. **Wiring**:
	- Every `@DBOS.step` decorated function wraps work in try/except that converts caught business exceptions to `FailureRecord`. DBOS handles system-level exceptions before they reach this code.
	- The supervisor handoff (VT-3.4) wires hard-limit breaches directly to `agent_hard_limit_breach`.
	- Scheduled triggers (VT-3.5) wrap handlers; failures here typically route to `escalate_to_fazal` (no owner conversation context).
6. **Backoff and circuit breaker**:
	- `retry_with_backoff`: jittered exponential 1s, 2s, 4s, 8s, 16s with ±25% jitter. Capped at 5 attempts.
	- Circuit breaker per-vendor: if `external_api_error` for same vendor exceeds 10 failures in 60s, breaker opens; all calls return `external_api_error: circuit_open` for 5 minutes.
	- DBOS already provides workflow-level retries; this is an additional vendor-level circuit on top.
7. **Escalation paths**:
	- `escalate_to_owner`: invokes `send_whatsapp_template` with `agent_stuck` template (VT-13.3 approval pending). Owner's reply re-enters via Twilio webhook → Pre-Filter → orchestrator-agent.
	- `escalate_to_fazal`: posts to Reports' OpsAssistant Telegram channel. Message includes run_id, tenant_id, failure_type, last 5 `pipeline_steps` for the run.
8. **Tests** in `apps/team-orchestrator/tests/orchestrator/test_error_handling.py`:
	- Each business failure type tested independently → router selects expected strategy.
	- **DBOS system-error test:** synthetic Railway crash mid-step → DBOS auto-resumes → NO business framework invocation (proves layer separation).
	- Backoff math test (mocked clock).
	- Circuit breaker: 11 synthetic external errors in 60s → breaker opens.
	- Escalation: `escalate_to_owner` triggers WhatsApp template send; `escalate_to_fazal` triggers Telegram post.
	- `unknown_error` always escalates regardless of attempt count.

## Architectural rules to enforce in this subtask
- **Pillar 8 (no patchwork):** every business exception caught MUST become a `FailureRecord` with a classified type. CodeX rejects PRs that catch and swallow.
- Router is deterministic. New failure types require updating `failures.py`; new strategies require updating `strategies.py`.
- `unknown_error` ALWAYS escalates. No retries. Forces classification.
- Agents (orchestrator-agent + specialists) do NOT invoke this framework directly. It's wrapped around their invocations by the harness.
- Circuit breakers scoped per-vendor + per-environment. Dev state never leaks to prod.
- All escalations logged. Escalations without `pipeline_steps` event are bugs.
- **Two error layers explicit:** system (DBOS-handled) and business (this framework). CodeX rejects PRs that conflate.
