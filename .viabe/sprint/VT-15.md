## What CoderC must produce
A single new file at `docs/concept-team-pillars.md` containing the inviolable architectural pillars for the Team product. The file structure is mandatory and mechanical so future agents can locate any pillar in seconds.
File structure:
1. **Top banner** declaring the source-of-truth hierarchy: this file > `concept-team.md` > `CLAUDE.md` (Team sections) > `Viabe_Team_Technical_Reference_v1_0.md`. Banner is identical wording to the banner that VT-1.2 will add to the other three files.
2. **Why this file exists** (~150 words). Reference the R6→R9 drift in Reports, the patchwork pattern, the retroactive nature of Reports' `concept.md`, and the lesson: pillars must precede code, not follow it.
3. **The 8 pillars**, each with these four sections in this order:
	- **Rule** — one-line statement (e.g. "Tier separation is sacred").
	- **Rationale** — the failure mode this rule prevents, citing Reports' specific incidents (VB-XX) where applicable.
	- **Examples of violations** — concrete patterns future agents and engineers can pattern-match against. Use Reports-side examples where they exist (regex scrubs in `pdf_generator.py`, hardcoded business-type dicts, Python clamps in producers, gate-bypass shortcuts).
	- **Verification check** — a CI grep, test fixture, code-review checklist item, or runtime invariant that proves the rule holds. Must be concrete. "We'll be careful" is not a verification check.
4. **The 8 pillars are**:
	1. Tier separation is sacred (LangGraph routes; agent reasons; tools execute deterministically).
	2. Reasoning lives in the agent, not in the tool (only `self_evaluate`, `classify_owner_message`, optionally `draft_message_variants` may call LLMs).
	3. Tenant isolation is structural, not procedural (RLS + typed wrappers + agent context isolation — three independent layers).
	4. Retrieve, don't calculate (no business-type Python dicts, no invented formulas, no per-vertical cheat sheets in prompts).
	5. No fine-tuning, no model-weight modification (improvements via prompts and retrieval only).
	6. K-anonymity is a build-time invariant, not a runtime check (L3 patterns built only over groups ≥1 0 with ward/city-tier coarsening at construction).
	7. The owner is the source of truth on subscriber identity (customer reconstitution requires owner-verified identity within 7-day SLA).
	8. Patchwork is forbidden (no regex scrubs, no value clamps, no hardcoded constants in producers; every fix routes through the correct architectural layer + memory store).
5. **Governance protocol** section restating the Type 1 / 2 / 3 framework from `concept-team.md` §16–18. Pillars are Type 3. Amendments require Fazal + Clau formal sign-off in Notion before code change.
6. **Source-of-truth banner** repeated at the bottom for emphasis, and a closing paragraph stating that this file is the highest authority for Team product architecture.
Word count target: 2,500–4,000 words for the full document. Each pillar gets ~250–400 words of treatment.

## Architectural rules to enforce in this subtask
- The pillars file is itself the embodiment of Pillar 8 (no patchwork). It must be precise, structured, and complete — not a summary of `concept-team.md` with bullet points. If a pillar can't be verified by a concrete check, the verification section says "manual code review" and names the reviewer (Fazal for Type 3, CoderX for Type 2, CoderC for Type 1).
- Cross-references to Reports' incidents must be accurate. If you cite VB-134, the Reports incident must actually be VB-134, not approximated. Pull the right task IDs from `Viabe_AI_Technical_Reference_v2_0.md`.
- Banner wording must be exact and reusable. VT-1.2 will paste the same banner into the other three files; if the wording in this file is sloppy, three more files inherit the sloppiness.
