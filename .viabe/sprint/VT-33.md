## What CoderC must produce
The v1.0 system prompt for the Sales Recovery Agent. Lives in `apps/team/src/agent/prompts/sales_recovery_v1.md` (markdown for readability), loaded at agent construction time. Capped at 4,000 tokens. Token count measured and recorded in the file's frontmatter.
The prompt has these mandatory sections in this order:
1. **Identity** (~150 words): "You are the Sales Recovery Agent for [tenant business name], a specialist in winning back dormant customers via WhatsApp campaigns." States the agent's narrow scope: re-engagement only. Names the product ("Viabe Team") and the tenant context (subscriber business).
2. **Scope and explicit out-of-scope** (~250 words): "Your job is to: identify dormant customer cohorts, propose a campaign plan, and produce structured output. Your job is NOT to: send messages directly (the orchestrator does), modify the customer ledger (tools do), make refund decisions (deterministic SQL does), reason about reputation/marketing/operations (separate Phase 2 specialists)." Each forbidden action is a bullet with a reason and a pointer to the right place.
3. **Available tools** (~600 words): enumerates the 11 (or 12) MCP tools with one-paragraph descriptions each. Tools are listed by exact name. Includes when to use each, and explicit warnings against asking for tools that aren't listed (the SDK will refuse). Calls out which tools are LLM-backed (`self_evaluate`, `classify_owner_message`, `draft_message_variants` if approved) so the agent doesn't loop on them.
4. **Required output schema** (~400 words): describes `CampaignPlan` structure (full schema lands in VT-4.6). The prompt says: "Your final output MUST be valid JSON conforming to the CampaignPlan schema. Do not produce free-form prose as final output. Do not produce partial or commented JSON."
5. **Refusal patterns** (~200 words): explicit instruction "Never refuse a task within scope. If a task seems out of scope, return a CampaignPlan with `status: out_of_scope` and a `reason` field. If you genuinely cannot proceed (e.g., the tenant has zero customers in the ledger), return `status: insufficient_data` with the missing-data reason." This prevents the model's default refusal patterns from leaking through.
6. **Pillar discipline** (~400 words): explicit prose forbidding:
	- Inventing numbers (footfall, churn, persona traits) — always retrieve via tools or knowledge layers.
	- Per-vertical heuristics from training data ("cafes typically have a 30% return rate" is forbidden — retrieve from L4 skill corpus or admit uncertainty).
	- Overstating attribution confidence ("this campaign will recover 50K rupees" is forbidden; "the historical pattern suggests 30K-60K with low confidence" is correct).
	- Retention pressure language in proposed message templates ("this is your last chance" is forbidden).
	- False certainty about customer intent ("this customer wants discount X" is forbidden; "this customer's pattern suggests price-sensitivity" is correct).
7. **Hard-limit awareness** (~150 words): the agent is told the hard limits (80K tokens / 25 tool calls / depth 8 / 5min wall clock) and told that termination is unilateral by the orchestrator. The agent is instructed to prioritize structured output even under tight budgets, and to call `self_evaluate` before final return.
8. **Self-evaluate requirement** (~150 words): "Before returning your final CampaignPlan, you MUST call `self_evaluate` with the draft. If `self_evaluate` returns `revise`, address its feedback and call once more. If it returns `revise` a second time, return your draft anyway with `self_evaluate_status: failed` so the orchestrator can decide." (Self-evaluate logic itself is VT-4.5; the prompt sets the contract.)
9. **Frontmatter** at top of file: `version: 1.0`, `token_count: <measured>`, `model: claude-opus-4-7`, `last_reviewed_by_fazal: <date>`. Frontmatter is parsed at load time; mismatched model triggers a startup error.

## Architectural rules to enforce in this subtask
- Pillar 1+2: the prompt is the structural enforcer of tier separation. Every "do not" clause maps to a pillar.
- Pillar 4: the "retrieve, don't calculate" prose is verbatim. CodeX confirms in review.
- Pillar 7: honesty rules in prompt prose are the agent's first defense. The composer (VT-3.7) is the second.
- Pillar 8 (no patchwork): prompt edits go through versioned files (`sales_recovery_v1.md`, `_v2.md`, etc.). No silent inline edits. Major version bumps require Type 2 governance.
- Token count is measured (`tiktoken`-equivalent or Anthropic's tokenizer). The hardcoded 4,000-token cap is a CI-checked invariant.
- The prompt explicitly forbids using Reports' Director / VRI / pipeline concepts. Sales Recovery Agent has no awareness of Reports product. Cross-product leakage in prompts is forbidden.
- Hindi + English literacy: the prompt is authored in English, but explicitly instructs the agent that owner messages may be in Hindi or mixed-language and should not be "corrected."
