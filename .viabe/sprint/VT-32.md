## What CoderC must produce
The Anthropic Agent SDK skeleton wrapped as a callable function consumable by the orchestrator's specialist dispatch. At this stage the agent has no real tools wired, no system prompt content, no context bundle — those come in subsequent subtasks. This subtask is purely the SDK plumbing.
1. **Install the Anthropic Agent SDK** in `apps/team/pyproject.toml`. Pin the version. Document in `docs/team/dependency-pins.md`. Reuse the existing ANTHROPIC_API_KEY env var.
2. **Agent module** at `apps/team/src/agent/sales_recovery.py`:
	- Function `run_sales_recovery_agent(context: SalesRecoveryContext) -> AgentResult` is the entry point called by `dispatch_to_specialist()` (VT-3.4).
	- Constructs an Agent SDK instance with: model (`claude-opus-4-7` for Phase 1; pinned in `apps/team/config/models.yaml`), system prompt placeholder (real content lands in VT-4.2), tool list placeholder (empty stub list — real tools wire up after VT-5 subtasks), max_tokens 80000, max_thinking_tokens (per Anthropic SDK spec).
	- Calls the Agent SDK's run loop and returns the structured result.
3. **`AgentResult` dataclass** in `apps/team/src/agent/types.py`:
	- `status: Literal['completed' | 'terminated' | 'refused' | 'invalid']`.
	- `terminated_by: Literal['tokens' | 'tool_calls' | 'depth' | 'wallclock'] | None`.
	- `output: dict | None` — the structured `CampaignPlan` (lands in VT-4.6) when status == completed.
	- `tokens_used: int`, `tool_calls_made: int`, `wallclock_ms: int`, `cost_paise: int`.
	- `raw_messages: list[dict]` — full message history for trace + observability.
	- `terminated_reason: str | None` — plain-language explanation when terminated.
4. **Stub canary**: with no tools wired and a placeholder system prompt ("You are a placeholder agent. Reply with the JSON `{\"status\": \"placeholder\"}`. Do nothing else."), the agent run completes successfully against the real Anthropic API. This proves the SDK plumbing works before any real prompt or tool exists.
5. **LangGraph node wrapping**: `apps/team/src/agent/langgraph_node.py` exports `sales_recovery_node(state)` that calls `run_sales_recovery_agent`, captures `AgentResult`, and returns a state update for LangGraph. This is the actual node imported by VT-3.4's dispatch.
6. **Tests** in `apps/team/tests/agent/test_sales_recovery_skeleton.py`:
	- Stub canary test: invocation against the real API succeeds and returns `status='placeholder'`.
	- Skipped-by-default integration test (env-flag gated to avoid burning API quota on every CI run).
	- Mocked unit tests verifying `AgentResult` shape, terminated_by population on hard-limit breach, raw_messages capture.
	- LangGraph node test: `sales_recovery_node` correctly translates `AgentResult` into state updates.

## Architectural rules to enforce in this subtask
- Pillar 1: this module is Tier 2 only. It does NOT call the database directly, does NOT send WhatsApp messages, does NOT reach into LangGraph state directly. It receives a typed context (VT-4.3), produces a typed result, and returns. The orchestrator handles persistence + side-effects.
- Pillar 2: the SDK invocation is the LLM boundary. Tools called from inside the agent must be Tier 3 MCP tools (VT-5). Calling an LLM from inside a tool is a Pillar 2 violation that CodeX rejects.
- Model selection lives in `apps/team/config/models.yaml`, not hardcoded in agent code. Switching models is a config change.
- The placeholder system prompt is explicitly placeholder. CodeX rejects PRs that grow it past stub status — real prompt content goes through VT-4.2.
- The integration test is env-flag gated. CI does not burn API quota by default.
- `cost_paise` calculation is mandatory in `AgentResult`. Phase 1 uses Anthropic's billing API or a deterministic table to convert tokens → USD → INR paise. Documented in `docs/team/llm-cost-attribution.md`.
