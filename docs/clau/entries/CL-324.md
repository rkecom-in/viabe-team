# CL-324 — DECISION (Type 2, final): Memory substrate split — L1 hand-built (Postgres+pgvec

## Decision (Type 2, Fazal-confirmed 2026-05-22) — supersedes CL-57

The memory layer L1-L3 substrate is NOT a single tool. Split by layer:

**L1 (business knowledge graph): HAND-BUILT.** Postgres + pgvector + plain relational tables — exactly the CL-55 design, before CL-57 bolted Mem0 on top. Two core tables: l1_entities (id, tenant_id, entity_type, attributes JSONB, embedding pgvector, valid_from, valid_to, created_at) and l1_relationships (id, tenant_id, from_entity, to_entity, relationship_type, attributes JSONB, valid_from, valid_to). Recursive CTEs for 2-3 hop traversal. Multi-signal retrieval: pgvector similarity + relational entity filter + Postgres full-text. Tenant isolation via real Postgres RLS. No Neo4j, no new infrastructure, no LLM-per-write.

**L2 (episodic memory) + L3 (cross-tenant patterns): Mem0 is a candidate, NOT yet decided.** Decision deferred to a real validation spike at VT-7 time (post-launch). The spike installs Mem0 OSS, tests it against L2's actual retrieval contract, then commits or rejects. L2/L3 are post-launch quality layers, not VT-4-blocking (ship-thin already decided).

**owner_inputs:** unchanged — separate PII decision-memo, structured-intent over raw bodies, Fazal's call. Not a memory-layer-tool question.

## Why CL-57 was wrong

CL-57 chose Mem0 OSS for all of L1-L3 on reputation + pattern-match, never validated. Context7 verification (2026-05-22) found:

1. Mem0's graph memory requires a SEPARATE graph DB — Neo4j or Neptune (every graph example uses provider: neo4j with a bolt:// connection). pgvector is Mem0's VECTOR store only, not its graph store. CL-57's claim of 'no new infrastructure' was false — a real L1 knowledge graph on Mem0 means standing up Neo4j, the exact problem AGE was killed to avoid.
2. Mem0 INFERS relationships from conversational text via an LLM. L1's data is structured business facts already held as foreign-keyed rows — customers, transactions, campaigns. You do not need LLM inference to know a customer made a transaction; you have a foreign key. Mem0's core value (inferring structure from unstructured conversation) does not apply to L1.
3. Mem0 v3 removed the relations field from API responses — entity relationships are now consumed only via retrieval ranking, not readable as explicit edges. For an L1 KG queried deterministically with tenant scoping, that is a regression. The library is moving away from being a queryable graph.
4. CL-57 itself flagged that Mem0 lacks a first-class temporal (valid_from/valid_to) model — while CL-55 made time-awareness the explicit reason the new L1 design beat AGE. Unreconciled contradiction.

The core reasoning, independent of points 1/3/4: L1 is a structured-data problem wearing a 'knowledge graph' label. Memory libraries earn their keep extracting structure from unstructured conversation; L1's input is already structured. Mem0 keeps its value for L2/L3 where data is genuinely conversational and recall-shaped.

## Verification honesty note

The Neo4j-required and v3-relations-removed findings are Context7-verified. Two supporting points were inferred not verified: that there is NO Postgres graph-store option at all, and that v3 behavior applies identically to the OSS library. The decision does not depend on those — it rests on the structured-data-vs-inference argument, which holds regardless. Fazal accepted the recommended direction without requiring a confirmatory spike.

## Consequence

L1 hand-built is briefable NOW from the CL-55 spec — no spike, no infrastructure decision, no wait. This unblocks the substrate work. L2/L3 defer cleanly to post-launch VT-7.
