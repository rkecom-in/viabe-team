# VT-227 — twilio_inbound_replay TTL purge

## Why

VT-81 ship: migration 041 adds `twilio_inbound_replay` table tracking message_sid for 5-minute replay window. Without TTL purge, table grows unboundedly.

Per CL-416 (30-day retention), but for replay defense 24h is overkill; 5min suffices. Daily purge keeps table bounded + index lean.

## Brief contract

DBOS scheduled workflow at `apps/team-orchestrator/src/orchestrator/observability/twilio_replay_purge.py`:
- `@DBOS.workflow()` + `@DBOS.scheduled('0 3 * * *')` — daily 3 AM IST
- `DELETE FROM twilio_inbound_replay WHERE received_at < now() - interval '24 hours'`
- Logs row count purged (no PII)

Canary `apps/team-orchestrator/canaries/vt227_replay_purge.py`:
- A1: seed 100 rows (50 > 24h, 50 < 24h); run workflow; assert 50 deleted, 50 retained
- A2: workflow registered + scheduled trigger confirmed

## Acceptance

- Workflow registered + scheduled
- Canary 2/2 PASS
- Manual run logged via Logfire

## Out of scope

- Webhook_metrics purge (separate row, may piggyback on VT-226)
- Inbound message body retention (CL-416 covers — different mechanism)
