# CL-29 — Orchestrator triggering: event-driven, plugin-mediated. NOT continuous-loop obse

Orchestrator is NOT a continuous-loop observer. It is event-driven and triggered only by: (1) Owner-initiated WhatsApp messages (founder interaction). (2) Scheduled triggers (cron-based: weekly approval, attribution check, day-39 evaluator, monthly report). (3) Integration plugins that we author per connected system. The plugins themselves are responsible for waking up the orchestrator when something materially happens in the connected system. Phase 1 plugins = the 9 ingestion methods. Phase 2-3 plugins = ERP/CRM/Accounting/Marketplaces — each plugin is responsible for its own polling/webhook/listener and pings the orchestrator only when events warrant. The orchestrator does NOT continuously poll external systems. The orchestrator is the brain; the plugins are the sensors. The brain wakes when a sensor reports something. Continuous-loop observation rejected as overkill: cost-prohibitive at Opus tier, and most external systems don't have events frequent enough to justify always-on listening anyway.
