## What CoderC must produce
The authenticated owner portal at `viabe.ai/team/dashboard`. Read-only views of tenant data. Authentication via WhatsApp magic-link (owner clicks a link delivered via WhatsApp; clicking confirms identity for a session).
1. **Authentication flow**:
	- Owner clicks "Open dashboard" link in any of the templates that include it (welcome / monthly report / approval).
	- Link opens a small Vercel-hosted page that triggers `POST /api/auth/whatsapp-link` with the link's signed token.
	- Backend validates token (signed JWT with tenant_id + 7-day expiry; signed with `INTERNAL_API_SECRET`-equivalent JWT secret in env).
	- On valid token: sets a session cookie (`viabe_team_session`); redirects to `/team/dashboard`.
	- Session cookie scoped to [viabe.ai/team/\*](http://viabe.ai/team/*) paths only; httpOnly + secure + samesite=Strict.
2. **Dashboard pages** (`apps/team-web/app/team/dashboard/...`):
	- `/dashboard` (index): hero metrics for current month + last 30 days; top 5 customers by lifetime spend; recent campaigns list (5).
	- `/dashboard/customers`: paginated customer list (using `query_customer_ledger` via VT-5.2; phone numbers MASKED in UI — last 4 visible only); filters by recency_band + spend_band.
	- `/dashboard/campaigns`: campaign history (using VT-5.4); each shows status + cohort + ARRR (closed) or pending.
	- `/dashboard/reports`: monthly reports list (from VT-9.6); each downloadable as PDF.
	- `/dashboard/settings`: read-only display of business profile (from VT-5.3) + plan tier + trial dates + privacy settings + DSR request initiation buttons (links to VT-8.6 endpoints).
3. **Data fetching**:
	- All API routes under `/api/team/*` validate session cookie → extract tenant_id → call typed wrappers (VT-8.1).
	- Phone numbers masked at API boundary; raw phones never sent to frontend.
	- Customer search by phone (last 4 digits) — partial match supported via VT-5.2.
4. **Phase 1 read-only constraints**:
	- NO write actions in Phase 1 portal. CodeX rejects PRs adding edit/delete buttons.
	- DSR initiation buttons are exceptions: clicking sends a request to /api/dsr/\* but that's owner-initiated, not owner-edit.
	- All actions (approvals, exclusions, refund decisions) flow through WhatsApp — portal is window, not control panel.
5. **Performance**:
	- SSR + edge caching for static parts.
	- Per-tenant data fetched on-demand with 30s cache.
	- Lighthouse score >85 on dashboard index.
6. **i18n**:
	- Hindi + English language toggle (matches `tenant.preferred_language` default; user can toggle in session).
	- All strings in `apps/team-web/locales/{en,hi}.json`.
7. **Tests** at `apps/team-web/__tests__/dashboard/*.test.tsx`:
	- Auth: invalid token → redirect to magic-link landing.
	- Dashboard index: renders metrics from synthetic backend.
	- Customers list: pagination + filter; phones masked.
	- Cross-tenant: session cookie for tenant_A cannot see tenant_B's data (verified via API mock with malicious cookie tampering).
	- Hindi locale: Hindi labels rendered.
	- Lighthouse: synthetic CI run >85.
## Architectural rules to enforce in this subtask
- Pillar 7: read-only Phase 1; CodeX rejects edit/delete UI.
- Pillar 8: one portal app under `apps/team-web/app/team/dashboard/`; no parallel admin / ops portals (those would be separate subdomain + separate auth).
- Phone masking at API boundary, not at frontend. Frontend never sees raw phone numbers (defense in depth against XSS / leak).
- Session cookie httpOnly + secure + samesite=Strict; non-negotiable.
- DSR action buttons (export + delete) DO submit; exception is documented.
- WhatsApp magic-link is the only auth in Phase 1. Password / OTP authentication is Phase 2.
- Lighthouse >85 is a Phase 1 commitment; lower scores fail CI.
