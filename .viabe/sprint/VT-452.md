# VT-452 — LLM-search discovery (PLAN-FIRST → gate; risk row, touches the GST-gate input)

## Problem (live e2e 2026-06-28)
Discovery's GBP leg finds the business but no GSTIN; the web/SERP leg returns 0 for a small-biz GSTIN not
SERP-indexed (RKeCom). So the owner must type the GSTIN. Google/ChatGPT DO find it → an LLM web-search of
public records can auto-surface the business + its **GSTIN** (and CIN/details).

## Proposed approach (for the gate to bless before build)
- Add an **LLM-discovery leg** to `onboarding/entity_match.fetch_candidates` (alongside web/gbp/cin), behind a
  flag (default off until blessed): an LLM (Anthropic, `claude-opus-4-8`) web-search/tool-use call —
  "find the GSTIN + CIN + registered name for `<business_name>` in `<city>` from public records" — returns
  candidate GSTIN(s) + name. Returned GSTINs are **candidates only** (HINTs), never trusted.
- **The authoritative GST verify (Sandbox `gstin/search`) STAYS the gate** — the LLM does FIND/SUGGEST; the
  owner-confirmed/auto-surfaced GSTIN is round-tripped through Sandbox + the VT-448 name-match before
  `gstin_verified`. No LLM output ever sets verified state. (This is why it's a risk row — the gate must
  confirm the trust boundary is unchanged.)
- Provenance: an LLM-found GSTIN is a `source:'llm'` candidate; same pick→verify spine as web/gbp.
- Cost/latency: bounded (one LLM call, timeout, graceful-degrade to the existing legs on miss/failure).
- Privacy: business-level identity only (no proprietor PII into the LLM beyond the public name/city).

## Acceptance (post-gate build)
"RKeCom Services Pvt Ltd"/Mumbai → the LLM leg surfaces a GSTIN candidate → owner confirms → Sandbox verify →
gstin_verified. The Sandbox gate + name-match remain the sole verify authority. Adversarial-verify the trust
boundary (no LLM-supplied GSTIN/name bypasses Sandbox). Plan → Cowork gate → build → gate → Fazal push.
