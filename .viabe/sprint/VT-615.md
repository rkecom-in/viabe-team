# VT-615 — approval arm-durability race + brain busy-blindness

## Root cause (VT-611 read-only traces, 2026-07-07 — 2 interacting bugs, NEITHER is VT-594)
NL approval recognition WORKS (`owner_inputs/approval_reply.py:classify_approval_reply` direct-tested: "haan theek hai,
bhej do unhe" → `approved`). The drop is upstream/structural:

**(A) Arm-durability race** — `agent/tools/request_owner_approval.arm_pause_request` sends the approval **template at
:311 BEFORE the `pending_approvals` INSERT at :371**. The INSERT can lose the migration-128 one-open-per-tenant unique
race → rolled back at :380 **while the template already went out** → no unresolved row. Then the owner's "haan bhej do"
hits `runner.try_resume_pending_approval` → `find_open_for_tenant` returns None → the reply falls through to
`dispatch_brain` → re-spawns SR → re-emits the template; `pending_approvals.decision` stays NULL, **campaign never
sends.** (Also the queue-busy branch :246-264 writes no row.)

**(B) Brain blind to open approvals** — no `dispatch_brain` context block or tool references `pending_approvals` (grep:
zero refs across all context builders + `orchestrator_agent_system.md`), and `_classify_terminal` (`dispatch.py:1027`)
gives a re-drafted `campaign_plan` ABSOLUTE precedence → the collapse path overrides the whole reply with a fixed
queue_busy template (`dispatch.py:1354-1377`), silently discarding anything else the turn produced (e.g. a Sheets
answer). This is the load-bearing mechanism for the duplicate-reply + a second way the turn mishandles a busy state.

They interact: a compound ask that escalates without arming a durable row is one way to create (A)'s no-row precondition.

## Fix (plan-first, send/approval risk path)
- **(A)** Reorder `arm_pause_request`: INSERT the `pending_approvals` row **before** sending the template (arm-then-send),
  so a lost unique-race never leaves a sent template with no row; handle the queue-busy branch to not send an
  un-backed template. Guard the resume lookup / `find_open_for_tenant` timing.
- **(B)** Inject open-`pending_approvals` state into the brain's context (so it knows an approval is pending and does not
  re-spawn SR / routes the reply to resume), and fix `_classify_terminal`'s collapse-always-wins so a compound turn
  doesn't discard the brain's own answer. **NOTE:** enforce's `task_store.has_active_task`/`pending_questions` (no-op in
  legacy per `triage_seam.py`) is exactly this busy-awareness — the enforce re-run measures how much of (B) it already
  solves before we hand-patch legacy.

## (A) implementation design — ARM-THEN-SEND (build-ready, 2026-07-08 root-cause session)
`request_owner_approval.arm_pause_request` (src/orchestrator/agent/tools/request_owner_approval.py). Confirmed the
current order is: 0 idempotency → 0b queue-busy refuse → resolve owner_phone → **summary send (:289)** → **template
send (:307)** → **INSERT (:347)** → UniqueViolation refuse (:372, which literally calls the dropped campaign an
"accepted residual"). Also confirmed the arm path is LIVE on BOTH legacy AND the enforce manager workflow
(`request_owner_approval_node` persists into `pending_approvals` — workflow.py:223, message_ids.py:21), so (A) bites
regardless of dispatch mode. `owner_message_sid` is NOT load-bearing for matching (resolve COALESCEs it;
find_open matches on tenant/run) → safe to INSERT NULL then UPDATE. The conn is **autocommit** (the UniqueViolation
handler says so) → a committed INSERT can't be rolled back, only DELETEd.

**Reorder:** INSERT the row FIRST (owner_message_sid=NULL) → UniqueViolation refuses HERE, before any send (no
phantom template, no dropped campaign) → THEN summary send (unchanged, still before template) → THEN template send →
on template success UPDATE owner_message_sid → emit `approval_armed` after send-success.

**The ONE unresolved sub-question (resolve at build, do NOT hand-wave — it's why this wasn't rushed):** the
send-FAILURE cleanup. INSERT-first changes the documented contract "status='error' → NO pending_approvals row"
(:135-137) — on send-fail a row now exists and would block the one-open-per-tenant queue until the timeout sweep.
Three options, pick after verifying: (a) explicit `DELETE FROM pending_approvals WHERE tenant_id=%s AND id=%s` on
the caller conn — REQUIRES verifying the pending_approvals RLS policy permits tenant-role DELETE (insert works, so
likely, but VERIFY) ; (b) resolve the row to a terminal state via the tested wrapper (line ~924) — but check the
migration for the one-open unique-index condition (if it's `WHERE decision IS NULL`, setting decision frees the
queue) AND that a terminal state doesn't fire an owner "cancelled" notification (`_HANDLED_OUTCOMES` includes
'cancelled'); (c) leave it for the timeout sweep — simplest, self-healing, but blocks the tenant queue for
timeout_hours on a (rare) Twilio failure AND breaks the "error→no row" contract for any downstream reader. Find the
migrations dir + the migration-128 unique index + the pending_approvals RLS policy first; that decides (a) vs (b).

**Tests to add/adjust** (9 files touch arm_pause_request — run all): new `test_race_loser_refuses_BEFORE_any_send`
(insert_raises=UniqueViolation ⇒ status='refused' AND send_fn NOT called AND summary NOT sent); new
`test_template_send_failure_leaves_no_open_row` (send_fn fails ⇒ chosen cleanup ran, queue not blocked);
`test_chat_summary_sent_before_template` still passes (summary+template both after INSERT, order preserved).
