# VT-623 — the primary win-back delegation fails intermittently (3 heads)

The `plan_queue_new_specialist_while_busy` + `plan_queue_status_check_while_pending` scenario fails are
NOT scenario bugs — they are downstream of the manager's PRIMARY function (delegate a win-back campaign
to Sales-Recovery, arm the approval) failing intermittently. Measured on deployed dev (seeded onboarded
tenant, 8 customers / 6 lapsed): the SR ask "set up a win-back campaign for my lapsed customers" resolves
to one of THREE broken outcomes on a minority of runs, plus the correct outcome (drafted plan + armed
approval) on the majority.

## Head 1 — VT-490 redaction backstop RAISES → SR lane crash → human-escalation. ✅ FIXED (dev, 47bbe43)
`context_builder.py` (dormant-cohort prompt builder) RAISED a ValueError whenever a customer's
display_name/business_name carried an 8+ digit run (`\+?\d{8,}`) — common in real AND seeded data. That
crashed the whole sales_recovery lane → VT-602 → the owner got "needs a human" instead of a plan.
Observed: `ValueError: VT-490 redaction backstop: a phone-shaped value was blocked from the dormant-cohort
prompt section (CL-390)`. **Fix:** redact the run IN PLACE (phone never reaches the LLM, CL-390 intact;
customer stays targetable by customer_id) + warn on redaction. Backstop test updated to assert
redact-not-raise (9/9). Landed 47bbe43.

## Head 2 — SpecialistNoOutputError (status='invalid', no output). ✅ FIXED (dev, pending deploy)
Root cause (agent-traced, verified in code): `sales_recovery.py:906` — `_parse_placeholder_output(text)`
returns None when Sonnet-5's terminal turn wraps the CampaignPlan JSON in a preamble / trailing note /
UNANCHORED code fence (`_CODE_FENCE_RE` at :151 is whole-string-anchored; loose brace-extraction is
deliberately refused). This bites only when `stop_reason == "end_turn"` (a COMPLETE but prose-wrapped emit,
so the VT-596 #2 max_tokens continuation never fires). It was the ONE terminal path with no recovery —
every other failure mode already retries (schema VT-596 #3) or server-heals (VT-498/499/501). ~1/6 runs.
**Fix (landed dev, tests 49 passed):** a ONE-shot corrective retry gated to `stop_reason=="end_turn"` (mirrors
the VT-596 #3 schema retry) — re-prompt "re-emit ONLY the JSON, no prose/fence". Emit the FailureRecord +
go invalid ONLY on the second miss, so a RECOVERED run routes no false failure to the error router/alerts.
No loose brace-extraction; downstream validation/gates untouched. Bounded (one-shot flag + _MAX_TURNS=50).

## Head 3 — approval armed but owner gets D1 (enforce hand-off RACE). 🔍 root-caused, fix DEFERRED (risk)
NOT a summary-send skip (the prompt's hypothesis, DISPROVEN by the agent): the legacy arm path always
`interrupt()`s → `final_status="paused"`, and D1 requires `completed`, so legacy armed→D1 is impossible. The
co-occurrence is an ENFORCE-mode race: a win-back classifies `new_task` → `triage_seam` FIRE-AND-FORGETs an
async manager-task workflow (`DBOS.start_workflow`, workflow.py:1013) + returns `skip_legacy_dispatch=True`
→ the webhook run keeps `final_status="completed"` and the D1 check (runner.py:1206, VT-606 extended it to
the hand-off) fires BEFORE the async task records its owner reply → "I'm on it" races ahead of the real plan
summary. ~1/6 = scheduling jitter.
**Candidate fix:** `runner.py:1206` — add `and not skip_legacy_dispatch` to the D1 gate (the manager task
owns the reply). **DEFERRED — real risk:** this trades away the VT-606 "never leave the owner silent on an
enforce hand-off" net. The manager task's `paused_approval`/rejected/timeout terminals DO notify the owner
(`_notify_owner_of_terminal`, `_block_owner_unreachable`), but the `escalate` terminal (workflow.py:912)
just breaks with a VTR incident — NO owner-facing send. So the safe fix = gate D1 off `skip_legacy_dispatch`
AND add an owner ack to the manager task's escalate/block terminals. That's a careful change on the enforce
path (VT-606) — do it focused, not as a tail-end one-liner (rushing risks owner-silence-on-escalate, worse
than the current racing D1). Fully specified here for the follow-up.
