## What CoderC must produce
The per-tenant cost dashboard. Aggregates LLM token costs, external API costs, and infra costs from `pipeline_log` events. Surfaces in a Fazal-only dashboard accessible via Telegram bot command `/costs <tenant_id>` or `/costs all`.
1. **Cost attribution at source**:
	- Every external_api_call event in pipeline_log includes `cost_paise` field in payload.
	- LLM costs computed per-call: tokens used × model rate from `apps/team/config/model_pricing.yaml`. Pricing pinned per model; revisions are Type 1 governance with effective_at timestamp (so historical costs remain accurate).
	- Twilio per-message cost: from Twilio's pricing API (Phase 1 uses static config from VT-2.5; Phase 2 may sync periodically).
	- Razorpay fees: 2% of transaction value Phase 1 (their MDR; documented).
	- Resend cost: tiny; Phase 1 free tier; Phase 1.5+ tracks.
	- Apify cost: per-actor-run cost; static config in `model_pricing.yaml`.
2. **Aggregation module** at `apps/team/src/observability/cost_dashboard.py`:
	- `get_tenant_cost(tenant_id, since, until) -> TenantCostBreakdown`: aggregates pipeline_log events for the tenant in window; returns breakdown by category (LLM, Twilio, Razorpay, Apify, infra-allocated).
	- `get_workspace_cost_summary(since, until) -> WorkspaceCostSummary`: aggregates across all tenants; flags top-10 cost outliers.
	- Materialized view `tenant_cost_daily` (migration `039_tenant_cost_views.sql`) refreshed every hour for fast queries; raw events queried directly for sub-hour precision.
3. **Telegram bot commands** (extends VT-3.7's OpsAssistant):
	- `/costs <tenant_id>`: returns last-7-day breakdown.
	- `/costs <tenant_id> month`: last-30-day breakdown.
	- `/costs all`: top-10 cost-runaway tenants in last 7 days, with absolute spend + ratio against ARRR.
	- `/costs anomaly`: tenants whose 7-day cost is >2x their 28-day average.
4. **Cost-runaway alerts**:
	- Daily 9 AM IST scheduled trigger: scans for tenants whose 7-day cost > 50% of their plan's monthly fee (e.g., ₹1,250 cost on a ₹2,499 founding plan).
	- Telegram alert with breakdown; suggests investigation.
	- Hard cost ceiling: per-tenant per-run cost cap of ₹50 at agent dispatch time (enforced by VT-4.4 hard limits). Run terminated if exceeded; tenant flagged.
5. **ARRR-vs-cost view**:
	- `get_tenant_unit_economics(tenant_id, since, until)` returns `arrr_paise / cost_paise` ratio. Useful for refund calibration (day-39 evaluator) and ops investigation.
6. **Tests** at `apps/team/tests/observability/test_cost_dashboard.py`:
	- Synthetic tenant with 100 LLM calls + 20 Twilio sends → aggregate cost matches expected.
	- Per-category breakdown matches.
	- Top-10 outliers correctly ranked.
	- Anomaly detection: synthetic tenant with 7-day cost 3x of 28-day avg → flagged.
	- Cost-runaway threshold: synthetic tenant exceeding 50% of plan fee → alert fired.
	- Hard cost ceiling: VT-4.4 integration test confirms ₹50 dispatch cap enforces.
	- Materialized view refresh: synthetic events written; view refreshed; aggregate matches.
	- Cross-tenant: query for tenant_A doesn't include tenant_B costs.
## Architectural rules to enforce in this subtask
- Pillar 7: cost transparency. Tenants visible to Fazal; eventually visible to tenant via portal (Phase 2).
- Pillar 8: one dashboard, one aggregation module. CodeX rejects parallel cost calculators (e.g., per-feature shadow tracking).
- Cost data sourced from pipeline_log (single source of truth).
- model_pricing.yaml is versioned; updates carry effective_at.
- ₹50 per-run cost ceiling is a Phase 1 Type 2 commitment; raising affects unit economics.
- 50%-of-plan-fee runaway threshold is Type 2.

### Eight required assertions (all must PASS — any fail = canary FAILED; per-assertion verbatim evidence captured in `pre-merge-result` per the supplement-pattern VT-102 established)

1. **Seed real `pipeline_log` events succeed.** Insert ~50 synthetic `external_api_call` events for canary tenant_A across 5 categories (LLM-Anthropic, Twilio, Razorpay, Resend, Apify) with varying `cost_paise`. Verify via direct SELECT that 50 rows exist with the canary's run_id markers.

2. **`get_tenant_cost()` against real DB returns expected aggregate.** Call `get_tenant_cost(canary_tenant_A, since, until)`. Assert: the returned `TenantCostBreakdown` has totals matching the seeded data within ±1 paise. Per-category breakdown matches per-category sums of the seeded events.

3. **Materialized view refresh works end-to-end.** `REFRESH MATERIALIZED VIEW CONCURRENTLY tenant_cost_daily`. Then SELECT from the view for canary_tenant_A in the day window. Assert: rows exist; aggregate matches the raw-event sum from #2.

4. **Cross-tenant isolation under real RLS.** Insert 20 more cost events for canary_tenant_B. `get_tenant_cost(canary_tenant_A, ...)` returns the original tenant_A total (NOT inflated by tenant_B). `get_tenant_cost(canary_tenant_B, ...)` returns tenant_B's total. Both verified via direct SQL counts.

5. **Top-10 outliers correctly ranked.** Seed 12 synthetic tenants with deliberately varied costs (₹50 / ₹100 / ₹200 / ... / ₹600 per tenant). Call `get_workspace_cost_summary(since, until)`. Assert: top-10 returned IS the 10 highest-cost tenants in descending order; 2 lowest-cost are excluded.

6. **Anomaly detection flags the right tenant.** Seed canary_tenant_anomaly with 28-day baseline cost X paise/day for days -28 to -8, then 3X paise/day for days -7 to today. Call the anomaly-detection function. Assert: canary_tenant_anomaly is flagged; canary_tenant_A (steady cost) is NOT flagged.

7. **ARRR-vs-cost ratio calculation matches.** Seed canary_tenant_unit with ARRR rows (campaigns table, not pipeline_log) totaling ₹1,000 and cost events totaling ₹250. Call `get_tenant_unit_economics(canary_tenant_unit, since, until)`. Assert: returns ratio = 4.0 ± 0.01.

8. **Cleanup is real-DB cleanup (best-effort, NOT a skip).** Canary cleanup at exit deletes all `component='canary'`-tagged rows + `pipeline_log` rows under canary run_ids + campaigns rows under canary tenant_id. Run a final SELECT COUNT(*) for residual canary rows; assert count is 0 OR log a warning with the count if cleanup partially failed (residuals get swept by 90-day retention from VT-102; canary's own pass/fail does not depend on cleanup). The cleanup attempt itself counts as a real-DB action — assertion verifies it ran AND its outcome.
