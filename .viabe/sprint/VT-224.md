# VT-224 — Admin HTTP endpoints suite

## Why this row exists

VT-212 walk kept hitting "needs Railway shell to invoke Python method" — `setup_push`, `pull_sample`, `tenant_oauth_tokens` shape inspection, etc. Shell access is a high-friction operator path; doesn't scale to multi-tenant production where ops needs to troubleshoot integrations safely.

This row ships an admin HTTP endpoint suite that exposes connector lifecycle + diagnostic operations behind a single admin token, callable from anywhere with `curl`.

## Scope

### A. Admin auth substrate

1. New env `TEAM_ADMIN_API_TOKEN` (Fazal drops a 32-byte token into Railway + `.viabe/secrets/supabase-dev.env`)
2. `apps/team-orchestrator/src/orchestrator/api/admin/_auth.py` — middleware that requires header `X-Team-Admin-Token` matching env. 403 on mismatch.
3. All admin routes mounted under `/api/orchestrator/admin/...`

### B. Endpoints

4. **`POST /admin/connector/setup_push`** — body `{tenant_id, connector_id, spreadsheet_id?}`. Calls connector's `setup_push` (or successor in VT-222 = `register_drive_push_channel`). Returns channel descriptor + script/config blob (where applicable).

5. **`POST /admin/connector/pull_sample`** — body `{tenant_id, connector_id, spreadsheet_id?, range?}`. Calls `pull_sample`. Returns row + col count + column headers + first-row preview. Limited to 10 rows max.

6. **`GET /admin/connector/token_shape?tenant_id=...&connector_id=...`** — returns the `tenant_oauth_tokens` row shape (refresh present? scope? expiry?). NEVER returns raw token values; only shape.

7. **`GET /admin/connector/drive_channels?tenant_id=...`** — returns active Drive Push channels for the tenant (VT-222 substrate).

8. **`POST /admin/connector/drive_channels/{channel_id}/renew`** — manually renew a Drive Push channel.

9. **`POST /admin/workflow/replay`** — body `{workflow_id, run_id?}`. Replays a DBOS workflow execution from its last persisted step. Useful for re-trying failed ingestion runs.

10. **`GET /admin/health/integration_agent`** — returns connector substrate health: # active oauth tokens, # active Drive channels, last ingestion timestamp per tenant.

### C. Rate limit + audit

11. **Rate limit: 10 req/sec per admin token.** Reject 429 if exceeded.
12. **Audit log: every admin call writes to `admin_audit_log` table** with timestamp, endpoint, tenant_id, source IP. New migration `039_admin_audit_log.sql`.

### D. Documentation

13. **`docs/clau/admin-endpoints-runbook.md`** — table of endpoints with curl examples. Token rotation procedure. Audit log query examples.

### E. Substrate hardening for VT-222 + future

14. After this row lands, future connector lifecycle work uses these endpoints, never Railway shell. Adopt as Standing pattern.

## Acceptance criteria

1. All 7 endpoints respond with 200 to valid admin token + reject 403 on missing/wrong token
2. `setup_push` endpoint returns Apps Script body (or post-VT-222 Drive channel descriptor) — confirms parity with previous shell-exec workflow
3. `pull_sample` endpoint returns row + col counts only; no raw cell values
4. Rate limit triggers 429 on burst > 10 req/sec
5. Audit log row created for every admin call
6. Endpoints work against the running Railway deployed orchestrator (no shell required)
7. Runbook covers all endpoints + token rotation

## Rule #15 Canary

`apps/team-orchestrator/canaries/vt224_admin_endpoints.py` — assertions:
- A1: Missing token → 403
- A2: Wrong token → 403
- A3: Valid token + setup_push → 200 + valid response shape
- A4: Valid token + pull_sample → 200 + row count + col count fields present
- A5: 11 requests in 1s → at least 1 returns 429
- A6: Each successful call writes audit log row

## Out of scope

- Web UI for admin operations (admin endpoints are CLI-first for Phase 1)
- Bulk operations (per-tenant only)
- Connector listing (separate row when needed)
- Admin role separation (single shared token for Phase 1; multi-admin = Phase 2)
