# CL-210 — spawn_sales_recovery handoff guards

=== TENANT_ID SOURCE GREP RESULT ===
build_sales_recovery_context(tenant_id, run_id, trigger_reason) at context_builder.py:181 takes tenant_id as a parameter. SalesRecoveryContext(tenant_id=tenant_id, ...) at context_builder.py:261 echoes it directly to bundle. Spawn tool would call build_sales_recovery_context with tenant_id=state['tenant_id']. Therefore bundle.tenant_id == state['tenant_id'] always — the originally-briefed guard 'assert state['tenant_id'] == bundle.tenant_id' compares a value to itself. Outcome (a) from the seam-gate report: tautological.

=== WHY THE TAUTOLOGY IS NOT ACCEPTABLE ===
Zero signal. Worse than no guard: creates false sense of testing. 'I have a cross-tenant assertion in the spawn tool' sounds like Pillar 3 work, reads like Pillar 3 work in PR review, contributes nothing in practice. Pattern Claude is allergic to per CL-183 (adjacent verification anti-pattern) — same failure class, applied to assertions instead of probes.

=== REAL PILLAR 3 LANDING POINTS ===
Cross-tenant row leakage is covered by *tenant_guard.assert_tenant_scoped inside each _build_* function on raw DB rows. Currently dormant (substrates absent per CL-190), wired when L1/L2 land. This is the architecturally correct location for cross-tenant assertions — close to the data, not at handoff boundary.

What the handoff tool CAN guard meaningfully: the case where state was somehow built without tenant_id or run_id. Configuration error, upstream propagation bug, future regression. CL-195 put these fields in STATE precisely so missing-ness can be detected fail-loud at use sites.

=== GUARD RESTRUCTURE ===
Drop tautological compare. Replace with:

    if state.get('tenant_id') is None:
        raise TenantIsolationError(
            "spawn_sales_recovery: tenant_id missing from state"
        )
    if state.get('run_id') is None:
        raise TenantIsolationError(
            "spawn_sales_recovery: run_id missing from state"
        )

trigger_reason does NOT get a None check — it has explicit read-site fallback per CL-195 ('weekly_cadence' literal). Different field, different policy by design.

=== RATIONALE ===
- Real signal: catches the propagation-failed case (which is exactly what the Step 6 seam gate today averted from happening silently in production).
- Fail-loud: TenantIsolationError, not silent None propagation.
- Pairs correctly with assert_tenant_scoped for downstream row-level work.
- Aligned with CL-195 (state-as-truth for these fields) and Pillar 3 (no cross-tenant leakage, no silent tenant absence).
