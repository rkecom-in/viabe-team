# VT-343 — SupportBot Phase 2 (deferred from VT-88 Phase 1)

VT-88 Phase 1 (#357) closed the silent-drop on the CLEAR unresolved terminals
(aborted_hard_limit / escalated): every unresolved terminal acks the owner, the 2nd+ in 24h
escalates to Fazal (deterministic counter). Phase 2:

## Scope
1. **completed-no-send (LAUNCH-RELEVANT residual):** a run ending status='completed' with ZERO
   owner-facing output is the SAME silent-drop — Phase 1 doesn't catch it (it keys on the
   terminal status, not the send outcome). Needs per-run send tracking (an outbox/send signal
   correlated to run_id) → then count it as unresolved too. **The owner still gets silence on
   this path** until it lands — a real launch-relevance, not just nice-to-have.
2. **SLA enforcement:** 4h business-hours (10am–7pm IST) / 24h otherwise; a second Fazal alert
   on breach. Unblocks the ack copy promising a time ("within Xh").
3. **Fazal /resolve command:** mark an escalation resolved + send `support_resolved` to the
   owner (or skip with `--no-followup`). + the `support_resolved` template (NEEDS-FAZAL).
4. **3rd-escalation fatigue flag:** 3+ escalations in 7 days → the alert mentions "consider
   proactive outreach".

## Hardening nits (Cowork VT-88 #357 subagent, 20260606T010500Z)
A. **Duplicate Fazal alert on DBOS replay** — `_alert_fazal_safe` isn't gated on
   `record_escalation`'s insert + isn't a DBOS step, so a workflow replay re-fires the Telegram
   ping. Bounded (duplicate ping to FAZAL, not the owner — no Pillar-7 owner violation). Fix:
   have `record_escalation` return an inserted-bool; gate the alert on it.
B. **Test gaps:** `sender_phone=None` ack-target (falls back to the tenant whatsapp number —
   benign for owner-surface) + the idempotency / double-escalate (same run reprocessed) path.
