# CL-333 — DECISION (Clau-owned, DB schema within locked stack): VT-7.1-R l1_entities pgvec

DECISION (Clau-owned, DB schema within locked stack): VT-7.1-R l1_entities pgvector index — HNSW, vector_cosine_ops opclass, built in the SAME PR/migration as the table. Rationale: (1) IVFFlat's lists parameter is untunable against an empty table — picking it bakes in a guess for non-existent data; HNSW has no such parameter. (2) Deferring the index (Claude Code option 3) would mean the VT-7.1-R recursive-CTE traversal + multi-signal retrieval code gets developed and CI-tested against exact-distance-scan behavior that silently changes when the index later lands — same class of testing-validity gap as the #39/#40 dead-canary thread. Index ships with the table so CI exercises production behavior. (3) Opclass vector_cosine_ops is a REQUIREMENT not a default: voyage-4-lite embeddings are cosine-similarity designed; the retrieval query must use the cosine distance operator (<=>) to match the index opclass — index/query operator mismatch is a silent index-not-used bug. ALSO: confirmed Claude Code's Step-0 path-drift catch — brief said apps/team-orchestrator/migrations/ which does not exist and is forbidden by migrations/README.md (Pillar 8: one migration history, one runner). Correct path is repo-root /migrations/, runner apps/team-orchestrator/scripts/apply_migrations.py. Brief path was a Clau error; Claude Code Step-0 verification caught it correctly. This sets the index pattern future pgvector tables inherit.
