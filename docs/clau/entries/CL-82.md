# CL-82 — RLS canonical mechanism is GUC, not auth.jwt()

DECISION (Fazal 2026-05-18, in response to audit findings CL-71 + CL-77): GUC is the canonical RLS-tenant-resolution mechanism for the backend runtime. NO JWT anywhere in our backend path.

Rationale: In Phase 1 (and probably never), no client touches Postgres directly. team-web client uses Supabase Auth for read-only frontend operations against the Supabase publishable key; all backend operations route through the orchestrator (Python) and ingestion-worker (Python), both of which connect with a dedicated Postgres role and set the tenant context via `SELECT set_config('app.current_tenant_id', <tenant_id>, false)` on connection checkout. The orchestrator never sees a Supabase Auth JWT.

Fix PR scope (will be the C1+DC4 root-cause PR):
1. Replace every RLS policy that uses `auth.uid()`, `auth.jwt()`, or claim-extraction patterns with `current_setting('app.current_tenant_id', true)::uuid`. Audit shipped migrations (000a-013) — the helper `app_current_tenant()` already does this; check all policies use it consistently.
2. Introduce a typed connection wrapper in apps/team-orchestrator (e.g., `tenant_connection(tenant_id) -> ContextManager[Connection]`) that sets the GUC on checkout and resets on return. Every @DBOS.step that writes a tenant-scoped table uses it.
3. _lookup_tenant (cross-tenant by design) and rate_limit reads/writes against the workspace-sentinel bucket use an explicit service-role connection helper (no GUC) and document why.
4. End-to-end test: create a non-superuser Postgres role, set the GUC, run apply_transition, assert the row was written. This is the test that proves the production runtime works. The current 63 orchestrator tests run as superuser — they wouldn't catch GUC absence.

Notion: update VT-20 (32 RLS policies + 7 attack tests) body to reflect GUC pattern and mark JWT-based pattern as superseded. The 7-attack-test count and structure remain accurate post-update.

SUPERSEDES: Notion VT-20 spec language about `auth.jwt()->>'tenant_id'::uuid` as canonical. Per concept-team-pillars.md Pillar 3 spirit, RLS enforcement is structural — the mechanism (JWT vs GUC) is implementation detail, but documentation must match reality.
