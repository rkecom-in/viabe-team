# VT-195 — L1 tenant context substrate (moat anchor)

## Why this row exists

Per Fazal directive 2026-05-27: the non-negotiable product moat is "agents change and perform based on type of business, owner, applications used, etc. Something that makes them impossible to replicate."

L0 memory (workspace-wide cohort patterns; VT-126) exists. L2-L4 memory is Mem0-deferred per CL-324. The missing tier is **L1 — per-tenant knowledge graph** that the Context Composer reads when assembling the orchestrator-agent's context bundle. Without L1, the agent is generic; with L1, it's specifically calibrated to RKeCom (or any tenant).

VT-7 ("Knowledge Architecture") is the parent epic — Backlog at Sprint 7. VT-195 advances the L1 portion to Sprint 2 because the moat won't function without it.

## Pre-implementation contract

**Concept doc gates the brief.** `docs/clau/l1-tenant-context-design.md` (Cowork-drafted, Fazal-reviewed) lands BEFORE this row gets a brief-ready signal. The doc covers:

1. L1 schema — structured columns for predictable per-tenant attributes (business_archetype, owner_persona summary, integration_map, working_hours, escalation_thresholds, communication_preferences) + JSONB `owner_curated_context` field for free-form notes
2. Derived stats refresh logic (drop_off_history, customer_response_rate, attribution_summary — nightly batch from existing observability data)
3. Schema validated against 3 archetypes (RKeCom electronics + 2 hypotheticals)
4. Context Composer read path: how `assemble_context_bundle(tenant_id)` queries L1 and merges into agent invocation context
5. Write path: who updates L1 (Cowork-curated initial seed + agent-curated updates via L0 → L1 promotion + owner-curated explicit updates via dashboard)
6. RLS + tenant isolation contract
7. Coexistence with L0 (workspace-wide) — L1 reads are tenant-scoped only

Concept doc lands as `docs/clau/l1-tenant-context-design.md`. Fazal reviews. Then VT-195 brief-ready dispatch with concrete schema + acceptance criteria.

## Scope (provisional — finalized post concept-doc)

- Migration: new `tenant_l1_context` table + RLS policies
- Backend: `assemble_context_bundle(tenant_id)` helper reads L1 and produces context block for agent invocation
- `dispatch_brain` integration: pass L1 context block as system message content
- Seed data: Cowork drafts initial RKeCom L1 row (Fazal reviews + edits)
- Dashboard surface: read-only view of L1 context (write surfaces in VT-198 owner-feedback work)

## Rule #15 Canary (mandatory, finalized post concept-doc)
