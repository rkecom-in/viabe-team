## Concept
A single migration runner connects to whatever `DATABASE_URL` is injected (dev = Seoul, prod = Mumbai
via `railway run --environment`). A wrong-env run is the catastrophic failure (dev seed → prod real-PII
DB, or migrations at the wrong target). VT-362 makes that structurally impossible — the runner must
state its intended env and the connected DB must prove it matches before any migration applies.

## Requirement (Fazal hard requirement, CL-431 Refinement E)
`apply_migrations.py` must REFUSE to apply unless the connected database matches an explicitly-passed
expected environment. No silent apply.

## Invariants (implemented)
1. Required `--expected-env {dev|prod}` (or `EXPECTED_ENV`). No default — missing aborts (exit 2).
2. Sentinel table `app_environment` (one row, singleton-enforced, name ∈ {dev,prod}). Steady-state:
   read + assert `name == expected-env`, abort BEFORE any migration on mismatch (exit 3).
3. Bootstrap (fresh DB, no sentinel): proceed only when `--expected-env` is explicit AND a non-secret
   host-substr identity check passes (print the connection host — host is not the password — and
   require it contain the expected substring). The guard then creates + stamps `app_environment`.
4. Never print the DSN/password — only host + sentinel value + applied count (CL-431).
5. Idempotent + forward-only preserved (`schema_migrations` tracking unchanged).

## Acceptance (dev-only — proven)
- `--expected-env prod` against a dev-stamped DB → ABORTS (sentinel mismatch), 0 migrations applied.
- Missing `--expected-env` → ABORTS (exit 2).
- `--expected-env dev` against dev → proceeds (bootstrap stamp + apply).
- Bootstrap without `--expected-host-substr`, or with a wrong substr → ABORTS.
- Programmatic `apply()` (no expected_env) still works for test fixtures; never stamps the sentinel.

## Out of scope (VT-231 cutover)
First guarded Mumbai prod run (`--expected-env prod`, host-ref check, stamps sentinel='prod').
