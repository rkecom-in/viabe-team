# VT-302 — concession DETECTION → request_owner_approval (VT-268's deferred half)

## CLOSURE (2026-06-03, Fazal)
Closed 2026-06-03 (Fazal). Superseded by VT-268's capability-lock: the agent provably cannot
send-to-customer / offer a price concession (no such tool; raises at graph build). Standalone
discount-detection is therefore unnecessary now. **Revisit ONLY if a 1:1 customer-send path is
later added** — at which point the discount→`request_owner_approval` gate must be built as part
of that work (don't ship a 1:1 send capability without it).

## Why
VT-268 (PR-D, merged) LOCKED the capability boundary: the agent holds no send-to-customer /
accounts-book-write / ledger-write tool (fail-closed runtime guard), so it cannot send a discount
to a customer at all today. That makes the boundary SAFE now. The remaining half — actively
DETECTING a discount/price-concession in agent output and routing it through
`request_owner_approval` (VT-47) — was deferred because:
- there is no agent-reachable 1:1 customer-send path today (the send tools are not on the agent
  surface), and
- campaign plans carry no discount field, and
- concession detection needs owner-set discount TERMS, which is the VT-195-phase-3 owner-terms UX
  that does not exist yet.

## What (when unblocked)
- Owner-set discount/concession terms (the VT-195-ph3 owner-terms surface).
- Detect a proposed concession (campaign plan discount field and/or a future 1:1 concession-send
  path) and route it through `request_owner_approval` — fail-CLOSED, conservative inference:
  when uncertain whether something is a concession, GATE it (Cowork ruling 2026-06-03).
- Canary: a proposed concession cannot reach the customer without an owner `approved` resume.

## Out of scope
- The capability lock (done in VT-268).

## Blocked-by
VT-195 phase-3 owner-terms UX + an actual agent-reachable customer-send / campaign-discount field.
Roster-only; do NOT build until those land (Cowork tracked-follow-up, 2026-06-03).
