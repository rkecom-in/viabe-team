# VT-617 — tool-call hard limit was truncating conversational multi-tool turns

## Root cause
The orchestrator-agent tool-call hard limit was **5** (`orchestrator_agent_driver.py:41`, a VT-125
"thin router" assumption). Under CL-443 (conversational-primary), `dispatch_brain` (route:none, ~90%
of turns) is the PRIMARY surface and legitimately does inline multi-tool work — a multi-field onboarding
message needs `read_onboarding_state + record_answer×3 + next_required_question` (6 calls). At 5 the run
truncated mid-save (`HardLimitExceeded tool_calls observed=6 limit=5`) → owner saw a "hiccup saving" snag
that then repeated (the multi_field stuck-loop, judge progression=2).

**Second bug (the reason the first raise did nothing):** `dispatch_brain`'s callback enforces via
`_NullDriver` (`dispatch.py:1662`), whose limits are a **SECOND copy** read from env with a **hardcoded
"5" default** — a DRY violation that SHADOWED the module constant. Raising the constant to 10 alone left
the brain path still tripping at 5 (verified: deployed 6774305 kept tripping `limit=5` at 11:44-47).

## Fix (landed — commits 6774305, 7ba6962 on dev)
1. `ORCHESTRATOR_TOOL_CALL_HARD_LIMIT = 5 → 10` (driver constant). Runaway still bounded by the token
   (10k), wall-clock (120s), cost (₹5), depth (3) guards — tool-call COUNT was the redundantly-tight axis.
2. `_NullDriver` env defaults now reference the module constants (single source of truth) so the two
   copies can never diverge again.

## Verified
Deployed 7ba6962, re-ran subset3: **zero tool_call HardLimit trips** (was tripping every multi-field turn).
The tool-call truncation is fixed.

## NOT fixed by this (see VT-618)
Raising the tool-call limit shifted the binding limit to **cost** (`cost_paise=518>500`). The onboarding
conductor still fails — but now via a mid-loop cost-abort that orphans a `tool_use` → 400 → deterministic
floor. That is a SEPARATE abort-handling bug: **[[VT-618]]**.
