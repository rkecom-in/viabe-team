# VT-230 — Fix /team/onboard unauthorized redirect target

## Why

Surfaced by Fazal manual walk 2026-05-29 16:50 IST against dev env (`viabe-team-dev.vercel.app/team/onboard`). Unauthenticated visit returns 404.

Root cause: `apps/team-web/app/(app)/team/onboard/page.tsx:25` catches `UnauthorizedError` and calls `redirect('/login')`. The `/login` route does not exist. The Ops Console magic-link login lives at `/team/ops/login` (VT-203 shipped that path).

Result: legitimate onboarding flow is blocked at the front door. Any non-authenticated visitor (including Fazal himself before signing in) gets a 404 instead of a login screen.

## Three options

**Option A — point onboard to `/team/ops/login` (recommended)**
- Single login surface — magic link → JWT cookie path-scoped to `/team/ops`
- Cookie path needs to widen to `/team/*` OR onboard route gets moved under `/team/ops/onboard`
- Tradeoff: widening cookie path crosses into VT-203 LOCK 2 territory; better to move onboard

**Option B — create new top-level `/login` route at `app/(marketing)/login/page.tsx`**
- Marketing-style login — simple email + magic link, no ops-console framing
- After auth → redirect to `/team/onboard`
- Cleaner separation; doubles as a future tenant-onboarder login path
- More code (new route + new callback + new cookie config)

**Option C — change onboard redirect to use existing `/team/ops/login` even without cookie-path-widening**
- Magic link callback would set viabe_ops_jwt with path=/team/ops, then redirect to /team/onboard
- /team/onboard's requireFazal reads viabe_ops_jwt — but path=/team/ops means cookie won't be sent for /team/onboard requests
- **Doesn't work without widening cookie path** — same blocker as A

## Brief contract — choose Option A with cookie path widening

CC implements:

1. **VT-203 cookie path adjustment** at `apps/team-web/app/api/ops/login/callback/route.ts`:
   ```ts
   // OLD: path: '/team/ops'
   // NEW: path: '/team'
   ```
   This makes the JWT cookie available across `/team/ops/*` AND `/team/onboard` AND `/team/dashboard/*`.

2. **`/team/onboard/page.tsx` redirect target**:
   ```ts
   // OLD: redirect('/login')
   // NEW: redirect('/team/ops/login?next=/team/onboard')
   ```

3. **`/team/ops/login` page** accepts `?next=<path>` query param and forwards it to callback. Callback redirects to `next` after issuing JWT (validated against allowlist `/team/ops`, `/team/onboard`, `/team/dashboard` only — open-redirect defense per VT-203 LOCK 1).

4. **Update `(app)/team/dashboard/page.tsx`** + any other `requireFazal()`-using route — same redirect pattern.

5. **Canary `apps/team-orchestrator/canaries/vt230_onboard_auth_flow.py`**:
   - A1: GET `/team/onboard` unauthenticated → 302 to `/team/ops/login?next=/team/onboard` (NOT 404)
   - A2: full flow — login → callback → redirect to `/team/onboard` → 200 with operator JWT cookie
   - A3: invalid `next` param → callback ignores it, redirects to default `/team/ops`

## Acceptance

- `/team/onboard` unauthenticated → 302 not 404
- After login + callback → land on requested page
- Cookie path widened to `/team` covers ops + onboard + dashboard
- Canary 3/3 PASS

## Out of scope

- Multi-operator login (covered by VT-228)
- Tenant-onboarder login (different user class; future row)
- Marketing-style `/login` page (Option B — deferred)

## Scope locks

- **DO NOT** create a separate `/login` route at root — preserves single-source-of-truth for auth surface
- Open-redirect allowlist for `?next` param is mandatory (per VT-203 LOCK 1)
- Cookie path `/team` not `/` — keep auth surface tight, don't widen to /api/*
