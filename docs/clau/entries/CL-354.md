# CL-354 — BLOCKER (VT-150 / PR #47 fix-1 deviation #1): Claude Code moved register_purge_s

BLOCKER (VT-150 / PR #47 fix-1 deviation #1): Claude Code moved register_purge_scheduler() to AFTER launch_dbos() in main.py, labeling it 'production-equivalent (queued vs immediate poller start)'. This is a rationalization that walks back the original VT-150 architecture. The whole point of the lifespan-import design was to register the @DBOS.scheduled workflow BEFORE DBOS.launch() so it is included in the launch-time app_version hash (DBOSRegistry.compute_app_version); recovery filters pending workflows by app_version (_recovery.py get_pending_workflows). Register-after-launch means the launch-time app_version did NOT include the purge workflow — registry and launched instance disagree. 'Queued vs immediate' is not the load-bearing difference; 'in the launch-time hash vs not' is, and that is what the deviation note glosses. ROOT of the deviation: Claude Code hit a real DBOS test-fixture bug (_destroy clears _executor_field but not _launched — stale registry across pytest fixture cycles; register-before-launch then hits a 'launched' branch submitting to a None executor and raises). Correct fix is in the FIXTURE TEARDOWN (clear _launched, or fresh instance per test) — NOT bending production main.py to dodge a test crash. Tail-wagging-dog. OPEN VERIFICATION before this is final: whether @DBOS.scheduled workflows are even subject to the recovery app_version filter — if scheduled workflows are exempt, deviation #1 is ugly but benign. Need updated main.py + substrate conftest fixture to confirm. PR #47 HELD pending that verification. If app_version filter applies to scheduled workflows: deviation #1 must be reverted, fixture teardown fixed instead.
