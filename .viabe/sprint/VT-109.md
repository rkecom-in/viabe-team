- Razorpay account in Live mode requires merchant KYC: PAN + GST + bank account + business proof. Fazal owns this; outside engineering scope but blocking.
## What CoderC must produce
Verification that the Razorpay integration works end-to-end in Live mode. Sandbox testing happens during VT-10.1 development; this subtask validates against the actual Razorpay Live API with real money flows (small test transactions to Fazal's own account).
1. **Razorpay Live activation**:
	- Fazal completes Razorpay merchant KYC: PAN, GST, current account proof, business registration proof.
	- Razorpay sales team activates Live mode (typically 3-7 business days from submission).
	- Live API keys received: `RZP_KEY_ID_PROD` + `RZP_KEY_SECRET_PROD`.
	- Webhook URL whitelisted: `https://viabe.ai/api/team/razorpay/webhook` registered in Razorpay dashboard with `RZP_WEBHOOK_SECRET_TEAM_PROD`.
2. **End-to-end Live verification**:
	- 3 plan IDs created in Razorpay Live dashboard: `FOUNDING_RZP_PLAN_ID_PROD`, `STANDARD_RZP_PLAN_ID_PROD`, `PRO_RZP_PLAN_ID_PROD` with prices ₹2,499 / ₹4,999 / ₹9,999 respectively.
	- Test subscription: Fazal subscribes via the production sign-up flow with a real card; verifies webhook fires with correct event types.
	- Test refund: Fazal triggers a refund via VT-10.5 refund executor; verifies refund processed in Razorpay Live + funds returned within 5 business days.
	- Test cancellation: subscription cancelled mid-cycle; verifies prorated billing OR full-period charging per chosen Razorpay subscription mode (Phase 1: full-period, no proration).
3. **Production webhook verification**:
	- Razorpay's Live webhook signature uses different secret from sandbox; environment-specific config from VT-10.1.
	- Test signature with synthetic + real Live events.
	- Verify idempotency: replay a real webhook event manually via Razorpay dashboard's "Resend webhook"; second delivery is idempotent.
4. **Compliance verification**:
	- Razorpay merchant agreement signed (Fazal owns).
	- PCI-DSS SAQ-A self-assessment completed by Fazal (per VT-10.3 PCI posture document).
	- GST invoicing configured in Razorpay (subscription invoices include GST line item per Indian tax rules).
5. **Production cutover checklist** at `docs/runbooks/razorpay-live-cutover.md`:
	- Live API keys in Vercel/Railway prod env (separate from dev env).
	- Plan IDs in `apps/team/config/plans.yaml` for production.
	- Webhook URL registered + secret rotated.
	- First 5 production subscriptions monitored manually via Telegram alert.
	- Rollback plan documented (revert env vars to sandbox; cancel any test subscriptions).
6. **Tests** at `apps/team/tests/billing/test_razorpay_live.py`:
	- These are NOT unit tests — they require real Razorpay Live keys. Excluded from CI; run manually pre-launch.
	- Test subscription creation with Fazal's real card.
	- Test refund + cancellation flows.
	- Test webhook idempotency on real events.
	- Test GST invoice generation.
## Architectural rules to enforce in this subtask
- Pillar 7: production financial flows non-negotiable. Launch gated on Live verification.
- Sandbox vs production strict separation; CodeX confirms env-var configuration.
- KYC + merchant agreement out of CoderC scope; Fazal owns.
- Live verification tests excluded from CI (real money flows); manual pre-launch only.
- Cutover runbook + rollback plan documented.
