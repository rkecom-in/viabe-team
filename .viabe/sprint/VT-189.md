# VT-189 — VT-OpsConsole-V2 (Full admin control panel)

## Why this parent exists

Per Fazal direction (2026-05-26): the Ops UI vision is much larger than VT-123's launch-essential MVP. The full admin control panel must allow Fazal to **monitor and control behavior, resolve issues, override, debug, trace, track, train, and empower owner agents and flows** across all tenants. This is a Phase-2 build that ships AFTER the Reports-Jun15 launch gate is cleared by VT-123 MVP.

VT-123 (existing row, reframed to Sprint 1 MVP scope) ships the launch-essential subset (3 views + [resolve] button) so the launch milestone is hit. VT-189 is the multi-sprint parent for the full vision.

## Vision

This is a parent row; sub-tasks will be enumerated as the design firms up. The capability surface (in rough scope-priority order):

1. **Debug** — full step-by-step pipeline replay with input/output envelopes per step, tool-call inspection, agent reasoning trace, error envelope drill-down.
2. **Trace** — cross-run correlation: "find all runs that hit this step_kind and failed in the last 7 days." Logfire trace-id integration for OTel-grade tracing.
3. **Track** — owner-level + tenant-level + campaign-level KPIs (success rates, cost per resolution, attribution rates, escalation rates). Trending over time.
4. **Train** — corrective feedback loop: when an agent decision was wrong, admin marks it + the feedback enters L1 KG (or a structured-feedback table) so future agent runs improve.
5. **Override** — manual intervention surface: pause an in-flight run, redirect a step, force-set an attribution, manually unblock a stuck DBOS workflow, manually trigger a refund/escalation.
6. **Monitor** — live dashboard of in-flight runs across all tenants. Health signals (DBOS workers, rate-limit consumption, cost burn-rate, error spikes, Anthropic 429s).
7. **Resolve** — full PII unmask path (operator-JWT) for issue triage; full DSR ticket workflow; manual cleanup tooling.
8. **Empower** — per-tenant agent configuration UI: which agents are active, which tools they have, custom prompt overrides, owner-input policies, escalation thresholds.

## Architectural rules

- **NO bypass of orchestrator pillars.** Override actions go through the same RLS / audit / cost-accounting paths the agents go through; the admin UI is a privileged client, not a back-door.
- **Pillar 7 (Consumer trust) — every override + resolve is audit-logged.** No silent admin actions.
- **Pillar 3 (tenant isolation) preserved.** Admin user with cross-tenant visibility must explicitly select a tenant context per action; cross-tenant queries (e.g., aggregate metrics) are read-only and logged.
- **CL-88 client-direct pattern** — admin UI reads heavy data via JWT-direct (low latency); writes/overrides go through backend API (for transaction integrity + audit).
- **Pillar 8 (single source of truth)** — admin UI does NOT cache; reads fresh from canonical tables. Logfire is the canonical trace store; pipeline_steps is the canonical step store; no parallel admin-only state.
