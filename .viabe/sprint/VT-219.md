# VT-219 — NEXT_PUBLIC_SUPABASE_* naming consistency

## Why this row exists

Fazal flagged (2026-05-28 18:55 IST) during VT-218 Vercel env-var setup: three call sites disagree on env var names.

- `apps/team-web/lib/supabase-client.ts` → `NEXT_PUBLIC_SUPABASE_URL` + `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` (load-bearing — auth + Supabase client)
- `apps/team-web/lib/ops/stream.ts` → same as above (load-bearing — Realtime stream)
- `apps/team-web/app/api/team/health/route.ts` → `NEXT_PUBLIC_TEAM_SUPABASE_URL` (one isolated health endpoint)
- `docs/clau/dev-env-runbook.md` Vercel matrix → documents the `TEAM_` variant — wrong for majority of call sites
- `.viabe/secrets/supabase-dev.env` → uses `NEXT_PUBLIC_SUPABASE_*` (matches the load-bearing call sites)

Inconsistency surfaced when Fazal was populating Vercel env vars and saw the runbook mismatch the env file. Fazal would have set the wrong variant in Vercel, then `requireFazal()` + Realtime stream would have failed silently at runtime.

## Scope

Standardize on `NEXT_PUBLIC_SUPABASE_URL` + `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` (no `TEAM_` prefix). Reasons:
- Matches the env file that's already populated
- Matches the load-bearing call sites
- Minimizes churn (fixes 1 endpoint + 1 doc vs renaming 2 load-bearing modules + the env file)

### Files to edit

1. **`apps/team-web/app/api/team/health/route.ts`** — replace `NEXT_PUBLIC_TEAM_SUPABASE_URL` references with `NEXT_PUBLIC_SUPABASE_URL`. Check for `NEXT_PUBLIC_TEAM_SUPABASE_PUBLISHABLE_KEY` or similar — if present, rename too.
2. **`docs/clau/dev-env-runbook.md`** — Vercel env-var matrix table: rename the `NEXT_PUBLIC_TEAM_SUPABASE_*` rows to `NEXT_PUBLIC_SUPABASE_*`.
3. **Sweep**: grep entire repo for `NEXT_PUBLIC_TEAM_SUPABASE` — fix any other stragglers.

### Verify

Grep after fixes: `grep -rn "NEXT_PUBLIC_TEAM_SUPABASE" apps/ docs/` should return zero hits.

## Acceptance criteria

1. Zero `NEXT_PUBLIC_TEAM_SUPABASE_*` references anywhere in repo (code + docs)
2. `/api/team/health` route reads from `NEXT_PUBLIC_SUPABASE_URL` (matches everywhere else)
3. Runbook Vercel matrix table has consistent names matching `.viabe/secrets/supabase-dev.env`
4. No regression on `lib/supabase-client.ts` or `lib/ops/stream.ts` (load-bearing — Fazal-style smoke check post-merge)
5. Vercel env vars set per the corrected runbook will populate all consuming paths

## Rule #15 Canary

`/api/team/health` is the only changed runtime code. Existing tests for health endpoint (if any) should still pass. Add or extend a test that asserts the route returns the URL value when `NEXT_PUBLIC_SUPABASE_URL` is set.

If no existing test for health endpoint: add `apps/team-web/app/api/team/health/route.test.ts` with the env-set/env-unset matrix (2 assertions).

## Out of scope

- Renaming the env file (already correct)
- Anything not env-var-naming-related
