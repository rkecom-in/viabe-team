## What CoderC must produce
Four distinct edge-case handlers, each triggered by a classifier intent. The webhook (VT-3.3) routes; this subtask implements each handler.
1. **Exclusion request handler** at `apps/team/src/owner_surface/edge_cases/exclusion.py`:
	- Triggered by intent `exclusion_request`.
	- Owner says: "exclude customer 9876543210" / "customer 98765 ko exclude karo, woh angry hai" / "don't message Rajesh again" / etc.
	- Classifier (VT-5.11) extracts customer phone or name + optional reason.
	- Handler resolves the customer via tenant-scoped lookup (phone match if provided; fuzzy name match if name only); on ambiguous match (multiple customers fit name), asks via VT-6.2 clarifying flow.
	- On match: sets `customers.opt_out_status='owner_excluded'`, `opt_out_reason=<owner_reason>`. The customer remains in the ledger but campaigns skip them.
	- Note: this is OWNER-side exclusion (different from CONSUMER-side opt-out in VT-8.5 which uses `opt_out_status='opted_out'`). The two flags carry different semantics for audit.
	- Sends confirmation via free-form (24h window assumed since owner just messaged) or template fallback if window closed.
2. **Adhoc campaign request handler** at `apps/team/src/owner_surface/edge_cases/adhoc_campaign.py`:
	- Triggered by intent `adhoc_campaign_request`.
	- Owner says: "send a campaign now to my dormant customers" / "festive offer for everyone who came in last 30 days" / etc.
	- Handler queues an immediate orchestrator dispatch (not waiting for next Monday's weekly cadence) with `trigger_reason='owner_initiated'`. The agent (VT-4) produces a `CampaignPlan`; the standard weekly approval flow takes over (the campaign goes through VT-9.3 approval like any other).
	- Per-week budget impact: an adhoc campaign counts against the per-week budget of 2 (combined with weekly cadence requests). If budget exhausted, the handler responds: "You've used both campaign requests this week. Try again next Monday or earlier if you want to skip this week's regular campaign."
3. **Status query handler** at `apps/team/src/owner_surface/edge_cases/status_query.py`:
	- Triggered by intent `status_query`.
	- Owner says: "how many customers do I have?" / "what was last campaign's result?" / "how many opt-outs this month?"
	- Handler routes the specific query to a small templated response set (NOT the agent — these are simple SQL aggregations):
		- Customer count → `query_customer_ledger` count.
		- Last campaign → `get_recent_campaigns` first result with attribution.
		- Opt-out count → SQL on `customers.opt_out_status`.
		- Trial / billing status → `get_business_profile` phase + dates.
		- Other status questions → fallback: "For detailed answers, check your Viabe Team portal at [viabe.ai/team/dashboard](http://viabe.ai/team/dashboard)."
	- Free-form WhatsApp response with the answer.
	- This handler does NOT invoke the agent; it's deterministic templated SQL response. CodeX confirms.
4. **Template error followup handler** at `apps/team/src/owner_surface/edge_cases/template_error_followup.py`:
	- Triggered by intent `template_error_followup`.
	- Owner says: "the message I got didn't make sense" / "the template was wrong" / "bug in your message"
	- Handler creates a `template_error_report` row (migration `030_template_error_reports.sql`) with the owner's complaint + the most recent outbound template_id within the past 7 days from this tenant's campaigns.
	- Auto-acknowledges the owner: "Thanks for flagging that. Fazal will review and get back to you."
	- Sends Telegram alert to Fazal with the report details.
5. **Tests** at `apps/team/tests/owner_surface/test_edge_cases.py`:
	- Exclusion: "exclude 9876543210" → customer flagged owner_excluded.
	- Exclusion ambiguous name: "don't message Rajesh" with 3 Rajeshes → clarifying flow triggered.
	- Adhoc happy path: "send campaign now" → orchestrator dispatch queued.
	- Adhoc budget exhausted: 3rd request in same week → declined with friendly message.
	- Status: "how many customers?" → templated count response.
	- Status fallback: random unsupported question → fallback portal mention.
	- Template error: complaint logged, Telegram alert fired, owner acknowledged.
	- All Hindi variants pass.
	- Cross-tenant: tenant_A's exclusion can't affect tenant_B.
## Architectural rules to enforce in this subtask
- Pillar 7: owner-driven exclusions are authoritative; agent / system can't reverse them.
- Pillar 8: one router for edge cases; per-intent handlers in their own files. CodeX rejects PRs that pile multiple intents into one handler.
- Status queries are deterministic; CodeX rejects PRs that route status to the agent (cost + quality concern).
- Owner-excluded customers carry `owner_excluded` flag distinct from consumer opt-out (`opted_out`). Audit + reconstitution behavior differ.
- Adhoc campaigns count against per-week budget alongside weekly cadence (VT-9.3).
- Template error reports go to Fazal directly (no automated triage in Phase 1).
