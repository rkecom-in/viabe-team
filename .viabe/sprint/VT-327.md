# VT-327 — kg_events outbox in DSR purge

The VT-82 ultracode review surfaced a PRE-EXISTING broader gap (Cowork 20260605T063000Z):
the `kg_events` transactional outbox (migration 082) is durable — the drain only stamps
`drained_at`, never deletes the row — and is NOT in `dsr_purge._PURGE_ORDER`. The inbound
path `create_tenant_if_unknown` emits `business_name` (owner subject data) into the
TENANT_CREATED payload, so that PII SURVIVES a tenant DSR indefinitely. (VT-82's own
signup emit is already PII-free — business_type only — so this is the inbound-path /
broader-outbox gap, not VT-82's.)

## Scope
1. Add `kg_events` to `dsr_purge._PURGE_ORDER` (FK-correct: check `kg_events_processed`
   ledger ordering — purge the child first if it FKs kg_events) so a tenant DSR hard-
   deletes the tenant's outbox rows. (Alternative considered: purge-on-drain — but
   DSR-sweep is the right home, consistent with episodic_events/platform_listings.)
2. Real-PG DSR canary: seed a kg_events row (PII-bearing payload), purge the tenant,
   assert it's hard-deleted + a co-resident tenant's kg_events untouched.

## Defense-in-depth follow-up (optional, flag): minimal TENANT_CREATED payload on the
inbound create_tenant_if_unknown path too (drop business_name, like VT-82 did).
