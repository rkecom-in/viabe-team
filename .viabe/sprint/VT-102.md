## What CoderC must produce
The `pipeline_log` table: append-only structured event store. Every event in the orchestrator, agent, tool, webhook, scheduled trigger, refund flow, etc. writes a row. PII redacted at write time. Tenant-scoped where applicable. Indexed for typical query patterns.
1. **Schema** at migration `038_pipeline_log.sql`:
	- Table `pipeline_log`:
		- `id` UUID PK
		- `run_id` UUID NOT NULL
		- `tenant_id` UUID NULL (workspace-level events have NULL)
		- `event_type` TEXT NOT NULL (categorical taxonomy: `webhook_received`, `webhook_signature_verified`, `agent_dispatched`, `tool_invoked`, `tool_completed`, `db_write`, `external_api_call`, `external_api_response`, `error`, `phase_transition`, `scheduled_trigger_fired`, `delivery_attempted`, `payment_event`, `consent_event`, ~30 total)
		- `severity` TEXT (debug / info / warn / error / critical)
		- `component` TEXT (which module emitted: `orchestrator`, `agent.sales_recovery`, `tool.send_whatsapp_message`, etc.)
		- `payload` JSONB (event-specific structured data, PII-redacted)
		- `duration_ms` INT NULL (for events with measurable duration like external API calls)
		- `created_at` TIMESTAMPTZ DEFAULT now()
	- Indexes: `(run_id, created_at DESC)`, `(tenant_id, created_at DESC) WHERE tenant_id IS NOT NULL`, `(event_type, created_at DESC)`, `(severity, created_at DESC) WHERE severity IN ('error', 'critical')`.
	- RLS: DENY UPDATE/DELETE (append-only). SELECT scoped: tenant-bound rows visible to tenant's owner; workspace-level via service role only.
	- Retention: 90 days; nightly delete of older rows. Post-delete, summary statistics retained in `pipeline_log_aggregates` (Phase 2).
2. **Writer** at `apps/team/src/observability/log.py`:
	- `log_event(event_type, run_id, tenant_id, severity, component, payload, duration_ms=None) -> None`. The sole writer.
	- Async fire-and-forget: writes happen on a background task; logging never blocks the calling code.
	- On write failure: logs to stderr with structured marker; writes a sentinel row to a separate `pipeline_log_failures` table (Phase 2; Phase 1 stderr-only acceptable).
	- Schema-validates payload against event_type-specific schema (defined in `apps/team/src/observability/event_schemas.py`). Invalid payloads logged but written with `payload_validation_failed=true` flag.
3. **PII redaction at write time**:
	- `payload` passes through `apps/team/src/privacy/pii_redactor.py` (the same module VT-12.4 builds).
	- All phone-like patterns hashed via E.164 hash from VT-7.1.
	- All message bodies (campaign content, owner replies, customer messages) hashed.
	- Customer names tokenized (e.g., `"Rajesh Kumar"` → `"<customer_name>"`).
	- CodeX rejects PRs that bypass the redactor.
4. **Query API** at `apps/team/src/observability/query.py`:
	- `query_run(run_id) -> list[PipelineLogEvent]`: all events for a run, ordered chronologically.
	- `query_tenant_recent(tenant_id, since, limit=100) -> list[PipelineLogEvent]`: tenant's recent events for owner-portal-style debugging.
	- `query_errors_recent(since, severity_min='error', limit=100) -> list[PipelineLogEvent]`: workspace-level error scan; service-role only.
	- `query_event_type(event_type, since, limit=100) -> list[PipelineLogEvent]`: type-specific scan.
5. **Coverage requirements**:
	- Every external API call: `external_api_call` event before, `external_api_response` after, with duration_ms.
	- Every DB write: `db_write` event with table_name + operation_type (insert/update/delete) + tenant_id.
	- Every error: `error` severity event with stack trace (redacted) + error_class + recovery_action.
	- Every phase transition: `phase_transition` event with from_phase + to_phase + reason.
	- Every scheduled trigger fire: `scheduled_trigger_fired` event.
6. **Tests** at `apps/team/tests/observability/test_pipeline_log.py`:
	- Append-only enforcement: synthetic UPDATE/DELETE attempt → RLS-blocked.
	- PII redaction: payload with phone → hashed in row.
	- Query API: synthetic 50 events for run_id → query_run returns chronologically ordered.
	- Cross-tenant: tenant_A's owner can't query tenant_B's events.
	- Workspace-level events: tenant_id NULL → not visible to tenant owners; visible via service role.
	- Schema validation: invalid payload for known event_type → logged with validation_failed flag, not rejected outright.
	- Performance: 10K events/min sustained insert rate; query with run_id index returns <100ms.
	- Async fire-and-forget: log call returns before write completes; calling code unblocked.
## Architectural rules to enforce in this subtask
- Pillar 7: append-only structural via RLS.
- Pillar 8: one log table; one writer; one query API. CodeX rejects shadow loggers.
- PII redaction non-negotiable. CodeX rejects raw-PII writes.
- Schema validation soft (logs warning, doesn't reject) so observability remains best-effort during code drift.
- 90-day retention is a Type 2 commitment.
- Async fire-and-forget; logging cannot block.

## Canary (Rule #15 — mandatory)

Per `docs/clau/discipline-rules.md` §Rule #15 (Fazal-Standing 2026-05-25), this row ships with a real-API canary that lives at `apps/team-orchestrator/canaries/vt102_pipeline_log.py`. The canary is NOT a pytest test — it is invoked explicitly with the env loaded; CI does not auto-run it.

The canary must source `.viabe/secrets/supabase-dev.env` in a subshell and hit the real Supabase dev DB. No mocks. No skips — a missing key, no network, or DB-down condition is a canary FAILURE, never a SKIP.

### Seven required assertions (all must PASS; any fail = canary FAILED)

1. **Real INSERT succeeds.** Call `log_event(event_type='canary_test', run_id=<fresh uuid4>, tenant_id=<test tenant uuid>, severity='info', component='canary', payload={'k': 'v'})`. Then direct SQL `SELECT * FROM pipeline_log WHERE run_id = <that uuid>` returns the row. Assert row exists AND every column has the expected value.

2. **PII redaction at write time.** Build a payload containing a phone `+919876543210` in a string field, a `customer_name` key with value `Rajesh Kumar`, and a nested `body: 'Hi I want to cancel'`. Call `log_event` with it. Direct SQL SELECT the row's `payload` JSONB. Assert:
   - `+919876543210` is absent from the JSONB literal
   - A hash/token form is present in its place (e.g., `phone_tok_<hex>`)
   - `customer_name` value is NOT `Rajesh Kumar` literally; redaction marker present
   - `body` value is NOT `Hi I want to cancel` literally; hashed form present
   These are positive assertions — assert raw absent AND redacted present.

3. **RLS enforces cross-tenant blocking.** Open a DB connection with `SET app.current_tenant_id = '<tenant_A uuid>'`. Call `log_event` writing a row tagged tenant_A. Open a second connection with `SET app.current_tenant_id = '<tenant_B uuid>'`. Execute `SELECT * FROM pipeline_log WHERE run_id = <that uuid>`. Assert: 0 rows returned. Then switch back to tenant_A's session var and SELECT again: 1 row returned. Both must hold.

4. **Indexed queries return quickly.** Insert ~100 synthetic rows for one run_id over a 5-minute window (vary `event_type`, `severity`, `created_at`). Call `query_run(run_id)`. Assert: returns all ~100 rows in chronological order in under 100ms wall-clock. Use `time.perf_counter()`; report observed milliseconds in canary output.

5. **Append-only enforcement.** As a non-service-role connection (the tenant `app_role` session), attempt `UPDATE pipeline_log SET payload = '{}'::jsonb WHERE id = <existing id>`. Assert: raises `permission denied for table pipeline_log` (or equivalent Postgres error). Attempt `DELETE FROM pipeline_log WHERE id = <existing id>`. Assert: same. The privileged retention sweep (next assertion) still works via service role — that's the design.

6. **Failure isolation.** Patch the writer's DB connection to a deliberately broken URL (`postgres://broken:broken@nowhere:1/missing`). Call `log_event(...)`. Assert: the call returns without raising to the caller (orchestrator pipeline survives observability failures per the brief's writer spec). Assert stderr captured a structured warning marker. Assert NO row was written.

7. **Retention sweep honors 90 days.** Insert 3 synthetic rows with `created_at` set to (a) 91 days ago, (b) 89 days ago, (c) now. Invoke the retention sweep job manually (the function the nightly cron will call). Assert: row (a) is deleted; rows (b) and (c) remain.
