# VT-519 — welcome non-delivery: premise corrected + silent failures instrumented

## Reconciliation (Rule #14 — the dispatched root-cause is NOT supported by the code)
Cowork's task said the welcome routes `compose_owner_output(intent="welcome") → customer_send.agent_send_draft → Gate-0 → ownership_verified`. Exhaustive trace says otherwise:
- **Signup welcome is UNGATED:** `run_signup` → `_default_welcome` (signup.py:508) → `owner_surface.send_owner_template` → `twilio_send.send_template_message` (`team_welcome2`, Meta-approved). No `is_agent_eligible`/ownership anywhere on this path.
- **`is_agent_eligible` (the VT-517 gate) lives ONLY on customer-facing paths:** customer_send, customer_send_choke, customer_inbound, campaign/execute. The owner paths (signup welcome, owner_send, dispatch_brain owner reply in runner.py/dispatch.py) never call it.
- `compose_owner_output_tool` only COMPOSES; it does not send.
⇒ Building the prescribed "exempt owner messages from the ownership gate" would fix a NON-bug and leave the welcome still broken. NOT built.

## Two other candidates checked on deployed dev
- **dev_send_guard (VT-476):** mocks non-allowlisted dev sends. RULED OUT — booleans on dev: `EXPECTED_ENV_is_prod=False`, `allowlist_nonempty=True`, `fazal_3267_allowlisted=True`. 3267 gets a REAL Twilio send, not a mock.
- ⇒ The welcome to 3267 reaches real Twilio and fails/doesn't-deliver THERE (template content_sid resolution on the dev Twilio acct, a 4xx, a `TEAM_TWILIO_FROM_NUMBER` KeyError raise, or accepted-but-Meta-undelivered). The 18:17 failure was SILENT: `_default_welcome` only `logger.warning`s and `send_template_message` persists no record → nothing in the VT-515 feed, nothing queryable.

## What shipped (the real, in-scope fix: make the silent failure OBSERVABLE — Cowork item #3)
1. **signup.py** welcome block: on `sent == False` OR a raising send, emit a first-class `debug_event` (`operation=welcome_not_delivered` / `welcome_send_raised`, `impact=owner_no_welcome`, `vendor=twilio`). The next signup's welcome failure is now diagnosable in the VT-515 feed with the actual reason.
2. **customer_send.py** Gate-0 skip: the `SKIP_NOT_ONBOARDED` skip (incl. VT-517 ownership=false) was SILENT (audit-blocker #1/#5). Emit a fail-soft `debug_event` (`operation=gate0_skip_not_onboarded`) carrying the SPECIFIC unmet prereqs (`unmet_prerequisites`). Never raises into the gate; the skip is unchanged.

Both emits are fail-soft (lazy import, try/except) — observability never blocks a send/gate.

## ROOT CAUSE FOUND (2026-07-01, Cowork task 184500Z — mined existing evidence, no re-run)
Reached the sealed Twilio creds via `railway ssh` (run ON the container — creds by-reference, never echoed) and queried the Message record for the 18:17 welcome:
- **2026-06-30 18:17:06Z → to ..3267, from ..4223, status=undelivered, err=63049.** Same pattern every welcome (12:03:43Z too); the message ~20s prior each time (OTP/verify) **delivered**. So WABA sender + number are fine — the `team_welcome2` template specifically is rejected.
- **Twilio 63049 = "Meta chose not to deliver this WhatsApp marketing message."** Meta throttles/declines MARKETING-category template delivery to low-engagement/new recipients (and bans it entirely for US numbers since 2025-04-01). team_welcome2 is categorized **Marketing** on Meta; the post-signup welcome is **transactional** → it must be **Utility**.

### The fix (Meta-side — NOT a repo/Railway change; CC has no Twilio/Meta console per CL-431)
Recategorize `team_welcome2` to **Utility** on the Twilio/Meta console (or approve a Utility-category welcome template + hand CC the content_sid — CC then points `_default_welcome` at it). The VT-404 "invites your reply" copy likely tripped Meta's marketing classifier; a plainly-transactional utility welcome delivers to first-contact users.

### Observability follow-up (separate; 63049 is ASYNC)
Twilio ACCEPTS the send (success=True at send-time) then marks it undelivered via status callback — so the VT-519 send-time emit can't see 63049. The real fix is a Twilio **StatusCallback** handler that records delivery-failed/undelivered (+error_code) to debug_events. Recommend a follow-up VT.

### Separate bug surfaced in the 18:17 logs (follow-up)
`tm_audit_log insert failed (fail-soft): ForeignKeyViolation ... Key (tenant_id)=(6467fde6-...) is not present in table tenants` — the conversational agent (team_manager/memory_write) emits a tm_audit row with a tenant_id absent from tenants (a pre-create/placeholder context). Fail-soft so it harms nothing, but it means tm_audit silently drops those rows. Recommend a follow-up VT.
