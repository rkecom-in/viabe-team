# VT-22 — INTERNAL_API_SECRET wiring across services with constant-time compare

## Parent task
VT-Foundation.

## Concept reference
- `concept-team.md` §10 (credential discipline: 64-char shared secret, different per environment, constant-time compare, never logged).
- `concept-team-pillars.md` Pillar 8 (no patchwork; secret handling structural, not retrofit).
- Reports' `INTERNAL_API_SECRET` pattern from `Viabe_AI_Technical_Reference_v2_0.md` §2.3 — the same discipline applies, scoped to Team services.

## Dependencies
- VT-2.5 Done (Vercel + Railway environments exist; secrets can be set per environment).
- VT-2.1 Done (`apps/team-web/`, `apps/team/` exist; their config layers are where the secret is consumed).

## What CoderC must produce
The internal-API-secret discipline shared across Team services. Two distinct secret values (dev vs prod). Constant-time comparison wherever the secret is checked. Never logged. The secret guards every internal API call between `apps/team-web/` and `apps/team/`, plus any future inter-service calls.
1. **Generate two 64-character random secrets**: one for dev, one for prod. Use `openssl rand -hex 32` or equivalent. Store in 1Password's `Viabe Team` vault as `INTERNAL_API_SECRET_DEV` and `INTERNAL_API_SECRET_PROD`. The two values must NEVER be the same; CI grep enforces this in VT-2.7.
2. **Configure environment variables** (no code in this step — just env settings):
	- Vercel `viabe-team-web` Preview/Development env: `INTERNAL_API_SECRET = <dev value>`.
	- Vercel `viabe-team-web` Production env: `INTERNAL_API_SECRET = <prod value>`.
	- Railway `team-orchestrator-dev`: `INTERNAL_API_SECRET = <dev value>`.
	- Railway `team-orchestrator-prod`: `INTERNAL_API_SECRET = <prod value>`.
3. **Constant-time comparison helper** in `packages/team-shared/`:
	- TypeScript: `packages/team-shared/src/auth/compare-secret.ts` exporting `compareInternalSecret(provided: string): boolean`. Uses `crypto.timingSafeEqual` with explicit length-padding. Returns false (not throws) on mismatch, length difference, or missing env var.
	- Python: `packages/team-shared/python/team_shared/auth/compare_secret.py` exporting `compare_internal_secret(provided: str) -> bool`. Uses `hmac.compare_digest`. Same return semantics.
	- Both helpers read the secret from env at call time, NOT at module load time, so test-time env mutation works.
4. **Middleware/decorator**:
	- Next.js: a route middleware in `apps/team-web/src/middleware.ts` (or per-route handler) that intercepts requests to `/api/internal/*` paths. Reads `x-internal-secret` header, compares via the helper, returns 401 on mismatch. Logs the failure with run_id (no secret contents) to `pipeline_log` (table from VT-2.3). On match, passes through.
	- Python: a FastAPI/orchestrator dependency in `apps/team/src/auth/internal_secret.py` that does the equivalent for inbound HTTP routes prefixed with `/internal/`.
5. **Logging discipline**:
	- Add a logging filter to both apps that scrubs any string matching the 64-char hex pattern from log output. Belt-and-braces: even if a developer accidentally `console.log`s the env var, the filter catches it.
	- Document the rule in `docs/runbooks/secrets-handling.md`: `INTERNAL_API_SECRET` MUST NEVER appear in logs, error messages returned to clients, or response bodies. Loud failure modes (5xx with redacted message) are correct; quiet leaks are catastrophic.
6. **Negative tests**:
	- `apps/team/tests/security/test_internal_secret.py`: missing header → 401, wrong value → 401, correct value → 200. Plus a timing test: 100 calls with wrong-but-progressively-closer secrets show no measurable timing variance (within tolerance).
	- `apps/team-web/__tests__/internal-secret.test.ts`: same coverage on the Next.js side.
	- Log-scrubbing test: emit a log line containing the secret value, capture the output, assert the secret does not appear.
7. **Rotation runbook entry** in `docs/runbooks/secrets-handling.md`: how to rotate the secret without downtime. Process: generate new secret → update prod env in Vercel + Railway → redeploy both services in lockstep — there is no "warm rollover" window because the comparison is exact. Plan a maintenance window of ≤5 minutes for prod rotation. Rotation owner: Fazal. Rotation frequency: every 90 days OR immediately on suspected leak.

## Architectural rules to enforce in this subtask
- Pillar 8 (no patchwork): the secret is compared via the shared helper, not by ad-hoc string equality. If a future PR introduces `if (req.headers['x-internal-secret'] === process.env.INTERNAL_API_SECRET)`, CodeX rejects on review and the PR adds a lint rule.
- Constant-time comparison is non-negotiable. `===` (TS) and `==` (Python) on the secret are forbidden in code. CI grep flags violations.
- The secret is read from env at call time. Hard-coding the secret as a module-level constant defeats hot rotation.
- Different values for dev vs prod is structural, not optional. CI fails if `INTERNAL_API_SECRET_DEV == INTERNAL_API_SECRET_PROD` in the 1Password vault export (verified via Pipeline Engineer's checklist; not stored in repo).
- Log scrubbing is a backstop, not a primary defense. The primary defense is never logging the secret in the first place.
- 401 response on mismatch returns NO information about why: not "wrong secret," not "missing header," just "unauthorized." The structured `pipeline_log` event captures the diagnosis for ops review.

## Acceptance criteria
- Two distinct 64-char hex secrets exist in 1Password's `Viabe Team` vault (`INTERNAL_API_SECRET_DEV`, `INTERNAL_API_SECRET_PROD`).
- Vercel and Railway env vars set in all four environments (preview/prod for web, dev/prod for services).
- `packages/team-shared/src/auth/compare-secret.ts` and `packages/team-shared/python/team_shared/auth/compare_secret.py` exist; both use constant-time comparison; both read env at call time.
- Next.js middleware on `/api/internal/*` enforces the secret; Python FastAPI dependency on `/internal/*` enforces it.
- Negative tests pass: missing header, wrong value, correct value behave correctly. Timing test shows variance within tolerance.
- Log-scrubbing filter active in both apps; test confirms accidental log emissions are scrubbed.
- `docs/runbooks/secrets-handling.md` documents rotation procedure; Fazal is named rotation owner.
- A deliberately-broken commit using `===` for secret comparison fails CI lint (negative test on the lint rule).
- PR title references `(VT-Foundation)`; reviewed by CoderX (mandatory — secret-handling is security-critical).

## Out of scope
- Public-facing auth (Twilio webhook signature verification, Razorpay webhook signature) — those have their own per-vendor secret patterns.
- Supabase JWT auth — different mechanism, lives at the database layer.
- Frontend→backend auth for the owner portal — that's `viabe.ai/team` user auth, separate flow in VT-9.7.
- Secret rotation automation — manual rotation in Phase 1 per the runbook; automation is a Phase 1.5 ask if needed.
- Audit logging of all secret comparisons — only failures are logged (success path is high-volume and not interesting).
