# VT-185 — DSR-purge coverage for pipeline observability tables

## Why this scope was rewritten (v1.0 → v2.0)

v1.0 of this row called for "nightly aggregation: >90d pipeline_steps → Pipeline_Run_Summary + drop source rows" per the old design-doc §2.1 retention spec. That spec was rooted in **CL-21 (2026-05-12) — Step record retention 90 days then aggregated**, which is now **SUPERSEDED** per `docs/clau/decisions-ledger.md` line 23: *"retention now governed by LOCKED standing privacy decisions"* (the CL-390 privacy cluster).

The new retention regime is **lifetime-of-relationship retention, no time-based deletion** (CL-416, filed 2026-05-26 alongside this rewrite; matches the CL-330 pattern for owner_inputs). Under that regime:
- **No nightly 90-day aggregation-and-drop.** Step records persist for the lifetime of the tenant relationship.
- **DSR-purge becomes the ONLY deletion path** for pipeline observability tables, just as it is for owner_inputs + L1 KG vector rows per CL-390's "Consequence to track" clause.

The old v1.0 scope is incompatible with the locked privacy posture and **would have caused a compliance regression** if shipped (deleting customer step records on a 90-day timer with no DSR-driven retention exception is a posture mismatch with the privacy notice). This row now sets up the correct deletion path.

## Scope (v2.0)

Per CL-416 + CL-390 + the DSR-purge architecture (`scripts/dsr_purge.py` + PR #51):

1. **Extend `_PURGE_ORDER` in `scripts/dsr_purge.py`** to cover the 3 pipeline observability tables:
   - `pipeline_steps` (delete by tenant_id + customer_id linkage via phone_token_resolutions)
   - `pipeline_runs` (delete by tenant_id + run_id linkage from pipeline_steps; or directly if a customer_id surfaces on pipeline_runs in VT-187)
   - `phone_token_resolutions` (delete by tenant_id + token where token resolves to the DSR-subject phone)

2. **Deletion ordering matters** — phone_token_resolutions deletes LAST so that pipeline_steps purge can use the token→phone lookup during purge. Or implement a 2-pass: identify all subject tokens via phone_token_resolutions; delete pipeline_steps/runs referencing those tokens; THEN delete phone_token_resolutions.

3. **Audit log** — every DSR purge writes to `privacy_audit_log` (existing, migration 008) with the table, row count, tenant_id, DSR ticket ID, timestamp. (Aligns with VT-184's per-resolution audit pattern.)

4. **Dry-run mode** — `dsr_purge.py --dry-run --ticket <id>` returns the count of rows that WOULD be purged across all 3 tables without committing.

5. **Idempotency** — re-running a DSR purge after partial failure must be safe (only purge rows not already purged).

## Out of scope (deferred / NOT this row)

- **Compaction/rollup for query performance** — if Ops UI queries get slow on full-history retention, a separate row VT-19X can add a materialized rollup table WITHOUT deleting source rows. That's a performance optimization, not a retention/privacy concern. **DO NOT bundle into this row.**
- **Cross-tenant data deletion (e.g., tenant offboarding)** — separate concern; the tenant-deletion path lives in VT-179X (TBD if not already filed).
- **Anthropic retention coordination** — Anthropic's own retention policy on data sent to their API is governed by the contract with them + CL-390; not this row's scope.
