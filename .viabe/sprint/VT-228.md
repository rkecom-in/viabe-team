# VT-228 — Dynamic operator allowlist table

## Why

VT-203 shipped with FAZAL_OWNER_UUID hardcoded in the magic-link callback (security tightening caught in fix-2). This works for Phase 1 single-operator (Fazal). Phase 2 needs multi-operator (Cowork roles, support staff).

## Brief contract

Migration NNN_operator_allowlist.sql:
```sql
CREATE TABLE public.operator_allowlist (
  user_id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  granted_by UUID NOT NULL REFERENCES auth.users(id),
  granted_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  revoked_at TIMESTAMPTZ,
  notes TEXT
);
INSERT INTO public.operator_allowlist (user_id, granted_by, notes)
  VALUES ((SELECT id FROM auth.users WHERE email = 'fazal@viabe.ai'), null, 'Initial seed — VT-203 → VT-228 migration');
```

VT-203 callback updated:
- Replace `if (user_id !== FAZAL_OWNER_UUID) return 403`
- With `if (! exists in operator_allowlist where user_id=$1 and revoked_at is null) return 403`

Admin endpoint to grant/revoke (gated by existing TEAM_ADMIN_API_TOKEN, VT-224 pattern):
- `POST /api/orchestrator/admin/operator/grant` { user_id, notes }
- `POST /api/orchestrator/admin/operator/revoke` { user_id, reason }

## Acceptance

- Migration lands
- VT-203 callback uses table lookup, not env constant
- Admin endpoints work via VT-224 admin-token surface
- Canary `vt228_operator_allowlist.py`:
  - A1: user in allowlist → 200 + JWT issued
  - A2: user not in allowlist → 403
  - A3: revoked user → 403
  - A4: admin grant endpoint adds row + verifies
  - A5: admin revoke endpoint marks revoked_at

## Out of scope

- Per-operator role differentiation (everyone is "operator" tier; differentiation = future row)
- Audit log of operator actions (CL-220 audit-enabled flag covers separately)
