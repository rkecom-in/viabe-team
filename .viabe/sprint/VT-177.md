## Scope

1. **Migration script** at `scripts/migrate_launch_tracker.py`:
   - Reads `NOTION_API_TOKEN` from env
   - Queries Notion `POST /v1/databases/413be4ab-870d-4895-bf35-dfd579142001/query` (paginated; 45 rows expected, possibly more if Notion has accumulated milestones since CL-15)
   - For each row, extracts: Milestone (title), Milestone ID (auto-increment integer), Launch Gate, Track, Owner, Status, Target Date, Completed Date, Linked VT Tasks (text), Blocking Notes, Acceptance Check, plus page block content if any
   - Writes one `.viabe/launch-tracker/MS-<N>.md` per row, where N = Milestone ID (auto-increment)
   - Idempotent: re-running should produce byte-identical files (deterministic sort, deterministic formatting)
   - Dry-run flag `--dry-run` prints planned writes without touching disk

2. **Sprint-file-style schema** for each `.viabe/launch-tracker/MS-<N>.md`:
   ```yaml
   ---
   ms_id: MS-<N>
   title: <Milestone>
   status: <Not Started | In Progress | Blocked | Completed | Deferred>
   launch_gate: <Reports-Jun15 | Team-Jul15-Soft | Team-Aug15-Full | Foundation-Ongoing>
   track: <Vendor Approvals | Code Build | Marketing-Acquisition | Reports Bugfix | Launch Ops | Templates>
   owner: <Fazal | Claude Code | Vendor External | Clau | Shared>
   target_date: YYYY-MM-DD
   completed_date: YYYY-MM-DD | null
   linked_vt_tasks: <comma-separated string from Notion's Linked VT Tasks property>
   notion_legacy_id: <UUID>
   created: 2026-05-26
   last_updated: 2026-05-26T<HH:MM>:00+05:30
   ---

   # MS-<N> — <title>

   ## Acceptance Check
   <text from Acceptance Check property; empty if absent>

   ## Blocking Notes
   <text from Blocking Notes property; empty if absent>

   ## Status history
   - 2026-05-26: migrated from Notion (data source 413be4ab-870d-4895-bf35-dfd579142001)
   ```

3. **README** at `.viabe/launch-tracker/README.md`:
   - Schema documentation (the YAML frontmatter shape above)
   - How to query overdue + due-soon (Python pattern reading frontmatter; will be used by VT-121's alert script)
   - How to update milestone status (edit YAML, commit; same pattern as `.viabe/sprint/`)
   - Re-running the migration script (deterministic; safe to re-run if Notion shifts)
   - Why the script exists post-migration: in case we add follow-up milestones, the script proves the extraction path; not for ongoing sync

4. **Migration execution** (this PR):
   - CC runs `python3 scripts/migrate_launch_tracker.py` once locally (with NOTION_API_TOKEN provisioned in env)
   - Generated files committed alongside the script
   - PR includes both the script + the 45 (or however many) generated files

5. **Acceptance check**:
   - File count matches Notion row count (CC reports both in pre-merge-result)
   - Spot-check 3-5 milestones manually: content in repo matches Notion source
   - Schema parseability: all files parse as valid YAML + markdown via `pytest tests/test_launch_tracker_schema.py`
   - `scripts/migrate_launch_tracker.py --dry-run` is idempotent (re-run shows zero diffs)
   - Secret-leak grep on diff: clean

6. **Credential ask for CC (surface at PICKUP)**:
   - Need `NOTION_API_TOKEN` in local env (not in repo, not in CI). Cowork doesn't have it; Fazal provisions on CC's terminal OR CC reads from existing `.env.local` if it's there.
   - Cowork has Notion MCP access (different auth — Notion-side OAuth tied to Fazal's account); CC's script needs a Notion integration token, which is what NOTION_API_TOKEN should be. If CC doesn't have one already, this is a Fazal-action precondition.
