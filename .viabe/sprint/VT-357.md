## Parts + gates
1. **completed-no-send (LAUNCH-RELEVANT) — [PLAN-FIRST]:** a run ending `status='completed'`
   with ZERO owner-facing output is the same silent-drop VT-88 closed for the unresolved
   terminals, but Phase 1 keys on terminal STATUS, not the send outcome. The send-ledger
   (mig 049 `send_idempotency_keys` / `campaign_messages`) has **no `run_id` correlation**.
   DESIGN FORK for the plan: **(a) inline result-threading** — the terminal handler
   (`maybe_escalate_support` call site in `runner.py`) reads the run's ACTUAL owner-send outcome
   from the dispatch result and treats completed-no-send as unresolved-equivalent; **(b) a sweep
   + a `run_id` column migration** threaded through every send call site.
   **Cowork lean:** inline (a) IF the threaded value is the ACTUAL send outcome (not a derived
   guess); the sweep+migration (b) only if inline can't be made trustworthy.
2. **SLA enforcement — [marker-migration PRE-APPROVED, build solo when this row opens]:** 4h
   business-hours (10am–7pm IST) / 24h otherwise → a 2nd Fazal alert on breach. Needs a
   breach-alerted MARKER column on `escalations` (a migration — else every sweep re-alerts) + an
   IST-window scheduled sweep (the `scheduled_triggers.py` `@DBOS.scheduled` pattern). Unblocks
   the ack copy promising a time ("within Xh").
3. **Fazal /resolve command — [FAZAL-GATED]:** a greenfield Telegram command ingress (none
   exists today) parsing `/resolve <escalation_id> [--no-followup]` → mark the escalation
   resolved (`escalations.status='resolved'`, `resolved_by`) + `record_ops_audit` + send
   `support_resolved` to the owner (unless `--no-followup`). The `support_resolved` template is
   **NEEDS-FAZAL** (Meta Content SIDs + copy not yet provided).
