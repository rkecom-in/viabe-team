# VT-347 — get_business_profile grant-dependency note (pre-existing, defensive)

Flagged by the VT-342 RLS subagent (20260606T024500Z) — PRE-EXISTING, unchanged by VT-342, out
of scope there.

`get_business_profile` reads `tenant_connector_status` + `l1_entities` via
`_safe_query_undefined`, which catches only `UndefinedTable` (graceful-None when a table is
absent) — NOT `insufficient_privilege`. The app_role SELECT grant on those two tables relies on
mig 015's `ALTER DEFAULT PRIVILEGES` (they were created after 015), not an explicit per-table
GRANT. If that default-privilege grant were ever missing for one of them, the read would raise
`insufficient_privilege` (a 500) instead of degrading to None.

**Options:** (a) widen `_safe_query_undefined` to also treat `insufficient_privilege` as a
graceful-None (defensive); and/or (b) add explicit `GRANT SELECT … TO app_role` on
`tenant_connector_status` + `l1_entities` (belt over the default-privilege braces). LOW —
the grants are present today (the VT-342 canary + owner_dashboard read both pass).
