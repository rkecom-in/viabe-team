# VT-236 — Extend operator JWT cookie TTL 1h → 7d

## Why

Fazal directive 2026-05-29 19:25 IST. Current operator JWT cookie TTL is 1 hour. For a single-operator (Phase-1) ops console with Fazal as the only authorized user via FAZAL_OWNER_UUID allowlist, re-authenticating hourly is operationally painful.

Trade-off: longer-lived JWTs are slightly less secure (leaked-cookie attack window widens). For internal tooling with one trusted user behind a per-tenant RLS-gated stored proc, 7 days is acceptable.

## Brief contract

CC implements 4 small changes:

### 1. Cookie TTL — callback route

`apps/team-web/app/api/ops/login/callback/route.ts` line 20:
```ts
// OLD
const COOKIE_TTL_SEC = 60 * 60 // 1 hour

// NEW
const COOKIE_TTL_SEC = 60 * 60 * 24 * 7 // 7 days
```

### 2. Cookie TTL — finalize-hash route

`apps/team-web/app/api/ops/login/finalize-hash/route.ts` line 21: same change.

### 3. JWT inner exp claim — `lib/auth/operator-jwt.ts`

Find the `issueOperatorJwt()` function. Extend the `exp` claim from 1h to 7d. Keep the `iat` + `sub` + `operator_claim` shape unchanged.

If the function takes a `ttlSec` parameter, default it to `60*60*24*7`. If TTL is hard-coded, update the literal.

### 4. Update VT-203 + VT-233 canaries

Both canaries assert cookie `maxAge=3600`. Update to `maxAge=604800` (7 days in seconds).

## Acceptance

- Cookie set with `Max-Age=604800` instead of `3600` (verify via DevTools after a fresh magic-link login)
- Operator JWT inner `exp` claim is 7 days from `iat`
- VT-203 + VT-233 canaries PASS post-update
- Fresh login → cookie persists for 7 days (no forced re-login after 1h)

## Out of scope

- Refresh token mechanism (not needed at 7d TTL for single-operator phase)
- Sliding-window TTL extension on each request (not worth the complexity for phase-1)
- Multi-operator session management (covered by VT-228 dynamic allowlist)
- Cookie shape changes beyond `maxAge` (HttpOnly + Secure + SameSite=Lax + path=/team preserved)

## Scope locks

- **DO NOT** change cookie shape — only the `maxAge` and JWT `exp`
- **DO NOT** weaken the FAZAL_OWNER_UUID allowlist (VT-203 fix-2 security lock)
- **DO NOT** widen cookie path or domain
- Open-redirect allowlist on `?next` mandatory (VT-203 LOCK 1)
- Audit log entry shape unchanged (CL-220 substrate)

## Rule #15 canary

Existing VT-203 + VT-233 canaries cover the surface. New TTL assertion in canary:
- A4 / A5 update: assert `Set-Cookie` header includes `Max-Age=604800` (not 3600)

No new canary file needed.
