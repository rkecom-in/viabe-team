# VT-454 — Dev OTP harness (the auto-drive enabler)

Both the signup OTP and the ownership OTP route through the orchestrator's `auth/twilio_verify` (start/check
verification). Its **mock mode** (`TEAM_TWILIO_VERIFY_MOCK_MODE=1` → OTP = `_mock_otp()` = `VT250_MOCK_OTP` or
`123456`) is the harness — CC drives both OTP steps with the known code, no Fazal.

**Already prod-safe — NO code change needed.** `TEAM_TWILIO_VERIFY_MOCK_MODE` is in
`auth/prod_safety._PROD_FORBIDDEN_FLAGS`, and `assert_prod_safe_flags()` runs at boot (main.py): under
`EXPECTED_ENV=prod` the process REFUSES TO BOOT if the flag is on; under dev it's a no-op. VERIFIED: prod→raises,
dev→allowed. So enabling the harness can never leak to prod (fail-closed, VT-362/VT-434 guard).

**Action:** set `TEAM_TWILIO_VERIFY_MOCK_MODE=1` on Railway DEV only (done; output-suppressed per Rule-18). The
win-back SEND is UNAFFECTED (it uses the separate `TEAM_TWILIO_MOCK_MODE` send flag, left OFF — the real customer
send still fires for Fazal's sign-off). Dev OTP = `123456` for the auto-drive.
