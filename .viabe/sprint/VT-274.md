# VT-274 — ledger idempotency-key disambiguation (Cowork N1, VT-273 review)

VT-273's idempotency key `entry_key = sha256(tenant:customer:date:amount:type)`
makes re-photographing the same ledger a no-op (no double-count). KNOWN LIMITATION
(accepted Phase-1 default, Cowork N1): two GENUINELY-separate identical entries
(same customer, same day, same amount, same type) collapse into ONE — an
under-count.

Accepted because under-count << the re-ingest double-count it prevents. Future
fix (only if it matters in practice): add a stable within-source ordinal to the
key (e.g. the entry's row position within the source upload) so two real
identical entries get distinct keys while re-ingest of the same upload stays
idempotent. Needs the ingestion adapter to carry a deterministic per-entry
ordinal. Do NOT build until data shows it matters. Documented in
migrations/061_customer_ledger_entries.sql + integrations/ledger.py.
