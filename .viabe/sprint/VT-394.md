## Contract (Fazal 2026-06-13: "close the stage1 OTP open gap"). PLAN-FIRST (auth-class, CL-429).

**The gap:** `apps/team-web/lib/auth/otp-rate-limit.ts` is an in-process fixed-window Map (5 req /
15-min per-IP AND per-phone). On Vercel serverless it's **per-instance, not global** → bypassable
across instances at scale. Note (from the file): this is layered ON TOP of **Twilio Verify's own
native per-number rate-limiting + brute-force protection** — so it's a thin extra cap, not the only
defense. The file itself names the intended fix: "a shared store (`rate_limit_buckets` / Redis) is a
deliberate follow-up."

**Plan the design (CC's call — surface options in plan.md before building):**
- **Preferred direction:** reuse existing infra — a Postgres `rate_limit_buckets` table (Supabase),
  OR move authoritative enforcement to the **orchestrator's `/api/orchestrator/owner/verify-start`**
  (team-web already proxies OTP there; the orchestrator is a SINGLE long-running Railway process, so
  enforcement there is global by construction). Avoid adding a new vendor (Upstash/Vercel KV) unless
  there's a concrete reason — call it out if so.
- Atomic increment + windowed expiry; fail-closed on the limiter's own error is NOT acceptable for a
  pre-auth endpoint (don't lock out all signups if the store hiccups) — choose a sane degrade and
  state it. Keep per-IP AND per-phone, 5/15-min (or confirm the D4 numbers).
- Phone keys stay tokenized/hashed (CL-390 — never raw phone in the limiter store).

## Acceptance
- The limiter is shared/global across instances (proven: two instances/requests share the counter).
- Per-IP + per-phone enforced; phone is hashed at rest; a real over-limit test trips it.
- If a DB-backed store: migration via `scripts/migration_id_allocate.py` (never hand-pick); RLS/scope
  as appropriate; new table added to DSR `_PURGE_ORDER` if it holds any identifier.

## Merge class
Auth-class (pre-auth OTP throttle) → **plan-first + Cowork gate** (CL-429), canary if it touches a new
store/migration. NOT self-merge.
