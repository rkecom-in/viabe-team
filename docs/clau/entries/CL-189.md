# CL-189 — orchestrator_agent terminal behaviour: Option B

DECISION: VT-3.4 PR 2/3 orchestrator_agent terminal behaviour — Option B (explicit conditional edge with route function).

IMPLEMENTATION:
- Add route_after_orchestrator conditional edge from orchestrator_agent.
- Inspects last AIMessage for tool_calls; if spawn_sales_recovery present → continue to handoff path (already wired in PR #26). If no tool call → route to new orchestrator_terminal node.
- orchestrator_terminal sets terminated_without_spawn=True in state, goes to END.
- AgentGraphState gets a new field: terminated_without_spawn: bool (default False).

WHY B over A: silent fall-through is unsafe alongside PR 2/3's DBOS resume + billing + tenant isolation. Need two observable exit states in state, not parse-message-tails forensics.

WHY B over C: 'orchestrator answers without delegating' will become a legitimate Phase 2 path (greetings, FAQ, off-topic refusal, routing to specialists not yet built per Pillar 8). Encoding it as error today is a future-Clau tax.

CAVEAT (load-bearing for brief): route_after_orchestrator's tool-call detection depends on langgraph create_agent's tool-call surface, which depends on underlying chat model response format. CL-178/183 discipline applies — Context7 validation on the exact surface before brief ships. Verification target: how does create_agent expose tool_calls in the final AIMessage state? Aggregate signals (Context7 returning, docs mentioning tool_calls) do NOT substitute for confirming the specific extraction path the route function will use.

OPEN QUESTION RESOLVED: this was the deferred open question from CL-184. Now closed.
