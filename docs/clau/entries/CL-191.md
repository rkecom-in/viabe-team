# CL-191 — VT-3.4 PR 2/3 scope locked: VT-34 only

## Decision

VT-3.4 PR 2/3 scope = VT-34 SalesRecoveryContext bundle contract + bounded constructor + cross-tenant isolation test + DBOS resume + LangGraph checkpoint test + CL-188 routing additions (orchestrator_terminal node + route_after_orchestrator + terminated_without_spawn state field).

VT-71 (full Context Composer with L0+L1-L4 Mem0 integration) does NOT ship in PR 2/3. Deferred to its own dedicated PR after VT-7.1 (Mem0 substrate) lands.

## Handling missing L1/L2 substrate

VT-7.1 (Mem0 OSS L1-L3 substrate) has not shipped. L1 KG schema and L2 episodic tables do not exist on disk. The bundle constructor in PR 2/3 reads against them with safe-empty fallbacks:

- `business_profile`: empty dict `{}` when L1 Mem0 collection does not exist or returns no entries
- `customer_ledger_summary`: empty dict `{}` when L2 episodic does not exist or returns no entries
- Other L1/L2-sourced fields: same pattern
- Emit a TECH DEBT marker in the bundle output (a dedicated field like `data_completeness: dict[str, bool]`) so downstream readers know what's empty by design vs by error

When VT-7.1 ships, the same constructor reads real data without code changes. Bundle contract stays stable; only the data sources change.

## Why not pull VT-71 forward

VT-71 would add Mem0 client wiring + L0 custom integration + retrieval logic for all 4 layers. That's scope explosion: a separate sprint's worth of work being smuggled into PR 2/3.

The natural evolution is: VT-34 (contract) → VT-7.1 (Mem0 substrate) → VT-71 (composer wires them together). Each is a clean PR.

## Counter-argument considered

'VT-34 depends on VT-7 partial; if L1/L2 don't exist, the constructor reads against nothing.' True. Resolved by safe-empty fallbacks + TECH DEBT marker. The bundle contract ships now; the data fills in as substrate lands.

## Supersedes

- CL-184's forward-pointer to 'VT-71' for PR 2/3 — the artifact is VT-34, not VT-71
- VT-34 spec's 'depends on VT-7 partial' — dependency relaxed via safe-empty fallbacks

## What PR 2/3 ships

1. SalesRecoveryContext dataclass per VT-34 spec
2. build_sales_recovery_context() constructor with safe-empty fallbacks
3. 8K-token cap enforcement
4. Cross-tenant context contamination test
5. DBOS auto-resume mid-specialist test
6. LangGraph checkpoint test (state persists across resume)
7. orchestrator_terminal node (CL-188)
8. route_after_orchestrator conditional edge function (CL-188)
9. terminated_without_spawn state field (CL-188)
10. TECH DEBT marker for empty L1/L2 data, pointing to VT-7.1
