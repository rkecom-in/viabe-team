# CL-377 — CORRECTION: CL-373 closeout tracker wrongly lists DBOS Step-1b as 'NOT STARTED'.

CL-373 (CLOSEOUT TRACKER, written 2026-05-23 15:33) lists 'TEAM STEP 1b: DBOS workflow_inputs' as 'NOT STARTED — needs a Step-0 investigation before it can be scoped.' This is FALSE as of tracker-write time.

Ground truth (gh pr view 47): PR #47 'fix(privacy): purge Twilio Body from dbos.workflow_status.inputs (VT-150-fix-1)' was MERGED 2026-05-22T22:15:23Z, commit b96e08e, 1902 lines. It contains the Step-0 investigation (referenced page 368387c2-cc5a-8128-a339-c2eb87a03f82, Branch B) AND the fix. The DBOS sink work was complete ~17 hours before CL-373 was written. Whoever authored CL-373 missed PR #47.

MECHANISM of #47 (material for privacy notice): Body in dbos.workflow_status.inputs is REPLAY-CRITICAL — DBOS deserialises status['inputs'] at execute_workflow_by_id and re-invokes on recovery, so redaction-on-serialize would break recovery. Fix is therefore NOT redaction — it is a scheduled purge of terminal-state rows: 2h retention (WORKFLOW_INPUT_RETENTION_SECONDS), 30-min cadence, worst-case Body lifetime ~2.5h post-terminal. In-flight rows never deleted.

CONSEQUENCE — the DBOS sink is BOUNDED, not eliminated. Raw WhatsApp body IS persisted in plaintext in the DBOS durability layer for up to ~2.5h after a workflow terminates, then purged. The owner_inputs brief Component 2's reported claim 'raw body never persisted' remains FALSE and must be rewritten to: raw body transiently retained <=2.5h in DBOS durability layer post-terminal, then purged; never long-term, never in the structured-intent table. The privacy notice RETAINED section must disclose the <=2.5h window truthfully.

ALSO: #47 merged un-reviewed during the pre-branch-protection window (protection went live 2026-05-23 ~19:53; #47 merged 2026-05-22 22:15) — consistent with #44, not a new breach. Un-reviewed merges on main from the unprotected window now known to include #44 and #47. #47's tests are strong (substrate tests assert in-flight preservation); no re-audit called.

ACTION TAKEN: Brief 3 (DBOS Step-0) CANCELLED — would re-derive solved+merged work. Carried open item from #47 itself: conductor recovery path — deployment is local-recovery-only today so purge suffices; if DBOS conductor is wired for production, cloud-managed retention must be re-confirmed against the 2h SLA.
