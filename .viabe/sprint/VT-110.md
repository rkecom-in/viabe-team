## What CoderC must produce
Apify account approval, actor selection + verification, and rate-limit + budget configuration. Apify is the data-scraping infrastructure used by Methods 7 + 8. This subtask validates the actors against Apify's terms of service, configures rate limits, and verifies output format.
1. **Apify account setup**:
	- Fazal owns Apify account creation + billing tier selection (Phase 1: starter tier ~$49/month estimated).
	- API token issued: `APIFY_API_TOKEN_PROD` in Vercel/Railway prod env.
	- Account scoped to Viabe Team only; not shared with Reports product (separate Apify account or separate API token namespace).
2. **Actor selection + verification**:
	- **Food platform actor**: Zomato/Swiggy listing scraper. Apify marketplace has multiple options; select one with active maintenance, decent pricing per run, and ToS-compatible scraping (publicly visible listings only).
	- **Google Business Profile actor**: Apify's official Google Maps actor or a community equivalent. Restricted to publicly visible business info (name, address, hours, photos count, ratings count) — NEVER reviews-with-PII.
	- Each actor verified against Apify ToS + the source platform's robots.txt + ToS. Documented at `docs/runbooks/apify-actors.md` with rationale for each selection.
3. **Rate limit + budget configuration**:
	- Per-tenant per-week limit: 1 food-platform run + 1 GBP run. Enforced via VT-3.5 scheduled trigger budget.
	- Workspace-wide monthly budget: $200/month (~₹17,000). Enforced via Apify's per-account spend limit + monthly cost-runaway alert (extends VT-12.3 cost dashboard).
	- Per-actor-run cost: ~$0.05-0.20 typical; budgeted in `apps/team/config/model_pricing.yaml`.
4. **PII discipline verification**:
	- VT-6.10/6.11 specify context-only data ingestion; this subtask verifies Apify actor outputs match.
	- Synthetic test: run actor against a known business; output examined for any PII (customer names, phone numbers, identifiable review content). Any PII findings → actor rejected; alternative selected.
	- Documented at `docs/team/apify-pii-verification.md`.
5. **Compliance posture**:
	- Apify scraping happens at orchestrator level (server-side); no client-side scraping.
	- Owner-tenant data flow: Apify scrapes business listings the tenant has explicitly identified as theirs (e.g., "my restaurant on Zomato is at URL X"); we don't scrape competitors or random businesses.
	- This is a Phase 1 Type 3 commitment: Apify only for self-listings, not competitive intelligence.
6. **Production cutover checklist** at `docs/runbooks/apify-cutover.md`:
	- Apify API token in prod env.
	- Selected actor IDs in `apps/team/config/apify_actors.yaml`.
	- Budget alerts configured.
	- Rate limits enforced.
	- PII verification completed + documented.
	- First 10 production runs monitored via Telegram.
## Architectural rules to enforce in this subtask

- Rate limits per-tenant + workspace-wide.
