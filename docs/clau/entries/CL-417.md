# CL-417 — α-sequencing + canonical schema guardrail

## Decision

Two coupled architectural rules for pipeline observability:

**Rule A (α-sequencing):** **VT-187 schema normalization lands BEFORE VT-180 writer.** VT-180 (`write_step()`) writes canonical per-field columns (parent_step_id, tokens_input/output, status, model_used, tool_calls, step_name, decision_rationale, step_seq) from day one. Writing to JSONB envelopes pre-VT-187 multiplies the consumer-migration cost when VT-187 lands (7 confirmed consumers + Composer + the writer itself).

**Rule B (canonical schema guardrail):** The canonical schema for `pipeline_runs`, `pipeline_steps`, `phone_token_resolutions` is **the normalized per-field shape in the Pipeline Observability design doc §2.1**. The JSONB-envelope columns currently on main (`trigger_payload`, `terminal_state_metadata`) are an **interim stand-in** being converged by VT-187. **After VT-187 lands, no new envelope-only read/write paths may be added** — new fields go in canonical columns. Until VT-187 lands, new code SHOULD write canonical-shape-ready (i.e., separate envelope fields rather than blob-everything-into-JSONB).

## Why this is a Standing decision

VT-178's STEP-0 audit (2026-05-26) surfaced significant schema drift between design-doc §2.1 (per-field) and actual on-main (JSONB-driven). Clau reviewed: VT-187 path = clean substrate; β alternative (ship VT-180 with JSONB workaround, normalize later) = forced multi-site refactor across every downstream consumer. The argument *"a necessary migration is big → do it later when it's bigger"* is the failure mode this guardrail prevents.

Cowork pushed back on the schedule risk: VT-187 could balloon and slip Reports-Jun15. Fazal called α — accept the schedule risk over the late-refactor debt. Decision locked.

The canonical-schema guardrail (Rule B) is necessary because without it, β creeps back through the door even after α is chosen — engineers default to JSONB-everything-because-it's-easier, and the schema normalization gets eroded over time. The guardrail enforces the convergence.

## Why the missing per-field columns matter (Clau's analysis)

Not cosmetic:

- **`parent_step_id`** — required for Anthropic Agent SDK think→act→observe loop nesting (without it, VT-182 callback writes flat traces; replay can't reconstruct the nested loop)
- **`tokens_input/output`** — required for cost accountability per CL-322 (cost per step + per tenant + per agent rollups; without separate columns, every cost query becomes a JSONB extraction with no index)
- **`status`** — required for the query that the entire VT-123 Ops UI replay feature depends on ("show failed steps in run X"); JSONB-extraction equivalent costs full-table-scan
- **`step_name`** — required for human-readable replay (without it, the Ops UI shows step_kind enums; with it, shows "Sales Recovery: invoke Composer")
- **`tool_calls`** (separate from input_envelope) — required for per-tool-call cost rollup + tool-usage telemetry

The design doc is not blessed-JSONB; VT-187 is the path back to spec.

## Consequences across the codebase

- **VT-180 (writer)** — `depends_on` extended to include VT-187. API signature already targets canonical columns (no API change needed, just sequencing).
- **VT-181/182/183/184/186** — transitively pulled forward by VT-180's VT-187 dependency. Order: VT-178 → VT-187 → VT-180 → (VT-181/182/183 in parallel) → VT-184 → VT-186.
- **VT-179 (envelopes)** — envelope Pydantic models target the canonical shape (one model per step_kind). Validates against the field set VT-187 lands.
- **CI gate (eventually)** — VT-186 may add `gate-no-new-jsonb-envelope-only-paths` once VT-187 has landed.
- **Design doc** — does NOT get revised to bless JSONB. VT-187 converges reality back to the doc.

## How future sessions must treat this

**DO:** Treat the design-doc §2.1 per-field shape as canonical. Write code that targets canonical columns. Reference this CL when extending pipeline observability or filing follow-up rows.

**DO NOT:** Add new JSONB-envelope-only fields post-VT-187. Don't argue for β sequencing post-decision. Don't ship VT-180 before VT-187 lands. Re-litigation = bounce.

**Edge case:** If a brand-new field truly is unstructured (e.g., free-form debugging blob during incident response), it goes in `terminal_state_metadata` or `error_envelope` JSONB AS A KNOWN EXCEPTION + the row brief documents why it's not canonical. Two examples allowed; more than two → file a row to split out the canonical shape.
