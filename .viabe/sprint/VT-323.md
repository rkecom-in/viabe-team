# VT-323 — episodic_events missing from DSR `_PURGE_ORDER` (L2 PII survives tenant delete)

Found by Cowork's adversarial review of the L2 lifecycle during VT-311 (20260604T113200Z); CC-confirmed
(20260604T113600Z). **Pre-existing — NOT introduced by VT-311.** Surfaced to Fazal for the build call; rostered
Backlog pending that authorization (new scope, not part of the dev-queue grant).

## The gap (confirmed)
- `episodic_events` is **NOT** in `dsr_purge._PURGE_ORDER` (covers L1 l1_relationships/l1_entities + 11 other
  tables, but not L2). Confirmed: zero match in dsr_purge.py.
- **Zero** `DELETE FROM episodic_events` anywhere in `src/` — the only L2 mutations are VT-311 retention
  soft-delete (`deleted_at`) + VT-76 reconstitution (`referenced_entity_id`→sentinel). Neither hard-deletes.
- `episodic_events.tenant_id REFERENCES tenants(id)` has **no ON DELETE CASCADE**, and DSR ANONYMIZES the
  tenants row (does not delete it) → nothing cascades.
- **Net:** a tenant DSR-delete leaves the entire L2 episodic store behind, and `payload` CAN carry PII at rest
  (the outbox-payload pattern). L2 (VT-66 / mig 083) was added after the purge order was written.

## Scope (when authorized)
1. Add `episodic_events` to `_PURGE_ORDER` at the FK-correct position — it's a leaf (references tenants, has no
   child tables pointing at it), so DELETE near the end of the per-tenant table sweep, before the tenants-row
   anonymize.
2. Real-DB DSR-purge canary: seed a tenant + episodic rows → run the DSR purge → assert L2 rows are
   HARD-DELETED (count 0), and the unscoped-DELETE guard (VT-154 pattern) still holds.
3. Re-verify the full `_PURGE_ORDER` is FK-safe with the new entry (no orphan/violation).

## Priority / gating
High, privacy + launch-gate. **NOT Jun15-gating** (CL-422 blocks real customer data on dev until VT-231 Mumbai
prod, so not yet exploitable) but it IS a **before-real-data gate** — must close before any real tenant DSR.
