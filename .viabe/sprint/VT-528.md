# VT-528 — B5 capability-truth registry

## What
A CODE registry (`capability/registry.py`) declaring the contract of every capability the system can perform — mode / effect_class / policy_rail / verifier / rollback / prerequisites / environments. Encodes two Phase-1 theses **at import, fail-closed** (a bad declaration crashes boot, the `run_control/registry` pattern).

## Built (0 migrations)
- **Effect-class taxonomy** (net-new, the plan's): `send | db_mutation | connector | campaign | advisory`. Modes: `concierge | supervised_auto | full_auto`.
- **`CapabilitySpec`** frozen dataclass — data only, no tenant facts (CL-390); CODE not DB (product contract, versioned, no runtime-edit use-case — same call as `activation_registry`/`integrations/registry`). Env availability resolves against the caller's `EXPECTED_ENV`, not a DB toggle.
- **Import-time invariants (fail-closed):** (1) **no self-certify** — any effectful capability (non-advisory) MUST name a `verifier` registered in `VERIFIER_REGISTRY`; (2) **every effect owner-policy-gated** — effectful ⇒ `policy_rail=True`; (3) declared verifier must exist; (4) `prerequisites`, if set, must be a real `activation_registry` key (reuse, don't fork — the N1 lesson); (5) valid effect_class/mode/environments.
- **Verifier framework** — deterministic-first (mirrors `vtr_classifier`/`approval_reply`, NOT an LLM self-eval per capability). Reference verifier `real_send_evidence`: a send is verified by its TRANSPORT RECEIPT (a real Twilio message_sid, not a `MKDEV…` dev-mock SID) — the specialist's word is never the proof.
- **API:** `resolve` (fail-closed KeyError), `is_available(env)`, `requires_policy_rail`, `verify(key, evidence)` (the "evidence before completion" gate — advisory ⇒ ok, effectful ⇒ registered verifier), `all_capabilities`.
- **Seed:** `sales_recovery.winback_send` (send; verifier=real_send_evidence; policy_rail; prereq=sales_recovery) + `sales_recovery.advice` (advisory) — exercises both the effectful-must-verify and advisory-exempt paths. Registry is extensible.

## Verify
ruff clean; dep-less import OK (no smoke-trap — activation_registry chain is light); 12/12 — resolution, env gating, the send verify gate (real SID ok / MKDEV-mock fails / missing fails), advisory-ok, policy-rail lookup, and every fail-closed invariant (missing-verifier, missing-policy-rail, unknown-verifier, unknown-prereq, bad-env) + the live registry re-validated.

## NOT in scope (follow-ups, noted)
Concrete verifiers beyond the send reference (campaign/db_mutation/connector); the connector `health_check`/`test_connection` (net-new on `connectors/base.py`); wiring consumers — the send path calling `verify()` before marking a step done, OC1 reading `requires_policy_rail` (OC1/VT-533), the manager reading `is_available` for capability selection (B3). The registry ships as the declared-contract substrate + gate, ready for those consumers (activation_registry shipped the same way).
