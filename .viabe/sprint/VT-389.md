## Contract (Fazal 2026-06-13: "get the gaps closed")

**Bug (verified on dev `0874370`):** every onboarding draft-confirm emits a
`business_profile_confirmed` KG event (`onboarding/draft_profile.py:93`), but
`kg_population.py:122` `_HANDLERS` has no handler for it → `process_kg_event` returns
`"failed"` ("unknown event_type", `:163`) and writes a `failed` row to
`kg_events_processed` on every confirm. The authoritative L1 write is fine (L1 is
authoritative per the `draft_profile.py:84` comment); the emit is dead code that silently
accrues failures.

**Decide + implement (CC's call, log it):** either
(a) ADD a real `business_profile_confirmed` handler that projects the confirmed fields into
the KG (if there's a genuine KG consumer for them), OR
(b) REMOVE the dead emit + drain if L1 is the only intended sink.
Lead with (b) unless CC finds a real KG reader that needs the projection — don't build a
handler for a projection nothing consumes.

## Acceptance
- An onboarding confirm no longer writes a `failed` row to `kg_events_processed` (test the
  confirm path end-to-end, assert zero `failed` for that event_type).
- If (a): the projected entity is queryable; if (b): the emit is gone and a test asserts no
  `business_profile_confirmed` event is produced.
- No change to the authoritative L1 `business_profile` write.
