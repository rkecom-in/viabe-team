## What CoderC must produce
The inbound-message ingress. A Next.js API route at `apps/team-web/app/api/team/twilio/webhook/route.ts` that accepts Twilio's webhook POST, verifies the signature, looks up the tenant from sender's WhatsApp number, performs idempotency check via DBOS workflow ID, and starts a DBOS workflow that routes to the Pre-Filter Gate (VT-3.8).
**Architectural shift from prior framing:** webhook does NOT perform intent classification anymore. Classification was a Tier 1 thin-router concern (old VT-49); now it's absorbed into the two-stage filter (VT-3.8 deterministic pre-filter for routine ~70%, VT-3.9 orchestrator-agent brain for ~30%). The webhook's only job is ingress + tenant resolution + DBOS workflow start.
1. **Webhook endpoint** at `apps/team-web/app/api/team/twilio/webhook/route.ts`:
	- Accepts Twilio's `application/x-www-form-urlencoded` body. Fields used: `From`, `To`, `Body`, `MessageSid`, `NumMedia`, `MediaUrl0`.
	- Returns `200 TwiML` (empty `<Response/>`) on success. Twilio retries on 5xx; we NEVER return 5xx for application errors (return 200 + log + queue internal retry).
2. **Twilio signature verification**:
	- Validate `X-Twilio-Signature` header against request body + public webhook URL.
	- Use the official `twilio-python` library's `RequestValidator` (called from a Python-backed verification endpoint, OR use Twilio's TS SDK if available). No bespoke crypto.
	- Auth token from `TWILIO_AUTH_TOKEN` env (set in VT-2.5).
	- Reject 403 on mismatch.
3. **Tenant lookup**:
	- Query `tenants WHERE whatsapp_number = $From`. If multiple matches, use the most recent.
	- If no match: two paths — (a) if sender previously submitted via `signup_holding`, route to sign-up completion (VT-OwnerSurface VT-9.2); (b) otherwise, log and ignore.
4. **DBOS workflow start (replaces old 'classifier dispatch')**:
	- After tenant resolution, call into `apps/team-orchestrator` via internal API (signed with `INTERNAL_API_SECRET`).
	- The orchestrator endpoint starts a `@DBOS.workflow` decorated `pipeline_run` with deterministic workflow_id derived from `MessageSid` (idempotency: same MessageSid → same workflow_id → DBOS short-circuits if already executed).
	- The workflow's first step routes the event into Pre-Filter Gate (VT-3.8).
5. **Idempotency**:
	- Twilio retries on 5xx and timeout. DBOS workflow_id derived from `MessageSid` provides exactly-once semantics natively. No separate processed-messages table needed.
	- Additional belt-and-suspenders: also write `MessageSid` to `twilio_inbound_events` table on first handle; check on subsequent calls.
6. **Rate limiting**:
	- Per-tenant: max 30 inbound/min. Excess returns 200 + logs `rate_limit_exceeded` step_record + drops.
	- Workspace: max 500 inbound/min (catastrophic protection).
	- Both implemented as Postgres advisory locks or Redis token bucket.
7. **Tests** in `apps/team-web/tests/api/twilio-webhook.test.ts` (plus integration test in `apps/team-orchestrator/tests/orchestrator/test_webhook_ingress.py`):
	- Valid signature + known tenant → DBOS workflow started with deterministic workflow_id; pipeline_steps shows `webhook_received` step.
	- Invalid signature → 403.
	- Unknown sender → logged + ignored, 200 response, no DBOS workflow started.
	- Duplicate `MessageSid` → second call short-circuits via DBOS idempotency (first workflow's result returned).
	- Rate limit triggers correctly.
	- Image attachment (`NumMedia > 0`) flagged in workflow input for downstream ingestion routing.
	- **DBOS auto-resume test:** webhook starts workflow, simulate Railway crash mid-workflow, verify DBOS auto-resumes on restart.

## Architectural rules to enforce in this subtask
- **Pillar 1 (revised):** webhook is deterministic ingress only. NO reasoning. NO intent classification. NO prose generation. Verifies signature → resolves tenant → starts DBOS workflow. The workflow handles routing via Pre-Filter Gate (VT-3.8) and orchestrator-agent (VT-3.9).
- **Pillar 3 (tenant isolation):** tenant lookup uses the typed-wrapper pattern (VT-PrivacyArchitecture VT-8.1).
- **Pillar 8 (no patchwork):** signature verification uses Twilio's official library. Bespoke HMAC forbidden.
- The endpoint never returns 5xx for application errors. 5xx triggers Twilio retry, which causes duplicate processing.
- DBOS workflow_id derived deterministically from `MessageSid` provides exactly-once semantics. Re-deriving the same workflow_id is the contract.
- Sensitive content (PII like phone numbers in body) NOT logged plaintext. Body hashed for `pipeline_steps`; full body lives in `pipeline_runs.trigger_payload` JSONB with phone tokenization applied by VT-122 observability writer.
