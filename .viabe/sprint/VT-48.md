## What CoderC must produce
The `schedule_followup` MCP tool. Lets the agent schedule a future invocation (e.g., "follow up on this campaign in 3 days if no owner response") that the orchestrator's scheduler runs at the appointed time. Idempotent on a stable key.
1. **Tool implementation** at `apps/team/src/agent/tools/schedule_followup.py`:
	Input schema:
	- `follow_up_type: Literal['campaign_followup', 'attribution_check', 'reengagement_reminder', 'other']`
	- `fire_at: datetime` — when the follow-up should run. Must be at least 30 minutes in the future, no more than 90 days out.
	- `follow_up_key: str` — stable key for idempotency. Convention: `<follow_up_type>_<related_entity_id>` (e.g., `campaign_followup_<campaign_id>`).
	- `payload: dict` (max 4KB serialized) — structured context for the future invocation.
	- `cancel_if: list[str] | None` — list of conditions that would cancel the scheduled follow-up before firing (e.g., `['campaign_status_in:[approved, sent, cancelled]']`). The scheduler evaluates these conditions deterministically before firing.
	Output schema:
	- `status: Literal['scheduled', 'duplicate_key', 'error']`
	- `scheduled_id: UUID | None`
	- `existing_fire_at: datetime | None` (when status=duplicate_key, returns the existing row's fire_at)
	- `error_envelope: ErrorEnvelope | None`
2. **Idempotency**:
	- Composite UNIQUE constraint on `(tenant_id, follow_up_key)`. Same key twice returns `duplicate_key` with the existing fire_at.
	- The agent uses this to safely retry: "schedule a follow-up if not already scheduled" — the second call is a no-op.
3. **Scheduler invocation**:
	- The orchestrator's scheduler (VT-3.5) polls for `scheduled_followups` rows where `fire_at <= now() AND fired_at IS NULL`.
	- Before firing: evaluates `cancel_if` conditions; if any match, marks `cancelled_at` and skips.
	- Fires by re-invoking the orchestrator with `trigger_reason='scheduled_followup'` + `payload`.
	- Marks `fired_at` after successful invocation.
4. **Migration `014_scheduled_followups.sql`**: `id UUID PK`, `tenant_id`, `run_id_origin` (the run that scheduled it), `follow_up_type`, `follow_up_key`, `fire_at`, `payload JSONB`, `cancel_if JSONB`, `scheduled_at`, `fired_at NULL`, `cancelled_at NULL`, `cancel_reason TEXT NULL`. UNIQUE`(tenant_id, follow_up_key)`. RLS scoped.
5. **Validation**:
	- `fire_at` between 30 minutes and 90 days from now; otherwise `error_envelope.code='invalid_fire_at'`.
	- `payload` serialized size ≤4KB; otherwise `error_envelope.code='payload_too_large'`.
	- `cancel_if` conditions parseable by the scheduler's evaluator; otherwise `error_envelope.code='invalid_cancel_condition'`.
6. **Tests** at `apps/team/tests/agent/tools/test_schedule_followup.py`:
	- Happy path: schedule a follow-up; row exists with correct fields.
	- Idempotency: same key twice returns `duplicate_key`.
	- Future-time validation: `fire_at` 5 minutes from now → error; 100 days → error.
	- Cancel condition evaluation: synthetic scheduled row with `cancel_if='campaign_status_in:[approved, sent]'`; campaign updated to approved before fire_at → cancelled, not fired.
	- Payload size: 5KB → error.
	- Cross-tenant: tenant_A cannot schedule against tenant_B (RLS-blocked).
	- Scheduler integration: synthetic clock advance → `fire_at` reached → orchestrator invocation triggered with payload.

## Architectural rules to enforce in this subtask
- Pillar 1: tool enqueues; scheduler runs. CodeX rejects in-tool execution.
- Pillar 8: idempotency via composite UNIQUE. Same key always means same scheduled row.
- `cancel_if` is a deterministic evaluator. The scheduler reads conditions and matches against current state. CodeX rejects LLM-based condition evaluation.
- Time bounds: 30min minimum prevents accidental immediate fires; 90d maximum prevents zombie schedules. Both bounds are Type 2 to change.
- The agent specifies `follow_up_key` deliberately. This is a contract: the agent commits to using the same key when scheduling the same logical follow-up. Idempotency depends on agent discipline + UNIQUE constraint.
